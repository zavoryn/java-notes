# Redis 面试题大全

> 综合整理自 小林coding、JavaGuide、javabetter、牛客网等平台 2024-2025 年高频面试题。
> 标注频率：🔥高频 / 🔥🔥必考 / 🔥🔥🔥核心必背

## 一、Redis 基础

### 1. 什么是 Redis？🔥🔥 ⭐

Redis（**RE**mote **DI**ctionary **S**erver）是基于 C 语言开发的开源 NoSQL 内存数据库，读写速度极快。核心特性：

- **基于内存**：数据存储在内存中，读写速度纳秒级
- **KV 键值对**：存储格式为 key-value
- **丰富的数据类型**：String、Hash、List、Set、ZSet、Bitmap、HyperLogLog、GEO、Stream
- **功能全面**：支持事务、持久化、Lua 脚本、发布订阅、多种集群方案

**典型应用场景**：缓存、分布式锁、消息队列、延时队列、排行榜、计数器、Session 共享

---

### 2. Redis 为什么这么快？🔥🔥🔥 ⭐⭐

| 原因 | 说明 |
|------|------|
| **纯内存操作** | 数据在内存中，访问速度纳秒级，磁盘是毫秒级 |
| **单线程 + IO 多路复用** | 避免上下文切换和锁竞争，一个线程处理多个连接 |
| **高效的数据结构** | SDS、ziplist、quicklist、skiplist、hashtable 等针对场景优化 |
| **简洁的通信协议** | RESP 协议实现简单、解析快、二进制安全 |

**单线程为何还能 10 万 QPS？**
- CPU 不是 Redis 瓶颈，瓶颈在内存和网络
- 单线程避免了锁竞争、上下文切换开销
- IO 多路复用（epoll）让单线程同时处理大量连接

---

### 3. Redis 和 Memcached 的区别？🔥🔥 ⭐

| 特性 | Redis | Memcached |
|------|-------|-----------|
| **数据类型** | String/Hash/List/Set/ZSet 等 | 简单 KV |
| **持久化** | ✅ RDB + AOF | ❌ 纯内存 |
| **集群** | ✅ 原生支持 Cluster | ❌ 需客户端分片 |
| **线程模型** | 单线程（6.0 网络 IO 多线程） | 多线程 |
| **过期删除** | 惰性删除 + 定期删除 | 仅惰性删除 |
| **高级功能** | 事务/Lua/发布订阅/分布式锁 | 无 |

---

### 4. 为什么用 Redis 作为 MySQL 的缓存？🔥🔥 ⭐

**高性能**：Redis 基于内存，读操作直接从内存获取，速度比磁盘快几十倍。

**高并发**：Redis 单机 QPS 可达 10 万+，MySQL 单机 QPS 很难超过 1 万。

**功能丰富**：除了缓存，还能做分布式锁、限流、消息队列等。

---

### 5. 为什么用 Redis 不用本地缓存？🔥 ⭐

| 特性 | 本地缓存 | Redis（分布式缓存） |
|------|---------|-------------------|
| 数据一致性 | 多实例数据不一致 | 数据集中管理，一致 |
| 内存限制 | 受限于单机内存 | 独立部署，内存更大 |
| 数据丢失 | 宕机即丢失 | 可持久化 |
| 可扩展性 | 差 | 水平扩展（集群） |

## 二、数据类型与底层数据结构

### 6. Redis 有哪些数据类型？各自的应用场景？🔥🔥🔥 ⭐⭐

| 数据类型 | 底层实现 | 应用场景 |
|---------|---------|---------|
| **String** | SDS（简单动态字符串） | 缓存对象、计数器、分布式锁、Session |
| **List** | quicklist（3.2+） | 消息队列、文章列表 |
| **Hash** | ziplist/listpack + hashtable | 缓存对象、购物车 |
| **Set** | intset + hashtable | 点赞、共同关注、抽奖 |
| **ZSet** | ziplist/listpack + skiplist + hashtable | 排行榜、延时队列 |
| **Bitmap** | String | 签到、统计活跃用户 |
| **HyperLogLog** | 概率算法 | UV 统计（12KB 内存估算亿级基数） |
| **GEO** | ZSet + GeoHash | 附近的人、打车距离 |
| **Stream** | radix tree | 消息队列（5.0 新增，支持消费组） |

---

### 7. String 的底层实现 SDS 与 C 字符串的区别？🔥🔥 ⭐⭐

Redis 没有直接用 C 字符串，而是自己实现了 **SDS（Simple Dynamic String）**：

| 特性 | C 字符串 | SDS |
|------|---------|-----|
| 获取长度 | O(n) 遍历 | O(1) 直接读 len |
| 缓冲区溢出 | 可能溢出 | 自动扩容，安全 |
| 二进制安全 | 靠 `\0` 判断结束 | 靠 len 判断，可存二进制 |
| 内存分配 | 每次修改都 realloc | 空间预分配 + 惰性释放 |

SDS 有 5 种类型（sdshdr5/8/16/32/64），根据字符串长度自动选择以节省内存。

---

### 8. ZSet 底层为什么用跳表而不用红黑树/B+树？🔥🔥 ⭐⭐⭐

| 对比 | 跳表 | 红黑树 | B+ 树 |
|------|------|--------|-------|
| 范围查询 | ✅ 链表直接遍历 | ❌ 需中序遍历 | ✅ 叶子链表 |
| 插入/删除 | ✅ 概率平衡，简单 | ❌ 需旋转+染色 | ❌ 节点分裂合并 |
| 实现复杂度 | 简单 | 复杂 | 复杂 |
| 内存数据库适配 | ✅ 不关心 IO | ✅ | ❌ 为磁盘 IO 设计 |

**核心原因**：
1. 跳表实现简单，不需要旋转维护平衡
2. 范围查询高效（ZSet 核心操作 `ZRANGE` 等）
3. Redis 是内存数据库，不需要 B+ 树的 IO 优化
4. 内存占用可控（通过概率调节索引层数）

---

### 9. 购物车用 String 还是 Hash？排行榜怎么实现？🔥 ⭐

**购物车 → Hash**：
```
Key: cart:{userId}
Field: {商品ID}
Value: {数量}
操作：HSET/HGET/HDEL/HINCRBY
```

**排行榜 → ZSet**：
```
Key: rank:game
Score: 分数
Member: 用户ID
操作：ZADD（添加）/ ZREVRANGE（排名）/ ZREVRANK（查排名）/ ZINCRBY（加分）
```

## 三、线程模型

### 10. Redis 是单线程还是多线程？🔥🔥🔥 ⭐⭐

**核心结论：Redis 的命令执行是单线程的，但 Redis 程序本身不是单线程的。**

| 版本 | 线程情况 |
|------|---------|
| **Redis 2.6** | 主线程 + 2 个后台线程（关闭文件、AOF 刷盘） |
| **Redis 4.0** | +1 个后台线程（lazy free 异步释放内存） |
| **Redis 6.0** | +N 个 IO 线程（多线程处理网络读写，**命令执行仍是单线程**） |

**Redis 6.0 多线程**：
- 默认关闭，需配置 `io-threads 4`
- 只处理**网络 IO 读写**，不处理命令执行
- 4 核 CPU 建议设 2-3 线程，8 核建议 6 线程
- 性能提升至少一倍

---

### 11. Redis 6.0 之前为什么不用多线程？🔥🔥 ⭐

1. **单线程可维护性高**：避免并发 bug、死锁
2. **瓶颈不在 CPU**：内存操作极快，瓶颈在内存大小和网络 IO
3. **多线程开销**：上下文切换、锁竞争可能反而降低性能

## 四、持久化

### 12. Redis 持久化有哪些方式？🔥🔥🔥 ⭐⭐⭐

| 特性 | RDB（快照） | AOF（追加日志） | 混合持久化（4.0+） |
|------|-------------|----------------|-------------------|
| **原理** | 定时将内存数据以二进制写入磁盘 | 每条写命令追加到日志文件 | AOF 重写时前半部分 RDB + 后半部分 AOF |
| **数据安全** | ❌ 可能丢失两次快照之间的数据 | ✅ 取决于 fsync 策略 | ✅ 较好 |
| **恢复速度** | ✅ 快（直接加载二进制） | ❌ 慢（逐条执行命令） | ✅ 快（RDB 部分加载快） |
| **文件大小** | ✅ 小（二进制压缩） | ❌ 大（文本命令） | 中等 |
| **触发方式** | `save`（阻塞）/ `bgsave`（后台） | 每次写操作后 | AOF 重写时自动 |

---

### 13. AOF 的三种写回策略？🔥🔥 ⭐⭐

```
写命令 → AOF 缓冲区(aof_buf) → 内核缓冲区(page cache) → 磁盘
                                ↑                       ↑
                          write()系统调用           fsync()系统调用
```

| 策略 | fsync 频率 | 数据安全 | 性能 |
|------|-----------|---------|------|
| **Always** | 每次写命令都 fsync | 最高（最多丢 1 条） | 最差 |
| **Everysec** | 每秒 fsync 一次 | 最多丢 1 秒数据 | 折中（**推荐**） |
| **No** | 由 OS 决定 | 不可控 | 最好 |

---

### 14. AOF 重写机制是什么？🔥🔥 ⭐⭐

**目的**：压缩 AOF 文件，减小体积。

**原理**：不是读取旧 AOF 文件，而是**直接读取当前数据库状态**，将每个键值对用一条命令记录到新 AOF 文件。

**流程（bgrewriteaof）**：
1. 主进程 fork 子进程（利用写时复制 COW）
2. 子进程将当前数据写入新 AOF 文件
3. 主进程同时将新写命令追加到**AOF 重写缓冲区**
4. 子进程完成后，主进程将重写缓冲区内容追加到新文件
5. 新文件替换旧文件

---

### 15. RDB 快照时数据能修改吗？🔥🔥 ⭐⭐

**可以。** 使用 `bgsave` 通过 fork 子进程创建快照，利用**写时复制（COW）**：

- fork 时父子进程共享内存（只读）
- 主进程执行写操作时，被修改的页会复制一份副本
- 子进程将副本数据写入 RDB 文件
- 主进程仍然可以正常修改原数据

---

### 16. 什么是混合持久化？🔥 ⭐⭐

Redis 4.0 新增。在 AOF 重写时：
- **前半部分用 RDB 格式**写入全量数据（加载快）
- **后半部分用 AOF 格式**写入增量命令（数据丢失少）

**优点**：兼具 RDB 的恢复速度和 AOF 的数据安全性
**缺点**：AOF 文件可读性差；不兼容 Redis 4.0 之前的版本

---

## 五、过期删除与内存淘汰

### 17. Redis 的过期删除策略？🔥🔥🔥 ⭐⭐

Redis 使用**惰性删除 + 定期删除**组合策略：

| 策略 | 方式 | 优点 | 缺点 |
|------|------|------|------|
| **惰性删除** | 访问 key 时检查是否过期 | CPU 友好 | 内存不友好（过期 key 不访问就一直占内存） |
| **定期删除** | 每隔一段时间随机抽查 20 个 key，过期比例 > 25% 则继续 | 折中 | 频率难控制 |

**定期删除细节**（Redis 7.2）：
- 每次随机抽取 20 个 key
- 过期比例超过 25% 则继续抽查
- 单次执行时间不超过 25ms
- 频率由 `hz` 参数控制（默认 10 次/秒）

---

### 18. Redis 内存淘汰策略有哪些？🔥🔥🔥 ⭐⭐

当 Redis 运行内存达到 `maxmemory` 阈值时触发。

| 策略 | 淘汰范围 | 说明 |
|------|---------|------|
| **noeviction**（默认） | — | 不淘汰，写入报错 |
| **allkeys-lru** | 所有 key | 淘汰最久未使用的（**最常用**） |
| **allkeys-lfu**（4.0+） | 所有 key | 淘汰最少使用的 |
| **allkeys-random** | 所有 key | 随机淘汰 |
| **volatile-lru** | 设置了过期时间的 key | 淘汰最久未使用的 |
| **volatile-lfu**（4.0+） | 设置了过期时间的 key | 淘汰最少使用的 |
| **volatile-ttl** | 设置了过期时间的 key | 淘汰最早过期的 |
| **volatile-random** | 设置了过期时间的 key | 随机淘汰 |

> **选型建议**：
> - 热点数据明显 → `allkeys-lru` 或 `allkeys-lfu`
> - 有明确冷热数据分离 → `volatile-lru`
> - 数据都重要、不能丢 → `noeviction` + 扩容

---

### 19. LRU 和 LFU 的区别？Redis 怎么实现的？🔥🔥 ⭐⭐

| 算法 | 全称 | 淘汰依据 | 问题 |
|------|------|---------|------|
| **LRU** | Least Recently Used | 最久未访问 | 偶尔访问一次的数据长期占内存（缓存污染） |
| **LFU** | Least Frequently Used | 访问频率最低 | ✅ 解决缓存污染 |

**Redis 的近似 LRU**：
- 不维护全局链表（节省内存）
- 在对象头中记录最后访问时间
- 淘汰时随机采样 5 个 key，淘汰最久未使用的

**Redis 的 LFU 实现**：
- 对象头 24bit 的 lru 字段分为：高 16bit 存访问时间 + 低 8bit 存访问频次（logc）
- 衰减机制：随着时间推移，计数器会递减，避免旧热点数据永远不被淘汰

## 六、高可用架构

### 20. Redis 主从复制原理？🔥🔥 ⭐⭐

```
主库 (Master)                          从库 (Slave)
    │                                      │
    │  1. 全量复制（首次同步）               │
    │     → 生成 RDB 发送给从库              │
    │                                      │
    │  2. 增量复制（后续同步）               │
    │     → 发送 replication offset         │
    │        之后的写命令                    │
    │                                      │
    │  注意：主从复制是【异步】的             │
```

**全量复制流程**：
1. 从库连接主库，发送 `PSYNC` 命令
2. 主库执行 `bgsave` 生成 RDB 文件
3. 主库将 RDB 文件发送给从库
4. 从库清空旧数据，加载 RDB 文件
5. 主库将 RDB 生成期间的写命令发送给从库（通过 replication buffer）

**注意：主从复制是异步的**，不保证强一致性。

---

### 21. Redis 哨兵（Sentinel）是什么？🔥🔥 ⭐⭐

**功能**：监控主从节点、自动故障转移。

**工作流程**：
1. **监控**： Sentinel 每秒向主从节点发送 PING
2. **主观下线**：某个 Sentinel 认为节点不可达
3. **客观下线**：超过半数 Sentinel 认为主节点不可达
4. **选举新主节点**：根据优先级 → 复制偏移量 → runid 最小
5. **故障转移**：通知其他从节点复制新主节点，通知客户端新地址

---

### 22. Redis Cluster 原理？哈希槽？🔥🔥 ⭐⭐

**核心：16384 个哈希槽（Hash Slot）**

```
key → CRC16(key) % 16384 → 哈希槽编号 → 对应的 Redis 节点
```

**特点**：
- 每个节点负责一部分哈希槽
- 支持在线扩缩容（迁移哈希槽）
- 节点间通过 Gossip 协议通信
- 至少需要 3 主 3 从

**为什么是 16384 个槽？**
- 槽太多 → 节点间传播的心跳包太大
- 槽太少 → 数据倾斜时不够灵活
- 16384 是权衡的结果

---

### 23. 什么是集群脑裂？怎么解决？🔥 ⭐⭐

**脑裂**：网络故障导致哨兵认为主节点下线，选举出新主节点；旧主节点恢复后变成从节点，清空本地数据重新同步，导致脑裂期间写入的数据丢失。

**解决方案**：
```conf
# 至少有 N 个从节点连接
min-slaves-to-write 1
# 主从复制延迟不超过 N 秒
min-slaves-max-lag 10
```

不满足条件时主节点拒绝写入，避免脑裂期间数据写入旧主节点。

## 七、缓存三大问题

### 24. 什么是缓存穿透？怎么解决？🔥🔥🔥 ⭐⭐⭐

**定义**：请求的数据在缓存和数据库中都不存在，每次请求都打到数据库。

```
恶意请求 → Redis 未命中 → MySQL 未命中 → 返回 null → 下次继续穿透
```

**解决方案**：

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| **布隆过滤器** | 把所有可能存在的数据哈希到一个 bitmap，请求前先判断 | ✅ 空间小；❌ 有误判率、不能删除 |
| **缓存空值** | 查不到也在 Redis 缓存一个 null 值，设短过期时间 | ✅ 简单；❌ key 多变时浪费内存 |
| **接口校验** | 参数合法性检查，拦截非法请求 | ✅ 第一道防线 |

---

### 25. 什么是缓存击穿？怎么解决？🔥🔥🔥 ⭐⭐⭐

**定义**：某个**热点 key** 过期的瞬间，大量并发请求直接打到数据库。

```
热点 key 过期 → 大量并发请求同时发现缓存 miss → 同时查 DB → DB 压力暴增
```

**与缓存穿透的区别**：穿透是数据不存在，击穿是数据存在但缓存刚好失效。

**解决方案**：

| 方案 | 原理 |
|------|------|
| **互斥锁** | 缓存 miss 时，用 `SETNX` 抢锁，只有一个线程查 DB 并更新缓存 |
| **逻辑过期** | 缓存不设物理过期，存一个逻辑过期时间，过期后异步更新 |
| **热点 key 永不过期** | 手动管理过期和更新 |

---

### 26. 什么是缓存雪崩？怎么解决？🔥🔥🔥 ⭐⭐⭐

**定义**：大量 key 同时过期，或 Redis 宕机，导致请求全部打到数据库。

**与缓存击穿的区别**：雪崩是大量 key 同时失效，击穿是单个热点 key 失效。

**解决方案**：

| 原因 | 解决方案 |
|------|---------|
| **大量 key 同时过期** | 过期时间加随机值，避免集中过期 |
| **Redis 宕机** | Redis 集群（哨兵/Cluster）、多级缓存（本地缓存 + Redis） |
| **通用** | 服务降级、限流、提前预热 |

## 八、缓存与数据库一致性

### 27. Cache Aside 模式是什么？为什么先更新数据库再删缓存？🔥🔥🔥 ⭐⭐⭐

**Cache Aside（旁路缓存）— 最常用的缓存读写策略：**

- **读**：先读缓存 → miss 则读数据库 → 写入缓存
- **写**：先更新数据库 → 再删除缓存

**为什么不能先删缓存再更新数据库？**

```
线程 A: 删除缓存
线程 B: 读缓存 miss → 读数据库(旧值) → 写入缓存(旧值)
线程 A: 更新数据库(新值)
→ 结果: 缓存是旧值，数据库是新值 ❌
```

**先更新数据库再删缓存也可能不一致（但概率极低）：**

```
线程 A: 读数据库(旧值) [缓存刚好失效]
线程 B: 更新数据库(新值) → 删除缓存
线程 A: 写入缓存(旧值)
→ 结果: 缓存是旧值 ❌ 但这种情况发生概率极低（缓存写入远快于数据库）
```

---

### 28. 如何保证缓存和数据库最终一致？🔥🔥 ⭐⭐

| 方案 | 说明 |
|------|------|
| **延迟双删** | 先删缓存 → 更新数据库 → 延迟一段时间 → 再删缓存 |
| **监听 binlog** | 使用 Canal 监听 MySQL binlog，异步更新/删除缓存 |
| **消息队列重试** | 删除缓存失败时，发到 MQ 异步重试 |
| **设置过期时间** | 给缓存设 TTL，最终会自动失效 |

## 九、分布式锁

### 29. 如何基于 Redis 实现分布式锁？🔥🔥🔥 ⭐⭐⭐

**加锁**：
```bash
SET lock_key unique_value NX PX 10000
# NX: key 不存在才设置（互斥）
# PX 10000: 过期时间 10 秒（防止死锁）
# unique_value: 客户端唯一标识（防止误删别人的锁）
```

**解锁（Lua 脚本保证原子性）**：
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**三个必要条件**：
1. 互斥性（NX 保证）
2. 设置过期时间（PX 保证，防止死锁）
3. 唯一标识（防止误删其他客户端的锁）

---

### 30. Redis 分布式锁有什么问题？Redlock 是什么？🔥🔥 ⭐⭐⭐

**问题**：
- **锁超时**：业务执行时间超过锁过期时间，其他线程获得锁 → 两个线程同时持有锁
- **主从切换**：主节点加锁后未同步到从节点就宕机 → 从节点升为主后其他客户端也能加锁

**Redlock（红锁）算法**：
- 使用 N 个（推荐 5 个）独立的 Redis 主节点
- 客户端向所有节点申请加锁
- 超过半数（N/2 + 1）节点加锁成功，且总耗时 < 锁过期时间 → 加锁成功
- 失败则向所有节点释放锁

**Redlock 的争议**：Martin Kleppmann 和 Antirez（Redis 作者）有过著名争论。实际生产中更多使用 **Redisson** 的分布式锁（看门狗机制自动续期）。

## 十、Redis 实战

### 31. Redis 大 key 如何处理？🔥🔥 ⭐⭐

**什么是大 key**：
- String 类型 value > 10KB（严格标准 > 1MB）
- 集合类型元素 > 5000 个

**危害**：
1. 客户端超时阻塞
2. 网络阻塞（1MB key × 1000 QPS = 1GB/s 流量）
3. 删除大 key 阻塞主线程
4. 集群内存倾斜

**发现方法**：
- `redis-cli --bigkeys`
- `SCAN` + `MEMORY USAGE`
- RdbTools 分析 RDB 文件

**处理方法**：
- 拆分大 key（如 Hash 按字段拆分）
- 异步删除：`UNLINK` 代替 `DEL`（Redis 4.0+）
- 开启 `lazyfree-lazy-eviction`

---

### 32. Redis 热 key 如何处理？🔥 ⭐⭐

**发现**：`redis-cli --hotkeys`（需 LFU 策略）、MONITOR 命令（短期使用）

**处理**：
- **本地缓存**：热 key 缓存到 JVM 本地（Caffeine）
- **读写分离**：主写从读
- **分片打散**：给 key 加随机后缀，分散到多个 key

---

### 33. Pipeline 有什么用？🔥 ⭐

**问题**：每条命令都有一次网络往返（RTT），大量命令时网络延迟累积。

**Pipeline**：将多条命令打包一次发送，减少网络 RTT。

```
普通模式：命令1 → 等响应 → 命令2 → 等响应 → ...  （N 次 RTT）
Pipeline：[命令1,命令2,...] → 一次性发送 → 一次性返回  （1 次 RTT）
```

**注意**：
- Pipeline 是客户端功能，不是服务端功能
- 非原子操作（与事务不同）
- 控制单次 Pipeline 的命令数量（建议 < 500）

---

### 34. Redis 事务支持回滚吗？🔥🔥 ⭐

**不支持回滚。**

```bash
MULTI          # 开启事务
SET key1 val1  # 入队
INCR key2      # 入队（key2 不是数字，执行时会报错）
SET key3 val3  # 入队
EXEC           # 执行：key1 设置成功，key2 报错，key3 仍然执行成功
```

**为什么 Redis 不支持回滚？**
1. Redis 作者认为事务错误通常是编程错误，应该在开发阶段发现
2. 支持回滚与 Redis 追求简单高效的设计主旨不符

**替代方案**：使用 **Lua 脚本**（原子执行多条命令，但也不支持回滚）。

### 35. 如何用 Redis 实现延时队列？🔥 ⭐

**方案一：ZSet**
```
ZADD delay_queue <执行时间戳> <任务内容>
ZPOPMIN delay_queue → 检查时间戳 → 未到时间放回
```

**方案二：Redisson 延时队列（推荐）**
- 基于 ZSet 实现
- 持久化、支持 ACK
- 避免使用 Redis 过期事件监听（时效性差、可能丢消息）

## 十一、性能优化

### 36. 大量 key 集中过期怎么办？🔥 ⭐

**问题**：大量 key 同时过期，定期删除线程占用主线程，导致请求延迟增加。

**解决**：
1. 设置过期时间时加随机偏移量
2. 开启 `lazyfree-lazy-expire`，异步删除过期 key

---

### 37. 常见导致 Redis 阻塞的原因？🔥 ⭐

| 原因 | 说明 |
|------|------|
| O(n) 命令 | `KEYS *`、`HGETALL`、`SMEMBERS` 等，n 越大越慢 |
| 大 key | 读写/删除大 key 耗时长 |
| `SAVE` 生成 RDB | 阻塞主线程，应使用 `BGSAVE` |
| AOF `appendfsync always` | 每次写入都 fsync |
| `FLUSHALL` | 删除所有数据 |
| Swap | 内存不足导致操作系统换页 |

---

### 38. Redis 的使用规范？🔥 ⭐

1. 使用连接池，避免频繁创建/关闭连接
2. 尽量使用 O(1) 命令，O(n) 命令注意 n 的大小
3. 使用批量操作减少网络开销（MGET/MSET/Pipeline/Lua）
4. 避免使用 Redis 事务（用 Lua 脚本代替）
5. 给 key 设置合理的过期时间
6. 禁止长时间开启 MONITOR
7. key 命名规范：`业务名:对象名:id`
