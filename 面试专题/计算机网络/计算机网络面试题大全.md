## 一、网络体系结构

### 1. OSI 七层模型和 TCP/IP 四层模型？🔥🔥🔥 ⭐

| OSI 七层 | TCP/IP 四层 | 功能 | 协议示例 |
|----------|------------|------|---------|
| 应用层 | 应用层 | 为应用程序提供服务 | HTTP、FTP、DNS、SMTP |
| 表示层 | 应用层 | 数据格式转换、加密压缩 | SSL/TLS、JPEG |
| 会话层 | 应用层 | 建立/管理/终止会话 | RPC、SQL |
| 传输层 | 传输层 | 端到端可靠传输 | TCP、UDP |
| 网络层 | 网络层 | 路由选择、IP 寻址 | IP、ICMP、ARP |
| 数据链路层 | 网络接口层 | 帧的封装、MAC 寻址 | Ethernet、PPP |
| 物理层 | 网络接口层 | 比特流传输 | 光纤、双绞线 |

> **面试关键**：OSI 是理论模型，TCP/IP 是实际使用的工业标准。常说的"五层模型"是两者的折中教学模型。

---

### 2. 每一层的数据单位叫什么？🔥 ⭐

| 层 | 数据单位 |
|----|---------|
| 应用层 | 报文（Message） |
| 传输层 | 报文段（Segment）/ 用户数据报（Datagram） |
| 网络层 | 分组 / 数据报（Packet） |
| 数据链路层 | 帧（Frame） |
| 物理层 | 比特（Bit） |

---

### 3. 从输入 URL 到页面显示发生了什么？🔥🔥🔥 ⭐⭐⭐

这是一道**综合题**，覆盖了计算机网络几乎所有核心知识点：

```
1. URL 解析 → 判断是搜索还是 URL
2. DNS 解析 → 域名 → IP 地址（先查缓存：浏览器→OS→路由器→ISP）
3. TCP 三次握手 → 建立连接
4. TLS 握手（HTTPS）→ 协商加密密钥
5. 发送 HTTP 请求 → 请求行 + 请求头 + 请求体
6. 服务器处理请求 → 返回 HTTP 响应
7. 浏览器解析渲染 → HTML→DOM树，CSS→CSSOM树，合成渲染树，布局，绘制
```

**DNS 解析顺序**：浏览器缓存 → OS 缓存 → 路由器缓存 → ISP DNS → 递归/迭代查询

---

### 4. DNS 用的是 TCP 还是 UDP？🔥🔥 ⭐

**默认使用 UDP（端口 53）**，但当响应数据超过 512 字节时改用 TCP：

| 场景 | 协议 | 原因 |
|------|------|------|
| **普通查询** | UDP | 快速，请求/响应小 |
| **区域传送**（主从同步） | TCP | 数据量大，需要可靠传输 |
| **响应超过 512B** | TCP | UDP 有长度限制 |

## 二、HTTP / HTTPS

### 5. HTTP 和 HTTPS 的区别？🔥🔥🔥 ⭐⭐

| 维度 | HTTP | HTTPS |
|------|------|-------|
| **端口** | 80 | 443 |
| **安全性** | 明文传输，不安全 | SSL/TLS 加密，安全 |
| **证书** | 不需要 | 需要 CA 证书 |
| **连接过程** | TCP 三次握手即可 | TCP 握手 + TLS 握手 |
| **性能** | 较快 | 略慢（加密开销） |
| **成本** | 免费 | 证书需要费用 |

---

### 6. HTTP 状态码有哪些？🔥🔥🔥 ⭐

| 类别 | 状态码 | 含义 |
|------|--------|------|
| **1xx** | 100 Continue | 继续发送请求 |
| **2xx** | 200 OK / **204 No Content** / **206 Partial Content** | 成功 |
| **3xx** | **301 永久重定向** / **302 临时重定向** / **304 Not Modified** | 重定向 |
| **4xx** | **400 Bad Request** / **401 Unauthorized** / **403 Forbidden** / **404 Not Found** | 客户端错误 |
| **5xx** | **500 Internal Server Error** / **502 Bad Gateway** / **503 Service Unavailable** | 服务端错误 |

> **高频追问**：301 和 302 的区别？301 搜索引擎会更新索引，302 不会。301 浏览器会缓存，302 每次都请求。

---

### 7. GET 和 POST 的区别？🔥🔥🔥 ⭐

| 维度 | GET | POST |
|------|-----|------|
| **参数位置** | URL 中（查询字符串） | 请求体中 |
| **数据量** | 有长度限制（浏览器限制 URL 长度） | 理论上无限制 |
| **安全性** | 参数暴露在 URL | 相对安全 |
| **幂等性** | ✅ 幂等（多次请求结果相同） | ❌ 非幂等 |
| **缓存** | 可被缓存 | 不缓存 |
| **编码** | URL 编码 | 多种编码（form/multipart/json） |
| **浏览器行为** | 可收藏、有历史记录 | 不可收藏、无历史 |

> **注意**：本质上 GET 和 POST 都是 HTTP 请求，技术上 GET 也可以有 body，POST 也可以有 URL 参数。区别是约定和浏览器实现。

---

### 8. HTTP 1.0 / 1.1 / 2.0 / 3.0 的区别？🔥🔥🔥 ⭐⭐

| 特性 | HTTP/1.0 | HTTP/1.1 | HTTP/2.0 | HTTP/3.0 |
|------|----------|----------|----------|----------|
| **连接** | 短连接 | 长连接（Keep-Alive） | 多路复用 | 基于 QUIC |
| **队头阻塞** | 有 | 有（TCP 层） | 有（TCP 层） | ❌ 无（UDP） |
| **头部压缩** | ❌ | ❌ | ✅ HPACK | ✅ QPACK |
| **服务器推送** | ❌ | ❌ | ✅ | ✅ |
| **传输层** | TCP | TCP | TCP | **UDP（QUIC）** |

**HTTP/1.1 核心改进**：
- 长连接（Connection: keep-alive）
- 管道化（pipelining，但仍有队头阻塞）
- Host 头（支持虚拟主机）
- 分块传输（Transfer-Encoding: chunked）

**HTTP/2.0 核心改进**：
- 二进制帧（不再是文本协议）
- 多路复用（一个 TCP 连接并行多个流）
- 头部压缩（HPACK 算法）
- 服务器推送

**HTTP/3.0（QUIC）**：
- 基于 UDP，解决 TCP 队头阻塞
- 0-RTT 建连
- 内置 TLS 1.3
- 连接迁移（用 Connection ID 代替四元组）

---

### 9. HTTPS 的 TLS 握手过程？🔥🔥🔥 ⭐⭐⭐

**TLS 1.2 握手（RSA 密钥交换）**：

```
Client                                    Server
  |------ ClientHello（支持的加密套件、随机数1） ---->|
  |<----- ServerHello（选定的加密套件、随机数2、证书） --|
  |------ 验证证书，生成预主密钥，用公钥加密发送 ------>|
  |<----- Server 确认                                |
  |        双方用 随机数1+随机数2+预主密钥 生成会话密钥    |
  |====== 对称加密通信 ===============================|
```

**TLS 1.3 改进**（1-RTT 握手）：
- 精简密码套件，只保留 ECDHE
- 支持 0-RTT 恢复
- 更快、更安全

---

### 10. Cookie、Session、JWT 的区别？🔥🔥🔥 ⭐⭐

| 维度 | Cookie | Session | JWT |
|------|--------|---------|-----|
| **存储位置** | 客户端（浏览器） | 服务端 | 客户端 |
| **安全性** | 较低（可伪造） | 较高 | 中（签名防篡改） |
| **跨域** | 受同源策略限制 | 依赖 Cookie | ✅ 天然支持 |
| **服务器压力** | 无 | 占内存/Redis | 无 |
| **扩展性** | 一般 | 分布式需共享 | ✅ 无状态，易扩展 |

**Cookie 工作流程**：服务器 Set-Cookie → 浏览器保存 → 每次请求自动携带

**Session 工作流程**：服务器创建 Session（存内存/Redis）→ 返回 SessionID（通过 Cookie）→ 客户端携带 SessionID → 服务器查找 Session

**JWT 结构**：`Header.Payload.Signature`
- Header：算法类型
- Payload：用户信息（不要放敏感数据）
- Signature：`HMAC(header + payload, secret)`

---

### 11. HTTP 长连接和短连接？🔥 ⭐

- **短连接**（HTTP/1.0 默认）：每次请求都建立新的 TCP 连接，完成后关闭
- **长连接**（HTTP/1.1 默认）：TCP 连接建立后保持，复用传输多个请求

```
短连接：请求1 → TCP握手 → 传输 → TCP挥手 → 请求2 → TCP握手 → ...
长连接：TCP握手 → 请求1 → 请求2 → ... → TCP挥手
```

**Keep-Alive 超时机制**：服务器一般设置 `Keep-Alive: timeout=60`，60s 内无新请求则关闭连接。

---

### 12. HTTP 请求报文和响应报文的结构？🔥 ⭐

**请求报文**：
```
POST /api/user HTTP/1.1          ← 请求行（方法 URL 版本）
Host: example.com                ← 请求头
Content-Type: application/json
Content-Length: 25

{"name":"tom","age":18}          ← 请求体
```

**响应报文**：
```
HTTP/1.1 200 OK                  ← 状态行（版本 状态码 原因短语）
Content-Type: text/html          ← 响应头
Content-Length: 128

<html>...</html>                 ← 响应体
```

---

### 13. 什么是 HTTP 的无状态？如何解决？🔥 ⭐

**无状态**：服务器不保存客户端的任何状态信息，每次请求都是独立的。

**解决方案**：
1. **Cookie**：客户端存储少量信息，每次请求自动携带
2. **Session**：服务端存储会话信息，通过 SessionID 关联
3. **Token（JWT）**：服务端签发令牌，客户端携带，服务端验证签名

---

### 14. URI 和 URL 的区别？🔥 ⭐

| 概念 | 说明 | 示例 |
|------|------|------|
| **URI** | 统一资源标识符，包含 URL 和 URN | `mailto:test@example.com` |
| **URL** | 统一资源定位符，是 URI 的子集 | `https://example.com/path?q=1` |
| **URN** | 统一资源名称 | `urn:isbn:0451450523` |

**URL 结构**：`scheme://host:port/path?query#fragment`

## 三、TCP 连接

### 15. TCP 三次握手的过程？🔥🔥🔥 ⭐⭐⭐

```
Client                                    Server
  |------ SYN（seq=x） ------------------>|  （CLOSED → SYN_SENT）
  |                                      （LISTEN → SYN_RCVD）
  |<----- SYN+ACK（seq=y, ack=x+1） ------|
  |（ESTABLISHED）                          |
  |------ ACK（ack=y+1） ---------------->|  （ESTABLISHED）
```

**过程**：
1. 客户端发送 **SYN** 报文（SYN=1, seq=x），进入 SYN_SENT
2. 服务器回复 **SYN+ACK**（SYN=1, ACK=1, seq=y, ack=x+1），进入 SYN_RCVD
3. 客户端发送 **ACK**（ACK=1, ack=y+1），双方进入 ESTABLISHED

---

### 16. 为什么是三次握手而不是两次/四次？🔥🔥🔥 ⭐⭐⭐

**三次原因**：
1. **确认双方的收发能力**：三次是确认双方都能收发的最小次数
2. **防止历史连接（最主要）**：如果是两次，服务器收到延迟的旧 SYN 会立即建立连接浪费资源；三次握手让客户端可以拒绝旧的 SYN
3. **同步序列号**：双方需要交换初始序列号（ISN），各需要一个 SYN+ACK

**不是四次的原因**：服务器端的 SYN 和 ACK 可以合并为一次发送（第二次握手），所以不需要四次。

---

### 17. TCP 四次挥手的过程？🔥🔥🔥 ⭐⭐⭐

```
Client                                    Server
  |------ FIN（seq=u） ------------------>|  （ESTABLISHED → FIN_WAIT_1）
  |<----- ACK（ack=u+1） -----------------|  （CLOSE_WAIT）
  |（FIN_WAIT_2）                          |
  |          ... 服务器继续发送剩余数据 ...      |
  |<----- FIN+ACK（seq=w, ack=u+1） -------|  （LAST_ACK）
  |------ ACK（ack=w+1） ---------------->|  （TIME_WAIT）
  |     等待 2MSL 后 → CLOSED              （CLOSED）
```

**过程**：
1. 客户端发送 **FIN**，表示不再发送数据
2. 服务器回复 **ACK**，进入 CLOSE_WAIT（可能还有数据要发）
3. 服务器发完剩余数据，发送 **FIN**
4. 客户端回复 **ACK**，进入 TIME_WAIT

---

### 18. 为什么需要 TIME_WAIT 等待 2MSL？🔥🔥🔥 ⭐⭐⭐

**两个原因**：

1. **确保最后一个 ACK 能到达服务器**：如果 ACK 丢失，服务器会重发 FIN。客户端在 2MSL 内可以收到重发的 FIN 并重新发送 ACK
2. **让本连接的所有报文在网络中消失**：避免新连接收到旧连接的延迟报文

**MSL（Maximum Segment Lifetime）**：报文最大生存时间，一般 30s~60s，2MSL 约 1~2 分钟。

---

### 19. 为什么四次挥手不能像三次握手那样合并？🔥🔥 ⭐⭐

**核心原因**：TCP 是全双工通信，两个方向的关闭是独立的。

- 三次握手时，服务器端的 SYN 和 ACK 可以合并（因为服务器收到 SYN 时还没开始传数据）
- 四次挥手时，服务器收到 FIN 后可能还有数据没发完，不能立即关闭，所以先回 ACK，等数据发完再发 FIN

> **特殊情况**：如果服务器没有数据要发了，确实可能合并为三次挥手（FIN 和 ACK 一起发）。

---

### 20. TCP 和 UDP 的区别？🔥🔥🔥 ⭐⭐

| 维度 | TCP | UDP |
|------|-----|-----|
| **连接** | 面向连接（三次握手） | 无连接 |
| **可靠性** | 可靠（确认、重传、排序） | 不可靠 |
| **传输方式** | 字节流 | 数据报 |
| **效率** | 较慢 | 快 |
| **头部开销** | 20 字节 | 8 字节 |
| **流量控制** | ✅ 滑动窗口 | ❌ |
| **拥塞控制** | ✅ | ❌ |
| **适用场景** | 文件传输、HTTP、邮件 | 视频直播、DNS、游戏 |

---

### 21. TCP 如何保证可靠传输？🔥🔥🔥 ⭐⭐

| 机制 | 说明 |
|------|------|
| **校验和** | 检测数据在传输过程中是否被篡改 |
| **序列号** | 对每个字节编号，保证有序 |
| **确认应答（ACK）** | 收到数据后发送确认 |
| **超时重传** | 发送方在一定时间内未收到 ACK 则重传 |
| **滑动窗口** | 流量控制，接收方告知发送方窗口大小 |
| **拥塞控制** | 慢开始、拥塞避免、快重传、快恢复 |

---

### 22. TCP 的流量控制和拥塞控制的区别？🔥🔥 ⭐⭐

| 维度 | 流量控制 | 拥塞控制 |
|------|---------|---------|
| **目的** | 保护接收方 | 保护网络 |
| **对象** | 发送方 vs 接收方 | 发送方 vs 网络 |
| **机制** | 滑动窗口（接收窗口 rwnd） | 拥塞窗口 cwnd |
| **判断依据** | 接收方的缓冲区剩余空间 | 网络的拥塞程度（丢包/延迟） |

**拥塞控制四步**：
1. **慢开始**：cwnd 从 1 开始，每轮翻倍（指数增长）
2. **拥塞避免**：达到慢开始阈值 ssthresh 后线性增长
3. **快重传**：收到 3 个重复 ACK 立即重传（不等超时）
4. **快恢复**：快重传后 ssthresh = cwnd/2，cwnd = ssthresh，进入拥塞避免

---

### 23. TCP 粘包和拆包？🔥🔥 ⭐

**粘包**：发送方发送的多个数据包被接收方当作一个数据包接收
**拆包**：发送方发送的一个大数据包被接收方当作多个数据包接收

**原因**：TCP 是字节流协议，没有边界概念。

**解决方案**：

| 方案 | 说明 |
|------|------|
| **固定长度** | 每条消息固定长度，不足补齐 |
| **分隔符** | 用特殊字符标记消息边界（如 `\n`） |
| **长度前缀** | 消息头中指明消息体长度（最常用） |
| **TLV 格式** | Type-Length-Value 编码 |

---

### 24. 什么是半连接队列？SYN Flood 攻击？🔥🔥 ⭐⭐

**半连接队列（SYN Queue）**：服务器收到 SYN 后，在回复 SYN+ACK 但还没收到第三次 ACK 时，连接放在半连接队列中。

**SYN Flood 攻击**：攻击者大量发送 SYN 但不完成第三次握手，填满半连接队列，导致正常连接无法建立。

**防御方案**：
- SYN Cookie：不分配资源，将状态编码在 SYN+ACK 的序列号中
- 减小 SYN+ACK 重传次数
- 增大半连接队列
- 防火墙/CDN

---

### 25. CLOSE_WAIT 过多是什么原因？🔥🔥 ⭐

**CLOSE_WAIT 状态**：收到对方的 FIN 并回复 ACK 后，等待自己应用层调用 close()。

**原因**：对方关闭了连接，但己方代码没有正确关闭 Socket。

**排查**：
1. 检查代码是否在合适时机调用了 `close()`
2. 是否有死循环或阻塞导致没有执行到 close
3. 连接池配置是否合理

---

### 26. TIME_WAIT 过多怎么处理？🔥🔥 ⭐

**原因**：大量短连接频繁创建和关闭。

**解决方案**：
1. 使用长连接（Keep-Alive）
2. 设置 `tcp_tw_reuse = 1`（允许复用 TIME_WAIT 连接，仅客户端）
3. 调小 `tcp_fin_timeout`（缩短 TIME_WAIT 时间）
4. 使用连接池

## 四、IP 协议

### 27. IP 协议有哪些？🔥 ⭐

| 类型 | 范围 | 说明 |
|------|------|------|
| **A 类** | 1.0.0.0 ~ 126.255.255.255 | 大型网络 |
| **B 类** | 128.0.0.0 ~ 191.255.255.255 | 中型网络 |
| **C 类** | 192.0.0.0 ~ 223.255.255.255 | 小型网络 |
| **D 类** | 224.0.0.0 ~ 239.255.255.255 | 组播 |
| **E 类** | 240.0.0.0 ~ 255.255.255.255 | 保留 |

**特殊 IP**：
- `127.0.0.1`：环回地址
- `0.0.0.0`：本机所有 IP
- `255.255.255.255`：广播地址
- `10.x.x.x`、`172.16~31.x.x`、`192.168.x.x`：私有地址

---

### 28. IPv4 不够用怎么办？🔥 ⭐

| 方案 | 说明 |
|------|------|
| **NAT（网络地址转换）** | 多个内网 IP 共享一个公网 IP |
| **IPv6** | 128 位地址，号称"地球上每粒沙子都能分配一个" |
| **CIDR（无类别域间路由）** | 灵活划分子网，减少地址浪费 |

**NAT 类型**：
- 静态 NAT：一对一映射
- NAPT：端口地址转换，多对一（最常用）

---

### 29. ARP 协议的作用？🔥 ⭐

**ARP（Address Resolution Protocol）**：将 IP 地址解析为 MAC 地址。

```
主机A 要发数据给 192.168.1.5：
1. 查本地 ARP 缓存 → 没找到
2. 广播 ARP 请求：谁是 192.168.1.5？请告诉我你的 MAC
3. 192.168.1.5 单播 ARP 响应：我是，MAC 是 xx:xx:xx:xx:xx:xx
4. 主机A 缓存该映射，发送数据帧
```

---

### 30. 为什么有了 IP 地址还需要 MAC 地址？🔥 ⭐

| 维度 | IP 地址 | MAC 地址 |
|------|---------|----------|
| **层级** | 网络层 | 数据链路层 |
| **特点** | 逻辑地址，可变 | 物理地址，固定（烧录在网卡） |
| **作用** | 跨网络寻址（路由） | 同一网络内寻址（局域网） |
| **长度** | 32 位（IPv4） | 48 位 |

> **类比**：MAC 地址像身份证号（不变的唯一标识），IP 地址像邮政地址（会随位置变化）。

---

### 31. ICMP 协议是什么？ping 用了什么协议？🔥 ⭐

**ICMP（Internet Control Message Protocol）**：网络层的控制协议，用于发送差错报告和查询消息。

**ping 命令**：使用 ICMP Echo Request（类型 8）和 Echo Reply（类型 0）。

```
ping 过程：
1. 源主机发送 ICMP Echo Request
2. 目标主机回复 ICMP Echo Reply
3. 根据往返时间计算延迟
```

**traceroute**：利用 ICMP 超时报文（TTL 递增）追踪路由路径。

## 五、网络安全

### 32. 常见的网络攻击有哪些？🔥🔥🔥 ⭐⭐

| 攻击类型 | 原理 | 防御 |
|---------|------|------|
| **XSS（跨站脚本）** | 注入恶意脚本到网页 | 输入过滤、CSP、HttpOnly Cookie |
| **CSRF（跨站请求伪造）** | 伪造用户请求 | CSRF Token、Referer 校验、SameSite Cookie |
| **SQL 注入** | 拼接恶意 SQL | 参数化查询、ORM、输入验证 |
| **DDoS** | 大量请求淹没服务器 | 流量清洗、CDN、限流 |
| **DNS 劫持** | 篡改 DNS 解析结果 | DNSSEC、HTTPS、可信 DNS |
| **中间人攻击（MITM）** | 拦截篡改通信内容 | HTTPS、证书校验 |

---

### 33. XSS 攻击详解？🔥🔥 ⭐

**类型**：
- **存储型 XSS**：恶意脚本存入数据库（如评论中插入 `<script>`），其他用户浏览时执行
- **反射型 XSS**：恶意脚本在 URL 参数中，用户点击链接时执行
- **DOM 型 XSS**：前端 JS 直接将不可信数据插入 DOM

**防御**：
1. 输入输出编码/转义（`<` → `&lt;`）
2. Content-Security-Policy（CSP）头
3. Cookie 设置 `HttpOnly`（防止 JS 读取 Cookie）
4. 使用框架内置的 XSS 防护（React/Vue 默认转义）

---

### 34. CSRF 攻击详解？🔥🔥 ⭐

**原理**：用户已登录 A 网站，访问恶意 B 网站时，B 网站自动向 A 发送请求（携带 Cookie）。

**防御**：
1. **CSRF Token**：服务端生成随机 Token，表单提交时验证
2. **Referer/Origin 检查**：验证请求来源
3. **SameSite Cookie**：`Set-Cookie: SameSite=Strict`（Chrome 默认 Lax）

---

### 35. 对称加密和非对称加密？🔥🔥 ⭐

| 维度 | 对称加密 | 非对称加密 |
|------|---------|-----------|
| **密钥** | 加密解密用同一密钥 | 公钥加密、私钥解密 |
| **速度** | 快 | 慢 |
| **算法** | AES、DES、ChaCha20 | RSA、ECC、DSA |
| **密钥分发** | 困难（需要安全传输密钥） | 简单（公钥公开） |

**HTTPS 中的混合加密**：
1. 非对称加密协商出对称密钥（会话密钥）
2. 之后用对称密钥加密通信数据（快）

---

### 36. 什么是 DNS 劫持？如何防御？🔥 ⭐

**DNS 劫持**：攻击者篡改 DNS 服务器的解析结果，将域名指向恶意 IP。

**防御**：
- 使用可信 DNS（如 8.8.8.8、114.114.114.114）
- DNSSEC（DNS 安全扩展，签名验证）
- HTTPS（即使 IP 被篡改，证书校验会失败）
- HTTPDNS（绕过本地 DNS，直接向 HTTP 接口查询）

## 六、其他高频题

### 37. WebSocket 和 HTTP 的区别？🔥🔥 ⭐

| 维度 | HTTP | WebSocket |
|------|------|-----------|
| **通信方式** | 请求-响应（单向） | 全双工（双向） |
| **连接** | 短连接（1.1 可长连接） | 持久连接 |
| **开销** | 每次请求携带完整头部 | 握手后数据帧极小 |
| **场景** | 普通 API 请求 | 即时通讯、实时推送、在线游戏 |

**WebSocket 建连**：通过 HTTP Upgrade 请求升级协议：
```
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
```

---

### 38. RPC 和 HTTP 的区别？🔥🔥 ⭐

| 维度 | HTTP | RPC |
|------|------|-----|
| **协议** | 标准协议 | 自定义协议（如 Dubbo、gRPC） |
| **性能** | 头部大，文本协议 | 头部小，二进制协议，更快 |
| **服务治理** | 需要额外框架（如 Spring Cloud） | 内置服务发现、负载均衡 |
| **跨语言** | ✅ 天然支持 | 取决于框架（gRPC 跨语言） |
| **适用场景** | 对外 API、前后端通信 | 微服务内部调用 |

---

### 39. Nginx 负载均衡算法？🔥 ⭐

| 算法 | 说明 |
|------|------|
| **轮询（Round Robin）** | 默认，依次分配 |
| **加权轮询** | 按权重分配（性能好的机器权重高） |
| **IP Hash** | 同一 IP 始终分配到同一后端（会话保持） |
| **最少连接（Least Connections）** | 分配给当前连接数最少的服务器 |
| **URL Hash** | 同一 URL 分配到同一后端（缓存友好） |

### 40. 如何排查网页打开慢的问题？🔥 ⭐⭐

**排查步骤**：

```
1. 网络 → ping/dns 查询是否慢？CDN 是否命中？
2. TCP/TLS → 连接建立是否耗时？证书链是否太长？
3. 服务器 → 响应时间？数据库慢查询？后端服务超时？
4. 传输 → 资源是否太大？是否压缩（gzip）？是否开启 HTTP/2？
5. 渲染 → JS 是否阻塞？CSS 是否阻塞？图片是否懒加载？
```

**工具**：Chrome DevTools（Network/Waterfall）、Fiddler、Wireshark、Lighthouse
