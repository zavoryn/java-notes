## 一、设计模式

### 1. 常用设计模式有哪些？🔥🔥🔥 ⭐

| 类型 | 模式 | 说明 |
|------|------|------|
| **创建型** | 单例、工厂、抽象工厂、建造者、原型 | 对象创建 |
| **结构型** | 适配器、装饰器、代理、外观、桥接、组合、享元 | 类与对象的组合 |
| **行为型** | 策略、观察者、模板方法、责任链、状态、命令 | 对象间通信 |

---

### 2. 单例模式？🔥🔥🔥 ⭐⭐

**双重检查锁（DCL）**：
```java
public class Singleton {
    private volatile static Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {                    // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {            // 第二次检查
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**为什么用 volatile？** 防止指令重排。`new Singleton()` 不是原子操作（分配内存→初始化→引用赋值），重排可能导致其他线程拿到未初始化的对象。

**其他实现**：
- **饿汉式**：类加载时创建，线程安全但可能浪费资源
- **静态内部类**：延迟加载，线程安全，推荐
- **枚举**：最安全，防反射和序列化破坏

---

### 3. 工厂模式和抽象工厂模式？🔥🔥 ⭐

| 模式 | 说明 | 示例 |
|------|------|------|
| **简单工厂** | 一个工厂类根据参数创建不同产品 | `BeanFactory.getBean("name")` |
| **工厂方法** | 每个产品对应一个工厂 | `Collection.iterator()` |
| **抽象工厂** | 创建一族相关产品 | GUI 工厂（Button+TextField） |

> Spring 中的 `BeanFactory` 和 `ApplicationContext` 都是工厂模式的典型应用。

---

### 4. 策略模式？🔥🔥 ⭐

**核心**：定义一系列算法，将每个算法封装起来，使它们可以互相替换。

```java
interface Strategy { void execute(); }
class StrategyA implements Strategy { public void execute() { /* 算法A */ } }
class StrategyB implements Strategy { public void execute() { /* 算法B */ } }

class Context {
    private Strategy strategy;
    public void setStrategy(Strategy s) { this.strategy = s; }
    public void run() { strategy.execute(); }
}
```

**应用**：Spring 的 `ResourceResolver`、Comparator 排序、支付方式选择

---

### 5. 观察者模式？🔥🔥 ⭐

**核心**：一对多依赖关系，一个对象状态改变时，所有依赖它的对象都会收到通知。

**应用**：
- Spring Event 机制（`ApplicationEvent` + `@EventListener`）
- Vue/React 的响应式系统
- Java 的 `Observable` / `Observer`（已废弃）
- Guava EventBus

---

### 6. 代理模式？🔥🔥 ⭐

| 类型 | 说明 | 示例 |
|------|------|------|
| **静态代理** | 编译时确定代理类 | 手动编写代理类 |
| **JDK 动态代理** | 运行时生成代理（基于接口） | Spring AOP |
| **CGLIB 代理** | 运行时生成子类（不要求接口） | Spring AOP |

> **Spring AOP 默认规则**：有接口用 JDK 动态代理，无接口用 CGLIB。Spring Boot 2.x 默认全部用 CGLIB。

---

### 7. 责任链模式？🔥 ⭐

**核心**：将请求沿着处理链传递，每个处理者决定是否处理或传递给下一个。

**应用**：
- Servlet Filter 链
- Spring Interceptor
- Netty 的 ChannelPipeline
- OAuth2 认证流程

---

### 8. 模板方法模式？🔥 ⭐

**核心**：父类定义算法骨架，子类实现具体步骤。

```java
abstract class AbstractClass {
    public final void templateMethod() {  // final 防止重写
        step1();
        step2();
        step3();
    }
    protected abstract void step1();
    protected abstract void step2();
    private void step3() { /* 公共逻辑 */ }
}
```

**应用**：Spring 的 `AbstractApplicationContext.refresh()`、`JdbcTemplate`、`RestTemplate`

## 二、分布式系统基础

### 9. CAP 定理？🔥🔥🔥 ⭐⭐

| 属性 | 说明 |
|------|------|
| **一致性（Consistency）** | 所有节点看到的数据一致 |
| **可用性（Availability）** | 每个请求都能得到非错误响应 |
| **分区容错性（Partition tolerance）** | 网络分区时系统仍能运行 |

**核心结论**：网络分区不可避免，因此只能在 **CP** 和 **AP** 之间选择：

| 方案 | 代表 | 场景 |
|------|------|------|
| **CP** | ZooKeeper、Etcd、HBase | 金融、配置中心（数据不能错） |
| **AP** | Eureka、Cassandra、DynamoDB | 电商、社交（允许暂时不一致） |

---

### 10. BASE 理论？🔥🔥 ⭐

| 原则 | 说明 |
|------|------|
| **基本可用（Basically Available）** | 故障时允许响应时间增加或功能降级 |
| **软状态（Soft State）** | 允许中间状态（数据副本间允许暂时不一致） |
| **最终一致性（Eventually Consistent）** | 经过一段时间后，数据最终达到一致 |

> BASE 是 CAP 中 AP 的延伸，是对大规模分布式系统的实践总结。

---

### 11. 分布式 ID 生成方案？🔥🔥 ⭐

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **UUID** | 128 位随机数 | 简单，无依赖 | 无序，索引性能差，太长 |
| **数据库自增** | auto_increment | 简单有序 | 单点瓶颈，扩展困难 |
| **号段模式** | 批量获取 ID 段 | 减少数据库压力 | 依赖数据库 |
| **Redis INCR** | 原子自增 | 高性能 | 依赖 Redis |
| **Snowflake** | 时间+机器+序列号 | ✅ 有序、高性能、不依赖外部 | 时钟回拨问题 |
| **Leaf（美团）** | 号段 + Snowflake | ✅ 生产级方案 | — |

**Snowflake 结构**（64位）：`1位符号 + 41位时间戳 + 10位机器ID + 12位序列号`

---

### 12. 分布式锁的实现方式？🔥🔥🔥 ⭐⭐

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **Redis SETNX** | `SET key val NX EX 30` | 简单高效 | 不可重入、锁续期问题 |
| **Redisson** | 看门狗自动续期 | ✅ 可重入、自动续期 | 依赖 Redis |
| **ZooKeeper** | 临时顺序节点 | 可靠、可 watch | 性能较低 |
| **MySQL** | 唯一索引 / FOR UPDATE | 简单 | 性能差 |

**Redisson 看门狗**：默认锁 30s，每 10s（1/3 时间）自动续期，客户端宕机则锁自动释放。

---

### 13. 分布式事务解决方案？🔥🔥🔥 ⭐⭐

| 方案 | 原理 | 一致性 | 性能 |
|------|------|--------|------|
| **2PC（两阶段提交）** | 协调者先 prepare，再 commit/rollback | 强一致 | ❌ 阻塞、单点 |
| **TCC** | Try-Confirm-Cancel，业务层面实现 | 最终一致 | ⚠️ 侵入性强 |
| **SAGA** | 长事务拆为多个本地事务 + 补偿 | 最终一致 | ✅ 适合长事务 |
| **本地消息表** | 本地事务 + 消息表 + 定时投递 | 最终一致 | ✅ 实用 |
| **RocketMQ 事务消息** | 半消息 + 本地事务 + 二次确认 | 最终一致 | ✅ 推荐 |

## 三、高并发系统设计

### 14. 如何设计一个秒杀系统？🔥🔥🔥 ⭐⭐⭐

**核心挑战**：瞬时高并发、防超卖、防刷。

**架构设计**：

```
客户端 → CDN → Nginx（负载均衡）→ 网关（限流/鉴权）
    → 本地缓存（库存预热）→ Redis（扣减库存）
    → MQ（异步下单）→ 订单服务 → 数据库
```

**关键设计**：

| 环节 | 方案 |
|------|------|
| **流量削峰** | MQ 异步处理、验证码分散请求 |
| **防超卖** | Redis 原子扣减（`DECR`）+ 数据库乐观锁 |
| **防刷** | IP 限流、用户维度限流、验证码、答题 |
| **库存预热** | 秒杀前将库存加载到 Redis |
| **兜底** | 熔断降级、限流、服务隔离 |
| **高可用** | 多级缓存、降级方案、灾备 |

**防超卖 SQL**：
```sql
UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0;
```

---

### 15. 如何设计一个短链系统？🔥🔥 ⭐⭐

**核心流程**：长链 → 短链 → 302 重定向到长链

**短链生成**：
1. 唯一 ID（Snowflake）→ Base62 编码
2. MD5/SHA256 哈希 → 取前 6~8 位

**存储方案**：
- MySQL + 分库分表（按短链前缀分）
- Redis 缓存热点映射（加速查询）

**跳转方案**：HTTP 301（永久，浏览器缓存）或 **302**（临时，可统计点击量）

---

### 16. 如何设计一个限流器？🔥🔥 ⭐

| 算法 | 说明 | 特点 |
|------|------|------|
| **固定窗口** | 固定时间窗口计数 | 简单，有临界突发问题 |
| **滑动窗口** | 滑动时间窗口计数 | ✅ 更平滑 |
| **漏桶** | 请求以固定速率处理 | 流量均匀，但无法应对突发 |
| **令牌桶** | 固定速率生成令牌，请求需获取令牌 | ✅ 允许突发流量 |

> **Sentinel**（阿里）默认用滑动窗口；**Guava RateLimiter** 用令牌桶。

---

### 17. 熔断降级是什么？🔥🔥 ⭐

**熔断**：当下游服务错误率超过阈值时，自动切断请求，直接返回降级结果。

**三种状态**：
```
Closed（正常）→ 错误率超阈值 → Open（熔断）
                                    ↓ 等待超时
                               Half-Open（半开，放少量请求探测）
                                    ↓ 成功则 → Closed
                                    ↓ 失败则 → Open
```

**降级策略**：返回默认值、返回缓存数据、关闭非核心功能。

**框架**：Sentinel（阿里）、Hystrix（Netflix，已停更）、Resilience4j

### 18. 什么是微服务？优缺点？🔥🔥 ⭐

| 维度 | 说明 |
|------|------|
| **优点** | 独立部署、技术栈灵活、故障隔离、按需扩缩容 |
| **缺点** | 分布式复杂度（网络延迟/数据一致/分布式事务）、运维成本高、服务间调用链路复杂 |

**微服务核心组件**：

| 组件 | 功能 | 框架 |
|------|------|------|
| **注册中心** | 服务注册与发现 | Nacos、Eureka、Consul |
| **配置中心** | 集中管理配置 | Nacos、Apollo |
| **网关** | 路由、鉴权、限流 | Spring Cloud Gateway |
| **负载均衡** | 请求分发 | Ribbon、LoadBalancer |
| **链路追踪** | 调用链监控 | SkyWalking、Zipkin |
| **熔断降级** | 保护服务 | Sentinel |

### 20. 负载均衡策略？🔥🔥 ⭐

| 策略 | 说明 |
|------|------|
| **轮询（Round Robin）** | 依次分配 |
| **加权轮询** | 按权重分配 |
| **随机** | 随机选择 |
| **最少连接** | 分配给当前连接数最少的服务器 |
| **一致性哈希** | 同一请求落到同一服务器（会话保持） |

**四层 vs 七层负载均衡**：
- **四层（LVS）**：基于 IP+端口转发，性能高
- **七层（Nginx）**：基于 HTTP 协议转发，功能丰富

---

### 21. 分库分表策略？🔥🔥 ⭐

**垂直拆分**：按业务模块拆分数据库/表（如用户库、订单库）

**水平拆分**：按数据行拆分到不同表/库

| 策略 | 说明 | 适用 |
|------|------|------|
| **Hash 分片** | `shard = hash(key) % N` | ✅ 数据均匀 |
| **范围分片** | 按 ID/时间范围 | 适合时间序列 |
| **一致性哈希** | 减少数据迁移 | 动态扩容 |

**分库分表中间件**：ShardingSphere、MyCat

**问题与解决**：

| 问题 | 方案 |
|------|------|
| 分布式 ID | Snowflake |
| 跨库 JOIN | 冗余字段、应用层组装 |
| 跨库事务 | 分布式事务（TCC/SAGA） |
| 跨库分页 | 各分片查询后合并排序 |

---

### 22. 什么是读写分离？🔥🔥 ⭐

**原理**：主库负责写操作，从库负责读操作，主从通过 binlog 同步。

**问题**：主从延迟导致读到旧数据

**解决方案**：
1. **强制走主库**：关键业务读操作直接查主库
2. **中间件判断**：根据 last_insert_id 或 GTID 判断从库是否已同步
3. **半同步复制**：至少一个从库确认接收后才返回

## 五、缓存设计

### 23. 缓存穿透、雪崩、击穿？🔥🔥🔥 ⭐⭐⭐

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| **穿透** | 查询不存在的数据，缓存和 DB 都没有 | 布隆过滤器 / 缓存空值（设短 TTL） |
| **雪崩** | 大量缓存同时过期 或 缓存服务宕机 | 随机过期时间 / 多级缓存 / 熔断降级 |
| **击穿** | 热点 Key 过期，大量请求打到 DB | 互斥锁 / 逻辑过期（不设 TTL） |

---

### 24. 缓存和数据库一致性？🔥🔥🔥 ⭐⭐

| 策略 | 说明 | 问题 |
|------|------|------|
| **先更新 DB 再更新缓存** | — | 并发写导致缓存与 DB 不一致 |
| **先更新缓存再更新 DB** | — | DB 更新失败导致不一致 |
| **先更新 DB 再删缓存** | ✅ 推荐 | 极端并发下仍可能不一致 |
| **延迟双删** | 更新 DB → 删缓存 → 延迟 → 再删 | 基本解决不一致 |

**最佳实践**：`先更新 DB + 再删缓存` + `消息队列保证最终一致`

---

### 25. 布隆过滤器原理？🔥 ⭐

**数据结构**：一个很长的二进制位数组 + 多个哈希函数。

**操作**：
- **添加**：元素经多个哈希函数计算位置，设为 1
- **查询**：所有哈希位置都为 1 → **可能**存在；任一位为 0 → **一定**不存在

**特点**：
- 有**误判率**（假阳性），但不会漏判
- 不支持删除
- 空间效率极高

**应用**：缓存穿透防护、爬虫 URL 去重、邮箱垃圾邮件过滤

## 六、场景设计题

### 26. 如何设计一个微信朋友圈？🔥🔥 ⭐

**核心功能**：发动态、时间线 Feed、点赞/评论

**Feed 流方案**：

| 方案 | 说明 | 适用 |
|------|------|------|
| **推模式（写扩散）** | 发帖时推送到所有粉丝的收件箱 | 粉丝少（普通用户） |
| **拉模式（读扩散）** | 读时从关注列表拉取 | 粉丝多（大 V） |
| **推拉结合** | 大 V 用拉模式，普通用户用推模式 | ✅ 微博等平台 |

**存储**：动态内容 → MySQL 分库分表，时间线 → Redis Sorted Set

### 29. 如何设计单点登录（SSO）？🔥 ⭐

**核心流程**：
1. 用户访问子系统 → 未登录 → 重定向到 SSO 服务器
2. SSO 服务器登录成功 → 生成 Token → 写入 Cookie（SSO 域）
3. 重定向回子系统并携带 Token
4. 子系统验证 Token → 建立局部会话
5. 访问其他子系统 → SSO Cookie 已存在 → 免登录

**Token 管理**：JWT / Redis Session + Token
