## 一、MySQL 基础架构与存储引擎

### 1. MySQL 一条 SQL 语句的执行流程是怎样的？🔥🔥🔥 ⭐⭐

**完整流程：**

```
客户端 → 连接器（权限验证） → 查询缓存（8.0已移除） → 解析器（词法/语法分析）
→ 优化器（执行计划选择） → 执行器（权限校验→调用引擎接口） → 存储引擎（读写数据）
```

**详细说明：**

| 组件 | 功能 |
|------|------|
| **连接器** | 负责与客户端建立连接、获取权限、维持和管理连接。长连接默认 8 小时超时（`wait_timeout`） |
| **查询缓存** | MySQL 8.0 已移除。之前以 SQL 语句为 key 缓存结果，但失效非常频繁，命中率低 |
| **解析器** | 词法分析（识别关键字、表名、列名）+ 语法分析（判断 SQL 是否合法），生成抽象语法树 |
| **优化器** | 决定使用哪个索引、JOIN 表的连接顺序等，生成执行计划 |
| **执行器** | 校验权限，根据执行计划调用存储引擎 API 读写数据 |

> **追问：为什么 MySQL 8.0 移除了查询缓存？**
> 因为查询缓存在高并发场景下效率极低——任何对表的修改都会导致该表所有缓存失效，对于频繁更新的表，缓存命中率可能不到 5%。MySQL 8.0 推荐使用应用层缓存（如 Redis）替代。

### 2. InnoDB 和 MyISAM 有什么区别？🔥🔥🔥 ⭐⭐

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| **事务支持** | ✅ 支持 ACID 事务 | ❌ 不支持 |
| **锁粒度** | 行级锁（并发高） | 表级锁（并发低） |
| **外键** | ✅ 支持 | ❌ 不支持 |
| **崩溃恢复** | ✅ redo log 保证 crash-safe | ❌ 无崩溃恢复机制 |
| **MVCC** | ✅ 支持多版本并发控制 | ❌ 不支持 |
| **索引结构** | 聚簇索引（数据和主键索引存储在一起） | 非聚簇索引（索引和数据文件分离） |
| **全文索引** | MySQL 5.6+ 支持 | ✅ 支持 |
| **COUNT(*)** | 需要遍历（因为 MVCC，不同事务看到的行数不同） | 直接读取存储的行数，极快 |
| **存储文件** | `.ibd` 文件 | `.MYD`（数据）+ `.MYI`（索引） |
| **适用场景** | 需要事务、高并发读写 | 以读为主、不需要事务的查询分析场景 |

> **结论：MySQL 5.5 之后默认引擎为 InnoDB，绝大多数场景应选择 InnoDB。**

---

### 3. MySQL 为什么采用插件式存储引擎架构？🔥 ⭐

MySQL 的存储引擎是**表级别**的，每张表可以使用不同的存储引擎。这种架构的好处：

1. **灵活性**：不同业务场景选择最适合的引擎（如 OLTP 用 InnoDB，OLAP 用 ColumnStore）
2. **可扩展性**：可以自定义存储引擎（如 Facebook 的 RocksDB 引擎）
3. **解耦**：上层 Server 层（解析器、优化器、执行器）与底层存储引擎完全解耦

### 4. 什么是聚簇索引？什么是非聚簇索引（二级索引）？🔥🔥 ⭐⭐

**聚簇索引（Clustered Index）：**
- **数据行和主键索引存储在同一棵 B+ 树中**
- 叶子节点直接包含完整的行数据
- 一张表只能有一个聚簇索引（通常是主键）
- InnoDB 中如果没有定义主键，会选择唯一非空索引，如果也没有，则自动生成一个 6 字节的隐藏主键（`ROW_ID`）

**非聚簇索引（Secondary Index / 二级索引）：**
- 叶子节点存储的是**主键值**而非行数据
- 查询非索引列时需要**回表**：先通过二级索引找到主键 → 再通过主键的聚簇索引找到完整行数据
- 一张表可以有多个二级索引

```
聚簇索引结构：
  根节点 → 分支节点 → 叶子节点（包含完整行数据，通过链表相连）

二级索引结构：
  根节点 → 分支节点 → 叶子节点（包含索引列值 + 主键值）
```

## 二、SQL 基础

### 5. SQL 的执行顺序是怎样的？🔥🔥 ⭐

**SQL 书写顺序：**
```sql
SELECT → FROM → JOIN → ON → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT
```

**SQL 实际执行顺序：**
```sql
FROM → ON → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
```

理解执行顺序有助于优化：WHERE 在 GROUP BY 之前执行，所以 WHERE 中不能使用聚合函数；HAVING 在 GROUP BY 之后，可以使用聚合函数。

### 6. CHAR 和 VARCHAR 的区别？🔥 ⭐

| 特性 | CHAR | VARCHAR |
|------|------|---------|
| **存储方式** | 定长，不足部分用空格填充 | 变长，按实际长度存储 |
| **最大长度** | 255 字符 | 65535 字节 |
| **额外字节** | 无 | 1-2 字节记录长度 |
| **性能** | 固定长度，检索效率高 | 需计算长度，稍低 |
| **适用场景** | MD5、手机号等定长数据 | 姓名、地址等变长数据 |

> **注意：** VARCHAR(50) 中的 50 是字符数，不是字节数。在 utf8mb4 编码下，一个字符最多占 4 字节。

### 7. DATETIME 和 TIMESTAMP 的区别？🔥 ⭐

| 特性 | DATETIME | TIMESTAMP |
|------|----------|-----------|
| **存储空间** | 8 字节 | 4 字节 |
| **范围** | 1000-01-01 ~ 9999-12-31 | 1970-01-01 ~ 2038-01-19 |
| **时区** | 不受时区影响，存储原始值 | 受时区影响，存储为 UTC，查询时转为当前时区 |
| **默认值** | 无 | CURRENT_TIMESTAMP |
| **适用场景** | 记录固定时间（如生日） | 记录操作时间（如创建/更新时间） |

> **坑点：** TIMESTAMP 存在 2038 年问题，超过 2038-01-19 03:14:07 UTC 会溢出。建议新项目优先使用 DATETIME。

### 8. 数据库三大范式是什么？🔥 ⭐

| 范式 | 要求 | 说明 |
|------|------|------|
| **第一范式 (1NF)** | 字段不可再分 | 每列都是原子值，不能有嵌套表 |
| **第二范式 (2NF)** | 消除部分依赖 | 非主键字段必须完全依赖主键，不能只依赖主键的一部分（针对联合主键） |
| **第三范式 (3NF)** | 消除传递依赖 | 非主键字段不能依赖于其他非主键字段 |

> **实际开发中的权衡：** 为了性能，有时会故意反范式化（冗余字段），用空间换时间。例如在订单表中冗余商品名称，避免每次 JOIN 查询。

### 9. JOIN 有哪些类型？🔥🔥 ⭐

```sql
-- 内连接：只返回两表匹配的行
SELECT * FROM A INNER JOIN B ON A.id = B.a_id;

-- 左外连接：返回左表所有行 + 右表匹配行（右表无匹配则为 NULL）
SELECT * FROM A LEFT JOIN B ON A.id = B.a_id;

-- 右外连接：返回右表所有行 + 左表匹配行
SELECT * FROM A RIGHT JOIN B ON A.id = B.a_id;

-- 全外连接（MySQL 不直接支持，用 UNION 模拟）
SELECT * FROM A LEFT JOIN B ON A.id = B.a_id
UNION
SELECT * FROM A RIGHT JOIN B ON A.id = B.a_id;

-- 交叉连接：笛卡尔积
SELECT * FROM A CROSS JOIN B;
```

> **面试陷阱：** 如果 JOIN 条件没有 ON 子句或 ON 条件永远为真，会产生笛卡尔积（A 表 × B 表行数），导致性能灾难。

## 三、索引

### 10. 为什么 MySQL 使用 B+ 树作为索引结构？🔥🔥🔥 ⭐⭐⭐

**对比各数据结构：**

| 数据结构 | 问题 |
|---------|------|
| **哈希表** | 不支持范围查询（`WHERE age > 18`），只适合等值查询 |
| **二叉搜索树** | 树的高度不均衡，最坏情况退化成链表（O(n)） |
| **AVL/红黑树** | 平衡的二叉树，但高度仍然太大（数据量大时层数深，IO 次数多） |
| **B 树** | 每个节点都存数据，非叶节点也存数据，导致每个磁盘块存储的键值少，树变高 |
| **B+ 树** ✅ | 非叶节点只存索引不存数据 → 每个磁盘块能存更多索引 → 树更矮 → IO 更少 |

**B+ 树的核心优势：**

1. **IO 次数少**：非叶节点只存索引键值（不存数据），一个 16KB 的页可以存 ~1170 个索引指针，3 层 B+ 树就能存约 2000 万行数据
2. **范围查询高效**：叶子节点通过双向链表连接，范围查询只需找到起点后顺序扫描
3. **查询性能稳定**：所有数据都在叶子节点，每次查询都要走到叶子节点，路径长度相同

```
B+ 树结构（3层约存 2000 万行）：
        [根节点: 索引键 + 指针]
              ↓ ↓ ↓
    [中间节点: 索引键 + 指针] × N
              ↓ ↓ ↓
  [叶子节点: 完整数据行] ← 双向链表连接 →
```

### 11. 什么是覆盖索引？🔥🔥 ⭐⭐

当查询的列全部包含在索引中时，不需要回表查询聚簇索引，直接从索引中返回数据，称为**覆盖索引**。

```sql
-- 假设有联合索引 (name, age)
-- 覆盖索引：查询列都在索引中，不需要回表 ✅
SELECT name, age FROM user WHERE name = '张三';

-- 非覆盖索引：需要查询 id 以外的列，必须回表 ❌
SELECT name, age, email FROM user WHERE name = '张三';
```

在 EXPLAIN 的 Extra 列中出现 `Using index` 就表示使用了覆盖索引。

### 12. 什么是最左前缀匹配原则？🔥🔥🔥 ⭐⭐

联合索引 `(a, b, c)` 实际上等于创建了三个索引：`(a)`, `(a, b)`, `(a, b, c)`。

**匹配规则：**
- 从索引最左列开始匹配
- 遇到范围查询（`>`, `<`, `BETWEEN`, `LIKE`）会停止匹配后续列

```sql
-- 联合索引 (a, b, c)

-- ✅ 可以命中索引
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3

-- ❌ 不能命中索引（缺少最左列 a）
WHERE b = 2 AND c = 3

-- ⚠️ 只有 a 列能命中（遇到 > 范围查询后停止）
WHERE a = 1 AND b > 2 AND c = 3
```

> **MySQL 优化器的智能之处：** `WHERE a = 1 AND c = 3 AND b = 2` 这种顺序不影响索引使用，优化器会自动重排为 `(a, b, c)` 顺序。关键是列是否存在，不是书写顺序。

### 13. 索引失效的场景有哪些？🔥🔥🔥 ⭐⭐

| # | 场景 | 示例 | 解决方案 |
|---|------|------|---------|
| 1 | **对索引列使用函数或运算** | `WHERE YEAR(create_time) = 2024` | 改为 `WHERE create_time >= '2024-01-01'` |
| 2 | **隐式类型转换** | `WHERE varchar_col = 123`（字符串列用数字查） | 统一类型：`WHERE varchar_col = '123'` |
| 3 | **LIKE 以通配符开头** | `WHERE name LIKE '%张'` | 使用全文索引或 ES；`LIKE '张%'` 可以命中 |
| 4 | **OR 条件中有一个无索引** | `WHERE indexed_col = 1 OR non_indexed = 2` | 给所有 OR 列加索引 |
| 5 | **不满足最左前缀** | 联合索引 `(a,b)` 但只查 `WHERE b = 2` | 遵循最左前缀或建立新索引 |
| 6 | **NOT IN / NOT EXISTS / !=** | `WHERE status != 1` | 视数据分布判断，有时优化器会放弃索引 |
| 7 | **优化器认为全表扫描更快** | 返回数据量超过表的 30% | 一般不需要处理，优化器的判断通常是正确的 |

> **快速判断方法：** 使用 `EXPLAIN` 查看 `type` 列，如果是 `ALL` 表示全表扫描，`index` 表示索引全扫描，`range` 表示索引范围扫描。

### 14. 什么是回表查询？什么是索引下推（ICP）？🔥🔥 ⭐⭐

**回表查询：**
通过二级索引找到主键值 → 再回到聚簇索引查找完整行数据的过程。

```
二级索引 B+ 树查找 name='张三' → 得到主键 id=5
→ 回到聚簇索引 B+ 树查找 id=5 → 得到完整行数据
```

**索引下推（Index Condition Pushdown，ICP）：**
MySQL 5.6 引入的优化。在联合索引 `(name, age)` 的场景下：

```sql
SELECT * FROM user WHERE name LIKE '张%' AND age = 25;
```

- **无 ICP**：存储引擎通过二级索引找到所有 `name LIKE '张%'` 的主键 → 全部回表 → Server 层再过滤 `age = 25`
- **有 ICP**：存储引擎在二级索引中直接判断 `age = 25` 是否满足 → 只对满足条件的记录回表

ICP 减少了回表次数，在 EXPLAIN 的 Extra 列中显示 `Using index condition`。

## 四、事务

### 15. 事务的 ACID 特性是什么？分别怎么实现的？🔥🔥🔥 ⭐⭐⭐

| 特性 | 含义 | 实现机制 |
|------|------|---------|
| **A - 原子性** | 事务中的操作要么全部成功，要么全部回滚 | **undo log**（回滚日志）：记录数据修改前的状态，回滚时通过 undo log 恢复 |
| **C - 一致性** | 事务执行前后，数据库从一个一致状态转到另一个一致状态 | 是**目标**而非实现，由 A+I+D 共同保证 |
| **I - 隔离性** | 并发事务之间互不干扰 | **MVCC + 锁机制**：读操作通过 MVCC 实现非锁定读，写操作通过锁保证互斥 |
| **D - 持久性** | 事务一旦提交，数据永久保存 | **redo log**（重做日志）：WAL 机制，先写日志再写磁盘，crash 后通过 redo log 恢复 |

> **一致性是最终目标**：一致性不是由单一机制保证的，而是原子性（undo log）、隔离性（锁+MVCC）、持久性（redo log）三者共同作用的结果。

---

### 16. 事务的隔离级别有哪些？🔥🔥🔥 ⭐⭐

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|---------|------|-----------|------|------|
| **READ UNCOMMITTED** 读未提交 | ✅ 可能 | ✅ 可能 | ✅ 可能 | 能读到未提交的数据，基本不用 |
| **READ COMMITTED** 读已提交 | ❌ 不会 | ✅ 可能 | ✅ 可能 | Oracle 默认级别 |
| **REPEATABLE READ** 可重复读 | ❌ 不会 | ❌ 不会 | ⚠️ 特殊情况下可能 | **InnoDB 默认级别** |
| **SERIALIZABLE** 串行化 | ❌ 不会 | ❌ 不会 | ❌ 不会 | 完全串行执行，性能差 |

**三种并发问题：**

```
脏读：事务A读到了事务B【未提交】的数据，事务B随后回滚
不可重复读：事务A两次读取同一行，中间事务B【修改并提交】了这行，两次结果不同
幻读：事务A两次执行相同查询，中间事务B【插入并提交】了新行，第二次多了数据
```

> **InnoDB 的 RR 级别如何解决幻读？**
> 通过 **MVCC（快照读）** 和 **Next-Key Lock（当前读）** 组合解决。但存在特殊场景的幻读漏洞：事务先快照读（无锁）后当前读（加锁）之间，另一个事务插入了新行。

### 17. 什么是 MVCC？Read View 的原理是什么？🔥🔥🔥 ⭐⭐⭐

**MVCC（Multi-Version Concurrency Control，多版本并发控制）：**
不通过加锁而是通过**数据版本链**来实现并发读写互不阻塞。

**核心组件：**

**① 隐藏字段（每行记录都有）：**
- `DB_TRX_ID`：最后一次修改该行的事务 ID
- `DB_ROLL_PTR`：回滚指针，指向 undo log 中该行的前一个版本

**② undo log 版本链：**
每次修改都会在 undo log 中保存旧版本，通过 `DB_ROLL_PTR` 形成链表：

```
当前行数据 (trx_id=5)
    ↓ roll_pointer
旧版本数据 (trx_id=3)
    ↓ roll_pointer
更旧版本数据 (trx_id=1)
```

**③ Read View（读视图）：**
事务执行快照读时生成的一致性视图，包含四个关键字段：

| 字段 | 含义 |
|------|------|
| `m_ids` | 生成 Read View 时，当前活跃（未提交）的事务 ID 列表 |
| `min_trx_id` | `m_ids` 中最小的事务 ID |
| `max_trx_id` | 下一个即将分配的事务 ID（当前最大事务 ID + 1） |
| `creator_trx_id` | 创建该 Read View 的事务 ID |

**可见性判断规则：**
对于版本链中某个版本的 `trx_id`：
1. `trx_id == creator_trx_id` → ✅ 自己的修改，可见
2. `trx_id < min_trx_id` → ✅ 该事务在 Read View 创建前已提交，可见
3. `trx_id >= max_trx_id` → ❌ 该事务在 Read View 创建后才开始，不可见
4. `min_trx_id <= trx_id < max_trx_id` → 检查是否在 `m_ids` 中：
   - 在 → ❌ 事务未提交，不可见，沿版本链继续找
   - 不在 → ✅ 事务已提交，可见

**RC vs RR 的区别：**
- **RC（读已提交）**：每次 SELECT 都生成新的 Read View → 能看到其他事务已提交的最新数据
- **RR（可重复读）**：只在事务中第一次 SELECT 时生成 Read View，后续复用 → 保证可重复读

## 五、锁机制

### 18. MySQL 有哪些锁？🔥🔥🔥 ⭐⭐

**按粒度分类：**

```
MySQL 锁体系
├── 全局锁：FTWRL（Flush Tables With Read Lock），整库只读，用于全库备份
├── 表级锁
│   ├── 表锁：LOCK TABLES ... READ/WRITE
│   ├── 元数据锁（MDL）：自动加，防止 DDL 与 DML 冲突
│   ├── 意向锁（IS/IX）：InnoDB 自动加，快速判断表中是否有行锁
│   └── AUTO-INC 锁：自增主键插入时的锁
└── 行级锁（InnoDB 独有）
    ├── 记录锁（Record Lock）：锁住单条索引记录
    ├── 间隙锁（Gap Lock）：锁住索引记录之间的间隙，防止插入
    └── 临键锁（Next-Key Lock）：记录锁 + 间隙锁，左开右闭区间
```

---

### 19. 意向锁是什么？有什么作用？🔥🔥 ⭐⭐

**意向锁（Intention Lock）是表级锁**，分为：
- **意向共享锁（IS）**：事务打算对某行加共享锁（S 锁）前，先在表级别加 IS 锁
- **意向排他锁（IX）**：事务打算对某行加排他锁（X 锁）前，先在表级别加 IX 锁

**作用：快速判断表中是否存在行锁。**

```
没有意向锁：事务A要加表锁 → 需要遍历每一行判断是否有行锁 → O(n) 效率极低
有意向锁：事务A要加表锁 → 检查表上是否有意向锁即可 → O(1) 快速判断
```

**兼容性矩阵：**

|  | IS | IX | S（表锁） | X（表锁） |
|--|----|----|----------|----------|
| **IS** | ✅ | ✅ | ✅ | ❌ |
| **IX** | ✅ | ✅ | ❌ | ❌ |
| **S（表锁）** | ✅ | ❌ | ✅ | ❌ |
| **X（表锁）** | ❌ | ❌ | ❌ | ❌ |

---

### 20. 什么是记录锁、间隙锁、临键锁？🔥🔥 ⭐⭐⭐

**记录锁（Record Lock）：**
- 锁住**单条索引记录**，不是数据行
- `SELECT * FROM t WHERE id = 5 FOR UPDATE` → 锁住 id=5 这条索引记录

**间隙锁（Gap Lock）：**
- 锁住索引记录之间的**间隙**，防止其他事务在间隙中插入新记录
- 唯一目的：**防止幻读**
- `SELECT * FROM t WHERE id > 5 AND id < 10 FOR UPDATE` → 锁住 (5, 10) 之间的间隙
- **间隙锁之间不互斥**，多个事务可以同时对同一间隙加间隙锁
- **间隙锁与插入操作互斥**

**临键锁（Next-Key Lock）：**
- = 记录锁 + 间隙锁
- 锁住的是一个**左开右闭区间**
- 是 InnoDB 在 RR 级别下的**默认加锁方式**
- `SELECT * FROM t WHERE id >= 5 FOR UPDATE` → 锁住 (前一个记录, 5] + (5, 下一个记录] 区间

> **特别说明：** 在唯一索引上等值查询匹配到记录时，Next-Key Lock 会退化为记录锁（因为不会插入相同值，不需要间隙锁）。

### 21. 当前读和快照读的区别？🔥🔥 ⭐

| 类型 | 说明 | 示例 | 加锁情况 |
|------|------|------|---------|
| **快照读** | 读取 MVCC 中的历史版本 | 普通 SELECT | 不加锁 |
| **当前读** | 读取最新已提交数据 | SELECT ... FOR UPDATE / LOCK IN SHARE MODE / INSERT / UPDATE / DELETE | 加锁（记录锁/间隙锁/临键锁） |

---

### 22. 乐观锁和悲观锁的区别？🔥🔥 ⭐

| 特性 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| **思想** | 假设一定会冲突，先加锁 | 假设不会冲突，提交时检查 |
| **实现** | `SELECT ... FOR UPDATE` | 版本号（version 字段）或 CAS |
| **SQL 示例** | `SELECT * FROM t WHERE id=1 FOR UPDATE` | `UPDATE t SET val=val+1, version=version+1 WHERE id=1 AND version=old_version` |
| **适用场景** | 写多读少、冲突频繁 | 读多写少、冲突较少 |
| **性能** | 加锁有开销，但保证一致性 | 无锁开销，但冲突时需要重试 |

### 23. 什么是死锁？如何避免？🔥🔥 ⭐⭐

**死锁条件：** 两个事务互相等待对方持有的锁。

```
事务A: 先锁 id=1 → 等待 id=2
事务B: 先锁 id=2 → 等待 id=1
→ 互相等待 → 死锁
```

**InnoDB 死锁检测：** 通过 wait-for graph（等待图）自动检测死锁，回滚代价较小的事务。

**避免死锁的方法：**
1. 尽量以相同的顺序访问资源（如都按 id 从小到大加锁）
2. 保持事务简短，减少持锁时间
3. 合理使用索引，避免行锁升级为表锁
4. 设置锁等待超时：`innodb_lock_wait_timeout`（默认 50 秒）

## 六、日志系统

### 24. redo log、undo log、binlog 的区别？🔥🔥🔥 ⭐⭐⭐

| 特性 | redo log | undo log | binlog |
|------|----------|----------|--------|
| **作用** | 保证持久性（crash-safe） | 保证原子性（回滚）+ MVCC | 主从复制、数据恢复 |
| **引擎** | InnoDB 独有 | InnoDB 独有 | Server 层，所有引擎通用 |
| **内容** | 物理日志（哪个页做了什么修改） | 逻辑日志（反向操作：INSERT→DELETE） | 逻辑日志（SQL 或行变更） |
| **写入方式** | 循环写（固定大小，会覆盖旧日志） | 随事务写入，不覆盖 | 追加写（不会覆盖） |
| **WAL 机制** | ✅ 先写日志再写磁盘 | — | — |

**关键理解：**
- **redo log** 保证的是 MySQL 异常重启后数据不丢失（持久性）
- **undo log** 保证的是事务回滚时能恢复到修改前的状态（原子性），同时是 MVCC 的核心
- **binlog** 是 MySQL Server 层的日志，主要用途是主从复制和基于时间点的数据恢复

---

### 25. 什么是两阶段提交？为什么需要两阶段提交？🔥🔥 ⭐⭐⭐

redo log 和 binlog 需要保持一致性，两阶段提交确保两者数据一致：

```
阶段一（prepare）：写入 redo log，状态标记为 prepare
阶段二（commit）：写入 binlog → 将 redo log 状态标记为 commit
```

**不用两阶段提交会出现的问题（以 UPDATE 为例）：**

| 情况 | redo log | binlog | 结果 |
|------|----------|--------|------|
| 先写 redo 后写 binlog，写 binlog 前崩溃 | ✅ 已写入 | ❌ 未写入 | 主从数据不一致（从库缺少该操作） |
| 先写 binlog 后写 redo，写 redo 前崩溃 | ❌ 未写入 | ✅ 已写入 | 主从数据不一致（主库少了该操作） |

**两阶段提交的恢复规则：**
- redo log 是 prepare 状态，且 binlog 完整 → 提交事务
- redo log 是 prepare 状态，但 binlog 不完整 → 回滚事务

---

### 26. 什么是 Double Write（双写）？🔥 ⭐⭐

**问题场景：** MySQL 的页大小是 16KB，而操作系统的一次 IO 通常以 4KB 为单位。如果正在写 16KB 的页时发生崩溃，可能出现**页撕裂（partial write）**——只写了一部分。

**Double Write 机制：**
1. 脏页刷盘前，先将页数据写入**共享表空间的 double write buffer**（2MB 连续空间）
2. 写入 double write buffer 成功后，再写入实际的数据文件
3. 如果实际写入时崩溃，恢复时可以从 double write buffer 找到完整页来恢复

> **注意：** redo log 记录的是"对某个页做了什么修改"，不是完整页数据。如果页本身已经损坏（撕裂），redo log 无法恢复。Double Write 解决的是"页本身损坏"的问题。

### 27. 如何使用 EXPLAIN 分析 SQL？🔥🔥🔥 ⭐⭐

```sql
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

**关键字段解读：**

| 字段 | 含义 | 关注点 |
|------|------|--------|
| **type** | 访问类型 | 从好到差：`system > const > eq_ref > ref > range > index > ALL`，目标至少达到 `range` |
| **key** | 实际使用的索引 | 如果为 NULL 说明没走索引 |
| **rows** | 预估扫描行数 | 越少越好 |
| **Extra** | 额外信息 | 重点关注以下值 |
| **possible_keys** | 可能用到的索引 | 与 key 对比，如果不一致说明优化器选择了其他索引 |

**Extra 列重要值：**

| 值 | 含义 |
|----|------|
| `Using index` | ✅ 覆盖索引，不需要回表 |
| `Using index condition` | ✅ 索引下推（ICP） |
| `Using where` | ⚠️ Server 层过滤，说明存储引擎返回了不需要的行 |
| `Using filesort` | ❌ 需要额外排序，消耗大 |
| `Using temporary` | ❌ 使用临时表，通常需要优化 |
| `Using join buffer` | ❌ JOIN 没有使用索引，使用了连接缓冲区 |

4. **架构优化**：
   - 读写分离
   - 分库分表

### 29. 大表如何优化？分库分表的策略？🔥🔥 ⭐⭐

**垂直拆分：**
- **垂直分库**：按业务拆分（用户库、订单库、商品库）
- **垂直分表**：将不常用的列拆到扩展表（如商品详情单独成表）

**水平拆分：**
- **水平分库**：相同表结构分布到不同数据库（如按用户 ID 取模分到不同库）
- **水平分表**：同一个库中拆分为多个结构相同的表（如 order_0, order_1, order_2）

**分片键选择原则：**
1. 数据分布均匀
2. 查询尽量能通过分片键定位到具体分片（避免广播查询）
3. 常用查询条件中的字段

**分库分表带来的问题：**
- 分布式事务
- 跨库 JOIN
- 全局唯一 ID
- 数据迁移
- 聚合查询困难

### 30. 分布式 ID 有哪些方案？🔥 ⭐⭐

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **UUID** | 128 位随机值 | 简单、无依赖 | 无序（B+ 树频繁页分裂）、太长（36 字符） |
| **数据库自增** | 单独的 ID 表 | 有序、简单 | 单点瓶颈 |


## 八、架构与高可用

### 31. MySQL 主从复制的原理？🔥🔥 ⭐⭐

```
主库 (Master)                    从库 (Slave)
    │                                │
    │  1. 事务提交，写入 binlog        │
    │──────────────────────────────→ │
    │      (从库 IO 线程拉取 binlog)    │
    │                                │ 2. 写入 relay log（中继日志）
    │                                │
    │                                │ 3. SQL 线程执行 relay log 中的事件
    │                                │    → 数据写入从库
```

**三种同步模式：**

| 模式 | 说明 | 优缺点 |
|------|------|--------|
| **异步复制** | 主库不等从库确认 | 性能最好，但主库崩溃可能丢数据 |
| **半同步复制** | 至少一个从库确认收到才返回 | 兼顾性能和数据安全 |
| **全同步复制** | 所有从库都执行完才返回 | 最安全但性能最差 |

**主从延迟的原因与解决：**
- 原因：从库单线程重放（MySQL 5.7+ 支持多线程并行复制）、大事务、网络延迟
- 解决：开启并行复制、拆分大事务、使用半同步复制

### 32. 读写分离的实现方案？🔥 ⭐

**实现方式：**
1. **代码层面**：在应用中配置主从数据源，写操作走主库，读操作走从库
2. **中间件**：使用 MyCat、ShardingSphere 等中间件自动路由
3. **代理层**：MySQL Router、ProxySQL 等

**需要解决的问题：**
- **主从延迟导致读到旧数据**：关键业务（如刚写入后立即读取）可以强制走主库
- **事务中的读写**：同一事务中的读写应走同一数据源

### 33. COUNT(1)、COUNT(*)、COUNT(列名) 有什么区别？🔥 ⭐

| 写法 | 含义 | 是否统计 NULL |
|------|------|--------------|
| `COUNT(*)` | 统计总行数 | ✅ 包含 NULL 行 |
| `COUNT(1)` | 统计总行数 | ✅ 包含 NULL 行 |
| `COUNT(列名)` | 统计该列非 NULL 的行数 | ❌ 不包含 NULL |

> **性能：** `COUNT(*)` 和 `COUNT(1)` 在 InnoDB 中没有本质区别，MySQL 优化器会自动选择最小的索引来遍历（减少 IO）。`COUNT(列名)` 如果该列有 NOT NULL 约束，则等同于 `COUNT(*)`。

### 34. NULL 值在 MySQL 中有哪些需要注意的？🔥 ⭐

1. **索引失效风险**：`WHERE col = NULL` 不会命中索引，必须用 `IS NULL` 或 `IS NOT NULL`
2. **比较运算**：`NULL = NULL` → NULL（不是 true），`NULL <> 1` → NULL（不是 true）
3. **聚合函数**：`COUNT(col)` 不统计 NULL 行，`SUM/AVG` 忽略 NULL 值
4. **排序**：`ORDER BY col ASC` 时 NULL 排在最前面
5. **DISTINCT**：`DISTINCT` 会将所有 NULL 视为相同值
6. **建议**：尽量使用 `NOT NULL` 约束 + 默认值，避免 NULL 带来的不确定性

### 35. ORDER BY 是怎么工作的？Using filesort 的优化？🔥 ⭐

**两种排序方式：**

1. **Using index（索引排序）**：如果 ORDER BY 的列有索引，直接利用索引的有序性，不需要额外排序
2. **Using filesort（文件排序）**：没有可用索引，需要在内存/磁盘中排序

**filesort 的两种算法：**
- **双路排序（旧版）**：第一次 IO 读取排序字段，排序后第二次 IO 读取完整数据
- **单路排序（现版）**：一次 IO 读出所有需要的字段到 `sort_buffer`，直接排序返回
  - `sort_buffer_size` 参数控制缓冲区大小
  - 数据量超过 `sort_buffer` 时使用临时文件（归并排序），性能下降

**优化方法：**
```sql
-- 利用联合索引避免 filesort
ALTER TABLE user ADD INDEX idx_age_name (age, name);
SELECT * FROM user WHERE age > 18 ORDER BY name;  -- 需要额外排序 ❌
SELECT * FROM user ORDER BY age, name;              -- 走索引排序 ✅
```

### 38. varchar(50) 中 50 的含义？最多能存多少？🔥 ⭐

- 50 表示**最多 50 个字符**（不是字节）
- MySQL 规定每行所有字段总长度不超过 65535 字节
- 在 utf8mb4 编码下，每个字符最多占 4 字节
- 所以 `VARCHAR(50)` 实际最多占用 `50 × 4 + 2(长度字节) = 202` 字节
- `VARCHAR` 最大能存的字符数受字符集和行总长度限制：`utf8mb4 下约 16383 个字符`

### 40. 如何解决深分页问题（LIMIT 1000000, 10）？🔥🔥 ⭐⭐

**问题：** `LIMIT 1000000, 10` 需要先扫描 1000010 行再丢弃前 1000000 行，效率极低。

**方案一：延迟关联（推荐）**
```sql
-- 先通过子查询在索引上定位 id（覆盖索引，无需回表）
SELECT * FROM user
INNER JOIN (SELECT id FROM user ORDER BY id LIMIT 1000000, 10) AS tmp
ON user.id = tmp.id;
```

**方案二：游标分页（最佳性能）**
```sql
-- 记录上一页最后一条的 id，下次从该 id 开始
SELECT * FROM user WHERE id > last_id ORDER BY id LIMIT 10;
```

**方案三：禁止深分页**
- 产品层面限制页数（如 Google 只显示前 10 页）
- 提供更精确的筛选条件缩小结果集

### 41. 什么是 WAL（Write-Ahead Logging）机制？🔥 ⭐

**WAL 的核心思想：先写日志，再写磁盘。**

具体流程：
1. 事务修改数据时，先将修改记录写入 redo log buffer（内存）
2. 事务提交时，将 redo log buffer 刷盘到 redo log file
3. 然后在合适时机将内存中的脏页（修改过的数据页）刷盘到数据文件

**好处：**
- 顺序写日志比随机写磁盘快得多（redo log 是顺序写，数据文件是随机写）
- 即使刷脏页前崩溃，也能通过 redo log 恢复数据
- 将随机 IO 转化为顺序 IO，大幅提升性能

### 42. Buffer Pool 的作用和优化？🔥 ⭐

**Buffer Pool 是 InnoDB 的内存缓存池**，缓存数据页和索引页，减少磁盘 IO。

**关键参数：**
- `innodb_buffer_pool_size`：建议设置为物理内存的 60%-80%
- `innodb_buffer_pool_instances`：多实例减少锁竞争（建议每个实例 >= 1GB）

**LRU 改进 — 冷热分区：**
InnoDB 的 LRU 不是简单的 LRU，而是分为**年轻代（young）** 和**老年代（old）**：
- 新读入的页先放入老年代（old 区头部，占 3/8）
- 只有在老代被再次访问且超过 `innodb_old_blocks_time`（默认 1 秒）后才移入年轻代
- 这防止了**全表扫描**将热数据挤出 Buffer Pool 的问题
