> 标注频率：🔥高频 / 🔥🔥必考 / 🔥🔥🔥核心必背

## 一、JVM 基础架构

### 1. 什么是 JVM？🔥🔥🔥 ⭐

JVM（Java Virtual Machine）是 Java 实现跨平台的基石。

**运行流程**：
1. Java 源代码（`.java`）→ `javac` 编译 → 字节码（`.class`）
2. JVM 加载字节码 → 解释器逐行翻译为机器码 / JIT 编译热点代码 → 操作系统执行

**核心特性**：
- **跨平台**：一次编译，处处运行（Write Once, Run Anywhere）
- **自动内存管理**：垃圾回收器自动回收不再使用的对象
- **JIT 即时编译**：热点代码编译为本地机器码缓存到 CodeCache，提升执行效率
- **跨语言**：Groovy、Kotlin、Scala 等语言也能在 JVM 上运行

---

### 2. JVM 的组织架构？🔥🔥 ⭐⭐

JVM 大致分为三个部分：

| 组件 | 职责 |
|------|------|
| **类加载器子系统** | 从文件系统、网络等来源加载 Class 文件到内存 |
| **运行时数据区** | 方法区、堆、虚拟机栈、程序计数器、本地方法栈 |
| **执行引擎** | 解释器 + JIT 编译器 + 垃圾回收器 |

**关键点**：每个 Java 程序对应一个独立的 JVM 进程，进程间完全隔离。

## 二、内存管理

### 3. JVM 的内存区域？🔥🔥🔥 ⭐⭐⭐

根据 JDK 8 规范，JVM 运行时内存分为五个部分：

| 区域 | 线程归属 | 存储内容 | 异常 |
|------|---------|---------|------|
| **程序计数器** | 线程私有 | 当前线程执行的字节码行号 | 唯一不会 OOM 的区域 |
| **虚拟机栈** | 线程私有 | 栈帧（局部变量表、操作数栈、动态链接、方法出口） | StackOverflowError / OOM |
| **本地方法栈** | 线程私有 | Native 方法的调用信息 | StackOverflowError / OOM |
| **堆** | 线程共享 | 对象实例、数组（GC 主要区域） | OutOfMemoryError |
| **方法区（元空间）** | 线程共享 | 类信息、常量、静态变量、JIT 编译后的代码 | OutOfMemoryError |

**补充**：
- **运行时常量池**：方法区的一部分，存放编译期生成的字面量和符号引用
- **直接内存**：不属于 JVM 规范，NIO 使用的堆外内存

---

### 4. JDK 1.6 / 1.7 / 1.8 内存区域的变化？🔥🔥 ⭐⭐

| 版本 | 方法区实现 | 字符串常量池 | 静态变量 |
|------|-----------|-------------|---------|
| **JDK 1.6** | 永久代（PermGen） | 永久代 | 永久代 |
| **JDK 1.7** | 永久代 | **移到堆中** | **移到堆中** |
| **JDK 1.8** | **元空间（Metaspace）**，使用本地内存 | 堆中 | 堆中 |

**为什么用元空间替代永久代？**
- 永久代大小受 `-XX:MaxPermSize` 限制，容易 OOM
- 元空间使用本地内存，大小仅受系统内存限制
- 统一 HotSpot 和 JRockit 虚拟机的实现

---

### 5. 堆和栈的区别？🔥🔥🔥 ⭐⭐

| 维度 | 堆（Heap） | 栈（Stack） |
|------|-----------|------------|
| **存储内容** | 对象实例、数组 | 局部变量、方法参数、对象引用 |
| **线程归属** | 线程共享 | 线程私有 |
| **生命周期** | GC 管理，不确定 | 随方法调用结束自动释放 |
| **存取速度** | 较慢 | 较快（LIFO） |
| **空间大小** | 大，动态扩展 | 小，固定 |
| **异常** | OutOfMemoryError | StackOverflowError / OOM |

**栈中存的是引用而非对象本身**：`MyObject obj = new MyObject()` 中 `obj` 是栈上的引用（8字节/64位系统），指向堆中的对象实例。

---

### 6. 堆的内存分区？🔥🔥🔥 ⭐⭐

```
Java 堆
├── 新生代（Young Generation）
│   ├── Eden 区（80%）
│   ├── Survivor 0 / From（10%）
│   └── Survivor 1 / To（10%）
└── 老年代（Old / Tenured Generation）
```

**默认比例**：`Eden : S0 : S1 = 8 : 1 : 1`，新生代 : 老年代 = `1 : 2`

**对象流转过程**：
1. 新对象先在 Eden 区分配
2. Eden 满时触发 Minor GC，存活对象复制到 Survivor 区
3. 每次 Minor GC 后年龄 +1，超过阈值（默认 15）晋升老年代
4. 大对象直接进入老年代

---

### 7. 对象什么时候进入老年代？🔥🔥 ⭐⭐

| 条件 | 说明 |
|------|------|
| **年龄达到阈值** | 默认 15（`-XX:MaxTenuringThreshold`），经历 15 次 Minor GC 仍存活 |
| **大对象直接分配** | 超过 `-XX:PretenureSizeThreshold` 阈值的对象直接进老年代 |
| **动态年龄判定** | Survivor 区中同年龄对象总大小超过 Survivor 空间一半，年龄 ≥ 该年龄的对象直接晋升 |
| **空间分配担保** | Minor GC 后 Survivor 放不下，通过担保机制进入老年代 |

---

### 8. 对象创建的过程？🔥🔥🔥 ⭐⭐

```
new 指令 → 类加载检查 → 分配内存 → 初始化零值 → 设置对象头 → 执行 <init>
```

1. **类加载检查**：检查类是否已被加载、解析和初始化
2. **分配内存**：指针碰撞（内存规整）或空闲列表（内存不规整）
3. **初始化零值**：保证实例字段可直接使用（int→0, boolean→false, 对象→null）
4. **设置对象头**：类元数据、哈希码、GC 分代年龄等
5. **执行 `<init>`**：构造方法，按程序员的意图初始化

**线程安全**：TLAB（Thread Local Allocation Buffer）为每个线程预留私有分配区域，避免并发冲突。

---

### 9. 对象的内存布局？🔥🔥 ⭐⭐

在 HotSpot 虚拟机中，对象由三部分组成：

| 部分 | 内容 | 大小（64位 JVM，开启压缩指针） |
|------|------|------|
| **对象头** | Mark Word（哈希码、GC 年龄、锁状态）+ 类型指针 | 12 字节 |
| **实例数据** | 对象的成员变量值 | 按字段类型计算 |
| **对齐填充** | 保证对象大小为 8 字节的倍数 | 补齐到 8 的倍数 |

**示例**：`new Object()` 在 64 位 JVM（开启压缩指针）下 = 12（对象头）+ 4（填充）= **16 字节**

---

### 10. 四种引用类型？🔥🔥 ⭐⭐

| 类型 | 类 | GC 行为 | 使用场景 |
|------|-----|---------|---------|
| **强引用** | 直接赋值 `A a = new A()` | 永不回收，除非引用断开 | 日常编程 |
| **软引用** | `SoftReference` | 内存不足时回收 | 缓存 |
| **弱引用** | `WeakReference` | 下次 GC 一定回收 | ThreadLocal、WeakHashMap |
| **虚引用** | `PhantomReference` | 随时可能回收，必须配合 ReferenceQueue | 跟踪 GC、管理堆外内存 |

---

### 11. 对象一定分配在堆中吗？🔥 ⭐⭐

**不一定**。

JDK 7 开始默认开启**逃逸分析**，如果对象没有逃逸出方法（即不被方法外部引用），可以在**栈上分配**：

```java
public void method() {
    Person p = new Person();  // 未逃逸，可能在栈上分配
    System.out.println(p.name);
}
```

**逃逸分析带来的优化**：
- **栈上分配**：减轻 GC 压力
- **同步消除**：对象未逃逸出线程，可消除同步锁
- **标量替换**：将对象拆解为基本类型直接使用

---

### 12. 内存溢出和内存泄漏？🔥🔥🔥 ⭐⭐

**内存溢出（OOM）**：程序申请内存时没有足够空间，抛出 `OutOfMemoryError`。

**内存泄漏**：程序使用完内存后未能释放，导致可用内存逐渐减少，最终引发 OOM。

| 内存溢出类型 | 触发条件 |
|-------------|---------|
| **堆溢出** | `java.lang.OutOfMemoryError: Java heap space` — 大对象分配或内存泄漏 |
| **栈溢出** | `StackOverflowError` — 递归调用过深或栈帧过大 |
| **元空间溢出** | `java.lang.OutOfMemoryError: Metaspace` — 动态生成类过多 |
| **直接内存溢出** | `java.lang.OutOfMemoryError: Direct buffer memory` — NIO 使用不当 |

**内存泄漏常见原因**：
1. 静态集合持有对象引用未清理
2. 数据库/IO/Socket 连接未关闭
3. ThreadLocal 未调用 `remove()`
4. 单例对象持有外部引用
5. 事件监听器未注销

## 三、垃圾回收

### 13. 如何判断对象是否存活？🔥🔥🔥 ⭐⭐⭐

**可达性分析算法**（JVM 使用）：

从一组 **GC Roots** 出发，沿引用链向下搜索，如果一个对象到 GC Roots 没有任何引用链，则该对象不可达，可以被回收。

**可作为 GC Roots 的对象**：
1. 虚拟机栈中引用的对象（方法的局部变量、参数）
2. 本地方法栈中 JNI 引用的对象
3. 方法区中类静态变量引用的对象
4. 运行时常量池中的常量引用的对象
5. 活跃线程的引用

**为什么不使用引用计数法？**
- 无法解决**循环引用**问题（A 引用 B，B 引用 A，计数器永不为 0）
- Python 等语言使用引用计数，但需要额外的循环引用检测机制

---

### 14. 垃圾收集算法？🔥🔥🔥 ⭐⭐⭐

| 算法 | 原理 | 优点 | 缺点 | 适用区域 |
|------|------|------|------|---------|
| **标记-清除** | 标记可回收对象，统一清除 | 简单 | 内存碎片 | 老年代（CMS） |
| **标记-复制** | 将存活对象复制到另一半空间 | 无碎片 | 浪费 50% 空间 | 新生代 |
| **标记-整理** | 存活对象向一端移动，清理边界外 | 无碎片 | 移动对象开销大 | 老年代 |
| **分代收集** | 新生代用复制，老年代用标记-整理/清除 | 综合效率高 | 实现复杂 | 整个堆 |

**新生代为什么用复制算法？**
- 新生代对象存活率低（约 98% 是垃圾），需要复制的对象少
- Eden : S0 : S1 = 8 : 1 : 1，只浪费 10% 空间

---

### 15. Minor GC / Major GC / Full GC 的区别？🔥🔥 ⭐⭐

| GC 类型 | 作用范围 | 触发条件 | 停顿时间 |
|---------|---------|---------|---------|
| **Minor GC（Young GC）** | 新生代 | Eden 区空间不足 | 短（几 ms ~ 几十 ms） |
| **Major GC** | 老年代（CMS 特有） | 老年代空间不足 | 较长 |
| **Mixed GC** | 新生代 + 部分老年代（G1 特有） | G1 选择回收价值高的 Region | 中等 |
| **Full GC** | 整个堆 + 方法区 | 老年代不足 / System.gc() / 元空间不足 | 最长（应尽量避免） |

**Full GC 触发场景**：
1. `System.gc()` 调用（建议触发，不保证执行）
2. 老年代空间不足
3. 方法区/元空间空间不足
4. Minor GC 后存活对象大于老年代剩余空间
5. CMS 出现 Concurrent Mode Failure

---

### 16. STW（Stop The World）了解吗？🔥🔥 ⭐⭐

**STW**：GC 过程中暂停所有用户线程，确保对象引用不被修改。

**为什么需要 STW？**
- 可达性分析必须确保对象引用关系不变
- 如果分析期间引用变化，会导致标记不准确（漏标或多标）

**安全点（Safepoint）**：
- 线程只能在特定位置暂停（方法调用、循环跳转、异常处理等）
- 类比：老王拉车只能在平地休息，不能在上坡/下坡时停下

---

### 17. 有哪些垃圾收集器？🔥🔥🔥 ⭐⭐⭐

```
      新生代收集器                    老年代收集器
┌─────────────────┐          ┌──────────────────┐
│  Serial         │ ───────→ │  Serial Old      │
│  ParNew         │ ───────→ │  CMS             │
│  Parallel       │ ───────→ │  Parallel Old    │
└─────────────────┘          └──────────────────┘

      整堆收集器
┌─────────────────────────────────────────────┐
│  G1（JDK 9 默认）    ZGC（JDK 11+）          │
└─────────────────────────────────────────────┘
```

| 收集器 | 算法 | 特点 | 适用场景 |
|--------|------|------|---------|
| **Serial** | 复制 | 单线程，STW | 客户端模式、小内存 |
| **ParNew** | 复制 | Serial 的多线程版 | 配合 CMS 使用 |
| **Parallel Scavenge** | 复制 | 吞吐量优先 | 后台计算任务 |
| **CMS** | 标记-清除 | 低停顿，并发收集 | 对延迟敏感的应用 |
| **G1** | 标记-整理 | 可预测停顿，Region 化 | 大堆内存（JDK 9 默认） |
| **ZGC** | 标记-整理 | 超低延迟（<10ms），TB 级堆 | 金融、高并发低延迟 |

---

### 18. CMS 收集器详解？🔥🔥🔥 ⭐⭐⭐

**CMS（Concurrent Mark Sweep）** 以获取最短停顿时间为目标。

**四个阶段**：

| 阶段 | STW? | 说明 |
|------|------|------|
| ① 初始标记 | ✅ 是 | 标记 GC Roots 直接引用的对象，速度很快 |
| ② 并发标记 | ❌ 否 | 从 GC Roots 遍历整个对象图，耗时最长但并发执行 |
| ③ 重新标记 | ✅ 是 | 修正并发标记期间变动的引用（使用增量更新） |
| ④ 并发清除 | ❌ 否 | 清除未标记对象，并发执行 |

**CMS 的缺点**：
1. **内存碎片**：使用标记-清除算法，不压缩
2. **浮动垃圾**：并发清除阶段用户线程产生的新垃圾，只能下次 GC 回收
3. **Concurrent Mode Failure**：预留空间不够时退化为 Serial Old，停顿时间长
4. **CPU 敏感**：并发阶段占用 CPU 资源

---

### 19. G1 收集器详解？🔥🔥🔥 ⭐⭐⭐

**G1（Garbage First）** 适合大堆内存、追求停顿时间可控的场景。

**核心设计**：
- 将堆划分为多个大小相等的 **Region**（1~32MB）
- 每个 Region 可以是 Eden / Survivor / Old / Humongous（大对象）
- 优先回收垃圾最多的 Region（Garbage First 名称由来）

**运行过程**：

| 阶段 | 说明 |
|------|------|
| **初始标记** | STW，标记 GC Roots 直接引用的对象 |
| **并发标记** | 并发遍历对象图，使用写屏障记录引用变化 |
| **最终标记** | STW，处理并发标记期间的引用变化 |
| **筛选回收** | STW，选择回收价值高的 Region，复制存活对象 |

**G1 vs CMS**：

| 维度 | CMS | G1 |
|------|-----|-----|
| 算法 | 标记-清除 | 标记-整理（无碎片） |
| 停顿预测 | 不支持 | 支持设置期望停顿时间 |
| 浮动垃圾 | 有 | 较少 |
| 内存碎片 | 严重 | 无（Region 间整理） |
| 收集范围 | 仅老年代 | 整堆（分区管理） |

---

### 20. ZGC 了解吗？🔥 ⭐⭐

**ZGC** 是 JDK 11 引入的低延迟垃圾收集器。

**核心特性**：
- 停顿时间 < 10ms，甚至 < 1ms
- 支持 TB 级堆内存
- 停顿时间不随堆大小增长而增加

**关键技术**：
- **指针染色（Colored Pointers）**：在指针中嵌入标记信息，不需要额外标记位
- **读屏障（Load Barrier）**：在读取对象引用时检查指针状态，确保引用正确
- **并发整理**：与应用线程并发执行对象移动

**适用场景**：金融交易系统、电商平台等对延迟极度敏感的场景。

---

### 21. 三色标记法？🔥 ⭐⭐

**三色标记法**是并发标记阶段使用的标记算法，将对象分为三种颜色：

| 颜色 | 含义 |
|------|------|
| **白色** | 尚未被访问，GC 结束后仍为白色的对象将被回收 |
| **灰色** | 已被访问但其引用还未全部扫描 |
| **黑色** | 已被访问且其所有引用都已扫描完毕 |

**标记流程**：GC Roots → 标灰色 → 扫描引用 → 子节点标灰 → 当前节点标黑 → 重复

**并发标记时的问题**：
- **漏标（对象消失）**：黑对象删除了对白对象的引用，同时灰对象新增了对白对象的引用
  - CMS 使用**增量更新**（Incremental Update）：记录新增引用，重新标记时处理
  - G1 使用 **SATB（Snapshot At The Beginning）**：记录删除引用，保证开始时的快照一致性

---

### 22. finalize() 方法了解吗？🔥 ⭐

对象被判定不可达后，如果重写了 `finalize()` 且未被调用过，会被放入 F-Queue，由 Finalizer 线程执行。

**注意**：
- `finalize()` 只会被系统调用**一次**
- 在 `finalize()` 中可以让对象重新被引用（"刀下留人"），但不推荐
- **JDK 9 已标记 `finalize()` 为 deprecated**，推荐使用 `try-with-resources` 或 Cleaner API

## 四、类加载机制

### 23. 类加载过程？🔥🔥🔥 ⭐⭐⭐

类的生命周期包括 7 个阶段：

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
│         └─── 连接 ───┘   │
```

| 阶段 | 说明 |
|------|------|
| **加载** | 通过类的全限定名获取 `.class` 二进制字节流，生成 Class 对象 |
| **验证** | 确保字节流符合 JVM 规范（文件格式、元数据、字节码、符号引用） |
| **准备** | 为**静态变量**分配内存并设置**默认初始值**（`static int a = 1` → a = 0） |
| **解析** | 将常量池的**符号引用**替换为**直接引用**（指针、偏移量等） |
| **初始化** | 执行 `<clinit>()` 方法（静态变量赋值 + 静态代码块），**父类优先** |

**关键区分**：
- 准备阶段：`static int a = 1` → a = 0（默认值）
- 初始化阶段：a = 1（代码期望的值）
- `static final int b = 1` → 准备阶段就赋值为 1（编译期常量）

---

### 24. 类加载器有哪些？🔥🔥🔥 ⭐⭐

| 类加载器 | 加载范围 | 实现语言 |
|---------|---------|---------|
| **启动类加载器（Bootstrap）** | `JAVA_HOME/jre/lib`（rt.jar 等核心类库） | C++ |
| **扩展类加载器（Extension）** | `JAVA_HOME/jre/lib/ext` | Java |
| **应用程序类加载器（Application）** | classpath 下的类库 | Java |
| **自定义类加载器** | 根据需求定制（网络、加密文件等） | Java |

**注意**：启动类加载器无法被 Java 程序直接引用。

---

### 25. 双亲委派模型？🔥🔥🔥 ⭐⭐⭐

**核心原则**：当一个类加载器收到加载请求时，先委托**父加载器**尝试加载，只有父加载器无法加载时，子加载器才会自己加载。

```
自定义类加载器 → 应用程序类加载器 → 扩展类加载器 → 启动类加载器
     （自底向上委派，自顶向下尝试加载）
```

**双亲委派的作用**：
1. **保证类的唯一性**：同一个类只被加载一次
2. **保证安全性**：防止自定义类覆盖核心类（如自定义 `java.lang.String`）
3. **简化加载流程**：大部分类由正确的加载器处理

**举例**：自定义 `java.lang.String` → AppClassLoader 委派给 BootstrapClassLoader → BootstrapClassLoader 已加载 JDK 的 String → 直接返回，自定义 String 不会被加载。

---

### 26. 如何破坏双亲委派模型？🔥 ⭐⭐

1. **重写 `loadClass()` 方法**（而不是 `findClass()`）
2. **SPI 机制**（如 JDBC）：接口在 `rt.jar` 由 Bootstrap 加载，实现类在 classpath 需要用线程上下文类加载器（Thread Context ClassLoader）加载
3. **OSGi / 模块化**：网状加载而非树状委派
4. **Tomcat**：每个 Web 应用有独立的 WebAppClassLoader，优先加载应用自己的类
5. **热部署**：Spring DevTools 自定义类加载器，优先加载新版本类

---

### 27. 类卸载条件？🔥 ⭐

一个类要被卸载，需**同时满足**三个条件：
1. 该类所有实例都已被回收
2. 加载该类的 ClassLoader 已被回收（**最难满足**）
3. 类对应的 `java.lang.Class` 对象没有任何地方被引用

### 28. JVM 常用参数？🔥🔥 ⭐⭐

**堆内存配置**：

| 参数 | 说明 | 示例 |
|------|------|------|
| `-Xms` | 初始堆大小 | `-Xms2g` |
| `-Xmx` | 最大堆大小 | `-Xmx4g` |
| `-Xmn` | 新生代大小 | `-Xmn1g` |
| `-XX:NewRatio` | 新生代:老年代比例 | `-XX:NewRatio=2`（老年代是新生代的 2 倍） |
| `-XX:SurvivorRatio` | Eden:Survivor 比例 | `-XX:SurvivorRatio=8` |

**GC 收集器配置**：

| 参数 | 收集器组合 |
|------|-----------|
| `-XX:+UseSerialGC` | Serial + Serial Old |
| `-XX:+UseParallelGC` | Parallel Scavenge + Serial Old |
| `-XX:+UseParallelOldGC` | Parallel Scavenge + Parallel Old |
| `-XX:+UseParNewGC -XX:+UseConcMarkSweepGC` | ParNew + CMS |
| `-XX:+UseG1GC` | G1 |
| `-XX:+UseZGC` | ZGC |

**GC 日志**：

```
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log
```

**推荐生产配置**：
```
-Xms4g -Xmx4g -XX:+UseG1GC -XX:MaxGCPauseMillis=200 
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heapdump.hprof
```

### 29. CPU 占用过高怎么排查？🔥🔥🔥 ⭐⭐

**排查步骤**：

```bash
# 1. 找到占用 CPU 最高的 Java 进程
top

# 2. 查看该进程中各线程的 CPU 占用
top -Hp <pid>

# 3. 将线程 ID 转为十六进制
printf "%x\n" <tid>

# 4. 导出线程堆栈
jstack <pid> > thread_dump.txt

# 5. 在堆栈中搜索十六进制线程 ID，定位问题代码
grep "0x<hex_tid>" thread_dump.txt
```

**常见原因**：死循环、频繁 GC、线程死锁、正则回溯、加密运算。

### 30. 内存飙高怎么排查？🔥🔥 ⭐⭐

```bash
# 1. 查看 GC 情况
jstat -gcutil <pid> 1000

# 2. 查看占用内存最多的对象类型
jmap -histo <pid> | head -20

# 3. 导出堆转储文件
jmap -dump:format=b,file=heap.hprof <pid>

# 4. 使用 MAT / VisualVM / JProfiler 分析 dump 文件
```

**分析方向**：
- 是否存在内存泄漏（对象持续增长无法回收）
- 是否有大量大对象直接进入老年代
- 年轻代/老年代比例是否合理

---

### 31. 频繁 Full GC 怎么办？🔥🔥 ⭐⭐

**常见原因及对策**：

| 原因 | 排查方法 | 解决方案 |
|------|---------|---------|
| **老年代空间不足** | `jstat -gc` 查看老年代使用率 | 增大老年代或排查长生命周期对象 |
| **内存泄漏** | dump 分析，找到无法回收的对象 | 修复代码中的引用未释放 |
| **大对象频繁分配** | `jmap -histo` 查看大对象 | 拆分大对象、分页处理 |
| **元空间不足** | 查看元空间使用情况 | 增大元空间或排查动态类生成 |
| **System.gc() 调用** | 代码搜索或 GC 日志 | 使用 `-XX:+DisableExplicitGC` 禁用 |
| **GC 参数不合理** | 分析 GC 日志 | 更换收集器（如 G1/ZGC）或调整参数 |

## 六、高频综合题

### 33. String s = new String("abc") 创建了几个对象？🔥🔥 ⭐⭐

- 如果常量池中没有 `"abc"`：**2 个** — 常量池中的字符串常量 + 堆中的 String 对象
- 如果常量池中已有 `"abc"`：**1 个** — 只有堆中的 String 对象

### 34. 解释执行和编译执行的区别？🔥 ⭐

| 方式 | 说明 | 速度 |
|------|------|------|
| **解释执行** | 解释器逐行将字节码翻译为机器码执行 | 较慢 |
| **编译执行（JIT）** | 将热点代码编译为本地机器码缓存到 CodeCache | 快 |

Java 两种方式**混合使用**：刚开始解释执行，JIT 发现热点代码后编译执行。

### 35. 空间分配担保？🔥 ⭐

Minor GC 前，JVM 检查老年代最大可用连续空间是否 > 新生代所有对象总空间：
- 如果大于，Minor GC 安全
- 如果小于，检查是否允许担保失败（`-XX:-HandlePromotionFailure`）
- 如果允许，检查老年代可用空间是否 > 历次晋升到老年代对象的平均大小
- 如果大于，尝试 Minor GC（有风险）
- 如果小于或不允许担保，改为 Full GC

### 36. JDK 默认使用什么垃圾收集器？🔥 ⭐

| JDK 版本 | 默认收集器 |
|---------|-----------|
| JDK 8 | Parallel Scavenge + Parallel Old |
| JDK 9~21 | G1 |
| JDK 22+ | G1（ZGC 可选） |

查看命令：`java -XX:+PrintCommandLineFlags -version`

---

### 37. 如何选择垃圾收集器？🔥 ⭐

| 场景 | 推荐收集器 |
|------|-----------|
| 小内存（<100MB）或无停顿要求 | Serial |
| 后台运算、吞吐量优先 | Parallel Scavenge + Parallel Old |
| 延迟敏感、堆 < 8GB | CMS（JDK 8）/ G1 |
| 大堆内存（>8GB）、停顿可控 | G1 |
| 超大堆（>100GB）、极低延迟 | ZGC |

---

### 38. JVM 调优的目标是什么？🔥 ⭐

1. **降低 GC 停顿时间**：选择合适的收集器，调整 Region 大小
2. **提高吞吐量**：减少 GC 频率，合理设置堆大小
3. **避免 Full GC**：排查内存泄漏，调整新生代/老年代比例
4. **减少内存占用**：逃逸分析、对象池、减少大对象

**调优原则**：
- 先监控分析，再调优
- 优先代码优化，再参数调优
- 每次只调一个参数，观察效果

### 39. Tomcat 的类加载机制？🔥 ⭐

Tomcat 在双亲委派模型基础上做了扩展：

```
Bootstrap ClassLoader
    ↑
Catalina ClassLoader（Tomcat 核心类库）
    ↑
Shared ClassLoader（多应用共享类库）
    ↑
WebApp ClassLoader（每个 Web 应用独立，优先加载自己的类 ← 破坏双亲委派）
    ↑
Jsp ClassLoader（JSP 热加载）
```

**隔离性**：不同 Web 应用的类互不干扰，同一类可以被不同 WebApp ClassLoader 分别加载。
