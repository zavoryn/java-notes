## 一、消息队列基础

### 1. 为什么要使用消息队列？🔥🔥🔥 ⭐

**三大核心作用：**

| 作用 | 说明 | 示例 |
|------|------|------|
| **解耦** | 系统间通过 MQ 通信，无需直接调用 | 订单系统下单后通知库存、物流、积分系统 |
| **异步** | 非核心逻辑异步处理，加快响应 | 用户注册后异步发送邮件、短信 |
| **削峰** | 高并发请求先入队列，消费者按能力处理 | 秒杀活动瞬间大量请求先入 MQ |

**引入 MQ 的代价：**
- 系统可用性降低（MQ 挂了影响业务）
- 系统复杂度增加（一致性问题、重复消费、消息丢失等）
- 运维成本增加

---

### 2. Kafka、RabbitMQ、RocketMQ 如何选型？🔥🔥🔥 ⭐⭐

| 特性 | RabbitMQ | RocketMQ | Kafka |
|------|----------|----------|-------|
| **单机吞吐量** | 万级 | 10 万级 | 10 万级+ |
| **时效性** | 微秒级 | 毫秒级 | 毫秒级 |
| **可用性** | 高（主从） | 非常高（分布式） | 非常高（分布式） |
| **消息可靠性** | 高 | 高（同步刷盘+同步双写） | 较高（副本机制） |
| **功能丰富度** | 极高（路由/交换机/延迟队列） | 高（事务/延迟/顺序/过滤） | 较低（核心 MQ 功能） |
| **支持 Topic 数** | 百万级 | 千级 | 百级（多了性能下降） |
| **消息回溯** | ❌ | ✅ 按时间 | ✅ 按 offset |
| **开发语言** | Erlang | Java | Scala/Java |
| **适用场景** | 中小规模、功能丰富需求 | 电商/金融在线业务 | 大数据/日志/流处理 |

**选型建议：**
- 日志收集/大数据 → **Kafka**
- 电商/金融/在线业务（事务消息、顺序消息）→ **RocketMQ**
- 功能丰富、中小规模 → **RabbitMQ**

---

### 3. 消息队列有哪些消息模型？🔥 ⭐

| 模型 | 说明 | 示例 |
|------|------|------|
| **点对点（P2P）** | 一条消息只能被一个消费者消费 | 任务分发 |
| **发布-订阅（Pub/Sub）** | 一条消息可被多个订阅者消费 | 事件通知、配置更新 |

**RocketMQ 的统一模型**：
- 同一消费者组内 = 点对点（消息负载均衡分配）
- 不同消费者组之间 = 发布订阅（每组都收到全量消息）

## 二、消息可靠性

### 4. 如何保证消息不丢失？🔥🔥🔥 ⭐⭐⭐

消息可能在三个阶段丢失，每个阶段都要考虑：

```
生产者 → MQ Broker → 消费者
  ↑           ↑          ↑
阶段1       阶段2      阶段3
```

**① 生产端（防发送失败）：**
- 使用同步发送 + 重试机制
- RocketMQ：同步发送等待 Broker ACK
- Kafka：设置 `acks=all`（等所有副本确认）

**② 存储端（防 Broker 丢失）：**
- RocketMQ：同步刷盘 + 主从同步复制
- Kafka：分区多副本（replica），`min.insync.replicas=2`
- RabbitMQ：消息持久化（`durable=true`）+ 镜像队列

**③ 消费端（防处理失败）：**
- **处理完业务逻辑后再确认（ACK）**
- 不要先 ACK 再处理，否则处理失败消息就丢了
- 处理失败让消息重新入队

---

### 5. 如何保证消息不被重复消费（幂等性）？🔥🔥🔥 ⭐⭐

**消息重复的原因**：生产者重试、消费者处理完但 ACK 失败导致重新投递。

**幂等性解决方案**：

| 方案 | 实现 |
|------|------|
| **全局唯一 ID + 去重表** | 消费前查去重表，已处理则跳过 |
| **数据库唯一约束** | 利用唯一索引防止重复插入 |
| **乐观锁（版本号）** | `UPDATE SET val=new, version=version+1 WHERE version=old` |
| **Redis SETNX** | 消费前 `SETNX messageId 1`，已存在则跳过 |
| **业务层面天然幂等** | 如 `UPDATE balance = balance - 100` 改为 `SET balance = 目标值` |

```java
// 典型的幂等消费伪代码
if (redis.setnx("msg:" + msgId, "1")) {
    processMessage(msg);  // 处理业务
    db.insert(msgId);     // 记录已处理
}
```

---

### 6. 如何保证消息的顺序性？🔥🔥 ⭐⭐

**Kafka**：
- 同一分区内消息天然有序
- 将需要顺序的消息用相同 key → 路由到同一 partition → 单线程消费

**RocketMQ**：
- **局部顺序**（常用）：相同业务 key 的消息发到同一 MessageQueue，Broker 对队列加锁保证单消费者顺序消费
- **全局顺序**：Topic 只设一个队列，性能低，一般不用

```java
// RocketMQ 顺序消息：根据订单ID选择队列
producer.send(msg, (mqs, msg, arg) -> {
    int index = orderId.hashCode() % mqs.size();
    return mqs.get(index);
}, orderId);
```

---

### 7. 消息积压怎么处理？🔥🔥 ⭐⭐

**原因**：生产速度 > 消费速度。

**紧急处理步骤**：
1. 修复消费者 bug，恢复消费能力
2. 如果 Queue 数量 > 消费者数量 → 扩容消费者
3. 如果 Queue 数量 ≤ 消费者数量 → 新建临时 Topic（10 倍 Queue），用消费者转发消息到临时 Topic，再用 10 倍消费者消费
4. 如果无法扩容 → 系统降级，关闭非核心业务减少消息生产
5. 消费完积压后恢复正常架构

## 三、Kafka 专题

### 8. Kafka 为什么这么快？🔥🔥🔥 ⭐⭐⭐

| 技术 | 说明 |
|------|------|
| **顺序写磁盘** | 追加写入日志文件，避免随机写的磁头寻道 |
| **零拷贝** | `sendfile()` 系统调用，数据从磁盘直接到网卡，不经用户态 |
| **PageCache** | 利用 OS 页缓存，读写接近内存速度 |
| **批量处理** | 生产者批量发送、消费者批量拉取，减少网络 IO |
| **压缩** | 支持gzip/snappy/lz4/zstd，减少网络传输量 |
| **分区并行** | Topic 分多个 Partition，分布在多台机器并行处理 |

**零拷贝对比**：
```
传统：磁盘 → 内核缓冲区 → 用户缓冲区 → Socket缓冲区 → 网卡 （4次拷贝）
零拷贝：磁盘 → 内核缓冲区 → 网卡 （2次拷贝）
```

---

### 9. Kafka 的分区和副本机制？🔥🔥 ⭐⭐

**分区（Partition）**：
- Topic 被分为多个 Partition，分布在不同 Broker 上
- 每个 Partition 是一个有序的、不可变的追加日志
- 分区数决定了并行度和最大消费者数

**副本（Replica）**：
- 每个 Partition 有多个副本（副本因子，如 3）
- 一个 Leader + 多个 Follower
- 所有读写都走 Leader，Follower 被动同步
- Leader 宕机时，从 ISR（In-Sync Replicas）中选新 Leader

**消费者组（Consumer Group）**：
- 同一分区同一时刻只能被组内一个消费者消费
- 分区数 = 消费者组内最大并行消费者数

---

### 10. Kafka 的 ACK 机制？🔥🔥 ⭐⭐

| acks 值 | 行为 | 可靠性 | 性能 |
|---------|------|--------|------|
| **0** | 发出去就不管 | ❌ 最低 | ✅ 最好 |
| **1** | 等 Leader 写入确认 | ⚠️ Leader 挂了可能丢 | 折中 |
| **-1/all** | 等所有 ISR 副本确认 | ✅ 最高 | ❌ 最慢 |

> **推荐生产配置**：`acks=all` + `min.insync.replicas=2` + 副本因子 3

---

### 11. Kafka 是 Pull 还是 Push？🔥 ⭐

**Kafka 采用 Pull 模式**。

- **Push 的问题**：Broker 控制推送速率，消费者处理不过来时容易堆积/拒绝服务
- **Pull 的优势**：消费者按自己能力拉取，可控制速度，可回溯消费
- **Pull 的劣势**：无消息时空轮询 → Kafka 用长轮询（`timeout` 参数）解决

## 四、RocketMQ 专题

### 12. RocketMQ 的架构是怎样的？🔥🔥 ⭐⭐

```
NameServer 集群（路由注册中心，无状态）
    ↕ 心跳
Broker（Master/Slave，消息存储）
    ↕ 发送/拉取
Producer（生产者）    Consumer（消费者）
```

| 组件 | 职责 |
|------|------|
| **NameServer** | 路由中心，维护 Topic→Broker 映射，无状态，集群部署 |
| **Broker** | 消息存储，分 Master（读写）和 Slave（只读），定时向 NameServer 心跳 |
| **Producer** | 发送消息，从 NameServer 获取路由后直接发 Broker |
| **Consumer** | 消费消息，从 NameServer 获取路由后从 Broker 拉取 |

---

### 13. RocketMQ 为什么不用 Zookeeper？🔥 ⭐⭐

1. **可用性优先**：ZK 满足 CP（选举时不可用），NameServer 无状态可随时扩展，满足 AP
2. **轻量级**：NameServer 实现极简，无需 ZK 的 ZAB 协议和事务日志
3. **弱依赖**：Producer/Consumer 本地缓存路由信息，NameServer 短暂不可用不影响运行

---

### 14. Broker 的存储机制？🔥 ⭐⭐

RocketMQ 采用混合存储结构，所有 Topic 共用一个 CommitLog：

| 文件 | 说明 |
|------|------|
| **CommitLog** | 消息主体文件，所有消息顺序写入，每文件默认 1G |
| **ConsumeQueue** | 消费队列索引，按 Topic/QueueId 组织，存储 offset+size+tagHash |
| **IndexFile** | 按 key 或时间查询的哈希索引 |

```
写入流程：消息 → 顺序写入 CommitLog → 异步构建 ConsumeQueue + IndexFile
读取流程：Consumer 读 ConsumeQueue 获取 offset → 从 CommitLog 读取完整消息
```

**读写优化**：PageCache + 顺序读写 + mmap 零拷贝

---

### 15. RocketMQ 的刷盘策略？🔥🔥 ⭐⭐

| 策略 | 说明 | 性能 | 可靠性 |
|------|------|------|--------|
| **同步刷盘** | 消息写入内存后等刷盘完成再返回 ACK | 较低 | ✅ 最高 |
| **异步刷盘** | 消息写入内存即返回，后台线程异步刷盘 | ✅ 高 | ⚠️ 宕机可能丢少量数据 |

---

### 16. 什么是事务消息？半消息？🔥🔥 ⭐⭐⭐

**场景**：下单扣库存 + 发消息通知下游，需要保证原子性。

**RocketMQ 事务消息流程**：
1. Producer 发送**半消息**（对 Consumer 不可见）到 Broker
2. Broker 返回确认，Producer 执行本地事务
3. 本地事务成功 → 发送 **Commit**（消息变为可消费）
4. 本地事务失败 → 发送 **Rollback**（消息被删除）
5. 如果 Broker 长时间未收到二次确认 → **事务回查** Producer 本地事务状态

> **RocketMQ 的分布式事务是最终一致性**，不是强一致。

---

### 17. RocketMQ 的延时消息？🔥 ⭐

**原理**：Broker 收到延时消息后先存到 `SCHEDULE_TOPIC_XXXX` 临时 Topic，定时任务轮询检查到期时间，到期后转发到目标 Topic。

**支持的延时级别**：
```
1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h
```

**典型应用**：订单超时自动取消（下单时发延时 30 分钟的消息，到期检查订单状态）

---

### 18. RocketMQ 的死信队列？🔥 ⭐

消息消费失败并达到最大重试次数后，进入**死信队列（DLQ）**：
- 死信队列 Topic：`%DLQ%{ConsumerGroup}`
- 用于存储无法正常消费的消息
- 可人工排查后重发或导出保存

---

### 19. RocketMQ 的负载均衡？🔥 ⭐

**Producer 端**：
- 从 Topic 的 MessageQueue 列表中轮询选择
- 开启 `sendLatencyFaultEnable` 后，会规避延迟高的 Broker

**Consumer 端**：
- 集群模式下，Queue 平均分配给同组的 Consumer
- 分配算法默认为平均分配（类似分页）
- 每 20s 执行一次 Rebalance

## 五、RabbitMQ 专题

### 20. RabbitMQ 的核心架构？🔥 ⭐

```
Producer → Exchange（交换机）→ Queue → Consumer
              ↓
         Binding（绑定规则）
```

| 组件 | 说明 |
|------|------|
| **Exchange** | 接收消息，按路由规则分发到 Queue |
| **Queue** | 存储消息，消费者从 Queue 取消息 |
| **Binding** | Exchange 和 Queue 之间的绑定关系 |
| **Routing Key** | 路由键，Exchange 根据 Key 决定消息去向 |

---

### 21. RabbitMQ 的交换机类型？🔥 ⭐

| 类型 | 路由规则 | 场景 |
|------|---------|------|
| **Direct** | 精确匹配 Routing Key | 点对点、路由 |
| **Fanout** | 广播到所有绑定的 Queue | 群发通知 |
| **Topic** | 通配符匹配（`*` 匹配一个词，`#` 匹配零或多个） | 发布订阅、日志路由 |
| **Headers** | 根据消息头匹配 | 复杂路由 |

---

### 22. RabbitMQ 如何保证高可用？🔥 ⭐

| 模式 | 说明 |
|------|------|
| **单机模式** | 开发测试用 |
| **普通集群模式** | Queue 只在一个节点，其他节点可转发请求。Queue 所在节点挂了消息丢失 |
| **镜像集群模式** | Queue 数据同步到多个节点，任一节点宕机不影响。**但性能开销大、扩展性差** |

---

### 23. RabbitMQ 的消息确认机制？🔥 ⭐

- **生产者确认**：`confirm` 模式，消息到达 Broker 后回调通知
- **消费者确认**：处理完消息后发送 ACK，Broker 才删除消息；不确认则重新投递

## 六、通用设计题

### 24. 如何设计一个消息队列？🔥 ⭐⭐⭐

**核心组件**：Producer → Broker → Consumer

**设计要点**：

| 要点 | 方案 |
|------|------|
| **RPC 通信** | 参考Dubbo：服务发现 + 高效序列化 |
| **消息持久化** | 顺序写文件（Kafka/RocketMQ 方案） |
| **高可用** | 多副本 + Leader 选举（Kafka 方案） |
| **可扩展** | Topic → Partition/Queue 分片，加机器扩容 |
| **消息可靠性** | 生产端重试 + 存储端刷盘 + 消费端确认 |
| **消息幂等** | 全局唯一 ID + 去重表 |
| **事务消息** | 半消息 + 本地事务 + 二次确认/回查 |

### 25. 多线程异步和 MQ 的区别？🔥 ⭐

| 维度 | 多线程 | MQ |
|------|--------|-----|
| CPU 消耗 | 占本机 CPU | 不占本机 CPU |
| 解耦 | 线程池内耦合 | 完全解耦 |
| 削峰能力 | 线程数有限，可能触发拒绝策略 | 消息堆积在 Broker |
| 复杂度 | 简单 | 引入中间件，增加运维 |
| 适用规模 | 小项目 | 大型互联网项目 |
