# 包装类（Wrapper Class）

> [!summary] 我的速记
> **核心考点 = 缓存池**：`Integer.valueOf(n)` 在 -128~127 范围内走缓存（复用对象），超出则 new。所以 `Integer i=127; Integer j=127; i==j` → true，但 128 就 false。
> **自动装箱** = 编译器插入 `valueOf()`；**自动拆箱** = 编译器插入 `intValue()`。
> **三目 NPE 陷阱**：`Integer n = flag ? 1 : null` → 自动拆箱 `null.intValue()` → NPE！

## 一、8 种基本类型对应的包装类

| 基本类型    | 包装类       | 内存占用          | 缓存范围         |
| ------- | --------- | ------------- | ------------ |
| byte    | Byte      | 1 字节          | -128 ~ 127   |
| short   | Short     | 2 字节          | -128 ~ 127   |
| int     | Integer   | 4 字节          | -128 ~ 127   |
| long    | Long      | 8 字节          | -128 ~ 127   |
| float   | Float     | 4 字节          | 无缓存          |
| double  | Double    | 8 字节          | 无缓存          |
| char    | Character | 2 字节          | 0 ~ 127      |
| boolean | Boolean   | 1 字节 ~ JVM 相关 | TRUE / FALSE |

> [!important] Float 和 Double 没有缓存池机制，每次都是新对象。

## 二、自动装箱与自动拆箱

### 2.1 概念

- **自动装箱（Auto-boxing）**：编译器自动将基本类型转换为对应的包装类对象
- **自动拆箱（Auto-unboxing）**：编译器自动将包装类对象转换为对应的基本类型

### 2.2 底层原理（反编译验证）

```java
// 源码
Integer i = 100;          // 自动装箱
int n = i;                // 自动拆箱

// 编译后等价于
Integer i = Integer.valueOf(100);   // 装箱 → valueOf()
int n = i.intValue();               // 拆箱 → xxxValue()
```

> [!abstract] 装箱走 `valueOf()`，拆箱走 `xxxValue()`。所以装箱对象受缓存池影响！

### 2.3 触发时机

- 赋值给包装类变量时 → 自动装箱
- 包装类对象和基本类型做运算时 → 自动拆箱
- 方法参数传递时（重载决议除外）
- **集合类**（如 `List<Integer>`）add 时 → 自动装箱

## 三、缓存池机制

### 3.1 Integer 缓存（-128 ~ 127）

```java
// Integer.valueOf() 源码简化版
public static Integer valueOf(int i) {
    // IntegerCache.low = -128, IntegerCache.high = 127
    if (i >= -128 && i <= 127)
        return IntegerCache.cache[i + 128];
    return new Integer(i);
}
```

> [!tip] 可调参数：`-XX:AutoBoxCacheMax=<size>` 可调大缓存上限，但不能调小到 127 以下。

### 3.2 各包装类缓存范围

| 包装类       | 缓存范围                    | 备注                  |
| --------- | ----------------------- | ------------------- |
| Byte      | 整个范围（-128 ~ 127）        | 所有值都缓存              |
| Short     | -128 ~ 127              | 不可调                 |
| Integer   | -128 ~ 127（可用 JVM 参数调大） | [[包装类#三、缓存池机制\|见上]] |
| Long      | -128 ~ 127              | 不可调                 |
| Character | 0 ~ 127（ASCII 范围）       | 不可调                 |
| Boolean   | TRUE / FALSE（只有两个实例）    | 全量缓存                |
| Float     | 无缓存                     | —                   |
| Double    | 无缓存                     | —                   |

## 四、== 与 equals 对比

```java
// == 对比（引用类型比地址，基本类型比值）
Integer a1 = 127;   // 装箱 → valueOf(127) → 缓存池取
Integer a2 = 127;   // 装箱 → valueOf(127) → 缓存池取（同一个！）
System.out.println(a1 == a2);        // true（缓存池中同一对象）

Integer b1 = 128;   // 超缓存范围 → new Integer(128)
Integer b2 = 128;   // 超缓存范围 → new Integer(128)（不同对象！）
System.out.println(b1 == b2);        // false（堆中两个不同对象）

// equals 对比
System.out.println(a1.equals(a2));   // true（比较内部 int 值）
System.out.println(b1.equals(b2));   // true（比较内部 int 值）
```

> [!warning] `==` 对包装类比的是**对象地址**，不是值。一定要用 `equals()` 或先拆箱再比。

## 五、为什么要有包装类

1. **泛型不支持基本类型**：`List<int>` 编译报错，必须写 `List<Integer>`
2. **提供工具方法**：`Integer.parseInt()`、`Integer.toBinaryString()` 等
3. **null 值语义**：int 默认是 0，Integer 可以是 null（表示"未赋值"）
4. **反射 / 序列化** 需要对象类型

## 六、valueOf() vs new Integer()

| 比较维度 | `Integer.valueOf(int)` | `new Integer(int)` |
|---------|----------------------|-------------------|
| 缓存 | 使用缓存池（-128~127） | 每次 new 新对象 |
| 是否推荐 | 推荐 | JDK9 起标记 `@Deprecated` |
| 内存 | 省内存（复用） | 浪费（每次都创建） |

```java
Integer v1 = Integer.valueOf(100);   // 从缓存池取，推荐
Integer v2 = new Integer(100);       // 强制创建新对象，已废弃
```

## 七、经典面试题

### 题目 1：127 判等

```java
Integer i1 = 127;
Integer i2 = 127;
System.out.println(i1 == i2);  // ?
```
**答**：`true`。127 在缓存池范围内，`valueOf()` 返回同一个对象。

### 题目 2：128 判等

```java
Integer i1 = 128;
Integer i2 = 128;
System.out.println(i1 == i2);  // ?
```
**答**：`false`。128 超出缓存范围，每次 new 新对象，地址不同。

### 题目 3：混用判等

```java
Integer a = 128;
int b = 128;
System.out.println(a == b);  // ?
```
**答**：`true`。包装类和基本类型用 `==` 比较时，包装类**自动拆箱**为 int，比的是值。

### 题目 4：三目运算符陷阱

```java
Integer a = null;
Integer b = (true ? a : 0);  // NPE!
```
**答**：三目表达式中，`a` 和 `0` 类型不一致时会自动拆箱，`a` 为 null 时拆箱抛 NPE。

> [!danger] 包装类参与运算、三目表达式时会自动拆箱，值为 null 抛 NPE！

## 原理深挖

### Integer 缓存池是怎么实现的？

```java
// Integer.valueOf() 源码
public static Integer valueOf(int i) {
    if (i >= -128 && i <= 127)
        return IntegerCache.cache[i + 128];
    return new Integer(i);
}
// IntegerCache 是 Integer 内部的静态内部类，启动时预创建 -128~127 共 256 个对象
```

Byte/Short/Long 也有缓存（-128~127），Character 缓存 0~127，Boolean 直接缓存 TRUE/FALSE 两个常量。只有 Float/Double 没有缓存——浮点数范围太大、精度问题，缓存意义不大。

### 三目运算符的 NPE 陷阱

```java
Integer i = true ? 1 : null;  // NPE!
// 三目运算符要求两边类型一致 → null 自动拆箱为 null.intValue() → NPE
// 正确写法：
Integer i = true ? Integer.valueOf(1) : null;
```

## 面试回答框架

> [!info] 问：说说包装类和自动装箱拆箱？

**① 一句话定位：** 8 种基本类型各对应一个包装类。包装类解决了三个问题：泛型必须用引用类型、提供工具方法（parseInt 等）、支持 null 语义。

**② 核心展开：** 自动装箱底层是 `valueOf()`，自动拆箱是 `xxxValue()`——是编译器插入的代码，不是 JVM 层面的特性。Integer.valueOf() 缓存 -128~127 的对象，所以 `Integer.valueOf(127) == Integer.valueOf(127)` 为 true，128 就 false。**判断内容必须用 equals。**

**③ 踩坑经验：** 三目运算符两边类型不一致时自动拆箱会 NPE；循环中大量自动装箱会频繁创建对象（用基本类型+Stream 避免）；数据库查出的 null 赋值给基本类型直接 NPE。

> [!question] 追问：`new Integer(127) == Integer.valueOf(127)` 结果？

> false。new 一定在堆上创建新对象，valueOf 从缓存池拿——地址不同。JDK9 后 new Integer() 已标记为 deprecated。
