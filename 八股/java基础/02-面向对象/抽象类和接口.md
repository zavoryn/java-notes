# 抽象类与接口

## 一、核心对比表格

| 对比维度      | 抽象类（abstract class）                 | 接口（interface）                                           |
| --------- | ----------------------------------- | ------------------------------------------------------- |
| **关键字**   | `abstract class`                    | `interface`                                             |
| **继承方式**  | 单继承（`extends`）                      | 多实现（`implements`）                                       |
| **构造方法**  | 有                                   | 无                                                       |
| **成员变量**  | 普通实例变量                              | 只能 `public static final` 常量                             |
| **方法**    | 可以有抽象方法和具体方法                        | JDK7: 只能抽象方法；JDK8+: default/static 方法；JDK9+: private 方法 |
| **访问修饰符** | 有（public/protected/private/default） | JDK7: 方法默认 `public abstract`，变量默认 `public static final` |
| **设计关系**  | is-a（是什么）                           | can-do（能做什么）                                            |
| **设计目的**  | 提取公共行为，作为模板                         | 定义能力契约                                                  |
| **初始化**   | 通过子类构造方法调用 `super()`                | 和实现类一起初始化                                               |

## 二、抽象类（Abstract Class）

### 定义

用 `abstract` 修饰的类，不能直接**实例化**，必须被子类继承并实现所有抽象方法后才能使用。

### 关键规则

> [!important]
> - 有抽象方法的类**必须**是抽象类，但抽象类**可以没有**抽象方法
> - `abstract` 方法不能是 `private`（子类不可见无法实现）、`static`（类方法无法多态）、`final`（禁止重写）
> - 有构造方法，供子类通过 `super()` 调用
> - 单继承

### 使用场景

- 多个相关类有**公共字段和公共行为**，需要提炼模板
- 希望限制实例化，只能通过子类间接创建
- 需要构造方法来初始化公共字段

### 示例

```java
public abstract class GameCharacter {
    protected String name;
    protected int hp;

    public GameCharacter(String name) {
        this.name = name;
        this.hp = 100;
    }

    // 具体方法：模板逻辑
    public void takeDamage(int damage) {
        this.hp -= damage;
        if (hp <= 0) {
            die();
        }
    }

    // 抽象方法：留给子类实现
    public abstract void attack();
    public abstract void die();
}

public class Warrior extends GameCharacter {
    public Warrior(String name) { super(name); }

    @Override
    public void attack() { System.out.println(name + " 挥剑攻击"); }

    @Override
    public void die() { System.out.println(name + " 倒下"); }
}
```

## 三、接口（Interface）

### 定义

接口是**能力的契约**，定义一组方法签名，实现类必须遵守。

### JDK 版本演进（面试重点）

| JDK 版本 | 接口能力 |
|---|---|
| **JDK 7 及以前** | 只能有：抽象方法（默认 `public abstract`）、常量（`public static final`） |
| **JDK 8** | 新增 **`default` 方法**（带默认实现，解决接口演化问题）、**`static` 方法**（工具方法） |
| **JDK 9** | 新增 **`private` 方法**（供 default 方法内部复用，避免代码重复） |

### JDK8 接口演进示例

```java
public interface Animal {
    // 常量
    String KINGDOM = "Animalia";  // 默认 public static final

    // 抽象方法（默认 public abstract）
    void eat();

    // default 方法：有默认实现，实现类可重写也可不重写
    default void sleep() {
        System.out.println("Zzz...");
    }

    // static 方法：属于接口本身，提供工具方法
    static void breathe() {
        System.out.println("Breathing...");
    }
}
```

### JDK9 接口新增：private 方法

```java
public interface Calculator {
    default int add(int a, int b) {
        log("加法");        // 复用 private 方法
        return a + b;
    }
    default int multiply(int a, int b) {
        log("乘法");
        return a * b;
    }

    // JDK9: private 方法，供 default 方法内部复用
    private void log(String operation) {
        System.out.println("执行: " + operation);
    }
}
```

### 使用场景

- 定义**不相关的类之间**的通用能力（如 `Comparable`、`Runnable`、`Serializable`）
- 实现**多继承能力**（类可以同时实现多个接口）
- 面向接口编程，松散耦合

## 四、经典面试问题

### Q1: 接口和抽象类什么时候选谁？

> [!tip] 选择原则
>
> - 需要**多继承**能力时 → 只能选**接口**
> - 类之间是 **is-a 关系**，有公共实现代码要共享 → 选**抽象类**
> - 类之间是 **can-do 关系**，不相关的类共享能力 → 选**接口**
> - 既需要公共实现又需要多态契约 → **抽象类 + 接口组合**使用

### Q2: 为什么要有 default 方法？

> 解决接口**向后兼容**问题。JDK8 要给 Collection 加 `stream()` 方法，如果改成抽象方法，所有实现类都要同步改——这是不可能的。`default` 方法允许在接口中增加有默认实现的新方法，老实现类无需改动。

### Q3: 一个类实现两个接口，两个接口有同名的 default 方法怎么办？

> 实现类**必须重写**这个冲突的方法，通过 `接口名.super.方法名()` 选择调用某个接口的实现：

```java
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello();  // 显式选择调用 A 的实现
    }
}
```

## 原理深挖

### 为什么 `default` 方法的设计是革命性的？

JDK8 要给 Collection 接口加 `stream()` 方法。如果加抽象方法，所有实现类（包括三方的）全部编译报错——Java 不可能要求全世界开发者同步改代码。`default` 方法带默认实现，老实现类无需改动即可编译通过。背后的设计原则是**向后兼容优于整洁性**。

### 接口为什么不能有构造方法？

因为接口不是类——JVM 规范里接口的字节码标识和类不同。接口没有实例字段（只有 `static final` 常量），构造方法的唯一作用是初始化实例字段，所以接口根本不需要它。深层原因是：接口代表的是「能力」而非「实体」，能力不需要构造，实体才需要。

### 多继承 default 方法冲突怎么解决？

```java
interface A { default void hello() { sout("A"); } }
interface B { default void hello() { sout("B"); } }
class C implements A, B {
    // 必须重写，手动选择
    @Override public void hello() { A.super.hello(); }
}
```

规则是「**类优先于接口**，**子接口优先于父接口**，冲突时必须手动指定」。

## 面试回答框架

> [!info] 问：抽象类和接口的区别？什么时候用哪个？

**① 一句话定位：** 抽象类是「是什么」（is-a），接口是「能干什么」（can-do）。抽象类提供模板代码，接口定义能力契约。

**② 核心对比展开：**

语法层面——抽象类有构造方法（给子类 super 调），单继承；接口没有构造方法，多实现。JDK8 给接口加了 default 和 static 方法，JDK9 加了 private 方法——这三次演进都是为了解决接口演化时的向后兼容问题。

继承层面——抽象类单继承限制了灵活性但适合紧密耦合的继承体系；接口多实现让一个类可以同时拥有多个能力，比如一个类既可以是 `Comparable` 又可以是 `Serializable`。

**③ 选型经验：**

如果需要复用代码（公共字段、模板方法）→ 抽象类。如果只定义契约、允许不相关的类共享能力 → 接口。大量场景是两者组合——`AbstractList` 实现 `List`，抽象类写通用逻辑（迭代器），接口定义契约。工作中我优先用接口，因为组合优于继承。

> [!question] 追问：default 方法的设计初衷是什么？

> 解决接口演化问题。JDK8 要在 Collection 加 stream 方法，如果加抽象方法，所有实现类都得改代码——这是不可能的。default 方法让接口可以加新功能而不破坏已有实现，本质是「向后兼容」的设计妥协。
