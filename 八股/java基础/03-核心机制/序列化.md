# Java 序列化

> [!summary] 我的速记
> **序列化** = 对象 → 字节流；**反序列化** = 字节流 → 对象。
> `Serializable` 是标记接口；`serialVersionUID` 做版本控制；`transient` 跳过字段。
> 反序列化**不走构造方法**，有安全风险（反序列化漏洞）。

## 一、什么是序列化 / 反序列化

### 核心定义

- **序列化（Serialization）**：将 Java 对象转换为**字节序列**的过程
- **反序列化（Deserialization）**：将字节序列**恢复为 Java 对象**的过程

```java
// 序列化：对象 → 字节流（可存入文件 / 数据库 / 网络传输）
ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("obj.dat"));
oos.writeObject(obj);
oos.close();

// 反序列化：字节流 → 对象
ObjectInputStream ois = new ObjectInputStream(new FileInputStream("obj.dat"));
Object obj = ois.readObject();
ois.close();
```

### 应用场景

| 场景 | 说明 |
|---|---|
| **网络传输** | RPC 调用、RMI 远程方法调用 |
| **持久化存储** | 将对象保存到文件或数据库 |
| **深拷贝** | 通过序列化实现对象的深拷贝 |
| **分布式缓存** | 对象存入 Redis / Memcached |
| **消息队列** | 对象作为消息体传输 |

## 二、Serializable 接口

### 定义

`java.io.Serializable` 是一个**标记接口**（Marker Interface），没有任何方法或字段，只起到**标记作用**——表明这个类的对象是可以被序列化的。

```java
// Serializable 接口源码
public interface Serializable {
    // 空接口！只是一个标记
}
```

### 使用示例

```java
public class User implements Serializable {
    // 强烈建议显式声明 serialVersionUID
    private static final long serialVersionUID = 1L;

    private String name;
    private int age;
    private transient String password;  // transient 字段不参与序列化

    // 构造方法、getter/setter 省略...
}
```

> [!tip] 如果没有实现 Serializable 接口，调用 `writeObject()` 时会抛出 `NotSerializableException`。

## 三、serialVersionUID（重要！）

### 定义

`serialVersionUID` 是序列化版本号，用于在反序列化时**验证发送方和接收方的类是否兼容**。

### 工作原理

1. 序列化时：`serialVersionUID` 写入字节流
2. 反序列化时：将字节流中的 `serialVersionUID` 与本地类的 `serialVersionUID` 比较
3. 如果**不相等** → 抛出 `InvalidClassException`，反序列化失败

### 为什么必须显式声明

> [!warning] 如果不显式声明，编译器会根据类的结构（字段、方法等）**自动生成** `serialVersionUID`。一旦类结构有任何变化（加字段、改方法签名），自动生成的值就会不同，导致之前序列化的数据**全部无法反序列化**。

显式声明 `serialVersionUID` 后，即使类结构有变化，只要版本号不变，反序列化仍然可以兼容（但可能丢失新增字段的数据）。

```java
// 强烈推荐的写法
private static final long serialVersionUID = 1L;  // 固定值

// IDEA 可以自动生成基于类结构哈希的 serialVersionUID
// private static final long serialVersionUID = -6849794470754667710L;
```

## 四、transient 关键字

### 定义

用 `transient` 修饰的字段**不参与序列化**。

### 使用场景

| 场景 | 示例 |
|---|---|
| 敏感信息 | 密码、银行卡号、身份证号 |
| 不可序列化的引用 | 包含 `InputStream`、`Connection` 等不可序列化对象的字段 |
| 冗余/可推导数据 | 通过其他字段计算出来的值（如年龄可从生日推导） |
| 性能优化 | 不需要传输的大数据字段 |

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private String name;
    private transient String password;       // 密码不序列化
    private transient Connection connection; // 连接对象不序列化

    // 反序列化后 password 为 null（引用类型的默认值）
}
```

> [!tip] transient 修饰的字段在反序列化后为其**类型的默认值**（对象 → null，int → 0，boolean → false）。

### 自定义序列化：writeObject / readObject

如果想对 transient 字段做特殊处理，可以实现私有方法：

```java
private void writeObject(ObjectOutputStream out) throws IOException {
    out.defaultWriteObject();          // 先序列化非 transient 字段
    out.writeObject(encrypt(password)); // 手动处理敏感字段
}

private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException {
    in.defaultReadObject();            // 先反序列化非 transient 字段
    this.password = decrypt((String) in.readObject()); // 手动恢复
}
```

## 五、Externalizable 接口

### 与 Serializable 的区别

| 对比 | Serializable | Externalizable |
|---|---|---|
| **自动程度** | 自动序列化 | 完全手动控制 |
| **必须实现的方法** | 无（标记接口） | `writeExternal()` 和 `readExternal()` |
| **性能** | 较慢（全部字段自动处理） | 较快（只处理指定字段） |
| **构造方法要求** | 不需要 public 无参构造 | **必须有 public 无参构造** |
| **transient 的影响** | 有影响 | 无影响（因为完全手动控制） |
| **默认序列化** | 全部非 transient 字段 | 不序列化任何东西（除非手动指定） |

```java
public class User implements Externalizable {
    private static final long serialVersionUID = 1L;

    private String name;
    private int age;

    // 必须有无参 public 构造方法
    public User() {}

    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeObject(name);  // 手动决定序列化什么
        out.writeInt(age);
    }

    @Override
    public void readExternal(ObjectInput in) throws IOException, ClassNotFoundException {
        name = (String) in.readObject();  // 手动恢复
        age = in.readInt();
    }
}
```

## 六、序列化的问题与风险

### 安全问题

> [!warning] Java 反序列化漏洞
> 反序列化时会调用对象的 `readObject()`，恶意构造的字节流可以**在反序列化过程中执行任意代码**。这是 Java 安全中最臭名昭著的漏洞类型之一（如 Spring Framework RCE、Fastjson 反序列化漏洞等）。

**防范措施**：
- 使用白名单限制可反序列化的类
- Jackson、Fastjson 关闭默认类型推断
- 使用安全的序列化框架

### 版本兼容问题

- 添加字段：旧数据反序列化时新字段为默认值
- 删除字段：旧数据中的字段值被丢弃
- `serialVersionUID` 是版本兼容的关键

### 性能问题

- Java 原生序列化性能较差（空间大、速度慢）
- 序列化结果不可读（二进制格式）

## 七、常见序列化框架对比

| 框架 | 格式 | 特点 |
|---|---|---|
| **Java 原生 Serializable** | 二进制 | JDK 自带，性能差，有安全风险 |
| **JSON（Jackson / Gson / Fastjson）** | 文本 | 可读性好，跨语言，Web 开发主流 |
| **Protobuf / gRPC** | 二进制 | Google 出品，体积小速度快，跨语言 |
| **Hessian** | 二进制 | 比原生序列化快，跨语言（Dubbo 早期用） |
| **Kryo** | 二进制 | 极快，常用于大数据（Spark） |
| **MsgPack** | 二进制 | 类似 JSON 但更紧凑 |
| **XML** | 文本 | 可读性好，体积大，已较少使用 |

## 面试话术

> Java 序列化是将对象转换为字节序列以便存储或传输，反序列化则是其逆过程。Java 原生序列化通过 `Serializable` 接口实现——它是一个标记接口，表示该类可以被序列化。
>
> 序列化中有几个关键点：**serialVersionUID** 是版本控制的核心，如果不显式声明，类结构变化后会导致之前的数据无法反序列化，所以强烈建议显式定义。**transient** 关键字可以让指定字段不参与序列化，反序列化后为默认值，常用于保护敏感信息。
>
> Java 原生序列化有几个问题：一是**性能差**，结果体积大速度慢；二是存在**安全漏洞**——反序列化过程中可能执行恶意代码；三是跨语言支持差。所以在实际项目中，Web 开发一般用 **JSON**（Jackson / Gson），微服务 RPC 一般用 **Protobuf**，大数据场景用 **Kryo**。
>
> `Externalizable` 是 `Serializable` 的子接口，提供完全手动的序列化控制，但需要实现两个方法和提供无参构造方法，使用较少。
