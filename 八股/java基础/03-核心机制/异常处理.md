# Java 异常处理

> [!summary] 我的速记
> **异常谱系：** `Throwable` → `Error`（不管）+ `Exception`（管）。Exception = checked（编译期必须处理）+ unchecked/RuntimeException（编译不管）。
> **finally 两条坑：** ① return 会覆盖；② 一定执行，除了 `System.exit(0)` 或 JVM 崩。
> **try-with-resources：** 实现 `AutoCloseable` 就能自动关流，不用写 finally。

## 一、Throwable 体系

> [!tip] 先看全局图谱

```
Throwable
├── Error                     ← 严重错误，程序无法处理
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   └── NoClassDefFoundError
└── Exception                 ← 程序可以处理
    ├── checked exception     ← 编译时必须处理（try-catch 或 throws）
    │   ├── IOException
    │   ├── SQLException
    │   ├── ClassNotFoundException
    │   └── InterruptedException
    └── RuntimeException      ← unchecked，编译期不检查
        ├── NullPointerException
        ├── IndexOutOfBoundsException
        ├── ClassCastException
        ├── IllegalArgumentException
        ├── ArithmeticException  (如 / 0)
        └── NumberFormatException
```

### Error vs Exception

| 对比 | Error | Exception |
|---|---|---|
| **严重性** | 严重，程序通常无法恢复 | 可以捕获和处理 |
| **示例** | `OutOfMemoryError`、`StackOverflowError` | `IOException`、`NullPointerException` |
| **处理方式** | 不应捕获，让程序终止 | try-catch 或 throws |
| **是否 checked** | unchecked | 分 checked 和 unchecked |

### Checked vs Unchecked

| 对比 | Checked Exception | Unchecked Exception |
|---|---|---|
| **编译期检查** | 必须处理（捕获或声明抛出） | 可选处理 |
| **父类** | `Exception`（非 RuntimeException） | `RuntimeException` 及其子类 |
| **设计意图** | 可预见的异常情况（文件不存在、网络中断） | 编程错误（空指针、数组越界） |
| **示例** | `IOException`、`SQLException` | `NullPointerException`、`IndexOutOfBoundsException` |

## 二、try-catch-finally 执行顺序（面试重点）

### 基本执行流程

```java
try {
    // 1. 执行 try 块
    return 1;
} catch (Exception e) {
    // 2. 如果 try 抛异常，匹配 catch 后执行
    return 2;
} finally {
    // 3. finally 一定执行（除非特殊例外）
    System.out.println("finally 执行");
}
```

### finally 一定会执行吗？

> [!important] 面试高频
> **绝大多数情况下 finally 都会执行**，但以下情况不会：

| 情况 | 说明 |
|---|---|
| `System.exit(0)` | JVM 直接退出，没有机会执行 finally |
| 守护线程退出 | 所有非守护线程结束后 JVM 退出，守护线程的 finally 可能不执行 |
| 无限循环 / 死锁 | try 中进入死循环或死锁，永远走不到 finally |
| JVM 崩溃 | 如 `OutOfMemoryError` 导致进程被 kill |
| 断电 / 系统崩溃 | 物理层面 |

### 经典面试题：finally 中的 return

> [!warning] 这是最经典的陷阱题

```java
// 问题：以下代码输出什么？
public static int test() {
    try {
        return 1;
    } finally {
        return 2;
    }
}
// 答案：返回 2
// finally 中的 return 会覆盖 try 中的 return
```

```java
// 进阶题：输出什么？
public static int test() {
    int x = 1;
    try {
        return x;          // 此时 x=1，返回值已暂存
    } finally {
        x = 2;             // x 变为 2，但返回值已经暂存为 1
        // 注意：如果没有 return，不会改变已暂存的返回值
    }
}
// 答案：返回 1（基本类型值已被暂存）

public static User test() {
    User user = new User("张三");
    try {
        return user;       // 返回的是引用
    } finally {
        user.setName("李四"); // 修改了引用指向的对象
    }
}
// 答案：返回的 User 对象 name 是 "李四"（引用类型，对象被修改了）
```

### finally 吞掉异常

```java
try {
    throw new RuntimeException("原始异常");
} finally {
    // 这里如果抛了异常，try 中的原始异常会被吞掉
    throw new RuntimeException("finally 异常");
}
// 最终抛出的只有 finally 中的异常，原始异常丢失！
```

## 三、try-with-resources（JDK7+）

### 传统方式的问题

```java
// 传统方式：手动关闭资源，代码冗长
InputStream is = null;
try {
    is = new FileInputStream("file.txt");
    // 读取数据
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (is != null) {
        try {
            is.close();  // close() 也可能抛异常！
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

### try-with-resources 简化版

```java
// JDK7+: 自动关闭资源，实现了 AutoCloseable 接口的都可以用
try (InputStream is = new FileInputStream("file.txt");
     OutputStream os = new FileOutputStream("out.txt")) {
    // 读取和写入
    byte[] buffer = new byte[1024];
    int len;
    while ((len = is.read(buffer)) != -1) {
        os.write(buffer, 0, len);
    }
} catch (IOException e) {
    e.printStackTrace();
}
// is 和 os 自动调用 close()，即使 close() 本身抛异常也不会丢失原始异常
```

### 要求

> 实现了 `java.lang.AutoCloseable` 接口的类都可以使用 try-with-resources。

常见可用类：`InputStream`、`OutputStream`、`Reader`、`Writer`、`Connection`、`Statement`、`ResultSet` 等。

## 四、常见 RuntimeException 速查表

| 异常 | 触发条件 | 示例 |
|---|---|---|
| `NullPointerException` | 对 null 调用方法或访问属性 | `null.toString()` |
| `ArrayIndexOutOfBoundsException` | 数组索引越界 | `arr[-1]` 或 `arr[arr.length]` |
| `StringIndexOutOfBoundsException` | 字符串索引越界 | `"abc".charAt(10)` |
| `ClassCastException` | 强制类型转换失败 | `(String) new Object()` |
| `IllegalArgumentException` | 方法接收非法参数 | `Thread.sleep(-1)` |
| `ArithmeticException` | 算术异常 | `int x = 1 / 0;` |
| `NumberFormatException` | 字符串转数字格式不对 | `Integer.parseInt("abc")` |
| `ConcurrentModificationException` | 集合遍历时修改 | foreach 中用 `list.remove()` |

## 五、自定义异常

```java
// 自定义 checked 异常（继承 Exception）
public class BusinessException extends Exception {
    private int code;

    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }

    public int getCode() { return code; }
}

// 自定义 unchecked 异常（继承 RuntimeException）
public class ServiceException extends RuntimeException {
    public ServiceException(String message) {
        super(message);
    }
}

// 使用
public void validateAge(int age) throws BusinessException {
    if (age < 0 || age > 150) {
        throw new BusinessException(400, "年龄不合法");
    }
}
```

## 原理深挖

### finally 一定会执行？有一个反直觉的例外

```java
try {
    System.out.println("try");
    System.exit(0);  // JVM 直接退出了
} finally {
    System.out.println("finally");  // 永远不会输出！
}
```

`System.exit(0)` 不是抛出异常，它直接调用 Runtime.getRuntime().exit()，用本地方法关闭 JVM。此时 JVM 不会执行任何 finally 块。守护线程被强制终止也是同样的道理——JVM 退出时不管守护线程的 finally。

### try-with-resources 的编译结果

```java
// 你写的
try (BufferedReader br = new BufferedReader(new FileReader("a.txt"))) {
    return br.readLine();
}

// 编译器实际生成的（简化版）
BufferedReader br = ...;
Throwable var1 = null;
try {
    return br.readLine();
} catch (Throwable t) {
    var1 = t; throw t;
} finally {
    if (br != null) {
        if (var1 != null) try { br.close(); } catch (Throwable t) { var1.addSuppressed(t); }
        else br.close();
    }
}
```

注意 `addSuppressed`——如果 try 体和 close 都抛了异常，close 的异常会被压制并附加到 try 的异常上，通过 `ex.getSuppressed()` 可以拿到。

## 面试回答框架

> [!info] 问：说说 Java 的异常体系？

**① 一句话定位：** Throwable 是根，分 Error（程序不处理）和 Exception。Exception 又分 checked（编译期强制处理）和 unchecked/RuntimeException（编译不检查）。

**② 核心展开：**
try-catch-finally 的执行：finally 一定会执行——除非 System.exit(0) 或 JVM 崩溃。finally 里写 return 会覆盖 try 的 return 并吞掉异常，这是实际开发中的禁忌。
一个经典面试题：`try { return 1; } finally { return 2; }` 结果是多少？答案是 2——finally 的 return 覆盖了 try 的。
JDK7 引入 try-with-resources，只要实现了 AutoCloseable 就能自动关资源。如果 try 体和 close 都抛异常，close 的异常会被 addSuppressed 附加到 try 异常上。

**③ 落地经验：**
实际开发的三条底线：不要吞异常（空 catch 块），不要用异常做流程控制（异常开销大），优先用 try-with-resources。Spring 的事务回滚默认只回滚 RuntimeException 和 Error——checked 异常需要 `@Transactional(rollbackFor = Exception.class)` 才会滚。
