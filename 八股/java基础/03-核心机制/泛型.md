# Java 泛型（Generics）

## 一、什么是泛型

把**类型参数化**，让代码在编译期就具备更强的类型约束能力。本质目的是在编译期进行类型检查，提高类型安全，减少强制类型转换。

```java
// 没有泛型之前（会有类型安全问题）
List list = new ArrayList();
list.add("hello");
Integer i = (Integer) list.get(0);  // 运行时 ClassCastException！

// 有了泛型之后（编译期就检查）
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);  // 无需强转，类型安全
```

## 二、Java 泛型是"伪泛型"

> [!important] 核心概念
> Java 泛型**只在编译期生效**，运行期会发生**类型擦除**。所以称为"伪泛型"（C# / C++ 是真泛型）。

### 类型擦除（Type Erasure）

泛型信息在编译后会被擦除，运行时没有泛型类型信息：

```java
List<String> stringList = new ArrayList<>();
List<Integer> integerList = new ArrayList<>();

System.out.println(stringList.getClass() == integerList.getClass());
// 输出 true——运行时都是 ArrayList.class，泛型参数被擦除

// 类型擦除后实际存储的是 Object
List<String> list = new ArrayList<>();
list.add("hello");
// 编译器自动插入 checkcast 指令，确保类型安全
String s = (String) list.get(0);  // 等价于类型擦除后的代码
```

### 类型擦除的四大影响

> [!warning]
> 1. **不能直接 new T()** ——运行时 T 被擦除为 Object，不知道具体类型
> 2. **不能直接拿到 T.class** ——运行时泛型信息不存在
> 3. **不能重载仅泛型不同的方法** ——类型擦除后签名相同，编译冲突
> 4. **基本类型不能作为泛型参数** ——只能使用包装类（如 `int` 不行，`Integer` 行）

```java
// 错误示例：不能重载仅泛型不同的方法
// public void print(List<String> list) {}
// public void print(List<Integer> list) {}  // 编译报错！擦除后签名相同

// 错误示例：不能直接 new T()
// public <T> T create() { return new T(); }  // 编译报错

// 错误示例
// List<int> list;  // 编译报错！基本类型不行
List<Integer> list;  // 正确
```

## 三、通配符与 PECS 原则

### 三种通配符

| 通配符           | 含义               | 使用场景      |
| ------------- | ---------------- | --------- |
| `?`           | 无界通配符            | 不确定或不关心类型 |
| `? extends T` | 上界通配符（T 及 T 的子类） | 只读，生产者    |
| `? super T`   | 下界通配符（T 及 T 的父类） | 只写，消费者    |

### PECS 原则

> [!tip] PECS 口诀（面试必考）
> **Producer Extends, Consumer Super**
>
> - 从集合**读取**数据 → 生产者 → `? extends T`
> - 向集合**写入**数据 → 消费者 → `? super T`
> - 既要读又要写 → 不要用通配符

```java
// Producer Extends: 只能读，不能写
List<? extends Number> numbers = new ArrayList<Integer>();
numbers.get(0);           // 读取 OK，返回 Number 类型
// numbers.add(1);        // 编译报错！无法确定具体子类型

// Consumer Super: 可以写，读只能读到 Object
List<? super Number> list = new ArrayList<Object>();
list.add(Integer.valueOf(1));   // 写入 OK
list.add(Double.valueOf(3.14)); // 写入 OK（Number 的子类型）
Object obj = list.get(0);       // 读取只能拿到 Object
```

### Collections.copy 源码中的 PECS

```java
public static <T> void copy(List<? super T> dest,   // dest 是消费者（写入）
                            List<? extends T> src) { // src 是生产者（读取）
    // ...
}
```

## 四、泛型类 / 方法 / 接口

### 泛型类

```java
public class Box<T> {
    private T content;

    public T get() { return content; }
    public void set(T content) { this.content = content; }
}

Box<String> stringBox = new Box<>();
Box<Integer> integerBox = new Box<>();
```

### 泛型方法

```java
// 泛型方法：类型参数在方法返回值之前声明
public static <T> T getFirst(List<T> list) {
    return list.isEmpty() ? null : list.get(0);
}

// 多类型参数
public static <K, V> Map<K, V> newHashMap() {
    return new HashMap<>();
}
```

### 泛型接口

```java
public interface Generator<T> {
    T next();
}

// 实现时指定具体类型
public class IntegerGenerator implements Generator<Integer> {
    @Override
    public Integer next() { return new Random().nextInt(100); }
}
```

### 泛型边界

```java
// 限定 T 必须实现 Comparable 接口
public static <T extends Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) > 0 ? a : b;
}
```

## 五、泛型擦除后的桥接方法

```java
public class Node<T> {
    private T data;
    public void setData(T data) { this.data = data; }
}

public class MyNode extends Node<Integer> {
    @Override
    public void setData(Integer data) { super.setData(data); }
    // 编译器会生成桥接方法：
    // public void setData(Object data) { setData((Integer) data); }
}
```

桥接方法确保了泛型擦除后多态依然正常工作。

## 原理深挖

### 为什么要做类型擦除？

Java 泛型是 JDK5 引入的。当时 Java 已经有大量的老代码使用 `List list = new ArrayList(); list.add("hello"); String s = (String) list.get(0)` 这种写法。如果设计成真泛型（像 C# 那样），所有老代码和老类库全部不兼容。**类型擦除是向后兼容的妥协**——编译后泛型信息消失，字节码和 JDK4 完全兼容。

这个设计决策的后果是：Java 泛型是纯粹的**编译期检查**，运行时没有任何类型保护，这也是为什么可以通过反射往 `List<String>` 里塞 Integer。

### 桥接方法的精妙之处

```java
// 泛型擦除前：MyNode 重写了 setData(Integer)
// 泛型擦除后：父类 Node 的 setData 变成了 setData(Object)
// 如果 MyNode 只有 setData(Integer)，调用 setData(Object) 时会找不到方法
// 编译器自动生成桥接方法：
//   public void setData(Object data) { this.setData((Integer) data); }
```

这是一行代码展示了 Java 编译器如何在泛型擦除后维持多态的正确性。

### 绕过类型擦除的方法

```java
// 想在泛型方法里创建 T 的实例？传 Class 对象进来
public <T> T create(Class<T> clazz) {
    return clazz.getDeclaredConstructor().newInstance();
}

// 想拿到泛型的具体类型？让子类继承时指定
// TypeToken/super type token 技巧
class TypeReference<T> {
    Type type = ((ParameterizedType) getClass()
        .getGenericSuperclass()).getActualTypeArguments()[0];
}
```

## 面试回答框架

> [!info] 问：讲讲 Java 泛型？

**① 一句话定位：** Java 泛型是把类型参数化，目的在编译期做类型检查，消除强制类型转换。它是伪泛型——只在编译期生效，运行时信息会被擦除。

**② 核心展开（挑 2 个深度点）：**

类型擦除是 JDK5 引入泛型时为了兼容老代码做的妥协。`List<String>` 和 `List<Integer>` 编译后都是 `ArrayList`，擦除后泛型参数位置都变成 Object。这带来四个限制：不能 new T()、拿不到 T.class、不能重载仅泛型不同的方法、基本类型不能做参数。

通配符和 PECS 原则是面试必考点。`? extends T` 是上界——生产者，只能读不能写；`? super T` 是下界——消费者，能写读出来是 Object。JDK 源码里 `Collections.copy` 就是 PECS 的教科书用法——源用 extends（读），目标用 super（写）。

**③ 落地经验：**

实际开发中泛型最常用的场景是集合（`List<User>`）和通用工具类（`Result<T>`）。一个常见坑是泛型方法重载——看起来参数不同，但擦除后方法签名一样，编译冲突。还有一个是老代码和泛型混用的 unchecked warning——表示可能丢失类型安全。

> [!question] 追问：为什么不能 new T()？

> T 擦除后变成 Object，JVM 运行时不知道具体类型，没法调用具体类的构造器。解决方案是传 Class 对象：`public <T> T create(Class<T> clazz)`。
