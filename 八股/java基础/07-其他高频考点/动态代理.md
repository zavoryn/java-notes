# 动态代理

> [!summary] 我的速记
> **JDK 代理** = 基于接口（`InvocationHandler` + `Proxy`），**必须有接口**。
> **CGLIB** = 基于继承（`Enhancer` + `MethodInterceptor`），**不能代理 final 类/方法**。
> **Spring AOP：** 有接口默认 JDK，无接口用 CGLIB。Spring Boot 2.x 起**默认 CGLIB**。

## 1. 什么是代理模式？

> 为其他对象提供一种**代理**以控制对这个对象的访问。

**静态代理**：编译期就确定代理类，每个目标对象需要手写一个代理类，代码冗余。
**动态代理**：运行时动态生成代理类，一份代码代理多个对象。

## 2. JDK 动态代理

> 基于**接口**，通过 `java.lang.reflect.Proxy` 和 `InvocationHandler` 实现。

### 核心组件

| 组件 | 作用 |
|------|------|
| `InvocationHandler` | 代理逻辑的调用处理器，每个代理实例关联一个 handler |
| `Proxy.newProxyInstance()` | 动态生成代理类并创建代理实例 |

### 示例代码

```java
// 1. 目标接口
public interface UserService {
    void save(String name);
}

// 2. 目标实现类
public class UserServiceImpl implements UserService {
    @Override
    public void save(String name) {
        System.out.println("保存用户: " + name);
    }
}

// 3. InvocationHandler
public class LogHandler implements InvocationHandler {
    private final Object target;  // 目标对象

    public LogHandler(Object target) { this.target = target; }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args)
            throws Throwable {
        System.out.println(">>> [JDK代理] 方法执行前: " + method.getName());
        Object result = method.invoke(target, args);
        System.out.println(">>> [JDK代理] 方法执行后: " + method.getName());
        return result;
    }
}

// 4. 创建代理对象
UserService target = new UserServiceImpl();
UserService proxy = (UserService) Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    target.getClass().getInterfaces(),   // 必须指定接口
    new LogHandler(target)
);
proxy.save("张三");
```

### 底层原理

- `Proxy.newProxyInstance()` 底层在运行时**动态生成一个代理类的字节码**
- 这个代理类实现了所有指定的接口
- 每次调用代理对象的方法，都会转发给 `InvocationHandler.invoke()`
- 生成的代理类名类似 `$Proxy0`

## 3. CGLIB 动态代理

> 基于**继承**，通过 `MethodInterceptor` 和 `Enhancer` 实现。

### 核心组件

| 组件 | 作用 |
|------|------|
| `Enhancer` | CGLIB 的字节码增强器，用于生成子类 |
| `MethodInterceptor` | 拦截器，类似 JDK 的 InvocationHandler |

### 示例代码

```java
// 1. 目标类（不需要接口！）
public class OrderService {
    public void create(String orderId) {
        System.out.println("创建订单: " + orderId);
    }
}

// 2. MethodInterceptor
public class LogInterceptor implements MethodInterceptor {
    @Override
    public Object intercept(Object obj, Method method, Object[] args,
            MethodProxy proxy) throws Throwable {
        System.out.println(">>> [CGLIB代理] 方法执行前: " + method.getName());
        Object result = proxy.invokeSuper(obj, args);  // 注意是 invokeSuper
        System.out.println(">>> [CGLIB代理] 方法执行后: " + method.getName());
        return result;
    }
}

// 3. 创建代理对象
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(OrderService.class);       // 设置父类
enhancer.setCallback(new LogInterceptor());
OrderService proxy = (OrderService) enhancer.create();
proxy.create("ORD-001");
```

### 底层原理

- CGLIB 通过 **ASM 字节码框架**动态生成目标类的子类
- 重写父类的非 final 方法，插入拦截逻辑
- ❌ 不能代理 `final` 类 / `final` 方法（因为没有子类能重写）

## 4. JDK 动态代理 vs CGLIB 对比

| 对比维度     | JDK 动态代理          | CGLIB 动态代理                 |
| -------- | ----------------- | -------------------------- |
| 实现原理     | 基于接口，运行时生成接口实现类   | 基于继承，运行时生成目标类的子类           |
| 是否需要接口   | **必须有接口**         | **不需要接口**（但不能是 final 类）    |
| 性能       | 反射调用（JDK 7+ 做了优化） | FastClass 机制，性能通常更好        |
| 创建代理速度   | 更快（直接生成字节码）       | 稍慢（ASM 生成子类字节码）            |
| final 方法 | 可以代理（因为基于接口）      | ❌ 不能代理 final 方法            |
| JDK 内置   | ✅ JDK 内置，无需额外依赖   | ❌ 需要引入 cglib 包             |
| 代理类命名    | `$Proxy0`         | `类名$$EnhancerByCGLIB$$xxx` |

## 5. Spring AOP 中的使用策略

```java
// Spring 默认代理策略：
// 1. 如果目标实现了接口 → 使用 JDK 动态代理
// 2. 如果目标没有实现接口 → 使用 CGLIB 代理

// 可配置强制使用 CGLIB
// Spring Boot: spring.aop.proxy-target-class=true
// 注解方式: @EnableAspectJAutoProxy(proxyTargetClass = true)
```

> Spring Boot 2.x 开始，**默认使用 CGLIB 代理**（`proxyTargetClass = true`），因为在实际项目中，大部分 Bean 没有定义接口。

## 原理深挖

### CGLIB 为什么不能代理 final？

CGLIB 的原理是生成目标类的子类，重写父类方法插入增强逻辑。final 类的语义就是「不能被继承」——JVM 在类加载时检查 ACC_FINAL 标志，直接拒绝。final 方法同样，子类不能重写。所以 CGLIB 对此无能为力——这不是技术限制，是 Java 语言规则。

### Spring Boot 2.x 为什么默认切到 CGLIB？

历史原因：JDK 动态代理只代理接口方法，如果 Bean 有自调用（this.method()），这个调用不走代理，AOP 增强不生效。CGLIB 代理的是子类，所有方法调用都经过代理对象。其次，Spring Boot 推崇「约定大于配置」的服务层写法——大多 Service 不定义接口直接用实现类。CGLIB 是这两个问题的通用解。

## 面试回答框架

> [!info] 问：JDK 动态代理和 CGLIB 的区别？

**① 一句话定位：** JDK 代理基于接口（`InvocationHandler`+`Proxy`），CGLIB 基于继承（`Enhancer`+`MethodInterceptor`）。

**② 核心对比：** JDK 代理必须目标有接口——没有接口就无能为力。CGLIB 不需要接口，但 final 类/方法无法代理（不能继承/重写）。调用性能上，CGLIB 的 invokeSuper 用 FastClass 机制通常更快，但创建代理对象 JDK 更快。Spring AOP 默认：有接口用 JDK，无接口用 CGLIB。Spring Boot 2.x 起默认 CGLIB——避免了自调用 AOP 失效的问题。

**③ 选型经验：** 目标有接口且希望轻量化用 JDK 代理；目标无接口或需要自调用增强用 CGLIB。两者都是运行时生成代理类，对调用方透明。
