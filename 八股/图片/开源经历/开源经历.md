持续参与开源社区建设，Alibaba/Nacos 代码贡献者之一，累计提交多个 PR。

### PR 1：Nacos Copilot 配置测试能力建设

### PR 3：Nacos AI 平台 MCP Registry 能力补齐

面向 Nacos AI 平台 MCP Registry 能力缺口，针对资源元数据无法落库、无法查询回填、无法级联清理的问题，推动并落地 MCP Resource Registry 全链路方案，实现资源描述独立管理与服务端自动装配，补齐 MCP Server 对 resources 的一等支持，提升注册中心对 AI 资源能力的表达完整性与可维护性。

## 面试问答准备

### Q：你在 Nacos 开源社区主要做了什么？

> 关联：[微服务 > 服务注册与发现](../微服务/微服务.md) | [微服务 > Nacos vs Eureka](../微服务/微服务.md)

主要参与了三个方向的贡献：

1. **配置测试能力**：Nacos 2.x 引入了 AI 配置管理（Copilot），但缺少配置校验机制。我实现了"测试连接"功能，用户填完 API Key、模型等配置后，可以即时验证连通性，不用保存后再发现问题。这涉及到后端新增校验接口，以及 LLM 服务的可达性检测。

2. **灰度配置扩展**：原来 Nacos 的灰度发布只支持 beta IP 和 tag 两种固定规则，无法满足复杂场景。我设计并实现了基于 labels 表达式的通用规则模型，兼容原有机制的同时支持更灵活的标签匹配，覆盖了规则发布、持久化、优先级匹配的全链路。

3. **MCP Registry 全链路**：Nacos AI 平台的 MCP 注册中心缺少对 resources 的支持，资源元数据无法持久化和查询。我从零推动了 MCP Resource Registry 的方案设计和落地，实现了资源的独立管理和服务端自动装配。

### Q：你参与的灰度配置优化，具体是怎么设计的？

> 关联：[微服务 > 自定义负载均衡策略](../微服务/微服务.md)

原来的灰度规则只有两种：beta（指定 IP）和 tag（指定标签），无法做到"凡是带有 `env=canary` 标签的服务实例都走灰度配置"这种灵活匹配。

我的方案是：

1. **引入 labels 表达式**：支持 `env=canary AND region=east` 这种组合条件
2. **兼容既有机制**：原有的 beta/tag 规则继续生效，新规则作为扩展接入
3. **全链路打通**：从规则定义 → 持久化存储 → 发布流程 → 优先级匹配 → 查询返回，每个环节都需要改造

### Q：MCP Registry 是什么？你解决了什么问题？

> 关联：[微服务 > 微服务](../微服务/微服务.md)

MCP（Model Context Protocol）是 AI 领域的一个开放协议，让 AI 模型能够调用外部工具和资源。Nacos 作为注册中心，需要支持 MCP Server 的注册和发现。

我解决的核心问题是：MCP Server 声明的 resources（如数据库表、文件系统目录等）在注册到 Nacos 后，元数据没有落库，导致无法查询、无法回填、清理时也无法级联处理。

我的方案实现了：

1. 资源描述的独立持久化存储
2. 注册/注销时的自动装配和级联清理
3. 查询接口支持资源信息的完整返回
