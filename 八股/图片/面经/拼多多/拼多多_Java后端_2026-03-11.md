## 一面（项目深挖 + 并发编程）

### 一、请介绍一下你的项目，说一下项目中用到的技术栈？

> 关联：[spring](../../spring/)、[微服务](../../微服务/)

项目介绍环节，围绕项目架构、技术选型展开。

### 二、并发编程三要素是什么？

> 关联：[juc并发编程](../../juc并发编程/)

**原子性**：一个操作或者多个操作要么全部执行且不被中断，要么都不执行。
**可见性**：多个线程访问同一个变量时，一个线程修改了这个变量，其他线程能立即看到修改的值。
**有序性**：程序执行的顺序按照代码的先后顺序。

### 三、创建线程的方式有哪些？

> 关联：[juc并发编程](../../juc并发编程/)

1. 继承 Thread 类
2. 实现 Runnable 接口
3. 实现 Callable 接口（配合 FutureTask）
4. 使用线程池（Executor 框架）

### 四、线程的状态及转换？

> 关联：[juc并发编程](../../juc并发编程/)

线程状态：NEW → RUNNABLE → BLOCKED → WAITING → TIMED_WAITING → TERMINATED

- NEW：线程对象创建后尚未启动
- RUNNABLE：调用 start() 后，包括就绪和运行中
- BLOCKED：等待获取锁
- WAITING：等待其他线程唤醒（wait/join/LockSupport.park）
- TIMED_WAITING：带超时的等待（sleep/wait(timeout)/join(timeout)）
- TERMINATED：线程执行完毕

### 五、sleep 和 wait 的区别？

> 关联：[juc并发编程](../../juc并发编程/)

| 区别点 | sleep | wait |
|--------|-------|------|
| 所属类 | Thread | Object |
| 释放锁 | 不释放 | 释放 |
| 使用位置 | 任意 | 必须在同步块中 |
| 唤醒方式 | 超时自动唤醒 | notify/notifyAll 或超时 |

### 六、ThreadPoolExecutor 的参数有哪些？各代表什么含义？

> 关联：[juc并发编程](../../juc并发编程/)

7 个核心参数：
1. **corePoolSize**：核心线程数
2. **maximumPoolSize**：最大线程数
3. **keepAliveTime**：非核心线程空闲存活时间
4. **unit**：存活时间单位
5. **workQueue**：任务等待队列
6. **threadFactory**：线程工厂
7. **handler**：拒绝策略

### 七、线程池的拒绝策略有哪些？

> 关联：[juc并发编程](../../juc并发编程/)

1. **AbortPolicy**（默认）：抛出RejectedExecutionException
2. **CallerRunsPolicy**：由调用线程执行任务
3. **DiscardPolicy**：直接丢弃任务
4. **DiscardOldestPolicy**：丢弃队列中最老的任务

### 八、线程是如何进行调度的？

> 关联：[juc并发编程](../../juc并发编程/)、[操作系统](../../操作系统/)

线程调度分为协同式和抢占式。Java 使用抢占式调度，线程的调度由操作系统决定，基于时间片轮转和优先级。

### 九、synchronized 和 ReentrantLock 的区别？

> 关联：[juc并发编程](../../juc并发编程/)

| 区别点 | synchronized | ReentrantLock |
|--------|-------------|---------------|
| 实现 | JVM 层面 | API 层面 |
| 锁释放 | 自动释放 | 手动 unlock |
| 可中断 | 不可 | 可中断 |
| 公平性 | 非公平 | 可选公平/非公平 |
| 条件变量 | 单一条件 | 多个 Condition |
| 锁绑定 | 不可绑定 | 可绑定多个条件 |

### 十、volatile 关键字的作用？

> 关联：[juc并发编程](../../juc并发编程/)

1. 保证可见性：修改后立即刷新到主内存
2. 禁止指令重排序：通过内存屏障实现
3. 不保证原子性

### 十一、CAS 是什么？有什么问题？

> 关联：[juc并发编程](../../juc并发编程/)

CAS（Compare And Swap）比较并交换，是一种乐观锁的实现方式。

存在的问题：
1. **ABA 问题**：值从 A 变为 B 再变为 A，CAS 检测不到。解决：AtomicStampedReference
2. **自旋开销**：长时间不成功会一直循环，消耗 CPU
3. **只能保证一个共享变量的原子操作**：解决：AtomicReference

### 十二、AQS 是什么？原理？

> 关联：[juc并发编程](../../juc并发编程/)

AQS（AbstractQueuedSynchronizer）是 Java 并发包的核心框架。

核心原理：
- 用 volatile int state 表示同步状态
- 内部维护一个 CLH 双向队列存储等待线程
- 独占模式下 state 为 0/1，共享模式下 state 表示可同时获取锁的线程数
- 模板方法模式，子类重写 tryAcquire/tryRelease 等方法

### 十三、公平锁和非公平锁的区别？

> 关联：[juc并发编程](../../juc并发编程/)

- **公平锁**：线程按等待顺序获取锁，先到先得，性能较低
- **非公平锁**：线程可以插队获取锁，性能较高，但可能导致饥饿

### 十四、可重入锁是什么？

> 关联：[juc并发编程](../../juc并发编程/)

可重入锁指线程获取锁后，可以再次获取该锁而不会产生死锁。synchronized 和 ReentrantLock 都是可重入锁。实现原理是通过计数器记录重入次数。

### 十五、CyclicBarrier 和 CountDownLatch 的区别？

> 关联：[juc并发编程](../../juc并发编程/)

| 区别点 | CyclicBarrier | CountDownLatch |
|--------|--------------|----------------|
| 计数方式 | 加计数 | 减计数 |
| 复用性 | 可复用（reset） | 不可复用 |
| 等待方 | 线程自己等待 | 外部线程等待 |
| 触发条件 | 所有线程都到达 | 计数器归零 |

### 十六、ConcurrentHashMap 的原理？

> 关联：[juc并发编程](../../juc并发编程/)

- JDK 1.7：Segment 分段锁，每个 Segment 继承 ReentrantLock
- JDK 1.8：CAS + synchronized，数组 + 链表 + 红黑树，锁粒度为桶（Node）

### 十七、Future 和 FutureTask 的区别？

> 关联：[juc并发编程](../../juc并发编程/)

- **Future**：接口，表示异步计算的结果，提供 get()、cancel()、isDone() 等方法
- **FutureTask**：Future 的实现类，同时实现了 Runnable 接口，可以提交给线程执行

### 十八、如何排查死锁？

> 关联：[juc并发编程](../../juc并发编程/)、[操作系统](../../操作系统/)

1. 使用 `jstack` 命令查看线程堆栈，会提示发现死锁
2. 使用 `jconsole` 或 `VisualVM` 图形化工具检测
3. 使用 ThreadMXBean 的 findDeadlockedThreads() 方法编程检测

### 十九、Linux 中如何排查 CPU 飙高问题？

> 关联：[操作系统](../../操作系统/)、[jvm虚拟机](../../jvm虚拟机/)

1. `top` 命令找到占用 CPU 最高的进程
2. `top -Hp <pid>` 找到占用 CPU 最高的线程
3. `printf "%x\n" <tid>` 将线程 ID 转为十六进制
4. `jstack <pid>` 查看线程堆栈，定位问题代码
5. 常见原因：死循环、频繁 GC、线程数过多

### 二十、如何保证线程安全？

> 关联：[juc并发编程](../../juc并发编程/)

1. **不可变对象**：final 修饰，String 等
2. **同步机制**：synchronized、ReentrantLock
3. **并发工具类**：Atomic 原子类、ConcurrentHashMap
4. **ThreadLocal**：线程本地变量
5. **无同步方案**：纯函数式编程，不共享状态

### 二十一、什么是线程上下文切换？

> 关联：[juc并发编程](../../juc并发编程/)、[操作系统](../../操作系统/)

线程上下文切换是指 CPU 从一个线程切换到另一个线程时，需要保存当前线程的执行状态（程序计数器、栈帧、寄存器等），并恢复目标线程的状态。上下文切换是有开销的，频繁切换会影响性能。
