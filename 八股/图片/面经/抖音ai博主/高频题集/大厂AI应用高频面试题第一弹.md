### 1. LLM Agent的定义是什么？和"LLM+工具调用"有什么本质区别？

LLM Agent根本不是"LLM+工具调用"的缝合怪，而是以大模型为决策大脑，具备自主感知、规划、执行、记忆、反馈闭环的数字自动化生命体。

7大核心组件：
- 感知模块（耳朵/眼睛）：接收用户输入、外部环境信号、工具返回结果
- 核心大脑（LLM基座）：负责理解、推理、决策
- 规划模块（脑子想象）：把复杂任务拆成可执行步骤
- 记忆模块（记忆）：短期上下文 + 长期知识检索
- 执行模块（手脚）：调用外部工具、API、数据库
- 反馈闭环（复盘）：结果校验、错误修正、策略迭代
- 调度模块（管家）：多Agent协同、任务分发、冲突仲裁

难度：★★★ | 岗位：通用 | 标签：#Agent #基础概念 | 公司：字节、阿里（高频）

### 2. ReAct框架的核心思想是什么？和纯推理或纯行动有什么区别？

ReAct = Reasoning + Acting，核心思想是让LLM在推理和行动之间交替迭代，而不是一次推理到底或盲目行动。

5步循环：Observe（观察）→ Reason（推理）→ Act（行动）→ Reflect（反思）→ 循环

例子：用户问"北京明天天气怎么样？穿什么合适？"
1. Observe：识别需要天气数据
2. Reason：决定调用天气API
3. Act：调用天气API获取数据
4. Reflect：验证数据合理性，判断还需穿衣建议
5. 循环：基于天气数据生成穿衣建议

与纯推理（CoT）的区别：纯推理只在脑子里想，ReAct会真正去执行动作获取信息；与纯行动的区别：纯行动不思考就做，ReAct先推理再行动，有反思环节。

难度：★★★ | 岗位：通用 | 标签：#Agent #ReAct #推理 | 公司：字节、阿里（高频）

### 3. 在Agent的设计中，"规划能力"至关重要。请谈谈目前有哪些主流方法可以赋予LLM规划能力？（例如CoT, ToT, GoT等）

本质：CoT是入门玩具，ToT是进阶标配，GoT是工业级最优解，学术派玩的花活再多，落地只看效率和成本。

主流方法：
1. **CoT（思维链）**：线性规划，把任务拆成1→2→3的单一路径。小白理解：按顺序写步骤，适合简单任务（算算数、答简答）。缺点：一根筋，错了就崩，无回溯。
2. **ToT（思维树）**：树状规划，一个任务拆出多分支路径，试错后选最优。小白理解：走迷宫，多条路试一遍，选能到终点的。适合：中等复杂任务（代码调试、多选项决策）。
3. **GoT（思维图）**：图结构规划，把规划节点复用、关联，打破线性/树状限制，支持跨步骤跳转、节点共享。小白理解：思维导图+流程图，步骤能来回跳，不用重复想。工业级首选：复杂业务流程（多工具联动、长链路自动化）。
4. **其他落地向方法**：Plan-and-Solve（先规划再执行）、分层规划（顶层定战略，底层拆细节），都是工程化简化版，别搞学术炫技，落地快才是王道。

简单任务用CoT省算力，复杂商业自动化用GoT，ToT过渡，规划能力直接决定Agent能不能替代真人干活，这是核心商业价值。

难度：★★★ | 岗位：通用 | 标签：#Agent #规划 #思维链 | 公司：字节、阿里（高频）

### 4. Agent的记忆系统怎么设计？短期记忆和长期记忆分别用什么技术？

记忆系统是Agent的"大脑存储"，分为三层：

**短期记忆：**
- 上下文窗口滑动：保留最近N轮对话
- 注意力缓存：对当前任务相关的关键信息加权
- 增量更新：只更新变化部分，不全量替换

**长期记忆：**
- 结构化存储：MySQL存用户画像、偏好、历史任务
- 非结构化存储：向量库（Milvus/Chroma/FAISS）+ ES做语义检索
- 记忆摘要：长对话压缩为关键摘要，减少token消耗
- 遗忘机制：按时间衰减+重要性打分，自动淘汰低价值记忆

**外部工具支撑：**
- Redis：热点会话缓存
- Milvus/Chroma/FAISS：向量检索
- MySQL：结构化数据持久化
- ES：关键词+混合检索
- MinIO：多媒体记忆存储

难度：★★★ | 岗位：通用 | 标签：#Agent #记忆系统 #RAG | 公司：字节、阿里（高频）

### 5. Agent的工具调用（Tool Use / Function Calling）核心逻辑是什么？

Function Calling核心逻辑5步：
1. 给LLM发送工具Schema（工具名、描述、参数定义）
2. LLM做语义匹配，判断需要调用哪个工具
3. LLM输出结构化JSON（工具名+参数）
4. 工程侧解析JSON，执行工具调用
5. 将结果回传LLM，LLM基于结果生成最终回答

例子：用户问"北京今天天气怎么样？"
1. LLM收到天气查询工具的Schema
2. LLM判断需要调用天气API
3. 输出JSON：`{"tool": "weather_api", "params": {"city": "北京", "date": "today"}}`
4. 工程侧调用天气API获取数据
5. 将天气数据回传LLM，生成自然语言回答

关键点：LLM不直接调用工具，只做决策和参数生成，工程层负责执行和错误处理。

难度：★★★ | 岗位：通用 | 标签：#Agent #FunctionCalling #工具调用 | 公司：字节、阿里（高频）

### 8. 你了解哪些Agent框架？怎么选型？怎么评估？

**用过的主要框架：**
- LangChain：流程编排、工具集成、多Agent协作
- LlamaIndex：数据索引、RAG、知识库检索
- MetaGPT：多角色协作、软件开发场景
- AgentScope：分布式Agent、高并发
- 自研框架：针对特定业务定制

**选型4维度：**
- 业务复杂度：简单问答用LangChain，复杂业务可能需要自研
- 算力成本：框架本身的token消耗和推理开销
- 落地效率：开发速度、文档质量、社区生态
- 维护成本：长期迭代、团队学习曲线

**评估指标：**
- 任务完成率
- 工具调用成功率
- 幻觉率
- 响应时延
- 系统稳定性
- ROI（投入产出比）
