# 阿里大模型AI智能体应用面试题（真难）

> 来源：抖音博主「阿俊AI」
> 注：本帖为80道系列合集的Q50-59

### 50. GraphRAG的核心流程是什么？

GraphRAG本质上是把"文本检索"升级成"结构化知识检索+语义生成"。流程6步：

1. **数据摄取：** 文档清洗、分块、去噪、版本化
2. **知识抽取：** 从文本里抽取实体-关系-属性-事件
3. **图谱构建：** 写入图数据库，形成节点/边/属性，同时保留溯源信息（Source_doc_id、chunk_id、timestamp、confidence）
4. **图索引与社区/主题组织：** 为图做索引（实体名称索引、路径索引）、社区划分、子图摘要
5. **查询理解与图检索：** 用户问题先做query understanding，识别意图/实体/约束条件，然后走节点召回→邻居扩展→路径搜索→子图裁剪
6. **基于证据生成：** 把检索出的路径、关系、节点属性、源文本片段一起喂给大模型生成答案

一句话总结：RAG是从文档块里捞答案，GraphRAG是先把知识结构化，再沿着关系链把答案推出来。

### 51. GraphRAG如何保证答案的可解释性？

- **路径可追踪：** 结论不是凭模型幻觉蹦出来的，可以展示A→关联→B→导致→C的reasoning path
- **源文档可回溯：** 每个节点和边都挂Source_chunk/source_doc，不仅知道"答了什么"，还知道"依据哪段文本答的"
- **证据分层：** 区分直接证据、间接证据、推断证据
- **结构化约束：** 回答按"结论→证据节点→关系路径→引用来源"的schema输出
- **冲突显式暴露：** 图上存在两条相互冲突的路径，直接把冲突点亮出来

GraphRAG的可解释性不是prompt调出来的，是图结构+溯源设计+输出协议一起做出来的。

### 52. GraphRAG的检索过程与向量检索如何结合？

成熟系统里它们本来就是混合架构：

1. **向量检索做粗召回：** 用embedding从海量chunk或实体描述里召回Top-K，快速缩小搜索空间
2. **图检索做结构扩展：** 以召回的实体/事件为起点，在图上做1-hop/2-hop邻居扩展、路径搜索、社区聚合、关系过滤
3. **子图与文本联合重排：** 按综合分数排序（向量相似度、节点重要度、路径长度、边置信度、时间新鲜度、业务权重）
4. **双通道注入上下文：** 图证据（节点/边/路径摘要）+文本证据（原始chunk/引用片段），图负责"关系"，文本负责"细节"

常见架构：Vector-first（先向量召回再图扩展）、Graph-first（先识别实体走图再补文本）、Hybrid rank（并行召回融合排序）。

### 53. GraphRAG在工程实践中面临的主要挑战是什么？

1. **抽取质量不稳定：** 同一个实体会抽出多个别名，同一种关系会抽出十几种表达
2. **实体消歧难：** 多义词在商业场景不解决就等着事故
3. **图谱更新复杂：** 文本变了，节点/边/社区摘要都可能要重建，是增量维护的地狱
4. **查询延迟高：** 图遍历/路径搜索/子图聚合都比单纯向量Top-K重
5. **成本高：** 构图/存储/更新/查询/评测成本都高
6. **评测难：** 价值体现在多跳推理/全局关系/解释性，若评测集只是FAQ级单跳问答则纯属浪费
7. **组织层面挑战：** 业务方总期待"100%正确"，忽略图谱只是让系统更可控

### 54. GraphRAG与RAG技术异同

| 维度 | RAG | GraphRAG |
| --- | --- | --- |
| 知识形态 | 文本块 | 实体-关系图+文本 |
| 检索方式 | 语义+BM25 | 图遍历+语义召回 |
| 路径推理 | 无 | 有 |
| 擅长场景 | FAQ、文档问答、局部知识召回 | 多跳推理、关系查询、全局分析 |
| 可解释性 | 引用chunk | 路径级可解释+溯源 |
| 更新成本 | 相对低 | 显著更高 |
| 商业门槛 | 快速落地 | 适合高价值复杂场景 |

### 55. 模型微调和模型蒸馏有什么区别？

**微调：** 让模型学你的任务。常见方式SFT、LoRA、QLoRA、DPO、RLHF。直接优化模型参数，提升风格/格式/领域知识适配，适合垂类任务稳定输出。

**蒸馏：** 让小模型学大模型。用teacher生成高质量样本，student学teacher的输出分布/logits或行为。核心价值是降本增效，用更小模型逼近大模型表现，适合高并发、低延迟、边缘部署。

本质区别：目的（微调求任务适配，蒸馏求压缩迁移）、监督源（微调靠人工/业务数据，蒸馏靠teacher输出）、商业诉求（微调解决"答得像我们"，蒸馏解决"同样好但更便宜"）。

### 56. GraphRAG的构建成本主要体现在哪些方面？

1. **知识抽取成本：** 用LLM抽实体关系，token消耗大
2. **数据治理成本：** 实体归一、关系规范、属性清洗、冲突消解
3. **存储成本：** 需同时存图+原文+向量索引+摘要缓存，一套系统三份数据副本
4. **更新维护成本：** 新文档需抽取、合并、重建局部图、刷新摘要
5. **查询计算成本：** 图搜索+多跳推理+rerank+生成，整条链路成本不低
6. **评测与调优成本：** 需用多跳命中率、解释质量、复杂查询成功率等指标
7. **人才成本：** 能串联图谱/检索/生成/更新/监控的复合型人才稀缺

### 57. 数据更新频繁时，RAG与GraphRAG的表现有何不同？

**RAG优势：** 增量更新简单，新文档切块后直接embedding入库，热更新快。缺点：多文档/多实体/多跳问题易碎片化。

**GraphRAG优势：** 对复杂关系和全局语义更强。缺点：更新链路长，新增文本可能引发新实体合并、关系重写、社区重算、摘要失效。

类比：RAG像冲锋枪（反应快），GraphRAG像重炮（威力大但装填慢）。

### 58. RAG与GraphRAG是否可以结合使用？

不但可以，而且成熟系统就应该结合：

1. RAG做入口，GraphRAG做深挖：先向量/BM25召回候选文档，再从文档中识别实体进入图推理
2. GraphRAG做骨架，RAG做细节补全：图负责找到关键关系链，原文chunk负责提供具体事实
3. 按问题类型路由：FAQ/定义类/操作类→普通RAG；因果/影响/关系/多跳分析类→GraphRAG
4. 统一重排层：文本证据和图证据一起送入reranker

最佳实践：RAG保证覆盖面，GraphRAG提升复杂问题上限。

### 59. 如何进行增量文本的RAG？

核心：只更新必要部分，并保证检索一致性。

1. **文档版本化：** 每篇文档含doc_id、version、updated_at、hash，先比hash，没变不重复处理
2. **差异检测：** 对比新旧版本，定位变更段落，而非全量重切
3. **局部切块重建：** 只对变更区域重新chunk，维护chunk稳定ID
4. **局部向量更新：** 新chunk重新embedding入库，旧chunk删除或标记失效
5. **索引一致性维护：** 同步刷新BM25、倒排索引、摘要缓存、rerank索引
6. **增量摘要与记忆更新：** 只重算受影响的文档摘要/知识卡片/FAQ缓存
7. **在线读写隔离：** 更新期间双索引切换或带版本号，避免读到半更新状态
