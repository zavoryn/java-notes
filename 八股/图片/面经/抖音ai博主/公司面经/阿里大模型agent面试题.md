# 阿里大模型Agent面试题（好难）

> 来源：抖音博主「阿俊AI」
> 注：本帖为面试题系列合集的Q29-38

### 29. 如果RAG相似度评分高，但是内容预期不符，该怎么解决？

这个问题本身就暴露出很多团队对RAG的理解还停留在"向量检索=语义理解"。其实差得很远。相似度高，只能说在embedding空间里靠得近，不代表它对当前业务问题真的有用。

我一般从五层看：

1. **查询理解错了：** 用户问的是"能不能退票"，系统检索成了"退票说明"；用户要的是裁决依据，你给他找了宣导文，这当然会偏。
2. **Chunk切分错了：** 把FAQ、免责声明、案例、规则正文切在一起，向量会被噪音牵着走。
3. **缺少元数据过滤：** 用户、时间版本、产品线、区域、权限没过滤，Top-K再准也可能是错库里的正确文本。
4. **只做召回不做重排：** ANN负责"粗找"，不是"最终判定"。真正线上要稳，必须有reranker。
5. **语料质量烂：** 重复文档、过期规则、营销话术、模板垃圾太多，向量库会越来越像垃圾场。

解决方案不是一句"调阈值"就完了，那是外行打法。正确做法是：先做query rewrite + intent parse，再做metadata filter，然后hybrid retrieval，召回后做rerank，最后做answerability check。如果还不够，就做hard negative数据集，把"看起来像、其实不对"的内容专门喂给检索链路优化。

一句话总结：RAG优化的目标从来不是相似度最大化，而是业务命中率最大化。连这个都分不清，别谈生产级检索。

### 30. AI场景题：设计一个机票业务，实现退票等客服业务，用AI该怎么实现？

这题很多人一张嘴就是"大模型客服自动处理退票"，一听就知道没碰过真交易系统。退票不是聊天问题，是资金、规则、合约、风控问题。LLM能参与，但绝不能天花板。

我会这么设计：

1. **第一层 - 对话理解层：** AI负责识别意图、提取槽位、多轮追问、会话记忆。比如识别出这是refund ticket，需要订单号、乘机人、航段、退票原因。
2. **第二层 - 业务编排层：** 把流程拆成标准节点：订单识别→身份校验→票规查询→退票资格判断→费用试算→用户确认→调用退款放行→结果通知与审计。
3. **第三层 - 规则裁决层：** 这一层绝不能交给LLM。是否可退、手续费、税费返还、非自愿退判定，这些必须由规则引擎+交易系统决定。
4. **第四层 - 知识解释层：** RAG负责解释：为什么能退、为什么要收费、规则依据是什么、预计多久到账、特殊场景怎么处理。
5. **第五层 - 执行与风控层：** 真正执行退票、发起退款、写审计日志、触发消息通知。高风险操作必须human-in-the-loop，比如大额退款、异常订单、跨人操作。

所以正确答案就一句话：AI在机票客服里最适合做"理解、编排、解释、辅助决策"，不适合做"最终裁决和资金执行"。谁让大模型直接退票金额、直接天花板能不能退，那不是创新，那是等事故。

### 31. 如果AI客服获取信息不全该怎么解决？

这题考的不是模型能力，考的是你有没有做过缺失信息下的闭环服务系统。正确思路不是"继续猜"，而是"建立槽位系统+业务补数能力+动态追问机制"。

我一般这么做：

1. **定义任务必填字段：** 比如退票最少要有：订单标识、乘机人信息、航段信息、当前操作人身份。
2. **区分"必须用户提供"和"系统可反查"：** 用户没说订单号，不代表拿不到。可以从登录态、手机号、历史会话、最近订单去反查。系统能查的别让用户再填，用户体验差到这种程度，再聪明都没用。
3. **动态追问，而不是脚本追问：** 每一轮只追最关键、信息增益最高的字段。不是机械地问一堆问题，而是根据当前缺失槽位动态补齐。
4. **引入结构化交互：** 订单列表选择、乘机人选择、航段确认、退票原因选项。自然语言不是万能输入方式，老系统都知道，表单和按钮才是降歧义神器。
5. **做置信度门控：** 信息不完整、意图不确定、订单冲突时，不允许进入执行链路，只能澄清或转人工。
6. **设置人工兜底：** 如果出现一个手机号对应多个订单、多乘机人冲突、退票原因影响规则判断且识别置信度低，直接转人工。不让AI硬撑，会自知无知，才是成熟系统。

### 32. 传统RAG架构在权限管理上存在哪些风险？

传统RAG最大的问题不是答错，是越权。而且很多团队越权了自己还不知道，因为他们只盯准确率，不盯数据边界。

常见风险：

1. **建索引时权限语义丢失：** 文档切chunk之后，只剩文本和向量，原始ACL、租户、密级没带进去。
2. **检索前没做权限过滤：** 先Top-K再想权限，这个顺序就是错的。因为只要越权内容进了上下文，泄露已经发生。
3. **Chunk继承权限错误：** 一个文档里不同段落权限可能不同，切chunk后粗暴继承整个文档权限，天然有坑。
4. **多租户混库：** 企业知识、客服知识、内部制度、公域FAQ一起进一个库，不出事才怪。
5. **缓存泄露：** 检索缓存、答案缓存如果不带用户和权限标签，A的答案可能被B命中。
6. **权限变更不同步：** 员工离职、角色收回、文档降权后，向量库和缓存还没刷新，残留越权窗口。
7. **存在性泄露：** 哪怕不直接返回原文，模型一句"我查到相关内部资料但不能展示"，都已经把不该暴露的信息暴露了。

所以我对传统RAG的评价一直很直接：它不是知识系统，它是没有访问控制意识的检索系统。在企业里，这种东西答得越快，泄露得越快。

### 33. 什么是上下文感知的访问控制（CBAC）？

CBAC，Context-Based Access Control，本质上是：不是只看你是谁，而是看你在什么上下文里、为什么访问、此刻风险如何。

这就比传统RBAC高一个维度。RBAC只看角色，太粗；ABAC看属性，已经进步了；CBAC更进一步，它把会话状态、操作目的、环境风险、任务上下文一起纳入决策。

一个完整的CBAC决策一般会看：

- **主体上下文：** 用户身份、角色、租户、设备、地理位置
- **资源上下文：** 文档密级、所属系统、敏感字段
- **行为上下文：** 查看、摘要、导出、执行退款
- **会话上下文：** 当前问题、历史对话、是否二次认证
- **环境上下文：** 时间、网络、终端风险、异常行为分析

在AI场景下它非常关键，因为同一个人在不同会话里，能看到的上下文和能调的工具不应该一样。比如客服主管可以看退票规则、可以查单个用户订单、可以批量导出高净值客户名单，但未必可以批量发起退款，必须额外审批。这就是上下文感知，不是"给你一个Admin你就横着走"，那叫野路子，不叫权限设计。

### 34. Agentic RAG在权限合规方面能发挥什么作用？

先说结论：Agentic RAG不会天然带来安全，但它给了你把安全做细、做深、做流程化的机会。

传统RAG很像"一把捞"，Agentic RAG可以变成"分步骤受控访问"。它的价值主要在这几个点：

1. **分步校验：** 从意图识别、资源发现、检索、重排、总结到输出，每一步都能做权限检查。
2. **动态最小权限：** Agent不必默认拿全部知识和全部工具，而是根据任务只申请当前需要的资源。
3. **分级读取：** 先看摘要，再看脱敏，最后看全文。权限不够时停在需要层，而不是一上来把全文喂给模型。
4. **工具调用可治理：** Agent要调用退款工具、订单工具、CRM工具，每一步都能审计、限流、审批。
5. **合规链路天然可审计：** 谁、在什么任务里、调用了什么资源、输出了什么内容，可以完整记录。这点对金融、政企场景特别重要。
6. **人工兜底更自然：** 高风险节点插入human approval非常顺手，这就是agent化流程的工程优势。

所以我的观点很明确：Agentic RAG的真正价值不是更聪明，而是更可控。

### 35. MCP协议是什么？

MCP，Model Context Protocol。你可以把它理解成模型应用和外部能力之间的标准化连接协议。

注意，是"标准化连接协议"，不是"又一种函数调用格式"。很多人把MCP理解成Function Calling的替代品，这认知就太浅了。

MCP解决的是三个问题：
- 外部能力怎么统一暴露
- 外部能力怎么统一发现
- 外部能力怎么统一调用

它一般覆盖三类能力：
- **Tools：** 可执行能力
- **Resources：** 可读资源
- **Prompts：** 可复用提示模板

一句话概括：MCP是大模型时代的能力接入协议，不是某个模型厂商私有的函数格式小把戏。

### 36. Function Calling和MCP有什么区别？

先把它们放在不同抽象层看。

**Function Calling**解决的是：模型如何表达"我要调用某个函数"的问题，重点在模型输出格式，比如函数名、参数JSON、调用结构化结果。

**MCP**解决的是：外部世界如何把工具、资源、提示能力用统一协议暴露给模型应用，重点在系统集成和能力治理，不在模型输出本身。

所以最精炼的区别是：
- **Function Calling：** 偏模型接口层
- **MCP：** 偏系统集成层

或者更狠一点说：Function Calling解决"模型怎么喊"，MCP解决"工具世界怎么被组织起来"。

再落地一点：Function Calling通常是单模型会话内的结构化调用机制，MCP更像跨工具、跨服务、跨应用的能力总线协议。它们不冲突，甚至经常配合：模型用Function Calling决策要调用什么，底层再通过MCP去接真正的工具服务。这才是完整答案。只会说"一个是函数调用，一个是协议"，那等于没答。

### 37. MCP协议体现在什么地方？

不要空谈标准，瞄准工程落点。MCP一般体现在四个层面：

1. **能力注册与暴露：** MCP Server对外声明有哪些能力
2. **能力发现：** MCP Client可以动态获取能力列表，而不是在业务代码里全部写死
3. **能力调用：** 请求参数、返回结果、异常结构都走统一协议
4. **通信会话：** 通过stdio或流式HTTP把请求、响应、上下文串起来

如果落到代码架构上，一般会有几个关键位置：mcp_client、adapter、tool_wrapper、auth/audit policy gate。

所以别问"它体现在某一个函数里吗"，这是没系统观。协议永远体现在边界、体现在集成层、体现在调用契约，不是体现在某一行if-else。

### 38. 项目中哪里集成了MCP协议？

在这个项目中，MCP不是抽象名词。对于小红书平台MCP，它承担的是小红书平台能力的标准化接入协议角色。系统通过mcp_client→adapter→tool_wrapper这条链路，把外部xhs-mcp服务接进Agent编排框架。

说白了就是：外部平台能力由MCP Server提供；项目内部通过mcp_client对接协议，adapter做内部接口抽象，xhs_tools再把这些能力暴露给LangGraph/LangChain。

这套设计是对的，因为它把协议细节隔离在adapter层，上层Agent不感知底层到底是MCP、HTTP还是Playwright。这才叫架构，不是把SDK直接塞进业务代码然后美其名曰"集成AI"。
