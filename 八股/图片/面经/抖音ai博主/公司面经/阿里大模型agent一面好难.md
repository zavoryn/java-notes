# 阿里大模型 Agent 一面好难

> 来源：抖音博主「阿俊AI」
> 原文链接：https://www.douyin.com/note/7620251836972622248

### 1. RAG 流程、编码模型了解（原理、优缺点），如何评估编码模型？

答：

**RAG 典型流程：** 文档切分与清洗 → 向量化（Embedding） → 建索引（向量库/混合检索） → 用户提问检索 Top-K → 将检索片段与问题拼进 Prompt → 模型生成答案 → 可加重排与引用校验。

编码模型侧重代码语义与结构，常见基于 Transformer 的代码预训练/指令微调。优点是补全、重构、跨文件理解强；缺点是对长上下文、私有库、弱类型语言易误判。

**评估可用：** HumanEval/MBPP 等功能正确率、代码相似度与可执行性、静态分析（类型/安全），以及业务场景的 Merge Request 通过率与线上缺陷率。

### 2. RAG 分类、多模态 RAG 框架、伪多模态 vs 多模态、CLIP 适用哪类及成因

答：

RAG 可按检索对象分为文本、表格、图谱、混合检索；按架构分单跳、多跳、Agentic RAG 等。

**多模态 RAG：** 常见框架思路在 LangChain、LlamaIndex、Haystack 等通过多模态嵌入与向量库实现。

- **伪多模态：** 先把图片 OCR/Caption 成文本再走文本 RAG
- **真多模态：** 图片/音频等直接入多模态编码器，与文本在同一嵌入空间对齐检索

**CLIP：** 属于图像-文本对齐的对比学习模型，适合"以图搜文/以文搜图"的跨模态检索与图文匹配类多模态 RAG，因其联合嵌入空间便于相似度检索。

### 3. RAG 怎么评估？评估体系里最重要的是什么？

答：可从检索与生成两方面：
- **检索：** Recall@K、MRR、nDCG
- **答案：** 忠实度（是否 grounded）、正确性、完整性
- **端到端：** 可用带标准答案的数据集或 LLM-as-judge（需校准）

体系中最重要的是**忠实度与可验证性**——答案必须能被检索证据支撑，否则再流畅也会幻觉；其次才是流畅性与相关性。

### 4. 传统 RAG 痛点、GraphRAG、难点、增量场景如何应对？

答：

**传统 RAG 痛点：** 切块破坏结构、长文档多跳推理弱、实体关系难表达、更新频繁时索引漂移与一致性差。

**GraphRAG：** 用图结构组织实体关系，支持社区摘要与全局问答。难点在高质量构图、图规模与查询延迟、增量更新与冲突消解。

**增量可：** 对新文档做实体链接与关系抽取后局部更新子图、版本化快照、定期合并与重算社区摘要，并对变更部分做向量与图双写。

### 5. 微调负责的工作、大模型微调最重要的是什么？

答：

**工作通常包括：** 数据构造与清洗、格式与指令模板设计、训练策略（全参/PEFT）、评测与对齐业务指标、部署与监控。

**最重要的是数据质量与任务定义**——数据决定能力边界，任务定义不清则指标再好也难上线；其次才是算力与算法细节。

### 6. 后训练方式有哪些？微调方式及做法？LoRA 原理与参数量

答：

**后训练常见：** SFT、偏好学习（DPO/RLHF）、安全对齐等。

**微调可分：** 全参数微调与参数高效微调（Adapter、Prefix、LoRA 等）。

**LoRA 原理：** 在低秩矩阵 A 和 B 上训练，前向用 W' = W₀ + BA 近似更新，冻结原 W₀。可训练参数量约为 2×r×d（每层每矩阵），当 r << d 时远小于全参。

### 7. DPO、DPO 与 PPO 区别

答：

**DPO（Direct Preference Optimization）：** 直接用偏好数据优化策略，使模型更倾向优选回复。通过将奖励模型目标改写为分类式损失，无需显式训练奖励模型与在线采样。

**PPO：** 属于 RLHF 管线：先训奖励模型，再用 PPO 对策略做强化学习，过程复杂，对超参与稳定性敏感。

**DPO 更简单稳定，数据效率高；PPO 更灵活可接多约束但工程成本高。**

### 8. 介绍一些 Agent 实现框架、有什么区别？LangGraph 适用场景？构建方式有哪几种？

答：

**常见有：** LangChain、Semantic Kernel、AutoGen、CrewAI、LangGraph 等。

**区别：** 编排模型（链式 vs 多智能体 vs 图状态机）、记忆与工具生态、可控性与调试体验。

**LangGraph 适用场景：** 有环、分支、人机协同、需明确状态与检查点复杂流程。

**构建方式：** 用图节点表示 LLM/工具/路由，边表示转移；通过状态 Schema、条件边、子图与 checkpoint 组合出工作流。

### 9. 场景题：客户输入一个软件或网页界面截图，怎么通过 RAG 的方式帮助用户了解界面每一个组件的作用？

答：针对用户上传界面截图解析组件功能并区分相似组件的需求，核心通过多模态 RAG 技术实现：

1. **截图解析：** 先通过 CV 轮廓检测、OCR 识别提取组件位置、文字内容、视觉特征（如是否含播放按钮、输入光标），将图像信息转化为结构化文本描述
2. **UI 组件知识库构建：** 构建包含组件定义、功能、视觉特征、区分规则的知识库（如图文框：静态内容，无播放控件；视频框：含进度条、播放按钮、动态帧序列），并基于 SentenceTransformer 生成向量嵌入，存入 FAISS 索引
3. **检索增强生成：** 将截图解析的文本特征作为检索词，从知识库中匹配最相似的组件信息，再结合 LLM 生成每个组件的功能解释
4. **相似组件区分：** 依托知识库中预定义的区分特征（如视频框独有播放控制栏、图片框无时间轴），通过检索结果对比，精准输出相似组件的核心差异

**整体流程：** 截图解析 → 向量检索 → LLM 生成，确保解释精准且贴合组件实际特征。

### 10. 算法题：2n+1 个数，配两对，找出单独的那个数

答：核心解法是利用异或运算的特性：相同数异或结果为 0，0 与任何数异或结果为该数本身。

**实现逻辑：** 初始化结果为 0，遍历所有数字，依次与结果做异或运算。由于成对数字异或后相互抵消（结果为 0），最终剩余的结果即为单独的数。
