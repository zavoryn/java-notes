# 百度AI应用开发一面

> 来源：抖音博主「阿俊AI」

### 1. System Message 和 User Message 应该怎么分工？
> 关联：[spring](../../spring/)

1. System Message：全局角色定位、行为约束、输出规范、权限边界、底层规则，全局生效、全程不变，定义Agent身份、禁止行为、格式标准、流程约束。
2. User Message：承载用户实时query、业务输入、会话上下文、临时指令，动态可变，仅负责业务诉求传递，不改动底层架构规则。
3. 工程隔离：System固化底层范式，User只做业务交互，避免用户输入篡改系统逻辑、越权篡改行为规则。

### 2. 结构化输出有哪些常见做法？各自的优缺点是什么？
> 关联：[spring](../../spring/)

1. 固定Prompt约束JSON：接入简单、零额外开发；容错差，模型易乱格式，复杂字段易缺漏。
2. 正则+语法校验兜底：低成本纠错、线上易兼容；嵌套复杂结构正则难覆盖，维护成本高。
3. 指令+Schema强制约束：基于JSON Schema定义字段类型、枚举；规范强、可校验，适配生产；Prompt冗余、消耗上下文。
4. 分层Parser后处理：模型自由输出+后端解析映射；鲁棒性最强；多一层开发与运维链路。

### 3. Prompt Injection是什么？为什么RAG/Agent场景更要小心？
> 关联：[spring](../../spring/)

Prompt Injection是用户/外部输入插入恶意指令，绕过系统约束、篡改角色逻辑、诱导越权执行。
RAG会把外部文档/用户问答直接灌入上下文，恶意指令随检索内容注入；Agent具备工具调用、流程执行能力，注入后可触发非法接口调用、数据泄露、越权操作，比纯对话场景破坏力更强、影响链路更深。

### 4. 如果文档里写着"忽略之前所有指令"，你会怎么处理？

1. 前置内容风控过滤，命中忽略指令、重置角色类关键词直接拦截过滤。
2. 分层上下文隔离：System Message做不可篡改底层基座，隔离检索文档与用户输入篡改空间。
3. 如规则强约束：设定外部输入无权限覆盖系统指令，模型层做优先级锁定。
4. 后置校验：检测输出是否偏离预设角色与规则，异常直接熔断重生。

### 5. 浏览器Agent怎么防网页里的隐藏提示词？

1. 页面DOM层过滤：删除隐藏元素、注释节点、透明文字、隐藏div等隐蔽文本。
2. 内容白名单抽取：只解析可视主体内容，过滤源码注释、隐藏脚本文本。
3. 输入脱敏规约：网页文本统一做指令关键词脱敏，拦截重置、篡改类敏感句式。
4. Agent侧权限隔离：网页内容仅作业务参考，无权修改系统指令与行为规则。

### 6. 为什么微调和对齐不能从根本上解决Prompt Injection？

1. 微调只能拟合固定分布，未知新型注入句式泛化拦截不足。
2. 对齐依赖训练数据，无法覆盖线上无穷多用户恶意构造话术。
3. 注入利用上下文优先级逻辑漏洞，是架构层问题，非模型能力问题。
4. 模型仍会遵循上下文靠前/权威指令逻辑，对齐无法打破上下文优先级。

### 7. Prompt Engineering中，如何系统优化提示词并把准确率做上去？
> 关联：[spring](../../spring/)

1. 分层架构：系统角色+约束规则+输出格式+示例Few-shot四层拆分固化。
2. 样本迭代：沉淀badcase做反例Prompt，补充边界场景示例。
3. 规则量化：明确枚举、范围、禁止行为，减少模型自由发挥空间。
4. 上下文精简：冗余信息裁剪、关键信息前置，降低模型歧义。
5. 闭环校验：输出后置规则校验、格式校验，不达标自动重绘重试。

### 8. 如果Prompt怎么改都上不去，你怎么判断应该切到RAG、Tool、Workflow还是微调？
> 关联：[spring](../../spring/)

1. 缺外部实时/私有知识→切RAG；
2. 需调用接口、执行业务操作→切Tool；
3. 固定多步骤强流程、分支判定明确→切Workflow；
4. 领域话术、固定范式批量偏差、规则无法穷举→切微调；
5. 先流程固化再知识补齐，最后模型微调，按工程成本由低到高选型。

### 9. badcase集应该怎么建，才能真的指导迭代？

1. 分层归类：歧义类、知识缺失类、格式错误类、指令越权类、工具调用错误类。
2. 字段标准化：记录query、原始Prompt、模型输出、预期结果、错误根因。
3. 版本绑定：关联Prompt版本、模型版本、检索链路版本。
4. 闭环复用：badcase灌入Few-shot、规则库、风控词库，迭代后回归复测。

### 10. 如果线上效果突然下降，你怎么判断是Prompt问题、模型版本问题，还是外部知识/工具链路问题？

1. 固定变量对照：同Query切旧Prompt、旧模型版本复现。
2. 链路排查：检索命中文档是否变更、工具接口返回是否异常。
3. 分层灰度：Prompt灰度、模型版本灰度、检索源灰度隔离验证。
4. 日志定位：看上下文构造、模型入参、工具返回、后处理日志，锁定故障层级。

### 11. 当工具很多时，如何控制上下文长度和调用稳定性？
> 关联：[spring](../../spring/)

1. 工具按需注册，非场景工具不灌入上下文。
2. 工具描述精简标准化，统一入参出参格式，压缩token。
3. 增加工具路由前置，先选候选子集再灌入模型。
4. 限制单轮最大调用次数，加超时、重试、熔断机制。
5. 会话上下文做滚动裁剪，保留关键流程舍弃冗余历史。

### 12. 工具太多时，应该优先做路由还是优先改工具粒度？

优先收敛工具粒度，再做路由。
先把细粒度原API聚合为业务动作级粗粒度工具，减少工具总数与描述token；粒度收敛后再做意图路由、分类匹配，否则工具泛滥会导致路由歧义、上下文爆炸，先路由无法解决基数过大的底层问题。

### 13. Tool Retrieval和普通知识检索有什么区别？

1. 检索对象：知识检索查文档文本；Tool Retrieval检索可用工具、能力描述。
2. 目标用途：知识检索补业务知识；工具检索匹配可执行动作。
3. 匹配维度：知识检索语义相似度；工具检索兼顾意图、参数、能力标签。
4. 链路闭环：工具检索直接关联调用入参、协议规范，可直接编排执行。

### 14. 为什么业务动作级工具通常比原子API更适合模型调用？

1. 原子API粒度太细，单业务需多步调用，易产生多轮错乱。
2. 业务动作封装复杂逻辑、参数兼容、异常处理，减少模型理解负担。
3. 大幅减少工具数量，降低上下文token与选择歧义。
4. 符合人类自然语言意图表达，模型更容易匹配业务动作，调用稳定性更高。
