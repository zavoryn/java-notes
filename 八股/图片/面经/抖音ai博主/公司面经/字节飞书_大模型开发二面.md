# 字节飞书大模型开发二面（合同审查项目）

> 来源：抖音博主「阿俊AI」

### 1. 合同审查的整体架构是怎样的？从文件上传到最终的推理输出，完整流程是什么

整体架构为文件接入层、文本解析层、预处理层、检索层、大模型推理层、结果输出层与管控层。完整流程为用户上传合同文件（PDF/Word/图片）→ 文件格式解析与文本提取 → 文本清洗、分段、结构化 → 向量化入库构建知识库 → 输入审查指令/问题 → 检索相关合同条款 → 上下文拼接进入大模型 → 模型执行风险识别/条款解读/合规性验 → 输出审查报告、风险、修改建议 → 结果结构化展示与日志留存。

### 2. 请具体说明RAG（检索增强生成）的完整流程

RAG完整流程为：离线构建知识库 + 在线检索生成。
1. 文档加载：读取合同、法规、判例等原处文件；
2. 文本预处理：去噪、分段、去重、关键词提取；
3. 向量化：通过Embedding模型将文本转为向量；
4. 向量入库：存入向量数据库，建立索引；
5. 用户Query向量化：将问题转为同一空间向量；
6. 检索召回：向量检索 + 关键词检索召回相关片段；
7. 重排精筛：通过交叉编码器或相似度排序过滤噪声；
8. 上下文构建：将召回片段与问题拼接成Prompt；
9. 大模型推理：基于检索内容生成回答；
10. 结果后处理：格式整理、去重、合规校验。

### 3. 指令微调中，三轮微调具体如何实施？涉及的损失函数如何选择？混淆矩阵如何应用？若数据存在不均衡问题，该如何处理

三轮微调采用渐进式微调策略：
1. 第一轮：基础指令对齐，用通用合同数据做全参数或LoRA调，学习基本格式与意图；
2. 第二轮：领域专项微调，聚焦风险点、条款类型、合规判断，强化专业能力；
3. 第三轮：对齐人类偏好，人工标注优质输出DPO/RLHF，提升准确性与实用性。

损失函数：分类任务用CrossEntropyLoss，回归用MSE，生成任务用NLLLoss/CrossEntropy。偏好对齐用DPO损失，关注分析与条款类型，指导数据补充与调值调整。数据不均衡：采用类别加权Loss、过采样少数类、欠采样多数类、合成样本（SMOTE）、Focal Loss。

### 4. 针对数据不均衡的二分类任务，可参考哪些评价指标？AUC指标在其中的作用是什么

常用评价指标：精确率（Precision）、召回率（Recall）、F1分数、Fβ分数、AUC、PR曲线、Kappa系数。AUC衡量模型对正负样本的区分能力，不受类别不均衡影响，能客观反映模型整体排序能力。在合同风险识别等场景中，AUC越高，说明模型越能把高风险样本排在前面，避免漏判重要风险。

### 5. XGBoost与LightGBM如何进行加权融合？具体的融合思路和操作方法是什么

加权融合思路：基于单模型表现分配权重，对预测概率做加权求和。操作方法：
1. 分别训练XGBoost与LightGBM，在验证集输出预测概率；
2. 计算单模型AUC/F1分配权重，或用网格搜索/线性回归优化权重；
3. 融合公式：最终概率 = w1 * xgb_prob + w2 * lgb_prob；
4. 也可采用堆叠融合（Stacking），以单模型输出作为特征，训练LR/逻辑回归做第二层融合；
5. 对分类任务，加权后设置最优阈值输出类别。

### 6. XGBoost的定义是什么？其核心优势有哪些？决策树、随机森林的学习率如何设置？XGBoost的层级设置及内存占用特点是什么？LightGBM为何采用叶子生长策略

XGBoost是极端梯度提升树，基于梯度提升框架，支持并行计算、正则化与二阶泰勒展开。核心优势：精度高、鲁棒性强、自带L1/L2正则防止过拟合、支持缺失值处理、列抽样、并行训练、可解释性强。

学习率设置：树模型一般初始0.1/0.3，文献复现时调低至0.01~0.1，配合更多树数量保证收敛。XGBoost采用按层生长，内存占用高，需存储每节点特征与梯度信息。LightGBM采用按叶子生长，优先分裂增益最大的叶子，收敛更快、精度更高，同时通过直方图优化降低内存占用，适合大规模数据。

### 7. 将XGBoost与LightGBM进行融合，需要考虑哪些关键因素

关键因素：
1. 数据一致性：确保训练集、验证集、特征工程完全一致；
2. 模型差异性：保留单模型独特差异，避免同质化导致融合增益低；
3. 权重分配依据：基于验证集AUC/F1/损失分配，避免主观赋值；
4. 阈值选择：不均衡数据下需重新确定分类阈值；
5. 推理性能：融合后延迟是否满足上线要求；
6. 过融合风险：避免在训练集上过拟合，需用独立测试集验证。

### 8. 如何使用LangGraph构建RAG系统？文件的处理方式、检索逻辑（如递归分段、向量检索+关键词检索重排）具体是怎样的

LangGraph构建RAG采用状态化、可中断、可循环的智能体流程：
1. 定义状态：存储文档、检索结果、对话历史、模型输出；
2. 节点拆分：文件加载→文本切分→向量化→检索→重排→推理→输出；
3. 条件边：判断是否需要多轮检索/自我修正。

文件处理：支持PDF/Word/txt/md，用PyPDF/Unstructured解析，递归按标题/段落分段，控制块大小与重叠字符。检索逻辑：
- 向量检索：召回Top-K相关片段；
- 关键词检索（BM25）补充召回；
- 混合结果送入重排模型（Reranker）精排；
- 最终选取高相关片段送入大模型。

### 9. RAG系统搭建过程中，面临的主要难点有哪些

主要难点：
1. 长文本处理：合同条款冗长，分段不当导致上下文断裂；
2. 复杂逻辑问题：多条款关联、跨章节推理能力不足；
3. 检索不精准：召回无关内容，引发模型幻觉；
4. 文本截断：Token超限丢失关键信息；
5. 显存溢出：Embedding + 大模型推理占用高；
6. 多轮对话理解：历史上下文管理困难；
7. 领域专业性强：通用Embedding对合同术语表达弱。

### 10. 在模型微调时，选择Llama 2进行微调的原因是什么

Llama 2的原因：
1. 开源可用，可商用版本降低落地成本；
2. 效果均衡，7B/13B参数适合私有化部署；
3. 指令遵循能力强，经过SFT微调，适合合同审查等任务；
4. 生成高效，支持LoRA/QLoRA高效微调，降低硬件要求；
5. 长文本支持较好，能处理合同类长文本理解；
6. 社区资源丰富，便于二次优化与问题排查。

### 11. 该项目是计划在内部上线还是对外部署？目前项目的完成度处于什么阶段

项目优先支持企业内网私有化部署，保证合同数据安全与合规；后续可封装为SaaS服务对外输出。目前已完成核心流程开发：文件解析、RAG识库、基础审查推理、风险点识别模块；前端界面、多格式导出、批量审查、权限管控等正在完善，整体处于可演示内测阶段，距离正式上线还需压测与迭代优化。

### 12. 你是如何对RAG系统进行评估的（如精确度、召回率、忠实率、相关性、RAGAS指标等）

采用人工评估 + 自动化指标结合：
1. 检索层：计算召回率、精确率、MRR，检验条款是否被正确召回；
2. 生成层：使用RAGAS体系，包括忠实率、答案相关性、上下文精度、上下文召回率；
3. 业务指标：风险点漏检率、错误解读率、格式合规率；
4. 人工评估：从准确性、完整性、专业性、可读性打分；
5. 长期通过A/B测试对比不同Embedding、分段策略、重排模型效果。

### 13. 当模型出现幻觉问题时，具体采用哪些解决方法

解决方法：
1. Prompt优化：强制要求引用原文条款，禁止无依据推断；
2. RAG增强：提升检索精准度，增加推理环节，过滤无关上下文；
3. 思维链引导：使用CoT分步推理，避免跳跃式结论；
4. 模型微调：用高质量合同审查数据SFT，减少虚构内容；
5. 校验机制：增加事实校验节点，比对生成内容与原文一致性；
6. 少样本示例：在Prompt中加入正确示例，规范输出格式与依据。

### 14. 当数据存在缺失、达到处理上限，或格式不完整（如非JSON格式）时，该如何处理

处理策略：
1. 缺失值：数值型用均值/中位数填充，类别型用默认值，文本型标记缺失；
2. 处理上限：做分批处理、流式读取，超长文本自动分段，避免内存溢出；
3. 格式不完整：非JSON依正则提取关键信息，无法解析则丢弃并记录日志；
4. 异常兜底：设置默认返回结构，保证接口不崩溃；
5. 数据校验：前置校验模块，提前过滤脏数据，提升系统稳定性。
