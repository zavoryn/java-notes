# 福报厂AI智能体应用面试题（贼难）

> 来源：抖音博主「阿俊AI」
> 注：本帖为面试题系列合集的Q20-28

### 20. Function Calling和MCP有什么区别

先把它们放在不同抽象层看：

**Function Calling**解决的是：模型如何表达"我要调用某个函数"的问题，重点在模型输出格式，比如函数名、参数JSON、调用结构化结果。

**MCP**解决的是：外部世界如何把工具、资源、提示能力用统一协议暴露给模型应用，重点在系统集成和能力治理，不在模型输出本身。

所以最精炼的区别是：
- **Function Calling：** 偏模型接口层
- **MCP：** 偏系统集成层

或者更狠一点说：Function Calling解决"模型怎么喊"，MCP解决"工具世界怎么被组织起来"。

再落地一点：Function Calling通常是单模型会话内的结构化调用机制，MCP更像跨工具、跨服务、跨应用的能力总线协议。它们不冲突，甚至经常配合：模型用Function Calling决策要调用什么，底层再通过MCP去接真正的工具服务。这才是完整答案。

### 21. MCP协议体现在什么地方

MCP从不是虚头巴脑的概念，真身就三件事：

1. **清单声明：** Server通过Manifest暴露tools、resources、prompts，把能力亮明白。
2. **传输规约：** stdio、流式HTTP做载体，JSON-RPC完成消息交互，干净利落。
3. **执行闭环：** Client拉取工具→转模型调用格式→模型返回指令→执行工具→回传结果，全链路标准化。

市面上Cursor、Claude Desktop、各类集成MCP的IDE，界面上的配置项、日志里的tools/list、tools/call，全是这套协议的表层皮。懂行的一眼就能看穿底层逻辑，没任何花活。

### 22. 怎么保证AI的准确度

别迷信大模型参数，也别指望换个模型就能解决幻觉。AI准确度从来不是模型给的，是工程硬堆出来的系统指标。没有银弹，只有层层设防的内功：

1. **约束输出：** 用JSON Schema、枚举、必填字段锁死格式，从根上杜绝胡编。
2. **RAG溯源：** 答案绑定检索片段，强制模型仅依据上下文生成，加引用实锤。
3. **工具兜底：** 计算、查库、实时数据全走API，绝不靠模型心算编造。
4. **验证护栏：** 二次模型校验、规则拦截，关键路径必须人在回路，不留漏洞。
5. **评测兜底：** 固定测试集回归、任务级指标、LLM评判校准，闭环迭代。
6. **策略优化：** 复杂任务拆分解耦，如实任务拉低温度，不确定就直说不知道。

小结：准确度靠数据、流程、评测、护栏四重加固，单换模型纯属自欺欺人。

### 23. 怎么实现智能体（Agent）

Agent从来没有玄学，就是一套感知、规划、行动、记忆、管控的工业级闭环工程：

1. **感知：** 用户输入、多模态数据、记忆、文档检索，打底输入源。
2. **规划：** ReAct思考（行动-观察）是基础思路，复杂场景上Planner或状态机。
3. **行动：** Function Calling调用能力，MCP做工具层兜底，执行API、代码、文件操作全标准化。
4. **记忆：** 短期靠上下文窗口，长期靠向量库+结构化存储，分层管理。
5. **控制流：** 加执行上限、超时、权限、审计日志，做重试降级，防止失控。
6. **编排：** 复杂任务上DAG工作流，支持并行、人工确认、状态持久化。

工程上凑齐工具定义、循环执行器、状态存储、安全护栏，就是能落地的Agent骨架。市面上吹的各种高级Agent，本质都是这套逻辑的包装。

### 24. AI助手会话记忆是怎么实现的

会话记忆的核心是分层治理——存储从来不是难点，萃取策略才是真内功：

1. **短期（会话内）：** 消息落DB/Redis，按session_id取最近K条或token截断，流式写入基础操作。滚动摘要——超上下文窗口就做历史压缩，摘要+近原文组合注入，平衡长度与信息。
2. **长期（跨会话）：** LLM/规则抽取事实、偏好，存入结构化表、向量库，新会话先检索再注入。
3. **用户级画像：** 聚合多会话标签，与会话解耦，实现全局个性化。

别纠结存MySQL还是向量库——什么时候抽、怎么去重合并、如何精准召回，才是区分新手与老手的关键，这才是记忆系统的核心壁垒。

### 25. 会话隔离是怎么实现的？@memoryId的底层原理

会话隔离就是多租户系统的基础基本功，没任何魔法，纯靠ID鉴权：

1. **强制数据隔离：** 所有消息、记忆、检索条件绑定session_id/tenant_id，查询强制加WHERE过滤。
2. **API层JWT鉴权：** 锁死用户权限，杜绝跨会话串数据，这是入行就该懂的底线操作。

至于@memoryId这类语法，不过是产品层的语法糖：解析后映射为服务端记忆命名空间、集合ID，底层就是普通字符串主键。所有读写操作绑定该ID做作用域隔离，与会话ID多对多关联。本质就是：解析→绑定资源ID→全链路过滤。玩不出任何高端操作，懂的都懂。

### 26. 会话上下文太多，怎么把内容联系起来

别傻到把全量聊天记录塞进上下文窗口，纯纯新手行为。老程序员处理海量上下文，靠的是全局+局部双连贯策略：

1. **层次压缩：** 远历史做摘要，近历史留原文，取舍有度。
2. **检索增强：** 问题向量化，从历史摘要、关键片段中召回相关内容，轻量化RAG兜底。
3. **实体治理：** 抽取人、项目，构建轻量知识图谱，先定位实体再拉关联内容。
4. **话题拆分：** 检测话题切换就开子线程，避免单线程无限堆叠。
5. **Prompt管控：** 滚动窗口+重要消息钉选，加权保留核心信息。

全局连贯靠检索+结构化记忆，局部连贯靠近邻消息，这是处理长对话的标准答案，没任何捷径。

### 27. RAG原理（索引、检索）

RAG就是一套标准化的离线建库、在线检索流水线，索引质量直接决定上限：

**索引（离线）：**
文档切块（按需选固定长度/语义切块）→清洗去重→打元数据（权限、来源、时间）→Embedding向量化→写入向量索引（HNSW/IVF），搭配BM25做混合检索，增量更新、版本控制，工程化落地。

**检索（在线）：**
用户Query向量化→混合检索Top-K→MMR保多样性→元数据做租户、合规过滤→重排序精排→片段注入Prompt，强制模型按片段生成并加引用。

这是RAG的核心骨架，所有优化都是围绕分块、向量、重排做细节打磨，脱离工程落地的概念都是空谈。

### 28. 怎么将用户输入的语言与准备好的文档进行匹配

语义匹配没什么高深玄学，本质就是把用户query和文档拉到同一维度做比对：

1. **核心匹配：** 同Embedding空间算向量相似度，这是底层根基。
2. **补漏匹配：** BM25关键词检索，兜底语义检索的漏召回。
3. **混合精排：** 向量+稀疏检索加权打分，再上重排模型精调。
4. **查询优化：** Query扩展、HyDE、指代消解，提升召回精度。
5. **跨语言兜底：** 多语言Embedding对齐，或先翻译统一语种，重点保障领域术语准确性。

不管是自然语言语义匹配还是跨语种匹配，底层逻辑都是统一向量空间+多层检索，懂工程的都能直接落地。
