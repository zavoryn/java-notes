### 3. 你怎么理解Agent项目的数据，和传统模型训练数据有什么不一样

核心差异：
- 实时性：Agent数据强实时，商户业务数据、接口返回数据时时刻刻动；训练数据多为静态离线数据；
- 智能性：训练数据教模型"懂知识"，Agent数据教智能体"会做事"；
- 迭代方式：Agent靠链路日志、失败案例、规则迭代快速优化，不用大规模标注；传统模型依赖高额标注成本、离线训练；
- 商业价值：Agent数据直接关联业务转化、运营效率，数据价值即时变现；训练数据是长期技术投入，回报周期极长。
简单说：训练数据是给模型"补课"，Agent数据是给智能体"复盘工作"。

### 4. 如果让你优化一个Agent系统，你通常先从哪里下手

外行优化改Prompt、换模型、加Few-shot；内行优化先抓架构瓶颈与链路短板，这就是工程化思维的碾压。

我的固定优化顺序：
1. 先排查任务链路瓶颈：大模型调用耗时、工具串行阻塞、无效多轮调用，这是Agent最大性能黑洞；
2. 再做异常与容错治理：工具脏数据、LLM输出格式错乱、任务死循环、超时卡死，90%线上事故都出在这里；
3. 其次优化记忆与上下文管理：上下文溢出、冗余历史拼接、无关信息干扰决策；
4. 最后才是Prompt优化、模型选型、检索策略调优。

绝大多数垃圾Agent系统，不是模型不够强，是架构烂、没容错、没状态管理，一堆临时拼接的逻辑堆在一起，稍微复杂场景直接崩。

### 5. 如果系统里需要接前端页面，你会怎么考虑React这类前端框架的接入方式

十年全栈视角，前后端一体化设计，拒绝割裂开发。
1. 架构分层：Agent核心逻辑完全内收在后端服务，前端只做交互渲染、指令输入、结果展示、流式输出渲染，绝不把决策、工具调用、任务拆解放在前端，避免安全泄露、逻辑失控；
2. 接入方案：React项目通过统一BFF层对接Agent接口，封装流式SSE/WebSocket双向通信，适配大模型流式输出、多任务实时进展展示；
3. 工程适配：接口统一鉴权、多租户隔离、请求加解密，前端做加载状态、错误兜底、优雅降级；
4. 扩展设计：采用组件化封装Agent对话框、任务看板、执行日志面板，后续快速复用在后台管理台、商户控制台、客服系统。
原则很明确：AI核心能力服务端闭环，前端只做纯粹的展示与交互，各司其职，才是可长期维护的工程方案。

### 6. 你了解哪些Agent架构，分别适合什么问题

| Agent架构 | 适配场景 |
| --- | --- |
| 单Agent直驱架构 | 轻量化最简架构，适合简单问答、单工具查询、低复杂度场景，开发快、成本低 |
| 规划式Agent（ReAct、CoT） | 思考-行动闭环，适合中等复杂度任务，需要分步调用工具、简单推理 |
| 分层编排Agent | 自研落地架构，意图层-规划层-执行层-记忆层解耦，适合企业级复杂业务、多步骤串联任务，强稳定性要求 |
| 多Agent分布式协作架构 | 各司其职、角色拆分，适合大型系统、跨模块业务、复杂运营闭环 |
| GraphRAG+Agent融合架构 | 适合知识关联强、需要溯源、复杂推理的行业场景 |
| Function Call原生智能体 | 适合标准化工具调用、接口闭环、强格式约束场景 |

没有最好的架构，只有最适配业务的架构，小白才会无脑只用ReAct，成熟架构师一定按需选型。

### 7. 多Agent协作里最难解决的是什么

表层问题是通信、分工；核心最难的是权限边界、任务冲突、全局一致性与不可控连锁反应。
1. 权限边界模糊：多个智能体职能重叠，重复执行、互相抢占任务；
2. 信息孤岛：各Agent独立记忆、独立上下文，信息不同步，决策矛盾；
3. 连锁副作用：一个Agent执行错误，引发其他子任务连环出错，无法快速追踪；
4. 全局目标偏离：子Agent只关注自身子任务，忽略整体业务目标，局部最优毁掉全局流程；
5. 调度复杂度爆炸：任务依赖、优先级、资源抢占，没有中心化调度治理完全玩不转。

算法圈只会聊协作逻辑，真正落地过生产的都清楚：多Agent最大的问题是分布式系统的老问题，只是套了一层大模型外壳。

### 8. Agent之间怎么做协调，你怎么看AtA这种思路

协调方案：采用中心化调度+去中心化执行的混合模式。顶层调度Agent作为总指挥，负责任务拆分、分工分配、进度管控、冲突拦截；业务执行Agent各司其职，独立完成原子子任务，只向上反馈结果，不跨角色乱调度；统一中央消息总线+全局任务状态中心，所有Agent共享标准状态，保证信息一致。

对AtA（Agent to Agent）看法：AtA本质是点对点自由通信，理念很理想，但生产落地局限性极大。优点：灵活性高、去中心化、适合轻量化探索场景；致命缺点：无管控、无熔断、无边界约束，复杂业务下极易出现对话爆炸、任务循环、信息冗余，完全不适合企业级稳定系统。结论：AtA适合Demo和学术探索，商用生产级多Agent，必须强中心化管控，自由协作就是埋雷。

### 9. 你怎么理解Skill，和Tool的区别是什么

本质是能力层级不同：
- Tool（工具）：是最小原子能力，纯执行层，比如查数据库、调接口、发短信、查报表，只有输入输出，没有决策能力；
- Skill（技能）：是封装后的业务能力，由多个Tool+规则+决策逻辑组合而成，是业务场景化的能力合集。
举例：查询订单接口是Tool；自动核对商户白名单并生成异常提醒是Skill，整合了订单查询、数据比对、规则校验、文案生成多个工具与逻辑。简单理解：Tool是零件，Skill是组装好的成品功能。

### 10. 你怎么做Agent的记忆

落地分层记忆体系，拒绝无脑拼接上下文，工业级标准设计：
1. 短时记忆：当前对话上下文，滑动窗口裁剪，自动过滤冗余内容，防止上下文溢出；
2. 中长期事实记忆：基于向量RAG存储用户/商户定制配置、业务规则、历史偏好，按需检索召回；
3. 结构化事务记忆：任务级状态存储，记录每一步执行记录、工具参数、错误日志，用于重试、回滚、溯源；
