### 1. 日常开发过程中都用过哪些AI的工具来提升效率的，哪些更好用些，MCP、Skill之类的？

真正能拉开和别人差距的，不是你用过多少工具，而是会用MCP和Rules/Skill：

**MCP（AI上下文连接器）：** 能把项目里的数据库、文档、其他工具，直接对接给AI。把数据库结构、项目文档、代码记录都对接给AI，它拿到这些信息后，写出来的代码直接就能用，不用反复修改纠正。本质就是让AI看懂项目，知道该怎么写代码。

**Cursor Rules（AI写代码的规矩）：** 在项目文件夹里建一个专门的文件，把项目的架构要求、命名规范（比如变量怎么命名）、不能做的事，都写进去。AI每次生成代码，都会遵守这些规矩，不会乱编。真正提高效率——不是一个人用AI快，而是整个团队写的代码都符合标准，不用互相改。

**Skill（AI任务模板）：** 把常用的工作，比如生成接口代码+测试代码+说明文档，做成一个模板，一键就能让AI完成，不用每次都跟AI说一堆话，相当于给AI定好固定流程。

大多数人用AI就是复制粘贴AI给的内容，根本没用到核心功能，提不了多少效率。

### 4. 讲一下如何设计提示词？有哪些设计原则？系统提示词和用户提示词有什么区别？能把系统提示词放到用户提示词中吗？

**提示词设计6大原则：**
1. 角色定义
2. 任务分解
3. 上下文注入
4. 输出格式约束
5. 负向约束
6. 少样本示例

**System Prompt vs User Prompt对比：**

| 维度 | System Prompt | User Prompt |
| --- | --- | --- |
| 简单理解 | 给AI定的"人设+规矩" | 用户实际问的问题 |
| 权重 | 高，优先级更高 | 低，受System约束 |
| 内容 | 角色定义、行为规范、输出格式 | 具体任务、问题、需求 |
| 持久性 | 整个会话持续生效 | 单次交互 |

**能把系统提示词放到用户提示词中吗？** 技术上可以但效果不一样，放入用户提示词后优先级变低，模型更容易"忘记"约束。

### 5. 讲一下项目中知识库检索中有哪些亮点？问题改写如何做？为什么使用三层递进检索？

**三层递进检索策略：** 原始问题→关键词→改写后问题

**三大亮点：**
1. **RRF多方式融合查找：** 多路召回结果融合排序
2. **Rerank精准筛选：** 精排后准确率提升20-30%
3. **Query Routing查询路由：** 根据问题类型智能选择检索策略

### 6. 怎么定义知识没检索到？（设置了一个召回率阈值）这个阈值如何设置？

影响准确率的核心因素：Embedding模型质量、Chunk切分策略、知识库数据质量、检索链路设计、Rerank模型选择。

召回率阈值设置需要结合业务场景：阈值过高会导致有效知识被误判为"未检索到"，阈值过低则噪声太多。通过离线评测集确定最优阈值，一般设置在0.7-0.85之间，同时结合相似度分数分布做动态调整。

### 7. 知识库检索准确率是多少？

- Top1准确率：94%+
- Top3准确率：98%+
- 无效召回率：<2%

### 8. 使用RAG过程觉得它有什么缺点？有哪些替代的技术？

**RAG的缺点（6点）：**
1. 检索质量依赖Embedding模型和Chunk策略
2. 长上下文信息可能被截断或分散
3. 多跳推理能力不足，复杂问题需要多次检索
4. 知识更新存在延迟
5. 对结构化知识（表格、图表）处理较弱
6. 检索噪声可能误导生成

**替代技术（6种）：**
1. **Fine-tuning（微调）：** 将知识直接融入模型参数
2. **Agent+工具调用：** 让模型主动调用工具获取信息
3. **KG+LLM（知识图谱+大模型）：** 结构化知识图谱辅助推理
4. **检索增强微调：** 结合检索与微调的优势
5. **长上下文模型：** 直接利用超长上下文窗口
6. **RAG+KG+Agent混合方案：** 多种技术组合的混合架构

### 9. CI/CD介绍一下？

**三个步骤：** 代码提交→CI持续集成→CD持续交付/部署

**CI持续集成包括：**
- ESLint/SonarQube代码质量检查
- Jest/Pytest单元测试
- 集成测试
- Docker镜像构建
- AI代码检查

**CD持续交付/部署包括：**
- 镜像仓库管理
- 测试环境部署
- 冒烟测试/E2E测试
- 灰度发布
- 生产环境部署
