### 你们系统里如果用了消息队列，怎么保证消息不丢失、不重复消费，并且实现最终一致性？
> 关联：[消息队列](../../消息队列/)

**答案：**
- 先保证生产者发送可靠，发送失败要重试，关键业务可以配合事务消息或本地消息表。
- Broker 这一层要做持久化和副本，避免机器故障导致消息丢失。
- 消费者要在业务处理成功后再 ack，不能先确认再执行业务。
- MQ 一般只能保证至少一次投递，所以消费者一定要做幂等。
- 幂等常见做法是业务唯一 ID、去重表、Redis 去重、数据库唯一索引。
- 最终一致性一般靠异步重试、死信队列、补偿机制来保证。

### 分布式事务有哪些常见方案？项目里一般怎么选？
> 关联：[微服务](../../微服务/)

**答案：**
- 常见方案有 2PC、TCC、事务消息、本地消息表、Saga。
- 2PC 强一致性比较强，但性能差、阻塞明显，互联网场景一般不常用。
- TCC 适合一致性要求高的场景，比如支付、账户，但开发成本比较高。
- 事务消息和本地消息表更常见，适合大部分互联网业务，走最终一致性。
- Saga 适合流程长、步骤多、需要补偿的业务。
- 如果是核心资金类业务，会偏向 TCC；如果是订单、积分、通知类，一般会选事务消息或本地消息表。

### 数据库出现主从延迟时，会带来哪些问题？一般怎么处理？
> 关联：[mysql](../../mysql/)

**答案：**
- 最典型的问题就是写后读不一致。刚下单，马上查订单列表可能查不到。
- 处理方式：关键查询直接走主库；写操作后的一小段时间内强制读主库；尽量走缓存。
- 如果主从延迟严重，还要排查慢 SQL、大事务、从库压力、复制线程、网络问题。

### 讲一下 InnoDB 的 MVCC 原理，它解决了什么问题？
> 关联：[mysql](../../mysql/)

**答案：**
- MVCC 是多版本并发控制，主要是为了解决读写冲突，提高并发性能。
- 核心有三个：隐藏字段、undo log、ReadView。
- 在 RC 下每次查询都会生成新的 ReadView，在 RR 下一个事务里通常复用同一个 ReadView。

### 为什么数据库索引底层通常使用 B+ 树，而不是红黑树或者哈希？
> 关联：[mysql](../../mysql/)

**答案：**
- 因为数据库是磁盘场景，最重要的是减少磁盘 IO。
- B+ 树是多叉树，树高低，查询时访问磁盘页的次数更少。
- 红黑树是二叉树，数据量大时树会更高，IO 次数更多。
- 哈希结构适合等值查询，但不适合范围查询和排序。

### Spring AOP 的底层原理是什么？JDK 动态代理和 CGLIB 有什么区别？
> 关联：[spring](../../spring/)

**答案：**
- Spring AOP 本质上就是动态代理，在目标对象外面生成一个代理对象，在方法前后做增强。
- JDK 动态代理是基于接口实现的，目标类必须实现接口。
- CGLIB 是基于继承实现的，不要求目标类有接口。
