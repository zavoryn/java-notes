# 微服务

##### Spring Cloud 5大组件有哪些？

**候选人**：

在早期，Spring Cloud的五大组件通常指的是：

- **Eureka**：服务注册中心。
- **Ribbon**：客户端负载均衡器。
- **Feign**：声明式的服务调用。
- **Hystrix**：服务熔断器。
- **Zuul/Gateway**：API网关。

随着Spring Cloud Alibaba的兴起，我们项目中也融入了一些阿里巴巴的技术组件：

- 服务注册与配置中心：Nacos。
- 负载均衡：Ribbon。
- 服务调用：Feign。
- 服务保护：Sentinel。
- API网关：Gateway。

**1.服务注册与发现 (Service Discovery)**

这是微服务架构的"通讯录"。

- **经典组件：Eureka**
  - **作用：** 服务提供者启动时向 Eureka Server 注册自己的 IP 和端口，消费者从 Eureka 获取服务列表，从而知道去哪里调用服务。它是基于 AP（可用性、分区容错性）原则设计的。
- **当前主流：Nacos** (Spring Cloud Alibaba)
  - **补充：** Nacos 不仅支持服务注册发现，还集成了配置中心功能，且支持 AP 和 CP 模式切换，功能更强大。

**2.客户端负载均衡 (Client-Side Load Balancing)**

解决了"从多个服务实例中选哪一个"的问题。

- **经典组件：Ribbon**
  - **作用：** 它是一个客户端负载均衡器，通常配合 RestTemplate 使用。它提供了轮询、随机、加权等多种算法，在客户端发起请求时直接选择目标服务器。
- **当前主流：Spring Cloud LoadBalancer**
  - **补充：** 由于 Ribbon 停止更新，Spring 官方推出了自己的 LoadBalancer 作为替代。

**3.服务熔断与降级 (Circuit Breaker)**

解决了"雪崩效应"的问题，保护系统稳定性。

- **经典组件：Hystrix**
  - **作用：** 当某个服务调用失败率达到阈值（如网络延迟、异常）时，Hystrix 会通过断路器阻断该服务的调用，直接返回兜底数据（降级），防止故障蔓延导致整个系统瘫痪。
- **当前主流：Sentinel** (Alibaba) 或 **Resilience4j**
  - **补充：** Hystrix 已不再开发新功能。Sentinel 提供了更完善的流量控制、熔断降级和系统负载保护，且有可视化的控制台，运维更加方便。

**4.API 网关 (API Gateway)**

这是系统的"统一入口"和"保安"。

- **经典组件：Zuul (1.x)**
  - **作用：** 负责请求路由转发、权限校验（鉴权）、限流等。Zuul 1.x 基于 Servlet 阻塞 I/O 模型。
- **当前主流：Spring Cloud Gateway**
  - **补充：** Spring 官方推出的第二代网关，基于 Spring WebFlux (Project Reactor) 和 Netty，是**非阻塞异步**模型，性能比 Zuul 1.x 强很多，支持长连接（WebSocket）。

**5.分布式配置中心 (Distributed Configuration)**

解决了"配置文件分散、难以管理"的问题。

- **经典组件：Spring Cloud Config**
  - **作用：** 将各个微服务的配置文件集中存储（通常放在 Git 仓库中），支持动态刷新配置（需配合 Bus 消息总线）。
- **当前主流：Nacos Config**
  - **补充：** Nacos 既是注册中心也是配置中心，配置修改后可以说秒级推送到客户端，架构复杂度比 "Config + Bus" 低很多。

---

总结与实际应用感悟

> **面试官，我在实际项目中是这样理解这套体系的：**

如果把微服务架构比作一家大公司：

1. **Eureka/Nacos** 是人力资源部（花名册），知道每个人在哪。
2. **Ribbon/LoadBalancer** 是项目经理，决定把任务派给哪个具体的员工（实例）。
3. **Hystrix/Sentinel** 是保险丝，当某个部门由于工作量过大"挂了"，它能保证不影响整个公司的运作，只针对该部门做降级处理。
4. **Zuul/Gateway** 是公司前台门禁，所有外部请求必须先经过它，验证身份后才能进入。
5. **Config/Nacos** 是公司规章制度发布栏，统一管理所有规则，实时更新。

在目前的新项目中，我更倾向于推荐使用 **Spring Cloud Alibaba (Nacos + Sentinel) + Spring Cloud Gateway** 的组合，这套组合在国内生态下社区活跃度更高，性能和易用性也更好。

##### 服务注册和发现是什么意思？Spring Cloud 如何实现服务注册发现？

**候选人**：

服务注册与发现主要包含三个核心功能：服务注册、服务发现和服务状态监控。

我们项目中采用了Eureka作为服务注册中心，它是Spring Cloud体系中的一个关键组件。

- **服务注册**：服务提供者将自己的信息（如服务名称、IP、端口等）注册到Eureka。
- **服务发现**：消费者从Eureka获取服务列表信息，并利用负载均衡算法选择一个服务进行调用。
- **服务监控**：服务提供者定期向Eureka发送心跳以报告健康状态；如果Eureka在一定时间内未接收到心跳，将服务实例从注册中心剔除。

**什么是服务注册与发现？**

在单体应用时代，我们调用服务通常是写死 IP 和端口（比如 `http://192.168.1.10:8080/api`）。但在微服务时代，服务实例是动态的：它们随时可能扩容、缩容、宕机或迁移，IP 地址是不固定的。

**服务注册与发现**就是为了解决**"动态获取服务地址"**的问题。我们可以把它类比为**"滴滴打车"的模式**：

- **服务注册（Registration）：**
  - **类比：** 司机出车时，打开 App 上线，系统记录下他的车牌号和当前位置。
  - **技术含义：** 服务提供者（Provider）在启动时，向"注册中心"汇报自己的 IP、端口、服务名等信息。
- **服务发现（Discovery）：**
  - **类比：** 乘客下单时，系统根据位置匹配附近的司机，并把司机信息发给乘客。
  - **技术含义：** 服务消费者（Consumer）向"注册中心"查询目标服务的可用列表，然后根据负载均衡算法选择一个实例进行调用。

---

**Spring Cloud 如何实现服务注册发现？**

Spring Cloud 为此提供了一套统一的抽象接口（标准），即 `Spring Cloud Commons` 中的 **`DiscoveryClient`** 接口。这使得我们可以无缝切换底层的实现（如 Eureka, Nacos, Consul, Zookeeper）。

具体的实现流程，通常包含以下 **4 个关键步骤**：

第一步：搭建注册中心 (Server)

我们需要一个服务端来存储"通讯录"。

- **做法：** 如果是用 **Eureka**，我们需要创建一个 Spring Boot 项目，引入 `spring-cloud-starter-netflix-eureka-server` 依赖，并在启动类上加 `@EnableEurekaServer` 注解。如果是 **Nacos**，则直接启动 Nacos Server 控制台即可。

第二步：服务注册 (Client - Provider)

服务提供者启动时，自动"上报"自己。

- **做法：** 引入对应的 Client 依赖（如 `spring-cloud-starter-netflix-eureka-client` 或 `spring-cloud-starter-alibaba-nacos-discovery`）。
- **配置：** 在 `application.yml` 中配置注册中心的地址。

```yaml
spring:
  application:
    name: order-service  # 这就是它在通讯录里的名字
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848
```

- **原理：** 应用启动时，客户端 SDK 会通过 HTTP 或 TCP 发送请求给注册中心。此后，还会定期发送**心跳（Heartbeat）**（Eureka 默认 30秒），告诉注册中心"我还活着"。如果不发心跳，注册中心会在一定时间后将其"剔除"。

第三步：服务发现 (Client - Consumer)

消费者需要"拉取"通讯录。

- **原理：** 消费者（如 OrderService 调用 UserService）启动时，会从注册中心全量拉取一份服务列表，缓存到本地（JVM 内存）。
- **增量更新：** 之后，消费者会定期（Eureka 默认 30秒）从注册中心拉取**增量更新**，保持本地缓存是最新的。
- **核心优势：** 这种**客户端侧发现（Client-side Discovery）**机制有一个巨大的好处：即使注册中心挂了，消费者依然可以使用本地缓存的地址列表继续工作，保证了高可用。

第四步：负载均衡调用

拿到一堆地址后，选哪一个？

- **实现：** Spring Cloud 自动整合了 **LoadBalancer**（老版本是 Ribbon）。
- **用法：** 当我们使用 `RestTemplate` 并加上 `@LoadBalanced` 注解，或者使用 `Feign` 时，框架会自动拦截请求，从本地的服务列表中，通过**轮询**或**随机**算法选出一个 IP 进行调用。

---

深入一点：实际开发中的注意点

在我的过往项目中，关于服务注册发现，有两点经验特别重要：

1. **AP 与 CP 的选择：**
   - **Eureka** 是典型的 **AP（高可用）** 设计，它优先保证注册中心能访问，即使节点间数据暂时不一致也没关系（服务注册信息稍微延迟一点同步是可以接受的）。
   - **Zookeeper** 是 **CP（强一致性）** 设计，如果 ZK 集群正在选主，可能会导致服务注册不可用。
   - **Nacos** 则支持在 AP 和 CP 之间切换，默认是 AP，这更符合微服务注册中心的实际需求。
2. **自我保护机制：**
   - 以前用 Eureka 时，常遇到网络波动导致大量服务报红（被标记为 Down），触发了**自我保护模式**（不再剔除服务）。这时候不能盲目重启，而要排查网络状况。Nacos 也有类似的阈值保护机制。

---

总结

> **面试官，简单总结一下：** Spring Cloud 通过 **`DiscoveryClient`** 接口定义标准。 服务启动时**自动注册**并维持**心跳**；消费者**拉取并缓存**服务列表；最后配合**负载均衡器**选址调用。 这种机制解耦了服务间的物理依赖，是微服务弹性伸缩的基石。

##### 我看你之前也用过nacos，你能说下nacos与eureka的区别？

**候选人**：

在使用Nacos作为注册中心的项目中，我注意到Nacos与Eureka的共同点和区别：

- **共同点**：两者都支持服务注册与发现，以及心跳检测作为健康检查机制。
- **区别**：
  1. Nacos支持服务端主动检测服务提供者状态，而Eureka依赖客户端心跳。
  2. Nacos区分临时实例和非临时实例，采用不同的健康检查策略。（临时心跳、非临时主动）
  3. Nacos支持服务列表变更的消息推送，使服务更新更及时。
  4. Nacos集群默认采用AP模式，但在存在非临时实例时，会采用CP模式；而Eureka始终采用AP模式。

##### 你们项目负载均衡如何实现的？

**候选人**：

在服务调用过程中，我们使用Spring Cloud的Ribbon组件来实现客户端负载均衡。Feign客户端在底层已经集成了Ribbon，使得使用非常简便。

当发起远程调用时，Ribbon首先从注册中心获取服务地址列表，然后根据预设的路由策略选择一个服务实例进行调用，常用的策略是轮询。

我们通常称之为**"服务端负载均衡"**和**"客户端负载均衡"**的结合。

具体的实现细节如下：

第一层：服务端负载均衡 (Nginx)

这是所有外部请求进入系统的"第一道关卡"。

- **位置：** 位于客户端（浏览器/App）和我们的后端网关（Spring Cloud Gateway）之间。
- **实现方式：** 我们使用 **Nginx** 作为反向代理。
- **工作原理：** 客户端发送请求到 Nginx 的 IP，Nginx 内部配置了 `upstream` 模块，维护了后端网关集群的 IP 列表。Nginx 按照配置的算法（通常是轮询或加权轮询）将请求转发给某一个网关实例。
- **特点：** 客户端不知道具体处理请求的服务器是谁，由 Nginx 统一分发。

第二层：客户端负载均衡 (Spring Cloud)

这是微服务内部调用（如订单服务调用库存服务）时的负载均衡机制。

- **位置：** 位于服务消费者（Consumer）一侧。
- **技术栈：**
  - 老项目使用 **Ribbon**。
  - 新项目（Spring Cloud 2020+ 版本）使用 **Spring Cloud LoadBalancer**。
  - 配合 **OpenFeign** 进行远程调用。
- **工作原理：**
  1. **拉取列表：** 消费者（比如订单服务）从注册中心（Nacos）拉取库存服务的所有健康实例列表，并缓存到本地。
  2. **本地选择：** 当订单服务要发起调用时，**它自己**根据负载均衡算法（如轮询），从本地列表中选出一个 IP 地址。
  3. **直连调用：** 选定 IP 后，直接发起 HTTP 请求，不再经过 Nginx 这种中间件。
- **特点：** 去中心化，性能更好，没有单点瓶颈。

我们使用的负载均衡策略 (算法)

在实际配置中，我们主要使用了以下几种策略：

- **默认策略：轮询 (Round Robin)**
  - **场景：** 绝大多数场景下，因为我们的服务器配置（CPU/内存）都是标准化的容器，性能一致，所以轮询是最简单有效的。
- **Nacos 权重策略 (Weighted)**
  - **场景：** 这是我们在使用 Spring Cloud Alibaba 时特别用到的。
  - **痛点：** 有时候我们新上线了一批服务器，配置比老服务器好；或者某台服务器有点"不稳定"，我们想减少它的流量。
  - **实现：** 我们在 Nacos 控制台上给不同的实例设置 `weight`（权重）。我们在代码中配置了基于 Nacos 权重的负载均衡策略（扩展了 `AbstractLoadBalancerRule`），让权重高的机器处理更多请求。
- **同集群优先 (Zone Avoidance / Cluster Preference)**
  - **场景：** 我们的服务部署在多个机房（比如北京机房、上海机房）。
  - **实现：** 配置 Nacos 的集群保护模式。如果订单服务在北京机房，它会优先调用同在北京机房的库存服务，避免跨地域的网络延迟。只有当北京机房的库存服务全挂了，才会去调用上海的。

---

结合实战的补充

> **面试官，这里我想补充一个我们在生产环境中遇到的坑：**

刚开始我们使用默认的 Ribbon **轮询策略**时，遇到过"慢请求堆积"的问题。某台服务器因为网络抖动处理变慢，但轮询策略依然傻傻地把请求发给它，导致这台机器彻底卡死（雪崩）。

后来我们针对关键核心链路，调研了 **"最小连接数" (Least Connections)** 策略（或者 Nacos 的 Sentinel 权重自适应），优先把请求转发给当前连接数最少的节点，这样能动态地避开性能变差的节点，提升了系统的整体稳定性。

---

总结

简单来说，我们的负载均衡是：**外网入口用 Nginx（服务端LB），内网微服务互调用 OpenFeign + LoadBalancer（客户端LB），策略以轮询为主，配合 Nacos 权重和同机房优先策略进行优化。**

##### Ribbon负载均衡策略有哪些？

**候选人**：

Ribbon提供了多种负载均衡策略，包括：

- **RoundRobinRule**：简单的轮询策略。
- **WeightedResponseTimeRule**：根据响应时间加权选择服务器。
- **RandomRule**：随机选择服务器。
- **ZoneAvoidanceRule**：区域感知的负载均衡，优先选择同一区域中可用的服务器。

##### 如果想自定义负载均衡策略如何实现？

**候选人**：

自定义Ribbon负载均衡策略有两种方式：

1. 创建一个类实现IRule接口，这将定义全局的负载均衡策略。
2. 在客户端配置文件中指定特定服务调用的负载均衡策略，这将仅对该服务生效。

在实际生产中，默认的轮询（Round Robin）或随机（Random）策略往往无法满足复杂的业务需求。我们通常在以下场景需要**自定义负载均衡策略**：

1. **灰度发布（金丝雀发布）：** 让特定用户（如 ID 为 100-200 的用户）访问新版本的服务实例。
2. **多版本共存：** 开发环境不想和测试环境混用，需要根据请求头（Header）路由到指定的开发机。
3. **同集群/同地域优先：** 减少跨机房的网络延迟。

针对目前主流的两种技术栈（老一代 Ribbon 和 新一代 Spring Cloud LoadBalancer），实现方式略有不同。我将分两部分来回答。

---

方案一：基于 Spring Cloud LoadBalancer (当前主流)

由于 Ribbon 已经进入维护模式，现在的 Spring Boot 2.4+ / Spring Cloud 2020+ 版本默认都使用 **Spring Cloud LoadBalancer**。

实现自定义策略的核心在于实现 **`ReactorServiceInstanceLoadBalancer`** 接口。

核心步骤

我们需要自定义一个类，去实现那个接口，重写 `choose` 方法。

```java
// 伪代码示例：只选择"版本号为 v2"的实例
public class GrayLoadBalancer implements ReactorServiceInstanceLoadBalancer {

    private final ServiceInstanceListSupplier serviceInstanceListSupplier;
    private final String serviceId;

    public GrayLoadBalancer(ObjectProvider<ServiceInstanceListSupplier> supplier, String serviceId) {
        this.serviceInstanceListSupplier = supplier.getIfAvailable();
        this.serviceId = serviceId;
    }

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        return serviceInstanceListSupplier.get(request).next()
            .map(instanceList -> {
                ServiceInstance chosen = filterInstanceByHeader(instanceList, request);
                return new DefaultResponse(chosen);
            });
    }
}
```

启用配置

```java
@LoadBalancerClient(name = "order-service", configuration = GrayRuleConfig.class)
public class WebConfig { ... }
```

方案二：基于 Ribbon (经典老项目)

如果维护的是老项目，核心接口是 **`IRule`**。

```java
public class MyGrayRule extends AbstractLoadBalancerRule {

```

面试加分项：结合 Nacos 做全链路灰度

我们实现了一个**"基于标签（Tag）的灰度路由"**。

1. **服务打标：** 在 Nacos 配置中，给新上线的 V2 版本实例配置元数据（Metadata）：`version: v2`。
2. **流量染色：** 网关（Gateway）识别到特定用户（如 Header 携带 `x-gray-user: true`），会将请求头透传给后续微服务。
3. **自定义策略：**
   - 我们的自定义负载均衡器会先检查 Request Header。
   - 如果有 `version=v2` 的标记，就遍历服务列表，只选择 Nacos Metadata 中包含 `version=v2` 的实例。
   - 如果没有对应的实例，再降级回退到 V1 版本，保证服务可用性。

##### 什么是服务雪崩，怎么解决这个问题？

**候选人**：

服务雪崩是指一个服务的失败导致整个链路的服务相继失败。我们通常通过服务降级和服务熔断来解决这个问题：

- **服务降级（预防）**：在请求量突增时，主动降低服务的级别，确保核心服务可用。（两条路可以走，正常返回，异常报网络异常）
- **服务熔断（解决）**：当服务调用失败率达到一定阈值时，熔断机制会启动，防止系统过载，每隔5秒重新尝试请求服务，没响应，继续熔断，如果微服务可达，关闭熔断机制，恢复正常请求。

我们项目早期使用的是 Spring Cloud Netflix 的 **Hystrix**，主要依赖**线程池隔离**。但线程池上下文切换会有一定的性能损耗。

后来升级到 Spring Cloud Alibaba 后，我们全面切换到了 **Sentinel**。

- **优势：** Sentinel 默认使用**信号量隔离**，性能更好。
- **可视化：** 我们在 Sentinel 控制台上配置了"慢调用比例"熔断规则。比如，当 SQL 查询超过 500ms 的请求比例达到 30% 时，自动触发熔断，非常灵活。

##### 你们项目中有没有做过限流？怎么做的？

在我的过往项目中，我们根据业务场景的不同，采用了**多级限流**的策略。我通常将限流分为**网关层限流**和**应用层限流**，针对特定场景还会使用**分布式限流**。

网关层限流 (Nginx / Spring Cloud Gateway)

- **Nginx 限流：**
  - 使用 `limit_req_zone` 模块，配置基于 IP 的限流。
  - **算法：** Nginx 默认使用的是**漏桶算法 (Leaky Bucket)**。强行削峰，出去的速度是恒定的。
- **Spring Cloud Gateway 限流：**
  - 集成了 Redis，使用 `RequestRateLimiter` 过滤器。
  - **算法：** **令牌桶算法 (Token Bucket)**。允许一定的突发流量。
  - **维度：** 根据 API 路径（URI）或者 用户 ID 进行限流。

应用层限流 (Sentinel —— 核心方案)

1. **QPS 直接限流：** 比如"创建订单"接口，单机极限是 500 QPS，配置阈值为 400。
2. **热点参数限流 (Hot Param)：** 对 `productId` 进行限流，只限制特定商品的请求。
3. **流控效果 - 排队等待 (Warm Up)：** 防止冷启动时流量瞬间打死服务。

自定义分布式限流 (Redis + Lua)

- 利用 Redis 的 `INCR` 和 `EXPIRE` 命令，编写 **Lua 脚本** 保证原子性。

被限流的请求怎么处理？

1. **对于前端用户：** 返回 HTTP 429 (Too Many Requests) 状态码。
2. **对于内部调用：** 触发 Feign 的降级逻辑（Fallback）。

**候选人**：

并发量大，防止用户恶意刷接口

- **版本1**：使用**Nginx**进行限流，通过漏桶算法控制请求处理速率，按照IP进行限流。
- **版本2**：网关限流：使用**Spring Cloud Gateway**的**RequestRateLimiter**过滤器进行限流，采用令牌桶算法。

如果是为了**平滑流量**，防止后端数据库被瞬间压垮，选**漏桶算法**（Nginx）。

如果是为了**应对突发流量**，同时又想限制平均速率，选**令牌桶算法**（Gateway/Guava）。

在实际微服务开发中，**Sentinel** 的滑动时间窗口算法结合了多种优势，且支持动态调整，是**首选方案**。

##### 限流常见的算法有哪些？

**候选人**：

常见的限流算法包括：

- **漏桶算法**：以固定速率处理请求，平滑突发流量。
- **令牌桶算法**：按照一定速率生成令牌，请求在获得令牌后才被处理，适用于请求量有波动的场景。

##### 什么是CAP理论？

**候选人**：

CAP理论是分布式系统设计的基础理论，包含一致性(Consistency)、可用性(Availability)和分区容错性(Partition tolerance)。在网络分区发生时，系统只能在一致性和可用性之间选择其一。

##### 为什么分布式系统中无法同时保证一致性和可用性？

**候选人**：

在分布式系统中，为了保证分区容错性，我们通常需要在一致性和可用性之间做出选择。如果系统优先保证一致性，可能需要牺牲可用性，反之亦然。

##### 什么是BASE理论？

**候选人**：

BASE理论是分布式系统设计中对CAP理论中AP方案的延伸，强调通过基本可用、软状态和最终一致性来实现系统设计。

##### 分布式服务的接口幂等性如何设计？

幂等：多次调用接口和方法不会改变业务状态，可以保证重复调用和单次调用结果一致

**候选人**：

我们通过Token和**Redis**来实现接口幂等性。用户操作时，系统生成一个Token并存储在Redis中，当用户提交操作时，系统会验证Token的存在性，并在验证通过后删除Token，确保每个Token只被处理一次。

分布式锁，性能较低

针对不同的业务场景，我会采用不同的策略，通常是组合使用：

方案一：数据库唯一索引 (Unique Index) —— 最强防线

适用于**插入类（Insert）**操作。

- **原理：** 在数据库表中建立唯一索引（比如 `order_id` 或者 `biz_type` + `biz_id`）。
- **关键点：** 捕获 `DuplicateKeyException` 后，**直接返回"成功"**。

方案二：Token 机制 (Redis) —— 防止前端重复提交

- **流程：**
  1. **获取 Token：** 客户端进入提交页面前，先调用后端获取一个全局唯一的 Token。
  2. **提交请求：** 客户端发起业务请求时，把 Token 带在 Header 或参数里。
  3. **校验与删除：** 使用 **Lua 脚本** 保证 `Get` 和 `Del` 的原子性。

方案三：乐观锁 (Optimistic Locking) —— 适用于更新操作

适用于**更新类（Update）**操作，比如库存扣减、余额更新。

```sql
UPDATE account
SET balance = balance - 100, version = version + 1
WHERE id = 123 AND version = 5;
```

方案四：状态机幂等 (State Machine) —— 业务逻辑控制

适用于有明确流转状态的业务，比如订单状态。

```sql
UPDATE order_table
SET status = 'PAID'
WHERE id = 1001 AND status = 'PENDING';
```

---

实际开发中的"坑"与经验

1. **"一锁、二判、三更新"** 在复杂的复合操作中，结合 **分布式锁（Redisson）**。
2. **幂等结果的返回** - 识别出重复请求时，**不能抛异常**，应直接返回 **Success**。

---

总结

我的设计原则是：**多层防护**。

- **最外层：** Token 机制挡住 90% 的无效点击。
- **中间层：** 分布式锁控制并发。
- **最底层：** 数据库唯一索引和乐观锁做最终兜底，保证数据绝对的一致性。

##### xxl-job路由策略有哪些？（分布式任务调度）

**候选人**：

xxl-job支持多种路由策略，包括轮询、故障转移和分片广播等。

我们选型了 **XXL-JOB** 作为核心的分布式任务调度平台。选择原因：

1. **轻量级与易用性：** 学习成本极低，开箱即用，拥有可视化管理后台。
2. **去中心化设计：** 调度中心和执行器分离，支持集群部署。
3. **社区活跃：** 国内应用广泛，文档齐全。

核心场景一：海量数据处理（分片广播）

- **解决方案：** 使用 **"分片广播" (Sharding Broadcast)** 路由策略。
- 每台机器通过 `XxlJobHelper.getShardIndex()` 获取分片序号和总数，SQL 加取模条件分配数据。

```sql
SELECT * FROM order_table
WHERE mod(id, #{shardingTotal}) = #{shardingIndex}
AND status = 'PENDING';
```

核心场景二：故障转移与容错

- **路由策略：** 配置为 **"故障转移"**。
- **失败重试：** 配置重试次数，配合邮件/钉钉报警。

---

面试加分项：遇到的坑与思考

1. **幂等性问题：** 业务代码层面利用数据库唯一索引或 Redis 分布式锁保证任务幂等性。
2. **长耗时任务处理：** 通过 XXL-JOB 触发"开始信号"，将任务丢入 MQ 或线程池异步处理。
3. **版本兼容性：** 确保 Admin 和 Executor 版本严格一致。

##### xxl-job任务执行失败怎么解决？

**候选人**：

1. 选择故障转移路由策略，优先使用健康的实例执行任务。
2. 设置任务重试次数。
3. 通过日志记录和邮件告警通知相关负责人。

**候选人**：
