# 从 URL 到页面

> [!tip] 模块速记
> **全链路 = DNS解析 → TCP握手 → TLS握手 → HTTP请求 → 服务端处理 → 浏览器渲染(DOM→CSSOM→Render→Layout→Paint→Composite) → 连接关闭。渲染优化关键：减少重排，利用GPU合成层，缩短关键渲染路径。**

---

## 一、从输入 URL 到页面展示——完整链路

这是面试的**经典压轴题**，考察对网络全链路的理解深度。

### 1.1 全流程拆解

```
用户在浏览器输入 https://www.example.com
                    │
    ┌───────────────┴────────────────┐
    │  1. URL 解析                    │
    │     解析协议(https)、域名、端口、路径
    └───────────────┬────────────────┘
                    ▼
    ┌───────────────┴────────────────┐
    │  2. DNS 解析                    │
    │     浏览器缓存 → hosts → LDNS → 根→TLD→权威
    │     获得 IP 地址：93.184.216.34
    └───────────────┬────────────────┘
                    ▼
    ┌───────────────┴────────────────┐
    │  3. TCP 三次握手                │
    │     SYN → SYN+ACK → ACK
    │     源端口随机，目标端口 443
    └───────────────┬────────────────┘
                    ▼
    ┌───────────────┴────────────────┐
    │  4. TLS 握手                    │
    │     证书验证 → 密钥协商 → 加密通信
    │     (HTTPS 专有，HTTP 跳过)
    └───────────────┬────────────────┘
                    ▼
    ┌───────────────┴────────────────┐
    │  5. 发送 HTTP 请求              │
    │     GET /index.html HTTP/1.1
    │     Host: www.example.com
    └───────────────┬────────────────┘
                    ▼
    ┌───────────────┴────────────────┐
    │  6. 服务端处理（可能经过 Nginx  │
    │     反向代理 → 应用服务器 → DB）│
    └───────────────┬────────────────┘
                    ▼
    ┌───────────────┴────────────────┐
    │  7. 服务端返回 HTTP 响应        │
    │     HTTP/1.1 200 OK
    │     Content-Type: text/html
    └───────────────┬────────────────┘
                    ▼
    ┌───────────────┴────────────────┐
    │  8. 浏览器解析渲染              │
    │     HTML解析 → CSS计算 → 布局   │
    │     → 绘制 → 合成               │
    └───────────────┬────────────────┘
                    ▼
    ┌───────────────┴────────────────┐
    │  9. 四次挥手（连接关闭或复用）   │
    │     长连接模式下此步延后
    └────────────────────────────────┘
```

### 1.2 面试回答模板（核心话术）

> **面试官：从输入 URL 到页面展示，发生了什么？**

"我分几个阶段来回答：

**第一阶段 —— DNS 解析**：浏览器先检查缓存（浏览器缓存 → hosts 文件），没有的话向本地 DNS 服务器发起递归查询。本地 DNS 服务器通过迭代查询——从根域名服务器 → .com 顶级域名服务器 → 权威域名服务器——最终拿到 IP 地址。

**第二阶段 —— 建立连接**：拿到 IP 后，TCP 三次握手建立连接——SYN → SYN+ACK → ACK。因为是 HTTPS，紧接着进行 TLS 握手——验证证书、协商加密套件、生成会话密钥。TLS 1.2 需要 2 个 RTT，TLS 1.3 只需要 1 个 RTT。

**第三阶段 —— 发送请求**：连接建立后，浏览器构造 HTTP 请求报文（请求行、请求头、请求体），通过已建立的 TCP 连接发送。中间可能经过 Nginx 反向代理、负载均衡器，最终到达应用服务器。

**第四阶段 —— 服务端处理**：服务端解析请求，执行业务逻辑（可能查数据库、调微服务），构造 HTTP 响应返回。

**第五阶段 —— 浏览器渲染**：浏览器收到 HTML 后，解析 HTML 构建 DOM 树、解析 CSS 构建 CSSOM 树、合并成 Render 树、布局计算、绘制、合成层显示。遇到外部资源（CSS/JS/图片）会发起额外请求。

**第六阶段 —— 连接关闭或复用**：HTTP/1.1 默认长连接，连接会保持一段时间复用。如果用完了或超时了，四次挥手关闭连接。"

> [!note] 功能对位口诀
> ```
> DNS     → 域名变IP的翻译官
> TCP     → 管送达的邮政局长
> TLS     → 管加密的保险箱
> HTTP    → 管说话格式的礼仪师
> Nginx   → 管分流的交通警察
> CDN     → 开在各地的连锁仓库
> ```

## 二、浏览器渲染详细流程

### 2.1 渲染六步

```
HTML文本
  │
  ▼
① 解析HTML → DOM树
  │         │
  │         ▼
  │    解析CSS → CSSOM树
  │         │
  ▼         ▼
② DOM + CSSOM → Render树（只包含可见元素）
  │
  ▼
③ Layout（布局/回流）：计算每个节点的位置和大小
  │
  ▼
④ Paint（绘制）：将节点绘制到屏幕上
  │
  ▼
⑤ Composite（合成）：将不同层合成最终画面（GPU加速）
  │
  ▼
⑥ 显示到屏幕
```

### 2.2 关键概念

**DOM 树**：HTML 解析为 DOM 节点树，表示文档结构。

**CSSOM 树**：CSS 解析为 CSSOM，表示样式规则。

**Render 树（渲染树）**：DOM + CSSOM 合并，只包含**需要显示的元素**（`display:none` 的元素不会出现在渲染树中，`visibility:hidden` 的会）。

**Layout（布局）**：计算每个可见节点的精确位置和大小——这是"几何计算"阶段。

**Paint（绘制）**：将节点绘制成像素——这是"填色"阶段。

**Composite（合成）**：多个图层合并成最终画面——GPU 负责，独立图层可以独立合成避免全量重绘。

## 三、重排（Reflow）vs 重绘（Repaint）

### 3.1 定义

| 操作                | 定义                 | 代价              | 触发条件                                                                  |
| ----------------- | ------------------ | --------------- | --------------------------------------------------------------------- |
| **重排/回流（Reflow）** | 重新计算元素的几何属性（位置、大小） | **高**（影响周围所有元素） | 改变布局属性：width/height/margin/padding/display/font-size/position/添加/删除元素 |
| **重绘（Repaint）**   | 重新绘制元素外观（不影响布局）    | **中**           | 改变外观属性：color/background/border-color/visibility/outline               |

> [!warning] 重排一定触发重绘，重绘不一定触发重排

### 3.2 如何减少重排

| 优化策略 | 说明 |
|----------|------|
| **批量修改样式** | 一次改 class，不要逐条改 style |
| **离线DOM操作** | 先 `display:none` → 批量修改 → `display:block`（只触发2次重排） |
| **使用 `transform` 替代 `top/left`** | transform 不触发布局计算，直接在合成层处理 |
| **避免逐条读取布局属性** | 读取 offsetWidth 等会强制同步布局，先批量读再批量写 |
| **使用 `DocumentFragment`** | 批量插入节点时用片段容器 |
| **CSS 动画用 `will-change`** | 提示浏览器提前创建独立图层 |

> [!tip] 面试加分
> `transform` 和 `opacity` 变化只触发 Composite（合成），不触发 Layout 和 Paint——这是 CSS 动画性能优化的核心原理。

## 四、关键渲染路径优化

### 4.1 什么是关键渲染路径？

从收到 HTML 到首次渲染出有意义内容的路径——需要尽快完成，让用户看到页面。

### 4.2 阻塞因素

| 阻塞类型 | 说明 | 优化 |
|----------|------|------|
| **CSS 阻塞渲染** | CSS 未加载完成，渲染树无法构建 | 关键CSS内联，非关键CSS异步加载 |
| **JS 阻塞解析** | JS 可能修改 DOM/CSSOM，解析器暂停 | `async`/`defer` 属性，关键JS前置 |
| **字体阻塞渲染** | 自定义字体未加载，文字可能闪烁 | `font-display: swap` |

### 4.3 优化策略

```
关键渲染路径优化思路：

1. 减少关键资源数量 → 内联关键CSS/JS，异步加载非关键资源
2. 减少关键资源大小 → 压缩、Tree-shaking、代码分割
3. 缩短关键路径长度 → 减少往返次数（CDN/预连接/缓存）
```

> [!note] 性能优化串联
> ```
> DNS 慢？    → DNS 预解析（<link rel="dns-prefetch">）+ CDN 加速
> TCP 慢？    → 升级 HTTP/2.0 多路复用 + TCP 调优参数
> TLS 慢？    → 升级 TLS 1.3 + Session 复用
> HTTP 慢？   → HTTP/2.0 + 头部压缩 + Gzip/Brotli + Keep-Alive
> 传输慢？    → CDN 就近访问 + 减小资源体积
> 服务端慢？  → Nginx 缓存 + 连接池 + 数据库优化
> 渲染慢？    → 关键CSS内联 + JS async/defer + 减少重排
> ```

## 五、面试回答模板

> **面试官：从输入 URL 到页面展示，发生了什么？**

"我分几个阶段来回答：DNS解析拿到IP → TCP三次握手建立连接 → TLS握手协商加密（HTTPS场景） → 发HTTP请求 → 服务端处理并返回响应 → 浏览器构建DOM树和CSSOM树合并成渲染树 → 布局计算 → 绘制 → 合成显示。长连接模式下连接复用，否则四次挥手关闭。

这个流程串起来就是整个 Web 技术栈的全链路地图——每个环节都有优化空间。"

> **面试官：浏览器渲染流程？什么是重排和重绘？**

"浏览器收到 HTML 后：解析 HTML 构建 DOM 树，解析 CSS 构建 CSSOM 树，两者合并成渲染树（只含可见元素），然后 Layout 计算几何属性，Paint 绘制像素，Composite 合成各层输出。

**重排**是重新计算几何属性（如改变 width/height），代价高因为影响周围元素。**重绘**是重新绘制外观（如改变 color/background），代价中等。**重排一定触发重绘，重绘不一定触发重排**。

优化核心是用 `transform`/`opacity` 替代 `top/left`（只触发合成层操作），批量修改样式避免逐条触发重排。"
