# DNS 与 CDN

> [!tip] 模块速记
> **DNS = 域名翻译官（递归+迭代），CDN = 全球连锁仓库（边缘缓存+就近访问），反向代理 = 服务端门面（负载均衡/SSL终止/安全防护）。DNS 用 UDP 端口 53，CDN 靠 CNAME 调度到最近节点。**

---

## 一、DNS 解析流程

### 1.1 完整解析过程

```
用户输入 www.example.com
        │
        ▼
① 浏览器缓存（Chrome: chrome://net-internals/#dns）
        │ 未命中
        ▼
② 操作系统缓存（/etc/hosts，Windows: C:\Windows\System32\drivers\etc\hosts）
        │ 未命中
        ▼
③ 本地 DNS 服务器（LDNS，递归查询开始）
        │
        ▼
④ 根域名服务器（.）
        │ 返回 .com 顶级域名服务器地址
        ▼
⑤ .com 顶级域名服务器（TLD）
        │ 返回 example.com 权威服务器地址
        ▼
⑥ example.com 权威域名服务器
        │ 返回 www.example.com 的 IP 地址
        ▼
⑦ 本地 DNS 服务器缓存结果，返回给客户端
```

### 1.2 递归查询 vs 迭代查询

```
递归查询（客户端 → 本地DNS）：
  "帮我查，查不到你别回来" —— 本地DNS 替客户端跑完全程

迭代查询（本地DNS → 各级DNS服务器）：
  "我不认识，但你可以去问 XXX" —— 只给线索，自己继续问
```

**完整比喻**：

> 你要找张三（www.example.com）：
> - 你先问小区物业（本地 DNS）："帮我找张三"（**递归查询**）
> - 物业问市档案馆（根 DNS）："张三在哪？"
> - 市档案馆说："不知道，但你可以去 XX 区查"（**迭代查询**）
> - 物业去 XX 区问 → 去 YY 街道问 → 找到"张三在 88 号 3 楼"
> - 物业回来告诉你："张三在 88 号 3 楼"（返回结果）

> [!note] DNS 基于 UDP 端口 53（常规查询），超 512 字节或区域传送用 TCP

### 1.3 DNS 用 UDP 还是 TCP？

> [!warning] 面试高频追问："DNS 用的是 UDP 还是 TCP？"

**答案：两种都用，但场景不同。**

| 场景 | 协议 | 原因 |
|------|------|------|
| **普通域名查询** | **UDP**（端口 53） | 查询请求小（通常 < 512 字节）、需要快速响应、丢包应用层重试即可 |
| **响应超过 512 字节** | 切 **TCP** | UDP 有长度限制，响应太大时会截断，客户端改用 TCP 重试 |
| **区域传送（Zone Transfer）** | **TCP** | 主从 DNS 服务器同步整个区域数据，数据量大、需要可靠传输 |
| **DNSSEC（DNS 安全扩展）** | **TCP** | 签名数据量大，常超 512 字节 |

```
普通查询流程（UDP）：
  客户端 → UDP 查询 → DNS 服务器
  └─ 如果响应 < 512 字节 → 直接返回 ✅
  └─ 如果响应 > 512 字节 → 返回截断标志 → 客户端改用 TCP 重试

区域传送流程（TCP）：
  主DNS服务器 ←── TCP 连接 ──→ 从DNS服务器
  └─ 同步整个 zone 数据（所有记录），必须可靠完整
```

> **面试话术：** "DNS 查询默认用 UDP，因为查询请求小、需要快速响应。但 UDP 有 512 字节限制，如果响应超过 512 字节，客户端会自动切换到 TCP 重试。另外 DNS 区域传送（主从同步）用的是 TCP，因为数据量大且必须可靠。所以严格来说 DNS 两种协议都用，日常查询用 UDP，大数据量场景用 TCP。"

**🔗 知识串联：**
- **串联 04 TCP vs UDP**：DNS 选 UDP 的原因和"为什么 DNS 用 UDP"是一样的逻辑——小数据包、快速响应、丢包可重试，详见 [[04-传输层协议对比/TCP vs UDP]]
- **串联 01 网络体系结构**：DNS 工作在应用层，底层用 UDP（[[04-传输层协议对比/TCP vs UDP]]）——小查询用 UDP 就够了，不需要 TCP 三次握手的开销
- **串联 04 UDP**：DNS 选用 UDP 而非 TCP，因为查询报文小、需要快速响应、丢包应用层重试即可
- **串联 05 HTTP**：HTTP 请求的 `Host` 头就是 DNS 解析出的域名，用于虚拟主机区分
- **串联 06 HTTPS**：TLS 证书绑定的是域名/IP，DNS 解析出 IP 后才能建立 TCP → TLS 连接
- **串联 从URL到页面**：[[07-网络应用技术/从URL到页面]] 全链路的第一步就是 DNS 解析

## 二、DNS 记录类型

| 记录类型 | 全称 | 作用 | 示例 |
|----------|------|------|------|
| **A** | Address | 域名 → IPv4 地址 | `www.example.com → 93.184.216.34` |
| **AAAA** | IPv6 Address | 域名 → IPv6 地址 | `www.example.com → 2606:2800:220:1:...` |
| **CNAME** | Canonical Name | 域名别名 → 另一个域名 | `cdn.example.com → cdn.cloudflare.com` |
| **MX** | Mail Exchange | 指向邮件服务器 | `example.com → mail.example.com（优先级10）` |
| **NS** | Name Server | 指定域名的权威DNS服务器 | `example.com → ns1.example.com` |
| **TXT** | Text | 存文本信息（SPF/DKIM/验证） | `"v=spf1 include:_spf.google.com ~all"` |
| **SRV** | Service | 指定特定服务的服务器和端口 | `_sip._tcp.example.com → 10 60 5060 sip.example.com` |
| **PTR** | Pointer | IP → 域名（反向解析） | `93.184.216.34 → www.example.com` |

> [!tip] 面试高频
> **CNAME** 最常考：它让一个域名指向另一个域名，常用于 CDN 加速（`cdn.example.com CNAME → cdn.cloudflare.com`，之后 Cloudflare 的 DNS 再解析到实际 IP）。

**🔗 知识串联：**
- **串联 01 IPv6**：AAAA 记录就是 IPv6 专用记录，详见 [[01-网络体系结构/网络体系结构]] 的 IPv6 vs IPv4
- **串联 CDN**：CDN 加速的第一步就是把域名 CNAME 到 CDN 服务商的域名

## 三、CDN 原理与反向代理

### 3.1 CDN（内容分发网络）

**核心思想**：把内容缓存到离用户最近的边缘节点，让用户就近获取。

```
没有 CDN：
  用户(上海) ───────────────→ 源站(北京)
            延迟 30ms，可能丢包

有了 CDN：
  用户(上海) → CDN节点(上海) ─ 回源 → 源站(北京)
            延迟 5ms           (仅首次/缓存未命中)
```

**CDN 工作流程**：
1. 用户请求 `cdn.example.com/image.jpg`
2. 本地 DNS 将域名解析到 CDN 的 DNS 调度系统
3. CDN 调度系统根据**用户 IP** 返回最近的边缘节点 IP
4. 用户向边缘节点请求资源
5. 边缘节点有缓存 → 直接返回；没有 → 回源站拉取 → 缓存 → 返回

**比喻**：在全国各地建仓库——北京的货物发上海要 2 天，但如果你在上海就有仓库，30 分钟就到。CDN 就是为互联网内容在全球各地建仓库。

### 3.2 反向代理（Reverse Proxy）

```
正向代理（代理客户端）：
  客户端 → 代理服务器 → 目标服务器
  "帮我去拿"——客户端知道目标是谁

反向代理（代理服务端）：
  客户端 → 反向代理 → 后端服务器集群
  "来找我吧"——客户端以为它访问的就是目标
```

**Nginx 作为反向代理的核心作用**：

| 功能 | 说明 | 比喻 |
|------|------|------|
| **负载均衡** | 将请求分发到多台后端服务器 | 大堂经理分配客人到不同服务员 |
| **SSL 终止** | 在代理层处理 HTTPS 加解密 | 门口安检统一处理 |
| **静态资源** | 直接返回静态文件 | 前台直接给访客宣传册 |
| **缓存** | 缓存响应，减少后端压力 | 常用东西放伸手可及处 |
| **安全防护** | 隐藏后端真实 IP，拦截恶意请求 | 门禁系统 |
| **限流** | 控制请求速率 | 排队系统 |

**🔗 知识串联：**
- **串联 02 TCP**：反向代理的瓶颈常在 TIME_WAIT（[[02-TCP连接管理/TCP连接管理]]），因为每个短连接都会产生 TIME_WAIT
- **串联 05 HTTP 缓存**：CDN 和 Nginx 缓存策略与浏览器缓存（[[05-HTTP协议/HTTP协议详解]]）的 Cache-Control/ETag 机制一致
- **串联 06 HTTPS**：反向代理做 SSL 终止，就是代理层负责 TLS 握手（[[06-HTTPS与安全认证/HTTPS与TLS详解]]），后端服务只需处理 HTTP
- **串联 07 Cookie与认证**：CDN 和反向代理需要注意 Cookie/Session 的跨域问题，详见 [[07-网络应用技术/Cookie与认证]]

### 3.3 负载均衡：L4 vs L7

> [!warning] 面试考点（小红书 SRE 直考）："四层负载均衡和七层负载均衡的区别？"

| 维度 | L4 负载均衡（四层） | L7 负载均衡（七层） |
|------|---------------------|---------------------|
| **工作层** | **传输层**（TCP/UDP） | **应用层**（HTTP/HTTPS） |
| **判断依据** | IP + 端口 | URL 路径、Host、Header、Cookie 等 |
| **比喻** | 快递分拣中心——只看收件地址（IP） | 前台接待——看你要找哪个部门（URL） |
| **性能** | **高**（只解析到传输层，无需解析内容） | 较低（需要解析 HTTP 请求内容） |
| **灵活性** | 低（只能基于 IP:Port 转发） | **高**（可以基于路径/头部/Cookie 路由） |
| **典型实现** | LVS、AWS NLB、F5 | Nginx、HAProxy、AWS ALB、Envoy |
| **SSL 终止** | 不处理（直接转发 TCP 流量） | ✅ 支持（解密后基于内容路由再加密转发） |
| **连接建立** | 客户端→LB→后端（TCP 直通） | 客户端→LB（解包）→LB→后端（新建 TCP） |

```
L4 负载均衡示例：
  客户端(10.0.0.1:8080) → L4 LB → 后端1(192.168.1.1:8080)
                              └→ 后端2(192.168.1.2:8080)
  只看目标 IP:Port，不关心 HTTP 请求内容

L7 负载均衡示例：
  客户端 → L7 LB
    ├─ /api/*    → 后端 API 集群
    ├─ /static/* → CDN / 静态资源服务器
    ├─ /admin/*  → 管理后台集群
    └─ Cookie: session=xxx → 粘性路由到特定后端
  可以根据 URL / Header / Cookie 精细路由
```

> **面试话术：** "四层负载均衡工作在传输层，只看 IP 和端口做转发，性能高但不够灵活，典型如 LVS。七层负载均衡工作在应用层，可以解析 HTTP 内容做精细化路由——按 URL 路径、Header、Cookie 等分流，还能做 SSL 终止，典型如 Nginx。实际生产中两者常组合使用：L4 做入口扛流量，L7 做精细化路由。"

**🔗 知识串联：**
- **串联 01 体系结构**：L4/L7 的"层"对应的就是 OSI 七层模型，详见 [[01-网络体系结构/网络体系结构]]
- **串联 05 HTTP**：L7 负载均衡能解析 HTTP 请求做路由，因为 HTTP 工作在第七层（应用层），详见 [[05-HTTP协议/HTTP协议详解]]
- **串联 04 NIO**：Nginx 既是 L7 负载均衡，又是 epoll 高并发的典型应用，详见 [[04-传输层协议对比/TCP vs UDP]]

## 四、面试回答模板

> **面试官：DNS 解析流程？**

"浏览器先查缓存（浏览器缓存 → hosts 文件），没有的话向本地 DNS 发起递归查询。本地 DNS 通过迭代查询从根域名服务器 → TLD 服务器 → 权威服务器逐步找到 IP。整个过程就像你问物业找人（递归），物业层层问各级机构（迭代），最终把地址告诉你。"

> **面试官：CDN 的原理？**

"CDN 在全球部署边缘节点缓存源站内容。用户请求时，DNS 调度系统根据用户 IP 返回最近的边缘节点。缓存命中直接返回，未命中则回源拉取后缓存。核心是'就近访问'——就像在全国各地建仓库，让用户从最近的仓库取货。"

> **面试官：反向代理的作用？**

"反向代理是服务端前面的'门面'，客户端感知不到后端集群。核心作用：负载均衡（分发请求）、SSL 终止（统一加解密）、静态资源直返、缓存、安全防护（隐藏后端 IP）、限流。微服务中的 API 网关本质上就是反向代理的升级版。"
