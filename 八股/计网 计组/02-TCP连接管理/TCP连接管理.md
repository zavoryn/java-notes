# TCP 连接管理

> [!tip] 模块速记
> **三次握手 = 双方确认初始序列号 + 防历史连接。四次挥手 = 全双工双方各自说"完了"。TIME_WAIT 等 2MSL 保最后一个 ACK 到达 + 防旧包迷路。SYN Flood 占半连接队列，SYN Cookie 不分配资源先验真伪。**

---

## 一、TCP 三次握手

### 1.1 过程详解

```
客户端                             服务端
  |                                  |
  |------- SYN (seq=x) ------------>|  第一次：客户端发SYN，进入SYN_SENT
  |                                  |          "喂，听得到吗？"
  |<--- SYN+ACK (seq=y, ack=x+1) ---|  第二次：服务端回SYN+ACK，进入SYN_RCVD
  |                                  |          "听得到，你说吧"
  |------- ACK (ack=y+1) ---------->|  第三次：客户端回ACK，进入ESTABLISHED
  |                                  |          "好的，我开始说了"
  |                                  |
  |<====== 数据传输 ================>|  双方都进入 ESTABLISHED 状态
```

### 1.2 每次握手后双方各"知道什么"？

> [!warning] 面试极高频追问："第一次握手后服务端知道了什么？第二次后呢？"

**① 第一次握手后，服务端知道了什么？**

客户端发了 SYN 给服务端：
- ✅ 服务端知道了：**客户端有发送能力**
- ✅ 服务端知道了：**自己有接收能力**（因为确实收到了）
- ❌ 服务端不知道：客户端是否能收？自己发的包客户端能不能收到？

**② 第二次握手后，客户端知道了什么？**

客户端收到了服务端的 SYN+ACK：
- ✅ 客户端知道了：**服务端有发送能力**
- ✅ 客户端知道了：**服务端有接收能力**（因为服务端正确回复了自己的 SYN）
- ✅ 客户端知道了：**自己发的 SYN 到达了服务端**
- ❌ 服务端还不知道：自己发的 SYN+ACK 客户端是否收到了？

**③ 第三次握手后，服务端知道了什么？**

服务端收到了客户端的 ACK：
- ✅ 服务端知道了：**客户端有接收能力**（因为客户端正确回复了自己的 SYN+ACK）
- ✅ 至此双方确认：**彼此的发送、接收能力都正常**

```
握手后能力确认一览：

                客户端知道的           服务端知道的
第一次后        （无）                客户端能发 ✓ 自己能收 ✓
第二次后        服务端能发 ✓ 服务端能收 ✓ 自己发的SYN到了 ✓
                                      （同上）
第三次后        （同上）              客户端能收 ✓ 自己发的SYN+ACK到了 ✓

三次握手后 → 双方确认彼此 ISN + 双方收发能力都正常 ✅
```

> **面试话术：** "第一次握手后服务端只知道了客户端能发、自己能收。第二次握手后客户端知道了服务端能发能收，自己的 SYN 也到了。但服务端还不知道客户端能不能收。第三次握手后服务端收到 ACK，才确认客户端也能收——至此双方收发能力全部确认完毕。所以三次是保证双方互相确认收发能力的最小次数。"

### 1.3 为什么是三次，不是两次？

**两次握手的问题**：

如果只有两次握手，客户端发 SYN，服务端回 SYN+ACK 后服务端就认为连接已建立（进入 ESTABLISHED）。此时如果第一个 SYN 是历史滞留报文（在网络中迷路了很久才到达），服务端会在收到这个失效的 SYN 后就傻傻地建立连接、分配资源——但客户端早已不需要这个连接了。这就是**历史连接问题**。

**三次握手恰好能解决**：客户端的第三次 ACK 可以告诉服务端"这个连接我确实要"。如果是历史 SYN，客户端收到 SYN+ACK 后发现不对，发 RST 拒绝即可。

**一句话总结**：三次握手的本质是**双方交换并确认彼此的初始序列号（ISN）**，并防止过期连接请求导致的服务端资源浪费。

### 1.3 打电话比喻（面试杀手锏）

```
我拨通你的电话：
我：  "喂，听得到吗？"          → SYN
你：  "听得到，你说吧"          → SYN + ACK
我：  "好的，我跟你说个事..."   → ACK

这才是完整的"三次确认"。
如果只有两次：
我："喂，听得到吗？"
你："听得到"
——我就开始说，但你可能已经挂了电话去忙别的了。
```

> [!note] 记忆口诀
> **"喂听说"**：喂(SYN)、听得到(SYN+ACK)、好开始说(ACK)

## 二、TCP 四次挥手

### 2.1 过程详解

```
客户端                             服务端
  |                                  |
  |------- FIN (seq=u) ------------>|  第一次：客户端发FIN，进入FIN_WAIT_1
  |                                  |         "我说完了"
  |<------ ACK (ack=u+1) -----------|  第二次：服务端回ACK，进入CLOSE_WAIT
  |        客户端进入FIN_WAIT_2       |         "好的，我知道了"
  |                                  |
  |<------ FIN (seq=v) -------------|  第三次：服务端发FIN，进入LAST_ACK
  |        (服务端把剩下的话说完)      |         "我也说完了"
  |------- ACK (ack=v+1) ---------->|  第四次：客户端回ACK，进入TIME_WAIT
  |                                  |         服务端收到后进入CLOSED
  |                                  |
  |---- 等待 2MSL 后 CLOSED -------->|
```

### 2.2 为什么挥手需要四次？

**核心：TCP 是全双工通信**。

- 客户端发 FIN 只是表示"我这边的数据发完了"——相当于"我说完了"。
- 服务端收到后回 ACK，表示"我知道你说完了"。
- 但是**服务端可能还有数据要发给客户端**——"你虽然完了，但我话还没说完"。
- 等服务端也说完了，再发 FIN——"我也说完了"。
- 客户端回 ACK 确认。

所以四次挥手的过程就像**两个人通电话**：
- 你："我说完了，挂了啊。" (FIN)
- 我："好的。" (ACK) —— 但我可能还有话说
- 我："我也说完了，挂了啊。" (FIN)
- 你："好的，拜拜。" (ACK) —— 然后等 2MSL 再挂

**为什么不能把第二次和第三次合并？**

因为服务端的 ACK 和 FIN 是分开发送的——ACK 是 TCP 协议栈自动回的，FIN 是应用层调用 close() 才发出的。两者之间有时间差。

> [!note] 记忆口诀
> **"我完了，我知道，我也完了，知道了"** + 等 2MSL

## 三、TIME_WAIT 状态详解

### 3.1 TIME_WAIT = 2MSL

Linux 上 MSL 通常是 30 秒，所以 TIME_WAIT = 60 秒。

**作用一：确保最后一个 ACK 能被服务端收到**

客户端发的最后一个 ACK 可能在网络中丢失。如果客户端直接 CLOSED，服务端收不到 ACK 会重发 FIN，但客户端已经关闭了，无法响应。所以客户端在 TIME_WAIT 期间保持端口占用，如果收到服务端重发的 FIN，可以重新发送 ACK。

**作用二：防止旧连接的数据包迷路到新连接**

假设客户端关闭后立即用相同端口发起新连接，而网络中还有上一个连接的滞留数据包在飘荡。如果这个滞留包在新连接建立后才到达，就会造成数据错乱。等待 2MSL 确保所有旧包都已经"自然死亡"。

**比喻**：就像你下班后不会立刻关手机，而是等多一会儿——万一领导还有事找你呢？等确认没消息了再安心关机。

### 3.2 TIME_WAIT 过多的问题及解决

**问题**：高并发短连接场景（如反向代理）会产生大量 TIME_WAIT，耗尽端口。

**解决**：
- `net.ipv4.tcp_tw_reuse = 1`（复用 TIME_WAIT 端口用于新连接）
- 调整 `tcp_tw_recycle`（谨慎使用，NAT 环境有问题）
- 使用长连接代替短连接
- 应用层调小 `tcp_fin_timeout`

## 四、SYN Flood 攻击

服务端收到 SYN 后进入 SYN_RCVD 状态，会分配一个半连接队列。攻击者大量发送 SYN 但不回复最后的 ACK，把服务端的半连接队列占满，导致正常请求无法建立连接。

**防御**：
- **SYN Cookie**——服务端不立即分配资源，而是将必要信息编码进 Cookie 返回给客户端，只有收到合法 ACK 后才真正分配资源。
- 增大半连接队列大小
- 限速/防火墙过滤
- SYN Flood 检测 + IP 黑名单

## 五、TCP Keepalive 机制

### 5.1 TCP Keepalive

TCP Keepalive 是**传输层**的保活探测机制，用于检测连接是否仍然存活。

**工作方式**：
- 连接空闲一段时间后（默认 2 小时），发送探测报文
- 如果对方回复 ACK，连接保持
- 如果连续多次无响应（默认 9 次，每次 75 秒），则认为连接断开，自动关闭

**参数**（Linux）：
- `tcp_keepalive_time`：空闲多久开始探测（默认 7200 秒 = 2 小时）
- `tcp_keepalive_intvl`：探测间隔（默认 75 秒）
- `tcp_keepalive_probes`：探测次数（默认 9 次）

> [!warning] 注意
> TCP Keepalive 默认 2 小时才触发，生产环境通常调小（如 30 秒），用于及时清理僵尸连接。

### 5.2 TCP Keepalive vs HTTP Keep-Alive

| 对比 | TCP Keepalive | HTTP Keep-Alive |
|------|---------------|-----------------|
| **层级** | 传输层 | 应用层 |
| **目的** | 检测连接是否存活 | 复用 TCP 连接发多个请求 |
| **机制** | 空闲时发探测包 | 不主动关闭连接，后续请求复用 |
| **比喻** | 定期打电话确认对方还在 | 去商店买完一样东西不走，继续买下一样 |

**一句话**：TCP Keepalive 是"确认连接还活着"，HTTP Keep-Alive 是"复用连接不关闭"。

## 六、CLOSE_WAIT 状态过多排查

### 6.1 什么是 CLOSE_WAIT？

四次挥手中，服务端收到客户端 FIN 后回 ACK，进入 CLOSE_WAIT 状态——等待应用层调用 close() 发出自己的 FIN。如果应用层迟迟不调用 close()，CLOSE_WAIT 就会一直存在。

### 6.2 排查方法

1. **统计 CLOSE_WAIT 数量**：`ss -ant state close-wait | wc -l`
2. **定位是哪个进程**：结合 `netstat` 或 `ss` 找出对应的 PID
3. **检查代码**：是否在异常分支忘记 close()？是否用了不正确的连接池配置？
4. **常见原因**：
   - 代码 bug：异常路径忘记关闭 socket
   - 连接池配置不当：未设置最大空闲时间
   - 对方突然断开但本地未感知到

> [!warning] 核心结论
> **CLOSE_WAIT 过多几乎一定是代码 bug**——应用层没有及时调用 close()。

## 八、面试回答模板

> **面试官：TCP 三次握手的过程？为什么是三次不是两次？**

"打个比方，就像打电话——'喂，听得到吗？'、'听得到，你说吧'、'好的，我跟你说个事'。为什么是三次？一是需要双方确认彼此的序列号，二是防止历史连接的 SYN 报文让服务端白白建立连接。如果是两次的话，服务端收到一个半年前发出的 SYN，可能傻傻地分配资源等着——因为客户端根本不要这个连接了。"

> **面试官：TCP 四次挥手的过程？为什么是四次？**

"四次挥手因为 TCP 是全双工通信，双方要各自关闭自己的发送方向。
过程是：客户端发 FIN 说'我说完了'，服务端回 ACK 说'知道了'，这时服务端可能还有数据要发；等服务端也说完了发 FIN，客户端回 ACK 确认，然后客户端进入 TIME_WAIT 等 2MSL 才真正关闭。
为什么不能合并成三次？因为服务端的 ACK 是 TCP 协议栈自动回的，而 FIN 要等应用层调用 close() 才发，中间可能有数据还在传——就像挂电话，你说完了不代表我也说完了，得各自说各自的'再见'。"

> **面试官：TIME_WAIT 的作用？TIME_WAIT 过多怎么办？**

"TIME_WAIT 等 2MSL 有两个作用：一是确保最后一个 ACK 能到达对方——如果丢了对方会重发 FIN，此时我还活着可以重发 ACK；二是防止旧连接的迷路数据包在新连接中造成混乱——等 2MSL 确保旧包自然死亡。

TIME_WAIT 过多常见于高并发短连接，解决方案包括开启 tcp_tw_reuse 复用端口、用长连接替代短连接、以及调小 tcp_fin_timeout。"

> **面试官：CLOSE_WAIT 过多怎么排查？**

"CLOSE_WAIT 是四次挥手中服务端收到客户端 FIN 后进入的状态，在等应用层调用 close() 发出自己的 FIN。如果大量 CLOSE_WAIT 堆积，**几乎一定是代码 bug**——程序在某个异常分支忘了关闭 socket 连接。排查步骤：先用 `ss -ant state close-wait` 统计数量并定位进程，然后检查代码中是否有未关闭连接的路径，同时排查连接池配置是否合理。"

> **面试官：什么是 SYN Flood 攻击？怎么防御？**

"SYN Flood 是攻击者大量发送 SYN 但不回最后的 ACK，把服务端的半连接队列占满，导致正常请求无法建连。防御方案是 SYN Cookie——服务端不立即分配资源，而是把必要信息编码进 Cookie 返回，只有收到合法 ACK 后才真正建连接。此外还可以增大半连接队列、限速、加 IP 黑名单等。"

> **面试官：TCP keepalive 和 HTTP keep-alive 有什么区别？**

"TCP keepalive 是传输层的保活探测机制——连接空闲一段时间后发探测包，多次无响应就认为断开。HTTP keep-alive 是应用层的连接复用——不发 Connection: close，让同一个 TCP 连接可以发多个 HTTP 请求，省去反复握手挥手的开销。一句话：TCP keepalive 是确认连接还活着，HTTP keep-alive 是复用连接不关闭。"
