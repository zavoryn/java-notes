## 二、面试满分话术

### 话术一：TCP 三次握手

> "打个比方，就像打电话——'喂，听得到吗？'、'听得到，你说吧'、'好的，我跟你说个事'。为什么是三次？一是需要双方确认彼此的序列号，二是防止历史连接的 SYN 报文让服务端白白建立连接。如果是两次的话，服务端收到一个半年前发出的 SYN，可能傻傻地分配资源等着——因为客户端根本不要这个连接了。"

### 话术二：TCP 可靠传输

> "TCP 的可靠性靠五板斧保证：序列号保证有序和去重、确认应答保证送达、超时重传兜底、滑动窗口做流量控制防止接收方被撑死、拥塞控制防止网络被堵死。拥塞控制又有四个算法——慢启动试探、拥塞避免稳增长、快重传不空等、快恢复不归零。这一套组合拳打下来，TCP 才能在不靠谱的 IP 网络之上提供靠谱的传输。"

### 话术三：HTTPS

> "HTTPS 的精髓是混合加密——用非对称加密来安全地交换一把对称密钥，之后用对称密钥高效地传输真正数据。为什么不用非对称全程加密？因为慢，比对称加密慢上千倍。整个过程就像一个开着的锁寄给你，你把自己的钥匙放进去锁上寄回来，我用我的私钥打开拿到你的钥匙——之后我们就用这把钥匙通信。加上 CA 证书就像给身份加了公章，防止中间人冒充。"

### 话术四：从 URL 到页面

> "这本质上是后端同学理解整个 Web 技术栈的'全链路地图'。DNS 把域名翻译成 IP，TCP 三次握手建立可靠通道，TLS 加密保障安全，HTTP 发送应用层请求，Nginx 反向代理做路由和负载均衡，应用服务器执行业务逻辑，最后浏览器渲染页面。把这个流程串起来，就对整个 Web 体系有了全景认识。"

## 三、记忆口诀

### OSI 七层速记

```
物数网传会表应
（物理→数据链路→网络→传输→会话→表示→应用）
```

### TCP 三次握手 —— "喂听说"

| 次数 | 发送方 | 报文 | 对话 |
|------|--------|------|------|
| 1 | 客户端 | SYN | "喂，听得到吗？" |
| 2 | 服务端 | SYN+ACK | "听得到，你说" |
| 3 | 客户端 | ACK | "好，开始说" |

### TCP 四次挥手 —— "我完了，我知道，我也完了，知道了"

| 次数 | 发送方 | 报文 | 对话 |
|------|--------|------|------|
| 1 | 客户端 | FIN | "我说完了" |
| 2 | 服务端 | ACK | "知道了" |
| 3 | 服务端 | FIN | "我也说完了" |
| 4 | 客户端 | ACK | "好，拜拜" + 等 2MSL |

```
1xx：信息提示还没完
2xx：请求成功已搞定
3xx：换个地址再请求
4xx：客户端你犯的错
5xx：服务端我背的锅
```

```
快重传——三声提醒即可重发
快恢复——减速但不归零
超时重传——停稳了从零再来
```

## 五、附录：高频面试追问

### Q1：TCP 粘包/拆包解决方案详解

**粘包产生原因**：
- 发送方：Nagle 算法将小包合并
- 接收方：应用层读取速度跟不上接收速度，TCP 是面向字节流，没有边界

**拆包产生原因**：
- 发送方的数据大于 MSS（最大报文段长度），TCP 会拆分
- 发送方的数据大于发送缓冲区大小

**解决方案**：

| 方案 | 说明 | 优缺点 | 代表 |
|------|------|--------|------|
| **定长消息** | 每个包固定长度，不足补齐 | 简单但浪费带宽 | 简单协议 |
| **分隔符** | 用特殊字符标记边界（如 `\r\n`） | 简单但分隔符可能出现在数据中 | HTTP、FTP |
| **消息头 + 消息体（TLV）** | 头部声明长度，Body 存数据 | **最常用**，灵活高效 | Netty 的 LengthFieldBasedFrameDecoder |
| **更复杂的协议** | 如 Protobuf/Thrift 等自带序列化和边界 | 最完善但有学习成本 | RPC 框架 |

> [!tip] 面试加分
> **TCP 粘包不是 TCP 的 bug，而是特性**——TCP 是面向字节流的设计，本来就没有边界概念。"粘包"是应用层没有正确解析字节流的问题，需要在应用层协议中定义边界。

### Q2：NAT 穿透原理

**NAT（Network Address Translation）**：内网设备用私有 IP（如 192.168.x.x），通过 NAT 转换成公网 IP 访问外网。NAT 只记录"出站连接"，外部主动访问内网会被拒绝。

**NAT 穿透问题**：两台都在 NAT 后面的设备（如两个家用电脑）无法直接通信——彼此都不知道对方的真实公网 IP + 端口。

**穿透方案**：

| 方案 | 原理 | 应用 |
|------|------|------|
| **STUN** | 通过公网中继服务器获取自己的公网映射地址 | VoIP、WebRTC |
| **TURN** | 所有流量通过中继服务器转发（兜底方案） | STUN失败时的备选 |
| **ICE** | 综合STUN+TURN+本地地址，按优先级尝试 | WebRTC标准框架 |
| **打洞（Hole Punching）** | 双方先通过中继服务器交换地址，然后同时向对方发包"打洞" | P2P下载、游戏 |
| **反向代理/VPN** | 通过第三方服务器中转流量 | 企业远程办公 |

> [!warning] 打洞原理
> A和B都想对方发一个包，NAT会为"出站"的包开放一个入站端口映射。双方同时发包后，各自的NAT都有了对方的映射记录，后续直接通信不再需要中继。

### Q3：如果让你设计一个网络协议，需要考虑哪些方面？

> [!note] 这是一道开放性设计题，考察系统思维

**核心考虑维度**：

| 维度 | 关键问题 | 参考 |
|------|----------|------|
| **可靠性** | 是否需要保证送达？ACK+重传？ | TCP的五板斧 |
| **连接管理** | 面向连接还是无连接？握手/挥手流程？ | TCP三次握手/四次挥手 |
| **数据边界** | 字节流还是消息边界？粘包如何处理？ | TLV格式/分隔符/定长 |
| **安全性** | 是否加密？对称还是非对称？证书认证？ | TLS混合加密 |
| **效率** | 头部大小？是否压缩？小包合并？ | HPACK/Nagle |
| **流量控制** | 接收方是否可能被撑死？滑动窗口？ | TCP rwnd |
| **拥塞控制** | 网络拥堵时如何退让？慢启动/快恢复？ | TCP拥塞控制/BBR |
| **扩展性** | 是否支持版本协商？功能扩展？ | TLS版本协商 |
| **多路复用** | 是否需要单连接多流？Stream机制？ | HTTP/2.0/QUIC |
| **连接迁移** | 切换网络是否需要重连？Connection ID？ | QUIC |

### Q4：HTTP 长连接 vs WebSocket 长连接的区别？

| 对比 | HTTP Keep-Alive 长连接 | WebSocket 长连接 |
|------|------------------------|------------------|
| **通信方向** | 半双工（客户端请求→服务端响应） | 全双工（双方随时发） |
| **主动性** | 服务端不能主动推送（只能等客户端请求） | 服务端可以主动推送 |
| **协议模型** | 仍是 HTTP 请求-响应模型 | 独立的全双工协议（建立后脱离HTTP） |
| **头部开销** | 每次请求仍带 HTTP 头部 | 建立后有头部，之后只 2~10 字节帧头 |
| **适用场景** | 普通请求复用连接 | 实时聊天、股票行情、游戏推送 |
| **连接建立** | HTTP请求自动建立 | HTTP Upgrade（101）切换协议 |

> [!warning] 核心区别
> HTTP 长连接是**连接复用**（省握手开销），但通信模式不变（仍是请求-响应）。WebSocket 长连接是**通信模式改变**（从请求-响应变成全双工）。

### Q5：浏览器同时能发多少 HTTP 请求？为什么 HTTP/2 不需要域名分片？

**HTTP/1.1**：
- 同一域名下浏览器限制 **6~8 个并发 TCP 连接**（Chrome 6 个）
- 超过限制的请求排队等待
- 历史优化：**域名分片**（sharding）——把资源分散到多个子域名（如 static1.example.com, static2.example.com），每个子域名都有独立的 6 连接上限，总并发数倍增

**HTTP/2.0**：
- 同一域名只需 **1 个 TCP 连接**，多路复用无并发限制
- 多个 Stream 在一个连接上并行传输，互不阻塞
- **域名分片不再需要**——反而有害（多个连接 = 多次 TLS 握手 + 多个拥塞控制窗口）

> [!tip] 面试加分
> HTTP/2 下域名分片反而降低性能：每个额外连接都需要独立的 TLS 握手和拥塞控制窗口，TCP 层面无法共享带宽感知。

### Q6：ping 的底层原理？ICMP 协议？

**ping 命令**：测试网络连通性的工具，发送 ICMP Echo Request 报文，目标回复 ICMP Echo Reply。

**ICMP（Internet Control Message Protocol）**：
- 工作在**网络层**（IP 之上），是 IP 协议的"报错和诊断助手"
- 不传输应用数据，只传输**控制信息**（错误报告、网络诊断）
- 主要报文类型：
  - **Echo Request/Reply**（类型 8/0）：ping 用的
  - **Destination Unreachable**（类型 3）：目标不可达
  - **Time Exceeded**（类型 11）：超时（traceroute 用的）
  - **Redirect**（类型 5）：路由重定向

**ping 工作流程**：
```
源主机 → ICMP Echo Request → 目标主机
目标主机 → ICMP Echo Reply → 源主机

源主机计算 RTT = 收到Reply时间 - 发送Request时间
```

> [!note] traceroute 原理
> traceroute 发一系列 TTL 递增的 ICMP/UDP 包，每个中间路由器 TTL 减到 0 时回复 Time Exceeded，从而逐跳探明路径。

> [!warning] ICMP ≠ 传输层
> ICMP 是网络层协议，不是传输层协议。它不提供端到端的服务，而是网络层的诊断和控制工具。
