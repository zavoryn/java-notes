# TCP 可靠传输

> [!tip] 模块速记
> **TCP 可靠传输靠五板斧：序列号(有序去重) + 确认应答(送达确认) + 超时重传(兜底) + 滑动窗口(流量控制) + 拥塞控制(防堵死)。拥塞控制口诀：慢启动猛踩油、拥塞避免稳增长、快重传三声提醒、快恢复减速不归零、超时从零再来。**

## 二、序列号与确认应答

- **序列号（Seq）**：每个字节一个序列号，保证有序和去重
- **确认应答（ACK）**：收到数据后回复 ACK，ack = 期望收到的下一个字节序号
- **累计确认**：ACK 确认该序号之前的所有字节都已收到

**比喻**：就像快递签收——你签收的是所有已到包裹的最后一个编号，快递公司就知道你收到了哪些。

> [!note] 捎带确认
> 当接收方要发数据时，把 ACK 信息"捎带"在数据包里一起发出去，不需要单独发一个 ACK 包——省了一次网络往返。

---

## 三、超时重传与快重传

### 3.1 超时重传（RTO）

发送方每发一个报文就启动一个定时器，如果在 RTO（Retransmission Timeout）内没收到 ACK，就重传。

**RTO 计算**：通过 RTT（Round-Trip Time）动态计算，采用 Jacobson/Karels 算法——不仅取 RTT 平均值，还要加上偏差值，保证 RTO 足够大以容忍网络抖动，又足够小以快速重传。

### 3.2 快重传（Fast Retransmit）

不需要等到超时，当发送方**连续收到 3 个重复 ACK**（Dup ACK）时，立即重传丢失的报文。

**比喻**：你是老师连续点名，学生1号到了、学生2号到了——每次点名你都报的是3号没到（重复ACK），喊了三遍3号都没来，你立刻意识到3号确实缺席了，不用等到下课后才去查考勤（超时重传），当场就让人去找。

## 四、流量控制——滑动窗口

> **核心问题**：发送方发太快，接收方处理不过来 → 数据堆积 → 丢包。滑动窗口让接收方告诉发送方"我还能收多少"。

### 4.1 完整过程（逐步示例）

```
发送方                                    接收方
  │  缓冲区大小：1000                         │  缓冲区大小：1000
  │                                         │
  │─────── 发送 400 字节 ───────→            │
  │                                         │  收到 400，缓冲区剩 600
  │←───── ACK + rwnd=600 ───────            │  "我还能收 600"
  │  发送窗口调成 600                          │
  │─────── 发送 500 字节 ───────→            │
  │                                         │  收到 500，缓冲区剩 100
  │←───── ACK + rwnd=100 ───────            │  "我还能收 100"
  │  发送窗口调成 100（悠着点发）               │
  │─────── 发送 100 字节 ───────→            │
  │                                         │  收到 100，缓冲区满了！
  │←───── ACK + rwnd=0 ────────             │  "我满了，别发了！"
  │  发送窗口 = 0，停止发送                    │
  │                                         │
  │  （接收方处理完数据，缓冲区空出来了）          │
  │←───── 窗口更新 rwnd=800 ────             │  "我又能收了，发吧"
  │  恢复发送！                                │
```

### 4.2 关键概念

| 概念 | 说明 |
|------|------|
| **rwnd（接收窗口）** | 接收方缓冲区剩余大小，每次 ACK 时带上 |
| **发送窗口** | = min(cwnd, rwnd)，取拥塞窗口和接收窗口的较小值 |
| **零窗口** | rwnd=0 时发送方停止发送，定期发**窗口探测报文**问"你缓过来没" |

> **面试话术：** "滑动窗口解决流量控制——防止发送方撑爆接收方。接收方每次回 ACK 时在 TCP 头部带 rwnd 告诉发送方'我缓冲区还剩多少'，发送方据此控制发送量。rwnd=0 就停发，定期探测等恢复。实际发送量 = min(cwnd, rwnd)，同时考虑接收方能不能收和网络能不能扛。"

## 五、拥塞控制——四大算法

> **核心问题**：网络本身拥堵了——不是接收方的问题，是中间的路由器/链路承受不了。发送方怎么知道？**靠丢包推断**。

### 5.1 四个算法 = 一个完整的故事（新手上路比喻）

想象你刚拿到驾照上了一辆新车，不知道路况怎么样：

#### 🚗 阶段一：慢启动（Slow Start）

```
你不知道路况，先试探着开：
第一辆车上路 → 没堵？太好了，发两辆！
两辆没堵？   → 发四辆！
四辆没堵？   → 发八辆！

cwnd: 1 → 2 → 4 → 8 → 16 → ...  （指数翻倍）
```

**规则**：cwnd 从 1 开始，每收到一轮 ACK 就**翻倍**。名字叫"慢启动"其实是"启动时从慢开始试探，但加速很快"。

#### 🚗 阶段二：拥塞避免（Congestion Avoidance）

```
车越来越多，你感觉不能再猛加了：
从"翻倍"改成"每轮+1"

cwnd: 16 → 17 → 18 → 19 → ...  （线性+1）
```

**触发条件**：cwnd 到达 **ssthresh（慢启动阈值）** 后切换。

#### ⚠️ 发生丢包！两种情况不同处理：

**情况 A：收到 3 个重复 ACK → 快重传 + 快恢复**

```
发送了 1,2,3,4,5 → 接收方收到 1,2,_,4,5（3丢了）
接收方连续回: ACK2, ACK2, ACK2（连喊三声"我要3！"）
→ 不等超时，立刻重传3！（快重传）

处理：ssthresh = cwnd/2, cwnd = ssthresh → 继续拥塞避免（快恢复）
比喻：感觉前面有点堵但没堵死 → 减速到一半继续开，不用停车重新起步
```

**情况 B：超时（等了好久都没 ACK）→ 严重拥塞**

```
发了包出去，等了好久好久了......什么都没回来
→ 说明网络严重拥堵

处理：ssthresh = cwnd/2, cwnd = 1 → 重新慢启动
比喻：直接熄火了 → 停下来，挂一档，从头起步
```

### 5.2 速查表

| 算法 | cwnd 变化 | 触发条件 | 比喻 |
|------|----------|----------|------|
| **慢启动** | 指数翻倍（1→2→4→8...） | 连接建立初期 | 起步猛踩油门试探 |
| **拥塞避免** | 线性+1（每轮） | cwnd >= ssthresh | 稳中求进 |
| **快重传** | 不改变 cwnd，立刻重传丢包 | 3个重复ACK | 三声提醒就重发 |
| **快恢复** | cwnd = cwnd/2，继续拥塞避免 | 快重传后 | 减速但不归零 |
| **超时重传** | cwnd = 1，重新慢启动 | 超时无回应 | 停稳了从零再来 |

```
cwnd
  │          ╱ ╱ ╱  拥塞避免（线性+1）
  │        ╱
  │      ╱
  │    ╱  慢启动（指数翻倍）
  │  ╱
  │╱
  └────────────────────────────→ 时间
      ↑              ↑
    ssthresh       ssthresh
                   (cwnd/2)

    收到3个DupACK:      超时:
    cwnd = cwnd/2       cwnd = 1 ← 重新慢启动
    继续拥塞避免
```

> **面试话术：** "拥塞控制是发送方感知网络拥塞并主动调节的机制，控制变量是 cwnd。四个算法：慢启动从1开始每轮翻倍试探网络容量，到阈值后切拥塞避免每轮+1稳增长。如果收到3个重复ACK说明丢了个别包，立刻快重传然后快恢复（cwnd减半不归零）；如果超时说明严重拥堵，cwnd归1重新慢启动。最后流量控制看rwnd怕接收方撑死，拥塞控制看cwnd怕网络堵死，实际发送取两者较小值。"

> [!note] 拥塞控制口诀
> ```
> 慢启动——起步猛踩油
> 拥塞避免——稳中求进
> 快重传——三声提醒即可重发
> 快恢复——减速但不归零
> 超时重传——停稳了从零再来
> ```

## 六、Nagle 算法详解

### 6.1 问题背景

TCP 是面向字节流的，应用层可能频繁发送很小的数据（如 1 字节），每次小数据都封装成 TCP 报文（头部 20 字节 + 数据 1 字节 = 21 字节），有效利用率极低——这就是"小包问题"（Tinygram Problem）。

### 6.2 Nagle 算法策略

**核心规则**：
1. 只有第一个小包可以立即发送（无论大小）
2. 之后的小包必须积累到足够大（达到 MSS）或收到上一个包的 ACK 后才能发送

**效果**：将多个小包合并为一个较大的包发送，减少网络上的小包数量。

### 6.3 Nagle 的利弊

| 方面 | 优点 | 缺点 |
|------|------|------|
| **网络效率** | 减少小包数量，降低带宽浪费 | |
| **延迟** | | 增加小包发送延迟（需等积累或ACK） |
| **实时场景** | | 交互式应用（游戏/远程终端）体验差 |

### 6.4 TCP_NODELAY

在延迟敏感的场景（游戏、实时交互），可以设置 `TCP_NODELAY` 选项**关闭 Nagle 算法**，允许小包立即发送。

> [!warning] Nagle + 延迟确认 = 双倍延迟
> 如果 Nagle 算法和延迟确认同时开启，可能产生 **200ms+ 的额外延迟**：Nagle 等ACK才发下一个包，但接收端延迟确认又拖着不回ACK——形成"互相等"的死锁。

## 七、延迟确认（Delayed ACK）

### 7.1 机制

接收方收到数据后不立即回 ACK，而是等待一小段时间（通常 40ms~200ms）：
- 如果这段时间内有数据要发给对方，就**捎带确认**（节省一个 ACK 报文）
- 如果没有数据要发，超时后才单独发 ACK

### 7.2 利弊

| 优点 | 缺点 |
|------|------|
| 减少纯 ACK 报文数量，节省带宽 | 增加发送方的等待时间 |
| 捎带确认提高效率 | 与 Nagle 组合可能造成双倍延迟 |

## 八、拥塞控制算法演进——BBR

### 8.1 传统算法的问题

Reno/CUBIC 等传统拥塞控制算法基于**丢包信号**——只要丢包就认为拥塞，然后大幅降低发送速率。问题：
- 高带宽长距离网络中，偶尔丢包不代表拥塞（可能是链路噪声）
- 丢包触发降速→恢复慢→带宽利用率低

### 8.2 BBR 算法核心思想

BBR（Bottleneck Bandwidth and Round-trip propagation time）由 Google 提出，**不再以丢包为信号**，而是直接测量：
- **Bw**：瓶颈带宽（管道能通过的最大速率）
- **Rtprop**：最小 RTT（管道的长度/延迟）

BBR 根据这两个参数决定发送速率，目标是将管道恰好填满但不溢出。

**比喻**：传统算法像"看到堵车就刹车"，BBR 像"精确测量管道容量后定量注水"。

> [!tip] 面试加分
> BBR 在高带宽长距离网络中性能远优于 CUBIC，Linux 4.9+ 已支持。YouTube 开启 BBR 后带宽利用率提升了 4%~30%。

## 九、TCP 校验和机制

TCP 头部有一个 16 位的 **Checksum** 字段，用于检测数据在传输过程中是否被篡改或损坏。

**计算范围**：TCP 头部 + TCP 数据 + 伪头部（含源IP、目的IP、协议号、TCP长度）

**工作方式**：
- 发送方对整段数据计算校验和并填入头部
- 接收方收到后重新计算，与头部中的值比较
- 不匹配 → 丢弃报文（不回 ACK，触发重传）

> [!note] 注意
> 校验和是**端到端**的验证——每一跳（路由器）只验证 IP 层校验和，TCP 校验和只在最终接收方验证。

## 十、面试回答模板

> **面试官：TCP 如何保证可靠传输？**

"TCP 的可靠性靠五板斧保证：序列号保证有序和去重、确认应答保证送达、超时重传兜底、滑动窗口做流量控制防止接收方被撑死、拥塞控制防止网络被堵死。拥塞控制又有四个算法——慢启动试探、拥塞避免稳增长、快重传不空等、快恢复不归零。这一套组合拳打下来，TCP 才能在不靠谱的 IP 网络之上提供靠谱的传输。"

> **面试官：Nagle 算法是什么？什么时候要关闭？**

"Nagle 算法把多个小数据包合并成一个较大的包发送，减少网络上的小包浪费。但会增加延迟——对于交互式应用（游戏、远程终端），需要设置 TCP_NODELAY 关闭 Nagle。特别要注意 Nagle 和延迟确认同时开启可能产生 200ms+ 的额外延迟，这是性能调优中的经典陷阱。"
