# HTTPS 与 TLS 详解

> [!tip] 模块速记
> **HTTPS = HTTP + SSL/TLS，解决窃听/篡改/冒充三大风险。混合加密：非对称传钥匙，对称传数据。CA证书防冒充（公证处盖章）。中间人攻击靠证书链防御。mTLS 双向认证在微服务 Service Mesh 中常用。**

---

## 〇、HTTP vs HTTPS 完整对比

| 维度 | HTTP | HTTPS |
|------|------|-------|
| **安全性** | 明文传输，不安全 | TLS 加密传输，安全 |
| **端口** | **80** | **443** |
| **证书** | 不需要 | 需要 CA 颁发的 SSL 证书（要钱/要配置） |
| **性能** | 无加密开销，更快 | TLS 握手有额外 RTT 开销（TLS 1.2 多 2-RTT，TLS 1.3 多 1-RTT） |
| **成本** | 零成本 | 证书费用 + 服务器 CPU 消耗（加解密） |
| **SEO** | 搜索引擎会降权 | Google 明确优先收录 HTTPS |
| **现代功能** | 越来越多功能不支持 | HTTP/2.0、Service Worker、地理位置等都**强制要求 HTTPS** |

> **面试话术：** "HTTP 和 HTTPS 的核心区别就是 HTTPS 在 HTTP 和 TCP 之间加了一层 TLS，解决了 HTTP 明文传输的三个问题：窃听、篡改、冒充。具体来说：①安全：HTTP 明文，HTTPS 加密；②端口：80 vs 443；③证书：HTTPS 需要向 CA 申请证书；④性能：HTTPS 有 TLS 握手开销，但 TLS 1.3 已优化到 1-RTT 甚至 0-RTT；⑤趋势：现代 Web 功能（HTTP/2、PWA）都强制要求 HTTPS，HTTP 基本在淘汰中。"

> [!warning] 面试避坑
> 不要只说"HTTPS 多了 TLS"——这太浅了。要能从安全、端口、证书、性能、趋势 5 个维度展开。

---

## 一、HTTP 的三大安全问题

| 风险 | 说明 | 比喻 |
|------|------|------|
| **窃听** | 明文传输，中间人可看到内容 | 明信片——任何人都能看 |
| **篡改** | 内容可能被中间人修改 | 明信片上内容被涂改 |
| **冒充** | 可能在与伪造的服务器通信 | 把信寄给了假地址 |

HTTPS = HTTP + SSL/TLS，解决以上三个问题：
- **加密** → 防窃听
- **数据完整性校验（MAC）** → 防篡改
- **身份认证（证书）** → 防冒充

---

## 二、HTTPS 的加密机制

HTTPS 采用**混合加密**——非对称加密 + 对称加密：

```
1. 非对称加密（公私钥）：安全交换对称密钥
2. 对称加密：用交换好的密钥加密真正的数据
```

**为什么混合？**
- 非对称加密安全但慢（RSA 比 AES 慢 1000 倍以上）
- 对称加密快但不安全（密钥怎么传给对方？）
- **取长补短**：用非对称加密安全地传递对称密钥，然后用对称密钥高效加密后续通话

**经典比喻——快递箱**：

> 你给朋友寄一个贵重物品：
> 1. 你先给朋友寄一个**空的、带锁的箱子**，锁是开着的（你的公钥）
> 2. 朋友拿到箱子后，把自己的**对称密钥纸条**放进去，锁上箱子（用你的公钥加密）
> 3. 箱子寄回给你，你用**你的私钥**开锁，取出朋友的对称密钥纸条
> 4. 之后你们就用这个**对称密钥**传递真正的贵重物品——又快又安全

这整个过程就是 TLS 握手 + 对称加密传输。

## 三、TLS 1.2 握手流程（四次通信）

```
客户端                                   服务端
  |                                        |
  |--(1) ClientHello -------------------->|
  |   【支持的加密套件、TLS版本、随机数1】   |
  |                                        |
  |<--(2) ServerHello --------------------|
  |   【选定的加密套件、随机数2】            |
  |<--(2) Certificate --------------------|  ← 服务端证书（含公钥）
  |<--(2) ServerHelloDone ----------------|
  |                                        |
  |--(3) ClientKeyExchange -------------->|  ← 用公钥加密的预主密钥
  |--(3) ChangeCipherSpec --------------->|  ← 通知切换加密模式
  |--(3) Finished ----------------------->|
  |                                        |
  |<--(4) ChangeCipherSpec ---------------|
  |<--(4) Finished -----------------------|
  |                                        |
  |<======= 对称加密传输数据 ==============>|
```

**简要说明**：
1. 客户端发送支持的加密套件、TLS 版本、第一个随机数
2. 服务端选择加密套件、发第二个随机数、发证书（含公钥）
3. 客户端验证证书、生成预主密钥并加密发送、通知切换加密
4. 服务端同样通知切换加密

最终**三个随机数**（客户端随机数 + 服务端随机数 + 预主密钥）共同生成会话密钥。

### 3.1 RSA 密钥交换 vs ECDHE 密钥交换

> [!warning] 面试直考（携程、小红书）："TLS 握手用的什么密钥交换算法？有什么区别？"

TLS 1.2 支持两种主要密钥交换方式：

| 维度 | RSA 密钥交换 | ECDHE 密钥交换 |
|------|-------------|---------------|
| **核心** | 客户端用服务端公钥加密预主密钥 | 双方各自生成临时密钥对，交换公钥算出共享密钥 |
| **前向安全** | ❌ 没有（私钥泄露→所有历史通信可解密） | ✅ 有（每次连接用不同临时密钥，私钥泄露不影响历史） |
| **性能** | RSA 运算较慢 | ECC 运算更快，密钥更短（256bit ≈ RSA 3072bit） |
| **TLS 1.3** | ❌ 已废弃 | ✅ 唯一保留的密钥交换方式 |

**ECDHE 密钥交换过程（面试要说得出）**：

```
客户端                          服务端
  |                                |
  |   ClientHello                  |
  |  【随机数C + 支持的曲线】 ──→   |
  |                                |
  | ← ServerHello + 证书           |
  |  【随机数S + 服务端临时公钥】    |
  |                                |
  |  客户端验证证书 ✓               |
  |  生成自己的临时密钥对            |
  |  用双方临时公钥算出共享密钥      |
  |  【客户端临时公钥】───────→     |
  |                                |
  |  服务端用双方临时公钥算出共享密钥 |
  |  （两边算出来的结果一样！）       |
  |                                |
  |  会话密钥 = PRF(随机数C, 随机数S, 共享密钥)  |
```

**前向安全（Forward Secrecy）**——面试关键点：
- RSA 交换：预主密钥用服务端**长期私钥**解密 → 私钥泄露 = 所有历史通信可被解密
- ECDHE 交换：每次连接用**临时密钥对**，连接结束即销毁 → 即使长期私钥泄露，历史通信也无法解密

> **面试话术：** "TLS 密钥交换主要有 RSA 和 ECDHE 两种。RSA 是客户端用公钥加密预主密钥发给服务端，但没有前向安全——私钥泄露所有历史通信都能解密。ECDHE 是双方各自生成临时密钥对，交换公钥后算出共享密钥，每次连接密钥不同，即使长期私钥泄露也不影响历史通信，这就是前向安全。TLS 1.3 已废弃 RSA，只保留 ECDHE。"

**🔗 知识串联：**
- **串联 03 TCP**：TLS 握手在 TCP 三次握手之后，增加了额外延迟（TLS 1.2 多 2-RTT），详见 [[02-TCP连接管理/TCP连接管理]]
- **串联 04 QUIC**：QUIC 把 TCP 握手和 TLS 握手合并，再加上 ECDHE 0-RTT 恢复，首次连接只需 1-RTT，详见 [[04-传输层协议对比/TCP vs UDP]]

## 四、TLS 1.3 的改进

- 握手从 2-RTT 精简到 **1-RTT**（去掉了一次往返）
- 0-RTT 恢复：之前连接过的话可以直接发数据
- 废弃不安全的加密算法（如 RSA 密钥交换，只保留 DH 系列）
- 大部分握手消息已经加密

> [!note] TLS 1.2 vs 1.3 对比
> | 版本 | RTT | 特点 |
> |------|-----|------|
> | TLS 1.2 | 2-RTT | RSA密钥交换仍支持 |
> | TLS 1.3 | 1-RTT (首次) / 0-RTT (恢复) | 只保留DH系列，废弃RSA |

## 五、数字证书与 CA 证书验证链

### 5.1 核心问题

客户端怎么知道收到的公钥确实是目标服务器的，而不是中间人伪造的？

**CA（Certificate Authority，证书颁发机构）** 就像**公证处**：
- 服务端把公钥和个人信息提交给 CA
- CA 用**自己的私钥**对服务端信息签名，生成证书
- 客户端用 CA 的**公钥**（预装于操作系统/浏览器中）验签

**比喻**：你的身份证（证书）由政府（CA）盖章签发，别人验证你的身份时通过政府的公章（CA公钥）来确认身份证的真伪。

### 5.2 CA 证书验证链详解

```
根证书（Root CA）
  │  • 预装在操作系统/浏览器中（自签名，绝对信任）
  │  • 签发中间证书
  │
  ▼
中间证书（Intermediate CA）
  │  • 由根CA签发
  │  • 为什么需要中间层？根CA离线保存保安全，日常签发由中间CA完成
  │  • 一个根CA可以有多个中间CA
  │
  ▼
服务器证书（Server Certificate / Leaf Certificate）
  │  • 由中间CA签发
  • 包含：域名、公钥、有效期、签发者信息
```

**验证流程**：
1. 客户端收到服务器证书 → 找到签发者是中间CA
2. 检查中间CA证书 → 找到签发者是根CA
3. 根CA证书已在本地信任库 → 信任链验证通过
4. 用根CA公钥验证中间CA → 用中间CA公钥验证服务器证书 → 确认公钥真实

> [!warning] 证书链断裂 = 不信任
> 如果中间CA证书没有随服务器证书一起发送，客户端可能无法构建完整信任链——浏览器会报"证书不受信任"错误。

## 六、中间人攻击详解（MITM）

### 6.1 完整攻击流程

```
正常HTTPS：客户端 ←─────── 服务器
中间人攻击：客户端 ←── 攻击者 ←── 服务器

攻击步骤：
① 攻击者拦截客户端的连接请求
② 攻击者伪装成目标服务器，向客户端发送自己的证书（伪造或自签名）
③ 如果客户端不验证证书（或信任了伪造的CA），客户端会用攻击者的公钥加密预主密钥
④ 攻击者用自己的私钥解密获得预主密钥 → 掌握了会话密钥
⑤ 攻击者再以客户端身份向真正的服务器建立HTTPS连接
⑥ 攻击者成为"中间人"——两端都以为是直接通信，实际都被攻击者截获
```

### 6.2 MITM 的前提条件

- 客户端不验证/忽略证书警告
- 客户端信任了伪造的CA（如企业内网安装的内部CA被滥用）
- ARP 欺骗 + DNS 欺骗（局域网内更容易实现）
- 代理工具配置不当

### 6.3 HTTPS 如何防御 MITM

- **CA 证书链验证**：浏览器严格验证证书链，伪造证书无法通过
- **证书透明度（CT）**：所有签发的证书必须记录到公开日志，防止 CA 滥签
- **HSTS**：强制浏览器只走 HTTPS，不接受降级到 HTTP
- **证书锁定（Certificate Pinning）**：App 内置特定证书公钥，拒绝其他证书

## 七、mTLS（双向 TLS 认证）

### 7.1 什么是 mTLS？

普通 TLS 只有**客户端验证服务端身份**（单向认证）。mTLS（Mutual TLS）要求**双方都验证对方身份**——服务端也要求客户端出示证书。

```
普通TLS：客户端验证服务端证书（单向）
mTLS：  客户端验证服务端证书 + 服务端验证客户端证书（双向）
```

### 7.2 mTLS 在微服务中的应用

在微服务/Service Mesh（如 Istio）架构中：
- 服务间通信不再只靠网络边界保护
- 每个微服务有自己的证书，通信时互相验证身份
- 即使内网被入侵，攻击者没有合法证书也无法冒充服务

**Istio 的 mTLS 实现**：
- 自动为每个服务生成证书（通过 Citadel 组件）
- Sidecar 代理（Envoy）自动完成 mTLS 握手
- 开启 mTLS 后，服务间流量全部加密 + 双向认证

> [!tip] 面试加分
> mTLS 是零信任网络的核心技术之一——"不再信任网络边界，只信任身份验证"。

## 七、🔗 知识串联（模块间）

- **串联 02 TCP**：TLS 握手发生在 TCP 三次握手之后——先建立 TCP 连接，再进行 TLS 协商。所以 HTTPS 首次连接需要 TCP 1-RTT + TLS 1~2-RTT，详见 [[02-TCP连接管理/TCP连接管理]]
- **串联 04 QUIC**：QUIC 把 TCP 握手和 TLS 握手合并为一步（0-RTT 恢复），从根本上减少了连接延迟，详见 [[04-传输层协议对比/TCP vs UDP]]
- **串联 05 HTTP**：HTTPS = HTTP + TLS，HTTP 的明文问题（窃听/篡改/冒充）由 TLS 层解决，详见 [[05-HTTP协议/HTTP协议详解]]
- **串联 07 DNS**：TLS 证书绑定的是域名/IP，DNS 解析出 IP 后才能建立连接。CDN 的 SSL 终止也是在反向代理层处理 TLS，详见 [[07-网络应用技术/DNS与CDN]]
- **串联 07 Cookie**：Cookie 的 `Secure` 属性确保只在 HTTPS 下传输，JWT 通过 HTTPS 防止令牌被窃取，详见 [[07-网络应用技术/Cookie与认证]]

## 八、面试回答模板

> **面试官：说一下 HTTPS 的原理？**

"HTTPS = HTTP + SSL/TLS，核心解决 HTTP 明文传输的三个问题：窃听、篡改、冒充。

加密方案采用**混合加密**——非对称加密交换对称密钥，对称加密传输实际数据。这就像用一把昂贵的保险锁（非对称）传递一把普通的钥匙（对称），之后用这把普通钥匙开日常的所有锁。为什么不是全程非对称？因为非对称加密比对称加密慢 1000 倍以上，全程用会严重影响性能。

具体流程是 TLS 四次握手：客户端先发起请求、服务端回传证书（含公钥）、客户端用公钥加密预主密钥发给服务端、双方用三个随机数生成会话密钥。

**证书的作用**是防止中间人攻击——CA 机构用私钥签名，客户端用 CA 公钥验签，确保你在跟真实的目标服务器通信。证书有信任链：根CA→中间CA→服务器证书，浏览器逐级验证。

TLS 1.3 把握手优化到了 1-RTT 甚至 0-RTT，并且去掉了不安全的加密套件。"

> **面试官：中间人攻击是什么？HTTPS 如何防御？**

"中间人攻击是攻击者同时与客户端和服务器建立连接，截获并可能篡改通信内容。前提是客户端信任了伪造的证书——如果证书链验证严格，伪造证书无法通过，HTTPS 就能有效防御。此外还有证书透明度（CT）防止 CA 滥签、HSTS 防止降级到 HTTP、Certificate Pinning 内置特定公钥等多重保障。"
