# TCP vs UDP

> [!tip] 模块速记
> **TCP = 打电话（先接通，保证每句听清），UDP = 发短信（直接扔出去不管收没收到）。QUIC = 在 UDP 上造 TCP（扬长避短），HTTP/3 用的就是 QUIC。UDP 想可靠？应用层自己搞。**

---

## 一、核心对比表

| 对比维度        | TCP               | UDP               |
| ----------- | ----------------- | ----------------- |
| **连接**      | 面向连接（三次握手）        | 无连接，直接发           |
| **可靠性**     | 可靠传输（确认+重传）       | 不保证送达             |
| **顺序**      | 保证有序              | 不保证有序             |
| **速度**      | 慢（握手+确认开销）        | 快（无额外开销）          |
| **数据边界**    | 面向字节流（无边界）        | 面向报文（有边界）         |
| **头部大小**    | 20~60 字节          | 8 字节              |
| **流量/拥塞控制** | 有滑动窗口和拥塞控制        | 无，发就完了            |
| **适用场景**    | HTTP、FTP、SSH、SMTP | DNS、音视频、直播、游戏     |
| **比喻**      | 打电话——先接通，保证每句话都听清 | 发短信——直接发，对方收没收到不管 |

### 1.2 常见端口号速查表

> **端口号是传输层的概念**——TCP 和 UDP 通过端口号区分同一台机器上的不同应用。

| 端口 | 协议 | 说明 | 记忆 |
|------|------|------|------|
| **20/21** | FTP | 文件传输（21 控制，20 数据） | File |
| **22** | SSH | 加密远程登录 | Secure Shell |
| **23** | Telnet | 明文远程登录（已淘汰） | — |
| **25** | SMTP | 发邮件 | Send Mail |
| **53** | DNS | 域名解析 | DNS=53 |
| **80** | HTTP | 网页 | 最基础 |
| **443** | HTTPS | 加密网页 | HTTP + SSL |
| **3306** | MySQL | 数据库 | — |
| **5432** | PostgreSQL | 数据库 | — |
| **6379** | Redis | 缓存 | — |
| **8080** | HTTP 备用 | 常见开发端口 | Tomcat 默认 |
| **27017** | MongoDB | 数据库 | — |

> [!tip] 面试记忆重点
> **22(SSH)、80(HTTP)、443(HTTPS)、3306(MySQL)、6379(Redis)** — 这 5 个最常考。

**🔗 知识串联：**
- **串联 05 HTTP**：HTTP 默认端口 80，HTTPS 默认端口 443——URL 不写端口时就用这些默认值
- **串联 02 TCP**：三次握手建立连接时，目的端口就在第一个 SYN 报文中
- **串联 07 DNS**：DNS 用的 UDP 端口 53——小查询用 UDP 就够了，不需要 TCP 的可靠性开销。详见 [[07-网络应用技术/DNS与CDN]]

**🔗 知识串联（补充）：**
- **串联 06 HTTPS**：HTTPS 基于 TCP 连接（先 TCP 三次握手，再 TLS 握手），而 QUIC 把两者合并——详见 [[06-HTTPS与安全认证/HTTPS与TLS详解]]
- **串联 07 Cookie**：Cookie/JWT 等认证机制通过 HTTP Header 传递，底层依赖 TCP/QUIC 连接——详见 [[07-网络应用技术/Cookie与认证]]

## 二、UDP 场景分析

**为什么 DNS 用 UDP？**
- DNS 查询通常一个包就能搞定（请求小、响应小）
- 不需要三次握手的开销
- 即使丢包，应用层简单重试即可

**为什么直播/视频会议用 UDP？**
- 实时性比可靠性更重要——丢掉一帧画面比卡顿等重传体验好
- 音频/视频数据本身允许少量丢失

**为什么游戏用 UDP？**
- 延迟敏感——等 TCP 重传的话角色已经死了
- 游戏状态本身就是高频更新的，旧数据过期了重传也没意义

### 更多场景补充

| 场景 | 选择 | 原因 |
|------|------|------|
| IoT 传感器数据上报 | UDP | 小数据包+TCP头部开销占比大 |
| 金融交易系统 | TCP | 数据准确性不可妥协 |
| 文件传输 | TCP | 必须完整无误 |
| 实时股票行情推送 | UDP | 实时性优先，旧行情无意义 |
| 视频点播（非直播） | TCP | 可缓冲，优先保证画质 |

## 三、QUIC 协议详解

### 3.1 为什么需要 QUIC？

HTTP/2.0 解决了**应用层**的队头阻塞，但**传输层（TCP）**的队头阻塞还在——TCP 一个包丢了，后续包即使到了也得等重传。QUIC 基于 UDP 在应用层实现可靠传输，彻底消除了传输层队头阻塞。

### 3.2 QUIC 核心特性

| 特性 | 说明 | 比喻 |
|------|------|------|
| **0-RTT 握手** | 如果之前连接过，可以直接发数据 | VIP 老客户免排队 |
| **无队头阻塞** | 不同 Stream 之间互相独立，一个流丢包不影响其他 | 高速公路多车道，一条堵了其他还通 |
| **连接迁移** | 切换网络（WiFi→4G）不需要重新握手 | 手机换卡不换号，朋友还能找到你 |
| **内置 TLS 1.3** | 安全与传输合并，不再分两层握手 | 安检和检票合并一步完成 |
| **拥塞控制可插拔** | 用户空间实现，可灵活替换算法 | 发动机可以换不同型号 |

### 3.3 QUIC 如何实现可靠传输？

QUIC 在 UDP 之上实现了类似 TCP 的可靠机制：

- **包序号单调递增**：不像 TCP 的序号基于字节偏移，QUIC 的包序号严格递增——ACK 直接确认具体包号，不再依赖累计确认
- **ACK 机制**：支持更精确的 ACK 范围（ACK Range），可以同时确认多个不连续的包
- **重传包新序号**：重传的包使用新序号（不像 TCP 重传用原序号），避免歧义
- **Stream 独立**：每个 Stream 有自己的流控，一个 Stream 丢包不影响其他 Stream

**对比表**：

| 特性 | TCP | QUIC |
|------|-----|------|
| 传输层 | TCP | UDP |
| 握手 | TCP三次握手 + TLS 1.3（1-RTT） | QUIC握手（0-RTT 恢复） |
| 队头阻塞 | TCP层面存在 | 无（Stream独立） |
| 连接迁移 | 不支持（换IP需重连） | 支持（Connection ID） |
| 序号机制 | 字节偏移，重传用原序号 | 包序号单调递增，重传用新序号 |
| 拥塞控制 | 内核实现，不可插拔 | 用户空间，可插拔 |

> [!tip] 面试加分
> HTTP/3 底层用 UDP，但 QUIC 在应用层实现了类似 TCP 的可靠传输——这是"扬长避短"的经典设计：取 UDP 的灵活（无队头阻塞、连接迁移），补 UDP 的不可靠（应用层加确认/重传）。

## 四、UDP 如何实现可靠传输

既然 UDP 不保证可靠，如果业务既要 UDP 的实时性又要可靠性怎么办？——**在应用层自己实现**。

### 4.1 常见方案

| 方案 | 核心思路 | 代表 |
|------|----------|------|
| **手动 ACK + 重传** | 发送方加序号，接收方回 ACK，超时重传 | KCP |
| **前向纠错（FEC）** | 发送冗余校验数据，丢包时可恢复无需重传 | QUIC 支持 |
| **KCP 协议** | 在 UDP 上实现 ARQ（自动重传请求），牺牲 10%~20% 带宽换取更低延迟 | 游戏加速器 |
| **QUIC** | 完整的可靠 UDP 实现，Google 出品 | HTTP/3 |
| **RUDP** | 微软提出的可靠 UDP，用于游戏 | Xbox Live |

### 4.2 KCP 简介

KCP 是一个快速可靠协议，核心优化：
- 比 TCP 的 RTO 更激进（更快重传）
- 支持选择性重传（不依赖累计确认）
- 不做慢启动（直接以配置的窗口大小开始）

**一句话**：KCP = TCP 的可靠逻辑 + UDP 的低延迟心态。

## 五、IO 模型：BIO / NIO / AIO

> [!warning] 面试高频盲区！携程、字节直考。
> Java 网络编程的 IO 模型是后端面试极高频题，尤其 BIO/NIO 区别 + epoll 原理。

### 5.1 三种 IO 模型对比

| 维度 | BIO（同步阻塞） | NIO（同步非阻塞） | AIO（异步非阻塞） |
|------|-----------------|-------------------|-------------------|
| **全称** | Blocking IO | Non-blocking IO | Async IO |
| **模型** | 一个连接一个线程 | 一个线程处理多个连接（多路复用） | 操作系统完成 IO 后回调通知 |
| **比喻** | 一对一专属服务员——坐着等客人点菜 | 一个服务员看多桌（叫号制） | 自助餐做好了叫号器响 |
| **线程模型** | 每个连接独占一个线程 | 线程通过 Selector 轮询多个 Channel | 回调机制，线程不阻塞 |
| **Java API** | `ServerSocket.accept()` | `Channel + Buffer + Selector` | `AsynchronousChannel` |
| **适用场景** | 连接数少且固定 | 连接数多、请求短（Netty、Nginx） | 连接数多且操作重（文件 IO） |
| **缺点** | 连接数 = 线程数，高并发扛不住 | 编程复杂，需处理半包/粘包 | Linux 支持不完善，实际少用 |

```
BIO 模型（一个连接一个线程）：
  while(true) {
      Socket client = server.accept();     ← 阻塞！等新连接
      new Thread(() -> {
          client.getInputStream().read();  ← 阻塞！等数据
      }).start();
  }
  问题：10000 个连接 = 10000 个线程 → 内存爆炸 + 线程切换开销

NIO 模型（一个线程管多个连接）：
  Selector selector = Selector.open();
  channel.register(selector, OP_READ);    ← 注册感兴趣的事件
  while(true) {
      selector.select();                  ← 等待事件（有数据才返回）
      for (SelectionKey key : selectedKeys) {
          if (key.isReadable()) { /* 读数据 */ }
      }
  }
  优势：1 个线程可以管 10000 个连接
```

> **面试话术：** "BIO 是同步阻塞——一个连接一个线程，accept 和 read 都阻塞，高并发扛不住。NIO 是同步非阻塞——用 Selector 多路复用，一个线程管多个连接，只在有数据可读时才处理，Netty 和 Nginx 都是这个模型。AIO 是异步非阻塞——操作系统完成 IO 后回调通知，但 Linux 的 AIO 实现不完善，生产环境基本用 NIO + epoll。"

### 5.2 IO 多路复用：select / poll / epoll

**核心问题**：NIO 怎么做到一个线程管多个连接？——靠操作系统的 **IO 多路复用**机制。

| 维度 | select | poll | **epoll** |
|------|--------|------|-----------|
| **数据结构** | bitmap（位数组） | 链表（pollfd 数组） | 红黑树 + 就绪链表 |
| **最大连接数** | 1024（FD_SETSIZE） | 无限制 | 无限制 |
| **时间复杂度** | O(n) 每次遍历所有 fd | O(n) 每次遍历所有 fd | **O(1)** 只返回就绪的 fd |
| **内存拷贝** | 每次调用全量拷贝 fd 集合 | 每次调用全量拷贝 | mmap 共享内存，只在注册时拷贝一次 |
| **触发方式** | 仅水平触发（LT） | 仅水平触发（LT） | 支持 LT + **ET（边缘触发）** |
| **比喻** | 点名——每次喊所有人名字 | 排队——逐个问有没有数据 | **叫号——谁好了直接叫号** |
| **适用规模** | 少量连接 | 中等 | **万级+ 高并发** |

> [!tip] 面试核心：为什么 epoll 比 select/poll 快？
> 1. **O(1) vs O(n)**：epoll 只返回就绪的 fd，不需要遍历所有连接（select/poll 每次都要遍历全部）
> 2. **mmap 共享内存**：select/poll 每次调用都把 fd 集合从用户态拷贝到内核态；epoll 只注册时拷贝一次
> 3. **无数量限制**：select 受限于 FD_SETSIZE=1024；epoll 无上限

### 5.3 LT（水平触发）vs ET（边缘触发）

```
LT（Level Triggered，水平触发）：
  "缓冲区有数据就一直通知你"
  → 数据没读完？下次 epoll_wait 还会通知
  → 编程简单，不容易漏事件
  → select / poll 只支持 LT

ET（Edge Triggered，边缘触发）：
  "数据到来的瞬间通知你一次"
  → 只在状态变化时通知一次，之后不再通知
  → 必须一次性读完（while 循环读到 EAGAIN）
  → 效率更高，但编程复杂
  → Nginx 使用 ET 模式
```

**比喻**：
- LT = 水壶响铃——水烧开了一直响，你关掉才停
- ET = 烟花——只放一次，没看到就没了

### 5.4 实际应用

| 中间件 | IO 模型 | 为什么 |
|--------|---------|--------|
| **Nginx** | epoll ET 模式 | 单 worker 进程处理数万并发，ET 模式效率最高 |
| **Netty** | Java NIO（Linux 底层 epoll） | 封装了 NIO，加了内存池、零拷贝等优化 |
| **Redis** | 单线程 + epoll | 命令都是内存操作极快，瓶颈在网络 IO 不在 CPU |
| **Tomcat** | NIO（默认） | 从 8.0 起默认 NIO 模型 |

> **面试话术：** "epoll 是 Linux 的 IO 多路复用机制，比 select/poll 快三个原因：O(1) 只返回就绪 fd 不用全量遍历、mmap 共享内存不用每次全量拷贝、无连接数限制。epoll 支持 LT 和 ET 两种触发模式，Nginx 用 ET 模式效率更高但编程更复杂。Java NIO 在 Linux 底层就是 epoll，Netty 在此基础上封装了内存池、零拷贝等优化。"

- **串联 02 TCP**：NIO 处理的就是 TCP 连接——Selector 监听的 Channel 对应 TCP Socket，详见 [[02-TCP连接管理/TCP连接管理]]

## 六、面试回答模板

> **面试官：说一下 TCP 和 UDP 的区别，什么场景用哪个？**

"TCP 是面向连接的、可靠的、有序的传输协议，像打电话——先拨通、确认对方在、然后逐句对话，每句话都确保对方听清。UDP 是无连接的、不保证送达的、无序的，像发短信或寄明信片——直接扔出去，对方收到没收到不管。

**TCP 适用**：对数据准确性要求高的场景，比如 HTTP 网页传输、文件下载、邮件、SSH 远程登录——少一个字节都不行。

**UDP 适用**：对实时性要求高、允许少量丢包的场景，比如直播、视频会议、在线游戏、DNS 查询——丢一帧画面比卡顿两秒好多了。

另外 TCP 头部最小 20 字节，UDP 只有 8 字节，对于小数据包场景（如 IoT 传感器数据）UDP 更高效。"

> **面试官：QUIC 和 TCP 的区别？为什么 HTTP/3 用 QUIC？**

"QUIC 基于 UDP 但在应用层实现了可靠传输——取了 UDP 的灵活（无队头阻塞、连接迁移），又补了 UDP 的不可靠（加确认/重传）。HTTP/3 用 QUIC 是因为 HTTP/2.0 只解决了应用层队头阻塞，TCP 层的队头阻塞还在：一个 TCP 包丢了会堵住所有 Stream。QUIC 每个 Stream 独立，一个丢包不影响其他流。再加上 QUIC 支持 0-RTT 恢复和连接迁移（WiFi 切 4G 不用重连），这些都是 TCP 做不到的。"
