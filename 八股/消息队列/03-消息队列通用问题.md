# 消息队列通用问题

> 无论 RabbitMQ 还是 Kafka，都会遇到：消息积压、消息乱序、消息重复。本章讲通用解决思路。

---

## 一、消息积压处理

> 面试追问："线上消息积压了几百万条怎么办？"

### 1.0 先理解：积压的本质是什么？

```
积压 = 生产速度 > 消费速度，持续一段时间

根本原因只有三类：
  A. 消费能力不足（最常见）
     → 消费者线程太少 / 处理逻辑慢（慢SQL、下游RT高）/ 消费者挂了
  B. 生产流量暴涨（突发）
     → 秒杀/大促/上游疯狂重试
  C. 故障（偶发）
     → 消费者假死 / 网络分区 / Broker 压力过大拒绝服务
```

**关键认知：Kafka 和 RabbitMQ 的积压处理思路完全不同！**

| 维度 | RabbitMQ | Kafka |
|------|----------|-------|
| **积压表现** | Queue 中 Ready 消息数飙升 | Consumer Lag（消费落后生产多少条） |
| **能否动态扩容消费者** | ✅ 可以，直接加消费者监听同一队列 | ⚠️ 受限于 Partition 数（一个 Partition 只能一个消费者线程） |
| **扩容限制** | 无硬限制 | **消费者数 ≤ Partition 数** |
| **积压后的影响** | 内存压力、可能触发 flow control | 磁盘压力（Kafka 存磁盘，不太怕积压） |
| **监控指标** | `rabbitmq_queue_messages_ready` | `kafka_consumer_group_lag` |

### 1.1 紧急处理方案（三级应急响应）

| 优先级 | 动作 | 说明 |
|--------|------|------|
| 🔴 第一 | **扩容消费者** | 加实例、加线程，最快见效 |
| 🟡 第二 | **跳过非关键消息** | 非核心消息 discarding，全力保核心 |
| 🟢 第三 | **拆分队列** | 原队列分片到 N 个临时队列，N 组消费者并行 |

#### 第一优先：快速恢复消费能力

```yaml
临时扩容方案:
  1. 增加消费者实例（水平扩容）
  2. 增加单个消费者的并发线程数
  3. 重启有问题的消费者（可能只是假死）
```

#### 第二优先：跳过非关键消息

```java
// 开启批量消费，只处理关键业务消息
if (不是核心业务消息) {
    channel.basicReject(deliveryTag, false); // 不入队，进DLX
    return;
}
processCoreMessage(message);
```

#### 第三优先：紧急扩容队列

```
原队列 → 临时转发程序 → 分片到 N 个临时队列 → N 组消费者并行处理

适用场景：消费者数已经到上限（Kafka 中 消费者数 = Partition 数了）
操作方式：新建临时 Topic（更多 Partition）→ 原消费者转投 → 批量启动消费者
```

### 1.2 Kafka 积压专项：lag = 100 万的排障 SOP

```
凌晨 2:00 报警：order-consumer-group lag 突破 100 万

Step 1 — 确认故障面（1 分钟）:
  kafka-consumer-groups --bootstrap-server xxx \
    --group order-consumer-group --describe
  → 确认是所有 Partition 都积压，还是个别 Partition

Step 2 — 查消费者是否活着（1 分钟）:
  → 查看消费者实例的 JVM 线程 / GC 日志 / 错误日志
  → 如果消费者假死：重启！最快见效

Step 3 — 定位消费慢的原因（核心）:
  3a. 消费者在处理什么？
      → 看日志中最新的消费 offset 对应的时间戳
      → 如果消费者已经在处理"1分钟前"的消息 → 消费速度勉强跟上
      → 如果还在处理"30分钟前"的消息 → 严重滞后
  3b. 处理逻辑哪里慢？
      → 查 APM（如 SkyWalking/Pinpoint）：看消费方法的耗时分布
      → 常见根因：慢SQL（缺索引/锁冲突）/ 下游RPC超时 / 消息处理中有同步阻塞IO
      → 看 GC：是否频繁 Full GC 导致 STW

Step 4 — 止血（对因下药）:
  情况A: 消费者线程太少
    → 先确认 Partition 数（kafka-topics --describe）
    → 增加消费者线程（单个消费者多线程消费不同 Partition）
    → 如果 Partition 不够：紧急增加 Partition（kafka-topics --alter）
      ⚠️ 注意：增加 Partition 后，有 key 的消息路由会变！

  情况B: 处理逻辑慢（慢 SQL / 下游超时）
    → 先跳过非关键消息（if 非核心 → commit offset 跳过）
    → 紧急调大下游超时时间 or 加缓存
    → 事后优化 SQL / 加索引 / 异步化

  情况C: 消费者硬件资源不足
    → 水平扩容消费者实例（K8s 加 Pod / 加机器）
    → 调整 JVM 参数（堆内存 / GC 策略）

Step 5 — 验证恢复:
  → 持续观察 lag 是否下降
  → lag 降为 0 → 告警恢复
```

### 1.3 RabbitMQ 积压的独特处理

```bash
# RabbitMQ 积压时查看队列状态
rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers

# 如果 consumers = 0 → 没有消费者监听！赶紧启动
# 如果 messages_unacknowledged 很高 → 消费者在处理但没 ACK，prefetch 太大 or 处理太慢
# 如果 messages_ready 暴涨 → 生产太快，需要扩容消费者
```

### 1.4 预防措施

| 措施 | 说明 |
|------|------|
| **监控告警** | 队列深度 > 阈值 → 立即告警 |
| **容量规划** | 提前评估峰值 QPS，预留 3-5 倍消费能力 |
| **消息过期** | 给队列设置 TTL，过期自动清理无用消息 |
| **消费者备集群** | 备好一组消费者实例，随时启动应对突发 |
| **降级开关** | 紧急情况可以丢弃非核心消息 |

## 二、消息顺序性

> "消息队列能保证消息顺序吗？"——经典坑题，答不好直接暴露水平。

### 2.0 核心结论（先记这句话）

> **同一 Partition / Queue 内严格有序，跨 Partition / 多消费者消费同一队列则无序。全局有序 = 牺牲并行度 = 只用一个 Partition。**

### 2.1 为什么会乱序？

```
Queue: [msg1, msg2, msg3]
  ├── Consumer-A 拿到 msg1（处理慢，要 5 秒）
  ├── Consumer-B 拿到 msg2（处理快，0.1 秒）
  └── Consumer-C 拿到 msg3
结果：msg2 先处理完，msg1 后处理完 → 乱序！
```

**结论：单队列 + 单消费者一定有序，多消费者一定无序。**

### 2.2 Kafka 顺序性专项

```
Topic: order-events (3 Partitions)
  Partition 0: [msg1_usrA, msg2_usrA, msg3_usrA]  ← usrA 的消息，有序 ✅
  Partition 1: [msg1_usrB, msg2_usrB]              ← usrB 的消息，有序 ✅
  Partition 2: [msg1_usrC, msg2_usrC]              ← usrC 的消息，有序 ✅

  → Producer 侧：用 userId 做 key → hash 到固定 Partition
  → Consumer 侧：每 Partition 只有一个线程消费
  → 结果：同一用户的消息在单个 Partition 内有序，不同用户之间无所谓顺序
```

**Kafka 保证顺序的四个关键点（缺一不可）：**

| 关键点 | 配置/做法 | 说明 |
|--------|----------|------|
| 1. **生产者按 key 分区** | `producer.send(record, callback)` 时指定 key | 同一 key → 同一 Partition |
| 2. **max.in.flight.requests.per.connection=1** | 同步发送 or 限制并发 | 防止发送端重试导致乱序 |
| 3. **消费者单线程处理** | 一个 Partition 只被一个线程消费 | 多线程消费同一 Partition → 乱序 |
| 4. **关闭自动提交** | `enable.auto.commit=false` | 处理成功再提交 offset |

```
⚠️ Kafka 最容易踩的坑：
  开启了 enable.auto.commit=true, 且 max.poll.records=500

  消费者一次拉 500 条 [msg1~msg500]
  处理 msg1 时抛异常 → 但 auto-commit 已经把 offset 提交到 500 了
  → 重启后从 501 开始消费 → msg1~msg500 永远丢了！
```

### 2.3 RabbitMQ 的顺序特性

| 场景 | 是否有序 | 原因 |
|------|---------|------|
| **单生产者 + 单Exchange + 单Queue + 单Consumer** | ✅ 有序 | 天然 FIFO |
| 多 Consumer 消费同一 Queue | ❌ 无序 | 并发消费，处理速度不同 |
| 消息有重试/重入队 | ❌ 无序 | 重试消息可能排队到后面 |
| 消息优先级 | ❌ 无序 | 高优先级消息插队 |

### 2.4 保证顺序的常用方案

#### 方案一：单队列单消费者（全局有序）

```
「创建订单」→「扣库存」→「支付」
       ↓ 同一队列，单消费者串行处理
   Queue: order_queue → Consumer(单线程)
```

> 缺点：吞吐量低，一个消费者处理所有消息。**Kafka 等价于整个 Topic 只用一个 Partition。**

#### 方案二：按业务 Key 哈希分区（局部有序，推荐 ✅）

```
用户1的消息 → hash(userId=1) % 3 = 0 → Partition_0 → Consumer_0
用户2的消息 → hash(userId=2) % 3 = 1 → Partition_1 → Consumer_1
用户3的消息 → hash(userId=3) % 3 = 2 → Partition_2 → Consumer_2
```

> **同一用户的消息始终落在同一个分区，由同一个消费者处理，保证局部有序。全局不强求有序。**

#### 方案三：内存排序 + 乱序容忍

```java
// 消费者收到消息后不立即处理，先按时间戳/序列号排序
ConcurrentSkipListMap<Long, Message> buffer = new ConcurrentSkipListMap<>();

void onMessage(Message msg) {
    buffer.put(msg.getSequence(), msg);
    // 只处理连续的序列号
    while (buffer.containsKey(expectedSeq++)) {
        process(buffer.pollFirstEntry().getValue());
    }
}
```
> 适用于消息偶尔乱序、可短暂缓冲的场景（如 IM 消息）。

#### 方案四：幂等化业务（终极方案）

> **与其费力保证顺序，不如让业务不在乎顺序。**
> 订单状态机：`待支付 → 已支付 → 已发货`，收到"已发货"时发现还是"待支付"状态，忽略 or 暂存，等"已支付"到了再推进。

### 2.5 真实场景：什么时候必须全局有序？

| 场景 | 是否需要全局有序 | 推荐方案 |
|------|----------------|----------|
| 用户聊天消息（A→B） | 只需同一对话有序 | **方案二（按对话 ID 分区）** ✅ |
| 数据库 binlog 同步 | **必须全局有序** | **方案一（单 Partition）** ⚠️ |
| 电商订单状态流转 | 只需同一订单有序 | **方案二（按订单 ID 分区）** ✅ |
| 秒杀库存扣减 | 必须全局有序（不超卖） | **方案一 or 用 Redis 原子操作** |
| 日志收集 | **不需要** | 随便发，性能优先 |

> 🔑 **面试关键认知：** "绝大多数业务场景不需要全局有序——只要同一实体（用户/订单/账户）的消息在它自己的 Partition 里有序就够了。全局有序是极端需求，代价是吞吐量降到单机水平。"

## 三、消息重复消费（幂等性）

### 3.1 重复消费的根因 — 图解

```
正常流程:
  Consumer 拉消息 → 处理成功 → 发 ACK → Broker 删消息

重复消费的三种经典场景:
  场景 A: Consumer 处理成功，ACK 因网络超时未到达 Broker
          → Broker 未收到 ACK → 重新投递 → Consumer 再次处理 ❌

  场景 B: Consumer 处理到一半挂了（OOM / kill -9 / 机器宕机）
          → Broker 未收到 ACK → 重新投递给另一个 Consumer → 重复处理

  场景 C: Producer 重发（Confirm 超时，Producer 不确定 Broker 收没收到）
          → Broker 收到两条相同内容的消息 → 两个 Consumer 各处理一遍

  场景 D [Kafka 特有问题]: Rebalance 时 offset 未提交
          → Consumer 被踢出 Group → 新 Consumer 从上次提交的 offset 开始
          → 重复消费上次提交到 Rebalance 之间的消息
```

**一个残酷的事实（面试金句）：**
> "在分布式系统中，消息至少投递一次（at-least-once）是常态，exactly-once 需要生产者+Broker+消费者三方协作才能实现。与其追求中间件的 exactly-once，不如让业务做到幂等——这是成本最低、最可靠的方案。"

### 3.2 幂等设计方案深度对比

| 方案 | 实现 | 可靠性 | 性能 | 适用场景 |
|------|------|--------|------|----------|
| **DB 唯一键** | `INSERT INTO consumed_record(msg_id)` + 唯一索引 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 需要强一致的金融/支付 |
| **Redis SETNX** | `setIfAbsent(msgId, 1, 24h)` | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 高并发业务首选 |
| **业务状态机** | 状态 `待支付→已支付→已发货` | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 有明确状态流转的实体 |
| **版本号/乐观锁** | `UPDATE SET v=v+1 WHERE id=? AND v=?` | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 并发更新同一实体的场景 |

### 3.2+ 每种方案的真实代码与坑

#### 方案一：DB 唯一键 — 注意"非幂等"操作

```java
// ❌ 错误写法：INSERT 成功后再做业务
INSERT INTO consumed_record VALUES ('msg_123');
// → 如果下面业务失败，消息已经"被标记为已消费"，无法重试！
doBusinessLogic();
return "success";

// ✅ 正确写法：业务和消息记录在同一个本地事务中
@Transactional
public void consume(Message msg) {
    // 1. 唯一键防重（利用数据库唯一约束）
    consumedMapper.insert(msg.getId());  // 重复则抛 DuplicateKeyException
    // 2. 执行业务逻辑
    orderService.cancel(msg.getOrderId());
    // 3. 事务提交 → 要么都成功，要么都回滚
}
```

#### 方案二：Redis SETNX — 注意过期时间和故障转移

```java
String msgId = message.getMsgId();
String key = "msg:consumed:" + msgId;

// SETNX + 设置过期时间（原子操作，Redis 2.6.12+）
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent(key, "1", Duration.ofHours(24));

if (Boolean.FALSE.equals(success)) {
    // 已经消费过，直接 ACK 跳过
    channel.basicAck(deliveryTag, false);
    return;
}

try {
    processMessage(message);
    channel.basicAck(deliveryTag, false);
} catch (Exception e) {
    // ⚠️ 处理失败必须删除 Redis 记录！否则消息永远无法重试
    redisTemplate.delete(key);
    channel.basicNack(deliveryTag, false, true); // 重新入队
}
```

**Redis 方案的风险：**
- `过期时间` 设太短：消息还没重试完 key 过期了 → 重复消费
- `Redis 主从切换`：主节点宕机，从节点还未同步 → 防重 key 丢失
- `Redis 集群模式`：不保证强一致 → 小概率丢失

#### 方案三：业务状态机 — 最优雅的方案

```java
// 订单取消幂等：无论消息来多少次，只取消一次
public void cancelOrder(String orderId) {
    Order order = orderMapper.selectById(orderId);

    // 只有"待支付"才能取消
    if (order.getStatus() != OrderStatus.PENDING) {
        // 已取消 / 已支付 / 已发货 → 直接返回成功（幂等）
        log.info("订单 {} 当前状态 {}，无需取消", orderId, order.getStatus());
        return;
    }

    // CAS 更新：WHERE status='PENDING'
    int rows = orderMapper.cancelIfPending(orderId);
    if (rows > 0) {
        // 取消成功 → 还库存
        inventoryService.restore(orderId);
    }
    // rows=0 → 被另一个消费者抢先取消了，幂等成功
}
```

### 3.2++ Kafka 的 Exactly-Once 语义

| 语义 | 含义 | 实现方式 |
|------|------|----------|
| **at-most-once** | 最多一次（可能丢） | `auto.commit=true`，拉了就提交 |
| **at-least-once** | 至少一次（可能重复） | `auto.commit=false`，处理完再提交（默认） |
| **exactly-once** | 精确一次 | **幂等生产者** + **事务** + **消费者幂等** |

```
Kafka Exactly-Once 的局限（面试必说）：
  ✅ 做到了"Kafka 内部"的精确一次
  ❌ 做不到"端到端"的精确一次
      → 消费者写 DB 时如果 DB 操作重复了，Kafka 管不了

  结论：Exactly-Once 靠 Kafka 只能保证一部分，业务幂等才是兜底方案。
```

### 3.3 综合方案（三层防护推荐）

```
┌─────────────────────────────────────────────┐
│         幂等防护链（多层保障）                 │
├─────────────────────────────────────────────┤
│  第一层：Redis SETNX                          │
│    - 消息 ID + TTL 24 小时                    │
│    - 快速过滤 99% 重复消息                     │
│                                              │
│  第二层：数据库唯一约束                         │
│    - msg_id 做唯一索引                         │
│    - 兜底保障，防止 Redis 数据丢失              │
│                                              │
│  第三层：业务状态判断                          │
│    - 检查当前状态是否允许本次操作               │
│    - 例如：已取消订单不允许再次取消             │
└─────────────────────────────────────────────┘
```

## 四、面试真题与回答框架

### 🔴 Q1：线上 Kafka 积压 100 万条，你怎么排查和解决？

> 频率：⭐⭐⭐⭐⭐ | 字节/美团必问题

**回答框架（5 步排查法）：**

> "我先确认是全局积压还是个别 Partition（`kafka-consumer-groups --describe`）。然后分三步：**查活着没 → 查慢在哪 → 对因下药**：
>
> 1. 消费者是否假死？看 JVM 线程 / GC / 错误日志
> 2. 消费慢的原因？看 APM 定位是慢 SQL、下游超时、还是 Full GC
> 3. 止血：慢 SQL → 跳过非关键消息；线程不够 → 扩容消费者；下游慢 → 调大超时 / 加缓存
> 4. 如果消费者数 = Partition 数了还不够 → 紧急加 Partition + 启动更多消费者
> 5. 事后加监控：lag > 阈值告警"

---

### 🔴 Q2：Kafka 能保证消息顺序吗？怎么做的？

> 频率：⭐⭐⭐⭐⭐

> "同一 Partition 内严格有序，跨 Partition 无序。保证顺序需要四个条件缺一不可：
> ① Producer 按 key 分区 → 同一 key 进同一 Partition
> ② max.in.flight.requests.per.connection=1 → 防止发送端重试乱序
> ③ 一 Partition 一消费线程 → 防止并发消费乱序
> ④ 手动提交 offset → 防止自动提交丢消息导致逻辑乱序"

**追问：一定要全局有序怎么办？**
> "整个 Topic 只用一个 Partition → 吞吐量降到单机。binlog 同步是少数必须全局有序的场景。"

---

### 🔴 Q3：消息重复消费了怎么办？幂等怎么设计？

> 频率：⭐⭐⭐⭐⭐

> "幂等三层防护：第一层 Redis SETNX 快速过滤 99%、第二层 DB 唯一键兜底、第三层业务状态机天然幂等。注意 Redis 方案要处理好异常回退——处理失败要删 key 允许重试。"

---

### 🔴 Q4：Kafka 的 exactly-once 和业务幂等有什么区别？

> 频率：⭐⭐⭐⭐ | 阿里喜欢问

> "Kafka exactly-once 保证的是 Kafka 内部（生产 → Broker → 消费提交 offset）的精确一次，但**管不了消费者写 DB 的重复**。端到端 exactly-once 需要业务幂等兜底。所以我的观点是：**与其追求中间件的 exactly-once，不如让业务逻辑天然幂等，成本更低、更可靠。**"

---

### 🟡 Q5：什么场景需要全局有序，什么场景局部有序就够？

| 场景 | 需要全局？ | 原因 |
|------|-----------|------|
| binlog 同步 | ✅ 必须 | 操作顺序错一个 → 数据永久不一致 |
| 秒杀扣库存 | ✅ 必须 | 超卖不可接受 |
| 订单状态流转 | ❌ 局部即可 | 只需要同一订单有序 |
| IM 消息 | ❌ 局部即可 | 只需要同一对话有序 |
| 日志收集 | ❌ 不需要 | 乱序不影响分析 |

---

### 🟡 Q6：多消费者消费同一队列为什么会乱序？怎么解决？

> "因为不同消费者的处理速度不同——A 处理慢 B 处理快，B 先完成。解决方案：**按业务 key 分区 + 单 Partition 单线程消费**，让同一实体的消息只被一个线程处理。"

---

### 🟢 Q7：Redis SETNX 做幂等有什么坑？

1. **处理失败要删 key**：否则永远无法重试那条消息
2. **TTL 不能太短**：消息可能因积压延迟数小时消费
3. **Redis 主从切换**：异步复制可能丢 key → 需要 DB 兜底
4. **集群模式下不保证强一致**：极小概率丢 key → DB 兜底

## 面试前 5 分钟速览

**积压处理（5 步 SOP）：**
- [ ] 先确认范围（全局 or 个别 Partition）→ 查消费者活着没 → 定位根因（慢SQL/下游/GC）
- [ ] 止血：扩容消费者 → 跳过非关键 → 紧急加 Partition。Kafka 消费者数 ≤ Partition 数
- [ ] 事后：lag 监控告警 + 容量规划 + 消费逻辑优化

**消息顺序：**
- [ ] 核心结论：**同一 Partition 内有序，跨 Partition 无序**
- [ ] Kafka 保证顺序四条件：key 分区 + max.in.flight=1 + 单线程消费 + 手动提交
- [ ] 方案：按业务 key 分区 → 局部有序，99% 场景够用
- [ ] 全局有序 = 单 Partition = 吞吐降到单机水平，只有 binlog 等场景才需要

**消息重复（幂等）：**
- [ ] 根因：ACK 超时 / 消费者崩溃 / Producer 重发 / Rebalance
- [ ] 三层防护：Redis SETNX（快）→ DB 唯一键（稳）→ 业务状态机（美）
- [ ] Redis 防重注意：失败要删 key！TTL 不能太短！需要 DB 兜底！
- [ ] exactly-once ≠ 端到端：Kafka 管不到消费者写 DB 的重复
