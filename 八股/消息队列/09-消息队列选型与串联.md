# 消息队列选型与串联

> 面试官："你为什么选 XXX 而不是 YYY？"——这是必考题。

---

## 一、RabbitMQ vs Kafka 全方位对比

| 维度 | RabbitMQ | Kafka |
|------|----------|-------|
| **定位** | 消息代理，AMQP 标准实现 | 分布式流平台，日志系统 |
| **消息模型** | Exchange → Queue（灵活路由） | Topic → Partition（顺序日志） |
| **推拉模式** | **Push**（Broker 推给 Consumer） | **Pull**（Consumer 主动拉取） |
| **吞吐量** | 万级/秒 | **百万级/秒** |
| **延迟** | **微秒级** | 毫秒级 |
| **持久化** | 内存+磁盘 | **磁盘顺序写入** |
| **消息回溯** | ❌ 消费即删除 | **✅ 按 offset 回溯** |
| **消息顺序** | 单 Queue 有序 | **Partition 内严格有序** |
| **路由能力** | **极强**（4 种 Exchange） | 弱（仅通过 Partition） |
| **集群扩展** | 对等集群，扩展性一般 | **水平扩展极强**，Controller 架构 |
| **协议** | AMQP 0-9-1 | 自定义二进制协议 |
| **生态** | 运维工具成熟 | Kafka Streams / Connect 生态丰富 |

## 二、Push vs Pull 深度对比

| | Push（RabbitMQ） | Pull（Kafka） |
|---|---|---|
| **发起方** | Broker 主动推送 | Consumer 主动拉取 |
| **优点** | 实时性好，消息立即送达 | Consumer 按自己能力消费，不会被打垮 |
| **缺点** | Consumer 处理慢会被消息淹没 | 有拉取间隔延迟，需调优 polling |
| **流量控制** | 需要设置 QoS（prefetch count） | Consumer 控制拉取速率本身是天然的流控 |
| **CPU 占用** | Consumer 空闲时几乎为零 | 需要轮询，有空转开销 |

**RabbitMQ QoS 配置（防止 Consumer 被淹没）：**

```java
channel.basicQos(1);  // 每次只推 1 条，处理完 ACK 后再推下一条
```

## 三、选型建议

```
选 RabbitMQ 的场景:
  ✅ 需要复杂路由（按不同规则分发到不同队列）
  ✅ 需要低延迟（微秒级）
  ✅ 企业应用集成，需要 AMQP 标准协议
  ✅ 中小规模，运维简单
  ✅ 需要消息确认、事务等完善的消息协议支持

选 Kafka 的场景:
  ✅ 海量日志采集和分发（ELK）
  ✅ 大数据流式计算（Flink/Spark）
  ✅ 需要消息回溯（重新消费历史数据）
  ✅ 超高吞吐（百万 QPS）
  ✅ 事件溯源架构
```

## 四、跨题串联

### 4.1 消息队列 vs Redis Stream（Redis 八股串联）

| 维度 | 消息队列（RabbitMQ/Kafka） | Redis Stream |
|------|---------------------------|--------------|
| 定位 | 专业消息中间件 | Redis 的轻量 MQ 功能 |
| 持久化 | 专业持久化（磁盘+副本） | RDB/AOF，数据可能丢失 |
| 路由 | Exchange 灵活路由 | 无路由能力 |
| 消费组 | Kafka 消费组 | Stream 消费组（类 Kafka） |
| 适用场景 | 核心业务消息 | 轻量级、快速接入 |

> **面试中如何串联：** "除了 RabbitMQ 和 Kafka，Redis Stream 也是一种选择。如果项目已经在用 Redis，且消息量不大、对可靠性要求不是极高，用 Redis Stream 可以降低架构复杂度。"

### 4.2 消息队列 vs 微服务（微服务八股串联）

- **服务间通信：** 同步调用（HTTP/RPC）→ 异步消息 → 降低服务耦合
- **分布式事务：** 消息队列+本地事务表 → 最终一致性 → 替代 Seata
- **事件驱动架构：** 订单服务发出"订单创建"事件 → 库存、积分、通知服务各自订阅
- **CQRS：** 写操作通过消息队列异步更新读模型

### 4.3 消息队列 vs 数据库（MySQL 八股串联）

- 消息队列不能替代数据库：消息消费完就没了
- 数据库不能替代消息队列：轮询/触发器无法满足高吞吐低延迟
- **事务消息模式：** 本地事务中同时写业务数据和消息表，保证一致性

## 六、面试话术模板

### 话术 1：如何保证消息不丢失

> "对于消息可靠性，我们从**生产、存储、消费**三个阶段做了全链路保障。
>
> **生产端：** 开启了 Publisher Confirm 机制，消息发送后异步等待 Broker 的确认回执，没有确认的进行重试。
>
> **存储端：** 队列和消息都设置了持久化（durable=true, deliveryMode=2），配合镜像队列或 Kafka 的多副本机制（acks=all, min.insync.replicas >= 2），确保 Leader 宕机也能切换。
>
> **消费端：** 关闭自动 ACK，改为手动确认。只有在业务处理成功后（比如数据库写入成功）才调用 basicAck，如果处理失败则 basicNack 重新入队。"

### 话术 2：RabbitMQ 和 Kafka 怎么选

> "这两个虽然都是消息中间件，但定位其实不同。
>
> RabbitMQ 是标准的 AMQP 消息代理，强项在于**复杂路由**和**低延迟**。比如订单系统的多级分发，direct、topic、fanout 四种交换机可以灵活组合。同时它是 Push 模式，消息实时性好，延迟在微秒级。
>
> Kafka 本质上是一个**分布式流平台**，核心是 append-only 的日志存储。它的强项是**海量吞吐**（百万级 QPS）和**消息回溯**能力。日志采集、实时计算、事件溯源这些场景，Kafka 是事实标准。
>
> 我们项目里有订单消息分发，路由规则比较复杂，而且对延迟要求高，所以选了 RabbitMQ。"

## 七、记忆口诀

```
削峰解耦一致性，三种场景记得清
交换队列绑路由，四种类型要分明
direct 精确 topic 模，fanout 广播 headers 少
persist 消息加队列，manualAck 手动做
死信 TTL 队列满，拒收消息也能进
积压紧急扩消费，降级跳过非关键
幂等全靠唯一键，Redis 记录 DB 查
Kafka 分区分段序，消费组内一人一
ISR 副本保可靠，acks 等于 all 最安全
Push 实时 Pull 自主，选型要看需求面
```

### 面试前快速回顾清单

- [ ] 三大用途：削峰、解耦、最终一致性
- [ ] 四种 Exchange：direct/topic/fanout/headers
- [ ] 三步防丢：Publisher Confirm → 持久化 → 手动 ACK
- [ ] 幂等三板斧：Redis SETNX + DB 唯一键 + 业务状态机
- [ ] 死信三来源：过期/拒收/队列满
- [ ] Kafka 四核心：Partition/Offset/ConsumerGroup/ISR
- [ ] 选型口诀：路由低延迟选 Rabbit，高吞吐回溯选 Kafka
