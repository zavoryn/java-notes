## 1. Building Effective Agents（构建高效的 AI Agent）

**来源**: Anthropic | **类型**: 经典基础文章

**总结**: 这是 Anthropic 发布的关于 AI Agent 设计模式的奠基性文章。文章从 LLM（大语言模型）出发，阐述了从简单到复杂的五种 Agent 工作流模式：

1. **Prompt Chaining（提示链）**: 将任务分解为多个步骤，每步的输出作为下一步的输入，可在中间加入检查点
2. **Routing（路由）**: 根据输入内容分类，将请求导向不同的处理路径
3. **Parallelization（并行化）**: 同时执行多个任务（分段或投票），提高效率
4. **Orchestrator-Workers（编排者-工人）**: 一个中心 Agent 动态分解任务并分配给多个子 Agent
5. **Evaluator-Optimizer（评估者-优化者）**: 一个 Agent 生成输出，另一个 Agent 评估并提供反馈，循环迭代

**核心观点**: 不要一开始就构建复杂的多 Agent 系统，而是从最简单的方案开始，只在确实需要时才增加复杂性。Agent 的价值在于其灵活性和适应性，而非复杂度。

## 2. Effective Context Engineering for AI Agents（AI Agent 的有效上下文工程）

**来源**: Anthropic | **类型**: 技术深度文章

**总结**: 本文提出"上下文工程"（Context Engineering）的概念，认为它比传统的"提示工程"（Prompt Engineering）更为重要和全面。上下文工程涵盖了所有将 Agent 所需信息放入其上下文窗口的技术。

**关键技术和策略**:

- **上下文压缩（Compaction）**: 当上下文窗口快满时，自动压缩历史信息，保留关键内容
- **结构化笔记（Structured Note-taking）**: 让 Agent 在工作过程中持续维护结构化的笔记，而非依赖原始对话历史
- **子 Agent 架构**: 通过创建独立的子 Agent 来分散上下文压力，每个子 Agent 有自己的上下文窗口
- **即时检索（Just-in-time Retrieval）**: 只在需要时才检索和加载相关信息，避免上下文窗口的浪费

**核心观点**: 上下文窗口是 Agent 最宝贵的资源。高效的上下文管理是构建实用 Agent 的关键，需要像管理内存一样精心管理 Agent 的上下文。

## 3. Don't Build Multi-Agents（不要构建多 Agent 系统）

**来源**: Cognition | **类型**: 观点文章 / 反面观点

**总结**: Cognition（Devin 的开发公司）发表了这篇具有争议性的文章，主张在大多数情况下不应构建多 Agent 系统。主要论点包括：

- **上下文共享问题**: 多个 Agent 之间共享上下文非常困难，信息传递过程中容易丢失关键细节（"电话游戏"问题）
- **协调成本高**: 多 Agent 系统的协调和通信开销往往超过其带来的并行收益
- **调试困难**: 多 Agent 系统的行为更难预测和调试，错误可能在 Agent 之间传播和放大
- **单体 Agent 更可控**: 一个精心设计的单体 Agent 通常比多个协作 Agent 更可靠

**核心观点**: 多 Agent 系统在理论上有吸引力，但在实践中，单体 Agent 加上良好的工具使用和上下文管理通常更有效。只有在任务确实可以完美并行且子任务间无依赖时，多 Agent 才值得考虑。

## 4. How We Built Our Multi-Agent Research System（我们如何构建多 Agent 研究系统）

**来源**: Anthropic | **类型**: 工程实践 / 案例分析

**总结**: 本文详细介绍了 Anthropic 构建 Claude Research 功能的多 Agent 架构经验。系统采用编排者-工人（Orchestrator-Worker）模式，由 Claude Opus 4 作为主导 Agent，Claude Sonnet 4 作为子 Agent。

**关键数据和发现**:

- 多 Agent 系统比单 Agent 系统性能提升 **90.2%**
- Token 使用量解释了 BrowseComp 评测中 **80%** 的性能差异
- 多 Agent 系统使用约比普通对话多 **15×** 的 token

**八条提示工程原则**:
1. 站在 Agent 的角度思考
2. 教会编排者如何分配任务
3. 根据查询复杂度调整投入
4. 工具设计和选择至关重要
5. 让 Agent 自我改进
6. 先广搜再聚焦
7. 引导思考过程
8. 并行工具调用大幅提升速度

**工程挑战**: Agent 有状态且错误会累积；调试需要新方法；部署需要彩虹部署策略；同步执行是瓶颈。

## 5. Code Execution with MCP（通过代码执行提升 MCP 效率）

**来源**: Anthropic | **类型**: 技术方案文章

**总结**: 本文探讨了如何通过代码执行环境来提升 MCP（Model Context Protocol）的效率。当 Agent 连接大量 MCP 服务器时，传统方式会导致严重的 token 浪费。

**核心问题**:
- 工具定义一次性加载到上下文窗口，消耗大量 token
- 中间结果必须经过模型处理，大数据会反复消耗上下文

**解决方案**: 让 Agent 通过编写代码来调用 MCP 工具，而非直接工具调用：

- **渐进式披露**: Agent 按需加载工具定义，而非一次性全部加载
- **上下文高效的结果处理**: 在执行环境中过滤和转换数据后再返回
- **隐私保护**: 中间结果默认留在执行环境中，支持 PII 脱敏
- **状态持久化和技能**: Agent 可以保存代码和状态供后续使用

**效果**: 将 token 使用从 150,000 降低到 2,000，节省 **98.7%**。

## 6. Practices for Governing Agentic AI Systems（治理 Agentic AI 系统的最佳实践）

**来源**: OpenAI | **类型**: 政策/治理白皮书

**总结**: 这篇白皮书提出了 Agentic AI 系统的定义和治理框架。

**Agentic AI 的四个维度**:
1. **目标复杂性**: 系统能处理多复杂的目标
2. **环境复杂性**: 系统能在多复杂的环境中运作
3. **适应性**: 系统对新情况的自适应能力
4. **独立执行**: 系统在多大程度上可以独立完成任务

**三类参与方**: 模型开发者、系统部署者、用户

**七项安全治理实践**:
1. 评估任务适合性
2. 限制行动空间并要求审批
3. 设置 Agent 默认行为
4. Agent 活动的可读性（透明度）
5. 自动监控
6. 可归因性
7. 可中断性和保持控制

**间接影响**: 采用竞赛风险、劳动力替代、攻防平衡变化、关联故障。

## 7. Introducing Deep Research（介绍 Deep Research）

**来源**: OpenAI | **类型**: 产品发布

**总结**: OpenAI 发布了 Deep Research 功能，这是一个能够自主进行多步互联网研究的 Agent 能力。基于优化后的 o3 模型，通过强化学习训练。

**核心能力**:
- 自主发现、推理和整合来自网络的信息
- 搜索、解读和分析大量文本、图片和 PDF
- 根据遇到的信息实时调整策略
- 输出完整引用和思考过程摘要

**性能**: Humanity's Last Exam 上达到 **26.6%** 准确率（当时 SOTA），GAIA 基准测试 SOTA。

**使用方式**: 在 ChatGPT 中选择"deep research"模式，输入查询即可。完成时间 5-30 分钟。

**局限性**: 仍可能产生幻觉，难以区分权威信息和谣言，置信度校准较弱。

## 8. Introducing ChatGPT Agent（介绍 ChatGPT Agent）

**来源**: OpenAI | **类型**: 产品发布

**总结**: OpenAI 推出 ChatGPT Agent，将 Operator 的网页交互能力、Deep Research 的信息综合能力和 ChatGPT 的对话能力统一到一个系统中。

**核心工具集**:
- 可视化浏览器（通过 GUI 交互）
- 文本浏览器（用于简单推理查询）
- 终端（代码执行）
- 直接 API 访问
- ChatGPT 连接器（Gmail、GitHub 等）

**性能数据**:
- Humanity's Last Exam: **41.6%**（新 SOTA）
- FrontierMath: **27.4%**
- BrowseComp: **68.9%**（比 Deep Research 高 17.4 个百分点）
- SpreadsheetBench: **45.5%**（对比 Copilot in Excel 的 20.0%）

**安全措施**: 明确用户确认、Watch Mode、主动风险缓解、隐私控制、安全浏览器接管。

**使用量**: Pro 用户每月 400 条消息，其他付费用户每月 40 条。

## 9. Computer-Using Agent（计算机使用 Agent）

**来源**: OpenAI | **类型**: 技术/产品发布

**总结**: OpenAI 推出 Computer-Using Agent（CUA），这是一种通过像素级 GUI 交互来操作计算机的 Agent 模型。CUA 结合了 GPT-4o 的视觉能力和强化学习训练的高级推理。

**工作原理**: 感知-推理-行动循环
- **感知**: 截图进入模型上下文
- **推理**: 通过思维链分析当前和过去的状态
- **行动**: 执行点击、滚动、输入等操作

**基准测试表现**:
- OSWorld（完整计算机使用）: **38.1%**（之前 SOTA 22.0%，人类 72.4%）
- WebArena（浏览器任务）: **58.1%**（之前 SOTA 36.2%）
- WebVoyager（在线网站）: **87.0%**

**安全**: 三层安全策略——防滥用、防模型错误、前沿风险评估。包括拒绝机制、黑名单、用户确认、Watch Mode、对抗性攻击防护等。

**核心价值**: CUA 不依赖特定 API，而是像人类一样操作 GUI，解决了"长尾"数字用例问题。

## 10. New Tools for Building Agents（构建 Agent 的新工具）

**来源**: OpenAI | **类型**: 开发者工具发布

**总结**: OpenAI 发布了一整套面向开发者的 Agent 构建工具。

**核心组件**:

1. **Responses API**: 新的 API 原语，结合了 Chat Completions 的简洁性和 Assistants API 的工具使用能力
2. **内置工具**:
   - Web Search（网页搜索）: GPT-4o search 在 SimpleQA 上达 90%
   - File Search（文件搜索）: 支持多种文件类型、查询优化、元数据过滤
   - Computer Use（计算机使用）: 基于 CUA 模型
3. **Agents SDK**: 开源的多 Agent 编排框架，支持：
   - Agent 定义和配置
   - 智能交接（Handoffs）
   - 护栏（Guardrails）
   - 追踪和可观测性
4. **可观测性工具**: 用于追踪和检查 Agent 工作流

**重要变化**: Assistants API 计划在 2026 年中弃用，迁移到 Responses API。Agents SDK 也支持其他提供商的模型。

## 11. Measuring AI Agent Autonomy in Practice（在实践中衡量 AI Agent 的自主性）

**来源**: Anthropic | **类型**: 研究/数据分析

**总结**: Anthropic 分析了数百万次人机交互数据，首次大规模实证研究了 AI Agent 在实际部署中的自主性水平。

**四大发现**:
1. **Claude Code 自主工作时间越来越长**: 最长会话的工作时间在三个月内翻倍（从不到25分钟到超过45分钟），且增长趋势平滑，表明不仅是模型能力提升，用户信任也在增长
2. **有经验的用户更频繁自动批准，但也更频繁中断**: 新用户约 20% 使用全自动批准，有经验用户超过 40%。但中断率也从 5% 上升到 9%，说明用户从"逐条审批"转向"监控+必要时干预"
3. **Claude 主动暂停的频率高于人类中断频率**: 在最复杂任务上，Claude 主动要求澄清的频率是用户中断的 **2 倍以上**
4. **Agent 在高风险领域有使用但尚未规模化**: 软件工程占近 50% 的 Agent 活动，医疗、金融、网络安全领域有新兴使用

**建议**: 投资部署后监控基础设施；训练模型识别自身不确定性；设计以用户监督为中心的产品；暂不应强制要求特定的交互模式。

## 12. Writing Effective Tools for AI Agents（为 AI Agent 编写有效的工具）

**来源**: Anthropic | **类型**: 技术实践文章

**总结**: 本文系统性地阐述了如何为 AI Agent 编写高效的工具，核心观点是工具设计需要从"确定性系统间的契约"转向"确定性系统与非确定性 Agent 间的契约"。

**开发流程**: 原型 → 评估 → 协作改进

**五大原则**:

1. **选择正确的工具**: 不是越多越好，优先构建针对高价值工作流的少量精准工具。例如用 `schedule_event` 代替 `list_users` + `list_events` + `create_event`
2. **命名空间**: 用前缀/后缀为工具分组（如 `asana_search`、`jira_search`），帮助 Agent 正确选择工具
3. **返回有意义的上下文**: 优先返回自然语言标识而非技术 ID（如用名字而非 UUID），支持 `concise`/`detailed` 响应格式切换
4. **优化 token 效率**: 实现分页、过滤、截断，设置合理的默认值。Claude Code 默认限制工具响应 25,000 token
5. **提示工程工具描述**: 像给新员工描述工具一样写工具说明，明确输入输出，消除歧义

**核心洞见**: 即使对工具描述做细微改进，也能显著提升 Agent 表现。Claude Sonnet 3.5 在优化工具描述后在 SWE-bench Verified 上达到 SOTA。
