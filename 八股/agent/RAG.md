# 后端要掌握的 RAG 面试题

来源：小红书笔记，作者：程序员流年

---

## 一、RAG 与 SFT 微调的区别？为什么选 RAG？

### 对比

| 维度 | SFT 微调 | RAG |
|------|----------|-----|
| 训练成本 | 高（需要标注数据 + GPU 训练） | 低（只需更新知识库） |
| 知识更新 | 需要重新训练 | 更新知识库即可 |
| 可解释性 | 差（知识编码在参数中） | 好（可追溯检索来源） |
| 幻觉问题 | 仍有幻觉风险 | 通过上下文约束降低幻觉 |

### 面试话术

"我理解 RAG 的核心，其实就是不给模型瞎编的空间，而是让它带着资料答题。大模型本质上是基于参数记忆去做概率生成的，如果它对某个知识不确定，也会生成一个'看起来合理'的答案，这就容易产生幻觉。RAG 做了两件关键的事：第一，它在生成前先做检索，把相关的真实资料找出来。第二，把这些资料拼进 prompt 里，让模型基于这些'证据'去回答。相当于给模型加了上下文约束，缩小了生成空间，不确定性自然就下降了。另外一个很重要的点是：RAG 解决了知识过期和长尾知识问题。模型不用把所有知识都记在参数里，只要知识库是最新的，检索出来的信息就是实时的。如果是在企业场景里，比如内部文档问答、FAQ 系统，RAG 比纯微调更稳定，也更容易更新。"

## 二、RAG 数据源有哪些，拿到数据后怎么处理？（Chunk 策略）

### 2.1 结构化数据

- MySQL / PostgreSQL（字段清晰、结构规范）
- 数据仓库
- FAQ 表 → 转为自然语言文本（如 "Q: xxx A: xxx"）或 schema-aware 检索
- 商品表、配置表

### 2.2 半结构化数据

- Excel、CSV、JSON
- Notion / Confluence
- 处理方式：扁平化，转成统一文本结构，保留 metadata（时间、作者、分类）

### 2.3 非结构化数据

- PDF、Word、PPT
- 网页、博客
- 处理方式：OCR + 文本提取 → 按 semantic chunk 切分

### 处理 Pipeline

```
数据清洗 → 语义 Chunk 切分（带 overlap） → Embedding → 存入向量数据库（带 metadata）
→ Hybrid Search（向量 + 关键词） → Rerank → Top-K
```

## 三、向量数据库怎么选？

### 选型维度

| 维度 | 说明 |
|------|------|
| 数据规模 | 几万~几十万：PGVector / ES 足够；千万级以上：Milvus 等专业向量数据库 |
| 检索性能 | 是否支持 HNSW 等 ANN 索引，查询延迟是否稳定 |
| Hybrid Search | 纯向量检索不够准，需要 BM25 + 向量混合检索 + metadata filter |
| 技术栈兼容 | 已有 ES 体系 → 直接用 ES 向量检索；已有 PG → PGVector |

### 常见方案

- **Elasticsearch**：支持稀疏+稠密混合检索，适合已有 ES 体系
- **Milvus**：专业向量数据库，适合大规模高性能场景，配合 BM25
- **PGVector**：适合小规模，已有 PostgreSQL 的场景

### 面试话术

"我主要从几个维度考虑。第一是数据规模，如果是几万到几十万规模，其实 PGVector 或 ES 就够了。但如果到千万级以上，我会优先选像 Milvus 这种专门做向量检索的数据库。第二是检索性能，RAG 对延迟比较敏感，所以我会关注是否支持 HNSW 等 ANN 索引，以及查询延迟是否稳定。第三是是否支持 Hybrid Search，因为很多业务场景纯向量检索不够准，我一般会结合 BM25 和向量检索一起用。第四是团队现有技术栈，比如如果公司已经大量使用 Elasticsearch，那直接用它做向量检索运维成本更低。整体来说，我不会单纯追求'最强'的向量库，而是根据数据规模、性能要求和运维成本综合选择。"

## 四、如何提升 RAG 的准确率？

### 核心框架

```
数据质量 → 检索质量 → 生成约束
```

### 4.1 提升数据质量

- **合理的 Chunk 切分**：切太大 → 噪声多；切太小 → 语义断裂
- **更好的数据清洗**：去重、去噪声、修复 PDF 结构，保证段落完整
- 脏数据会直接导致错误召回

### 4.2 提升检索质量（核心）

| 方法 | 说明 |
|------|------|
| 更好的 Embedding 模型 | 选择好的 embedding 模型，微调模型，做 embedding 对比评测 |
| Hybrid Search | 向量 + BM25，或 sparse + dense 混合，加权融合 |
| 加 Rerank 模型 | 向量召回 Top 20 + Rerank 二次排序，准确率提升非常明显 |
| 查询改写 | LLM 重写 query、多 query 并检索、意图识别 |

### 4.3 提升生成阶段准确率

| 方法 | 说明 |
|------|------|
| 强的 Prompt | 必须基于资料回答、条理清晰、禁止编造、结合上下文 |
| 限制 Top-K | K 太大 → 噪声，K 太小 → 信息不足，一般 3~5 比较稳定 |
| 输出引用 | 强制输出来源，增加可信度，减少幻觉 |

### 面试话术

"我一般会从三个阶段去优化：数据、检索、生成。第一是数据质量，比如做好清洗和合适的 chunk 切分，因为如果切分不好，后面再怎么优化也没用。第二是检索阶段，这是最关键的，我通常会：选合适的 embedding 模型、做 Hybrid Search（向量 + 关键词）、用 rerank 模型做二次排序、拉取 Top-K、对 query 做改写。第三是生成阶段，我在 prompt 里强调约束模型必须基于检索内容回答，如果资料中没有就明确说明，避免幻觉。我的理解是，RAG 的准确率更多取决于检索质量，而不是模型本身。把检索打得好，准确率会明显提升。"

## 五、RAG 如何提升检索性能，响应速度？

RAG 的延迟一般来自三块：**向量检索、Embedding、系统架构层**

### 5.1 向量检索层

- **使用 ANN 而不是精确检索**
  - 不要用 Flat 全量扫描
  - 使用 HNSW、IVF 等索引算法
  - 一般企业场景用 HNSW
  - 复杂度从 O(N) 降到接近 O(logN)
- **调整 HNSW 参数**
  - 合理设置 ef_search 和 M
  - 压缩到：延迟 < 50ms，召回率可接受
- **降低 Top-K**
  - Top-K 越大，排序越慢，数据传输越多
  - 一般 RAG 场景 3~5 就够了

### 5.2 Embedding 层

- **缓存热门 Query 的 Embedding**：很多问题会重复问

## 总结：5 道核心面试题

1. **RAG 与 SFT 微调的区别？为什么选 RAG？** — RAG 不给模型瞎编空间，带着资料答题，解决知识过期和长尾问题
2. **RAG 数据源有哪些，拿到数据后怎么处理？** — 结构化/半结构化/非结构化数据，清洗 → Chunk → Embedding → 向量库 → Hybrid Search → Rerank
3. **向量数据库怎么选？** — 从数据规模、检索性能、Hybrid Search、技术栈兼容四个维度综合选择
4. **如何提升 RAG 的准确率？** — 数据质量 → 检索质量 → 生成约束，核心在检索阶段
5. **RAG 如何提升检索性能，响应速度？** — 向量检索层（ANN/HNSW）、Embedding 层（缓存）、系统架构层优化
