# RAG 检索增强生成（Retrieval-Augmented Generation）

> RAG 的本质：**不让模型瞎编，而是让它带着资料答题。**

---

## 一、知识点讲解

### 1. RAG 通俗类比

| 场景 | 类比 | 解释 |
|------|------|------|
| 纯大模型（无 RAG） | **闭卷考试** | 只能靠"脑子里的记忆"（训练数据）作答，记不清就可能编答案 |
| 大模型 + RAG | **开卷考试 + 参考资料** | 先去"图书馆"找到相关章节，再基于资料组织答案 |

```
闭卷考试（纯LLM）                开卷考试（LLM + RAG）
┌──────────────────┐           ┌──────────────────┐
│  用户提问         │           │  用户提问         │
│      ↓           │           │      ↓           │
│  模型凭记忆回答    │           │  ① 检索相关文档   │
│      ↓           │           │      ↓           │
│  可能出现幻觉 ❌  │           │  ② 拼接到Prompt   │
└──────────────────┘           │      ↓           │
                               │  ③ 基于资料回答   │
                               │      ↓           │
                               │  有据可查 ✅      │
                               └──────────────────┘
```

**核心思想**：将"检索"和"生成"结合，用外部知识库弥补大模型的知识盲区和时效性问题。

### 2. RAG 完整四步流程

```
用户Query
   │
   ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 索引构建（Indexing）—— 离线阶段，提前做好           │
│  ┌──────┐   ┌──────────┐   ┌──────────┐   ┌──────────────┐ │
│  │ 文档 │ → │ Chunk切分 │ → │ Embedding │ → │ 向量数据库    │ │
│  │ 集合 │   │ 切成小块   │   │ 文本→向量  │   │ 存储索引      │ │
│  └──────┘   └──────────┘   └──────────┘   └──────────────┘ │
└─────────────────────────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 2: 检索召回（Retrieval）—— 在线阶段                    │
│  ┌──────┐   ┌──────────┐   ┌────────────────────────────┐  │
│  │Query │ → │ Embedding │ → │ 向量相似度搜索（ANN/HNSW）  │  │
│  │ 用户  │   │ Query→向量 │   │ 召回Top-K个相关Chunk       │  │
│  └──────┘   └──────────┘   └────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 3: 上下文增强（Augmentation）                          │
│  ┌──────────┐   ┌──────────┐   ┌──────────────────────┐    │
│  │ 原始Query │ + │ 检索结果  │ → │ 构造增强Prompt        │    │
│  │          │   │ Top-K    │   │ "参考资料：...\n问题：..."│   │
│  └──────────┘   └──────────┘   └──────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 4: 生成回答（Generation）                              │
│  ┌────────────────────┐   ┌──────────┐   ┌──────────────┐  │
│  │ 增强Prompt          │ → │ LLM生成  │ → │ 最终回答      │  │
│  │ (Query + Context)  │   │ 基于资料  │   │ 可追溯来源    │  │
│  └────────────────────┘   └──────────┘   └──────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**每步详解：**

| 阶段 | 关键操作 | 核心技术 | 耗时 |
|------|---------|---------|------|
| 索引构建 | 文档加载→切分→向量化→存储 | Chunk策略 + Embedding模型 + HNSW索引 | 离线，一次性 |
| 检索召回 | Query向量化→相似度搜索 | 余弦相似度 + ANN近似搜索 | 在线，<100ms |
| 上下文增强 | 拼接检索结果到Prompt | Prompt模板设计 + 上下文窗口管理 | 极快 |
| 生成回答 | LLM基于增强Prompt生成 | GPT-4/Claude等大模型 | 在线，1-5s |

### 3. Query 改写策略

用户原始Query往往不完整或有歧义，需要先改写再检索：

```
原始Query: "它的性能怎么样？"
   │
   ├── 指代消解：它在上下文中指什么？
   │   → "iPhone 15 Pro的性能怎么样？"
   │
   ├── 上下文补全：缺少背景信息
   │   → "iPhone 15 Pro的A17芯片性能相比上一代提升多少？"
   │
   ├── 多视角改写：从不同角度生成子Query
   │   → "iPhone 15 Pro跑分数据"
   │   → "iPhone 15 Pro游戏性能表现"
   │   → "iPhone 15 Pro与14 Pro性能对比"
   │
   └── HyDE（假设文档嵌入）：让LLM先猜一个答案
       → LLM生成假设答案 → 对假设答案做Embedding → 去检索
```

#### 各策略详解

**① 指代消解（Coreference Resolution）**

多轮对话中，用户说"它好不好"，需要结合历史上下文把"它"替换为具体实体。

```python
# 指代消解示例
chat_history = [
    {"role": "user", "content": "介绍一下iPhone 15 Pro"},
    {"role": "assistant", "content": "iPhone 15 Pro是..."},
    {"role": "user", "content": "它的性能怎么样？"}  # "它" → "iPhone 15 Pro"
]

# 通过LLM改写
rewritten_query = "iPhone 15 Pro的性能怎么样？"
```

**② 上下文补全（Context Completion）**

将最近几轮对话压缩成一个独立的、信息完整的Query。

**③ 多视角改写（Multi-Query / Multi-View）**

用一个Query生成多个不同视角的子Query，分别检索后合并结果，提高召回率。

```python
# 多视角改写示例
original_query = "Java内存模型"
rewritten_queries = [
    "Java Memory Model JMM规范",
    "JMM happens-before原则",
    "Java主内存与工作内存交互"
]
# 对每个子Query分别检索，合并去重
```

**④ HyDE（Hypothetical Document Embedding）**

核心思想：让LLM先"猜"一个答案，用假设答案的向量去检索，因为假设答案在语义空间中更接近真正的文档。

```python
# HyDE流程
query = "Java内存模型是什么？"
# Step1: LLM生成假设答案（可能不完全正确，但语义方向对）
hypothetical_answer = llm.generate(
    "请简要回答：" + query
)
# Step2: 对假设答案做Embedding（而非对query做Embedding）
hyde_embedding = embedding_model.encode(hypothetical_answer)
# Step3: 用假设答案的向量去检索
results = vector_db.search(hyde_embedding, top_k=5)
```

### 5. Chunk 策略详解

Chunk（文本分块）是 RAG 中影响效果最关键的环节之一。

#### 5.1 固定长度切分（Fixed-size Chunking）

```python
# 固定长度切分
def fixed_chunk(text, chunk_size=500, overlap=50):
    chunks = []
    for i in range(0, len(text), chunk_size - overlap):
        chunks.append(text[i:i + chunk_size])
    return chunks
```

- 优点：实现简单，效果稳定
- 缺点：可能在句子中间截断，破坏语义完整性

#### 5.2 基于分隔符切分（Delimiter-based Chunking）

```python
# 基于分隔符切分
def delimiter_chunk(text, separators=["\n\n", "\n", "。", "；"]):
    chunks = []
    current = ""
    for sep in separators:
        parts = text.split(sep)
        for part in parts:
            if len(current) + len(part) > chunk_size:
                chunks.append(current)
                current = part
            else:
                current += sep + part
    return chunks
```

- 按段落/句子/标点符号切分
- 优点：保留自然语义边界

#### 5.3 语义切分（Semantic Chunking）

```python
# 语义切分：计算相邻句子的Embedding相似度，相似度骤降处切分
def semantic_chunk(sentences, embedding_model, threshold=0.7):
    chunks = []
    current_chunk = [sentences[0]]
    for i in range(1, len(sentences)):
        sim = cosine_similarity(
            embedding_model.encode(sentences[i-1]),
            embedding_model.encode(sentences[i])
        )
        if sim < threshold:
            chunks.append(" ".join(current_chunk))
            current_chunk = [sentences[i]]
        else:
            current_chunk.append(sentences[i])
    chunks.append(" ".join(current_chunk))
    return chunks
```

- 优点：语义完整性最好
- 缺点：计算成本高（需要先做Embedding）

#### 5.4 递归切分（Recursive Chunking）

LangChain 默认推荐的策略，优先按大分隔符切，超过大小限制再按小分隔符切：

```python
# LangChain RecursiveCharacterTextSplitter 的逻辑
separators = ["\n\n", "\n", "。", " ", ""]  # 优先级从高到低
# 先按\n\n切，某块仍超长则按\n切，再超长按。切...
```

- 兼顾效率和语义完整性，工程实践首选

#### 5.5 父子文档（Parent-Child Chunking）

```
父文档（大Chunk，如2000 token）
├── 子Chunk 1（小Chunk，如200 token）  → 用于检索（更精准匹配）
├── 子Chunk 2（小Chunk）
├── 子Chunk 3（小Chunk）
└── 子Chunk 4（小Chunk）

# 检索时：用小Chunk匹配 → 返回对应的父文档给LLM
# 好处：小Chunk匹配更精准，大Chunk提供更完整的上下文
```

#### 策略对比表格

| 策略 | 原理 | 语义完整性 | 实现复杂度 | 适用场景 |
|------|------|-----------|-----------|---------|
| 固定长度 | 按字符/token数截断 | ⭐ | ⭐ | 快速验证、简单文档 |
| 基于分隔符 | 按段落/句子/标点 | ⭐⭐⭐ | ⭐⭐ | 结构化文档 |
| 语义切分 | Embedding相似度骤降处切分 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 高质量要求 |
| 递归切分 | 多级分隔符递归 | ⭐⭐⭐⭐ | ⭐⭐ | **工程实践首选** |
| 父子文档 | 小块检索+大块生成 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 长文档、需完整上下文 |


### 7. Embedding 本质解释

```
Embedding = 把文本翻译成"语义坐标"

"今天天气真好"  →  [0.12, -0.34, 0.56, ..., 0.78]  ← 1536维向量
"今天阳光明媚"  →  [0.13, -0.32, 0.55, ..., 0.79]  ← 1536维向量（非常接近！）
"我想吃火锅"    →  [-0.45, 0.67, -0.12, ..., 0.23] ← 1536维向量（距离远）
```

**关键性质：语义相似的文本 → 向量在高维空间中彼此接近。**

```
高维向量空间可视化（降维后）

        "天气好" ●
                 ● "阳光明媚"
            ● "晴朗"

                              ● "吃火锅"
                         ● "美食"
                    ● "火锅底料"

    语义相似 → 聚类在一起
    语义不同 → 距离较远
```

**为什么需要Embedding？**
- 计算机无法直接理解文本语义
- 向量化后可以用数学方法计算"语义相似度"
- 这是所有向量检索的基础

### 8. 余弦相似度 vs 欧氏距离

#### 欧氏距离（Euclidean Distance）

$$d(A, B) = \sqrt{\sum_{i=1}^{n}(a_i - b_i)^2}$$

衡量两个向量之间的**直线距离**。

#### 余弦相似度（Cosine Similarity）

$$\cos(\theta) = \frac{A \cdot B}{|A| \times |B|} = \frac{\sum a_i b_i}{\sqrt{\sum a_i^2} \times \sqrt{\sum b_i^2}}$$

衡量两个向量之间的**方向夹角**，值域 [-1, 1]，越大越相似。

```
为什么 RAG 用余弦相似度？

  向量A（短） ●        向量B（长） ●━━━━━━━━━●
        \              /
         \  θ角很小   /
          \          /
           \        /
            ● 原点

• 欧氏距离：受向量模长影响。B比A长，即使方向一样距离也大
• 余弦相似度：只看方向，不受模长影响

→ 文本向量经常模长不等（文档长短不同），但语义方向一致
→ 余弦相似度更适合衡量"语义相似性"
```

| 对比维度 | 余弦相似度 | 欧氏距离 |
|---------|-----------|---------|
| 衡量的是什么 | **方向**（角度） | **距离**（位置） |
| 受模长影响？ | **不受影响** | 受影响 |
| 值域 | [-1, 1] | [0, +∞) |
| RAG适用性 | **首选** ✅ | 需归一化后才可用 |
| 场景 | 文本语义相似度 | 空间位置/聚类 |

> **面试要点**：RAG选余弦相似度，核心原因是文本Embedding向量的模长不稳定（长短文档产生的向量模长不同），但方向能准确反映语义。

### 9. HNSW 算法原理

**HNSW = Hierarchical Navigable Small World（分层可导航小世界图）**

是当前最主流的 ANN（近似最近邻）索引算法。

#### 核心思想：分层图结构

```
Layer 2（高速公路层，稀疏）
    ●────────────────────────────●
    │                            │
Layer 1（国道层，中等密度）
    ●───────●───────●───────●───────●
    │       │       │       │       │
Layer 0（小路层，密集，包含所有节点）
    ●──●──●──●──●──●──●──●──●──●──●──●──●
    ↑
  查询入口（从最高层开始）

比喻：
• 高速公路：少量节点，长距离跳跃，快速缩小范围
• 国道：中等密度，进一步逼近目标区域
• 小路：密集连接，精确定位最近邻
```

#### 搜索过程

```
Step 1: 从Layer 2的入口节点开始
Step 2: 在当前层做贪心搜索，找到离Query最近的节点
Step 3: 跳到下一层，从上层的最近节点继续贪心搜索
Step 4: 直到Layer 0，得到最终近似最近邻
```

```
搜索示例：找离 ★ 最近的节点

Layer 2:  ●₁ ──── ●₂ ──── ●₃    贪心→ ●₂最近
                        ↓
Layer 1:  ●₄ ── ●₅ ── ●₆ ── ●₇  贪心→ ●₅最近
                      ↓
Layer 0:  ●₈─●₉─●₁₀─●₁₁─●₁₂     贪心→ ●₁₀最近 → 返回！
                  ★(查询点)
```

**复杂度：O(log N)**，相比暴力搜索的 O(N) 大幅提升。

| 对比 | 暴力搜索 | HNSW |
|------|---------|------|
| 时间复杂度 | O(N) | O(log N) |
| 准确率 | 100% | 95%+（近似） |
| 百万级数据耗时 | 数秒 | 毫秒级 |
| 内存占用 | 低 | 较高（需存图结构） |
| 适用场景 | 小数据量 | **生产环境首选** |

### 14. RAG 与 SFT 微调的区别

| 维度 | SFT 微调 | RAG |
|------|---------|-----|
| 核心思路 | 修改模型参数，把知识"记进脑子" | 不改模型，检索外部知识"翻资料" |
| 知识更新 | 需要重新训练（成本高） | 更新知识库即可（成本低） |
| 训练成本 | 高（标注数据 + GPU训练） | 低（只需Embedding + 向量存储） |
| 幻觉控制 | 仍有幻觉风险 | 通过上下文约束显著降低幻觉 |
| 可解释性 | 差（知识编码在参数中） | 好（可追溯检索来源） |
| 长尾知识 | 可能遗忘低频知识 | 只要知识库有就能检索到 |
| 时效性 | 受训练数据截止日期限制 | 知识库实时更新 |
| 部署门槛 | 需要GPU推理资源 | 只需向量数据库 + API |
| 适用场景 | 改变模型行为/风格/格式 | 知识密集型问答 |

**面试话术**：RAG解决的是"知识问题"（知道什么），SFT解决的是"能力问题"（会做什么）。两者互补，不互斥。

### 16. 混合检索（Hybrid Search）

**核心思想**：向量检索擅长语义匹配，BM25擅长关键词精确匹配，两者互补。

```
用户Query: "Java中HashMap的扩容机制"
            │
    ┌───────┴───────┐
    ↓               ↓
向量检索          BM25关键词检索
（语义相似）      （关键词匹配）
Top-20           Top-20
    │               │
    └───────┬───────┘
            ↓
      RRF融合排序（Reciprocal Rank Fusion）
            ↓
        合并Top-K结果
```

#### RRF 融合公式

$$RRF(d) = \sum_{r \in R} \frac{1}{k + rank_r(d)}$$

其中 $k$ 通常取 60，$rank_r(d)$ 是文档 $d$ 在检索系统 $r$ 中的排名。

```python
# RRF融合示例
def reciprocal_rank_fusion(results_list, k=60):
    """
    results_list: 多个检索系统的排序结果列表
    k: 平滑常数，默认60
    """
    rrf_scores = {}
    for results in results_list:
        for rank, doc in enumerate(results, start=1):
            if doc.id not in rrf_scores:
                rrf_scores[doc.id] = 0
            rrf_scores[doc.id] += 1.0 / (k + rank)

    # 按RRF分数降序排列
    return sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)
```

**为什么用RRF而不是直接平均分数？**
- 不同检索系统的分数范围不同（向量相似度0~1，BM25分数0~30+），无法直接比较
- RRF只用排名，归一化了不同系统的分数差异

### 17. Rerank 重排序

```
初检（向量召回，快速但粗糙）
Top-100/Top-50
        ↓
Rerank（Cross-Encoder，慢但精准）
Top-10/Top-5
        ↓
送入LLM生成
```

**为什么需要Rerank？**

向量检索用的是 **Bi-Encoder**（Query和Document分别编码后算相似度），速度快但精度有限。
Rerank用的是 **Cross-Encoder**（Query和Document拼接后一起编码），精度高但速度慢。

```
Bi-Encoder（召回阶段）:
  Query → Encoder → 向量Q     →  cos(Q, D) → 相似度
  Doc   → Encoder → 向量D  ↗
  ✅ 快：向量可预计算
  ❌ 粗糙：Query和Doc独立编码，交互不充分

Cross-Encoder（Rerank阶段）:
  [CLS] Query [SEP] Document [SEP] → Encoder → 相关性分数
  ✅ 精准：Query和Doc深度交互
  ❌ 慢：每对(Q,D)都要过一遍模型
```

**工程实践**：向量召回 Top-20 → Rerank 精排取 Top-5 → 送入LLM。

常用Rerank模型：Cohere Rerank、BGE-Reranker、bce-reranker-base。

### 18. Lost in the Middle 现象

**现象**：LLM在处理长上下文时，对中间位置的信息关注度明显低于开头和结尾。

```
Prompt中的文档排列：

  位置1  ████████  ← LLM关注度高
  位置2  ████████  ← LLM关注度高
  位置3  ░░░░░░░░  ← LLM关注度低 ⚠️ Lost in the Middle
  位置4  ░░░░░░░░  ← LLM关注度低 ⚠️
  位置5  ████████  ← LLM关注度高（结尾）

  呈U型注意力分布
```

**解决方案：**

1. **将最相关的文档放在开头和结尾**（重排策略）
2. **减少送入LLM的文档数量**（Top-5而非Top-20）
3. **压缩文档内容**（用LLM提取关键信息，去除冗余）
4. **长上下文窗口模型**（128K/200K的模型对此有改善）

```python
# 重排策略示例：将最相关的放在首尾
def lost_in_the_middle_rerank(ranked_docs):
    """将按相关度排序的文档重排为：最相关在开头，次相关在结尾，依次交替"""
    result = []
    for i, doc in enumerate(ranked_docs):
        if i % 2 == 0:
            result.insert(0, doc)   # 偶数位放前面
        else:
            result.append(doc)      # 奇数位放后面
    return result
```

```
传统RAG的局限：
1. 只能做局部检索，无法理解全局信息（如"总结全文核心观点"）
```

```
┌──────────────────────────────────────────────────────────────┐
```

### 23. 传统 RAG 的权限管理风险

传统RAG在权限管理方面存在7大风险：

| 编号 | 风险 | 说明 |
|------|------|------|
| 1 | **无权限过滤检索** | 用户检索时可能召回无权访问的文档内容 |
| 2 | **上下文泄露** | 有权限的文档在Prompt中暴露了无权文档的摘要信息 |
| 3 | **元数据缺失** | Chunk未绑定权限标签（部门/级别/角色），无法按权限过滤 |
| 4 | **跨租户数据泄露** | 多租户场景下，A租户可能检索到B租户的数据 |
| 5 | **间接推断攻击** | 用户通过精心构造Query，间接推断出无权访问的信息 |
| 6 | **日志审计缺失** | 检索行为未记录，无法追溯谁访问了什么 |
| 7 | **缓存穿透** | 检索结果缓存未按用户权限隔离，可能导致权限绕过 |

### 24. Agentic RAG 在权限合规方面的作用

```
Agentic RAG = Agent + RAG

传统RAG：
  用户Query → 直接检索 → 直接生成（无权限意识）

Agentic RAG：
  用户Query → Agent判断 → 权限校验 → 定向检索 → 生成
                    │
                    ├── 身份验证：确认用户身份和角色
                    ├── 权限过滤：只检索用户有权访问的文档
                    ├── 审计日志：记录每次检索行为
                    └── 结果脱敏：生成前检查是否包含敏感信息
```

**Agentic RAG如何解决权限风险：**

| 风险 | Agentic RAG解决方案 |
|------|-------------------|
| 无权限过滤 | Agent在检索前自动添加权限过滤条件（metadata filter） |
| 上下文泄露 | Agent在生成后检查回答是否包含无权信息 |
| 元数据缺失 | Agent自动为文档打标签（部门/角色/密级） |
| 跨租户泄露 | Agent强制添加tenant_id过滤 |
| 间接推断 | Agent识别可疑Query模式，拒绝回答 |
| 日志审计 | Agent记录完整检索链路 |
| 缓存穿透 | Agent按用户维度隔离缓存 |

### 25. 增量文本的 RAG 处理

当知识库需要持续更新时：

```
增量文本处理流程：

新文档到达
    │
    ▼
┌──────────────┐
│ 变更检测       │  ← 文档指纹（hash）比对，判断是否更新
│ 新增/修改/删除  │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 增量Chunk处理  │  ← 只处理变更部分，而非全量重建
│ 新增Chunk     │
│ 修改Chunk     │
│ 删除旧Chunk   │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 增量Embedding  │  ← 只对变更Chunk做向量化
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 增量写入向量库  │  ← upsert操作，更新已有向量
│ 维护版本号     │
└──────────────┘
```

**关键设计点：**

```python
# 增量处理伪代码
class IncrementalRAGUpdater:
    def process_document(self, doc):
        # 1. 计算文档指纹
        fingerprint = hash(doc.content)

        # 2. 比对是否变更
        old_fp = self.db.get_fingerprint(doc.id)
        if old_fp == fingerprint:
            return  # 无变更，跳过

        # 3. 删除旧Chunk的向量
        if old_fp:
            old_chunks = self.db.get_chunks(doc.id)
            self.vector_db.delete(old_chunks.ids)

        # 4. 切分 + Embedding + 写入
        new_chunks = self.chunker.split(doc.content)
        embeddings = self.embedding_model.encode(new_chunks)
        self.vector_db.upsert(new_chunks, embeddings)

        # 5. 更新指纹
        self.db.update_fingerprint(doc.id, fingerprint)
```

**时间调度策略**：
- 实时场景：Kafka消息驱动，文档变更即时触发
- 批量场景：定时任务（每小时/每天）扫描变更

### 26. 多模态 RAG

#### 伪多模态 vs 真多模态

```
伪多模态RAG（文本中介）：
  图片 → OCR提取文本 → 文本Embedding → 文本检索
  ❌ 丢失了视觉信息（布局、图表、颜色）

真多模态RAG（CLIP）：
  图片 → CLIP直接编码为向量 → 多模态检索
  ✅ 保留了视觉语义
```

#### CLIP 模型

**CLIP = Contrastive Language-Image Pre-training（对比语言-图像预训练）**

```
CLIP训练过程：
┌──────────┐     ┌──────────┐
│  图片编码器 │     │  文本编码器 │
│  (ViT)    │     │  (Transformer) │
└────┬─────┘     └────┬─────┘
     │                 │
  图片向量           文本向量
     │                 │
     └───对比学习──────┘
     "一只猫的图片" ≈ "a photo of a cat"
     拉近匹配对的距离，推远不匹配对的距离
```

**CLIP的核心能力：图文共享同一个向量空间，可以用文字搜图片，也可以用图片搜文字。**

```
多模态RAG流程：
  用户Query: "展示一下产品外观"
       │
       ▼
  CLIP文本编码 → 向量Query
       │
       ▼
  向量数据库检索（图文混合库）
       │
       ├── 匹配到图片向量 → 返回相关图片
       └── 匹配到文本向量 → 返回相关文档
```

## 二、串联理解记忆

### RAG 与 Memory

在 [[Memory]] 系统中，RAG 对应的是**情节记忆**的检索机制：

```
Memory系统
├── 短期记忆（对话上下文）  → 无需RAG，直接放在Prompt中
├── 长期记忆
│   ├── 语义记忆（通用知识）  → 模型参数 / 知识库 → RAG检索
│   └── 情节记忆（历史经验）  → 历史对话/日志 → RAG检索
└── RAG就是Memory的"搜索引擎"
```

### RAG 与 ReAct

在 [[ReAct]] 框架中，Observation阶段可以触发RAG检索：

```
ReAct循环：
  Thought: 用户问了一个专业知识问题，我需要查找资料
  Action: 调用RAG检索工具
  Observation: ← RAG返回的检索结果在这里
  Thought: 根据检索结果，答案是...
  Answer: 最终回答
```

[[MCP]] 可以将RAG检索封装为一个标准工具暴露给Agent：

```
MCP Server
├── Tool: rag_search
│   ├── input: query (string)
│   ├── input: top_k (number)
│   └── output: 检索结果列表
├── Tool: rag_hybrid_search
│   ├── input: query (string)
│   └── output: 混合检索结果
└── Resource: knowledge_base (知识库)
```

### 余弦相似度与数学基础

- 余弦相似度本质是向量内积的归一化形式
- 与 [[Agent总览]] 中提到的向量空间模型一脉相承
- 维度灾难：维度越高，任意两向量距离趋向一致 → 需要降维或选择合适维度

### HNSW 与跳表/多级索引的联系

```
HNSW的分层思想与数据结构中的跳表（Skip List）非常相似：

跳表：
  Level 3:  Head ──────────────────────────── Node
  Level 2:  Head ────────── Node ──────────── Node
  Level 1:  Head ── Node ── Node ── Node ──── Node
  Level 0:  Head - N - N - N - N - N - N - N - Node

HNSW：
  Layer 2:  Entry ─────────────────────────── Node（稀疏）
  Layer 1:  Entry ────────── Node ──────────── Node
  Layer 0:  Entry ── Node ── Node ── Node ──── Node（密集）

共同点：高层快速跳跃定位 → 逐层精确定位
区别：跳表是一维有序的，HNSW是多维近邻图
```

### 记忆口诀

```
RAG四步要记牢：索引检索和增强，最后生成才算好。
Chunk切分有诀窍：五百token五十叠，递归切分最可靠。
Embedding翻译官：文本变向量，余弦看方向。
HNSW高速路：分层跳跃log(N)，比暴力搜索快万倍。
混合检索是王道：向量加词频，RRF来融合。
Rerank二次排：召回二十精排五，Lost in Middle放两头。
GraphRAG新思路：实体关系建图谱，全局推理不再输。
```

## 三、面试题

### Q1: RAG流程是什么？

**答：RAG完整流程分四步：**

1. **索引构建（Indexing）**：离线阶段，将文档切分成Chunk，通过Embedding模型转化为向量，存入向量数据库并建立ANN索引
2. **检索召回（Retrieval）**：在线阶段，将用户Query同样做Embedding，在向量数据库中通过余弦相似度检索最相关的Top-K个Chunk
3. **上下文增强（Augmentation）**：将检索到的Chunk与原始Query拼接成增强Prompt，格式如"参考资料：{context}\n问题：{query}"
4. **生成回答（Generation）**：将增强Prompt送入LLM，模型基于检索到的资料生成回答

> 关键点：索引是离线预处理的，检索和生成是在线的。检索质量直接决定RAG效果。

---

### Q2: RAG与SFT微调的区别？为什么选RAG？

**答：**

| 维度 | SFT微调 | RAG |
|------|---------|-----|
| 知识存储 | 编码在模型参数中 | 存储在外部知识库 |
| 更新成本 | 需重新训练 | 更新知识库即可 |
| 可解释性 | 差 | 好（可追溯来源） |
| 幻觉控制 | 有限 | 通过上下文约束显著降低 |

**选RAG的原因**：
1. 知识时效性：知识库可实时更新，不受模型训练截止日期限制
2. 成本可控：不需要GPU训练，只需Embedding服务+向量数据库
3. 可追溯：每个回答都能追溯到具体文档来源
4. 长尾知识：只要知识库有就能检索到，不存在"遗忘"

> 补充：两者互补而非互斥。SFT改变模型的"行为方式"（怎么回答），RAG补充模型的"知识范围"（回答什么）。可以先用SFT调教输出风格，再用RAG注入领域知识。

### Q3: RAG数据源有哪些，拿到数据后怎么处理？

**答：**

**数据源**：
- 结构化：MySQL、数据仓库、FAQ表
- 半结构化：Excel、JSON、Notion、Confluence
- 非结构化：PDF、Word、PPT、网页、图片

**处理Pipeline**：

1. **文档解析**：文本PDF用PyMuPDF提取，扫描件用OCR，Word用python-docx，网页用BeautifulSoup
2. **数据清洗**：去除乱码、去重、标准化格式
3. **Chunk切分**：
   - 首选递归切分（LangChain的RecursiveCharacterTextSplitter）
   - 推荐参数：chunk_size=500 token，overlap=50 token
   - 按场景可选父子文档策略（小块检索+大块生成）
4. **Embedding**：中文场景推荐BGE-M3模型，生成向量
5. **存储**：写入向量数据库，同时存储metadata（来源、时间、权限、分类）

### Q4: 向量数据库怎么选？

**答：选型看三个维度——规模、生态、团队能力。**

- 已有ES集群 → 直接用Elasticsearch 8.x（支持kNN向量检索+BM25混合检索）
- 已有PostgreSQL + 数据量不大 → pgvector（SQL生态无缝融合）
- 数据量100万以内 + 快速验证 → Chroma
- 数据量100万~5000万 + 高性能要求 → Qdrant（Rust写的高性能）
- 数据量5000万以上 + 分布式需求 → Milvus（云原生架构）
- 纯研究/算法实验 → FAISS

> 我们项目是Java技术栈，最终选了Milvus，因为Java SDK完善、支持分布式、社区活跃。如果团队已经有ES，ES 8.x是最务实的选择。

### Q5: 如何提升RAG的准确率？

**答：三层优化框架——数据质量、检索质量、生成约束。**

**第一层——数据质量**：Garbage In = Garbage Out。清洗脏数据、去重、选好Chunk策略、丰富metadata

**第二层——检索质量**：
- Query改写：指代消解、多视角改写、HyDE假设文档嵌入
- 混合检索：BM25关键词 + 向量语义检索，用RRF融合
- Rerank重排序：向量召回Top-20，用Cross-Encoder精排到Top-5
- 选好Embedding模型：中文场景首选BGE-M3

**第三层——生成约束**：
- Prompt模板优化："仅根据以下参考资料回答，如果资料中没有相关信息，请回答'暂无相关资料'"
- 引用溯源：要求标注回答来自哪个文档
- 幻觉检测：对比生成内容与检索内容的覆盖率

### Q7: Chunk太大和太小的区别？怎么选Chunk大小？

**答：**

| Chunk大小 | 问题 |
|-----------|------|
| **太大**（>1000 token） | 检索精度下降（大量无关内容混入），LLM输入变长导致成本增加、速度变慢 |
| **太小**（<100 token） | 语义不完整（一句话被切断），缺乏上下文，LLM无法理解完整含义 |

**推荐配置**：
- chunk_size = 500 token（约350-400个中文字）
- chunk_overlap = 50 token（约10%重叠，防止边界信息丢失）
- 使用递归切分策略，优先在自然语义边界切分

**实际选择还需考虑场景**：
- FAQ问答：小Chunk（100-200），一个Q-A对就是一个Chunk
- 技术文档：中Chunk（400-600），保留完整段落
- 长篇报告：大Chunk（800-1000）或父子文档策略

### Q8: 余弦相似度和欧氏距离的区别？为什么RAG用余弦？

**答：**

- 欧氏距离衡量的是空间中两个点的**直线距离**，受向量模长影响
- 余弦相似度衡量的是两个向量的**方向夹角**，不受模长影响

**RAG用余弦的原因**：文本Embedding向量的模长不稳定——长文档和短文档的向量模长可能差很多，但语义方向可能一致。如果用欧氏距离，模长差异会干扰相似度判断。余弦相似度只看方向，更能准确反映语义相似性。

> 公式：cos(A,B) = (A·B) / (|A|×|B|)，值域[-1,1]，1表示方向完全一致。

### Q9: HNSW的原理？和暴力搜索的区别？

**答：**

HNSW（Hierarchical Navigable Small World）是分层可导航小世界图，核心思想是构建多层图结构：

- **高层**（稀疏）：少量节点，长距离连接，类似"高速公路"，快速缩小搜索范围
- **中层**：中等密度，类似"国道"，进一步逼近目标
- **底层**（密集）：包含所有节点，类似"小路"，精确定位

**搜索过程**：从最高层入口开始贪心搜索 → 找到最近节点后跳到下一层 → 逐层向下直到Layer 0 → 返回最近邻。时间复杂度O(log N)。

**和暴力搜索对比**：暴力搜索逐一计算所有向量的相似度，O(N)复杂度，百万级数据要数秒。HNSW是近似搜索，准确率95%以上，百万级数据只需毫秒。代价是内存占用较高（需存储图结构）和构建索引较慢。

### Q10: 如果RAG相似度评分高但内容不符怎么解决？

**答：这是一个经典的检索噪声问题。解决方案：**

1. **Rerank重排序**：向量召回的相似度只是初步筛选，用Cross-Encoder对Query和每个Chunk做深度交互，重新打分排序
2. **混合检索**：加入BM25关键词检索，语义相似+关键词匹配双保险，RRF融合
3. **Prompt约束**：明确告诉LLM"如果参考资料中没有相关信息，请直接说不知道"
4. **元数据过滤**：利用时间、类别、来源等metadata先过滤，缩小检索范围
5. **调整Top-K**：适当增大召回数量（Top-20→Top-50），再用Rerank精排
6. **人工标注反馈**：收集bad case，用标注数据微调Embedding或Rerank模型

---

### Q11: 传统RAG痛点、GraphRAG是什么？

**答：**

**传统RAG痛点**：
1. 只能做局部检索，无法回答全局性问题（如"总结所有文档的核心观点"）
2. 多跳推理能力弱（如"A公司的子公司B的CTO曾在哪里工作"）
3. 缺乏实体关系的结构化表示
4. 长文档被切碎后丢失全局逻辑

**GraphRAG**：在传统RAG基础上引入知识图谱，先用LLM从文档中抽取实体和关系构建图谱，再通过社区检测算法将图分层（社区摘要），查询时既可以做局部图遍历也可以做全局社区摘要聚合。

---

### Q12: GraphRAG的核心流程？

**答：六步流程：**

1. **数据摄取**：加载原始文档
2. **知识抽取**：用LLM识别文档中的实体（人/组织/事件）和关系（担任/属于/导致）
3. **图谱构建**：将实体作为节点、关系作为边，构建知识图谱
4. **图索引**：用Leiden等社区检测算法将图分层，为每个社区生成摘要
5. **查询**：全局查询走社区摘要聚合，局部查询走子图遍历
6. **生成**：LLM基于图上下文（关系路径/社区摘要）生成回答

---

### Q13: GraphRAG与RAG技术异同？

**答：**

| 维度 | RAG | GraphRAG |
|------|-----|----------|
| 知识表示 | 平坦文本Chunk | 结构化实体-关系图 |
| 检索方式 | 向量相似度 | 图遍历+社区摘要 |
| 全局理解 | 弱 | 强 |
| 多跳推理 | 弱 | 强 |
| 构建成本 | 低 | 高（需LLM抽取） |
| 更新维护 | 容易 | 复杂（图的增量更新） |
| 可解释性 | 中等 | 高（可追溯推理路径） |

---

### Q14: RAG+GraphRAG可以结合使用吗？

**答：可以，而且这是当前的最佳实践。**

通过路由策略判断问题类型：
- 简单事实型问题 → 走传统RAG（向量检索）
- 关系推理/全局总结型问题 → 走GraphRAG（图查询）

两个系统共享同一份文档，但索引方式不同：RAG做向量索引，GraphRAG做图谱索引。框架层面，LightRAG、Microsoft GraphRAG等已支持这种混合模式。

### Q15: 传统RAG在权限管理上存在哪些风险？

**答：七大风险：**

1. **无权限过滤检索**：用户可能检索到无权访问的文档
2. **上下文泄露**：有权限文档的Prompt中暴露了无权文档的摘要
3. **元数据缺失**：Chunk未绑定权限标签，无法按权限过滤
4. **跨租户数据泄露**：多租户场景下数据隔离不彻底
5. **间接推断攻击**：用户通过精心构造Query推断无权信息
6. **日志审计缺失**：检索行为未记录
7. **缓存穿透**：检索结果缓存未按用户权限隔离

---

### Q16: Agentic RAG在权限合规方面能发挥什么作用？

**答：**

Agentic RAG通过Agent的自主决策能力，在检索全链路嵌入权限控制：

1. **检索前**：Agent自动添加权限过滤条件（metadata filter），只检索用户有权访问的文档
2. **检索中**：Agent强制添加tenant_id过滤，防止跨租户泄露
3. **生成前**：Agent检查生成内容是否包含超出用户权限的敏感信息
4. **审计**：Agent记录完整检索链路（谁、什么时候、检索了什么、返回了什么）
5. **异常检测**：Agent识别可疑Query模式（间接推断攻击），拒绝回答

> 核心优势：传统RAG的权限控制是"被动过滤"，Agentic RAG是"主动管控"。

---

### Q17: 如何将用户输入的语言与准备好的文档进行匹配？

**答：**

1. **Embedding向量化**：将用户Query和所有文档Chunk都通过同一个Embedding模型编码为向量
2. **向量相似度搜索**：计算Query向量与所有Chunk向量的余弦相似度，取Top-K
3. **混合检索**：同时做向量语义检索和BM25关键词检索，用RRF融合
4. **Rerank精排**：用Cross-Encoder对Top-K结果做二次排序
5. **Query改写**：先对用户Query做指代消解、上下文补全、多视角改写，再检索

---

### Q18: 增量文本的RAG怎么做？

**答：**

1. **变更检测**：通过文档指纹（hash）比对，判断哪些文档是新增/修改/删除
2. **增量Chunk处理**：只处理变更部分，不是全量重建
3. **增量Embedding**：只对变更的Chunk做向量化
4. **增量写入**：通过upsert操作更新向量数据库
5. **版本管理**：维护文档和Chunk的版本号，支持回滚

触发方式可以是Kafka消息驱动（实时），也可以是定时任务（批量）。

### Q19: RAG怎么评估？

**答：常用评估指标：**

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| Recall@K | 前K个结果中包含了多少相关文档 | 相关文档数 / 总相关文档数 |
| MRR | 第一个正确结果的排名倒数的均值 | avg(1/rank) |
| nDCG | 考虑排序位置的增益 | 归一化折损累计增益 |
| 忠实度（Faithfulness） | 生成内容是否忠于检索资料 | 生成声明与检索内容的覆盖比 |
| 答案相关性 | 生成内容是否回答了用户问题 | 语义相似度评估 |
| 上下文相关性 | 检索内容是否与问题相关 | 人工标注或LLM评估 |

> 常用评估框架：Ragas、TruLens、DeepEval。

### Q20: Embedding模型选型考虑哪些因素？

**答：**

1. **语言能力**：中文场景选BGE系列，多语言选OpenAI或Cohere
2. **部署方式**：数据安全要求高必须本地部署（BGE），否则可用API（OpenAI）
3. **向量维度**：高维度表达力强但成本高（1536 vs 256）
4. **模型大小**：大模型效果好但推理慢，需平衡效果与延迟
5. **MTEB榜单排名**：参考公开评测榜单
6. **许可证**：商业项目注意开源协议（MIT/Apache/自定义）

### Q21: 向量相似度怎么判断？

**答：**

主要三种方式：

1. **余弦相似度**：衡量方向夹角，不受模长影响，RAG首选
2. **欧氏距离（L2）**：衡量空间距离，受模长影响，需归一化后使用
3. **内积（点积）**：向量已归一化时等价于余弦相似度

> 实际工程中，如果Embedding模型输出的向量已经归一化（如OpenAI的模型），内积和余弦等价，可以选更快的内积运算。

### Q22: GraphRAG如何保证答案的可解释性？

**答：**

1. **推理路径可视化**：图查询返回的是实体-关系路径，可以可视化展示"从A到B的推理链"
2. **社区摘要溯源**：全局查询基于社区摘要，可以追溯到具体社区的组成文档
3. **实体引用**：答案中提到的实体都有对应的图节点，可查看完整属性和关联关系
4. **LLM推理过程**：在Prompt中要求LLM展示推理步骤

### Q23: GraphRAG在工程实践中面临的主要挑战？

**答：**

1. **图谱构建成本高**：需要LLM逐文档抽取实体和关系，Token消耗大
2. **增量更新复杂**：新增文档可能需要修改已有图结构（新增/合并实体），不像RAG直接upsert向量
3. **存储成本**：需要图数据库（Neo4j）+ 向量数据库双套存储
4. **查询效率**：图遍历可能涉及多跳，路径越长延迟越高
5. **实体对齐困难**：不同文档中的同一实体（"张三" vs "张三（技术部）"）需要消歧合并
6. **社区检测参数敏感**：Leiden算法的分辨率参数影响社区粒度，需反复调优

### Q24：IVF_FLAT 和 HNSW 索引有什么区别？怎么选？

| 对比 | IVF_FLAT（倒排文件） | HNSW（分层可导航小世界） |
|------|---------------------|----------------------|
| 原理 | 先聚类再在子集中搜索 | 多层图结构，逐层逼近 |
| 查询速度 | 快（只搜索部分簇） | 极快（图遍历效率高） |
| 构建速度 | 快 | 慢 |
| 内存占用 | 较低 | 较高（需存储图结构） |
| 召回率 | 依赖 nprobe 参数，可调 | 高（通常 >95%） |
| 增量更新 | 支持较好 | 增量更新代价高 |
| 适用场景 | 数据量大、需要增量更新 | 数据量中等、追求高召回 |

选型建议：
- **数据量 > 1000万 + 需要频繁更新** → IVF_FLAT
- **数据量 < 1000万 + 追求高精度** → HNSW
- **资源受限** → IVF_FLAT（内存更友好）

### Q25：Milvus 中 Collection 和 Partition 怎么设计和选择？

Collection（集合）相当于数据库的表，Partition（分区）相当于表的分区：

```
Collection: 产品文档库
  ├── Partition: 产品A
  ├── Partition: 产品B
  └── Partition: 产品C

Collection: 内部制度库
  ├── Partition: 人事制度
  ├── Partition: 财务制度
  └── Partition: 技术规范
```

| 维度 | 多 Collection | 单 Collection 多 Partition |
|------|-------------|--------------------------|
| 隔离性 | 强（完全独立） | 中（共享索引，逻辑隔离） |
| 管理成本 | 高（每个独立维护） | 低（统一管理） |
| 跨域检索 | 不支持（需分别查再合并） | 支持（可跨 Partition 搜索） |
| 权限控制 | 简单（Collection 级别） | 复杂（需要 Partition 级别） |
| 资源占用 | 高（每个 Collection 独立索引） | 低（共享资源） |

设计原则：**数据结构和检索需求相同用 Partition，不同用 Collection**。

### Q26：RAGAS 评估框架是什么？怎么用？

RAGAS（Retrieval Augmented Generation Assessment）是专门评估 RAG 系统的框架，核心指标：

| 指标 | 评估什么 | 计算方式 |
|------|---------|---------|
| **Faithfulness（忠实度）** | 回答是否忠于检索内容 | 检查回答中每个声明是否可被检索内容支撑 |
| **Answer Relevance（答案相关性）** | 回答是否切题 | 回答与问题的语义相似度 |
| **Context Precision（上下文精确度）** | 检索内容中有多少是相关的 | 相关文档在检索结果中的排名位置 |
| **Context Recall（上下文召回率）** | 回答所需的信息是否都被检索到 | 答案中的信息是否可由检索内容支撑 |

```
使用流程:
  1. 准备测试集: (question, ground_truth_answer, contexts)
  2. 用 RAG 系统生成回答
  3. 用 RAGAS 自动评估各指标
  4. 分析短板 → 针对性优化
```

实践建议：不要只看总分，要看每个维度的得分。如果 Faithfulness 低，说明模型在"编造"——需要加强 Prompt 约束或 Rerank。如果 Context Recall 低，说明检索不够全面——需要调整 Chunk 策略或增加检索数量。

### Q27：Code-RAG 有什么特殊考虑？

Code-RAG 是 RAG 在代码领域的特殊应用，与普通文本 RAG 有显著差异：

| 维度 | 普通文本 RAG | Code-RAG |
|------|------------|----------|
| Chunk 策略 | 按段落/语义切分 | 按函数/类/模块切分（需 AST 解析） |
| Embedding 模型 | 通用文本模型 | 代码专用模型（如 CodeBERT） |
| 检索维度 | 语义相似 | 语义 + 结构（调用关系、继承关系） |
| 上下文窗口 | 段落级 | 需要跨文件（定义-引用关系） |
| 精确度要求 | 允许模糊匹配 | 一个符号错误就会导致代码不工作 |

关键设计点：
- **AST 感知切分**：用 Tree-sitter 等工具按语法结构切分，保持函数/类完整性
- **多层索引**：符号级（函数名、类名）+ 语义级（功能描述）+ 结构级（调用链）
- **上下文补全**：检索到某函数时，自动补充其依赖的 import、类型定义
- **验证机制**：检索结果经 LLM 生成代码后，用 Lint/编译/单测做自动验证

## 四、面试话术

### 话术1："请解释RAG的完整流程"

> RAG的完整流程分四步：索引构建、检索召回、上下文增强、生成回答。
>
> 首先是**索引构建**，这是离线阶段。我们把文档通过Chunk策略切成小块，比如500个token一个Chunk，有50个token的重叠防止边界信息丢失。切分策略上我们用的是递归切分，优先按段落、句子、标点这样的自然语义边界切。切好之后，用Embedding模型——我们中文场景用的BGE-M3——把每个Chunk编码成一个1024维的向量，然后存入向量数据库，建立HNSW索引。
>
> 第二步是**检索召回**。用户提了一个问题，我们先用同样的Embedding模型把Query编码成向量，然后在向量数据库里通过余弦相似度做近似最近邻搜索，召回Top-K个最相关的Chunk。实际生产中我们会做混合检索，向量语义检索加上BM25关键词检索，用RRF公式融合结果，这样既能匹配语义也能匹配关键词。
>
> 第三步是**上下文增强**。把检索到的Chunk按一定格式拼接到Prompt里，比如"参考资料：{chunk1}\n{chunk2}\n问题：{query}"。这里有个细节，学术上叫"Lost in the Middle"，就是LLM对中间位置的内容关注度低，所以我们会把最相关的Chunk放在开头和结尾。
>
> 最后是**生成回答**。把增强后的Prompt送给LLM，模型基于检索到的资料来生成回答。我们在Prompt里会加约束，比如"仅根据参考资料回答"，这样能显著降低幻觉。回答出来后，还会标注引用来源，做到可追溯。

### 话术2："RAG系统Chunk策略怎么设计？"

> Chunk策略是影响RAG效果最关键的环节之一。核心原则是：**切出来的每一块都要在语义上是完整的，同时大小要合适。**
>
> 先说Chunk大小：太大了，比如超过1000个token，检索的时候容易混入大量无关内容，而且送进LLM的输入变长，成本和延迟都上升。太小了，比如不到100个token，可能一句话都被切断了，上下文不够，LLM没法理解完整含义。我们的经验是500 token左右最平衡，配合50 token的重叠。
>
> 切分策略上，我们对比过几种方案：固定长度切分最简单但容易切断句子；语义切分效果最好但成本太高，要先把所有句子做Embedding再算相邻句子的相似度。我们最终选了**递归切分**，这也是LangChain的默认策略——优先按`\n\n`段落切，某段超长了就按`\n`行切，再超长按句号切。这样兼顾了效率和语义完整性。
>
> 还有一个高级策略叫**父子文档**：把文档切成大块（父文档2000 token）和小块（子文档200 token），检索时用小块匹配（更精准），返回时把对应的父文档给LLM（上下文更完整）。这个策略在我们长文档场景效果提升明显。
>
> 另外我们还在每个Chunk上附加了metadata——来源文档、时间、分类、权限等级。这样检索时可以做pre-filtering，先按条件过滤再搜索，既能提高精度也能做权限控制。

### 话术3："向量数据库怎么选？"

> 向量数据库选型我们是从三个维度来考虑的：**数据规模、技术栈生态、运维复杂度。**
>
> 先说规模。我们项目的文档量在500万级别，预计会增长到2000万。这个量级，Chroma就不够了——它是轻量级的，不支持分布式。FAISS是个算法库，不是数据库，没有持久化、没有并发控制，适合研究不适合生产。我们重点对比了Milvus、Qdrant和Elasticsearch。
>
> 再说技术栈生态。我们是Java技术栈，Milvus有完善的Java SDK（milvus-sdk-java），Qdrant是gRPC接口也可以用，ES就更不用说了，Java原生。另外我们团队已经有Elasticsearch集群在用，ES 8.x原生支持kNN向量检索，而且可以做BM25全文检索+向量检索的混合查询，用RRF融合，这个对我们非常重要，因为我们就是要做混合检索的。
>
> 最终我们选了**双引擎方案**：Elasticsearch做混合检索（向量+BM25），这是主检索引擎。Milvus做大规模纯向量检索的备选，如果未来数据量超过ES的处理能力，可以平滑迁移。
>
> 不过如果是从零开始，没有历史包袱，我建议根据数据规模来选：100万以下用pgvector（简单可靠），100万到5000万用Qdrant（Rust写的高性能），5000万以上用Milvus（分布式架构）。如果有ES运维经验，ES 8.x是最务实的选择，一套系统同时搞定向量检索和全文检索。
