## 一、知识点讲解

### 1. 场景设计题的本质

Agent 场景设计题考察的不是你能否写出完美代码，而是你的**思维模型**——面对一个真实业务场景，你能否：
- 准确理解问题的核心诉求
- 将复杂问题拆解为可落地的子问题
- 设计出合理的 Agent 架构和流程
- 清楚每个决策的 tradeoff

> **一句话**：面试官考的是你"像架构师一样思考"的能力，而不是"像程序员一样写代码"的能力。

### 2. 万能回答框架（5 步）

任何场景设计题都可以用这个框架组织回答：

```
Step 1: 需求分析
  "让我先理解一下这个场景的核心诉求..."
  → 明确用户是谁、核心痛点是什么、成功标准是什么

Step 2: 问题拆解
  "这个问题可以拆分为几个子问题..."
  → 将大问题分解为独立可解决的子问题

Step 3: 逐个击破
  "对于每个子问题，我的方案是..."
  → 用 Agent 知识体系（ReAct、Memory、Tool、Multi-Agent）逐一解决

Step 4: 综合流程
  "把这些方案串起来，整体流程是..."
  → 画出完整的数据流和控制流

Step 5: 优化演进
  "如果要进一步优化，我会考虑..."
  → 展示你懂得权衡和持续改进
```

> **关键**：不要跳步。先需求分析再方案设计，先整体架构再细节实现。面试官更看重你的思维过程，而非最终答案。

### 3. 面试官考察的三个层次

| 层次 | 考察能力 | 典型问题 | 判定标准 |
|------|---------|---------|---------|
| **认知层** | 是否理解问题 | "你觉得这个场景的难点在哪？" | 能准确识别核心矛盾 |
| **设计层** | 能否解决问题 | "你会怎么设计这个系统？" | 方案合理、可落地 |
| **取舍层** | 懂得权衡 | "如果延迟和准确性冲突怎么办？" | 能说出 tradeoff 和选择理由 |

> 大多数候选人卡在认知层就冲去设计层，导致方案偏离业务需求。**先理解问题，再解决问题**。

### 4. 分布式系统的五个根本矛盾（CAP 扩展）

Agent 系统本质上是分布式系统，以下五个矛盾在场景设计中反复出现：

| 矛盾 | 说明 | Agent 场景示例 |
|------|------|---------------|
| **一致性 vs 可用性** | 数据强一致会牺牲响应速度 | 多 Agent 间的状态同步 |
| **延迟 vs 可靠性** | 快速响应 vs 经过验证的可靠响应 | 直接回答 vs 多步验证 |
| **成本 vs 性能** | 更强的模型/更多的步骤 vs 更低的成本 | GPT-4o vs GPT-4o-mini |
| **实时性 vs 稳定性** | 即时响应 vs 稳定输出 | 流式输出 vs 完整生成 |
| **复杂度 vs 可维护性** | 更精细的控制 vs 更简单的架构 | 多 Agent 编排 vs 单 Agent |

### 5. 通用问题域拆解

任何 Agent 场景都可以从这五个维度展开：

```
┌─────────────────────────────────────┐
│         功能正确（Functional）         │  → 核心流程跑通
├─────────────────────────────────────┤
│         高可用（Availability）         │  → 异常处理、降级策略
├─────────────────────────────────────┤
│         系统性能（Performance）        │  → 延迟、Token 成本、并发
├─────────────────────────────────────┤
│         可扩展（Scalability）          │  → 新场景、新工具的接入成本
├─────────────────────────────────────┤
│         安全与监控（Security）          │  → 权限控制、审计日志、告警
└─────────────────────────────────────┘
```

## 二、Agent 场景设计真题详解

### 场景 1：设计一个机票业务客服系统

#### 需求分析

核心诉求：用户通过自然语言与客服系统交互，完成机票查询、预订、改签、退票等操作。

关键难点：
- 业务规则复杂（退改签规则因航司/舱位/票价而异）
- 涉及资金操作（预订、退票），容错率极低
- 用户情绪多样（投诉、紧急改签、价格质疑）
- 需要对接多个外部系统（航司 API、支付系统、票务系统）

#### 五层架构设计

```
┌─────────────────────────────────────────────┐
│              对话理解层                        │
│  意图识别 + 槽位提取 + 情感分析 + 上下文管理    │
├─────────────────────────────────────────────┤
│              业务编排层                        │
│  任务拆解 + 流程编排 + 多轮对话管理              │
├─────────────────────────────────────────────┤
│              规则裁决层                        │
│  退改签规则引擎 + 价格计算 + 权限校验            │
├─────────────────────────────────────────────┤
│              知识解释层 (RAG)                   │
│  航班信息检索 + 政策文档查询 + FAQ 匹配          │
├─────────────────────────────────────────────┤
│              执行与风控层                       │
│  API 调用 + 支付执行 + 操作审计 + 人工兜底       │
└─────────────────────────────────────────────┘
```

#### 各层详解

**对话理解层**：
- 意图识别：查询/预订/改签/退票/投诉/闲聊
- 槽位提取：出发地、目的地、日期、航司偏好、舱位等
- 情感分析：识别用户情绪（愤怒/焦虑/满意），触发不同处理策略

**业务编排层**：
- 根据 intent 选择对应的业务流程
- 管理多轮对话状态（已获取哪些信息、还缺哪些）
- 异常流程处理（信息不全时的追问策略）

**规则裁决层**：
- 这里**不适合用 LLM 直接决策**——退改签规则是确定性的、涉及资金的
- 使用传统规则引擎（Drools/自研）处理业务规则
- LLM 负责将自然语言转化为规则引擎的输入参数

**知识解释层**（RAG）：
- 航班信息、机场信息等实时数据：通过 API 获取
- 退改签政策、行李规则等文档：通过 RAG 检索
- 历史常见问题：FAQ 匹配

**执行与风控层**：
- API 调用：航司系统、支付系统、通知系统
- 操作审计：所有资金操作记录完整日志
- 人工兜底：高金额退票、投诉升级等场景转人工

#### AI 能力边界划分

| 适合 AI 做 | 不适合 AI 做 |
|-----------|-------------|
| 理解用户意图 | 最终资金执行 |
| 编排业务流程 | 退改签规则裁决 |
| 解释复杂政策 | 投诉最终处理 |
| 辅助决策（推荐方案） | 越权操作 |

> **核心原则**：AI 适合做"理解、编排、解释、辅助决策"，不适合做"最终裁决和资金执行"。

### 场景 2：设计旅行规划 Agent

#### 需求分析

用户给出模糊的旅行需求（如"下个月想去海边放松几天"），Agent 需要生成完整的旅行方案。

关键难点：
- 需求模糊，需要多轮追问
- 多维度约束（预算、时间、偏好、交通、住宿、景点）
- 信息来源多样（机票、酒店、景点、天气、签证）
- 方案需要全局最优而非局部最优

#### 按流程拆分子 Agent（编排者-工作者模式）

```
用户输入: "下个月想去海边放松几天"
        ↓
┌──────────────────┐
│ 意图理解 Agent    │ → 解析出: 目的地类型=海边, 时间=下个月, 目的=放松
└──────────────────┘
        ↓
┌──────────────────┐
│ 追问 Agent       │ → 追问: 预算范围? 出发城市? 同行人数? 有无特殊需求?
└──────────────────┘
        ↓ (获取完整信息后)
┌──────────────────────────────────────────┐
│             编排者 Agent                   │
│  并行调度以下子 Agent:                      │
│  ├── 交通查询 Agent → 搜索机票/火车/自驾方案  │
│  ├── 酒店筛选 Agent → 按位置价格评分筛选      │
│  └── 景点规划 Agent → 推荐景点并分类          │
└──────────────────────────────────────────┘
        ↓
┌──────────────────┐
│ 日程编排 Agent    │ → 整合交通+住宿+景点，生成日程表，处理时间冲突
└──────────────────┘
        ↓
┌──────────────────┐
│ 预算核算 Agent    │ → 汇总费用，与用户预算对比，给出优化建议
└──────────────────┘
        ↓
    完整旅行方案输出
```

#### 关键设计决策

1. **并行 vs 串行**：交通、酒店、景点查询可以并行，但日程编排必须串行（依赖前三者结果）
2. **冲突处理**：航班时间与景点安排冲突时，编排者 Agent 仲裁（优先级：交通 > 住宿 > 景点）
3. **预算约束**：预算核算 Agent 在方案生成后验证，超预算则触发重新规划（调整酒店档次或减少景点）
4. **记忆机制**：记住用户的偏好（如上次选了经济型酒店），下次推荐时优先匹配

### 场景 3：AI 客服获取信息不全怎么办

#### 问题分析

用户的问题可能缺少关键信息，Agent 需要判断：
- 哪些信息是必需的（缺失就无法执行）
- 哪些信息可以通过系统反查获取（不需要问用户）
- 何时应该追问 vs 何时应该主动查询

#### 解决方案

```
用户输入（信息不全）
    ↓
Step 1: 定义必填字段
    每个业务场景有一个必填字段清单：
    - 机票查询: [出发城市, 到达城市, 出发日期] ← 必填
    - 酒店预订: [城市, 入住日期, 离店日期] ← 必填
    ↓
Step 2: 区分"用户提供"和"系统可反查"
    ┌──────────────┬──────────────────────┐
    │ 必须用户提供   │ 系统可反查            │
    ├──────────────┼──────────────────────┤
    │ 出发/到达城市  │ 用户历史订单          │
    │ 出发日期      │ 用户会员等级          │
    │ 乘客人数      │ 常用出发城市(从历史)   │
    └──────────────┴──────────────────────┘
    ↓
Step 3: 动态追问
    - 一次只问一个问题（避免信息过载）
    - 优先问影响范围最大的问题
    - 提供选项降低用户认知负担
    例: "您想从哪个城市出发？北京还是上海？"
    ↓
Step 4: 结构化交互
    用卡片/表单等结构化 UI 辅助信息收集
    ┌─────────────────────────┐
    │ 请补充以下信息：           │
    │ 出发城市: [____]          │
    │ 到达城市: [____]          │
    │ 出发日期: [日期选择器]     │
    │        [确认]             │
    └─────────────────────────┘
    ↓
Step 5: 置信度门控
    LLM 对提取的信息给出置信度评分
    - 置信度 > 0.9: 直接执行
    - 0.7 < 置信度 ≤ 0.9: 确认后执行（"您是要从北京出发吗？"）
    - 置信度 ≤ 0.7: 追问（"请问您从哪个城市出发？"）
    ↓
Step 6: 人工兜底
    超过 3 轮追问仍未获取完整信息 → 转人工
```

### 场景 4：用户追问为何选该酒店，Agent 如何回溯决策

#### 问题分析

用户问"你为什么推荐这家酒店而不是那家？"，这考验 Agent 的**决策可解释性**——不仅要给出好的推荐，还要能解释推荐理由。

#### 解决方案

```
决策链日志结构：
{
  "recommendation_id": "rec_001",
  "hotel": "杭州西湖国宾馆",
  "decision_trace": {
    "step_1_filter": {
      "criteria": "价格 500-800, 评分 > 4.5, 距西湖 < 2km",
      "filtered_out": ["酒店A(价格超)", "酒店B(评分低)", "酒店C(距离远)"],
      "remaining": ["杭州西湖国宾馆", "酒店D", "酒店E"]
    },
    "step_2_rank": {
      "scoring_dimensions": {
        "价格匹配度": {"score": 0.85, "reason": "698元，在预算范围内偏上"},
        "位置评分": {"score": 0.95, "reason": "距西湖800米，步行10分钟"},
        "用户评价": {"score": 0.90, "reason": "4.8分，992条评价"},
        "风格匹配": {"score": 0.80, "reason": "中式风格，用户偏好'有特色'"}
      },
      "final_score": 0.88,
      "rank": 1
    },
    "step_3_compare": {
      "vs_hotel_D": "国宾馆位置更优(+0.15), 价格略高(-0.05)",
      "vs_hotel_E": "国宾馆评分更高(+0.10), 但距离稍远(-0.03)"
    }
  }
}
```

#### 回溯流程

1. **决策链日志记录**：每次推荐时，完整记录筛选、排序、对比的每一步及其依据
2. **自然语言解释**：将结构化的决策日志转化为用户可理解的自然语言
3. **候选对比**：展示被淘汰的候选项及其淘汰原因

```
Agent 回复示例：
"我推荐杭州西湖国宾馆，主要基于以下考量：

1. 位置优势：距西湖仅800米，步行约10分钟，在您的三个候选中距离最近
2. 口碑出色：4.8分（992条评价），是同价位中评分最高的
3. 风格匹配：您之前提到喜欢'有特色'的酒店，国宾馆是经典的中式园林风格

之所以没选酒店D，是因为它距西湖2.3公里，出行不太方便；
没选酒店E，是因为它评分4.3分，低于我设定的4.5分门槛。

如果您更看重性价比，酒店D也是个不错的选择，价格便宜约200元。"
```

### 场景 5：通过 RAG 帮助用户了解界面组件

#### 需求分析

用户在使用一个复杂的管理后台时，看到界面上某个组件不清楚其功能，想通过对话了解。例如："这个'审批流程配置'面板是干什么用的？"

#### 方案设计

```
用户提问: "这个面板是干什么的？"
(可能附带截图)
        ↓
┌──────────────────────────────────────┐
│          截图/上下文解析               │
│  方式1: 截图解析                      │
│    截图 → CV轮廓检测 → OCR文字提取    │
│    → 识别出组件名称/类型/位置          │
│  方式2: DOM 上下文（更可靠）           │
│    获取当前页面的 DOM 结构             │
│    → 提取组件ID/类名/aria-label       │
│    → 精确定位用户指向的组件            │
└──────────────────────────────────────┘
        ↓
┌──────────────────────────────────────┐
│          RAG 检索                     │
│  Query: 组件名称 + 组件类型 + 上下文   │
│  ↓                                   │
│  UI 组件知识库（预构建）               │
│  ├── 产品需求文档 (PRD)              │
│  ├── 设计规范文档                     │
│  ├── 用户手册                        │
│  └── FAQ 历史问答                    │
│  ↓                                   │
│  向量检索 + 关键词检索（混合检索）      │
└──────────────────────────────────────┘
        ↓
┌──────────────────────────────────────┐
│          LLM 生成回答                 │
│  输入: 用户问题 + 检索到的文档片段      │
│  输出: 用自然语言解释组件功能          │
│  + 使用示例 + 相关组件推荐             │
└──────────────────────────────────────┘
```

#### UI 组件知识库构建

| 数据来源 | 处理方式 | 向量化策略 |
|---------|---------|-----------|
| 产品需求文档 (PRD) | 按功能模块切片 | 语义向量 |
| 设计规范文档 | 按组件类型切片 | 组件名 + 语义向量 |
| 用户手册 | 按操作步骤切片 | 关键词 + 语义向量 |
| FAQ 历史问答 | 按 Q&A 对切片 | 问题向量 |

#### 相似组件区分

当存在多个相似组件时（如"审批流程配置"和"审批流程实例"）：
1. 利用用户当前页面上下文（URL、页面标题）缩小范围
2. 展示区分要点："您可能是在问以下两个功能之一..."
3. 如果仍无法区分，追问用户："您是在'设置'页面还是'审批列表'页面？"

## 三、串联理解记忆

### 场景设计题与所有 Agent 模块的综合运用

每个场景设计题都是对 Agent 知识体系的综合考查：

| 知识模块 | 在场景中的应用 |
|---------|--------------|
| [[八股/agent/ReAct]] | 每个 Agent 的核心推理循环 |
| [[八股/agent/Memory]] | 用户偏好的记忆、对话历史的维护 |
| [[八股/agent/MCP]] | Agent 与外部工具/系统的标准化交互 |
| [[八股/agent/7-Multi-Agent多智能体]] | 多 Agent 协作架构的设计 |
| [[八股/agent/8-Agent测试与评估]] | 系统上线后的评估和迭代策略 |

### 场景设计题与后端架构设计的关系

Agent 场景设计题和传统后端架构设计题有大量共通之处：

| 后端架构设计 | Agent 场景设计 |
|------------|--------------|
| 服务拆分 | Agent 角色划分 |
| API 设计 | Tool/接口设计 |
| 数据库选型 | Memory 存储选型 |
| 缓存策略 | LLM 响应缓存 |
| 限流降级 | Token 预算管理 |
| 分布式事务 | 多 Agent 状态一致性 |
| 监控告警 | Bad Case 检测 |

> **如果你后端架构设计能力很强，Agent 场景设计也不会差。** 核心思维模型是通用的。

### 每个场景都可以用 Agent 知识体系拆解

万能拆解模板：

```
1. 这个场景需要几个 Agent？（Multi-Agent 架构选择）
2. 每个 Agent 需要哪些 Tool？（工具设计）
3. Agent 之间怎么通信？（通信机制）
4. Agent 需要记住什么？（Memory 策略）
5. Agent 怎么做决策？（ReAct / 规则引擎）
6. 出错了怎么办？（鲁棒性设计）
7. 怎么评估效果？（评估体系）
8. 怎么持续改进？（Bad Case + Human Feedback）
```

## 四、面试题

### Q1：设计一个机票业务客服系统，用 AI 该怎么实现？

参见"场景 1：五层架构"的详细设计。回答要点：

1. **五层架构**：对话理解层 → 业务编排层 → 规则裁决层 → 知识解释层(RAG) → 执行与风控层
2. **AI 能力边界**：AI 做"理解、编排、解释、辅助决策"，不做"最终裁决和资金执行"
3. **规则引擎 vs LLM**：涉及确定性规则（退改签）用规则引擎，LLM 负责理解和解释
4. **人工兜底**：高金额操作、投诉升级等场景转人工
5. **记忆机制**：记住用户历史订单、偏好，提升服务体验

### Q2：设计旅行规划 Agent 如何拆解任务？

参见"场景 2"的详细设计。回答要点：

1. **编排者-工作者模式**：一个编排者 Agent 协调多个专职子 Agent
2. **按流程拆解**：意图理解 → 追问补全 → 并行查询(交通/酒店/景点) → 日程编排 → 预算核算
3. **并行与串行结合**：查询阶段并行，编排阶段串行
4. **冲突处理**：编排者仲裁，优先级为交通 > 住宿 > 景点
5. **记忆与优化**：记住用户偏好，持续优化推荐

### Q3：AI 客服获取信息不全该怎么解决？

参见"场景 3"的详细设计。回答要点：

1. **定义必填字段**：每个场景有必填字段清单
2. **区分来源**：哪些必须用户提供，哪些系统可反查
3. **动态追问**：一次只问一个问题，提供选项降低认知负担
4. **置信度门控**：高置信度直接执行，低置信度追问确认
5. **人工兜底**：超过 3 轮追问仍未获取完整信息则转人工

### Q4：用户追问为何选该推荐，Agent 如何回溯决策？

参见"场景 4"的详细设计。回答要点：

1. **决策链日志**：完整记录筛选、排序、对比的每一步及其依据
2. **结构化存储**：筛选条件、评分维度、候选对比全部结构化
3. **自然语言解释**：将决策日志转化为用户可理解的解释
4. **候选对比**：展示被淘汰的候选项及其淘汰原因
5. **提供替代方案**：如果用户对推荐不满意，给出替代选项

### Q5：客服 Agent 如何判断是否调用外部 API？

决策逻辑：

1. **意图识别**：LLM 识别用户意图是否需要实时数据（"查航班"需要，"解释退改签规则"不需要）
2. **信息充分性检查**：必填参数是否齐全（缺参数先追问，不盲目调用）
3. **缓存优先**：先检查缓存中是否有有效期内的数据
4. **成本评估**：API 调用是否有成本（付费 API 需评估必要性）
5. **降级策略**：API 不可用时的备选方案（历史数据、友好提示、转人工）

```
用户输入 → 意图识别
              ↓
         需要实时数据？
         ├── 是 → 信息充分？ → 调用 API
         │              └── 否 → 追问补全
         └── 否 → RAG 检索 / 知识库查询 / 直接回答
```

### Q6：通过 RAG 帮助用户了解界面组件？

参见"场景 5"的详细设计。回答要点：

1. **组件定位**：截图解析(CV+OCR) 或 DOM 上下文解析，精确识别用户指向的组件
2. **知识库构建**：PRD、设计规范、用户手册、FAQ 向量化存储
3. **混合检索**：向量检索 + 关键词检索，提高召回率
4. **LLM 生成**：结合检索结果和用户上下文，生成自然语言解释
5. **相似组件区分**：利用页面上下文缩小范围，必要时追问

### Q7：设计一个自动化 AI 漫画创作 Agent

这道题考察复杂创作流程的 Agent 设计能力：

```
用户输入: "帮我画一个关于AI面试的搞笑漫画"
                │
    ════════════╪════════════════════════════════════════
    Step 1: 故事脚本生成
    ══════════════════════════════════════════════════════
    • LLM 生成漫画脚本（分镜 + 对话 + 场景描述）

                │
    Step 2: 角色一致性保障
```

| 难点 | 解决方案 |

### Q8：Code Agent（代码 Agent）有哪些特殊挑战？

Code Agent 是 AI Agent 在编程领域的垂直应用，有独特的挑战：

| 挑战 | 说明 | 解决方案 |
|------|------|---------|
| **代码精确性** | 一个符号错误就会导致编译失败 | AST 感知 + 自动编译验证 |
| **上下文跨文件** | 修改一个函数可能影响多个文件 | 代码索引 + 依赖分析 |
| **安全风险** | Agent 直接执行代码有安全风险 | 沙箱执行（E2B/Docker）+ 权限最小化 |
| **版本管理** | 多次修改需要可追溯 | 自动 Git 提交 + Diff 审查 |
| **测试验证** | 生成的代码需要测试 | 自动生成单测 + 执行验证 |
| **增量修改** | 不能每次重写整个文件 | 精确的 Edit 操作（行级/函数级） |

```
Code Agent 的安全架构:

  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
  │  LLM 规划    │     │  沙箱执行    │     │  结果校验    │
  │  生成修改方案 │ ──→ │  Docker/E2B  │ ──→ │  Lint + 测试 │
  │              │     │  网络隔离    │     │  人工审查    │
  └──────────────┘     └──────────────┘     └──────────────┘
         ↑                                         │
         └───────── 失败重试 ←─────────────────────┘
```

- **AST 感知** → [[八股/agent/3-RAG检索增强生成]] 中的 Code-RAG
- **增量修改** → [[八股/agent/2-ReAct推理范式]] 中的反思机制

## 五、面试话术

### 话术 1："设计一个 AI 客服系统"

> 我会用**五层架构**来设计这个系统：
>
> 第一层是**对话理解层**，负责意图识别、槽位提取和情感分析。比如用户说"我要退票"，这一层识别出 intent=退票，并提取订单号、航班信息等槽位。
>
> 第二层是**业务编排层**，根据意图选择对应的业务流程。比如退票流程需要：验证订单 → 检查退票规则 → 计算退款 → 确认执行。这一层管理多轮对话状态，知道已经获取了哪些信息、还缺哪些。
>
> 第三层是**规则裁决层**，这里我用传统规则引擎而非 LLM。因为退改签规则是确定性的、涉及资金的，不能交给概率性的 LLM 来决策。LLM 负责将自然语言转化为规则引擎的输入参数。
>
> 第四层是**知识解释层**，用 RAG 检索航班信息、退改签政策、FAQ 等。用户问"为什么退票要扣手续费"，从知识库检索相关政策，LLM 用自然语言解释。
>
> 第五层是**执行与风控层**，负责调用外部 API（航司系统、支付系统），所有资金操作记录完整审计日志，高金额操作和投诉场景自动转人工兜底。
>
> 核心原则是：**AI 做理解、编排、解释，不做最终裁决和资金执行**。

### 话术 2："设计一个旅行规划 Agent"

> 我会用**编排者-工作者模式**来设计，把旅行规划拆分为多个专职子 Agent：
>
> 首先是**意图理解 Agent**，解析用户的模糊需求。用户说"下个月想去海边放松几天"，它需要提取出目的地类型=海边、时间范围=下个月、旅行目的=放松。
>
> 然后是**追问 Agent**，通过多轮对话补全必要信息：出发城市、预算范围、同行人数、住宿偏好等。一次只问一个问题，提供选项降低用户认知负担。
>
> 获取完整信息后，**编排者 Agent** 并行调度三个查询 Agent：交通查询 Agent 搜索机票火车方案，酒店筛选 Agent 按位置价格评分筛选住宿，景点规划 Agent 推荐景点并分类。这三个可以并行执行，提升效率。
>
> 查询结果汇总后，**日程编排 Agent** 整合交通、住宿、景点信息，生成日程表，处理时间冲突。编排者 Agent 负责仲裁冲突，优先级是交通 > 住宿 > 景点。
>
> 最后**预算核算 Agent** 汇总费用，与用户预算对比。如果超预算，触发重新规划——比如降低酒店档次或减少景点。
>
> 整个系统还配备了**记忆机制**，记住用户的偏好和历史选择，下次推荐时优先匹配。同时记录完整的决策链日志，当用户追问"为什么推荐这家酒店"时，可以回溯解释每一步的决策依据。
