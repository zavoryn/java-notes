# 后端 场景设计题 万能回答框架

来源：小红书笔记，作者：程序员流年

---

系统设计题都是有规律的，我认为可以用一套答题框架或是话术来应对这类问题。以后面试官再问场景设计题，就可以直接套用这个框架，实现一套有条理有逻辑的高级回复。

## 一、场景设计题的规律

### 场景题的本质：面试官是在考「思维模型」，而不是细节实现

场景题要答得好，离不开平时的学习和积累。我们答题的时候要有一个清晰的思路。考虑的全面，设计的巧妙都是很加分的点，让面试官看到你有系统且全面的设计思维。

面试官出场景题，比如：
- 如何设计一个分布式 ID 生成系统
- 如何设计一个高可用的短链接系统
- 如何设计一个排行榜系统
- 如何设计一个视频点赞系统
- 如何设计一个抢红包的系统

其实他们不是想听你"写代码"，而是想看你是否具备架构化思维模型，能把复杂问题抽象成几个核心维度去分析。

> 问题千变万化，但思维框架是固定的。

所以你会发现，大多数场景题都能被分解成以下几个「通用问题域」：

| 模块 | 常问内容 | 核心思维 |
|------|----------|----------|
| 功能正确 | 怎么保证不出错？ | 一致性、事务、幂等 |
| 高可用 | 挂了怎么办？ | 多副本、故障转移、降级 |
| 系统性能 | 撑得住高并发吗？ | 缓存、异步、分片、限流 |
| 可扩展 | 增加新业务怎么办？ | 模块化、解耦、分层 |
| 安全与监控 | 怎么监控，怎么恢复？ | 日志、监控、告警、审计 |

## 二、分布式系统的五个根本矛盾（CAP 定理的扩展）

场景设计都是为了解决问题、解决矛盾。所有大型分布式系统，无论是 ID 服务、支付系统还是消息系统，都要面对相同的五个"物理矛盾"：

| 根本矛盾 | 典型场景 | 对应解决思路 |
|----------|----------|-------------|
| 一致性 Vs 可用性 | 写入延迟、分布式事务 | CAP / BASE 思维、幂等重试 |
| 延迟 Vs 可靠性 | 同步写多副本还是异步 | 异步队列、最终一致性 |
| 成本 Vs 性能 | 多副本 Vs 节点数 | 动态扩缩容、热点分流 |
| 实时性 Vs 稳定性 | 秒杀、推送系统 | 限流、队列削峰 |
| 复杂度 Vs 可维护性 | 系统解耦 | 分层、事件驱动、接口抽象 |

所以，回答问题时，要根据具体的业务做出取舍。比如出于成本的考虑，一个业务才数千数万条数据，就没必要做分库分表。当数据增长到数千万时，出于性能的考虑，不得不做分库分表、分布式部署等等。

## 三、面试官想考察的点

### 1. 认知层（理解问题）
你是否能识别系统中的关键挑战。你要做到能拆解场景，比如一致性、高可用、幂等性等等。

### 2. 设计层（解决问题）
你是否知道主流架构方案（考你的技术广度和深度，不知道也没关系，回答要展现你的思路）：
- 举方案：缓存、分片、主从、异步、限流
- 说明为什么

### 3. 取舍层（懂得权衡）
你是否理解 trade-off。意思是在资源有限的情况下，为了达到一个目标而放弃另一个目标所做的权衡或取舍。这种权衡通常是"此消彼长"的关系，即在两者之间进行选择，不可能两者兼得，需要付出代价来换取利益。
- 分析优缺点
- 结合业务选择最合适的

## 四、场景设计回答框架

### 1. 需求分析
- a. 这个系统是干什么的？
- b. 我要实现什么功能？

### 2. 问题拆解（将复杂系统拆分成一个个小部分，很重要的思想）
- a. 这个系统需要重点考虑什么问题？
- b. 比如设计电商系统，要考虑超卖问题、一致性问题
- c. 再比如设计短链系统，要考虑高可用、如何重定向、日志记录

### 3. 逐个问题讲述方案设计
- a. 比如是否需要设计缓存层，几级缓存？
- b. 数据存储是考虑关系型数据库，还是 NoSQL？
- c. 业务场景是否需要实现强一致性？最终一致性行不行？

### 4. 综合讲述整个系统的运行
- a. 从入口（比如用户点击行为）到出口（比如落库）的整个流程

## 五、实战例子：设计一个短链系统（缩短 URL）

| 步骤 | 你该怎么思考 | 典型回答方向 |
|------|------------|-------------|
| 明确需求 | 唯一、短、可重定向、高可用 | 明确功能 + 非功能要求 |
| 识别挑战 | 高并发、唯一性、热点、过期清理 | |
| 拆解核心问题 | 拆解子问题：ID 生成、映射存储、缓存、负载均衡 | 拆成模块 |
| 做权衡 | 一致性 Vs 吞吐、持久化 Vs 成本 | 讲出 trade-off |
| 给方案 | Redis 缓存 + DB 持久化 + 异步更新 | 输出架构 |
| 优化与演进 | 增加灰度、多机房同步、监控 | 展示前瞻性 |

当然，以上这些步骤和细节不用回答的那么全，只要有条理的按照前面说的框架展开回答，展现出你清晰的逻辑和思维，把系统的关键点回答出来就好。

### 澄清需求（展现需求理解和关键点定位）

在动手设计前先明确需求与约束。在答题之前，一般面试官会把需求讲清楚，告诉你这个系统设计来是干嘛用的。示例问题和默认假设（面试时可向面试官确认或直接给出合理假设）：

**功能性需求：**
- 将长 URL 映射为短 ID（短链）
- 支持重定向（HTTP 301/302）
- 支持自定义别名（用户指定短码）
- 支持短链过期/失效和统计（点击量、来源）
- 支持批量创建/批量返回

**非功能性需求（SLA）：**
- QPS：例如 10k-100k QPS（根据假设）
- P99 重定向延迟 < 10ms
- 可用性：99.99%（多副本 + 自动 failover）
- 数据一致性：短链映射要强一致（不要出现映射丢失），但统计可以最终一致
- 存储：支持亿级短链（可水平扩展）

**约束 / 补充假设：**
- 短码长度目标 6-8 字符（Base62）
- 避免恶意词过滤
- 支持区域多活/跨机房可选（视业务需求）

### 问题拆解（展现清晰设计逻辑）

这部分可以一点一点思考，一点一点说出来，这是你思考的过程，也是向面试官体现你的技术思维。

**1. 短码生成策略（唯一、短、避免碰撞）**

自增 ID + Base62 映射：后端维护一个自增序列（全局或按分片），生成：id → Base62(id) 的短码。
- 优点：无碰撞、可预测长度、简单去重
- 缺点：暴露增长模式（可用随机打散或 hash 混淆）、单点分配需做分段分配（DB/Redis/ID 服务）
- 实现：每个短链服务有本地缓存区间（例如租一个 ID 段 1000），从 DB/Redis 原子 INCRBY 获取段，避免频繁请求中心。

**2. 存储设计（高吞吐、分片、持久化）**
- 主数据：关系型 DB（MySQL 分库分表）或分布式 Key-Value（Cassandra/ScyllaDB），取决于读写模式
- 分片策略：通过 id % shard_count 或 consistent-hash 确定分片
- 唯一性保证：short_code 建唯一索引（跨分片需预先分配或用协调服务），常见做法是短码来源是自增，从而唯一性由生成规则保证，减少跨节点冲突

**3. 缓存策略（提高重定向速度，降低 DB 负载）**
- Redis 缓存：short_code → long_url，TTL 长（或不过期，结合回写机制）
- 热点短链使用较长 TTL 并在边缘（CDN）缓存 301
- 缓存穿透：若不存在，返回 404 并短期缓存空结果或使用 Bloom Filter 过滤非法短码
- 缓存击穿：使用互斥锁 / singleflight 来避免并发穿透去 DB

**4. 高可用性 & 容错（节点失效、数据恢复）**
- 服务无状态：将短链服务设计为无状态，便于水平扩容和滚动升级
- 数据分片与副本：DB 使用主从或多副本保证持久性与读扩展；对 KV 存储使用多副本策略（例如 Cassandra）
- ID 分配高可用：如果使用集中式自增，做分段预分配（每个实例租用一个 ID 段），并持久化分配表防止重复
- 跨机房/多活：为低延迟可在多机房部署，采用不同前缀或 data center 位避免短码冲突；或者把读取流量优先路由到本地机房，写操作同步到主数据中心（增加复杂度）
- 宕机恢复：备份 DB，使用 WAL/CDC 恢复映射。短链是关键功能，数据恢复策略要清晰

**5. 监控与运维（指标、告警、容量规划）**

监控指标：
- QPS（创建/定向）、P50/P95/P99 延迟
- Redis 命中率
- DB 查询延迟与错误率
- ID 段耗尽频次、短码冲突或插入冲突次数
- 热点短链的流量分布
- CDN 命中率
- 失败率、错误码分布（404/500/409 等）

告警：
- Redis 掉线
- DB 主从延迟、时延上升
- 异常高并发（可能 DDOS）

### 完整流程（展现整合能力）

整体上，从入口到出口，短链系统是一个典型的读多写少型分布式系统。对"创建链路"强调唯一性、一致性与幂等；对"访问链路"强调极致读性能、缓存命中率和监控。我通过分层缓存（本地 + Redis + CDN）、分布式 ID、异步消息和分片数据库，实现了高可用、高性能、可扩展的短链服务。

> 以上我只是简单的给了一个总结逻辑，具体地大家可以自己练习一下。
