# 大模型基础

## 一、知识点讲解

### 1. 大模型推理原理

大语言模型（LLM）的推理过程本质上是**自回归生成**：根据已生成的 Token，预测下一个 Token 的概率分布，逐步生成完整回答。

#### 1.1 Self-Attention 机制

Self-Attention 是 Transformer 的核心，使模型能关注输入序列中的所有位置：

```
输入: [我, 喜欢, AI]
             │
    ┌────────▼────────┐
    │ Q = X · W_Q     │  Query：当前 Token 想了解什么
    │ K = X · W_K     │  Key：当前 Token 能提供什么信息
    │ V = X · W_V     │  Value：实际携带的信息内容
    └────────┬────────┘
             │
    Attention(Q,K,V) = softmax(Q·K^T / √d_k) · V
```

| 要素 | 说明 | 类比 |
|------|------|------|
| Q（Query） | 当前 Token 的查询向量 | "我在找什么" |
| K（Key） | 每个 Token 的索引向量 | "我有什么信息" |
| V（Value） | 每个 Token 的内容向量 | "我的实际内容" |
| √d_k | 缩放因子，防止点积过大 | 温度调节器 |

**Multi-Head Attention**：将 Q/K/V 投影到多个子空间，分别计算 Attention 后拼接。不同 Head 关注不同语义维度（语法、语义、位置等）。

#### 1.2 Prefill & Decode 两阶段

LLM 推理分为两个截然不同的阶段：

```
用户输入: "请解释什么是Agent"
                 │
    ══════════════╪═══════════════════════════════
    Prefill 阶段（预填充）          Decode 阶段（逐Token生成）
    ═══════════════════════════    ═══════════════════════════
    • 一次性处理全部输入 Token      • 每次只生成 1 个 Token
    • 计算密集型（大量矩阵运算）    • 访存密集型（读KV Cache）
    • 类比：读完整个题目            • 类比：逐字写答案
    • 可并行处理                    • 严格串行
    ═══════════════════════════    ═══════════════════════════
         ↓ 生成 KV Cache →              → 逐 Token 输出
```

| 对比 | Prefill | Decode |
|------|---------|--------|
| 瓶颈 | 计算力（GPU FLOPS） | 显存带宽（HBM带宽） |
| 并行度 | 高（所有输入Token并行） | 低（每次1个Token） |
| 耗时 | 与输入长度成正比 | 与输出长度成正比 |
| KV Cache | 生成并存储 | 读取并追加 |

#### 1.3 KV Cache

KV Cache 是推理加速的核心技术，避免重复计算已处理 Token 的 K/V 向量：

```
无 KV Cache（每次重新计算所有Token的Attention）:
  Step 1: 计算 [T1] 的 Attention
  Step 2: 计算 [T1, T2] 的 Attention        ← T1 重复计算
  Step 3: 计算 [T1, T2, T3] 的 Attention    ← T1, T2 重复计算
  ...

有 KV Cache（只计算新Token，复用历史）:
  Step 1: 计算 T1 → 缓存 K1, V1
  Step 2: 只计算 T2 → 缓存 K2, V2 → 用 [K1,K2] [V1,V2] 做Attention
  Step 3: 只计算 T3 → 缓存 K3, V3 → 用 [K1,K2,K3] [V1,V2,V3]
  ...
```

**KV Cache 显存占用公式**：

```
KV Cache = 2 × num_layers × hidden_dim × seq_len × batch_size × dtype_size

示例（Llama-2-7B，FP16）:
  = 2 × 32 × 4096 × 2048 × 1 × 2 bytes
  ≈ 1 GB（仅KV Cache，不含模型权重）
```

| 问题 | 说明 |
|------|------|
| 显存瓶颈 | 长上下文（100K+ Token）时，KV Cache 可能比模型本身还大 |
| OOM 风险 | Batch Size × Seq Len 过大导致显存溢出 |
| 解决方案 | PagedAttention（vLLM）、MQA/GQA（模型级优化） |

### 2. 推理优化技术

#### 2.1 FlashAttention

FlashAttention 是一种**IO 感知的精确 Attention 算法**，通过减少 HBM（显存）读写次数来加速计算：

```
标准 Attention:
  HBM → SRAM: 加载完整 Q, K, V        （大量数据搬运）
  SRAM → HBM: 写回 S = Q·K^T          （中间矩阵）
  HBM → SRAM: 重新加载 S              （再次搬运）
  SRAM → HBM: 写回 softmax(S)·V 的结果

FlashAttention:
  HBM → SRAM: 分块加载 Q, K, V 的小块  （减少搬运量）
  SRAM 内部: 完成Attention计算          （利用SRAM高带宽）
  SRAM → HBM: 只写回最终结果           （减少中间写出）
```

| 优势 | 说明 |
|------|------|
| 速度提升 | 2-4x 加速（减少 HBM 读写次数） |
| 显存节省 | O(N) 显存（标准 Attention 为 O(N²)） |
| 精确计算 | 数学结果与标准 Attention 完全一致 |

#### 2.2 PagedAttention（vLLM 核心）

PagedAttention 借鉴操作系统**虚拟内存分页**思想管理 KV Cache：

```
传统方式：为每个请求预分配最大长度的连续显存
  → 大量浪费（预分配了可能用不到的空间）
  → 碎片化（不同请求释放后产生空闲间隙）

PagedAttention：将 KV Cache 分成固定大小的"页"
  → 按需分配（用多少分配多少页）
  → 消除碎片（页可以不连续存储）
  → 支持共享（相同前缀的请求共享页）
```

| 对比 | 传统预分配 | PagedAttention |
|------|-----------|---------------|
| 显存利用率 | 20%-40% | 90%+ |
| 碎片问题 | 严重 | 几乎消除 |
| Batch Size | 受限于预分配 | 显存允许范围内最大化 |
| 类比 | 给每个程序预分配全部内存 | 操作系统按需分页 |

#### 2.3 Continuous Batching

传统 batching 等最慢的请求完成后才能处理下一批，Continuous Batching 在请求完成后立即插入新请求：

```
Static Batching:
  Request A: ████████████（生成完成但等待其他请求）
  Request B: ████████████████████（最慢的请求）
  Request C: ████████（生成完成但等待其他请求）

Continuous Batching:
  Request A: ████████████
  Request B: ████████████████████
  Request C: ████████
  Request D:             ██████████（A完成后立即插入）
  Request E:                     ████████（C完成后立即插入）
```

#### 2.4 Speculative Decoding（投机解码）

用**小模型快速生成草稿**，大模型**一次性并行验证**多个 Token：

```
小模型（Draft Model）快速生成 k 个候选 Token: [T1, T2, T3, T4]
大模型（Target Model）一次性验证:
  T1 ✓ → T2 ✓ → T3 ✗ → 接受 T1, T2，从 T3 开始重新生成

理想情况（小模型预测准确率高）:
  原本逐 Token 生成需要 4 步 → 投机解码只需 1 步（验证通过）
```

### 3. 推理框架选型

| 框架 | 核心特点 | 适用场景 |
|------|---------|---------|
| **vLLM** | PagedAttention、Continuous Batching、吞吐量最优 | 生产环境高吞吐服务 |
| **SGLang** | RadixAttention（前缀缓存共享）、编程式控制 | 多轮对话、结构化生成 |
| **TensorRT-LLM** | NVIDIA 官方、Kernel 级优化 | NVIDIA GPU 极致性能 |
| **Ollama** | 本地部署简单、GGUF 量化 | 本地开发/个人使用 |
| **TGI** | HuggingFace 官方、FlashAttention 集成 | 快速部署 HF 模型 |

**选型决策树**：

```
需要极致推理性能？
├── 是 → NVIDIA GPU？ → 是 → TensorRT-LLM
│                    → 否 → vLLM
└── 否 → 本地开发？ → 是 → Ollama
       → 快速部署？ → 是 → TGI
       → 多轮对话为主？ → 是 → SGLang
       → 通用生产环境 → vLLM（默认推荐）
```

### 4. 训练与微调范式

#### 4.1 训练三阶段

```
┌─────────────────────────────────────────────────────────────┐
│  Stage 1: 预训练（Pre-training）                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 数据：海量无标注文本（TB级）                               │ │
│  │ 目标：学习语言规律和世界知识                               │ │
│  │ 方法：预测下一个Token（Next Token Prediction）             │ │
│  │ 成本：极高（数万GPU·月）                                  │ │
│  │ 类比：读完整个图书馆                                       │ │
│  └─────────────────────────────────────────────────────────┘ │
│                          ↓                                    │
│  Stage 2: 有监督微调（SFT, Supervised Fine-Tuning）            │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 数据：高质量的指令-回答对（万~十万级）                       │ │
│  │ 目标：学会按指令格式回答                                   │ │
│  │ 方法：在预训练模型基础上，用标注数据微调                     │ │
│  │ 成本：中等（数~数百GPU·小时）                              │ │
│  │ 类比：学会按考试要求答题                                   │ │
│  └─────────────────────────────────────────────────────────┘ │
│                          ↓                                    │
│  Stage 3: 对齐训练（Alignment）                                │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 目标：让模型输出符合人类偏好（安全、有帮助、诚实）            │ │
│  │ 方法：RLHF / DPO / GRPO 等                                │ │
│  │ 成本：较高（需要人类偏好数据或强化学习训练）                  │ │
│  │ 类比：学会"如何更好地与人沟通"                              │ │
│  └─────────────────────────────────────────────────────────┘ │
```

#### 4.2 SFT（有监督微调）

```
原理：用 "指令-回答" 对微调预训练模型
数据格式：
  {"instruction": "解释什么是Agent", "output": "Agent是一种..."}

关键决策：
  ┌─────────────────────────────────────────┐
  │ 全量微调（Full Fine-tuning）              │
  │ • 更新所有参数                            │
  │ • 效果最好但成本高                        │
  │ • 需要多卡训练                            │
  ├─────────────────────────────────────────┤
  │ 参数高效微调（PEFT）                      │
  │ • 只更新少量参数（如 LoRA）               │
  │ • 单卡即可训练                            │
  │ • 效果接近全量微调                        │
  └─────────────────────────────────────────┘
```

#### 4.3 RLHF（基于人类反馈的强化学习）

```
三步走流程:

Step 1: SFT 模型
  预训练模型 + 指令数据 → SFT模型

Step 2: 训练奖励模型（Reward Model）
  同一问题生成多个回答 → 人类排序 → 训练RM学习偏好

Step 3: PPO 强化学习
  SFT模型生成回答 → RM评分 → PPO优化SFT模型
```

#### 4.4 DPO（直接偏好优化）

DPO 绕过奖励模型，**直接用偏好数据优化策略模型**：

```
RLHF:  偏好数据 → 训练RM → PPO优化（复杂，不稳定）
DPO:   偏好数据 → 直接优化（简单，稳定）

数据格式:
  {"prompt": "...", "chosen": "更好的回答", "rejected": "较差的回答"}

DPO Loss: 让模型增大 chosen 的概率，减小 rejected 的概率
```

| 对比 | RLHF | DPO |
|------|------|-----|
| 复杂度 | 高（需要训练RM + PPO） | 低（一步到位） |
| 稳定性 | 不稳定（RL训练难） | 稳定（分类损失） |
| 训练成本 | 高 | 相对较低 |
| 效果上限 | 更高（理论上） | 接近RLHF |

#### 4.5 GRPO（群组相对策略优化）

DeepSeek 提出的 RL 训练方法，**不需要奖励模型**，通过组内相对比较来优化：

```
GRPO 流程:
  1. 对同一问题，模型生成 G 个回答
  2. 用规则奖励（如准确性、格式）对每个回答打分
  3. 在组内进行相对排名（不需要人类偏好）
  4. 用相对排名作为优势函数优化模型

优势: 无需训练奖励模型，降低了训练复杂度
应用: DeepSeek-Math, DeepSeek-R1
```

#### 4.6 LoRA（低秩自适应）

LoRA 是最常用的参数高效微调方法：

```
原始权重矩阵 W (d×d)，冻结不动
           ↑
新增低秩分解矩阵: ΔW = A × B
  A: d×r（r << d，如 r=8）
  B: r×d

最终: Y = (W + ΔW) · X = (W + A×B) · X

可训练参数: 2 × d × r（相比原始 d×d 大幅减少）
```

| 参数 | 说明 | 典型值 |
|------|------|-------|
| r（秩） | 低秩矩阵的维度 | 8, 16, 32, 64 |
| alpha | 缩放因子 | 通常等于 r 或 2r |
| target_modules | 应用 LoRA 的层 | q_proj, v_proj 等 |

| 变体 | 说明 |
|------|------|
| **QLoRA** | 4-bit 量化 + LoRA，单卡可训练 65B 模型 |
| **LoRA+** | 对 A 和 B 使用不同的学习率 |
| **rsLoRA** | 用 √r 而非 r 进行缩放 |

### 5. Prompt Engineering 基础

#### 5.1 核心 Prompt 技术

| 技术 | 说明 | 示例 |
|------|------|------|
| **Zero-shot** | 不给示例，直接提问 | "请翻译：Hello" |
| **Few-shot** | 给几个示例再提问 | "apple→苹果, banana→香蕉, cherry→?" |
| **CoT（Chain-of-Thought）** | 引导模型分步推理 | "请一步步思考..." |
| **角色设定** | 给模型指定角色 | "你是一个资深的Java架构师..." |
| **结构化输出** | 要求JSON/表格格式 | "请以JSON格式输出..." |

#### 5.2 System Prompt 设计原则

```
好的 System Prompt 结构:

1. 角色定义: "你是一个专业的XX助手"
2. 能力边界: "你可以做A、B、C，不能做X、Y"
3. 行为规范: "回答时遵循以下规则..."
4. 输出格式: "请用以下格式输出..."
5. 安全约束: "不要泄露敏感信息..."
6. 示例引导:（可选）给出期望的交互示例
```

### 6. 上下文工程（Context Engineering）

上下文工程是 2025 年出现的新概念，关注**如何高效利用有限的上下文窗口**：

```
┌──────────────────────────────────────────────────┐
│              Context Window（上下文窗口）            │
│  ┌──────┐ ┌──────┐ ┌───────┐ ┌──────┐ ┌───────┐ │
│  │System│ │User  │ │History│ │ RAG  │ │  Tool │ │
│  │Prompt│ │Input │ │对话历史│ │检索结果│ │ Results│ │
│  └──────┘ └──────┘ └───────┘ └──────┘ └───────┘ │
│                                                    │
│  ←─────── 总长度不超过 Context Window ──────────→ │
└──────────────────────────────────────────────────┘
```

#### 6.1 上下文压缩策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **滑动窗口** | 只保留最近 K 轮对话 | 通用场景 |
| **摘要压缩** | 用 LLM 摘要历史对话 | 长对话 |
| **关键信息提取** | 提取事实、偏好等结构化信息 | 需要精确记忆 |
| **检索增强** | 按需检索相关历史 | 大规模对话 |
| **渐进式披露** | 根据任务阶段逐步提供信息 | 复杂任务 |

#### 6.2 渐进式披露（Progressive Disclosure）

渐进式披露是 Skill 系统中重要的设计模式——**不一次性把所有信息塞给 LLM，而是按需提供**：

```
传统方式（全部塞入）:
  Prompt = 完整文档 + 所有工具描述 + 全部历史对话
  → Token 消耗巨大，LLM 容易 Lost in the Middle

渐进式披露（按需提供）:
  Step 1: 只给任务概述和核心工具 → Agent 决定需要什么
  Step 2: 按需加载相关文档片段   → 减少噪音
  Step 3: 按需查询具体工具       → 精准执行
```

### 7. 流式输出机制

| 机制 | SSE (Server-Sent Events) | WebSocket |
|------|--------------------------|-----------|
| 方向 | 服务端 → 客户端（单向） | 双向 |
| 协议 | HTTP/1.1+ | 独立协议 |
| 断线重连 | 内置支持 | 需自行实现 |
| 适用场景 | LLM 流式生成 | 实时交互、Multi-Agent |
| 复杂度 | 低 | 中 |

**SSE 在 LLM 流式输出中的典型应用**：

```
Client: POST /chat/completions (stream: true)
Server:
  data: {"choices": [{"delta": {"content": "你"}}]}
  data: {"choices": [{"delta": {"content": "好"}}]}
  data: {"choices": [{"delta": {"content": "，"}}]}
  ...
  data: [DONE]
```

## 三、面试题

### Q1：LLM 推理的 Prefill 和 Decode 阶段有什么区别？

Prefill 阶段一次性处理全部输入 Token，计算密集型（大量矩阵运算），生成 KV Cache。Decode 阶段逐 Token 生成输出，访存密集型（读 KV Cache），严格串行。Prefill 的瓶颈是 GPU 计算力，Decode 的瓶颈是显存带宽。这个两阶段特性决定了推理优化的方向不同——Prefill 优化靠并行计算（如 Tensor Parallelism），Decode 优化靠减少显存读写（如 PagedAttention）。

### Q2：什么是 KV Cache？为什么它很重要？

KV Cache 是在推理过程中缓存已计算 Token 的 Key 和 Value 向量，避免重复计算。它将时间复杂度从 O(n²) 降到了 O(n)。但 KV Cache 的显存占用随序列长度线性增长，长上下文场景（100K+ Token）下可能超过模型本身的显存。这就引出了 PagedAttention（vLLM）、MQA/GQA 等优化技术。在实际部署中，KV Cache 的管理直接影响最大并发数和推理吞吐量。

### Q3：FlashAttention 的核心思想是什么？

FlashAttention 是一种 IO 感知的精确 Attention 算法。核心思想是：GPU 的计算速度远快于显存读写速度（SRAM 带宽远高于 HBM），所以通过分块计算（Tiling），将数据分块加载到 SRAM 中完成 Attention 计算，减少对 HBM 的读写次数。这带来了 2-4x 的加速，同时将显存占用从 O(N²) 降到 O(N)，且计算结果与标准 Attention 完全一致（精确计算，非近似）。

### Q4：vLLM 的 PagedAttention 是怎么工作的？

PagedAttention 借鉴操作系统虚拟内存分页思想管理 KV Cache。传统方式为每个请求预分配最大长度的连续显存，浪费严重（20%-40% 利用率）。PagedAttention 将 KV Cache 分成固定大小的"页"，按需分配，页之间不需要连续存储。这消除了显存碎片，将利用率提升到 90%+，同时支持跨请求的 KV Cache 共享（如相同 System Prompt 的请求可以共享前缀的页）。

### Q5：SFT、RLHF 和 DPO 有什么区别？

三者是 LLM 训练的不同阶段/方法。SFT 用"指令-回答"对微调预训练模型，让模型学会按指令格式回答。RLHF 分三步：先 SFT，再用人类偏好数据训练奖励模型，最后用 PPO 强化学习优化。DPO 是 RLHF 的简化版，跳过奖励模型训练，直接用偏好数据（chosen/rejected 对）优化策略模型。DPO 更简单稳定，RLHF 理论上限更高但训练更复杂。实际项目中 DPO 越来越主流。

### Q6：LoRA 的原理是什么？为什么它这么受欢迎？

LoRA（Low-Rank Adaptation）的核心思想是：模型微调时的权重变化矩阵 ΔW 具有低秩特性，可以用两个小矩阵 A（d×r）和 B（r×d）的乘积来近似表示，其中 r << d。这样可训练参数从 d² 降到 2dr，大幅降低训练成本。LoRA 受欢迎的原因：① 训练成本低，单卡可训练大模型；② 原始权重冻结，多个 LoRA 适配器可共享同一基础模型，切换成本极低；③ 效果接近全量微调。变体有 QLoRA（4-bit 量化 + LoRA）进一步降低门槛。

### Q7：什么是上下文工程？和 Prompt Engineering 有什么区别？

Prompt Engineering 关注的是"怎么写好单次 Prompt"，是对话层面的优化。上下文工程是更宏观的概念，关注的是"如何在有限的上下文窗口中，高效地组织和管理所有信息"——包括 System Prompt、对话历史、RAG 检索结果、工具返回等所有需要塞入上下文的内容。核心策略包括：滑动窗口、摘要压缩、检索增强、渐进式披露等。在 Agent 场景中，上下文工程尤为重要，因为 Agent 的多轮循环会快速消耗上下文窗口。

### Q8：SSE 和 WebSocket 在 LLM 应用中怎么选？

SSE 适合单向流式输出场景（LLM 生成文本流式返回给客户端），基于 HTTP，实现简单，天然支持断线重连，是大多数 ChatGPT 类应用的首选。WebSocket 适合双向实时交互场景（如 Multi-Agent 系统中 Agent 之间实时通信，或需要客户端在生成过程中发送取消指令），协议更复杂但功能更强大。在实际项目中，大多数 LLM 流式服务用 SSE 就够了，只有在需要双向通信时才用 WebSocket。

### Q9：GRPO 和 DPO/RLHF 有什么区别？

GRPO（Group Relative Policy Optimization）是 DeepSeek 提出的训练方法。与 RLHF 需要训练奖励模型、DPO 需要人类偏好对（chosen/rejected）不同，GRPO 通过让模型对同一问题生成多个回答，用规则奖励函数（如准确性、格式正确性）打分，然后在组内进行相对排名来优化模型。GRPO 不需要人类标注偏好数据，也不需要训练额外的奖励模型，降低了训练门槛。DeepSeek-R1 就是用 GRPO 训练的。

### Q10：推理框架 vLLM 和 SGLang 怎么选？

vLLM 核心优势是 PagedAttention + Continuous Batching，吞吐量最优，是生产环境部署的默认选择。SGLang 的核心是 RadixAttention——自动识别和缓存共享前缀（如多轮对话中相同的 System Prompt 和历史），在前缀复用场景下性能更优。选型建议：通用生产环境选 vLLM（生态成熟、社区活跃），多轮对话为主或需要编程式控制生成流程选 SGLang，NVIDIA GPU 且追求极致性能选 TensorRT-LLM。

## 四、面试话术

### 话术 1："请介绍一下你对大模型推理原理的理解"

> 大模型推理本质上是自回归生成——根据已生成的 Token 预测下一个 Token。推理分为两个阶段：Prefill 阶段一次性处理全部输入 Token，是计算密集型；Decode 阶段逐 Token 生成输出，是访存密集型。Prefill 的瓶颈在 GPU 算力，Decode 的瓶颈在显存带宽。
>
> 这个两阶段特性引出了几个关键优化：KV Cache 避免重复计算已处理 Token 的 K/V 向量；FlashAttention 通过减少 HBM 读写次数实现 2-4x 加速；vLLM 的 PagedAttention 借鉴操作系统分页思想管理 KV Cache，将显存利用率从 20%-40% 提升到 90%+。此外 Continuous Batching 允许请求完成后立即插入新请求，Speculative Decoding 用小模型草稿+大模型验证来加速 Decode。
>
> 在实际部署中，推理框架选型也很重要。vLLM 吞吐量最优，是生产环境默认选择；SGLang 的 RadixAttention 在多轮对话场景有优势；TensorRT-LLM 提供 NVIDIA GPU 上的极致性能。

### 话术 2："请介绍一下大模型训练的几个阶段"

> 大模型训练一般分三个阶段。第一阶段是预训练，用海量无标注文本学习语言规律，成本极高。第二阶段是 SFT（有监督微调），用高质量的指令-回答对让模型学会按格式回答。第三阶段是对齐训练，让模型输出符合人类偏好。
>
> 对齐训练有三种主要方法：RLHF 是传统方法，用人类偏好数据训练奖励模型，再用 PPO 强化学习优化，效果好但复杂且不稳定。DPO 是简化方案，跳过奖励模型，直接用偏好对优化策略模型，更简单稳定。GRPO 是 DeepSeek 提出的方法，不需要人类偏好数据，通过组内相对比较优化，降低了数据标注成本。
>
> 在实际微调中，LoRA 是最常用的参数高效微调方法，核心是冻结原始权重，只训练低秩分解矩阵 ΔW=A×B，将可训练参数从 d² 降到 2dr。QLoRA 进一步用 4-bit 量化降低门槛，单卡可以训练 65B 模型。

### 话术 3："你在项目中怎么管理 LLM 的上下文？"

> 上下文管理是 Agent 工程化中的核心问题。我的思路是分层管理：首先是 System Prompt 的设计，明确角色、能力边界和输出格式。然后是上下文窗口的分配策略，根据任务类型合理分配给 System Prompt、对话历史、RAG 检索结果和工具返回。
>
> 具体到压缩策略，根据场景选择：长对话用摘要压缩，需要精确记忆用关键信息提取，大规模历史用检索增强。对于复杂任务，采用渐进式披露——不一次性把所有信息塞给 LLM，而是根据任务进展按需加载。这既能节省 Token 消耗，也能减少 Lost in the Middle 问题。
>
> 在流式输出方面，大多数场景用 SSE 就够了，它基于 HTTP、实现简单、天然支持断线重连。只有在 Multi-Agent 实时通信等需要双向交互的场景才用 WebSocket。
