# MySQL 调优

> [!summary] 我的速记
> **调优 = SQL 调优（慢查询→Explain→优化）+ 架构调优（分库分表）+ 配置调优（参数）。**
> **SQL 执行顺序：FROM→WHERE→GROUP BY→HAVING→SELECT→ORDER BY→LIMIT。**
> **COUNT(*) = COUNT(1) > COUNT(列)。InnoDB 的 COUNT 选最小索引扫。**

---

## 一、SQL 语句执行顺序

### 1.1 书写顺序 vs 执行顺序

```
你写的顺序：            MySQL 实际执行顺序：
SELECT ...              FROM → ON → JOIN →
FROM ...                WHERE →
JOIN ... ON ...         GROUP BY →
WHERE ...               HAVING →
GROUP BY ...            SELECT →
HAVING ...              DISTINCT →
ORDER BY ...            ORDER BY →
LIMIT ...;              LIMIT
```

### 1.2 记忆口诀：F-W-G-H-S-O-L

| 字母 | 关键字 | 作用 |
|------|--------|------|
| **F** | FROM | 确定数据来自哪张表 |
| **W** | WHERE | 过滤原始行（先过滤再分组省开销） |
| **G** | GROUP BY | 分组 |
| **H** | HAVING | 过滤分组后的结果 |
| **S** | SELECT | 选出需要的列 |
| **O** | ORDER BY | 排序 |
| **L** | LIMIT | 截取行数 |

> 🧠 **为什么 WHERE 中不能用 SELECT 的别名？** 因为 SELECT 在 WHERE 之后执行——别名还没定义。但 ORDER BY 在 SELECT 之后，可以用别名。

## 二、COUNT 函数深析

### 2.1 三种 COUNT

| 写法          | 含义             | NULL 处理             |
| ----------- | -------------- | ------------------- |
| `COUNT(*)`  | 统计所有行          | 包含 NULL             |
| `COUNT(1)`  | 统计所有行（每行放常量 1） | **和 COUNT(*) 完全等价** |
| `COUNT(列名)` | 统计该列非 NULL 的行  | **跳过 NULL 值**       |

### 2.2 为什么 COUNT(*) 比 COUNT(主键) 快？

```
COUNT(主键)：走聚簇索引（主键索引）→ 每页包含完整行数据 → 页大行少 → IO 多
COUNT(*)：   优化器选最小的二级索引 → 只存索引列+主键 → 页小行多 → IO 少
```

> 🧠 **COUNT(*) 会选最小的索引来扫——通常是某个二级索引，页 16KB 能装的行比主键索引多得多。**

### 2.3 MyISAM vs InnoDB

| | MyISAM | InnoDB |
|---|---|---|
| COUNT(*) | **O(1)** — 元数据中存了总行数 | **O(n)** — 必须实时扫描 |
| 原因 | 无 MVCC，行数固定 | MVCC 下不同事务看到不同行数 |

> 🧠 **InnoDB COUNT 慢是 MVCC 的代价——为了事务隔离，必须实时数一遍。**

## 三、慢查询分析

### 3.1 三步定位法

```bash
# ① 开启慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过1秒记录

# ② 分析慢查询日志
mysqldumpslow -s t -t 10 slow.log   -- 按时间排序 TOP 10
# 或用 pt-query-digest（Percona Toolkit）

# ③ EXPLAIN 分析具体 SQL
EXPLAIN SELECT * FROM orders WHERE user_id = 100;
```

### 3.2 常见 SQL 优化

| 问题              | 优化                                                                                       |
| --------------- | ---------------------------------------------------------------------------------------- |
| **SELECT \***   | 只取需要的列→覆盖索引                                                                              |
| **索引列用函数**      | `WHERE YEAR(create_time)=2024`→`WHERE create_time BETWEEN '2024-01-01' AND '2024-12-31'` |
| **隐式类型转换**      | `WHERE phone=13800138000`（phone 是 varchar）→ `WHERE phone='13800138000'`                  |
| **大偏移量**        | `LIMIT 1000000,10`→`WHERE id>1000000 LIMIT 10`（延迟关联）                                     |
| **OR 导致索引失效**   | 改为 `UNION ALL`                                                                           |
| **JOIN 小表驱动大表** | 小结果集放驱动表，关联列建索引                                                                          |
| **批量操作**        | 多条 INSERT 合并→减少交互次数                                                                      |

## 四、分库分表

### 4.1 垂直拆分 vs 水平拆分

```
垂直拆分（按业务模块）：
  单体库 → 用户库 + 订单库 + 商品库
  优点：解耦业务  缺点：跨库 JOIN 困难

水平拆分（按数据行）：
  一张 5000 万行表 → 5 张 1000 万行表（分布在不同库/表）
```

### 4.2 四种水平分片方案

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **范围分片** | `id 1~1000万→库0` | 扩容简单 | 数据不均匀（新数据热） |
| **哈希取模** | `hash(id)%4→库N` | 绝对均匀 | 扩容要全量迁移 |
| **一致性哈希** | 哈希环+虚拟节点 | 扩缩只影响相邻 | 实现略复杂 |
| **映射表** | 查映射表找分片 | 最灵活 | 多一次查询 |

> 🧠 **核心矛盾：数据均匀 vs 扩容方便。** 中小规模用哈希取模，大规模用一致性哈希或 ShardingSphere。

## 五、Explain 关键速查

| 列 | 含义 | 评判 |
|----|------|------|
| **type** | 访问类型 | system>const>eq_ref>ref>**range**>index>ALL。range 是底线 |
| **key** | 实际使用的索引 | NULL = 没走索引 |
| **rows** | 预估扫描行数 | 越小越好，最好 < 1000 |
| **Extra** | 额外信息 | Using index✅ / Using filesort❌ / Using temporary❌❌ |
| **key_len** | 用到的索引字节数 | 判断联合索引用了几列 |

## 面试满分话术

> **面试官：SQL 调优怎么做？**

> 慢查询排查分三步：①开 slow_query_log 记录慢 SQL → ②用 mysqldumpslow 或 pt-query-digest 找到 TOP N → ③EXPLAIN 分析执行计划。
>
> Explain 重点看：type（range 是底线，ALL/index 要优化）、key（是否命中期望索引）、Extra（Using filesort 和 Using temporary 必须优化）、rows（扫描行数）。
>
> SQL 执行顺序是 FROM→WHERE→GROUP BY→HAVING→SELECT→ORDER BY→LIMIT，这个顺序解释了为什么 WHERE 中不能用 SELECT 的别名。COUNT(*) 和 COUNT(1) 完全等价，优化器会选最小的二级索引来扫。分库分表的核心矛盾是数据均匀 vs 扩容方便——中小规模用哈希取模，大规模用一致性哈希。
