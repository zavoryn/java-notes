# MySQL 日志系统与主从复制

> 如果把 MySQL 比作一家银行，redo log 是每一笔交易的实时流水账，binlog 是对外报送的正式账单，undo log 则是退款的原始凭证。三条日志各司其职，共同守护数据安全。

---

## 一、redo log —— InnoDB 的保险单

### 1.1 基本概念

| 维度   | 说明                                    |
| ---- | ------------------------------------- |
| 所属层级 | InnoDB 引擎层（存储引擎独有）                    |
| 日志类型 | **物理日志**：记录"数据页上做了什么修改"               |
| 核心机制 | WAL（Write-Ahead Logging，先写日志再写磁盘）     |
| 写入方式 | **顺序写**（追加到日志文件末尾）                    |
| 存储位置 | `ib_logfile0`、`ib_logfile1`，循环写（固定大小） |
| 核心作用 | **崩溃恢复（crash-safe）**：保证事务的持久性（D）      |
|      |                                       |

### 1.2 WAL机制 —— 先记账，后搬货
（**Write-Ahead Logging**） 
比喻：你是一个仓库管理员，每次有人来提货，你有两种做法：

- **方案A（随机写）**：每次直接跑到对应货架位置更新库存卡。仓库里有几千个货架，跑来跑去脚都断了。
- **方案B（WAL + 顺序写）**：先在本子上按顺序记一笔（redo log），等晚上仓库关门了，再集中把所有修改同步到各个货架（刷盘）。

**方案B 比方案A 快 100 倍以上**，原因：
1. 顺序写是连续 I/O，磁盘磁头无需来回寻道
2. 随机写每次都要定位到不同数据页，寻道时间占大头
3. redo log 先写入日志缓冲区（`innodb_log_buffer_size`，默认 16MB），批量刷盘

```
┌─────────────────────────────────────────────────────────┐
│                    事务执行流程                           │
│                                                         │
│  ① 修改数据页（Buffer Pool 中）                           │
│       ↓                                                 │
│  ② 写 redo log buffer（内存）                            │
│       ↓                                                 │
│  ③ 提交事务时 redo log 刷盘（顺序写，极快）                  │
│       ↓                                                 │
│  ④ 返回客户端"提交成功"                                   │
│       ↓                                                 │
│  ⑤ 后台线程慢慢将脏页刷到磁盘（随机写，慢但异步）              │
└─────────────────────────────────────────────────────────┘
```

### 1.3 崩溃恢复 —— 为什么 redo log 能保证数据不丢

```mermaid
崩溃发生 → 重启 → 读取 redo log → 重放已提交但未刷盘的修改 → 数据恢复
```

**关键**：即使数据页没来得及刷到磁盘，只要 redo log 里有记录，重启后就能恢复。

参数控制：
- `innodb_flush_log_at_trx_commit`：
  - **= 1**（推荐）：每次提交都刷盘，最安全，crash-safe
  - **= 0**：每秒刷一次，可能丢 1 秒数据
  - **= 2**：每次提交写 OS 缓存，每秒刷盘，MySQL 崩溃不丢但 OS 崩溃可能丢

**面试金句**：
> "redo log 是 InnoDB 的保险单。只要保险单还在，出了事故（崩溃）就能理赔（恢复）。redo log 的顺序写比数据页的随机写快 100 倍，这就是 WAL 的威力。"

## 二、undo log —— 数据的时光机

### 2.1 基本概念

| 维度   | 说明                             |
| ---- | ------------------------------ |
| 所属层级 | InnoDB 引擎层                     |
| 日志类型 | **逻辑日志**：记录"修改前的旧版本数据"         |
| 存储位置 | 回滚段（Rollback Segment），在系统表空间中  |
| 核心作用 | ① **事务回滚** ② **MVCC（多版本并发控制）** |

### 2.2 事务回滚 —— 后悔药机制

```
UPDATE user SET age = 26 WHERE id = 1;  (原 age = 25)

→ undo log 记录：将 id=1 的 age 恢复为 25
→ 如果事务回滚（ROLLBACK），执行 undo log 中的逆向操作
→ 年龄变回 25，就像什么都没发生过
```

### 2.3 MVCC 版本链 —— 时间旅行

```
┌──────────────────────────────────────────────────────────┐
│                    undo log 版本链                        │
│                                                          │
│  当前记录（id=1, age=26）                                 │
│      ↑ roll_pointer                                      │
│  历史版本1（id=1, age=25）← trx_id=100                   │
│      ↑ roll_pointer                                      │
│  历史版本2（id=1, age=24）← trx_id=99                    │
│      ↑                                                   │
│  历史版本3（id=1, age=23）← trx_id=88                   │
│                                                          │
│  每条记录的 roll_pointer 指向它的上一个版本，形成链         │
│  MVCC 读视图（ReadView）决定你能看到链上的哪个版本          │
└──────────────────────────────────────────────────────────┘
```

**时间旅行原理**：
- 事务 A 开始前，数据是 age=25
- 事务 B 把 age 改成 26，提交了
- 事务 A 再读，因为 MVCC 的 **可重复读** 隔离级别，事务 A 沿着 `roll_pointer` 链找到自己开始时的版本（age=25），读到的是"过去"的值

**面试金句**：
> "undo log 是数据的时光机。它把每次修改前的旧版本都记录下来，链条上的每个节点都是一次'穿越点'。回滚时穿越到零，MVCC 读时穿越到你事务开始的那个时间点。这就是 undo log 一鱼两吃：原子性靠它回滚，隔离性靠它提供旧版本。"

## 三、binlog —— Server 层的正式账单

### 3.1 基本概念

| 维度   | 说明                                       |
| ---- | ---------------------------------------- |
| 所属层级 | **Server 层**（所有引擎共用）                     |
| 日志类型 | **逻辑日志**：记录 SQL 语句或行变更的逻辑操作              |
| 核心作用 | ① **主从复制** ② **数据恢复**（基于时间点恢复）           |
| 写入方式 | 追加写（文件写满后切换新文件）                          |
| 存储位置 | `mysql-bin.000001`、`mysql-bin.000002`... |
|      |                                          |

### 3.2 redo log vs binlog 对比

| 对比维度 | redo log         | binlog        |
| ---- | ---------------- | ------------- |
| 所属层级 | InnoDB 引擎层       | Server 层      |
| 日志类型 | 物理日志（页修改）        | 逻辑日志（SQL/行变更） |
| 写入方式 | 循环写（固定大小）        | 追加写（无限增长）     |
| 用途   | 崩溃恢复（crash-safe） | 主从复制、数据恢复     |
| 谁产生  | 只有 InnoDB 有      | 所有存储引擎都有      |
| 文件大小 | 固定（如 4 个文件各 1GB） | 不断增长，需定期清理    |

### 3.3 binlog 三种格式

| 格式            | 记录内容                     | 优点             | 缺点                                      |
| ------------- | ------------------------ | -------------- | --------------------------------------- |
| **STATEMENT** | SQL 语句原文                 | 日志量小           | 可能导致主从不一致（如 `NOW()`、`LIMIT` 无 ORDER BY） |
| **ROW**       | 每行数据的具体变更                | **最安全**，保证主从一致 | 日志量大（改 10 万行就记 10 万行）                   |
| **MIXED**     | 默认 STATEMENT，特殊语句自动切 ROW | 折中方案           | 仍可能遇到边界 case                            |
|               |                          |                |                                         |

**面试重点**：
> "ROW 格式最安全，因为它不记录 SQL 语句，而是记录每一行数据的变化前和变化后。就像摄像头录像，不记录你说了什么命令，而是录下每一帧画面。这样即使从库的 SQL 环境不同（比如时区、字符集），也能得到完全一致的数据。"

### 3.4 binlog 刷盘策略

```
sync_binlog = 0  → 每次提交不刷盘，交给 OS（最不安全）
sync_binlog = 1  → 每次提交都刷盘（推荐，最安全）
sync_binlog = N  → 每 N 次提交刷一次（折中）
```

**面试金句**：
> "binlog 是 Server 层的正式账单，可以向外部（从库、审计系统）公开。redo log 是 InnoDB 私有的内部账本。两者分工明确：redo log 保证单机不丢数据，binlog 保证集群数据一致。"

## 四、两阶段提交 —— redo 和 binlog 的握手协议

### 4.1 为什么需要两阶段提交

**核心问题**：redo log 和 binlog 是两个独立的日志系统，如果分别独立写入，可能出现逻辑不一致。

**灾难场景**：假设先写 redo 再写 binlog
```
① 写 redo log 成功（数据页修改已记录）
② 写 binlog 时 MySQL 崩溃！
→ 重启后：redo log 让主库恢复了数据，但 binlog 没有这条记录
→ 从库拿 binlog 同步，从库没有这条数据
→ 主从数据不一致！
```

### 4.2 两阶段提交流程

```
┌─────────────────────────────────────────────────────────┐
│                    两阶段提交完整流程                      │
│                                                         │
│  1. InnoDB 层：写 redo log，标记为 PREPARE 状态           │
│                      ↓                                  │
│  2. Server 层：写 binlog                                 │
│                      ↓                                  │
│  3. InnoDB 层：将 redo log 标记为 COMMIT 状态             │
│                                                         │
│  口诀：prepare → binlog → commit                          │
│          (预提交)   (写账单)    (确认提交)                 │
└─────────────────────────────────────────────────────────┘
```

### 4.3 崩溃恢复的判断逻辑

```
重启时检查：

情况 1：redo log 里有 prepare，binlog 里也有
→ 该事务已完成，重放 redo log，事务生效 ✅

情况 2：redo log 里有 prepare，binlog 里没有
→ 写 binlog 前崩溃，回滚该事务，保证主从一致 ✅

情况 3：redo log 里只有 commit（完整了）
→ 正常已提交，重放即可 ✅
```

**一句话总结**：
> "两阶段提交就像签字盖章：prepare 是甲签了字，binlog 是乙也签了字，commit 是盖上公章。缺任何一个章，合同（事务）都不算生效。这样保证 redo 和 binlog 的内容永远逻辑一致。"

## 五、主从复制原理

### 5.1 架构全景图

```
┌────────────── 主库 (Master) ──────────────┐
│                                            │
│  ① 事务提交 → 写 binlog                     │
│  ② Binlog Dump 线程：监听从库请求             │
│     → 发现 binlog 有更新 → 推送给从库         │
│                                            │
└───────────────────┬────────────────────────┘
                    │  网络传输 binlog
                    ↓
┌────────────── 从库 (Slave) ───────────────┐
│                                            │
│  ③ IO 线程：接收 binlog → 写入 relay log     │
│            (中继日志，从库的本地的 binlog 副本)  │
│                                            │
│  ④ SQL 线程：读取 relay log → 重放执行        │
│             (把主库的操作在自己身上再做一遍)     │
│                                            │
└────────────────────────────────────────────┘
```

### 5.2 三大线程

| 线程                 | 所在位置 | 职责                          |
| ------------------ | ---- | --------------------------- |
| **Binlog Dump 线程** | 主库   | 监听从库请求，把 binlog 变更推送过去      |
| **IO 线程**          | 从库   | 接收主库推送的 binlog，写入 relay log |
| **SQL 线程**         | 从库   | 读取 relay log，解析并在从库上重放执行    |

### 5.3 三种复制方式

| 复制方式           | 工作机制                           | 数据一致性         | 性能影响  |
| -------------- | ------------------------------ | ------------- | ----- |
| **异步复制**（默认）   | 主库提交完就返回，不等从库                  | 可能丢数据         | 性能最高  |
| **半同步复制**      | 主库提交后，等至少 1 个从库确认收到 binlog 才返回 | 不丢数据（但可能主从延迟） | 有一定损耗 |
| **全同步复制**（组复制） | 所有从库都确认后才返回                    | 强一致           | 性能最差  |

**比喻**：
> - 异步复制 = 群发通知，不管对方看没看到
> - 半同步复制 = 发微信，等对方显示"已读"
> - 全同步复制 = 打电话，等对方亲口说"收到"

### 5.4 面试金句

> "主从复制本质是 binlog 的传输和重放。主库的 Binlog Dump 线程是送货员，从库的 IO 线程是收货员（把货放进 relay log 仓库），SQL 线程是搬运工（从仓库取出货物摆上货架）。三个线程各司其职，形成一条数据同步流水线。"

## 六、主从延迟 —— 为什么从库总是慢一步

### 6.1 五大原因

| 原因        | 详细说明                        | 解决方案           |
| --------- | --------------------------- | -------------- |
| **从库性能差** | 从库机器配置低于主库                  | 从库配置不低于主库      |
| **大事务**   | 主库执行 10 分钟的大事务，从库也要重放 10 分钟 | 拆大事务为小事务       |
| **单线程回放** | MySQL 5.6 及以前，SQL 线程只有 1 个  | 升级 5.7+ 开启并行复制 |
| **网络延迟**  | 主从跨机房/跨地域部署                 | 同机房部署或使用半同步    |
| **锁冲突**   | 从库上有查询持有锁，阻塞 SQL 线程         | 读写分离 + 隔离      |

### 6.2 并行复制（MySQL 5.7+）

```
MySQL 5.6：按库并行（多个数据库可以并行回放）
MySQL 5.7：按组提交并行（LOGICAL_CLOCK）
         → 同一个组提交内的事务互不冲突，可以并行回放
MySQL 8.0：按 WRITESET 并行
         → 基于行的冲突检测，并行度更高
```

**核心思想**：主库上能并行提交的事务，从库上也能并行重放。

**面试金句**：
> "MySQL 5.6 之前的从库像一个收银员，只有一个窗口排队；5.7 的并行复制等于开了多个窗口，只要顾客买的东西不冲突（不在同一个组提交），就能同时结账。8.0 的 WRITESET 更狠，直接按商品分类开窗口，并行度达到理论最大值。"

### 6.3 监控主从延迟

```sql
-- 在从库上执行
SHOW SLAVE STATUS\G;

-- 关键字段：
-- Seconds_Behind_Master: 主从延迟秒数（0 表示无延迟）
-- Slave_IO_Running: IO 线程是否运行
-- Slave_SQL_Running: SQL 线程是否运行
```

## 七、分库分表策略

> "当一张表的数据量超过 2000 万行，MySQL 的性能开始断崖式下跌，分库分表就成了必选项。分库分表就像搬家——要么按房间功能拆分（垂直拆分），要么把东西分装到多个箱子（水平拆分）。"

### 7.1 垂直拆分 —— 按业务模块

```
┌─────────────────────────────────────────┐
│              单体大库                     │
│  user 表  order 表  product 表  pay 表    │
└─────────────────────────────────────────┘
                  ↓ 垂直拆分
┌──────────┐  ┌──────────┐  ┌──────────┐
│  用户库   │  │  订单库   │  │  商品库   │
│  user    │  │  order   │  │ product  │
│  user_ext│  │  pay     │  │ inventory│
└──────────┘  └──────────┘  └──────────┘

优点：解耦业务，每个库独立扩展
缺点：跨库 JOIN 困难，分布式事务
```

### 7.2 水平拆分 —— 按数据行

**一张 5000 万行的表 → 拆成 5 张 1000 万行的表（分布在不同库/表）**

#### 方案一：范围分片

```
id 1~1000万    → 库0
id 1000~2000万 → 库1
id 2000~3000万 → 库2
...

优点：简单直观，扩容时直接加新库
缺点：数据不均匀（新用户活跃，旧用户沉寂），存在热点
```

#### 方案二：哈希取模

```
hash(id) % 4 = 0 → 库0
hash(id) % 4 = 1 → 库1
hash(id) % 4 = 2 → 库2
hash(id) % 4 = 3 → 库3

优点：数据绝对均匀
缺点：扩容时需要重新哈希，数据迁移量大
```

#### 方案三：一致性哈希

```
        0
  ┌──────┐
  │ 节点A │    → 把节点映射到一个哈希环上
  │      │      id 也映射到环上
  │ 节点B │      id 顺时针遇到的第一个节点就是归属
  └──────┘
       2^32-1

虚拟节点：每个物理节点在环上映射多个位置
→ 避免数据倾斜
→ 扩容时只需迁移相邻节点的部分数据
```

#### 方案四：映射表

```
┌──────────────┐
│  分片映射表    │
│ id_range → db │
│ 1~1000w  → db1│
│ 1000~2000→ db2│
└──────────────┘

优点：灵活，可以任意调整
缺点：每次查询多一次映射表查询，映射表成单点瓶颈
```

### 7.3 面试金句

> "分库分表的核心矛盾是 **数据均匀 vs 扩容方便**。哈希取模最均匀但扩容最难，范围分片扩容最易但最不均匀，一致性哈希是两者的折中。实际项目中，中小规模用哈希取模够用，大规模用一致性哈希或基于 ShardingSphere 的分片算法。"

## 八、SQL 语句执行顺序

### 8.1 书写顺序 vs 执行顺序

```
书写顺序（你看到的）：
SELECT DISTINCT column, AGG_FUNC()
FROM table
  JOIN another_table ON ...
WHERE ...
GROUP BY ...
HAVING ...
ORDER BY ...
LIMIT ...;

执行顺序（MySQL 实际做的）：
FROM → ON → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
```

### 8.2 记忆口诀

| 口诀         | 含义      |
| ---------- | ------- |
| **F**rom   | 先确定数据来源 |
| **W**here  | 过滤原始行   |
| **G**roup  | 分组      |
| **H**aving | 过滤分组后结果 |
| **S**elect | 选出需要的列  |
| **O**rder  | 排序      |
| **L**imit  | 限制条数    |

**口诀**：**F**rank **W**ent **G**o **H**ome **S**tudying **O**n **L**ine
或中文：**从**哪里 **条**件 **分**组 **过**滤 **选**什么 **排**序 **限**制

### 8.3 为什么会不一样

```
为什么 WHERE 在 GROUP BY 前面？
→ 因为 WHERE 过滤的是原始行，必须先过滤再分组，节省分组开销

为什么 HAVING 在 GROUP BY 后面？
→ 因为 HAVING 过滤的是聚合后的结果，必须先有聚合结果才能过滤

为什么 SELECT 中的别名不能在 WHERE 中用？
→ 因为 SELECT 在 WHERE 之后执行，别名还没定义
→ 但可以在 ORDER BY 中用（ORDER BY 在 SELECT 之后）
```

## 九、COUNT 函数的深度辨析

### 9.1 三种 COUNT 的本质

| 写法          | 含义                            | NULL 处理      |
| ----------- | ----------------------------- | ------------ |
| `COUNT(*)`  | 统计所有行数                        | 包含 NULL      |
| `COUNT(1)`  | 统计所有行数（每行放个常量 1，统计非 NULL 的 1） | 等价于 COUNT(*) |
| `COUNT(列名)` | 统计该列非 NULL 的行数                | **跳过 NULL**  |

### 9.2 COUNT(\*) = COUNT(1) 完全等价

```sql
-- MySQL 优化器将 COUNT(*) 和 COUNT(1) 视为等价
-- 执行计划完全相同，查询优化器会选择最小的索引来扫描

EXPLAIN SELECT COUNT(*) FROM orders;
EXPLAIN SELECT COUNT(1) FROM orders;
-- 两条语句的 Extra 都是 "Select tables optimized away" 或走相同索引
```

### 9.3 为什么 COUNT(\*) 比 COUNT(主键) 快

```
COUNT(主键)：必须遍历聚簇索引（主键索引），每个页包含完整行数据
             页大 → 一页存的行少 → 需要读更多页 → IO 多

COUNT(*)：   优化器会选择最小的二级索引（非聚簇索引）
             二级索引只存索引列+主键值，不存完整行
             页小 → 一页存更多行 → 需要读更少页 → IO 少
```

**比喻**：
> "COUNT(主键) 像去图书馆一本一本翻完整书籍统计，每本书好几百页。COUNT(*) 像只翻目录卡片，一张卡片只有书名，效率差几十倍。"

### 9.4 MyISAM vs InnoDB 的 COUNT 性能

```
MyISAM：把表的总行数存储在元数据中
        COUNT(*) = O(1)，瞬间返回

InnoDB：因为 MVCC，每行可能有多个版本，不同事务看到不同的行数
        必须实时扫描索引 → O(n)
        → 这是 InnoDB 为支持事务付出的代价
```

### 9.5 面试金句

> "COUNT(*) 和 COUNT(1) 在 MySQL 里等价，优化器会选择最小的二级索引来扫描。COUNT(列名) 会跳过 NULL 值，所以结果可能小于 COUNT(*)。MyISAM 的 COUNT(*) 是 O(1)，因为它在元数据里存了总行数，但 InnoDB 为了支持 MVCC 必须实时扫描——这就是事务能力的代价。"

## 面试满分话术

### 场景：面试官问 "说说你对 MySQL 日志系统的理解"

> "MySQL 有三条核心日志，我用一个银行柜台来打比方：
>
> **redo log 是柜员的实时流水账**。每做一笔操作，他不是立刻去金库更新存折——那太慢了——而是先在本子上按顺序记一笔。等一天营业结束，再统一去金库更新所有人的存折。这就是 WAL（先写日志再写磁盘），顺序写比随机写快 100 倍。如果突然停电，有流水账在，第二天就能恢复所有已完成的交易。这就是 crash-safe。
>
> **binlog 是银行对外报送的正式账单**。redo log 是 InnoDB 私有的内部账本，binlog 是 Server 层所有引擎共用的，用于主从复制和数据恢复。三种格式里 ROW 最安全——它不是记 SQL 语句，而是像监控摄像头一样记录每一行数据的实际变化。
>
> **undo log 是退款凭证**。每次修改前，它把旧版本存下来。需要回滚时，根据退款凭证恢复原状；需要读历史数据时（MVCC），沿着退款凭证链找到当时的版本。
>
> **两阶段提交**把这三者串在一起：redo prepare（预提交）→ 写 binlog → redo commit（确认提交），确保 redo 和 binlog 逻辑一致。任何一个环节崩溃，恢复时都能正确判断事务的最终状态。"

### 场景：面试官问 "主从延迟怎么解决"

> "主从延迟有五个常见原因，每个都有对应的解法：
>
> 1. **从库机器差** → 从库配置不要低于主库
> 2. **大事务** → 把大事务拆成小事务，比如一次删 10 万行改成每次删 1000 行
> 3. **单线程回放（核心原因）** → 升级 MySQL 5.7+ 开启并行复制。5.7 按组提交并行，8.0 按 WRITESET 并行，并行度更高
> 4. **网络延迟** → 主从尽量同机房部署，避免跨地域
> 5. **锁冲突** → 读写分离时，从库查询不要用 `FOR UPDATE`
>
> 监控方面，通过 `SHOW SLAVE STATUS` 关注 `Seconds_Behind_Master` 和两个线程的运行状态。"

## 记忆口诀

### redo log / binlog / undo log 三部曲

```
redo → 物理、引擎层、循环写、崩溃恢复
bin  → 逻辑、Server层、追加写、主从复制
undo → 逻辑、引擎层、回滚段、事务回滚 + MVCC

口诀：redo 保恢复，binlog 保同步，undo 保回滚
```

### 两阶段提交

```
prepare → binlog → commit
预备       写账单     确认

有备有据才确认，少了谁都不行
```

```
Dump 送（主库推送）

送货 → 收货 → 干活
```

```
Frank Went Go Home Studying On Line
```

## 三日志协作全景图

```
                     ┌─────────────┐
                     │  客户端请求   │
                     └──────┬──────┘
                            ↓
                     ┌─────────────┐
                     │  UPDATE ... │
                     └──────┬──────┘
                            ↓
            ┌───────────────┼───────────────┐
            ↓               ↓               ↓
     ┌──────────┐    ┌──────────┐    ┌──────────┐
     │ undo log │    │ redo log │    │  binlog  │
     │ 记录旧版本 │    │ 物理页修改 │    │ 逻辑SQL  │
     │  (回滚)   │    │  (crash)  │    │ (主从)   │
     └──────────┘    └────┬─────┘    └────┬─────┘
                          │              │
                          └── 两阶段提交 ──┘
                                 │
                     prepare → binlog → commit
```

> **一句话总结**：undo log 让你能反悔，redo log 让你不怕断电，binlog 让你能把数据复制到任何地方。三条日志，三重保障，这就是 MySQL 在性能和可靠性之间找到的最优解。
