# MySQL 存储引擎与表设计

> **一句话总结**：存储引擎决定了数据如何在磁盘上组织、如何被读写、支持哪些特性；表设计决定了数据如何映射到业务的正确性和效率。选对引擎 + 选对类型 = 地基打牢。

---

## 1. InnoDB vs MyISAM -- 装甲车 vs 跑车

> **InnoDB = 装甲车（安全但稍重），MyISAM = 跑车（快但不安全）**

想象你要运输一批贵重货物：
- **装甲车**（InnoDB）：防弹、防撞、有 GPS 追踪、出了事故能完整恢复。代价是车身重，每公里多烧点油。
- **跑车**（MyISAM）：轻量化、直线加速极快。但一旦撞车，货物可能散落一地，基本无法恢复。

| 维度                         | InnoDB（装甲车）                                       | MyISAM（跑车）                                |
| -------------------------- | ------------------------------------------------- | ----------------------------------------- |
| **事务（Transactions）**       | 支持 ACID 事务                                        | 不支持（你只能手动 `LOCK TABLES`，像用绳子临时捆一下）        |
| **锁粒度（Locks）**             | **行级锁**：只锁正在操作的那几行，其他人可以同时操作其他行——就像超市多个收银台同时工作    | **表级锁**：一有人写，全表排队——就像只有一个收银台的便利店          |
| **外键（Foreign Keys）**       | 支持外键约束，保证引用完整性                                    | 不支持（只能靠应用层自己维护，等于没有交警的十字路口）               |
| **崩溃恢复（Crash Recovery）**   | 有 **redo log**，就像飞机的黑匣子，崩溃后能回放重做，自动恢复             | 没有日志，崩溃后只能祈祷。通常需要 `myisamchk` 手动修复，还可能丢数据 |
| **MVCC（多版本并发控制）**          | 支持。读不阻塞写，写不阻塞读——像银行柜台，每个人拿到一个"时间戳号码"，读到的是自己取号时的快照 | 不支持。读写互相阻塞——读的时候不能写，写的时候不能读               |
| **全文搜索（Full-Text Search）** | MySQL 5.6+ 开始支持                                   | 原生支持，早期优势，现在差距已抹平                         |
| **存储限制**                   | 64TB/表（独立表空间）                                     | 256TB/表（但受文件系统限制）                         |
| **数据存储方式**                 | **聚簇索引**：数据和主键索引存在一起，主键就是数据的物理顺序                  | **非聚簇索引**：索引和数据分开存，索引文件存的是数据文件的指针         |
| **计数性能（COUNT(\*)）**        | 需要逐行扫描（因为有 MVCC，每行可能有多个版本）                        | 极快，因为单独维护了一个行数计数器                         |
| **适用场景**                   | OLTP（在线事务处理）：电商订单、银行转账、用户系统                       | OLAP（在线分析处理）：日志分析、数据仓库、只读报表               |

### 锁粒度的生动比喻

```
InnoDB 行锁：                    MyISAM 表锁：
┌─────────────────────┐         ┌─────────────────────┐
│ 行1: 张三在更新       │         │                     │
│ 行2: 李四在更新  ✓    │         │ 张三在更新           │
│ 行3: 王五在更新  ✓    │         │ 李四等着...  ✗       │
│ 行4: 赵六在查询  ✓    │         │ 王五等着...  ✗       │
└─────────────────────┘         │ 赵六等着...  ✗       │
像音乐节多个入口同时检票         └─────────────────────┘
                                像只有一个检票口，全堵住了
```

### 为什么 MyISAM 曾经流行？

早期的 Web 应用以 **读为主**（CMS、博客、论坛），MyISAM 的读性能确实出色。但随着互联网业务从"展示信息"变成"处理交易"，对事务、并发写入、数据安全的需求急剧上升，InnoDB 就成了标配。

## 2. 为什么 MySQL 5.5 后默认 InnoDB？

MySQL 5.5（2010年）是一个分水岭，就像功能机到智能机的跃迁。Oracle 收购 Sun 后，决定把 InnoDB 设为默认引擎，背后的逻辑非常清晰：

### 2.1 事务支持（ACID）-- 业务的底线

```sql
-- 转账场景：A 转 100 给 B
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- A 扣钱
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- B 加钱
COMMIT;

-- 如果第二条 SQL 失败了怎么办？
-- InnoDB：ROLLBACK，A 的钱不会少 --> 数据一致性有保障
-- MyISAM：第一条已经执行了，第二条失败了 --> A 的钱凭空消失！灾难！
```

没有事务，互联网的核心业务（支付、订单、库存）基本没法安全运行。

### 2.2 行级锁 -- 高并发的基础

一个电商秒杀场景，10000 人同时下单。MyISAM 的表锁让所有人排队，第 10000 个人要等前面 9999 人全部完成；InnoDB 的行锁让不同商品的行可以同时操作，吞吐量天差地别。

### 2.3 崩溃恢复 -- 数据安全的底线

> **比喻**：InnoDB 的 redo log 就像飞机的 **黑匣子**。即使飞机坠毁（MySQL 崩溃），黑匣子记录了坠毁前的一切，可以完整回放还原。

MyISAM 没有这个能力——就像没有黑匣子的飞机，坠毁后只能靠猜测。

### 2.4 MVCC -- 读写不冲突

> **比喻**：像 Git 版本控制。每次事务开始，InnoDB 给数据拍一张"快照"，后续的修改不影响快照中的内容。读事务看到的是"自己的版本"，写事务创建"新版本"。多个人同时读写互不干扰——就像 Git 多人协作，各自在分支上工作，最后合并。

### 2.5 外键约束 -- 数据完整性

```sql
-- 有外键约束
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
-- 如果试图删除一个还有订单的用户，InnoDB 会拒绝 --> 不会出现"幽灵订单"
-- MyISAM：你删你的，订单表完全不知情 --> 数据关系断裂
```

### 总结：为什么是 5.5？

| 版本 | 默认引擎 | 原因 |
|------|---------|------|
| < 5.5 | MyISAM | MySQL AB 时期，MyISAM 是自家亲儿子 |
| = 5.5 | **InnoDB** | Oracle 收购后，InnoDB 也是自家人了；而且互联网时代需要事务+行锁 |
| > 5.5 | InnoDB | 一统江湖，成为唯一真正的"生产级"引擎 |

## 3. 自增主键 vs UUID -- 顺序号牌 vs 随机抽签

> **自增主键 = 银行取号机（1,2,3,4...），UUID = 身份证号（每个人生来就有唯一编号）**

### 3.1 自增主键（AUTO_INCREMENT）

**优点：**
- **存储小**：4 字节（INT）或 8 字节（BIGINT），是 UUID 的 1/4 到 1/2
- **顺序写入**：新数据永远追加在 B+Tree 的最右边，无需页分裂——就像在书的最后一页继续写，不影响前面已经写好的内容
- **可读性好**：`id=10086` 比 `id=a3f2b8c1-d4e5-6789-abcd-ef0123456789` 直观太多
- **索引效率高**：主键索引是聚簇索引，顺序插入使得索引页填充率接近 100%

**缺点：**
- **分布式冲突**：分库分表后，两个库各自自增，会产生重复 ID。需要额外方案（如步长设置、分布式 ID 生成器）
- **暴露业务量**：用户注册后看到 `user_id=9527`，大致能猜到平台有多少用户。竞争对手可能利用这个信息
- **单点瓶颈**：插入性能受限于自增锁（虽然 InnoDB 有优化）

### 3.2 UUID（Universally Unique Identifier）

**优点：**
- **全局唯一**：在分布式系统中天生不冲突，不用协调
- **安全**：不可猜测，遍历 `user_id` 无法获取其他用户信息
- **分布式友好**：各节点独立生成，无需中心化协调

**缺点：**
- **存储大**：16 字节（CHAR(36) 则是 36 字节），占用更多磁盘和内存
- **随机写入导致页分裂**：这是最致命的问题！

```
自增主键插入（顺序）：              UUID 插入（随机）：
B+Tree 叶节点:                     B+Tree 叶节点:
┌───────┬───────┬───────┐         ┌───────┬───────┬───────┐
│ 1│2│3 │ 4│5│6 │ 7│8│9 │         │ a│b│c │ d│e│f │ ...   │
└───────┴───────┴───────┘         └───────┴───────┴───────┘
插入 id=10：追加到最后 ✓           插入 id="k..."：可能插到中间
                                  页已满 → **页分裂** → 数据搬移
                                  性能骤降，索引碎片化严重
```

> **比喻**：自增主键就像在一条街上按顺序盖房子（1号、2号、3号...），后来的永远盖在最末尾。UUID 则是在已建成的街道中间随机拆房重建——每次都要把邻居挤开（页分裂），最终整条街七零八落（索引碎片化）。

### 3.3 自增主键真的会导致主从自增 ID 冲突吗？

这是一个经典误解。**在单主多从的普通主从复制架构中，自增 ID 不会冲突**——因为写操作只在主库发生，所有 ID 由主库统一分配，从库只被动复制。

真正会冲突的场景是 **双主架构** 或 **分库分表后合并数据**：

```sql
-- 双主架构的解决方案：步长偏移法
-- 主库 A：
SET auto_increment_increment = 2;
SET auto_increment_offset = 1;   -- 生成 1, 3, 5, 7...
-- 主库 B：
SET auto_increment_increment = 2;
SET auto_increment_offset = 2;   -- 生成 2, 4, 6, 8...
```

### 3.4 解决方案：雪花算法（Snowflake）

> **一句话**：有序的分布式 ID，取两者的优点。

```
雪花算法 64 位 ID 结构（Twitter Snowflake）：
┌──────────┬────────────┬───────┬──────────┐
│ 1bit 不用 │ 41bit 时间戳 │ 10bit │ 12bit    │
│          │ (毫秒级)    │ 机器ID │ 序列号    │
└──────────┴────────────┴───────┴──────────┘

特性：
- 趋势递增（时间戳在高位，整体有序，B+Tree 友好）
- 全局唯一（时间戳 + 机器 ID + 序列号保障）
- 高性能（本地生成，不依赖数据库，每秒可生成数十万）
```

此外还有 **有序 UUID（UUID v7）** 方案，将时间戳放在 UUID 前面，兼顾了全局唯一和索引友好。

### 3.5 选型决策树

```
需要分布式部署？
├── 是 → 数据量大吗？
│   ├── 是 → 雪花算法 / 有序 UUID
│   └── 否 → UUID 也可以
└── 否 → 需要隐藏业务量？
    ├── 是 → 雪花算法（对外暴露时 hash 一下）
    └── 否 → 自增主键（简单高效）
```

## 4. DATETIME vs TIMESTAMP -- 客观时间 vs 墙上的钟

> **DATETIME = 日历上的日期（客观不变），TIMESTAMP = 墙上的钟（会随时区变化）**

### 4.1 核心差异

| 维度       | DATETIME                      | TIMESTAMP                                                      |
| -------- | ----------------------------- | -------------------------------------------------------------- |
| **时间范围** | `1000-01-01` ~ `9999-12-31`   | `1970-01-01 00:00:01` UTC ~ `2038-01-19 03:14:07` UTC          |
| **存储大小** | 8 字节（MySQL 5.6.4+ 按精度 5-8 字节） | 4 字节                                                           |
| **时区**   | **无时区概念**——存什么拿什么，像照片一样定格那一刻  | **有时区转换**——存入时转为 UTC，读出时转为当前会话时区                               |
| **默认值**  | 不支持自动更新                       | 支持 `DEFAULT CURRENT_TIMESTAMP` 和 `ON UPDATE CURRENT_TIMESTAMP` |

### 4.2 时区转换的陷阱

```sql
-- 场景：你在北京时间（UTC+8）插入一条数据
INSERT INTO events (event_time_ts, event_time_dt)
VALUES ('2026-05-03 12:00:00', '2026-05-03 12:00:00');

-- 数据库中实际存储：
-- TIMESTAMP 列：存储 UTC 时间 2026-05-03 04:00:00 （减了8小时）
-- DATETIME 列：存储 2026-05-03 12:00:00（原样存储）

-- 当你把服务器时区改为 UTC+0 后再查询：
SELECT event_time_ts, event_time_dt FROM events;
-- TIMESTAMP 显示：2026-05-03 04:00:00（时区变了，显示也跟着变）
-- DATETIME 显示：2026-05-03 12:00:00（雷打不动，存什么就是什么）
```

> **比喻**：
> - **DATETIME**：你在北京拍了一张照片，照片上的时间是 12:00。这张照片拿到纽约看，还是显示 12:00——照片本身就是客观记录。
> - **TIMESTAMP**：你记录的是"太阳在头顶正上方的那一刻"。在北京显示 12:00，到了纽约自动显示为当地的"太阳在头顶"的时间——它表达的是一个**时刻**而不是一个**钟面读数**。

### 4.3 选型指南

```
你的数据代表什么？
├── 日志记录时间、订单创建时间
│   → TIMESTAMP（省空间，自动更新，时区感知）
├── 用户生日、历史事件、合同到期日（超过2038年）
│   → DATETIME（范围大，不受时区影响）
├── 跨时区应用（如 SaaS，用户遍布全球）
│   → TIMESTAMP（存入 UTC，展示时按用户时区转换）
└── 定时发布（"2026年10月1日晚上8点发布"）
    → DATETIME（这是一个"日历时间"，不随时区变化）
```

### 4.4 2038 年问题

TIMESTAMP 用 4 字节存储，采用的是 Unix 时间戳（从 1970-01-01 开始的秒数），有符号 32 位整数的最大值对应 2038-01-19 03:14:07。之后会溢出。

> MySQL 8.0+ 已经部分缓解，但如果你设计一个要跑到 2040 年的系统，请用 DATETIME。

## 5. CHAR vs VARCHAR -- 停车位 vs 弹性车位

> **CHAR = 固定停车位（每辆车都占一样大的位置，不管车大车小），VARCHAR = 弹性车位（车多大就占多大，旁边贴个标签记尺寸）**

### 5.1 核心差异

| 维度         | CHAR                          | VARCHAR                         |
| ---------- | ----------------------------- | ------------------------------- |
| **存储方式**   | 定长。不足部分用空格填充（存储时），查询返回时去掉尾部空格 | 变长。用 1-2 字节记录实际长度，后面跟数据         |
| **最大长度**   | 0-255 字符                      | 0-65535 字节（实际最大约 65532，因为开销）    |
| **额外开销**   | 无                             | 1 字节（≤255B 时）或 2 字节（>255B 时）存长度 |
| **空间效率**   | 固定长度数据省开销，变长数据浪费空间            | 变长数据省空间，固定长度数据反而多花 1-2 字节       |
| **读写性能**   | 稍快（定长，CPU 直接按偏移量定位——就像数组寻址）   | 稍慢（需要先读长度前缀，再读数据——多一步间接寻址）      |
| **更新时的碎片** | 定长，更新不改变行长度，不会碎片化             | 变长，更新可能导致行变长，原来的位置放不下 → 页分裂或行迁移 |

### 5.2 可视化对比

```
CHAR(10) 存 "Hello"：
┌──────────────────────────────────┐
│ H │ e │ l │ l │ o │ _ │ _ │ _ │ _ │ _ │   10 字节，空格填充
└──────────────────────────────────┘

VARCHAR(10) 存 "Hello"：
┌───┬─────────────────┐
│ 5 │ H │ e │ l │ l │ o │   6 字节（1 字节长度 + 5 字节数据）
└───┴─────────────────┘
```

### 5.3 更新场景的陷阱

```sql
CREATE TABLE demo (
    id INT PRIMARY KEY,
    name VARCHAR(10)
);
-- 插入 "John"（4 字节数据）
-- 更新为 "Alexander"（9 字节数据）
-- 如果原来位置空间不够 → 行迁移到新位置 → 原位置留一个"转发指针"
-- 转发指针 = 额外 IO，性能下降
```

> **比喻**：CHAR 就像图书馆按书架格子放书——每个格子固定大小，放薄册子浪费空间但查找快（直接走到第 N 格）。VARCHAR 像自助储物柜——每个包裹多大占多大，但需要贴标签，偶尔包裹变大了还得换柜子。

### 5.4 选型指南

```
适合 CHAR 的场景（定长或接近定长）：
├── 身份证号 CHAR(18)
├── 手机号 CHAR(11)
├── 状态码 CHAR(2)  -- '00','01','99'
├── 性别 CHAR(1)    -- 'M','F'
├── MD5 哈希 CHAR(32)
└── 邮政编码 CHAR(6)

适合 VARCHAR 的场景（长度变化大）：
├── 用户姓名 VARCHAR(50)     -- "李明" vs "亚历山大·冯·洪堡"
├── 收货地址 VARCHAR(200)    -- 长度差异巨大
├── 商品描述 VARCHAR(500)
├── 邮箱 VARCHAR(100)
└── URL VARCHAR(2048)
```

### 5.5 一个常见误区

```sql
-- 误区：VARCHAR(10) 一定比 CHAR(10) 省空间？
-- 来算一笔账：
-- CHAR(10) 存 10 个字符：10 字节
-- VARCHAR(10) 存 10 个字符：1 字节（长度）+ 10 字节（数据）= 11 字节

-- 结论：当数据长度固定或接近声明长度时，CHAR 反而更省空间！
```

## 6. InnoDB 行记录格式 -- 数据是如何"打包"的

InnoDB 把数据存在 16KB 的页中，行记录格式决定了如何在这些页中"打包"数据。

面试速答：InnoDB 行记录格式表示一行数据在页中的存储方式。常见格式有 Redundant、Compact、Dynamic、Compressed。现在主要使用 Compact 和 Dynamic。以 Compact 为例，一条记录由变长字段长度列表、NULL 值列表、记录头信息、隐藏列和真实数据组成。隐藏列包括事务 ID 和回滚指针，用于 MVCC。Dynamic 和 Compact 的主要区别在于大字段存储方式，Dynamic 会把 TEXT、BLOB 等大字段更多地放到溢出页中，当前页只保留指针。
### 6.1 四种格式对比

| 格式              | 引入版本       | 特点                                     | 适用场景          |
| --------------- | ---------- | -------------------------------------- | ------------- |
| **REDUNDANT**   | 最古老        | 最早的格式，已不推荐                             | 兼容老版本         |
| **COMPACT**     | MySQL 5.0  | 比 REDUNDANT 节省约 20% 空间，对长字段部分溢出        | 基本场景          |
| **DYNAMIC（默认）** | MySQL 5.7+ | 对 BLOB/TEXT 等大字段**完全溢出存储**，只保留 20 字节指针 | **推荐**，现代默认   |
| **COMPRESSED**  | MySQL 5.1  | 在 DYNAMIC 基础上支持页级压缩（zlib），以 CPU 换空间    | 磁盘紧张、数据可压缩的场景 |

### 6.2 COMPACT vs DYNAMIC -- 关键区别

```
COMPACT 格式（部分溢出）：
┌──────────────────────────────────────┐
│ 主键 │ 其他字段 │ BLOB 前 768 字节 │ 溢出页指针 │
└──────────────────────────────────────┘
        ↑ 前 768 字节还在主记录中

DYNAMIC 格式（完全溢出）：
┌──────────────────────────────────────┐
│ 主键 │ 其他字段 │  20 字节溢出页指针  │
└──────────────────────────────────────┘
        ↑ 大字段全部移到溢出页
```

> **比喻**：COMPACT 和 DYNAMIC 的区别就像**搬家打包**：
> - COMPACT：大件物品锯成两半，一半放主箱子里，一半放副箱子，主箱子里记录副箱子的位置
> - DYNAMIC：大件直接放专门的"大件区"，主箱子里只留一个便签写着"大件在仓库 B 区 3 排"

DYNAMIC 的优势：
- 主记录更紧凑，一个 16KB 页能放下更多行
- 对大字段的 B+Tree 索引扫描只需读取指针，不读取大字段本身
- 对包含 TEXT/BLOB 列的表，DYNAMIC 是最佳选择

### 6.3 查看与设置

```sql
-- 查看当前表的行格式
SHOW TABLE STATUS WHERE Name = 'your_table';

-- 创建时指定
CREATE TABLE products (
    id INT PRIMARY KEY,
    desc TEXT
) ROW_FORMAT=DYNAMIC;

-- 修改已有表
ALTER TABLE products ROW_FORMAT=DYNAMIC;
```

## 7. 前缀索引（Prefix Index）-- 给长字段建"简称索引"

> **前缀索引 = 给超长的人名建简称目录。全名 "亚历山大·谢尔盖耶维奇·普希金"，索引只记前几个字 "亚历山大..."，足够区分就行。**

### 7.1 为什么需要前缀索引？

InnoDB 索引键最大长度是 767 字节（`innodb_large_prefix` 开启后为 3072 字节）。对于 `VARCHAR(2000)` 或 `TEXT` 类型的列，完整索引键太长甚至不允许建索引。前缀索引取前 N 个字符建索引，兼顾了索引大小和区分度。

### 7.2 语法与原理

```sql
-- 对 url 列的前 50 个字符建索引
CREATE INDEX idx_url_prefix ON articles(url(50));

-- 查询时：
SELECT * FROM articles WHERE url = 'https://example.com/very/long/path/...';
-- 步骤：
-- 1. 在索引中找 url 前 50 个字符匹配的条目 → 快速定位候选行
-- 2. 回表读取完整 url → 精确比较是否完全匹配
-- 3. 返回匹配的行
```

### 7.3 如何选择前缀长度？核心原则：区分度

```sql
-- 计算全列的区分度（distinct 值 / 总行数）
SELECT COUNT(DISTINCT url) / COUNT(*) AS selectivity FROM articles;

-- 计算不同前缀长度的区分度，找到拐点
SELECT
    COUNT(DISTINCT LEFT(url, 10)) / COUNT(*) AS len10,
    COUNT(DISTINCT LEFT(url, 20)) / COUNT(*) AS len20,
    COUNT(DISTINCT LEFT(url, 30)) / COUNT(*) AS len30,
    COUNT(DISTINCT LEFT(url, 50)) / COUNT(*) AS len50
FROM articles;

-- 理想情况：找到区分度接近全列的那个长度，即为最优前缀长度
```

> **原则**：在区分度可接受的前提下，前缀越短越好（索引越小 = 内存缓存更多 = 更快）。

### 7.4 前缀索引的局限性

```
不能用于 ORDER BY 和 GROUP BY → 因为索引中只有前缀，不是完整值，无法排序
不能用于覆盖索引 → 必须回表取完整值做精确匹配
```

### 7.5 替代方案

对于确实需要完整索引的长字符串列，可以考虑：
- **反向索引**：对邮件地址 `user@domain.com` 取 `domain.com` 部分建索引
- **Hash 索引**：额外维护一个 CRC32/MD5 列，对该列建完整索引
- **全文索引**：如果是文本搜索场景，直接用 FULLTEXT 索引

## 面试满分话术

### 话术 1：InnoDB vs MyISAM

> "如果把 MySQL 的存储引擎比作交通工具，InnoDB 就是**装甲车**——它支持事务 ACID、行级锁保证高并发、有 redo log 实现崩溃恢复、MVCC 让读写不互相阻塞，代价是稍微重一点。MyISAM 是**跑车**——表级锁让它读写速度极快，但没有事务、不支持外键、崩溃了基本无法恢复。MySQL 5.5 后默认 InnoDB，本质是互联网从'展示型'进入'交易型'时代，对数据安全和并发的要求高了，装甲车虽然重，但能保证货物安全送达。"

### 话术 2：为什么不用 UUID 做主键？

> "UUID 做主键有两个致命问题。第一，它是随机值，插入 B+Tree 时不是顺序追加而是随机插入，会导致大量页分裂——就像一个已建满的街区非要拆一栋插进去，整条街都要挪位置，性能急剧下降。第二，16 字节的存储是自增 INT 的 4 倍，索引也相应膨胀。UUID 在分布式下有优势，但我通常会选**雪花算法**作为最佳折中——时间戳在高位保证整体有序不页分裂，机器 ID + 序列号保证全局唯一，本地生成不依赖数据库。"

### 话术 3：DATETIME vs TIMESTAMP

> "DATETIME 是无时区的，存什么拿什么，像个客观记录——适合生日、历史日期这些不随时区变化的场景。TIMESTAMP 会做时区转换，存入转 UTC、读出按会话时区转回，像个墙上的钟——适合记录'发生时刻'。跨时区应用必须用 TIMESTAMP，但要注意它的上限是 2038 年，如果系统要跑到 2040 年，考虑用 DATETIME。"

### 话术 4：CHAR vs VARCHAR

> "CHAR 是固定停车位——每格一样大，找第 N 个直接走过去，快但可能浪费空间。VARCHAR 是弹性车位——车多大占多大位置，需要多用一个标签记录尺寸。固定长度的数据用 CHAR 其实更省（没有那 1-2 字节长度前缀），比如身份证号；变化大的用 VARCHAR。另外 VARCHAR 更新变长时可能触发行迁移，对高频更新场景要注意。"
