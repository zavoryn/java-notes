# MySQL 事务、MVCC 与锁机制

---

## 一、事务基础

### 1.1 什么是事务？

事务是一组不可分割的 SQL 操作，要么全部成功，要么全部回滚——**原子性（Atomicity）**是事务最核心的承诺。

> 想象你去银行转账：从 A 账户扣 100 元，给 B 账户加 100 元。如果扣款成功但加款失败，这 100 块就凭空消失了——事务就是用来防止这种灾难的。

### 1.2 ACID 四大特性

| 特性 | 英文 | 含义 | 类比 |
|------|------|------|------|
| 原子性 | Atomicity | 要么全做，要么全不做 | 签合同——所有签字页缺一不可 |
| 一致性 | Consistency | 事务前后数据满足所有约束 | 天平——无论你怎么折腾，最终两边要平衡 |
| 隔离性 | Isolation | 并发事务互不干扰 | 自助取款机——你取钱时别人不能同时取同一笔 |
| 持久性 | Durability | 提交后数据永久保存 | 刻在石头上——写上去就擦不掉了 |

> **面试重点**：四个特性不是并列关系——C（一致性）是最终目标，AID 是实现手段。

## 二、四种事务隔离级别

### 2.1 并发问题三兄弟

在理解隔离级别之前，必须先认识"并发问题三兄弟"：

| 问题 | 别名 | 现象 | 通俗解释 |
|------|------|------|----------|
| 脏读 | Dirty Read | 读到别人**未提交**的数据 | 看到草稿——别人还在写呢你就拿去用 |
| 不可重复读 | Non-Repeatable Read | 同一事务内两次读**同一条**记录，结果不一样 | 翻书翻到一半，内容被人改了一行 |
| 幻读 | Phantom Read | 同一事务内两次读**同一个范围**，记录数不一样 | 翻书翻到一半，多了几页/少了几页 |

> **关键区分**：不可重复读是"一条记录内容变了"；幻读是"记录条数多了或少了"。一个关注**值的变化**，一个关注**行的增减**。

### 2.2 四种隔离级别详解

```
隔离级别从低到高：
READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE
    ↓                    ↓                   ↓                ↓
  最不安全              解决了脏读          解决了不可重复读     解决了幻读
  性能最好                                 (但幻读微妙)      最安全/最慢
```

| 隔离级别             | 脏读  | 不可重复读 | 幻读     | 实现原理                   | MySQL 默认   |
| ---------------- | --- | ----- | ------ | ---------------------- | ---------- |
| READ UNCOMMITTED | ✓   | ✓     | ✓      | 无锁无MVCC，直接读最新          | -          |
| READ COMMITTED   | ✗   | ✓     | ✓      | 每次查询创建新 ReadView       | -          |
| REPEATABLE READ  | ✗   | ✗     | ✗(大部分) | 事务开始时创建 ReadView + 临键锁 | ★ MySQL 默认 |
| SERIALIZABLE     | ✗   | ✗     | ✗      | 所有读加共享锁，串行执行           | -          |

**隔离级别越高，数据越安全，但并发越差——这是一把双刃剑。**

#### 2.2.1 READ UNCOMMITTED（读未提交）

```sql
-- 会话A
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
START TRANSACTION;
SELECT * FROM account WHERE id = 1;  -- 看到 balance=100

-- 会话B（同时进行，未提交）
START TRANSACTION;
UPDATE account SET balance = 200 WHERE id = 1;
-- 还没提交！

-- 会话A 再次查询
SELECT * FROM account WHERE id = 1;  -- 看到 balance=200 ← 脏读！
```

**一句话总结**：读不加锁，能看到别人正在改的数据（未提交的脏数据）。生产环境基本不会用。

> 就像一个作家正在书房里写草稿，你直接推门进去看——他可能写完又划掉，你看到的根本不算数。

#### 2.2.2 READ COMMITTED（读已提交）—— 解决了脏读

```sql
-- 会话A
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 100

-- 会话B 提交修改
UPDATE account SET balance = 200 WHERE id = 1;
COMMIT;

-- 会话A 再次查询
SELECT balance FROM account WHERE id = 1;  -- 200 ← 同一事务内两次值不同，不可重复读！
```

**实现原理**：每次 `SELECT` 都创建一个新的 ReadView，所以能看到其他事务已提交的修改。

> **RC 是 Oracle 的默认隔离级别**。它通过"每次都拍新照片"解决了脏读，但因为照片一直在变，带来了不可重复读。

#### 2.2.3 REPEATABLE READ（可重复读）—— MySQL 默认

```sql
-- 会话A
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;
SELECT balance FROM account WHERE id = 1;  -- 100

-- 会话B 提交修改
UPDATE account SET balance = 200 WHERE id = 1;
COMMIT;

-- 会话A 再次查询
SELECT balance FROM account WHERE id = 1;  -- 还是 100 ← 可重复读！
```

**实现原理**：事务开始时创建一个 ReadView，**整个事务都用这一个 ReadView**。就像每个事务在开始时拍了一张数据库的快照，之后所有的读都看这张快照。

> **MySQL 的 REPEATABLE READ 通过"快照读 + 临键锁"几乎解决了幻读**，这和标准 SQL 规范中的定义有所不同。标准 SQL 中的 RR 只能解决不可重复读。

#### 2.2.4 SERIALIZABLE（串行化）—— 终极安全

```sql
-- 所有 SELECT 默认变成 SELECT ... FOR SHARE（读加共享锁）
-- 事务必须排队执行，不能并发
```

**实现原理**：所有读操作自动加共享锁，写操作自动加排他锁。事务完全串行化执行。

> **性能最差**：就像食堂只有一个打饭窗口，所有人必须排队。

### 2.3 修改隔离级别

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置全局隔离级别
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 设置当前会话隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

## 三、MVCC 原理——多版本并发控制

### 3.1 一句话理解 MVCC

> **MVCC 就像给数据拍快照——每个事务看到的是属于自己那个时刻的版本。**

你不需要加锁去读数据，因为数据库为你保存了数据的历史版本。每个人看自己那个时刻的"存档"，互不干扰。

### 3.2 MVCC 的三大核心组件

```
┌──────────────────────────────────────────────┐
│  MVCC 三大组件                                │
│                                              │
│  ① 隐藏列（3个）                              │
│     DB_TRX_ID  ──── 最后修改事务ID            │
│     DB_ROLL_PTR ──── 回滚指针（指向undo log） │
│     DB_ROW_ID   ──── 行ID（无主键时）         │
│                                              │
│  ② Undo Log 版本链                           │
│     每次修改都生成一条undo记录                 │
│     通过 DB_ROLL_PTR 串成历史版本链           │
│                                              │
│  ③ ReadView（读视图/快照）                    │
│     记录了当前活跃事务的快照                   │
│     用来判断哪个版本对当前事务可见             │
└──────────────────────────────────────────────┘
```

#### 3.2.1 三个隐藏列（InnoDB 每一行都有）

```sql
-- 你建的表：
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(20),
    age INT
);

-- InnoDB 实际存储的样子：
-- ┌────┬──────┬─────┬────────────┬──────────────┬──────────┐
-- │ id │ name │ age │ DB_TRX_ID  │ DB_ROLL_PTR  │DB_ROW_ID │
-- ├────┼──────┼─────┼────────────┼──────────────┼──────────┤
-- │ 1  │ 张三 │ 25  │    100     │  0x12345678  │    1     │
-- └────┴──────┴─────┴────────────┴──────────────┴──────────┘
```

| 隐藏列 | 大小 | 作用 |
|--------|------|------|
| **DB_TRX_ID** | 6 字节 | 最近一次**修改**（INSERT/UPDATE）这行数据的事务 ID |
| **DB_ROLL_PTR** | 7 字节 | 回滚指针，指向 undo log 中该行数据的上一个版本 |
| **DB_ROW_ID** | 6 字节 | 单调递增的行 ID（当表没有主键时，InnoDB 用它来创建聚簇索引） |

> **注意**：DELETE 不是真的删除，而是打了一个"删除标记"，事务提交后才由 purge 线程真正清理。所以在 MVCC 中，DELETE 也算一种"修改"。

#### 3.2.2 Undo Log 版本链

每次 UPDATE 都会在 Undo Log 中记录修改前的值，通过 `DB_ROLL_PTR` 指针串成链：

```
假设事务 100 插入一行：(id=1, name='张三', age=25)
事务 200 修改 age→26
事务 300 修改 age→27

最新数据（在B+树叶子节点）：
┌──────┬──────┬──────┬───────────┬──────────────┐
│  id  │ name │ age  │ DB_TRX_ID │ DB_ROLL_PTR  │
├──────┼──────┼──────┼───────────┼──────────────┤
│  1   │ 张三 │  27  │    300    │      ↓       │
└──────┴──────┴──────┴───────────┴──────┼───────┘
                                        │
                          ┌─────────────┘
                          ↓
Undo Log 版本链：
                          ┌───────────┬──────────────┐
                          │  age=26   │   trx_id=200 │──→ │age=25│trx_id=100│
                          └───────────┴──────────────┘    └──────┴──────────┘
                              版本2                            版本1（初始）
```

> 形象比喻：就像 Git 的版本历史——当前 commit 是最新版，每个 commit 都指向前一个 commit，你可以回退到任意历史版本。

#### 3.2.3 ReadView——事务的"快照"

ReadView 有四个关键字段：

| 字段 | 含义 |
|------|------|
| `m_ids` | 创建 ReadView 时，**正在活跃的**（未提交的）事务 ID 列表 |
| `min_trx_id` | `m_ids` 中的最小值 |
| `max_trx_id` | 创建 ReadView 时，系统下一个要分配的事务 ID（即全局最大值+1） |
| `creator_trx_id` | 创建这个 ReadView 的事务 ID |

### 3.3 ReadView 可见性判断（核心算法）

对于版本链上的某个版本，假设它的 `DB_TRX_ID = trx_id`：

```
if (trx_id == creator_trx_id):
    → 可见（自己改的当然能看到）

elif (trx_id < min_trx_id):
    → 可见（修改这条数据的事务在创建ReadView之前就提交了）

elif (trx_id >= max_trx_id):
    → 不可见（修改这条数据的事务在创建ReadView之后才开始）

elif (min_trx_id <= trx_id < max_trx_id):
    if (trx_id in m_ids):
        → 不可见（事务还没提交）
    else:
        → 可见（事务已提交）
```

> **核心逻辑**：顺着版本链从新到旧遍历，找到第一个对当前事务可见的版本，那就是我们应该读到的数据。

### 3.4 快照读 vs 当前读

这是面试中非常高频的考点，必须彻底分清：

|                   | 快照读（Snapshot Read） | 当前读（Current Read）                                                         |
| ----------------- | ------------------ | ------------------------------------------------------------------------- |
| **含义**            | 读历史版本快照            | 读最新版本，且锁定                                                                 |
| **依赖**            | MVCC               | 锁机制                                                                       |
| **SQL 示例**        | 普通的 `SELECT`       | `SELECT ... FOR UPDATE`、`SELECT ... FOR SHARE`、`UPDATE`、`DELETE`、`INSERT` |
| **是否加锁**          | 不加锁                | 加锁                                                                        |
| **RC 下 ReadView** | 每次 SELECT 新建       | -                                                                         |
| **RR 下 ReadView** | 事务开始时创建一次          | -                                                                         |

```sql
-- 快照读（MVCC，不加锁）
SELECT * FROM user WHERE id = 1;

-- 当前读（加锁，读到最新提交的版本）
SELECT * FROM user WHERE id = 1 FOR UPDATE;  -- 加排他锁（X锁）
SELECT * FROM user WHERE id = 1 FOR SHARE;    -- 加共享锁（S锁）
UPDATE user SET age = 30 WHERE id = 1;        -- 隐式当前读
DELETE FROM user WHERE id = 1;                -- 隐式当前读
```

> **关键理解**：`UPDATE` 和 `DELETE` 内部会先做一次"当前读"获取最新数据，然后再修改。所以它们不走 MVCC 快照，而是走锁机制。这也解释了为什么 RR 下"当前读"需要用临键锁来防幻读。

## 四、RR vs RC——ReadView 创建时机的本质区别

这可能是面试中最常见的问题：**"MySQL 的 RC 和 RR 隔离级别有什么区别？"**

### 4.1 核心区别：ReadView 创建时机

|                   | READ COMMITTED               | REPEATABLE READ        |
| ----------------- | ---------------------------- | ---------------------- |
| **ReadView 创建时机** | **每次 SELECT 都创建新的 ReadView** | **事务开始时创建一次 ReadView** |
| **导致的现象**         | 不可重复读（同事务两次读结果可能不同）          | 可重复读（同事务多次读结果相同）       |
| **比喻**            | 每看一次就拍一次新的快照                 | 进门时拍一张快照，整个事务只看这一张     |

### 4.2 图解

```
RC（读已提交）：
事务A开始 ── 拍照1(ReadView)读数据 ── 事务B提交修改 ── 拍照2(新ReadView)读数据(能看到B的修改)
                ↑                                        ↑
            ReadView#1                               ReadView#2  ← 不一样的快照

RR（可重复读）：
事务A开始 ── 拍照(ReadView)读数据 ── 事务B提交修改 ── 还看同一张照片吧(看不到B的修改)
                ↑                                        ↑
            ReadView#1                               ReadView#1  ← 同一张快照
```

> **一句话总结**：RC 是"实时"地看，能看到别人提交的新数据；RR 是"拍照"看，整个事务期间看到的世界是凝固的。

### 4.3 代码示例

```sql
-- RC 下的不可重复读
-- 事务A
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
BEGIN;
SELECT age FROM user WHERE id = 1;  -- age=25

-- 事务B 提交
UPDATE user SET age = 26 WHERE id = 1;
COMMIT;

-- 事务A 再查
SELECT age FROM user WHERE id = 1;  -- age=26 （不可重复读！）

-- ================================

-- RR 下的可重复读
-- 事务A
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
SELECT age FROM user WHERE id = 1;  -- age=25

-- 事务B 提交
UPDATE user SET age = 26 WHERE id = 1;
COMMIT;

-- 事务A 再查
SELECT age FROM user WHERE id = 1;  -- age=25 （可重复读！）
```

## 五、RR 如何解决幻读？

这是面试中的高频追问：**"RR 真的完全解决了幻读吗？"**

答案是：**对于快照读——解决了；对于当前读——需要临键锁配合解决。**

### 5.1 快照读天然防幻读（MVCC 机制）

```sql
-- RR 下，事务开始时创建 ReadView
BEGIN;
SELECT COUNT(*) FROM user WHERE age > 20;  -- 10条（快照读，看的是事务开始时的版本）

-- 事务B 插入一条 age=25 的新记录并提交
INSERT INTO user(name, age) VALUES ('李四', 25);
COMMIT;

-- 事务A 再次查询
SELECT COUNT(*) FROM user WHERE age > 20;  -- 还是10条！（MVCC 快照不变）
```

> ReadView 在事务开始时创建后就不再变化。其他事务插入的新数据的 `DB_TRX_ID` 一定大于 ReadView 的 `max_trx_id`，根据可见性规则，对当前事务不可见——所以幻读**在快照读下被 MVCC 天然解决**。

### 5.2 当前读靠临键锁防幻读（锁机制）

```sql
-- RR 下，使用当前读
BEGIN;
SELECT * FROM user WHERE age > 20 FOR UPDATE;
-- ↑ 会加上 Next-Key Lock，锁定 age>20 这个范围
--    其他事务无法向这个范围插入数据！

-- 事务B 尝试插入
INSERT INTO user(name, age) VALUES ('李四', 25);
-- ↑ 被阻塞！因为 age=25 落在被锁定的范围内
```

> **"快照读靠 MVCC，当前读靠临键锁"**——这句话是 RR 解决幻读的精髓。

### 5.3 一张图总结

```
        RR 防幻读的两种武器
              ↓
    ┌─────────┴──────────┐
    ↓                     ↓
  快照读                 当前读
 (普通SELECT)          (FOR UPDATE/
                        UPDATE/DELETE)
    ↓                     ↓
  MVCC                 临键锁
 (ReadView不变)       (Next-Key Lock)
    ↓                     ↓
 天然防幻读            锁间隙防INSERT
```

### 5.4 RR 下幻读的"漏网之鱼"

> **面试反杀要点**：RR 并不能 100% 解决幻读！

```sql
-- 场景：在一个事务中先快照读，再当前读
BEGIN;
-- 第一步：快照读（没有幻读）
SELECT * FROM user WHERE id > 10;  -- 返回 3 条

-- 事务B 插入了一条 id=15，并提交
-- （可以插入，因为第一步是快照读没有加锁）

-- 第二步：当前读（可能有幻读！）
UPDATE user SET name = 'new' WHERE id > 10;
-- 虽然这一步会加上临键锁阻止新的插入
-- 但是刚才事务B插入的 id=15 已经被当前读读到了！
-- 结果是第一步快照读看到3条，第二步当前读影响了4条
```

所以准确地说：**RR 通过 MVCC + 临键锁，在绝大多数场景下避免了幻读，但并不能 100% 消除。**

## 六、锁的类型——InnoDB 的锁体系

### 6.1 按粒度分类

```
                    锁的粒度
                       │
        ┌──────────────┼──────────────┐
        ↓              ↓              ↓
      表锁            页锁           行锁
   (Table Lock)   (Page Lock)   (Row Lock)
     ┌──┴──┐        (已淘汰)       ┌──┴──┐
     ↓     ↓                       ↓     ↓
  读锁   写锁                   共享锁  排他锁
  (S)    (X)                     (S)    (X)
```

| 锁粒度    | 适用引擎                 | 特点           | 并发度 |
| ------ | -------------------- | ------------ | --- |
| **表锁** | MyISAM 常用，InnoDB 也支持 | 锁整张表，开销小，加锁快 | 低   |
| **页锁** | BDB（已淘汰）             | 锁一个数据页       | 中   |
| **行锁** | InnoDB               | 只锁需要的行，开销大   | 高   |
|        |                      |              |     |
|        |                      |              |     |

### 6.2 行锁的四种类型

```
              行锁的类型
                  │
    ┌──────┬──────┼──────┬──────────┐
    ↓      ↓      ↓      ↓          ↓
  记录锁  间隙锁  临键锁  插入意向锁
```

#### 6.2.1 记录锁（Record Lock）

**锁住索引中的一条记录**，阻止其他事务对该记录的 UPDATE 和 DELETE。

```sql
-- 对 id=1 的记录加记录锁
SELECT * FROM user WHERE id = 1 FOR UPDATE;
```

```
索引树（简化）：
    [id=1] ← 锁住了这一条
    [id=3]
    [id=5]
```

> 就像你在自习室占了一个座位，别人不能再坐这个位置。

#### 6.2.2 间隙锁（Gap Lock）

**锁住索引记录之间的间隙**，阻止其他事务在这个间隙中 INSERT。**只在 RR 级别下生效。**

```sql
-- 假设表中 id 有 1, 3, 5, 7, 10
-- 对 id>5 AND id<10 范围加间隙锁
SELECT * FROM user WHERE id > 5 AND id < 10 FOR UPDATE;
```

```
索引树：                              锁住的范围：
    [id=1]                              ← 间隙1: (1,3)
    [id=3]                              ← 间隙2: (3,5)
    [id=5] ──────┐
    [id=7]       ├── 间隙锁：锁住 (5,7) 和 (7,10) 这两个间隙
    [id=10] ─────┘
```

> **间隙锁锁的是间隙，不是记录**。这意味着它锁住的是"两个记录之间不存在但可能被插入的空间"。

**间隙锁的特点**：
- **不互斥**：两个事务可以同时在同一个间隙上持有间隙锁（因为间隙锁的目的是阻止 INSERT，不是阻止 SELECT）
- **防 INSERT**：任何想向该间隙插入数据的操作都会被阻塞
- **只在 RR 级别生效**：RC 下没有间隙锁

#### 6.2.3 临键锁（Next-Key Lock）——默认行锁

**临键锁 = 记录锁 + 间隙锁**，锁住一条记录和它前面的间隙。

> **生动比喻**：临键锁 = 锁住自身（记录锁）+ 锁住你和左边邻居之间的空隙（间隙锁）
> 就像你在电影院里占座：不仅占了自己的座位，还把左边和你座位之间的过道封锁了，别人没法在旁边加塞一把椅子。

```
假设 id 值：1, 5, 10, 15

Next-Key Lock 锁住的范围：
    (-∞, 1]
    (1, 5]
    (5, 10]
    (10, 15]
    (15, +∞)  ← 最后一个叫"上界伪记录"

解释：
  (-∞, 1]  表示：( -∞ 到 1 的间隙) + (记录 1 本身)
  (1, 5]   表示：( 1 到 5 的间隙)   + (记录 5 本身)
```

> **临键锁是 InnoDB 在 RR 级别下默认的行锁算法**，它通过"左开右闭"的区间锁设计，同时防止了不可重复读（记录锁部分）和幻读（间隙锁部分）。

**何时会退化为记录锁或间隙锁？**

| 情况                      | 退化结果       | 原因                   |
| ----------------------- | ---------- | -------------------- |
| 查询命中**唯一索引**的**精确等值**匹配 | 退化为**记录锁** | 唯一索引保证不会新增重复值，不需要锁间隙 |
| 查询未命中任何记录（等值查询）         | 退化为**间隙锁** | 没有记录可锁，只锁间隙防 INSERT  |
| 范围查询                    | 保持**临键锁**  | 范围查询需要锁定区间防止新数据插入    |
|                         |            |                      |

```sql
-- 场景1：等值查询命中唯一索引 → 记录锁
SELECT * FROM user WHERE id = 5 FOR UPDATE;  -- id 是主键
-- 退化：只锁 id=5 这条记录

-- 场景2：等值查询未命中 → 间隙锁
SELECT * FROM user WHERE id = 6 FOR UPDATE;  -- id=6 不存在
-- 退化：锁住 (5, 10) 这个间隙

-- 场景3：范围查询 → 临键锁
SELECT * FROM user WHERE id > 5 FOR UPDATE;
-- 不退化为间隙锁：锁住 (5, 10] 这个区间
```

#### 6.2.4 插入意向锁（Insert Intention Lock）

**特殊的间隙锁**，在 INSERT 操作之前自动加在目标间隙上。多个事务可以在同一个间隙上持有插入意向锁（它们之间不互斥），但插入意向锁和间隙锁/临键锁互斥。

```sql
-- 事务A
BEGIN;
SELECT * FROM user WHERE id > 5 FOR UPDATE;  -- 加了临键锁，包含间隙 (5,10)

-- 事务B（被阻塞）
INSERT INTO user(id, name) VALUES (7, '测试');
-- 事务B 会先在间隙 (5,10) 上生成一个插入意向锁
-- 但间隙 (5,10) 已被事务A的临键锁锁定
-- 所以事务B被阻塞，等待事务A释放
```

> 插入意向锁就像一个"排队号"——你想插入但被别人的间隙锁挡在外面了，于是拿了个号等着。同一间隙上可以排多个号（多个事务都想插入不同位置），但只要有间隙锁在，你们都进不去。

### 6.3 意向锁——表级的"告示牌"

> **"意向锁是个告示牌——有人在这个表里占了行锁，你来加表锁先看一眼告示牌。"**

在没有意向锁的世界里，如果要给一张表加表锁，数据库必须逐行检查有没有行锁——**O(n) 的开销**。有了意向锁，只需要检查表上有没有意向锁——**O(1)**。

| 意向锁 | 全称 | 含义 | SQL触发 |
|--------|------|------|---------|
| **IS** | Intention Shared Lock | 事务打算在某些行上加**共享锁** | `SELECT ... FOR SHARE` |
| **IX** | Intention Exclusive Lock | 事务打算在某些行上加**排他锁** | `SELECT ... FOR UPDATE`、`INSERT`、`UPDATE`、`DELETE` |

```
表 user：
┌────────────────────────────┐
│  表级：IX（意向排他锁）     │  ← 告示牌："表里有行被X锁锁住了"
│  ┌────┬──────┬─────┐      │
│  │ id │ name │ age │      │
│  ├────┼──────┼─────┤      │
│  │ 1  │ 张三 │ 25  │      │
│  │ 2  │ 李四 │ 30  │ ← X锁│
│  │ 3  │ 王五 │ 35  │      │
│  └────┴──────┴─────┘      │
└────────────────────────────┘

当事务想在 id=2 上加 X 锁时：
  1. 先在表上加 IX 锁（告示牌）
  2. 再在行 id=2 上加 X 锁

当事务想对整个表加 X 锁时：
  1. 看到表上的 IX 告示牌 → 有人锁了行！
  2. 放弃加表锁（被阻塞）
  3. 不需要逐行检查，省时省力
```

## 七、锁的兼容性

### 7.1 兼容矩阵

| 请求\已有 | IS | IX | S | X |
|-----------|----|----|---|---|
| **IS** | ✓ | ✓ | ✓ | ✗ |
| **IX** | ✓ | ✓ | ✗ | ✗ |
| **S** | ✓ | ✗ | ✓ | ✗ |
| **X** | ✗ | ✗ | ✗ | ✗ |

> **核心记忆**：
> - **X 锁是"终极排斥"**——X 锁和任何锁（包括 X 自己）都互斥。写操作独享。
> - **S 锁是"读友不读敌"**——S 和 S 兼容（读读不冲突），但 S 和 X 不兼容（读写冲突）。
> - **意向锁之间不互斥**——IS 和 IX 可以同时存在（多个事务都在锁行，只是互相通知）。

### 7.2 通俗理解

```
X 锁（排他锁）= 独占一间会议室，锁了之后谁也进不来
S 锁（共享锁）= 阅览室，多个人可以同时进去看书，但进去后不能改造（不能写）
IS/IX（意向锁）= 会议室门口的"使用中"指示灯（告示牌）

灯亮着 → 有人在用 → 想锁整个楼层的人一看灯亮就放弃了
灯不亮 → 没人占行 → 可以放心锁整个楼层
```

## 八、死锁

### 8.1 什么是死锁？

两个或多个事务**互相持有对方需要的锁**，都等着对方释放，形成闭环等待，谁也无法继续。

```sql
-- 死锁现场重现
-- 事务A
BEGIN;
UPDATE user SET age = 26 WHERE id = 1;  -- 获得 id=1 的 X 锁
-- 此时暂停一下

-- 事务B
BEGIN;
UPDATE user SET age = 27 WHERE id = 2;  -- 获得 id=2 的 X 锁

-- 事务A 继续
UPDATE user SET age = 28 WHERE id = 2;  -- 想要 id=2 的 X 锁，等待事务B释放 ⏸

-- 事务B 继续
UPDATE user SET age = 29 WHERE id = 1;  -- 想要 id=1 的 X 锁，等待事务A释放 ⏸

-- 💀 死锁！事务A等B，事务B等A
-- MySQL 检测到死锁后，会回滚其中一个事务（通常是持锁最少的事务）
```

```
图解：死锁的循环等待
    持有 id=1 的锁
    想要 id=2 的锁
    事务A ──────────────────→ 事务B
      ↑                        │
      │        持有 id=2 的锁   │
      │        想要 id=1 的锁   │
      └────────────────────────┘
              死锁环
```

### 8.2 死锁的四个必要条件

| 条件 | 解释 | 如何破坏 |
|------|------|----------|
| **互斥** | 资源一次只能被一个事务持有 | 无法破坏（锁的天然属性） |
| **持有并等待** | 事务持有资源的同时等待其他资源 | 一次性申请所有锁 |
| **不可剥夺** | 已获得的锁不能被强制拿走 | 设置锁超时 |
| **循环等待** | 形成事务-资源的环形等待链 | **统一加锁顺序**（最常用） |

### 8.3 如何避免死锁

```sql
-- ❌ 糟糕的做法（不同事务加锁顺序不一致）
-- 事务A：先锁 id=1，再锁 id=2
-- 事务B：先锁 id=2，再锁 id=1 → 可能死锁

-- ✅ 正确的做法（统一加锁顺序）
-- 事务A：先锁 id=1，再锁 id=2
-- 事务B：先锁 id=1，再锁 id=2 → 事务B在第一步就等待，不会死锁
```

**避免死锁的四个实用技巧**：

1. **统一加锁顺序**：所有事务按相同顺序访问表和行
2. **缩短事务**：事务越小，持有锁的时间越短，死锁概率越低
3. **尽量按主键操作**：按主键可以精确定位记录锁，减少间隙锁范围冲突
4. **降低隔离级别**：RC 级别没有间隙锁，死锁概率大幅降低（但要权衡数据一致性）

### 8.4 如何排查死锁

```sql
-- 1. 查看 InnoDB 状态（包含最近一次死锁信息）
SHOW ENGINE INNODB STATUS\G

-- 在输出中搜索 "LATEST DETECTED DEADLOCK" 部分：
-- LATEST DETECTED DEADLOCK
-- ------------------------
-- *** (1) TRANSACTION:
-- Transaction 12345
-- 该事务持有的锁
-- 该事务等待的锁
-- *** (2) TRANSACTION:
-- ...

-- 2. 查看当前锁等待
SELECT * FROM information_schema.INNODB_LOCKS;        -- MySQL 5.x
SELECT * FROM performance_schema.data_locks;           -- MySQL 8.0+

-- 3. 查看锁等待关系
SELECT * FROM information_schema.INNODB_LOCK_WAITS;    -- MySQL 5.x
SELECT * FROM performance_schema.data_lock_waits;      -- MySQL 8.0+

-- 4. 查看当前事务
SELECT * FROM information_schema.INNODB_TRX;

-- 5. 设置锁等待超时（默认 50 秒）
SET innodb_lock_wait_timeout = 10;  -- 设置为 10 秒
-- 超时后自动回滚，防止无限等待
```

> **InnoDB 的死锁检测是自动的**——它会在构建等待图时发现环，然后选择回滚代价最小的事务（`UNDO` 日志量最少的事务）自动回滚。默认开启（`innodb_deadlock_detect = ON`）。

## 十、面试满分话术

### 话术 1：介绍 MVCC

> "MVCC 是多版本并发控制，InnoDB 通过三个隐藏列（DB_TRX_ID、DB_ROLL_PTR、DB_ROW_ID）和 undo log 版本链，以及 ReadView 读视图，实现了不加锁的一致性读。每个事务看到的都是属于自己的历史快照，就像在看 Git 的某个 commit——不同的事务可以基于不同的版本工作，互不干扰。ReadView 通过比较事务 ID 的大小和活跃事务列表，来判断版本链上的哪一个版本对当前事务可见。"

### 话术 2：解释 RR 和 RC 的区别

> "RC 和 RR 的核心区别在于 ReadView 的创建时机：RC 是每次 SELECT 都创建新的 ReadView，所以能看到其他事务已提交的修改，产生了不可重复读；RR 是事务开始时创建一个 ReadView，整个事务期间都用这一份，所以实现了可重复读。打个比方，RC 是一边走一边拍新照片，世界一直在变；RR 是进门时拍一张照片，整个事务期间只看这张照片，世界是凝固的。另外 RR 还引入了间隙锁和临键锁来解决当前读下的幻读问题。"

### 话术 3：解释临键锁

> "临键锁是 InnoDB 在 RR 级别下默认的行锁算法，它是记录锁和间隙锁的组合——锁住一条记录本身，同时锁住它左边那段空隙。形象地说就是'锁住自身加左边邻居'。这样设计的原因是：记录锁防止别人修改这条记录，间隙锁防止别人在这个范围插入新记录，从而同时解决了不可重复读和幻读。唯一索引的等值查询会退化为记录锁，等值查询未命中会退化为间隙锁，范围查询则保持临键锁。"

### 话术 4：解释意向锁

> "意向锁是表级锁，但它不阻塞行级操作。它的作用就像在表上立了一块告示牌：'此表里有人在用行锁'。当有人想对整个表加表锁时，看一眼告示牌就行，不需要逐行检查——把 O(n) 降到了 O(1)。IS 表示某事务计划加行级共享锁，IX 表示计划加行级排他锁。意向锁之间互相兼容，但和表级的 S/X 锁有兼容规则。"

### 话术 5：解释死锁

> "死锁是两个事务互相持有对方需要的锁，形成循环等待。解决思路有两个方向：预防和检测。预防上，核心就是'统一加锁顺序'——所有事务都按主键递增的顺序访问数据，打破循环等待的条件。另外缩短事务时间、尽量按主键操作也能降低概率。检测上，MySQL InnoDB 有自动的死锁检测机制，会回滚代价较小的事务，我们可以通过 SHOW ENGINE INNODB STATUS 来查看最近一次死锁的详细信息。"
