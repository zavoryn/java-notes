# 微服务完整篇

> 面试定位：Spring Cloud + 微服务架构是 Java 后端面试必考模块，涵盖注册中心、配置中心、网关、远程调用、限流熔断、分布式事务、链路追踪等核心组件，以及 CAP/BASE 等分布式理论基础。

## 一、单体架构 → 微服务架构演进

### 1.1 单体架构的痛

单体架构 = 所有功能模块打包在一个 WAR/JAR 中，部署在一个进程里。

```text
┌──────────────────────────────────────────┐
│              单体应用 WAR                  │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────────┐ │
│  │ 订单  │ │ 用户  │ │ 商品  │ │  支付     │ │
│  └──────┘ └──────┘ └──────┘ └──────────┘ │
│           同一个进程，同一个数据库            │
└──────────────────────────────────────────┘
```

**痛点：**

| 痛点 | 说明 |
|------|------|
| 部署耦合 | 改一行代码 → 整个应用重新部署，部署窗口 2 小时 |
| 扩展困难 | 订单模块高负载，只能复制整个应用，无法按模块弹性伸缩 |
| 技术锁定 | 一开始选了 Spring MVC，后面想用 Go 写某个服务完全不可能 |
| 故障扩散 | 支付模块 OOM → 整个应用挂掉，连注册登录都用不了 |
| 团队协作 | 100 人改同一个代码仓库，Merge 冲突天天上演 |

### 1.2 微服务怎么解决问题

**比喻：从"大超市"拆成"专卖店"。**

> 一个大超市（单体），卖生鲜、家电、服装全挤在一起。收银台排队（线程池满了），整个超市瘫痪。后来拆成生鲜店、家电店、服装店一个个专卖店（微服务），独立经营、独立补货（独立部署）、独立排队（独立线程池）。生鲜店排长队，家电店照样逛。

```text
┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
│ 订单服务 │  │ 用户服务 │  │ 商品服务 │  │ 支付服务 │
└───┬────┘  └───┬────┘  └───┬────┘  └───┬────┘
    │           │           │           │
    └───────────┴─────┬─────┴───────────┘
                      │
              ┌───────┴───────┐
              │   注册中心      │
              │  (Nacos)       │
              └───────────────┘
```

**微服务核心优势：**

| 优势 | 说明 |
|------|------|
| **独立部署** | 订单服务改一行代码，只部署订单服务，1 分钟上线 |
| **独立扩展** | 秒杀时订单服务 100 个实例，用户服务只要 3 个 |
| **技术异构** | 订单用 Java，推荐算法用 Python，搜索引擎用 Go |
| **故障隔离** | 支付服务挂了，用户还能浏览商品、加购物车 |
| **团队自治** | 订单团队 6 个人，有自己的代码仓库、自己的迭代节奏 |

> **面试要点：不要把微服务吹上天。面试官最想听的是你清楚"微服务带来的新问题"——分布式事务、服务发现、配置管理、链路追踪、运维复杂度。这才说明你真的做过。**

## 三、注册中心 Nacos

### 3.1 注册中心是干什么的？

**比喻："黄页电话簿"。**

> 以前查饭店电话要翻黄页电话簿。饭店（服务提供者）把自己的电话（IP+端口）登记到黄页（注册中心），你（服务消费者）要订餐时翻黄页查电话，然后打过去。饭店倒闭了（服务下线），黄页把它划掉（剔除）。新饭店开业（新实例上线），黄页加上去。你不需要记住每家饭店的电话，翻黄页就行。

```text
服务提供者（订单服务启动）
    │
    ├─→ 向 Nacos 注册：我是订单服务，地址 192.168.1.10:8080
    │
    └─→ 定时发送心跳（5秒一次）：我还活着！我还能接客！

服务消费者（用户服务）
    │
    ├─→ 问 Nacos：订单服务的地址是什么？
    │
    └─→ Nacos 返回：[192.168.1.10:8080, 192.168.1.11:8080, ...]
```

### 3.2 Nacos 核心功能

**1. 服务注册与发现**

```yaml
# 每个微服务启动时配置
spring:
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848    # Nacos 地址
        service: order-service          # 服务名
```

服务提供者启动 → 向 Nacos 注册自己 → 服务消费者从 Nacos 拉取服务列表 → 通过服务名调用。

**2. 健康检查 —— 心跳机制**

- 服务实例每 **5 秒** 向 Nacos 发送一次心跳
- Nacos **15 秒** 没收到心跳 → 标记为不健康
- Nacos **30 秒** 没收到心跳 → 从服务列表中剔除

```
时间线:  0s    5s   10s   15s   20s   25s   30s
        ─♥────♥────♥────────────✗────────────✗
        注册   心跳  心跳   不健康标记       剔除
```

**3. CP 模式 vs AP 模式 —— 这是高频面试题**

Nacos 支持两种一致性协议切换：

| 模式 | 协议 | CAP 满足 | 适用场景 | 实例类型 |
|------|------|----------|----------|----------|
| **AP** | Distro 协议 | AP | 一般服务发现 | 临时实例（默认） |
| **CP** | Raft 协议 | CP | 关键元数据 | 持久实例 |

**比喻：**

> **AP 模式（临时实例）**：像 KTV 包间预订。客人（临时实例）打电话订包间，超时没来自动取消。KTV 即使内部系统有点不一致（这个前台说有空房，那个前台说没空房），但不能把客人晾外面不管（优先可用性）。
>
> **CP 模式（持久实例）**：像房产证登记。房子（持久实例）登记后永久有效，即使房主不在家。所有房管局的数据必须严格一致（一致性优先），你宁愿说"系统维护中请稍等"也不能出现两个人持有同一套房的情况。

```java
// 临时实例（默认，AP 模式）—— 服务停止后自动剔除
spring.cloud.nacos.discovery.ephemeral = true   // 默认值

// 持久实例（CP 模式）—— 即使服务挂了也保留记录
spring.cloud.nacos.discovery.ephemeral = false
```

> **面试要点：为什么默认用 AP？** 因为微服务场景下，宁可返回一个可能已经不健康的实例（客户端重试解决），也不能因为 Nacos 集群内部一致性同步而卡住整个服务发现（注册中心挂了所有服务都不可用）。

### 3.3 保护阈值

Nacos 有一个"保护阈值"机制（默认 0）：

```text
设置保护阈值 = 0.8（80%）
   └─→ 含义：即使 80% 的实例都是不健康的，也返回所有实例（包括不健康的）
       └─→ 目的：防止因注册中心自身网络抖动导致大量实例被误剔除
           └─→ 宁可让你调到一个可能不健康的实例，也不让你无实例可调
```

## 四、网关 Gateway

### 4.1 网关是干什么的？

**比喻："写字楼前台/门禁"。**

> 一个大写字楼，有 100 个公司。如果没有前台，每个访客要自己记住 100 个公司在几楼几号——有的在 15 楼 1503，有的在 23 楼 2308，找起来要命。有了前台（网关），你只需要去一楼前台："我找订单服务的张三"，前台查表告诉你电梯卡刷 15 楼，或者直接帮你刷卡。"前台还有安检功能——可疑的人拦住，访客登记（鉴权），高峰期限流放行。"

**网关的核心职责：**

```text
请求 → Gateway
         │
         ├─ 1. 鉴权：token 校验，没登录直接返回 401
         ├─ 2. 路由：/order/** → 订单服务，/user/** → 用户服务
         ├─ 3. 限流：这个用户 1 秒请求了 1000 次，拦下
         ├─ 4. 跨域：统一处理 CORS，不用每个服务自己配
         ├─ 5. 日志：记录每个请求的入参和响应时间
         ├─ 6. 负载均衡：订单服务有 3 个实例，轮询分发
         │
         └─→ 转发到对应微服务
```

### 4.2 Gateway 路由配置

```yaml
spring:
  cloud:
    gateway:
      routes:
        # 路由规则 1：订单服务
        - id: order-service
          uri: lb://order-service          # lb = load balance，走负载均衡
          predicates:
            - Path=/order/**               # 路径以 /order/ 开头的请求
          filters:
            - StripPrefix=1                # 去掉第一层路径 /order

        # 路由规则 2：用户服务
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/user/**
          filters:
            - StripPrefix=1
```

**断言（Predicate）是"匹配规则"，过滤器（Filter）是"附加操作"。**

> 比喻：Predicate 像安检员的判断——"包里有液体"（路径以 /order 开头），Filter 像具体的处理——"没收液体"（StripPrefix 去掉一层路径）或"登记身份证"（AddRequestHeader 加请求头）。

### 4.3 全局过滤器 —— 鉴权实战

```java
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        String token = request.getHeaders().getFirst("Authorization");

        // 1. 如果有 token，解析并传给下游服务
        if (StringUtils.hasText(token)) {
            // 解析 token 得到 userId
            String userId = JwtUtil.parseToken(token);
            // 通过请求头传给下游服务
            ServerHttpRequest newRequest = request.mutate()
                .header("X-User-Id", userId)
                .build();
            return chain.filter(exchange.mutate().request(newRequest).build());
        }

        // 2. 没 token，直接拦截
        ServerHttpResponse response = exchange.getResponse();
        response.setStatusCode(HttpStatus.UNAUTHORIZED);
        return response.setComplete();
    }

    @Override
    public int getOrder() {
        return -1;  // 过滤器顺序，越小越先执行
    }
}
```

### 4.4 Gateway vs Zuul

| 对比维度 | Spring Cloud Gateway | Netflix Zuul |
|----------|---------------------|-------------|
| 底层实现 | **WebFlux + Netty**（响应式、非阻塞） | Servlet 2.5（阻塞 IO） |
| 性能 | 高吞吐，适合高并发 | 线程模型受限 |
| 维护状态 | Spring 官方维护，活跃更新 | Netflix 停止维护 |
| Spring Boot 兼容 | 2.x / 3.x 完美支持 | 需要额外适配 |

> **记忆点：Gateway = Netty + 响应式，Zuul = Servlet + 阻塞。现在基本都用 Gateway。**

## 五、远程调用 OpenFeign

### 5.1 什么是 Feign？

**Feign = 声明式 HTTP 客户端。** 你写一个接口 + 注解，Feign 帮你生成实现类，发 HTTP 请求。

**比喻："外卖 App"。**

> 你想吃宫保鸡丁（调用订单服务），不用自己跑到餐馆（不用自己拼接 URL、处理 HTTP 连接、解析响应），打开外卖 App（Feign 接口），点一下"下单"（调用 `createOrder` 方法），外卖 App 自动帮你下单、付款、配送（Feign 帮你发 HTTP 请求、序列化参数、反序列化响应）。

### 5.2 代码对比：传统方式 vs Feign

```java
// ========== 传统方式：手写 HTTP 调用 ==========
public Order createOrder(Order order) {
    String url = "http://192.168.1.10:8080/order/create";  // 硬编码地址
    RestTemplate restTemplate = new RestTemplate();
    HttpHeaders headers = new HttpHeaders();
    headers.setContentType(MediaType.APPLICATION_JSON);
    HttpEntity<Order> request = new HttpEntity<>(order, headers);
    ResponseEntity<Order> response = restTemplate.postForEntity(url, request, Order.class);
    return response.getBody();
}

// ========== Feign 方式：声明式调用 ==========
@FeignClient(name = "order-service")     // 指定服务名，自动从 Nacos 获取地址
public interface OrderFeignClient {

    @PostMapping("/order/create")
    Order createOrder(@RequestBody Order order);
}

// 使用时：
@Autowired
private OrderFeignClient orderFeignClient;   // Feign 自动生成实现类

public void doSomething() {
    Order order = orderFeignClient.createOrder(new Order());  // 像调本地方法一样
}
```

### 5.3 Feign 的负载均衡

Feign 默认集成 **Spring Cloud LoadBalancer**（替代了老版本 Ribbon），自动做负载均衡：

```text
orderFeignClient.createOrder()
    │
    └─→ Feign 拦截请求
        │
        └─→ LB 从 Nacos 拿到 order-service 的实例列表
            [192.168.1.10:8080, 192.168.1.11:8080, 192.168.1.12:8080]
            │
            └─→ 轮询选一个实例发送 HTTP 请求
```

### 5.4 Feign 日志与超时配置

```yaml
# 开启 Feign 日志（开发调试用）
logging:
  level:
    com.example.feign.OrderFeignClient: DEBUG   # 只对这个接口开启

feign:
  client:
    config:
      default:                 # 全局配置
        connectTimeout: 5000   # 连接超时 5 秒
        readTimeout: 10000     # 读取超时 10 秒
      order-service:           # 针对特定服务
        connectTimeout: 2000   # 订单服务单独配置 2 秒
```

### 5.5 Feign 拦截器 —— 传递认证信息

```java
@Component
public class FeignAuthInterceptor implements RequestInterceptor {
    @Override
    public void apply(RequestTemplate template) {
        // 从当前请求上下文获取 token，添加到 Feign 请求头
        String token = RequestContextHolder.getRequestAttributes() != null
            ? ((ServletRequestAttributes) RequestContextHolder.getRequestAttributes())
                .getRequest().getHeader("Authorization")
            : null;
        if (token != null) {
            template.header("Authorization", token);
        }
    }
}
```

> **这解决了什么问题？** 用户请求 → Gateway → 用户服务 → Feign → 订单服务。如果不传 token，订单服务拿不到用户信息，鉴权失败。Feign 拦截器把 token 一路透传下去。

## 六、Sentinel 限流熔断降级

### 6.1 核心概念：限流、熔断、降级的区别

| 机制 | 触发条件 | 行为 | 比喻 |
|------|---------|------|------|
| **限流** | QPS 过高 / 并发线程数过多 | 直接拒绝多余请求 | 餐厅满座，门口挂"客满"牌 |
| **熔断** | 慢调用/异常比例超过阈值 | 主动断开，快速失败 | 保险丝熔断，防止火灾 |
| **降级** | 服务不可用或熔断后 | 返回兜底数据 | 大厨请假，今天只卖预制菜 |

### 6.2 三种机制深入解析

**比喻总纲："餐厅排队"。**

> 一个网红餐厅（微服务），门口有个排队系统（Sentinel）：
>
> —— **限流**：餐厅只有 50 个座位。第 51 个客人来了，服务员说"不好意思客满了，请在外面等"（直接返回限流提示）。这叫"QPS 限流"。
>
> —— **熔断**：后厨（订单服务）开始连续做菜失败（慢调用比例 > 50%），经理意识到后厨出问题了，直接把后厨闸拉了（熔断开启），新来的单不往里面送了，直接告诉客人"今天不做这个菜"（快速失败）。过 5 秒，经理派一个人进去试做一道菜（半开状态），如果做好了，重新开闸（熔断关闭）。
>
> —— **降级**：后厨熔断期间，客人非要吃这个菜，经理说"这道菜今天做不了，给您换成隔壁的预制菜行不行？"（降级处理，返回兜底数据）。

### 6.3 Sentinel 限流规则

```java
// 编程式定义限流规则（也可以在 Sentinel Dashboard 控制台配置）
private void initFlowRules() {
    List<FlowRule> rules = new ArrayList<>();

    FlowRule rule = new FlowRule();
    rule.setResource("createOrder");         // 资源名（接口名）
    rule.setGrade(RuleConstant.FLOW_GRADE_QPS);  // 限流维度：QPS
    rule.setCount(100);                      // 阈值：每秒 100 个请求
    rules.add(rule);

    FlowRule threadRule = new FlowRule();
    threadRule.setResource("createOrder");
    threadRule.setGrade(RuleConstant.FLOW_GRADE_THREAD);  // 限流维度：并发线程数
    threadRule.setCount(20);                 // 阈值：同时处理 20 个请求
    rules.add(threadRule);

    FlowRuleManager.loadRules(rules);
}
```

**QPS 限流 vs 并发线程数限流：**

| 维度 | 含义 | 适用场景 | 例子 |
|------|------|----------|------|
| QPS | 每秒请求数 | 防止流量突增 | 秒杀接口，每秒最多 1000 请求 |
| 并发线程数 | 同时处理的请求数 | 防止慢接口拖死线程池 | 导出报表接口同时最多 5 个 |

### 6.4 Sentinel 熔断规则

```java
private void initDegradeRules() {
    List<DegradeRule> rules = new ArrayList<>();

    // 策略一：慢调用比例
    DegradeRule slowRule = new DegradeRule("createOrder")
        .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
        .setCount(500)                       // 慢调用阈值：500ms
        .setSlowRatioThreshold(0.5)          // 慢调用比例 > 50% 触发熔断
        .setMinRequestAmount(10)             // 统计窗口内至少 10 个请求
        .setStatIntervalMs(10000)            // 统计窗口 10 秒
        .setTimeWindow(5);                   // 熔断 5 秒后进入半开状态

    // 策略二：异常比例
    DegradeRule errorRule = new DegradeRule("createOrder")
        .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
        .setCount(0.3)                       // 异常比例 > 30% 触发熔断
        .setMinRequestAmount(10)
        .setStatIntervalMs(10000)
        .setTimeWindow(5);

    // 策略三：异常数
    DegradeRule errorCountRule = new DegradeRule("createOrder")
        .setGrade(CircuitBreakerStrategy.ERROR_COUNT.getType())
        .setCount(50)                        // 1 分钟内异常数 > 50 触发熔断
        .setMinRequestAmount(10)
        .setStatIntervalMs(60000)
        .setTimeWindow(5);

    DegradeRuleManager.loadRules(rules);
}
```

**熔断状态机：**

```text
                             熔断时间到期
     CLOSED（关闭/正常）  ──────────────────→  HALF_OPEN（半开）
          │                                        │
          │ 慢调用/异常超阈值                    探测成功 │  探测失败
          │                                        │
          ↓                                        ↓
     OPEN（打开/熔断）                ←──────────────────┘
     快速失败
```

### 6.5 降级处理

```java
@RestController
public class OrderController {

    @SentinelResource(
        value = "createOrder",
        blockHandler = "createOrderBlockHandler",      // 限流/熔断的兜底
        fallback = "createOrderFallback"               // 异常（RuntimeException）的兜底
    )
    @PostMapping("/order/create")
    public Result createOrder(@RequestBody Order order) {
        // 正常业务逻辑
        return Result.success(orderService.createOrder(order));
    }

    // 被 Sentinel 限流或熔断时调用
    public Result createOrderBlockHandler(Order order, BlockException e) {
        return Result.error(429, "当前下单人数过多，请稍后再试");
    }

    // 业务抛出异常时调用
    public Result createOrderFallback(Order order, Throwable e) {
        log.error("下单失败", e);
        return Result.error(500, "下单失败，请重试");
    }
}
```

> **blockHandler vs fallback 的区别**：blockHandler 处理 Sentinel 限流/熔断异常，fallback 处理业务异常。面试时说清这一点，说明你真的用过。

## 七、分布式事务 Seata

### 7.1 分布式事务的困境

**比喻："旅行社订票"。**

> 你在一个旅行 App 上订"机票+酒店"套餐：
> - 机票系统扣款成功
> - 酒店系统预订失败（房间被抢走了）
>
> 这时候怎么办？机票已经扣款了，不能让人家到了机场没酒店住。分布式事务要解决的问题就是：**多个服务之间的数据一致性**。

```text
订单服务 createOrder()
    │
    ├─→ 库存服务 reduceStock()    ← 成功：库存 -1
    │
    └─→ 账户服务 reduceBalance()  ← 失败：余额不足
            │
            └─→ 问题：库存已扣，但没收到钱。数据不一致！
```

### 7.2 Seata AT 模式（最常用）

**AT = Automatic Transaction，自动事务。** 核心思想：**改进版两阶段提交**。

```text
┌──────────────────────────────────────────────────────────┐
│                    Seata TC（事务协调者）                    │
│                      全局事务管理                            │
└──────┬───────────────────────┬──────────────────────────┘
       │                       │
  ┌────┴────┐             ┌────┴────┐
  │ 库存 RM   │             │ 账户 RM   │   ← RM = Resource Manager（资源管理器）
  │          │             │          │
  │ ┌──────┐ │             │ ┌──────┐ │
  │ │undo_log│             │ │undo_log│  ← AT 模式的关键：undo_log 表
  │ └──────┘ │             │ └──────┘ │
  └─────────┘             └─────────┘
```

**AT 模式执行流程（一阶段 + 二阶段）：**

```text
一阶段（执行 + 记录回滚日志）：
    ┌──────────────────────────────────────┐
    │ 1. 库存 RM：执行 UPDATE stock SET    │
    │    count = count - 1 WHERE id = 100  │
    │ 2. 同时记录 undo_log：                │
    │    更新前的值 count = 10（用于回滚）    │
    │ 3. 账户 RM：同样操作                   │
    │ 4. 一阶段提交（只提交本地事务）          │
    │ 5. 释放本地锁                         │
    └──────────────────────────────────────┘

二阶段（TC 判断全局结果）：
    全部成功 → 异步删除 undo_log
    任一失败 → 异步执行 undo_log 中的回滚 SQL
```

> **AT 模式 vs 传统 XA 两阶段提交核心区别：** AT 一阶段就提交本地事务、释放本地锁，避免了 XA 长时间锁表的问题。回滚依赖 undo_log 记录。

### 7.3 TCC 模式

**TCC = Try / Confirm / Cancel**，适合对一致性要求极高的场景（如金额扣减）。

| 阶段 | 操作 | 库存服务 | 账户服务 |
|------|------|---------|---------|
| **Try** | 预留资源 | 冻结库存 1 件 | 冻结余额 100 元 |
| **Confirm** | 确认提交 | 真正扣减库存 | 真正扣减余额 |
| **Cancel** | 回滚 | 解冻库存 | 解冻余额 |

```text
        Try（冻结）
           │
    ┌──────┴──────┐
    │  全部成功？   │
    └──────┬──────┘
      是 ╱   ╲ 否
        ╱     ╲
   Confirm   Cancel
   （扣减）    （解冻）
```

**TCC vs AT 对比：**

| 维度 | AT 模式 | TCC 模式 |
|------|---------|----------|
| 侵入性 | 低（加注解 + undo_log 表） | 高（需要实现 Try/Confirm/Cancel 三个方法） |
| 性能 | 好（一阶段就释放锁） | 一般（资源要冻结到二阶段） |
| 一致性 | 最终一致 | 强一致 |
| 适用场景 | 一般分布式事务 | 金额扣减、库存扣减等严格场景 |

### 7.4 Seata AT 模式使用示例

```java
// 入口方法加 @GlobalTransactional
@Service
public class OrderServiceImpl implements OrderService {

    @Autowired
    private StockFeignClient stockFeignClient;

    @Autowired
    private AccountFeignClient accountFeignClient;

    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    @Override
    public void createOrder(Order order) {
        // 1. 创建订单（本地事务）
        orderMapper.insert(order);

        // 2. 扣减库存（远程调用，Seata 自动管理）
        stockFeignClient.reduceStock(order.getProductId(), order.getCount());

        // 3. 扣减余额（远程调用）
        accountFeignClient.reduceBalance(order.getUserId(), order.getTotalAmount());

        // 三个操作：任何一个失败，全局回滚
    }
}
```

**每个服务需要配置：**
```yaml
seata:
  tx-service-group: my_tx_group     # 事务分组
  service:
    vgroup-mapping:
      my_tx_group: default           # 映射到 Seata Server 集群
```

**每个数据库添加 undo_log 表：**
```sql
CREATE TABLE `undo_log` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `branch_id` bigint(20) NOT NULL,
  `xid` varchar(100) NOT NULL,
  `context` varchar(128) NOT NULL,
  `rollback_info` longblob NOT NULL,
  `log_status` int(11) NOT NULL,
  `log_created` datetime NOT NULL,
  `log_modified` datetime NOT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `ux_undo_log` (`xid`,`branch_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;
```

> **面试要点：AT 模式为什么比 XA 好？** XA 事务在第一阶段锁定资源直到第二阶段结束。假设第二阶段网络延迟 10 秒，数据库锁就持有 10 秒。AT 模式第一阶段就提交并释放锁，第二阶段异步回滚即可，吞吐量远高于 XA。

## 八、CAP 定理

### 8.1 定理定义

> **CAP 定理**：在一个分布式系统中，Consistency（一致性）、Availability（可用性）、Partition Tolerance（分区容错性）三者只能同时满足两个。

```text
                C（一致性）
               /\
              /  \
             /    \
            /  不可能同时满足
           /        \
          /          \
   A（可用性）────── P（分区容错性）
```

### 8.2 三个概念

| 字母 | 全称 | 含义 | 通俗解释 |
|------|------|------|----------|
| **C** | Consistency | 一致性 | 所有节点在同一时刻看到相同的数据 |
| **A** | Availability | 可用性 | 每个请求都能得到非错误的响应（不一定是最新数据） |
| **P** | Partition Tolerance | 分区容错性 | 网络分区发生时，系统仍能正常运作 |

**比喻：**

> **C（一致性）**：银行账户余额。你刚转了 100 块，立刻查询，必须显示余额减少了 100。不能出现"转了但查不到"的情况。
>
> **A（可用性）**：微博热搜。服务器再忙也得让你刷到内容，哪怕看到的是 5 分钟前的。
>
> **P（分区容错性）**：微信发消息。你和朋友在不同城市，中间有段网线被挖断了（网络分区），但你发出的消息朋友最终一定能收到——不能因为网络断了就罢工。

### 8.3 为什么 P 必须选

在网络通信中，分区容错是必须面对的客观现实。两个数据中心之间的网络随时可能抖动或断开。**不选 P 意味着你要假设网络永远不会出问题——这在分布式系统中是不可能的。**

> 因此实际的权衡是：**当网络出现分区时，你选择 C 还是 A？**

```text
┌──────────────────────────────────────────────────────┐
│  网络分区发生（两个数据中心之间的网线断了）                   │
│                                                      │
│  数据中心 A  ←──x──→  数据中心 B                       │
│                                                      │
│  选 C：拒绝写请求，保证两边数据一致                         │
│     └─→ "系统维护中"（牺牲可用性）                        │
│                                                      │
│  选 A：允许两边各自写入                                  │
│     └─→ 数据暂时不一致（牺牲一致性）                       │
│     └─→ 网络恢复后再同步 → 最终一致                       │
└──────────────────────────────────────────────────────┘
```

### 8.4 常见系统的 CAP 选择

| 系统 | 核心选择 | 说明 |
|------|----------|------|
| **ZooKeeper** | **CP** | 选主、分布式锁等场景，一致性优先 |
| **Eureka** | **AP** | 注册中心，宁可返回旧数据也不能拒掉请求 |
| **Nacos** | **AP（默认）/ CP（切换）** | 注册发现走 AP，配置管理走 CP |
| **Redis Cluster** | **AP** | 缓存场景，可用性优先 |
| **MySQL 主从** | **AP** | 主从复制有延迟，但不会拒绝读请求 |

## 九、BASE 理论

### 9.1 BASE 是什么

BASE 是 CAP 定理中 **AP（牺牲一致性换可用性）** 的实践指导：

| 字母 | 全称 | 含义 |
|------|------|------|
| **B** | Basically Available | 基本可用 |
| **A** | Soft State | 软状态（允许系统存在中间状态） |
| **E** | Eventually Consistent | 最终一致性 |

### 9.2 三个概念深入

**1. 基本可用（Basically Available）**

系统出现故障时，允许损失部分功能，保证核心可用。

> 例子：双十一淘宝，下单按钮正常，但"查看商品评论"功能暂时关闭。核心功能（下单）可用，次要功能降级。

```text
正常情况：
  首页 → 搜索 → 商品详情 → 下单 → 支付 → 评价

大促期间（基本可用）：
  下单 ✅（核心，必须保证）
  支付 ✅（核心，必须保证）
  评价 ❌（非核心，暂时关闭）
  推荐 ❌（非核心，数据降级到昨天）
```

**2. 软状态（Soft State）**

数据存在中间状态，不需要时刻保持强一致。

> 例子：你在朋友圈发了一条消息，你的朋友可能过 3 秒才看到。这 3 秒里，你的朋友圈和他的朋友圈状态不一致——这就是"软状态"。

**3. 最终一致性（Eventually Consistent）**

不强求实时一致，但保证最终会一致。

> 例子：支付宝转银行卡，提示"转账成功，预计 2 小时内到账"。2 小时内余额不一致，但 2 小时后肯定对得上。

```text
时间线：
t0: 转账 100 元
    └─→ 支付宝显示：已扣 100（你的"软状态"）
    └─→ 银行卡显示：余额不变（银行的"软状态"）
    →
t1（2小时后）:
    └─→ 银行卡显示：余额 +100（"最终一致"）
```

### 9.3 CAP vs BASE vs ACID

| 理论 | 级别 | 核心理念 | 代表系统 |
|------|------|----------|----------|
| **ACID** | 单机数据库 | 强一致，事务隔离 | MySQL InnoDB |
| **CAP** | 分布式理论 | C/A/P 三者只能取二 | 理论指导 |
| **BASE** | 分布式实践 | 放弃强一致，追求最终一致 | NoSQL、微服务 |

> **ACID → BASE 的转变**：就像从一个人管钱（单机 ACID）变成多个人一起管钱（分布式 BASE）。一个人的账本绝对精确（一致），但一旦生病就没人记账了（单点故障）。多个人写允许暂时不一致，但对账的时候（最终一致）能对上。

## 十、分布式配置中心 Nacos Config

### 10.1 为什么需要配置中心

**比喻："统一遥控器"。**

> 一个智能家居有 10 个设备（微服务），每个设备有自己的设置（线程池大小、超时时间、开关）。如果没有配置中心，你要一个一个去改——爬到设备旁边按按钮。有了配置中心（统一遥控器），在手机上点一下，所有设备自动更新设置。这就是"分布式配置中心"。

```text
没有配置中心：
  修改线程池大小 → 逐个改 application.yml → 重启 10 个服务 → 部署 1 小时

有配置中心（Nacos Config）：
  修改线程池大小 → 在 Nacos 控制台改 → 10 个服务自动刷新 → 1 分钟完成
```

### 10.2 Nacos 配置中心使用

**配置优先级（bootstrap.yml）：**
```yaml
spring:
  cloud:
    nacos:
      config:
        server-addr: localhost:8848       # Nacos 地址
        file-extension: yaml              # 配置文件格式
        group: DEFAULT_GROUP
        namespace: dev                    # 命名空间（开发/测试/生产隔离）
        shared-configs:                   # 共享配置（所有服务通用）
          - data-id: common-config.yaml
            group: DEFAULT_GROUP
            refresh: true
```

**Nacos 中的配置文件约定：**
```text
Data ID = ${spring.application.name}-${spring.profiles.active}.${file-extension}

示例：
  order-service-dev.yaml    → 订单服务开发环境配置
  order-service-prod.yaml   → 订单服务生产环境配置
```

### 10.3 动态刷新 @RefreshScope

```java
@RestController
@RefreshScope   // ← 关键注解：配置变更时重建 Bean
public class ConfigController {

    @Value("${order.timeout:5000}")    // :5000 是默认值
    private int timeout;

    @GetMapping("/config/timeout")
    public int getTimeout() {
        return timeout;   // 配置在 Nacos 改了，这里自动刷新
    }
}
```

**@RefreshScope 的原理：**
```text
1. Nacos 配置变更 → 推送通知给客户端
2. @RefreshScope 标记的 Bean 在下次访问时重新创建
3. 重新注入 @Value 的值 → 拿到最新配置
4. 旧 Bean 被 GC 回收

注意：@RefreshScope 的 Bean 是懒加载的（第一次访问才创建）
```

> **面试要点：@RefreshScope vs @ConfigurationProperties。** @Value 必须加 @RefreshScope 才能刷新，而 @ConfigurationProperties 天生支持动态刷新，因为它是通过 setter 注入的。

## 十一、服务链路追踪

### 11.1 为什么需要链路追踪

**比喻："快递物流跟踪"。**

> 你网购了一件商品（发起一个请求），经历了揽件（Gateway）→ 中转站 A（用户服务）→ 中转站 B（订单服务）→ 中转站 C（支付服务）→ 签收。如果商品丢了（请求超时），你需要知道卡在哪个环节。快递单号（TraceId）贯穿全程，扫描枪记录每个节点的时间（Span）。

```text
一次请求的调用链：
  TraceId: abc123
    │
    ├─ SpanId: 1  Gateway      耗时 2ms
    │    │
    │    ├─ SpanId: 1.1  用户服务   耗时 50ms
    │    │    └─ SpanId: 1.1.1  数据库查询  耗时 30ms
    │    │
    │    └─ SpanId: 1.2  订单服务   耗时 200ms
    │         ├─ SpanId: 1.2.1  库存服务  耗时 80ms
    │         └─ SpanId: 1.2.2  支付服务  耗时 1500ms ← 问题在这里！
```

### 11.2 Spring Cloud Sleuth + Zipkin

**Sleuth**：负责生成 TraceId 和 SpanId，记录调用链路。
**Zipkin**：负责收集、存储、展示链路数据。

```xml
<!-- 依赖 -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-sleuth-zipkin</artifactId>
</dependency>
```

```yaml
spring:
  sleuth:
    sampler:
      probability: 1.0          # 采样率：1.0 = 100%（开发环境），生产 0.1
  zipkin:
    base-url: http://localhost:9411  # Zipkin Server 地址
```

**Sleuth 自动注入的日志信息：**
```text
2026-05-05 10:30:15.123 INFO [order-service,abc123,abc123.1.2,true]
                            └─服务名──┘ └─TraceId┘ └──SpanId──┘ └─是否上报─┘
```

### 11.3 SkyWalking

SkyWalking 是更重量级的 APM（应用性能监控）方案：

| 对比 | Sleuth + Zipkin | SkyWalking |
|------|-----------------|------------|
| 实现方式 | 代码侵入（依赖包） | Java Agent 探针（零侵入） |
| 数据收集 | HTTP → Zipkin Server | gRPC → SkyWalking OAP |
| 指标维度 | 调用链 | 调用链 + JVM 指标 + 服务拓扑 + 慢 SQL |
| 存储 | 默认内存 | Elasticsearch / H2 / MySQL |
| 告警 | 需配合其他工具 | 内置告警规则 |

```text
SkyWalking 架构：
  Application（JVM + Agent探针）
       │  gRPC
  ┌────┴─────────────────────────┐
  │  SkyWalking OAP Server        │  ← 接收、分析、聚合数据
  │  （Observability Analysis     │
  │   Platform）                   │
  └────────────┬─────────────────┘
               │
       ┌───────┴───────┐
       │               │
  Elasticsearch    SkyWalking UI
   (数据存储)        (可视化界面)
```

### 11.4 链路追踪的核心概念

| 概念 | 含义 | 比作快递 |
|------|------|----------|
| **TraceId** | 一次完整请求的唯一标识 | 快递单号 |
| **SpanId** | 一次调用的唯一标识 | 每个中转站的扫描记录 |
| **Parent SpanId** | 父调用的 SpanId | 从哪个站点发出的 |
| **Annotation** | 关键时间点（CS, SR, SS, CR） | 到达时间、发出时间 |

```text
Span 的生命周期注解：
  Client Send (CS)      Server Receive (SR)     Server Send (SS)     Client Receive (CR)
  客户端发出请求  ──────→    服务端收到   ──────→    服务端处理完   ──────→  客户端收到响应
      ↓                        ↓                      ↓                    ↓
  ┌─────── 网络传输 ──────┐ ┌──────── 服务处理 ────────┐ ┌──────── 网络传输 ─────────┐
  │      耗时 = SR-CS     │ │   耗时 = SS-SR           │ │      耗时 = CR-SS         │
  └───────────────────────┘ └─────────────────────────┘ └────────────────────────────┘
```

## 十四、面试满分话术

### 话术 1：介绍微服务架构经验（开场高频）

> "我参与的项目从单体 Spring Boot 应用拆分为 Spring Cloud 微服务，使用 Nacos 做注册中心和配置中心，Gateway 做统一网关，Feign 做服务间远程调用，Sentinel 做限流熔断，Seata 做分布式事务，Sleuth + Zipkin 做链路追踪。拆分过程中最大的感受是：微服务不是银弹，它解决了单体扩展难的问题，但也带来了分布式事务、服务发现、配置管理、运维部署等新问题，需要配套完善的 DevOps 体系。"

### 话术 2：CAP 定理（理论基础高频）

> "CAP 定理说的是分布式系统中一致性、可用性、分区容错性三者只能同时满足两个。实际上 P 是必须选的，因为网络分区是无法避免的客观现实。所以我们实际是在 C 和 A 之间做权衡。比如 Nacos 作为注册中心默认走 AP 模式——宁可返回一个可能不健康的实例，也不能因为内部一致性同步导致整个服务发现不可用。而 ZooKeeper 走 CP 模式，适合分布式锁、选主等一致性要求高的场景。"

### 话术 3：Sentinel 限流熔断（实战高频）

> "我们用 Sentinel 做了三层保护：第一层在 Gateway 网关做全局 QPS 限流，防止整体流量过载；第二层在核心服务接口上做并发线程数限流，防止慢接口拖死线程池；第三层对非核心依赖做熔断降级——比如评价服务不可用时，商品详情页返回缓存的评价数据而不是直接报错。熔断规则我们用的是慢调用比例策略，统计窗口内超过 50% 的调用响应时间大于 500 毫秒就触发熔断，5 秒后进入半开探测。"

### 话术 4：分布式事务 Seata（难度加分项）

> "分布式事务我们用 Seata 的 AT 模式，它本质上是对传统两阶段提交的改进。第一阶段就提交本地事务、释放数据库锁，同时记录 undo_log。第二阶段根据 TC 的全局判断——全部成功则异步删除 undo_log，任一失败则根据 undo_log 回滚。相比 XA，AT 模式避免了长时间锁表的性能问题。对于金额扣减这种强一致性场景，我们会改用 TCC 模式，通过 Try-Confirm-Cancel 三阶段来保证。"

## 十五、记忆口诀

### Spring Cloud 组件口诀（顺口溜）

```text
注（Nacos）册中心找服务，配（Config）置动态随时改。
网（Gateway）关拦路验身份，调（Feign）用远程像本地。
哨（Sentinel）兵站岗限流量，海（Seata）神镇场保一致。
追（Sleuth/Zipkin）踪链路查瓶颈，微服架构齐活了。
```

**拆解记忆：**

| 一句话 | 组件 | 核心功能 |
|--------|------|----------|
| 注册中心找服务 | Nacos Discovery | 服务注册与发现 |
| 配置动态随时改 | Nacos Config | 配置管理 + @RefreshScope |
| 网关拦路验身份 | Gateway | 路由 + 鉴权 + 限流 + 跨域 |
| 调用远程像本地 | OpenFeign | 声明式 HTTP 调用 + 负载均衡 |
| 哨兵站岗限流量 | Sentinel | 限流 + 熔断 + 降级 |
| 海神镇场保一致 | Seata | 分布式事务 |
| 追踪链路查瓶颈 | Sleuth/Zipkin | 链路追踪 |

### CAP / BASE 口诀

```text
CAP → C一致 A可用 P分区 → 最多选两个 → P必须选 → 剩下 C 和 A 二选一
BASE → 基本可用 + 软状态 + 最终一致 → AP 的工程实践

记忆链：CAP 是理论 → BASE 是折中方案 → Seata AT 是具体实现
```

## 十六、跨题串联

### 16.1 微服务 ↔ Spring Boot

| 关联点 | 说明 |
|--------|------|
| **自动配置** | Spring Boot 的 `@EnableAutoConfiguration` 是 Spring Cloud 集成的基础，Nacos/Gateway 都依赖它 |
| **启动类** | `@SpringBootApplication` + `@EnableDiscoveryClient`（启用服务发现）|
| **配置文件** | `application.yml` → `bootstrap.yml`（配置中心优先加载）|
| **健康检查** | Spring Boot Actuator 提供 `/actuator/health`，供 Nacos 心跳检测 |

> **面试话术**："Spring Boot 是微服务的快速开发框架，Spring Cloud 在 Boot 基础上提供了服务治理能力。Boot 解决单个微服务怎么快速写，Cloud 解决多个微服务怎么协作。"

### 16.2 微服务 ↔ 消息队列

| 关联点 | 说明 |
|--------|------|
| **异步解耦** | 订单创建后发消息通知库存扣减，替代 Feign 同步调用，提升吞吐 |
| **削峰填谷** | 秒杀请求写入 MQ 队列，后端缓慢消费，配合 Sentinel 限流 |
| **最终一致性** | 用 MQ 实现分布式事务的最终一致性（替代 Seata） |

```text
方案对比：
  Feign 同步调用 → 强一致 → 性能差 → 配合 Seata 使用
  MQ 异步通知  → 最终一致 → 性能好 → 配合本地消息表使用
```

### 16.3 微服务 ↔ Redis

| 关联点 | 说明 |
|--------|------|
| **分布式 Session** | Gateway 鉴权后，token → userId 映射存 Redis，微服务共享登录态 |
| **分布式锁** | 秒杀扣库存用 Redis 分布式锁（Redisson），配合 Sentinel 限流 |
| **缓存预热** | 商品详情页数据缓存到 Redis，减少对商品服务的调用 |
| **计数/限流** | Sentinel 可以结合 Redis 做分布式限流（单机限流不够精确） |

> **面试话术**："微服务是架构骨架，Redis 是它最常用的基础设施之一。分布式 Session、分布式锁、热点缓存这三大场景，在微服务中几乎必然用到 Redis。"

### 16.4 微服务 ↔ 数据库

| 关联点 | 说明 |
|--------|------|
| **分库分表** | 每个微服务有自己的数据库，订单服务用订单库，用户服务用用户库 |
| **读写分离** | 每个微服务数据库可以做主从复制，读多写少的服务查从库 |
| **跨库事务** | 分库后单机事务失效 → 需要 Seata 分布式事务 |

```text
数据库拆分演化：
  单体：一个 database，200 张表
    ↓
  微服务拆分：每个服务一个 database
    ├─ order_db（订单服务）
    ├─ user_db（用户服务）
    ├─ product_db（商品服务）
    └─ account_db（账户服务）

  拆分后的问题：
    1. 不能 JOIN 跨库表 → 通过 Feign 调用 + 代码拼接
    2. 分布式事务 → Seata AT / TCC / MQ 最终一致
    3. 数据一致性 → Canal + MQ 同步冗余数据
```

## 十七、微服务不是银弹 —— 引入微服务的新问题

| 问题 | 单体不存在 | 微服务引入 | 解决手段 |
|------|-----------|-----------|----------|
| 服务发现 | 无需 | 服务多了找不到彼此 | Nacos 注册中心 |
| 配置管理 | 一个文件 | 几十个服务配置文件分散 | Nacos 配置中心 |
| 远程调用 | 本地方法调用 | 网络调用、序列化、超时 | Feign + Ribbon |
| 分布式事务 | 本地事务搞定 | 跨服务数据不一致 | Seata / MQ 最终一致性 |
| 链路追踪 | 单线程调用栈 | 跨服务看不出调用链 | Sleuth / SkyWalking |
| 运维部署 | 一个 WAR 部署 | 几十个服务要 CI/CD | Docker + K8s + Jenkins |
| 调试 | 本地 debug | 跨服务难复现 | 日志 + 链路追踪 + 统一日志平台 |

> **面试加分句**："微服务解决的每一个问题，都引入了新的复杂度。没有最好的架构，只有最合适的架构。小团队小项目用单体 + 模块化往往效率更高。"
