# JVM 调优实战

> [!summary] 我的速记
> **调优三部曲：查找问题（jps/jstat/jmap/jstack/arthas）→ 分析根因（OOM/CPU飙升/频繁GC）→ 调整参数（堆大小/GC选择/GC调优）。**
> **常见参数口诀：** `-Xms` 初始堆、`-Xmx` 最大堆、`-Xmn` 新生代、`-XX:+UseG1GC` 选 GC、`-XX:+PrintGCDetails` 打日志。
> **OOM 排查：** `-XX:+HeapDumpOnOutOfMemoryError` → jmap dump → MAT 分析 → 找泄漏对象 → 查 GC Roots 引用链。

---

## 一、核心 JVM 参数速查

### 1.1 堆内存参数

| 参数 | 含义 | 示例 |
|------|------|------|
| `-Xms` | 初始堆大小 | `-Xms2g` |
| `-Xmx` | 最大堆大小 | `-Xmx4g` |
| `-Xmn` | 新生代大小 | `-Xmn1g` |
| `-XX:NewRatio` | 老年代/新生代比例 | `-XX:NewRatio=2`（老:新=2:1） |
| `-XX:SurvivorRatio` | Eden/Survivor 比例 | `-XX:SurvivorRatio=8`（Eden:S0=8:1） |
| `-XX:MaxTenuringThreshold` | 晋升老年代的年龄阈值 | `-XX:MaxTenuringThreshold=15` |

> 🧠 **最佳实践：** `-Xms` 和 `-Xmx` 设为相同值——避免堆动态伸缩带来的 GC 压力。生产环境堆一般不超过 75% 物理内存。

### 1.2 元空间参数

| 参数 | 含义 | 示例 |
|------|------|------|
| `-XX:MetaspaceSize` | 触发 Metaspace GC 的初始阈值 | `-XX:MetaspaceSize=256m` |
| `-XX:MaxMetaspaceSize` | 最大元空间 | `-XX:MaxMetaspaceSize=512m` |

### 1.3 GC 选择参数

| 参数 | 含义 |
|------|------|
| `-XX:+UseSerialGC` | 使用 Serial + Serial Old |
| `-XX:+UseParallelGC` | 使用 Parallel Scavenge + Parallel Old（吞吐量优先） |
| `-XX:+UseConcMarkSweepGC` | 使用 ParNew + CMS（JDK9 废弃） |
| `-XX:+UseG1GC` | 使用 G1（JDK9 起默认） |
| `-XX:+UseZGC` | 使用 ZGC（JDK11+） |

**GC 选型速查：**

| 堆大小 | GC 推荐 |
|--------|---------|
| < 4GB | Parallel（吞吐量） |
| 4~8GB | CMS（低停顿）/ G1 |
| > 8GB | G1 |
| > 32GB | G1 / ZGC |

### 1.4 GC 日志参数

```bash
# JDK 8
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/path/to/gc.log

# JDK 9+
-Xlog:gc*=info:file=/path/to/gc.log:time,uptime:filecount=10,filesize=100M
```

### 1.5 OOM 排查参数

| 参数 | 含义 |
|------|------|
| `-XX:+HeapDumpOnOutOfMemoryError` | OOM 时自动 dump 堆快照 |
| `-XX:HeapDumpPath=/path/to/dump.hprof` | dump 文件路径 |
| `-XX:OnOutOfMemoryError=script.sh` | OOM 时执行脚本（重启/告警） |

### 1.6 G1 调优参数

| 参数 | 含义 |
|------|------|
| `-XX:MaxGCPauseMillis=200` | 期望的最大 GC 停顿（默认 200ms） |
| `-XX:G1HeapRegionSize=4m` | Region 大小（1~32M，默认堆/2048） |
| `-XX:InitiatingHeapOccupancyPercent=45` | 触发 Mixed GC 的老年代占比阈值 |

## 二、排查工具大全

### 2.1 JDK 自带命令行

| 命令 | 作用 | 常用场景 |
|------|------|----------|
| `jps -l` | 列出所有 Java 进程 PID | 找到目标进程 |
| `jstat -gc <pid> 1000 10` | 每秒输出 GC 统计，共 10 次 | 实时监控 GC |
| `jmap -heap <pid>` | 堆内存概况 | 查看各区使用情况 |
| `jmap -histo:live <pid>` | 堆中对象统计（触发 FGC） | 快速定位大对象 |
| `jmap -dump:live,file=dump.hprof <pid>` | dump 堆快照 | 离线分析 |
| `jstack <pid>` | 线程栈信息 | 排查死锁、CPU 飙升 |
| `jinfo <pid>` | 查看/修改 JVM 参数 | 查看运行时配置 |

### 2.2 jstat 输出解读

```bash
$ jstat -gc <pid> 1000
 S0C    S1C    S0U    S1U      EC       EU        OC         OU       MC     MU
  ... Survivor 区    ...  Eden区使用  ... 老年代使用  ... Metaspace使用
     YGC   YGCT    FGC   FGCT     GCT
   新生代GC次数 耗时  FullGC次数 耗时  总GC耗时
```

> 🧠 **关注指标：** `FGC` 频繁增长 → 老年代有问题；`OU` 接近 `OC` → 可能要 OOM；`YGC` 太频繁 → 新生代太小。

### 2.3 Arthas（阿里开源——线上神器）

| 命令 | 作用 |
|------|------|
| `dashboard` | 实时面板（线程/内存/GC） |
| `thread -b` | 找出当前阻塞其他线程的线程 |
| `thread -n 3` | 找出 CPU 使用最高的 3 个线程 |
| `jad com.example.MyClass` | 反编译类，确认线上代码版本 |
| `watch com.example.Service method '{params,returnObj}'` | 观察方法入参和返回值 |
| `heapdump /tmp/dump.hprof` | 在线 dump（不触发 FGC） |
| `vmtool --action getInstances -c <class> --limit 10` | 查看某个类的实例 |

## 三、调优三步骤

```
① 发现问题 → ② 定位根因 → ③ 调整参数
```

### 3.1 发现问题

| 现象 | 表现 | 工具 |
|------|------|------|
| **CPU 飙升** | top 显示 Java 进程 CPU 100% | `top -Hp <pid>` → 找到高 CPU 线程 → `jstack` 转十六进制定位代码 |
| **频繁 YGC** | 响应 RT 抖动，GC 日志刷屏 | `jstat -gc` 观察 YGC 频率 |
| **频繁 Full GC** | STW 频繁，服务假死 | GC 日志 + `jstat` 看 FGC |
| **OOM** | 进程挂掉，日志有异常 | dump 文件分析 |
| **内存泄漏** | 老年代使用率持续增长不降 | 多次 dump 对比 |

### 3.2 CPU 100% 排查流程

```bash
# 1. 找到 Java 进程
top            # 或 jps -l

# 2. 找到进程内的高 CPU 线程
top -Hp <pid>   # 记录线程 ID（十进制），如 12345

# 3. 线程 ID 转十六进制
printf "%x\n" 12345   # → 0x3039

# 4. 从线程栈定位代码
jstack <pid> | grep -A 20 0x3039
# → 找到对应线程的执行栈，定位代码行
```

### 3.3 OOM 排查流程

```bash
# 1. 确保 OOM 时 dump（启动参数里加）
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof

# 2. 用 MAT (Memory Analyzer Tool) 打开 dump.hprof

# 3. MAT 分析步骤：
#   Leak Suspects Report → 看最大嫌疑对象
#   Dominator Tree → 按 retained heap 排序
#   Path to GC Roots → 看引用链，找到是谁在持有这个对象
```

> 🧠 **MAT 核心概念：**
> - **Shallow Heap** = 对象自身占的内存
> - **Retained Heap** = 对象自身 + 它引用的所有对象（如果该对象被回收，总计释放的内存）
> - **Dominator Tree** → 按 Retained Heap 排序 → 找到内存大户

### 3.4 频繁 Full GC 排查（调优前先找根因）

| 常见原因 | 现象 | 解决 |
|----------|------|------|
| **老年代太小** | OU 一直很高，FGC 频繁但能回收 | 调大老年代 / 整个堆 |
| **System.gc()** | 代码里显式调了 | 加 `-XX:+DisableExplicitGC` |
| **大对象频繁进老年代** | 大对象 > 新生代 | 调大新生代或 `PretenureSizeThreshold` |
| **元空间不够** | GC 日志显示 Full GC (Metadata GC Threshold) | 调大 `-XX:MaxMetaspaceSize` |
| **内存泄漏** | 老年代持续增长，Full GC 收不回 | MAT 找泄漏对象 |

## 面试满分话术

> **面试官：** 线上 CPU 飙升 100% 或 OOM 怎么排查？

**你的回答：**

> **CPU 100%：** 先用 `top` 找到 Java 进程 PID → `top -Hp <pid>` 找到高 CPU 的线程 ID → 转十六进制 → `jstack <pid> | grep 0x<hex>` 定位到具体代码行。如果是死循环或死锁，jstack 日志里会显示 `BLOCKED` 或循环执行的方法。
>
> **OOM 排查：** 首先确保启动参数里配了 `-XX:+HeapDumpOnOutOfMemoryError`，OOM 时自动 dump。拿到 dump 文件后用 MAT 打开——先看 Leak Suspects Report 找最大嫌疑对象，再看 Dominator Tree 按 Retained Heap 排序定位内存大户。最后通过 Path to GC Roots 看引用链，找到谁持有这个对象不放——90% 的情况是某个集合类（Map/List）往里面塞了没删。
>
> **高频场景：** 内存泄漏的典型特征是老年代持续增长，每次 FGC 回收不了多少——这是泄漏，不是配置问题。用 Arthas 的 `heapdump` 可以在不触发 FGC 的情况下 dump（因为 `jmap -histo:live` 会触发 FGC），再用 MAT 分析。
>
> **工具选型：** JDK 自带 jps/jstat/jmap/jstack 足够排查 80% 的问题。线上调优用 Arthas——dashboard 看全局、thread 定位 CPU、jad 确认代码版本、watch 观察方法调用。离线深度分析用 MAT。
