# JVM 内存结构与对象创建

> [!summary] 我的速记
> **JVM 5 大数据区：堆（共享，存对象）、虚拟机栈（私有，存栈帧）、方法区（共享，存类信息+常量）、程序计数器（私有，存指令地址）、本地方法栈（私有，Native方法）。**
> **堆又分两代：新生代（Eden:S0:S1=8:1:1）+ 老年代。JDK8 方法区用 Metaspace（本地内存），告别永久代。**
> **new 对象全流程：类加载检查 → 分配内存（TLAB优先）→ 初始零值 → 设对象头 → init方法。**

## 二、五大区域逐一深挖

### 2.1 程序计数器（PC Register）

| 属性 | 说明 |
|------|------|
| 线程 | **私有** |
| OOM | **不会**（唯一不会 OOM 的区域） |
| 内容 | 当前线程执行的**字节码行号指示器** |

```java
// 字节码示例
0: iconst_1      // PC = 0
1: istore_1      // PC = 1
2: iload_1       // PC = 2
3: ireturn       // PC = 3
```

> 🧠 **为什么需要 PC？** 线程切换后恢复执行时需要知道上次执行到哪一行。Native 方法时 PC 为 undefined。

---

### 2.2 虚拟机栈（VM Stack）

| 属性 | 说明 |
|------|------|
| 线程 | **私有** |
| 生命周期 | 和线程同生共死 |
| OOM | StackOverflowError（栈深度超限） / OutOfMemoryError（无法扩展） |

**栈帧结构（每个方法调用 = 一个栈帧）：**

```
┌──────────────────────┐
│  局部变量表            │  ← 方法参数 + 局部变量（基本类型存值，对象存引用）
│  (Local Variables)   │
├──────────────────────┤
│  操作数栈              │  ← 计算中间结果的临时空间（JVM是栈式指令集）
│  (Operand Stack)     │
├──────────────────────┤
│  动态链接              │  ← 指向运行时常量池中该方法的符号引用
│  (Dynamic Linking)   │
├──────────────────────┤
│  返回地址              │  ← 调用者 PC 的值（方法结束回到哪）
│  (Return Address)    │
└──────────────────────┘
```

```java
// 栈帧示例
public int add(int a, int b) {  // 栈帧入栈
    int sum = a + b;            // 局部变量表: a, b, sum
    return sum;                 // 操作数栈: 计算 a+b 的临时值
}                               // 栈帧出栈
```

**StackOverflowError 示例：**

```java
public void recursive() { recursive(); }  // 无限递归 → 栈帧一直入栈 → StackOverflow
// 默认栈深度约 10000，可通过 -Xss 调整
```

---

### 2.3 本地方法栈（Native Method Stack）

和虚拟机栈类似，只不过服务的是 **Native 方法**（C/C++写的）。HotSpot 把它和虚拟机栈合二为一。

---

### 2.4 堆（Heap）——面试重灾区

| 属性   | 说明                                            |
| ---- | --------------------------------------------- |
| 线程   | **共享**                                        |
| 创建时机 | JVM 启动时                                       |
| 内容   | 所有**对象实例**和**数组**                             |
| OOM  | `java.lang.OutOfMemoryError: Java heap space` |

**堆的分代结构（JDK8）：**

```
┌─────────────────────────────────────────────────┐
│                     Java 堆                      │
├────────────────────────┬────────────────────────┤
│      新生代 (1/3)       │     老年代 (2/3)        │
├────────┬──────┬────────┤                        │
│  Eden  │ S0   │  S1    │                        │
│ (8/10) │(1/10)│ (1/10) │                        │
└────────┴──────┴────────┴────────────────────────┘

比例：新生代 : 老年代 = 1 : 2
     Eden : S0 : S1 = 8 : 1 : 1
```

> 🧠 **对象的一生：** 新生 → Eden → 熬过一次 MinorGC → S0/S1 来回 → 年龄到 15（默认）→ 老年代。大对象直接进老年代（`-XX:PretenureSizeThreshold`）。

---

### 2.5 方法区（Method Area / Metaspace）

|     | JDK7 永久代（PermGen）                           | JDK8 Metaspace                             |
| --- | ------------------------------------------- | ------------------------------------------ |
| 位置  | JVM 堆内                                      | **本地内存**（堆外）                               |
| 大小  | 默认小，容易 OOM                                  | 默认无上限，用多少占多少                               |
| OOM | `java.lang.OutOfMemoryError: PermGen space` | `java.lang.OutOfMemoryError: Metaspace`    |
| 配置  | `-XX:PermSize` `-XX:MaxPermSize`            | `-XX:MetaspaceSize` `-XX:MaxMetaspaceSize` |

**方法区存什么：** 类的元信息（类名、字段、方法）、运行时常量池、静态变量、JIT 编译后的代码缓存。

> 🧠 **为什么从永久代改成 Metaspace？** ①永久代在堆内，GC 复杂；②永久代大小难预估，容易 OOM；③Metaspace 用本地内存，不挤占堆空间，默认无上限。

## 三、对象创建全过程（面试必画）

```java
User user = new User();
```

```
① 类加载检查
   └→ 检查 User 类是否已加载 → 没有则走类加载

② 分配内存
   ├── 指针碰撞（Bump the Pointer）：堆规整时，指针往后挪
   └── 空闲列表（Free List）：堆不规整时，从列表找
   优先 TLAB（Thread Local Allocation Buffer）：线程私有的分配缓冲区
     → 避免多线程分配内存时的竞争（CAS）

③ 初始化零值
   └→ 实例字段赋默认值（int→0, boolean→false, 对象→null）

④ 设置对象头
   └→ Mark Word + Klass Pointer + 数组长度（如果是数组）

⑤ 执行 <init>
   └→ 执行构造方法，赋程序员指定的初始值
```

**TLAB 是什么？**

每个线程在 Eden 区有一小块私有的分配缓冲区（默认 Eden 的 1%）。线程在自己的 TLAB 里分配对象**不需要加锁**——如果在 TLAB 里分配失败（剩余空间不够放新对象），才去 Eden 的共享区域用 CAS 分配。

> 🧠 **面试话术：** "TLAB 是 JVM 的优化——每个线程在 Eden 区有自己的私有缓冲区，避免多线程分配对象时的锁竞争。这体现了 JVM 对并发场景的考虑。"

## 四、对象内存布局

```
┌─────────────────┬──────────────────┬──────────────┐
│   对象头 Header  │  实例数据 Instance  │ 对齐填充 Padding │
├────────┬────────┤    Data           │              │
│Mark Word│Klass   │                   │              │
│64bit   │Pointer │                   │              │
│        │(压缩32)│                   │              │
└────────┴────────┴──────────────────┴──────────────┘
```

| 部分 | 内容 | 大小 |
|------|------|------|
| Mark Word | 锁信息、GC 年龄、hashCode | 8 字节（64 位） |
| Klass Pointer | 指向方法区中类元信息的指针 | 4 字节（开启压缩）/ 8 字节 |
| 实例数据 | 字段值 | 字段数量和类型决定 |
| 对齐填充 | 补齐到 8 字节倍数 | 0~7 字节 |

## 五、各区域 OOM 场景速查

| 区域 | 异常信息 | 常见原因 |
|------|----------|----------|
| **堆** | `Java heap space` | 对象太多/太大 → 泄漏或内存不够 |
| **栈** | `StackOverflowError` | 递归太深 / 方法调用链太长 |
| **Metaspace** | `Metaspace` | 动态生成类太多（CGLIB、动态代理） |
| **直接内存** | `Direct buffer memory` | NIO 的 ByteBuffer.allocateDirect 太多 |
| **GC Overhead** | `GC overhead limit exceeded` | GC 时间占总时间 98% 以上，但回收不到 2% 堆 |

## 六、逃逸分析与标量替换

> 来源：播面 [ID:1116](https://www.bomianfm.com/web/question/1116) | 本笔记原缺失此知识点

### 6.1 逃逸分析（Escape Analysis）

**逃逸分析是 JIT 编译器的一项分析技术**，判断对象作用域是否超出当前方法或线程。

| 逃逸状态 | 含义 | 示例 |
|----------|------|------|
| **不逃逸** | 对象仅在方法内使用 | 局部变量，方法结束即销毁 |
| **方法逃逸** | 对象被传给其他方法或作为返回值 | 方法参数传递、return 对象 |
| **线程逃逸** | 对象被多线程访问 | 赋值给静态变量、被其他线程引用 |

```
比喻：
  不逃逸 = 机密文档你看完就碎了（只有你知道）
  方法逃逸 = 你把文档递给同事（传给另一个方法）
  线程逃逸 = 你把文档贴在公告栏上（所有人可见）
```

### 6.2 标量替换（Scalar Replacement）

如果逃逸分析发现对象**不逃逸**且可拆解，JIT 编译器会**把对象拆成若干基本类型（标量），直接在栈/寄存器中分配**——不创建对象，也就无需 GC。

```java
// 优化前（程序员写的）
class Point { int x; int y; }
void myMethod() {
    Point p = new Point(10, 20);  // 对象在堆上
    int result = p.x + p.y;
}

// 标量替换后（编译器实际执行的）
void myMethod() {
    int x = 10;   // 对象被拆成标量，直接在栈上
    int y = 20;
    int result = x + y;  // 没有堆分配，没有 GC！
}
```

### 6.3 逃逸分析还能触发什么？

| 优化 | 说明 |
|------|------|
| **标量替换** | 不逃逸对象拆成标量 → 栈上分配，免 GC |
| **同步消除（锁消除）** | 单线程使用的对象 → 去掉 `synchronized` |
| **栈上分配** | HotSpot 没有真正实现对象栈上分配，而是通过标量替换模拟等效效果 |

> 🧠 **关键认知：** Java 7 起逃逸分析默认开启。HotSpot **没有真正的对象栈上分配**——标量替换是等效替代。大量短生命周期小对象通过这招避免了堆分配，是 JIT 最重要的优化之一。

## 七、字节码执行机制

> 来源：播面 [ID:1177](https://www.bomianfm.com/web/question/1177) | 本笔记原仅覆盖内存结构未涉及执行引擎

### 7.1 宏观流程：字节码到机器码

```
.class 字节码
    │
    ├── 类加载器 → 加载到方法区
    │
    ├── 运行时数据区 ← 程序计数器 + 虚拟机栈 + 堆
    │
    └── 执行引擎（核心！）
          ├── 解释器（Interpreter）：逐条翻译字节码 → 立即执行
          │     特点：启动快，效率低（每次都翻译）
          │
          ├── JIT 编译器（Just-In-Time）：识别热点代码 → 编译成机器码 → 缓存
          │     特点：编译耗时，但执行极快（接近 C/C++）
          │     热点判定：方法调用次数 / 循环回边次数超过阈值
          │
          └── 混合模式 + 分层编译：兼顾启动速度与峰值性能
```

### 7.2 微观执行：a + b 的字节码过程

```java
int a = 1;
int b = 2;
int c = a + b;
```

| 字节码 | 操作数栈 | 局部变量表 | 说明 |
|--------|---------|-----------|------|
| `iconst_1` | [1] | | 常量1压入栈顶 |
| `istore_1` | [] | [a=1] | 弹出栈顶 → 存入变量a |
| `iconst_2` | [2] | [a=1] | 常量2压入栈顶 |
| `istore_2` | [] | [a=1, b=2] | 弹出栈顶 → 存入变量b |
| `iload_1` | [1] | [a=1, b=2] | 加载a到栈 |
| `iload_2` | [1,2] | [a=1, b=2] | 加载b到栈 |
| `iadd` | [3] | [a=1, b=2] | 弹出a+b，压入结果3 |
| `istore_3` | [] | [a=1,b=2,c=3] | 弹出结果 → 存入c |

> 🧠 **JVM 是基于栈的指令集**（不是寄存器架构）——所有运算通过操作数栈完成。这保证了跨平台一致性（不依赖 CPU 寄存器），但指令更多。JIT 编译时会将栈操作映射为寄存器操作来优化。

### 7.3 解释器 vs JIT 编译器

| | 解释器 | JIT 编译器 |
|:---|:---|:---|
| 工作方式 | 逐条翻译执行 | 热点代码→机器码缓存 |
| 启动速度 | ✅ 快 | ❌ 慢（需编译） |
| 峰值性能 | ❌ 低 | ✅ 高（接近静态编译） |
| 资源消耗 | 低 | 高（编译占 CPU/内存） |

> 🧠 **总结：** JVM 执行字节码 = 类加载（字节码进方法区）→ 程序计数器指向指令 → 解释器逐条翻译 → 热点代码触发 JIT 编译优化。这个流程是新版面试高频题（尤其是 JIT、热点代码、栈式指令集架构），补充后覆盖完整。

## 面试满分话术

> **面试官：** 讲讲 JVM 内存结构？

**你的回答：**

> JVM 运行时数据区分为 5 个部分。**线程私有**的有三个：程序计数器（存当前字节码行号）、虚拟机栈（每个方法一个栈帧，存局部变量表/操作数栈/返回地址）、本地方法栈（Native 方法）。**线程共享**的有两个：堆（所有对象实例和数组，分新生代和老年代）和方法区（类元信息、运行时常量池、静态变量）。
>
> 堆的分代上，新生代用复制算法（Eden:S0:S1=8:1:1），对象在 S0/S1 之间来回复制，熬过 15 次 MinorGC 晋升到老年代（老年代采用标记整理或者标记清楚）。JDK8 把方法区的实现从永久代改成了 Metaspace——存在本地内存里，默认无上限，避免了永久代的 OOM 问题。
>
> 对象创建流程五步走——类加载检查、分配内存（优先用 TLAB 避免竞争，内存规整用指针碰撞否则用空闲列表）、初始化零值、设置对象头、执行构造方法 init。这个流程中 TLAB 是个容易被问到的优化点：每个线程在 Eden 有自己的私有缓冲区，分配对象时不需要 CAS 竞争。
