# GC 垃圾回收

> [!summary] 我的速记
> **GC 三问：哪些要回收（可达性分析+GC Roots）→ 怎么回收（算法）→ 谁来做（收集器）。**
> **四大算法：标记清除（碎片）、标记整理（慢但规整）、复制（快但浪费空间）、分代（组合最优）。**
> **面试最爱问 CMS vs G1：CMS（并发标记清除，碎片化STW不可控）→ G1（分区+可预测停顿，Region 单位回收）。**
> **ZGC（JDK11/15）：染色指针+读屏障，STW < 1ms。**

---

## 一、怎么判断对象该回收？

### 1.1 引用计数法（Java 不用）

```java
A.obj = B;  // B 的引用计数 = 1
B.obj = A;  // A 的引用计数 = 1
A = null;   // A 引用计数 = 0? NO! B 还引着 A
B = null;   // B 引用计数 = 0? NO! A 还引着 B
// → 循环引用 → 两个对象永远不会被回收 → 内存泄漏！
// Python 用了引用计数，额外加了一个循环检测器
```

> 🧠 **Java 不用引用计数，而用可达性分析——因为引用计数解决不了循环引用。**

---

### 1.2 可达性分析（GC Roots）— Java 用的方法

```
GC Roots 是什么？一组必须活跃的引用。

GC Roots 包含：
  ├── 虚拟机栈中引用的对象（局部变量表中的引用）
  ├── 方法区中 static 变量引用的对象
  ├── 方法区中常量引用的对象（String Table 中的引用）
  ├── 本地方法栈中 JNI 引用的对象
  ├── 被 synchronized 持有的对象（Monitor 对象）
  └── JVM 内部的引用（Class 对象、ClassLoader 等）

从这些"根"出发，沿着引用链遍历 → 能到达的 = 存活 → 不能到达的 = 可回收
```

```
GC Roots
  │
  ├──→ obj1 ──→ obj2  ← 存活！
  │
  └──→ obj3 ──→ obj4 ──→ obj5
                            ↑
                      obj6 ─┘  ← obj5 和 obj6 循环引用
                                  但只要GC Roots到不了 → 可回收！
```

## 二、四大回收算法

| 算法 | 原理 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **标记-清除** | 标记存活→统一清除 | 简单 | ❌碎片多 ❌效率不稳定 | 老年代（CMS 基础） |
| **标记-整理** | 标记存活→移到一端→清边界 | 无碎片 | ❌移动对象耗时（STW 长） | 老年代（Serial Old/Parallel Old） |
| **复制** | 分两块→存活从 A 复制到 B→清 A | ✅快 ✅无碎片 | ❌浪费一半空间 | **新生代** |
| **分代** | 新生代用复制，老年代用标记整理/清除 | ✅综合最优 | 实现复杂 | **JVM 默认** |

> 🧠 **新生代为什么用复制算法？** 新生代对象 98% 朝生夕死，每次只有少量存活——复制成本极低。老年代对象存活率高，用复制代价太大。

## 三、经典垃圾收集器——7 种

### 3.1 全景图

```
       新生代收集器              老年代收集器
         │                        │
  Serial ───┐              Serial Old ───┐
  ParNew ───┤              Parallel Old ─┤   组合线
  Parallel ─┘              CMS ───────────┘
   Scavenge
                            G1 (整堆)
                            ZGC (整堆)
```

| 收集器 | 代 | 线程 | 算法 | 特点 |
|--------|-----|------|------|------|
| **Serial** | 新生代 | 单线程 | 复制 | 最早，简单高效，Client 模式默认 |
| **ParNew** | 新生代 | 多线程 | 复制 | Serial 多线程版，唯一能和 CMS 配合的新生代 |
| **Parallel Scavenge** | 新生代 | 多线程 | 复制 | **吞吐量优先**，可控停顿时间 |
| **Serial Old** | 老年代 | 单线程 | 标记整理 | Serial 的老年代版 |
| **Parallel Old** | 老年代 | 多线程 | 标记整理 | Parallel Scavenge 的老年代搭档 |
| **CMS** | 老年代 | 多线程 | 标记清除 | **低停顿**，JDK9 废弃，JDK14 移除 |
| **G1** | 整堆 | 多线程 | 标记整理+复制 | **可预测停顿**，JDK9 默认 |
| **ZGC** | 整堆 | 多线程 | 染色指针 | **超低停顿<1ms**，JDK11 实验→JDK15 正式 |

---

### 3.2 CMS（Concurrent Mark Sweep）——低停顿经典

```
CMS 四个阶段：
  ① 初始标记 (STW)
     标记 GC Roots 能直接关联到的对象
     STW 极短

  ② 并发标记 (与用户线程并发)
     从 GC Roots 出发遍历对象图，标记所有可达对象
     耗时最长，但和用户线程并发执行

  ③ 重新标记 (STW)
     修正并发标记期间用户线程继续运行导致的标记变化
     比初始标记长但比并发标记短

  ④ 并发清除 (与用户线程并发)
     清除未被标记的对象
     不需要移动对象 → 产生内存碎片
```

**CMS 三大问题：**

| 问题 | 描述 | 后果 |
|------|------|------|
| **浮动垃圾** | 并发清除时新产生的垃圾无法清理 → 留到下次 GC | 需要预留空间，堆满 92% 就触发 GC |
| **碎片化** | 标记清除不整理 → 碎片越来越多 | 大对象分配失败 → 触发 Full GC（Serial Old 整理） |
| **CPU 敏感** | 并发阶段占用 CPU 核心 | 核心数少时性能下降明显 |

> 🧠 **CMS 怎么解决并发标记中的引用变化？** 三色标记 + 增量更新（Incremental Update）。黑色对象新增了对白色对象的引用 → 把黑色变回灰色 → 重新扫描。

---

### 3.3 G1（Garbage First）——面向大内存

```
G1 的核心创新：化整为零，Region 为单位

G1 把堆分成多个大小相等的 Region（默认 2048 个）：

│ R1 │ R2 │ R3 │ R4 │ R5 │ R6 │ R7 │ R8 │ R9 │ ...
  新生代 Region    老年代 Region    Humongous Region   空闲 Region
  (Eden/Survivor)                  (存巨型对象)
```

| 特性 | CMS | G1 |
|------|-----|-----|
| **回收单位** | 整个老年代 | **Region**——挑垃圾最多的 Region 回收 |
| **停顿预测** | ❌ 不可预测 | ✅ 用户指定 `-XX:MaxGCPauseMillis`，尽量满足 |
| **碎片处理** | ❌ 碎片化 → 偶尔 Full GC | ✅ 回收时局部复制整理 |
| **内存占用** | 小 | 大（Remember Set 占用 5-10% 堆空间） |
| **适用堆大小** | 4-8GB | **8GB+** |

**G1 的回收模式（三种）：**

| 模式 | 触发条件 | 回收范围 |
|------|----------|----------|
| **Young GC** | Eden 满了 | 新生代 Region |
| **Mixed GC** | 老年代使用率 > IHOP(默认45%) | 所有新生代 + 部分老年代 Region |
| **Full GC** | Mixed GC 跟不上分配速度 | 整堆（Serial Old，STW 长，尽量避免） |

> 🧠 **G1 怎么解决并发标记中的引用变化？** SATB（Snapshot-At-The-Beginning）算法——并发标记开始时对对象图拍快照。新产生的对象视为存活，保证不漏标。

---

### 3.4 ZGC（JDK11+）——超低停顿的未来

| 特性 | ZGC | G1 |
|------|-----|-----|
| 停顿时间 | **< 1ms**（所有 STW 操作 < 1ms） | 可预测，通常几十~几百 ms |
| 堆大小 | 支持 4TB+ | 推荐 < 100GB |
| 核心技术 | **染色指针**（Colored Pointers）+ 读屏障 | 记忆集（Remember Set） |
| 标记阶段 | 并发标记 | 并发标记 |
| 整理/压缩 | **并发重整**（Concurrent Compact） | 需要 STW |

> 🧠 **染色指针：** ZGC 把对象指针的 64 位中的 4 位拿来做标记（Finalizable/Remapped/Marked0/Marked1）。读屏障检查指针颜色 → 颜色不对就做相应处理。不需要额外的对象头来存 GC 信息——这就是 ZGC 的核心创新。

## 四、记忆集（Remember Set）和卡表（Card Table）

**问题：** 做 Young GC 时，新生代的对象可能被老年代引用（跨代引用）。如果每次都要扫描整个老年代来找这些引用，代价太大。

**解决：卡表（Card Table）**

```
老年代被分成 512 字节的"卡片"（Card）
每当老年代有引用指向新生代 → 对应的 Card 标记为"脏"
Young GC 时只需扫描标记为脏的 Card → 不需要扫描整个老年代
```

G1 更进一步：每个 Region 都有一个 Remember Set，记录了"哪些 Region 有引用指向我的 Region"。跨 Region 引用不需要全堆扫描。

## 五、空间分配担保机制

> 来源：播面 [ID:1166](https://www.bomianfm.com/web/question/1166) | Minor/Major/Full GC 区分 + 担保机制

### 5.1 Minor GC、Major GC、Full GC 区分

| GC 类型 | 回收区域 | 触发条件 | STW |
|---------|---------|----------|-----|
| **Minor GC** | 仅新生代 | Eden 满了 | ✅ 有，但很短 |
| **Major GC** | 仅老年代 | 老年代空间不足 | ✅ 有，比 Minor GC 长 |
| **Full GC** | 整堆（新生代+老年代+方法区） | 多种条件触发 | ✅ 有，最长 |

> ⚠️ **注意：** "Major GC" 在不同资料中含义不同——有的等同于 Full GC，有的特指老年代 GC。面试时建议区分 Minor GC vs Full GC，Major GC 做补充说明即可。

### 5.2 触发 Full GC 的常见条件

| 条件 | 说明 |
|------|------|
| **老年代空间不足** | 对象晋升时老年代没有足够连续空间 |
| **Metaspace 不足** | 类元数据占用超过 `MaxMetaspaceSize` |
| **System.gc()** | 显式调用（建议用 `-XX:+DisableExplicitGC` 禁止） |
| **Promotion Failed** | Minor GC 时存活对象太多，老年代装不下 |
| **Concurrent Mode Failure** | CMS 并发清理时老年代被填满 |
| **空间担保失败** | 老年代可用空间 < 历次晋升平均大小 → 直接 Full GC |
| **大对象分配失败** | 巨型对象直接进老年代但空间不够 |

### 5.3 空间分配担保机制

**目的：** Minor GC 前判断老年代能否安全容纳晋升对象，避免先做 Minor GC 发现不行又做 Full GC 的浪费。

```
Minor GC 前检查：
  │
  ├── 老年代可用连续空间 > 新生代所有对象总大小？
  │     └── 是 → 【绝对安全】→ 直接 Minor GC
  │
  ├── 老年代可用空间 > 历次晋升对象的平均大小？
  │     └── 是 → 【冒险】→ Minor GC（可能 Promotion Failed → 被迫 Full GC）
  │
  └── 否 → 【不安全】→ 直接 Full GC
```

**版本差异（面试高频！）：**
- **JDK 6u24 之前**：`HandlePromotionFailure` 参数有效，可手动控制
- **JDK 6u24 及以后**：参数失效，JVM 永远允许冒险，仅依据空间大小判断

> 🧠 **比喻：** 小剧场（新生代）散场 → 保安看体育馆（老年代）还有多少空座。空座 > 小剧场总人数 → 绝对安全。空座 > 历史平均转移人数 → 冒险试试。空座太少 → 直接清场扩容（Full GC）。

## 面试满分话术

> **面试官：** 讲讲你知道的 GC 收集器？CMS 和 G1 区别？

**你的回答：**

> **第一层（全景）：** Java 的垃圾收集器是一个家族。新生代有 Serial（单线程）、ParNew（多线程，唯一配 CMS 的）、Parallel Scavenge（吞吐量优先）。老年代有 Serial Old、Parallel Old、CMS（低停顿），以及整堆回收的 G1 和 ZGC。
>
> **第二层（CMS）：** CMS 是低停顿的标杆——目标是最小化 STW 时间。四阶段：初始标记（STW 极短）→ 并发标记（和用户线程一起跑）→ 重新标记（STW 修正）→ 并发清除。但它有三个致命问题：浮动垃圾（并发清除时新垃圾只能等下次→堆用到92%触发GC）、碎片化（标记清除不整理→大对象分配失败时退化 Serial Old 做 Full GC）、CPU 敏感（并发占用核心）。
>
> **第三层（G1）：** G1 的目标是可预测停顿——把堆分成大小相等的 Region，每次挑垃圾最多的 Region 回收（Garbage First）。它用 SATB 算法解决并发标记中的漏标问题，通过 Remember Set 避免全堆扫描跨代引用。G1 有 Young GC、Mixed GC 和 Full GC 三种回收模式。和 CMS 最核心的区别：CMS 关心**停顿短**，G1 关心**停顿可预测**；CMS 碎片化严重，G1 每次回收都局部整理。
>
> **第四层（ZGC）：** 如果面试官对前沿技术感兴趣——JDK11 引入的 ZGC 用染色指针和读屏障，把 STW 控制在 1ms 以内，甚至支持并发整理，是超大堆（TB 级）场景的未来方向。
