# 类加载机制与双亲委派

> [!summary] 我的速记
> **类加载 = 7 步：加载→验证→准备→解析→初始化→使用→卸载。核心是前 5 步。**
> **双亲委派 = 儿子先问爹：AppCL→ExtCL→BootCL。爹能加载爹来，爹不行儿子再上。**
> **为什么要打破？** Tomcat 隔离 Web 应用、SPI 接口由 BootCL 加载但实现由 AppCL 加载（反向委派）。

---

## 一、类加载的 7 个步骤

```
.class 文件
  │
  ├── ① 加载（Loading）
  │     通过全限定名获取二进制字节流 → 方法区生成 Class 对象
  │
  ├── ② 验证（Verification）
  │     魔术CAFEBABE、版本号、字节码安全检查
  │
  ├── ③ 准备（Preparation）
  │     给 static 变量分配内存、赋默认值（不是程序员赋的值！）
  │     static int x = 123; → 这步 x = 0
  │
  ├── ④ 解析（Resolution）
  │     符号引用 → 直接引用（"java/lang/String" → 内存地址）
  │
  ├── ⑤ 初始化（Initialization）
  │     执行 static{} 块，给 static 变量赋真实初值
  │     ← Class.forName 走到这！loadClass 不走这！
  │
  ├── ⑥ 使用（Using）
  │
  └── ⑦ 卸载（Unloading）
        类的 Class 对象被 GC → 方法区中类信息被回收
```

> 🧠 **准备 vs 初始化：** 准备阶段 static 变量 = 默认值（0/null），初始化阶段才 = 程序指定的值。这是面试中最容易被问倒的细节。

## 二、双亲委派模型（Parent Delegation）

### 2.1 三层类加载器

```
BootStrap ClassLoader（启动类加载器）
  ← C++ 实现，没有 Java 对象
  ← 加载 <JAVA_HOME>/lib 下的核心类（rt.jar、java.*）
        │
        ▼
Extension ClassLoader（扩展类加载器）
  ← sun.misc.Launcher$ExtClassLoader
  ← 加载 <JAVA_HOME>/lib/ext 下的扩展类
  ← JDK9 后改名为 PlatformClassLoader
        │
        ▼
Application ClassLoader（应用类加载器）
  ← sun.misc.Launcher$AppClassLoader
  ← 加载 classpath 下的应用类
```

### 2.2 工作流程（loadClass 源码）

```java
protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        // ① 先查自己加载过没有
        Class<?> c = findLoadedClass(name);

        if (c == null) {
            // ② 没加载过 → 交给父加载器
            try {
                if (parent != null) {
                    c = parent.loadClass(name, false);   // 向上委派
                } else {
                    c = findBootstrapClassOrNull(name);  // 到顶了，BootStrap加载
                }
            } catch (ClassNotFoundException e) {
                // ③ 父加载器加载不了 → 自己加载
                c = findClass(name);
            }
        }
        return c;
    }
}
```

> 🧠 **一句话流程：** 先检查缓存 → 没缓存就往上委派 → 父加载不了才自己找 → 找到就缓存。这就是"双亲委派"——但实际上是"向上委派"。

### 2.3 为什么要双亲委派？

**防止核心类被篡改。** 如果有人写了一个 `java.lang.String` 放到 classpath，没有双亲委派的话，应用加载器就会加载这个假的 String——JVM 里出现两个不同的 String 类，类型系统崩溃。

有了双亲委派：加载 `java.lang.String` → AppCL 往上问 ExtCL → ExtCL 往上问 BootCL → BootCL 发现 rt.jar 里有 → BootCL 加载 → 你写的假 String 永远不会被加载。

> 🧠 **核心思想：** "核心库由核心类加载器加载，应用层别想冒充。"

## 三、打破双亲委派——什么时候需要？

| 场景 | 为什么打破 | 怎么打破 |
|------|-----------|----------|
| **Tomcat** | 多个 Web 应用需要隔离各自的类（不同版本的 Spring），也要共享公共类 | 自定义 WebAppClassLoader，先自己找再委派 |
| **SPI（JDBC）** | 接口 `java.sql.Driver` 由 BootCL 加载，但实现 `com.mysql.Driver` 在 classpath | 线程上下文类加载器（Thread Context ClassLoader） |
| **OSGi** | 模块化，每个 bundle 有自己的类加载器 | 网状委派 |
| **热部署** | 需要替换已加载的类 | 自定义 ClassLoader 重新加载 |

### 3.1 SPI 如何打破——线程上下文类加载器

```
问题：
  java.sql.DriverManager 由 BootCL 加载
    → 它内部调 ServiceLoader.load(Driver.class)
    → 但 MySQL 的 Driver 实现类在 classpath（由 AppCL 加载）
    → BootCL 找不到 AppCL 加载的类！

解决：Thread.currentThread().getContextClassLoader()
  → 默认是 AppCL
  → SPI 代码通过它来加载实现类
  → 实现了"反向委派"：爹（BootCL）请儿子（AppCL）帮忙加载
```

> 🧠 **这就是今天上午讲的—JDBC 为什么曾经要写 `Class.forName`：** 因为 Driver 的 static {} 里自己注册，绕过了 SPI 的类加载问题。JDBC4 用 SPI 自动发现，底层就是线程上下文类加载器。

## 四、类的卸载条件

类需要同时满足三个条件才能被 GC 卸载：
1. 该类的所有实例已被 GC
2. 加载该类的 ClassLoader 已被 GC
3. 该类的 Class 对象没有被引用

> 🧠 **由 BootCL/ExtCL/AppCL 加载的类永远不会被卸载**——因为这些加载器本身永远不会被 GC。只有自定义 ClassLoader 加载的类才可能被卸载。这就是热部署的原理。

## 面试满分话术

> **面试官：** 讲讲类加载机制和双亲委派？

**你的回答：**

> **第一层（类加载步骤）：** 类加载分 5 个核心步骤——加载、验证、准备、解析、初始化。加载是把 .class 字节流读到方法区生成 Class 对象；验证是检查字节码合法性；准备阶段给 static 变量分配内存并赋默认零值；解析把符号引用变成直接引用；初始化执行 static 代码块和赋程序员指定的初值。
>
> **第二层（双亲委派）：** Java 有三层类加载器——BootStrap（核心库）、Extension（扩展库）、Application（classpath）。加载一个类时，从 AppCL 开始逐级往上委派，父加载器能加载就父来，父不行再自己加载。核心目的是防止核心类被篡改——你自己写的 java.lang.String 永远不会被 AppCL 直接加载，因为 BootCL 先拦截了。
>
> **第三层（打破场景）：** 双亲委派不是铁律。Tomcat 为了隔离不同 Web 应用（各自不同版本的 Spring），每个应用有自己的 WebAppClassLoader，先自己找再委派。SPI 机制更特殊——接口由 BootCL 加载但实现类在 classpath，靠线程上下文类加载器实现"反向委派"。热部署也是通过自定义 ClassLoader 重新加载来实现的。
>
> **第四层（和今天上午讲的串联）：** Class.forName 和 loadClass 的区别就在初始化这一步——forName 加载+初始化（执行 static 块），loadClass 只加载。JDBC 的 Driver 在 static 块里自注册，所以早期必须写 Class.forName，JDBC4 用 SPI 自动发现后可以省略——但面试里这个区别是经典考点。
