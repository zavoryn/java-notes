# Elasticsearch 面试题汇总

## 基础认知（必会）

### Q1：什么是 Elasticsearch？做什么用的？
> ES 是一个基于 Lucene 的分布式搜索引擎，核心能力是**近实时的全文检索**和**聚合分析**。
> 典型应用：电商商品搜索、日志分析（ELK）、搜索推荐、站内搜索。

### Q2：ES 和 MySQL 的区别？
> MySQL 是正向索引 → 按 ID 查数据快，事务支持好
> ES 是倒排索引 → 按关键词查数据快，聚合分析强
> 经典架构：MySQL 主存储 + ES 搜索索引，通过 Canal/MQ 同步

### Q3：倒排索引是什么？怎么工作的？
> 核心概念：**词条(Term) → 文档ID列表**的映射。
> 构建三步：分词 → 归一化 → 建倒排表。
> 内部结构：Term Dictionary → Term Index (FST) → Posting List（FOR压缩+Roaring Bitmap） → 磁盘文件

### Q4：为什么说 ES"近实时"而不是"实时"？
> 写入的数据先进入内存 buffer，**默认每 1 秒** refresh 成 segment 才能被搜索到。
> 这个 1 秒的延迟就是"近实时"的原因（可以调小 refresh_interval 但影响性能）。

## 写入/存储（重点）

### Q5：ES 写入一条文档的完整流程？
```
协调节点路由 → 主分片写 translog → 写内存 buffer → 返回成功 → 同步副本
后台：refresh(1s) → segment(可搜索) → flush → 持久化磁盘
```
> 关键机制：
> - **translog**：防丢失（类比 MySQL binlog）
> - **refresh**：数据变得可搜索
> - **flush**：数据落盘

### Q6：如何保证写入数据不丢失？
> translog（事务日志）机制，每次写操作都会 fsync 到磁盘。
> 宕机后重启会重放 translog 恢复未持久化的数据。
> 可配置：`index.translog.durability = request`（每个请求都刷盘）

### Q7：ES 索引的 Mapping 和 MySQL 的 Schema 有什么不同？
> MySQL Schema：严格的字段类型，可增删改
> ES Mapping：
> - 动态映射：自动推断类型（生产危险，可能猜错）
> - 显式映射：手动定义（生产必须）
> - ⚠️ Mapping **只能新增字段，不能修改已有字段类型**

### Q8：text 和 keyword 的区别？什么场景用哪个？
| text | keyword |
|:---|:---|
| 会被分词 | 不会被分词 |
| 适合全文搜索 | 适合精确匹配、聚合、排序 |
| 搜"人民"能命中"中华人民共和国" | 搜"人民"命中不了，必须完整匹配 |

## 搜索/查询（高频）

### Q9：ES 搜索的两阶段（Query then Fetch）？
> **Query 阶段**：协调节点广播到每个分片 → 各分片本地搜索返回 (doc_id, score) → 协调节点做全局排序取 topN
> **Fetch 阶段**：协调节点根据 doc_id 去对应分片取完整文档 → 返回客户端
> 这样能减少数据传输量，只传需要的文档。

### Q10：query 和 filter 的区别？
| query | filter |
|:---|:---|
| 计算 _score（相关性） | 不计算分数 |
| 较慢 | **较快**（有缓存） |
| 适合模糊匹配、搜索 | 适合精确匹配、范围过滤 |

> 💡 面试金句：**能用 filter 尽量用 filter，它不计算分数且有缓存。**

### Q11：深度分页为什么慢？怎么解决？
> 原因：`from + size` 需要每个分片取 from+size 条数据到协调节点排序，深度越深开销越大（如取第10000页，5个分片就是50000条）
>
> 解决：
> - **search_after**：基于上一页排序值继续查，实时游标（推荐）
> - **scroll**：快照模式，适合全量导出，非实时
> - **限制 from + size 范围**

### Q12：BM25 是什么？
> ES 5.0+ 默认相关性打分算法，是 TF-IDF 的改进版。
> 引入**字段长度惩罚**：短文档匹配得分更高（因为同样的词出现在短文档中更有信息量）。
> 公式包含：TF（词频）、IDF（逆文档频率）、docLen（文档长度归一化）。

## 集群/架构

### Q13：ES 集群有哪几种节点？
> - **Master Node**：管理集群元数据，不处理数据（可以做数据节点，但生产建议分离）
> - **Data Node**：存储数据、CRUD、搜索聚合
> - **Coordinating Node**：请求路由 + 结果聚合（每个节点默认都是）
> - **Ingest Node**：数据预处理（pipeline）

### Q14：主分片和副本分片的作用？为什么主分片数不能改？
> 主分片：数据存储容器，创建索引时确定。
> 副本分片：主分片的拷贝，提供**高可用**（容错）和**读扩展**（分担查询压力）。
>
> 不能改主分片数的原因：`shard = hash(_routing) % number_of_primary_shards`
> 如果改了分母，取模结果全变，旧数据的路由全部失效。

### Q15：集群健康状态的三种颜色分别代表什么？
| Green | 所有主/副本都正常分配 |
| Yellow | 主分片正常，但部分副本未分配（⚠️单节点集群必然是 yellow） |
| Red | 有主分片丢失，数据不完整（🔴线上事故级别） |

### Q16：ES 如何实现高可用？
> 1. 副本分片 → 主分片挂了副本接替
> 2. Master 选举 → 基于 Zen Discovery（7.x 后是协调层自动选举）
> 3. 最小主节点数：`discovery.zen.minimum_master_nodes` 防止脑裂（7.x 自动管理）

## 性能优化

### Q17：写入慢怎么优化？
> 1. 批量写入（Bulk API）
> 2. 增大 refresh_interval（比如 30s）
> 3. 写入时先设 `number_of_replicas = 0`，写完再加
> 4. 使用 SSD
> 5. 使用自动生成 ID（比指定 ID 快）
> 6. 减少分片数（分片越多，分段合并开销越大）

### Q18：查询慢怎么优化？
> 1. 用 filter 代替 query（不计算分数 + 有缓存）
> 2. 减少返回字段 `_source: ["title", "price"]`
> 3. 避免深度分页（用 search_after）
> 4. 合理规划分片数（不超过总数据/30GB）
> 5. 避免 `*` 前缀的通配符查询
> 6. 冷热分离架构（热节点 SSD 存近期数据）

## 场景设计（能力拔高）

### Q19：MySQL 数据如何同步到 ES？
> 方案一（推荐）：**Canal 监听 MySQL binlog** → 推送到 MQ（Kafka） → ES 消费者写入
> 方案二：业务代码双写（MySQL + ES），需注意一致性
> 方案三：定时任务全量/增量同步（简单但有延迟）

### Q20：ES 适合当主数据库吗？
> **不适合**。原因：
> 1. 不支持事务
> 2. 写入有近实时延迟（1s refresh）
> 3. 数据可靠性不如关系型数据库
> 4. 没有连接（JOIN）操作
> ES 应定位为"搜索索引"，MySQL 做主存储。
