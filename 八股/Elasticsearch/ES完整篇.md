## 1. ES 是什么 —— 一句话讲清楚

### 一句话定义

> **Elasticsearch 是一个基于 Lucene 构建的分布式、近实时（NRT）、RESTful 的全文搜索引擎。**

拆开来看，每个定语都是考点：

| 定语 | 含义 | 面试时怎么说 |
|:---|:---|:---|
| **基于 Lucene** | 底层引擎是 Apache Lucene，负责倒排索引和搜索的硬核算法 | ES 没有重新发明轮子，而是在 Lucene 之上封装了分布式能力 |
| **分布式** | 数据分片存储在多台机器上，天然水平扩展 | 一个索引可以拆成 N 个分片，分布在 N 个节点上，并行处理 |
| **近实时（NRT）** | 写入后默认 1 秒即可被搜索到 | 不是实时，有约 1 秒的 refresh 延迟，但对搜索场景足够 |
| **RESTful** | 通过 HTTP + JSON 交互，无语言绑定 | `GET /index/_search` + JSON body，任何语言都能调 |
| **全文搜索** | 按关键词搜内容，而不是按 ID 查行 | 你搜"Spring 面试题"，ES 会给你所有包含这些词的文档 |

### 现实类比 —— 图书馆管理员

> 把 ES 想象成一个**超级图书馆管理员**。
>
> - MySQL 是**书架的编号系统**：你要找"X-03-17 号书"，它从编号索引里直接定位，很快。
> - ES 是**索引卡片柜**：你要找"讲 Redis 的书"，它翻卡片，"Redis"这个词出现在第 3、17、42、88 本书里，直接给你书单。
>
> 两种索引，各有所长。面试时你不是去挑战你能不能替代 MySQL，而是去说明 **ES 擅长什么、MySQL 擅长什么、两者怎么配合**。

### ELK 栈

```
E → Elasticsearch   搜索+存储引擎（核心）
L → Logstash        数据采集管道（日志清洗、转换、入库）
K → Kibana          可视化展示（图表、仪表盘）
```
再加一个轻量级日志采集器 Beats（如 Filebeat 读文件），就是完整的日志平台。面试时只需记住：**"ELK 是企业级日志分析的标准方案。"**

## 2. 核心概念 —— MySQL 类比法

面试时用 MySQL 做类比，一是展现你的横向对比能力，二是面试官自己也用 MySQL，秒懂。

| Elasticsearch | MySQL | 说明 |
|:---|:---|:---|
| **Index**（索引） | **Database** | 一类文档的集合，如 `product_index` 放商品数据 |
| **Type**（已废弃） | Table | 7.x 以后一个 Index 只有一个 `_doc` Type，8.x 彻底移除 |
| **Document**（文档） | **Row** | 一条 JSON 记录，如 `{"id": 1, "title": "iPhone 16"}` |
| **Field**（字段） | **Column** | 文档里的属性，如 `title`、`price` |
| **Mapping**（映射） | **Schema** | 定义字段类型和分词方式，如 `title` 是 `text` 类型 |
| **DSL**（查询） | **SQL** | JSON 格式的查询语言，如 `{"match": {"title": "手机"}}` |
| **Shard**（分片） | **分库分表** | 数据水平拆成多片，分布式存 |

### 一个更形象的比喻

```
MySQL 像 Excel 表格：
  - 一行一行存，按行号找很快
  - 要在"备注"列里搜索包含"重要"的行？→ 全表扫描，逐行看

ES   像书的**附录索引**：
  - 把书中所有"重要词汇"和它们出现的页码列出来
  - 查"乐观锁" → 索引说：第 23 页、第 47 页、第 89 页
  - 不用翻全书，直接翻指定页码
```

### 一条数据你很少用"ID"去搜，更多是用"关键词"搜 —— **这就是 ES 存在的根本意义。**

## 3. 倒排索引 —— ES 快的内核

> ⚠️ 面试官问"ES 为什么快"，只要你没提到**倒排索引**四个字，基本就凉了。这是 ES 面试的**第一考点**。

### 3.1 正排 vs 倒排

```
【正排索引】MySQL 的方式
  文档1 → "Java 八股文整理很累"
  文档2 → "Redis 八股文面试题汇总"
  文档3 → "Java 并发编程核心知识"
  文档4 → "Redis 缓存雪崩解决方案"

  找包含"Redis"的文档 → 必须把 4 篇都读一遍 → O(n)

【倒排索引】ES 的方式
  "Java"    → [文档1, 文档3]    位置: [0, 0]
  "Redis"   → [文档2, 文档4]    位置: [0, 0]
  "八股文"  → [文档1, 文档2]    位置: [1, 1]
  "并发"    → [文档3]            位置: [2]

  找包含"Redis"的文档 → 直接读"Redis"这一行 → O(1)
```

### 3.2 构建过程（三步走）

```
原始文档："Java 八股文整理"
         ↓
① 分词（Tokenization）
   ["Java", "八股文", "整理"]
         ↓
② 归一化（Normalization）
   英文转小写："java"（中文通常不需要）
   去除停用词："的"、"了"、"是"
   词干提取："running" → "run"
         ↓
③ 建立倒排表（Posting List）
   "java"   → [文档1]  词频:1  位置:[0]
   "八股文" → [文档1]  词频:1  位置:[1]
   "整理"   → [文档1]  词频:1  位置:[2]
```

### 3.3 倒排索引的内部结构（面试高分细节）

一个完整的倒排索引在磁盘上有三层结构：

```
┌──────────────────────────────────────────┐
│          Term Index（词项索引）              │
│  FST（有限状态转换器）压缩前缀树             │
│  存内存中，快速定位 Term 在哪个 Block        │
│  比如查 "redis" → 定位到 Block 17           │
├──────────────────────────────────────────┤
│          Term Dictionary（词项字典）         │
│  所有不重复的 Term 排好序                   │
│  二分查找定位具体 Term                       │
│  Block 17 里有 "redis" → 指向 Posting List │
├──────────────────────────────────────────┤
│          Posting List（倒排列表）            │
│  文档ID列表 + 词频(TF) + 位置 + 偏移量      │
│  压缩存储：FOR 编码 + Roaring Bitmap        │
│  支持跳表加速合并                            │
│  [Doc1: TF=3, Pos=[2,5,9]], [Doc3: TF=1...]│
└──────────────────────────────────────────┘
```

**逐层解释：**

| 结构 | 在哪 | 做什么 | 为什么快 |
|:---|:---|:---|:---|
| **Term Index** | 内存 | FST 前缀压缩树，从字符串前缀定位到 Term Dictionary 的 Block | O(前缀长度)，内存查找极快 |
| **Term Dictionary** | 磁盘（有内存缓存）| 排序的 Term 列表，二分查找 | O(log N)，且排序利于压缩 |
| **Posting List** | 磁盘 | 存储文档 ID、词频、位置 | 跳表 O(log N) 合并 + 压缩编码减少 IO |

### 3.4 两个单词解释什么是 FST

> **FST（Finite State Transducer）** = 共享前缀的极小化有向无环图。
>
> 想象你把 `["mon", "monday", "month"]` 三个词塞进一棵树：
> ```
>     m → o → n → (终结)
>            ├→ d → a → y → (终结)
>            └→ t → h → (终结)
> ```
> "mon" 共同前缀只存一次，极致压缩。查询时按字符走边即可，时间复杂度 = 词的长度。

### 3.5 Posting List 的压缩与加速

**为什么需要压缩？** Posting List 可能非常长（比如"中国"出现在几千万篇文档里），不压缩磁盘IO爆炸。

**FOR 编码（Frame of Reference）：**
```
原始：  [100, 115, 132, 188, 300]
差值：  [100, 15, 17, 56, 112]
         ↑ 差值比原值小得多，用更少 bit 存储
分块后每个 Block 选最小位宽 → 压缩率极高
```

**Roaring Bitmap：**
```
文档ID分桶：高16位确定桶号，低16位用不同策略存储
  - 稀疏桶 → 用数组存低16位
  - 稠密桶 → 用 Bitmap（bit 位对应文档 ID）
两个 Posting List 做交集 → 直接做 Bitmap AND，极快
```

**跳表（Skip List）：**
```
Posting List:  [1] → [3] → [7] → [12] → [15] → [20] → [25] → ...
                       ↓                 ↓
                 Skip → [12]  ←───── Skip → [25]  ←── 跳跃指针

合并两个 Posting List 时：
  不需要逐个对比，通过跳表大步跳跃，
  快速跳过不会命中的区间，O(log N) 合并
```

> **一句话总结倒排索引**：把"从文档里面找词"反过来，变成"从词找文档"。就像书的附录索引，查"Redis"直接指向所有提到这个词的页码，不用翻完整本书。

## 4. 分片与副本 —— 数据怎么存的

### 4.1 基本概念

| 概念 | 角色 | 数量 | 可变性 |
|:---|:---|:---|:---|
| **Primary Shard**（主分片） | 数据容器，处理写入 | 创建索引时定，**不可改** | 不可变 |
| **Replica Shard**（副本分片） | 主分片的复制品，处理读取 | 可动态调整 | 可变 |

### 4.2 比喻 —— 印刷厂

> 把 ES 集群想象成一个印刷厂，每出本书（建索引）：
>
> - **主分片 = 原版印刷稿**：印刷厂定好每页排多少字（分片数），开始印刷就不能再拆散了。
> - **副本分片 = 复印本**：原版需要多印几份给不同读者同时看（读扩展），原版坏了还有备份（高可用）。
>
> 这也是为什么"主分片数创建后不能改" —— 你把一本书的排版（路由）重新改掉，旧页码全报废了。

### 4.3 分片路由公式（面试必问）

```
shard = hash(_routing) % number_of_primary_shards
```

- `_routing` 默认就是文档的 `_id`
- `number_of_primary_shards` 是创建索引时指定的主分片数
- **为什么不能改？** 因为一旦改了分母，取模结果全部改变，所有旧的文档都路由到错误的分片上，数据全乱了。

> **类比**：假设有 5 个分片，文档 ID=18 → `18 % 5 = 3`，文档放在分片 3。
> 现在新增分片变成 7 个 → `18 % 7 = 4`，你去分片 4 找文档但它在分片 3 里，找不到了。
>
> 这就是**一致性哈希要解决的问题（一致性哈希在节点增删时只影响少量数据，而简单取模影响全部）**，但 ES 选择了简单取模，所以约束你：主分片数不能变。

### 4.4 分片规划建议

```
单分片不超过 30 GB
  原因：Lucene 段（Segment）合并时性能急剧下降
  单个分片是一个 Lucene 实例，30GB 是生产经验的阈值

分片数 ≈ Ceil(数据总量 / 30GB) + 2 预留

# 举例：预计存 100GB 数据
100 / 30 ≈ 3.3 → 4 个分片 + 2 → 6 个分片

# 注意：不是越多越好！
分片多 → 每个搜索请求分发更多节点 → 并发度高
       但 → 小文件多，Lucene 段合并开销大，Master 管理负担大
```

## 5. 写入流程 —— 一条数据进去经历了什么

### 5.1 完整写入时序图

```
Client 发送写入请求
        │
        ▼
┌─────────────────┐
│  协调节点        │  ① 接收请求，hash(_routing) 算出目标分片
│  (Coordinating)  │  ② 将请求转发到主分片所在的节点
└────────┬────────┘
         │ 转发
         ▼
┌─────────────────┐
│  主分片          │  ③ 先写 translog（WAL 预写日志）
│  (Primary)       │  ④ 再写内存 buffer（此时还没有写入 segment）
│                  │  ⑤ 返回 ack 给协调节点 → Client 收到成功
└────────┬────────┘
         │ 并行同步
         ▼
┌─────────────────┐
│  副本分片        │  ⑥ 副本同步写入 translog + buffer
│  (Replica)       │  ⑦ 返回确认给主分片
└─────────────────┘

══════════ 后台异步操作 ══════════

⑧ refresh（默认每 1s）
   内存 buffer → 生成新的 Segment（可搜索！）
   但 segment 仍在内存中（或部分写入磁盘缓存）

⑨ flush（translog 达到 512MB 或每 5s，较保守）
   segment → fsync 持久化到磁盘
   清空 translog（已持久化的部分）
```

### 5.2 三个关键概念 —— 用快递分拣中心比喻

> 把 ES 的写入流程想象成一个**快递分拣中心**：
>
> | ES 概念 | 快递类比 | 作用 |
> |:---|---:|:---|
> | **translog** | 快递底单（一式两份，永远保留原始记录） | 哪怕电脑死机，底单在手就能恢复 |
> | **buffer** | 临时分拣筐（包裹先扔筐里） | 先堆着，还没按小区分类（不可搜索） |
> | **refresh** | 每 1 秒把筐里的快递按小区摆上货架 | 快递员能按小区找到包裹了（可搜索！） |
> | **segment** | 按小区分类好的货架（不可变） | 摆好就不再动了，只读 |
> | **flush** | 把货架信息录入永久仓库笔记本（磁盘） | 断电也不怕，底单撕掉 |

### 5.3 为什么是"近实时"而不是实时？

```
写入 → buffer（此时不可搜索）
      ↓ 等 refresh（默认 1s）
     buffer → segment（此时可搜索）

这个 1 秒延迟 = 近实时（NRT）

你可以调成 refresh_interval = 100ms → 更实时但写入性能下降
你也可以调成 refresh_interval = 30s → 牺牲实时性换吞吐量
```

### 5.4 如何保证数据不丢？

```
translog 机制：
  - 每次写请求必先写 translog
  - translog 默认每个请求都 fsync 到磁盘（index.translog.durability = request）
  - 如果设成 async（异步），丢数据的窗口 = 5s（translog.sync_interval）

宕机恢复流程：
  上次 flush 之后的 translog → 顺序重放 → 恢复未持久化的数据

    类比：MySQL 的 redo log / binlog
    translog 是 ES 的 WAL（Write-Ahead Log）
    先记日志（translog）→ 再改数据（buffer/segment）→ 最后落盘（flush）
```

### 5.5 写入性能优化总结

| 优化项 | 操作 | 原理 |
|:---|:---|:---|
| 批量写入 | 用 `_bulk` API，一批 1000-5000 条 | 减少网络往返 + 减少 translog fsync 次数 |
| 增大 refresh_interval | 设为 30s 或 -1（关闭） | 减少生成 segment 的频率，降低 IO |
| 写入时关闭副本 | `number_of_replicas = 0`，写完再加回来 | 避免写入时实时同步副本，减少网络开销 |
| 自动生成 ID | 不要手动指定 `_id` | ES 内部自动生成 ID 更快，命中 Lucene 优化路径 |
| SSD | 硬件层面 | 随机 IO 性能远优于机械盘 |

## 6. 搜索流程 —— Query Then Fetch

### 6.1 为什么分两阶段？

> **核心原因：减少数据传输**。
>
> 不需要把所有分片上的文档全部传回协调节点，只需要传 `doc_id + _score` 就够了。
> 等确定了"最终要哪 10 篇"，再去它们所在的分片上拉取完整内容。

### 6.2 Query Phase（查询阶段）

```
用户请求: "搜索 Java 面试题，返回最相关的 10 篇"

协调节点:
  ① 广播请求到所有相关分片（主分片或副本分片，轮询调度）
  ② 每个分片本地搜索：
     - 查倒排索引 → 找到所有匹配的 doc_id
     - BM25 打分 → 算出每个文档的 _score
     - 在本分片内排序 → 取 Top 10（协调节点要的 size）
     - 返回 [(doc_1, 9.8), (doc_3, 7.5), (doc_5, 6.2), ...] 共 10 个
  ③ 协调节点收集所有分片的 Top 10：
     分片0: [9.8, 7.5, 6.2, ...]
     分片1: [8.3, 7.1, 5.9, ...]
     分片2: [9.2, 8.8, 6.1, ...]
           ↓ 全局排序
     [9.8, 9.2, 8.8, 8.3, 7.5, 7.1, 6.2, 6.1, 5.9, ...]
                       ↓ 取前 10
     最终 doc_id 列表: [doc_A, doc_C, doc_F, ...]
```

### 6.3 Fetch Phase（取数据阶段）

```
协调节点:
  ④ 手上有最终 10 个 doc_id
  ⑤ 知道每个 doc_id 在哪个分片上
  ⑥ 向对应分片发请求: "请给我 doc_A, doc_C 的完整 _source"
  ⑦ 各分片返回完整文档 JSON

协调节点 → 组装结果 → 返回给客户端
```

### 6.4 深度分页问题（为什么 from + size 10000 会慢）

```
假设 5 个分片，用户请求: from=10000, size=10

Query Phase:
  每个分片需要返回 from + size = 10010 条数据给协调节点
  5 个分片 × 10010 = 50050 条数据 → 协调节点排序取 [10000:10010]

  问题：
    ① 网络传输 50050 条 (doc_id + score) → 量大
    ② 协调节点内存排序 50050 条 → 内存开销大
    ③ from 每增加 1000，开销线性增加

解决方案:
  search_after: 基于上一页最后一条的排序值继续翻（推荐，实时游标）
  scroll:       创建快照，适合全量导出（非实时，维护成本高）
  限制 from:    直接限制 from 最大值（简单粗暴）
```

### 6.5 相关性打分 BM25

```
ES 5.0 之前: TF-IDF
ES 5.0 之后: BM25（默认，更优）

BM25 核心思想:
  - TF（词频）：一个词在文档中出现越多次，文档越相关
    → 但有上限，出现 1000 次和 2000 次差别不大
  - IDF（逆文档频率）：一个词在所有文档中越少见，越有区分度
    → "的"出现再多次也没有区分度
  - 字段长度归一化：同样的词出现在短文档里比长文档里更"重要"
    → 一篇 100 字的文章出现 3 次"Redis" vs 一篇 10000 字的文章出现 3 次

公式（简化理解）:
  score = Σ IDF(term) × (TF(term, doc) × (k1 + 1))
           / (TF(term, doc) + k1 × (1 - b + b × docLen/avgDocLen))

  其中 k1 和 b 是调参因子，控制 TF 饱和度和长度惩罚力度

一句话: BM25 比 TF-IDF 更"聪明"，不会因为文档长就给更高分。
```

## 7. 查询 DSL —— ES 的 SQL 怎么用

### 7.1 基础查询类型

| 查询类型 | SQL 类比 | 说明 |
|:---|:---|:---|
| **match** | `LIKE '%关键词%'` | 全文搜索，会分词，相关度打分 |
| **term** | `= '精确值'` | 精确匹配，不分词，不评分 |
| **terms** | `IN (v1, v2, v3)` | 多值匹配 |
| **range** | `BETWEEN / > / <` | 范围查询 |
| **exists** | `IS NOT NULL` | 判断字段是否存在 |
| **prefix** | `LIKE 'abc%'` | 前缀匹配 |
| **wildcard** | `LIKE '%abc%'` | 通配符（性能差） |
| **bool** | `WHERE ... AND/OR` | 组合查询 |

### 7.2 Bool 查询 —— 最重要的组合查询

```json
{
  "query": {
    "bool": {
      "must": [
        {"match": {"title": "Java"}}
      ],
      "filter": [
        {"term": {"status": "published"}},
        {"range": {"price": {"gte": 50, "lte": 200}}}
      ],
      "should": [
        {"match": {"tags": "高并发"}},
        {"match": {"tags": "面试"}}
      ],
      "must_not": [
        {"term": {"deleted": true}}
      ],
      "minimum_should_match": 1
    }
  }
}
```

| 子句 | 含义 | 是否影响评分 | 是否有缓存 |
|:---|:---|:---|:---|
| **must** | 必须满足（AND） | **是** | 否 |
| **filter** | 必须满足，但不评分 | **否** | **是** |
| **should** | 应该满足（OR），可调权重 | **是** | 否 |
| **must_not** | 必须不满足（NOT） | 否 | 是 |

### 7.3 Filter vs Query —— 面试高频

> 面试金句：**"能用 filter 尽量用 filter。"**

| | query | filter |
|:---|:---|:---|
| 核心能力 | 搜索，计算相关度 | 过滤，精确匹配 |
| 计算 _score | 是 | 否 |
| 结果缓存 | 不缓存 | **自动缓存**（bit set） |
| 适用场景 | 模糊搜索、关键词匹配 | 状态过滤、时间范围、精确值匹配 |
| 性能 | 慢 | 快 |

**filter 缓存原理**：filter 的结果是一个 bit set（位图），对于每个文档，匹配则是 1，不匹配是 0。下次同样的 filter 条件直接从缓存取 bit set，不用重新执行，速度极快。

**典型写法**：
```json
{
  "query": {
    "bool": {
      "must": {"match": {"title": "Java 面试"}},
      "filter": {"term": {"status": "published"}}
    }
  }
}
```
`match` 负责搜索打分，`filter` 负责过滤条件。各司其职。

## 8. 聚合分析 —— ES 的 GROUP BY

### 8.1 三种聚合类型

```
① Bucket Aggregation（桶聚合 = 分组）
   类似 SQL 的 GROUP BY
   按城市分组 | 按价格区间分组 | 按日期直方图分组

② Metric Aggregation（指标聚合 = 统计计算）
   类似 SQL 的 COUNT | SUM | AVG | MAX | MIN

③ Pipeline Aggregation（管道聚合 = 对聚合结果再聚合）
   类似 SQL 子查询 / 窗口函数
   月环比、移动平均

三者关系: Metric 计算 Bucket 里的数据，Pipeline 在 Bucket 之上再算
```

### 8.2 聚合示例

```json
{
  "size": 0,
  "aggs": {
    "by_category": {
      "terms": {"field": "category", "size": 10},
      "aggs": {
        "avg_price": {"avg": {"field": "price"}},
        "top_seller": {
          "top_hits": {"size": 1, "sort": [{"sales": "desc"}]}
        }
      }
    }
  }
}
```

对应 SQL：`SELECT category, AVG(price), (SELECT TOP 1 ... ORDER BY sales DESC) FROM products GROUP BY category`

### 8.3 聚合的性能要点

- 聚合**吃内存**：桶太多 = 内存爆炸
- **doc_values**（列式存储）是聚合快的基础（keyword/number/date 默认开启）
- 避免在高基数（cardinality，如用户 ID）字段上做 terms 聚合
- `size: 0` 表示不需要返回文档，只返回聚合结果，极大减少网络传输

## 9. 性能调优 —— 生产落地必知

### 9.1 写入优化

| 优化项 | 操作方法 | 效果 |
|:---|:---|:---|
| **批量写入** | `_bulk` API，每批 1000-5000 条 | 减少网络往返和 translog fsync |
| **增大 refresh_interval** | `index.refresh_interval = 30s` 或 `-1` | 减少 segment 生成频率 |
| **写入时关副本** | `number_of_replicas = 0`，完事加回 | 避免实时同步副本的延迟 |
| **自动生成 ID** | 不要手动指定 `_id` | 命中 Lucene 更快路径 |
| **调大 translog** | `translog.flush_threshold_size = 1GB` | 减少 flush 频率 |
| **SSD** | 硬件保障 | 随机 IO 性能飞跃 |

### 9.2 查询优化

| 优化项 | 原理 |
|:---|:---|
| **filter 代替 query** | filter 不算分 + 自动缓存 bit set |
| **减少返回字段** | `_source: ["title", "price"]` 只取需要的列 |
| **search_after 替代深度分页** | 避免 from + size 的巨大开销 |
| **避免前缀通配符** | `*keyword` 无法利用倒排索引，全表扫描 |
| **合理分片数** | 参考 30GB 阈值，分片过多合并开销大 |
| **冷热分离** | 详见 9.3 |

### 9.3 冷热数据分离（Hot-Warm Architecture）

```
场景: 日志系统，每天 500GB 新数据
  - 最近 3 天数据需要频繁搜索 → Hot 节点（SSD，高配）
  - 3-30 天偶尔查看 → Warm 节点（HDD，低配）
  - 30 天以上归档 → 删除或存离线

配置:
  Hot 节点标签: node.attr.data: hot
  Warm 节点标签: node.attr.data: warm

  index.routing.allocation.include.data: hot  （新索引先放热节点）
  定时任务 → 更新 allocation 规则 → 旧索引迁移到 warm 节点

好处:
  ① 热数据在 SSD，查询飞快
  ② 冷数据在 HDD，成本大幅降低
  ③ 减少集群整体负载
```

### 9.4 Force Merge（强制段合并）

```
背景: 每次 refresh 生成一个新 segment
      写了一天可能有几百个 segment
      搜索时要遍历所有 segment → 慢

force merge: 把多个小 segment 合并成一个大 segment
  好处: 搜索更快，减少文件句柄
  代价: 消耗 IO 和 CPU，低峰期操作
  操作: POST /my_index/_forcemerge?max_num_segments=1
  注意: 只对不再写的索引做（如按天切的旧索引）
```

### 9.5 堆内存设置（Heap Size）

```
规则：
  - 不超过物理内存的 50%
  - 不超过 32 GB

原因：
  超过 32GB 后，JVM 指针从 4 字节变成 8 字节（压缩指针失效）
  内存利用率下降，GC 压力增大

  剩余 50% 物理内存交给 Lucene 做 OS Cache
  Lucene 依赖 OS Cache 缓存倒排索引和 doc_values
  内存不足 → 磁盘 IO 暴增 → 性能雪崩

面试原话: "ES 堆内存建议设为 31GB（安全阈值），
          留一半给 OS 做文件缓存，这是官方推荐的最佳实践。"
```

### 9.6 禁用 Swap

```
ES 进程被 swap 出去 → 一次 GC 可能要几百毫秒甚至秒级
→ 集群认为该节点挂了 → 触发分片重分配 → 连锁雪崩

设置: bootstrap.memory_lock: true (锁定物理内存)
```

## 10. 集群健康 —— Green / Yellow / Red

### 10.1 三种状态的含义

| 颜色 | 含义 | 严重程度 | 说明 |
|:---|:---|:---|:---|
| **Green** | 所有主分片和副本分片都正常分配 | 健康 | 生产期望状态 |
| **Yellow** | 所有主分片正常，但部分副本分片未分配 | 警告 | **单节点集群必然 Yellow**（无副节点放副本） |
| **Red** | 至少有一个主分片丢失/未分配 | 事故 | 数据不完整，需要立即处理 |

### 10.2 Yellow 的常见场景

```
场景 1: 单节点集群
  索引有 1 个主分片 + 1 个副本 → 副本无处分配 → Yellow
  解决: 增加节点，或将副本数设为 0（开发环境）

场景 2: 节点下线
  副本所在的节点挂了 → 副本未分配 → Yellow
  如果恢复快，自动重新分配；如果恢复慢，副本会迁移到其他节点

场景 3: 磁盘空间不足
  磁盘 watermark 达到阈值（默认 85%） → 禁止分配分片 → Yellow
  解决: 扩容磁盘或清理数据
```

### 10.3 Red 的应急处理

```
Red 意味着有主分片丢失：
  - 持有该主分片的节点永久下线且没有副本
  - 某些数据将永久丢失

应急步骤:
  ① _cluster/health 查看哪些索引是 Red
  ② _cat/shards 找出未分配的分片
  ③ reroute 尝试强制分配（allocate_stale_primary）
     ⚠️ 这意味着接受数据丢失！仅限万不得已
  ④ 从备份恢复（如果有快照）
```

## 12. 面试满分话术

> 当面试官问"你了解 ES 吗？"时，你不需要等他一个一个问题问，直接开讲，掌握节奏。

### 开场白

"我对 ES 比较熟悉。Elasticsearch 是一个基于 Lucene 的分布式搜索引擎，底层核心是**倒排索引**——把'从文档找词'变成'从词找文档'，所以全文搜索非常快。它是**近实时**的，写入数据默认每 1 秒 refresh 后就能被搜索到。

我主要在项目中用它做**商品搜索**（或日志分析/站内搜索），和 MySQL 配合：MySQL 做主存储保证事务，ES 做搜索索引通过 Canal 监听 binlog 异步同步。

我来讲几个核心点：倒排索引、写入流程、查询优化，你看关注哪个方面？"

### 倒排索引话术

"倒排索引的核心思路是**反过来**：正排索引是知道文档ID找文档内容，倒排索引是知道内容中的词，找到包含这个词的所有文档ID。

就像一个**书的附录索引**——查'乐观锁'这个词，索引告诉你第23页、47页、89页都有，不需要把整本书翻一遍。

内部是三层结构：最底层是 **Posting List** 存储文档ID和位置，用 FOR 编码和 Roaring Bitmap 压缩；中层是排好序的 **Term Dictionary**，二分查找；最上层是内存中的 **Term Index**，用 FST 前缀压缩树做前缀定位，查找时间复杂度是 O(词长度)。三段加起来：内存快速定位 + 压缩减少 IO + 跳表加速合并，所以能做到毫秒级搜索。"

### 写入流程话术

"一条文档写入 ES 的路径是：**协调节点路由 → 主分片写 translog（WAL）→ 写内存 buffer → 同步副本 → 返回成功**。后台每 1 秒 refresh 把 buffer 刷成 segment 变得可搜索，再定期 flush 把 segment 持久化到磁盘并清空 translog。

这里 translog 是关键——类比 MySQL 的 redo log，每次写操作都先记日志再操作数据。默认 translog 是请求级别刷盘的，宕机后重放 translog 恢复数据，保证不丢失。"

### 总结收尾

"总的来说，ES 就是一个**分布式 + 倒排索引 + RESTful** 的搜索引擎。用倒排索引解决搜索快的问题，用分片和副本解决分布式和高可用的问题，用 translog 和 refresh 机制在性能和可靠性之间取平衡。定位是 MySQL 的搜索助手，不是替代品。"

## 13. 记忆口诀

### 核心概念口诀
> **"索引像库文像行，字段就像一列桩。映射是 Schema 作定义，DSL 就是查文章。"**

### 倒排索引口诀
> **"分词建表词找文，字典索引倒排存。FST 前缀速定位，跳表合并压缩神。"**

### 写入流程口诀
> **"路由分发先记账（translog），内存缓冲排队忙。每秒 refresh 变可查，定期 flush 落硬盘。"**

### 性能优化口诀
> **"批写入大间隔，少副本多 flush。Filter 不评分有缓存，search_after 翻页不迷路。热节点 SSD 冷归档，堆不过半三十二。"**

### 集群健康口诀
> **"Green 全好 Yellow 半（主好在、副本缺），Red 主丢数据完。单节点必 Yellow 态，加节点副本全。"**

## 14. 跨题串联

> 面试时主动做跨题串联，是"高级水平"的信号。

### ES <-> MySQL

| 串联点 | 展开 |
|:---|:---|
| **索引结构** | MySQL 用 B+ 树（正排，按主键快），ES 用倒排索引（按关键词快），各有擅长 |
| **WAL 机制** | MySQL 有 redo log，ES 有 translog，都是先记日志再操作数据，保证崩溃恢复 |
| **数据同步** | Canal 监听 MySQL binlog → MQ → ES 消费，是最经典的"异构数据同步"方案 |
| **事务** | MySQL 支持 ACID，ES 不支持事务，所以 ES 只能是"搜索索引"而不是"主存储" |
| **Schema 变更** | MySQL ALTER TABLE 可以在线 DDL，ES Mapping 只能加字段不能改类型 |

### ES <-> Redis

| 串联点 | 展开 |
|:---|:---|
| **缓存与搜索** | Redis 做热点数据缓存（按 Key 查快），ES 做全文搜索（按内容查快），两者配合 |
| **近实时 vs 实时** | ES 是近实时（1s refresh），Redis 是纯内存实时，场景不同 |
| **持久化** | Redis RDB/AOF 持久化，ES translog + flush 持久化，都是异步 + 日志 |
| **倒排索引的跨界使用** | Redis 的 RediSearch 模块也实现了倒排索引，证明倒排索引是搜索的通用方案 |
| **哨兵与副本** | Redis Sentinel 监控主从切换，ES Master 选举管理分片分配，都是高可用机制 |

### ES <-> Lucene

| 串联点 | 展开 |
|:---|:---|
| **关系** | ES 是 Lucene 的分布式封装，每个 Shard 底层就是一个 Lucene 实例 |
| **倒排索引** | 倒排索引是 Lucene 的核心，ES 继承了全部能力 |
| **段（Segment）** | Lucene 的 Segment 不可变机制 → ES 需要 refresh 新建 segment → 近实时的根源 |
| **段合并** | Lucene 的 force merge 是合并 segment 减少碎片，ES 原样暴露 |
| **打分算法** | BM25 是 Lucene 实现的，ES 直接使用，5.0 后为默认 |

### ES <-> 消息队列 (Kafka/RocketMQ)

| 串联点 | 展开 |
|:---|:---|
| **数据同步管道** | MySQL binlog → Canal → MQ → ES Consumer，经典的异步入库链路 |
| **削峰填谷** | 大数据量写入 ES 时，先入 MQ 再慢慢消费，避免打爆 ES 集群 |
| **顺序性** | 同 ID 的更新必须顺序处理，MQ 的分区有序性保证这一点 |

### 一句话串联模板

> 面试官问 ES 时，你可以这样说：
>
> "ES 和 MySQL 是互补关系：**MySQL 管精准读写和事务，ES 管模糊搜索和分析**，通过 Canal + MQ 异步同步。
> 底层原理上和 MySQL 都是 WAL 日志机制保证不丢数据，但索引结构完全不同——MySQL 是 B+ 树正排，ES 是倒排。
> 和 Redis 一样都面临集群高可用的挑战，ES 用副本分片 + Master 选举来解决。"
