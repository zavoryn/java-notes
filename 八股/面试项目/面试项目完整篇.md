## 一、项目一：电商数据分析系统

### 1.1 项目概述

> 为某电商平台搭建一套**高性能、高可用**的数据采集与分析系统，支撑大促期间日均亿级埋点数据和百万级订单数据的实时采集、清洗、入库与多维分析，最终面向运营和商家提供秒级数据看板。

一句话版本：**"从零搭建了一套日均亿级数据量的电商数据分析系统，核心解决数据不丢、查得快、大促不崩三个问题。"**

### 1.3 核心亮点与难点

#### 亮点一：高可用数据采集架构 —— 数据 0 丢失

##### 技术方案

```
前端埋点 SDK ──→ Nginx ──→ 网关 ──→ 采集服务 ──→ RabbitMQ ──→ 消费服务 ──→ MySQL
                  │                    │
                  │  ① 本地缓存重试     │
                  │  (写入失败时)       │
                  │                    │
                  │  ② 滚动日志扫描     │
                  │  (每5分钟扫描)      │
                  │                    │
                  └── ③ 定时全量校验 ──→ 数据平台原始日志
                     (每天凌晨3点)       (HDFS/Hive)
```

**多级兜底机制：**

- **第1级 — 本地缓存重试**：采集服务写入 RabbitMQ 失败时，先将数据写入本地 `LinkedBlockingQueue` + 内存缓存，定期（每 30s）重试投递，最多重试 5 次。
- **第2级 — 滚动日志扫描补偿**：采集服务同时将原始数据写入滚动日志（按小时切分），补偿任务每隔 5 分钟扫描上一个小时的日志文件，找出未成功投递的数据重新入队。
- **第3级 — 定时全量校验**：每天凌晨 3 点，将 RabbitMQ 投递记录与数据平台的原始日志（HDFS/Hive）做全量对账，按 `traceId` 比对，发现缺失数据后补推。

##### 面试话术

> "我们采用的是多级兜底策略来保证数据零丢失。第一级是实时层：写入 MQ 失败的请求先落到本地内存队列，30 秒后重试，最多重试 5 次，这能覆盖 99.9% 的瞬时故障。第二级是近线层：所有原始数据同时写入本地滚动日志，每 5 分钟有一个补偿任务扫描上一小时的日志，跟 MQ 的 ACK 记录做差值，把漏掉的数据重新投递。第三级是离线层：每天凌晨用 Hive 做全量对账，用 traceId 关联两边数据，发现差异后逐条补推。这三层兜底下来，我们在大促期间实测做到了数据零丢失。"

#### 亮点二：分布式数据一致性 —— RabbitMQ 最终一致性

##### 技术方案

```
Producer                          RabbitMQ                         Consumer
   │                                 │                                │
   │ ① 发送消息（持久化 delivery_mode=2）│                                │
   ├────────────────────────────────→│                                │
   │                                 │ ② 消息持久化到磁盘               │
   │                                 │                                │
   │                                 │ ③ 推送消息                      │
   │                                 ├───────────────────────────────→│
   │                                 │                            ④ 业务处理
   │                                 │                            ⑤ 手动 ACK (channel.basicAck)
   │                                 │  ←────────────────────────────│
   │                                 │                                │
   │                                 │ ⑥ 如果 ACK 超时/NACK           │
   │                                 │    → 重新投递 (requeue=true)    │
   │                                 │                                │
   │                                 │ ⑦ 多次重试失败                   │
   │                                 │    → 进入死信队列 (DLX)          │
   │                                 │        │                       │
   │                                 │        └──→ 死信消费者          │
   │                                 │              │                 │
   │                                 │              └──→ 临时表补偿     │
   │                                 │                   (定时任务轮询) │
```

##### 关键设计点

| 机制 | 作用 | 配置 |
|------|------|------|
| 消息持久化 | 防止 Broker 重启丢消息 | `delivery_mode=2`，队列 durable=true |
| 手动 ACK | 保证消费端处理完才确认 | `autoAck=false`，业务处理后在 finally 中 ack |
| 死信队列 (DLX) | 捕获反复消费失败的消息 | `x-dead-letter-exchange`，超时或达到最大重试次数后转入 |
| 临时表补偿 | 死信队列兜不住的场景 | 消费端将失败消息写入 `t_compensation` 表，定时任务每 10 分钟扫描重试 |
| 幂等性 | 防止重复消费 | Redis 记录 `msgId`，消费前先 `SETNX`，24 小时过期 |

##### 面试话术

> "我们使用 RabbitMQ 来实现分布式环境下的最终一致性。消息在发送端开启持久化，Broker 重启也不丢。消费端采用手动 ACK，在业务处理成功后才确认。如果消费一直失败，达到最大重试次数后消息会进入死信队列，由专门的死信消费者写入临时补偿表。定时任务每 10 分钟扫描补偿表，对可以重试的消息重新投递。整套机制保证了消息至少被消费一次，配合 Redis 的幂等去重，最终实现了数据的一致性。这里的核心权衡是：宁可在极端情况下重复处理，也绝不能丢数据。"

#### 亮点三：高并发查询与性能调优 —— 500ms 以内

##### 技术方案

**查询链路全流程优化：**

```
用户请求
   │
   ▼
┌──────────────┐
│ ① Redis 缓存  │ ← 命中率 95%+，直接返回（< 5ms）
└──────┬───────┘
       │ 未命中
       ▼
┌──────────────┐
│ ② 预聚合表    │ ← 按天/商品/用户群体预聚合，避免实时 JOIN
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ ③ MySQL 分表  │ ← 时间范围分表 + 联合索引，单表 < 500 万行
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ ④ 写回 Redis  │ ← Cache Aside 模式
└──────────────┘
```

**优化策略详解：**

| 优化维度 | 具体措施 | 效果 |
|---------|---------|------|
| 写入幂等 | `REPLACE INTO` + Redis `SETNX` 去重 | 避免重复数据导致统计失真 |
| 联合索引 | `(create_date, product_id, user_group)` 联合索引覆盖 90% 查询 | 查询从全表扫描变为索引覆盖 |
| 预聚合 | 每小时跑一次预聚合任务，按天/商品/用户群体维度汇总 | 看板查询无需实时 JOIN，直接查汇总表 |
| 时间范围分表 | 按月分表 `t_order_stats_202501` | 单表数据量控制在 500 万以内 |
| Redis 缓存 | 热点数据（当天/近7天）提前预热到 Redis | 缓存命中率 95%+，命中时 < 5ms 返回 |
| SQL 优化 | 避免 `SELECT *`，只查必要字段；用 `EXPLAIN` 分析执行计划 | 减少回表查询 |

##### 面试话术

> "性能优化我们做的是全链路。写入侧用 REPLACE INTO 做去重插入，配合 Redis 的 SETNX 防止消息重复消费导致数据脏写。查询侧做了四层优化：第一层是 Redis 缓存，热点数据命中率 95% 以上，命中时 5 毫秒以内就返回。第二层是预聚合，我们按天、按商品、按用户群体提前跑好汇总数据，看板查询直接查预聚合表，避免实时 JOIN 大表。第三层是 MySQL 分表，按月分表，单表控制在 500 万行以内，配合联合索引，即便是回源到 MySQL 也能在百毫秒级返回。第四层是用 EXPLAIN 逐条分析慢 SQL，持续优化执行计划。整体效果是：95% 的查询在 Redis 命中，5 毫秒返回；剩余的包括回表也能控制在 500 毫秒以内。"

#### 亮点四：系统稳定性与大促保障

##### 技术方案

**大促保障全景图：**

```
                    ┌──────────────────────────┐
                    │      核心链路资源倾斜      │
                    │  (独立线程池 + 独立MQ队列) │
                    └────────────┬─────────────┘
                                 │
         ┌───────────────────────┼───────────────────────┐
         │                       │                       │
         ▼                       ▼                       ▼
┌─────────────────┐   ┌─────────────────┐   ┌──────────────────────┐
│   业务降级       │   │   接口限流       │   │  削峰填谷             │
│                 │   │                 │   │                      │
│ 非核心功能关闭   │   │ 令牌桶算法      │   │ ConcurrentHashMap    │
│ 如：导出报表     │   │ 每接口单独配置   │   │ 本地缓存 + 线程池     │
│ 历史数据查询     │   │ 超过阈值返回     │   │ 定时批量写入 MQ      │
│                 │   │ 429 + 排队提示   │   │ 单条变批量           │
└─────────────────┘   └─────────────────┘   └──────────────────────┘
```

**稳定性措施详解：**

| 措施 | 实现 | 目的 |
|------|------|------|
| 核心链路资源倾斜 | 订单数据用独立的线程池(核心 20/最大 50)和独立 MQ 队列 | 埋点数据量再大也不影响订单处理 |
| 业务降级 | 大促期间自动关闭非核心功能：报表导出、历史数据明细查询等 | 把资源让给核心数据链路 |
| 接口限流 | 基于 Guava RateLimiter 的令牌桶算法，接入层配置限流阈值 | 防止突发流量打垮下游 |
| 削峰填谷 | ConcurrentHashMap 本地缓存数据，每 200ms 或积攒 500 条时批量刷入 MQ | 单条变批量，MQ 写入 QPS 降低 500 倍 |
| 监控告警 | Prometheus + Grafana，埋点延迟/消费积压/错误率等核心指标 | 3 分钟内发现问题 |

##### 面试话术

> "大促保障我们做了四件事。第一，核心链路资源倾斜：订单数据走独立的线程池和 MQ 队列，跟埋点流量物理隔离，埋点再大也不会拖垮订单。第二，业务降级：大促期间自动关闭报表导出、历史明细查询等非核心功能。第三，接口限流：基于令牌桶算法，每个接口配置限流阈值，超出的请求直接返回 429 让客户端排队重试。第四，削峰填谷：用 ConcurrentHashMap 在本地先攒一批数据，每攒够 500 条或 200 毫秒刷一次 MQ，这样 MQ 的写入 QPS 降低了两个数量级。这套机制经历了两次双十一验证，系统零故障。"

### 1.4 面试官追问准备

#### 追问 1：为什么选 RabbitMQ 而不是 Kafka？

**回答：** "这个项目的数据有两个特点：一是业务数据（订单）量级不大但对可靠性要求极高，需要单条确认和死信队列；二是埋点数据量大但对延迟容忍度高。RabbitMQ 的灵活性更高，手动 ACK + DLX 的机制天然适合我们的订单场景。Kafka 更适合纯日志类的海量流式场景，它的分区有序性、高吞吐是优势，但在这个项目里我们更需要的是消息级别的确认机制和灵活的路由。实际上我们后面埋点链路也补充了 Kafka，形成了 RabbitMQ 管业务、Kafka 管流量的双通道架构。"

#### 追问 2：REPLACE INTO 和 INSERT ... ON DUPLICATE KEY UPDATE 的区别？为什么选 REPLACE INTO？

**回答：** "两者都能处理主键/唯一键冲突的情况。核心区别：
- `REPLACE INTO`: 先尝试 INSERT，如果冲突则 **DELETE 旧行 + INSERT 新行**，自增 ID 会变。
- `ON DUPLICATE KEY UPDATE`: 冲突时执行 **UPDATE**，ID 不变。

我们选 REPLACE INTO 是因为业务上需要全量覆盖（比如订单状态变更，新数据就是最新状态，不需要保留旧数据），而且我们的表没有用自增 ID 做主键，用的是业务 ID，所以 ID 变化不影响。另外 REPLACE INTO 语法更简洁，不需要写 UPDATE 子句。"

#### 追问 3：死信队列的消息如果也处理失败怎么办？

**回答：** "死信队列的消费者把失败消息写入 MySQL 临时表 `t_compensation`，记录原始消息体、失败原因、重试次数。有一个 XXL-JOB 定时任务每 10 分钟扫描这张表，查询 `retry_count < 10` 且 `next_retry_time < now()` 的记录重新投递。超过 10 次后标记为人工处理，同时通过企业微信发告警通知开发介入。我们线上跑了一年多，超过 10 次重试的情况非常罕见，一般都是下游服务长时间的不可用导致的。"

#### 追问 4：为什么用 ConcurrentHashMap 做本地缓存而不是直接用 Redis？

**回答：** "这是削峰填谷场景的特定选择。如果用 Redis 做批量缓冲，每次写入都是一次网络 IO，在大促高并发下网络开销不可忽略——本来就是为了减少 MQ 写入频率，结果又多了一层网络开销，得不偿失。ConcurrentHashMap 是进程内的，写入是纯内存操作，基本零延迟。缺点也很明显：服务重启会丢数据。所以我们在设计上做了取舍——只有当攒不够 500 条时，200ms 定时器强制刷盘，最大丢失量就是 200ms 的数据。另外服务重启前会通过 JVM ShutdownHook 把本地缓存刷完再关闭。"

#### 追问 5：预聚合是怎么做的？用的什么调度？

**回答：** "预聚合用了 XXL-JOB 的分片广播模式。每个小时第 5 分钟触发，比如凌晨 1:05 聚合前一个小时(0:00-1:00)的数据。分片广播的好处是：如果聚合任务需要扫描多张分表，每个执行器各负责几个分表，并行处理。聚合结果写入 `t_stats_hour` 表（按小时粒度），然后还有一个每天的汇总任务把小时数据聚合成天维度的 `t_stats_day`。Redis 缓存的预热是在聚合任务执行完后，通过回调把当日和近 7 天的数据写入 Redis。"

#### 追问 6：限流用的什么方案？令牌桶和漏桶的区别？

**回答：** "用的是 Guava RateLimiter，它是令牌桶算法的实现。核心区别：
- 令牌桶：固定速率生产令牌，请求拿不到令牌就拒绝，**允许突发流量**（桶里有攒的令牌时可以一次消费多个）。
- 漏桶：请求先进桶，固定速率流出，**不允许突发**，多余的溢出。

我们用令牌桶是因为业务上允许一定的突发——比如大促开始时流量突然上来，令牌桶里提前攒的令牌可以消化一波，不至于一上来就限流。配置上我们按接口维度的实际压测 QPS 来设阈值，留了 20% 的 buffer。"

#### 追问 7：手动 ACK 在 finally 里做，如果 finally 也抛异常呢？

**回答：** "极端情况下 finally 也抛异常（比如连接已经断了），这时消息没有 ACK，Broker 会重新投递给其他消费者（或同一消费者重新消费）。这回到了我们幂等性的保护——Redis 的 SETNX 确保同一条 msgId 不会重复处理。从业务角度看，这条消息最多是暂时处理不了，但绝不会丢。等到消费端恢复正常后，它会被重新消费。"

#### 追问 8：Redis 缓存和 MySQL 数据怎么保证一致性？

**回答：** "采用的是 Cache Aside 模式。读的时候先查缓存，命中直接返回，未命中查 MySQL 然后写回缓存。写的时候先更新 MySQL，然后**删除缓存**（而不是更新缓存）。删缓存而非更新缓存的原因：更新缓存可能写了一个中间态进去；删除则强制下次读的时候从 MySQL 重新加载最新数据。还有一种情况：先删缓存后更新 MySQL，如果在删缓存和更新 MySQL 之间来了一个读请求，会读到旧数据并写回缓存。所以我们采用的是先更新 MySQL 再删缓存的顺序。配合缓存过期时间（热点 30 分钟，非热点 10 分钟）兜底，即便极端情况出现短暂不一致，过期后也会自动修复。"

#### 追问 9：按时间分表后，跨月查询怎么办？

**回答：** "用 MyBatis 拦截器实现分表路由。根据查询条件中的时间范围，动态计算出需要查询哪些表。如果是 `2025-01-15` 到 `2025-03-10` 的查询，路由到 `t_data_202501`、`t_data_202502`、`t_data_202503` 三张表，分别查询后在应用层合并结果。对于排序和分页的跨表查询，会先在各表内查询并排序，然后在应用层做归并排序取 TopN。这也是为什么我们提前做预聚合——看板查询大部分走预聚合表而不需要跨表。"

#### 追问 10：你说数据零丢失，真的能做到吗？如何验证？

**回答：** "我们说的零丢失是指业务层面——在兜底机制全覆盖的时间窗口内，没有数据永久丢失。验证方式分三层：
1. 实时的：监控 RabbitMQ 的 `publish` 和 `consumer_ack` 差值，差值 < 1000 视为正常。
2. 近线的：补偿任务扫描日志后记录 `compensated_count`，如果这个值突然变大说明有问题。
3. 离线的：全量对账后输出差异报告，`diff_rate` 需要 < 0.001%（万分之一）。实际运行下来我们 diff_rate 长期为 0。
诚实地讲，极端情况如机房断电导致内存中未来得及刷盘的数据确实会丢，但这种最多丢 200ms（削峰缓存窗口）+ 30s（重试窗口）的数据。"

#### 追问 11：项目的 QPS 和 RT 是多少？大促峰值多少？

### 1.5 遇到的坑和解决方案

| 问题 | 现象 | 根因 | 解决方案 | 教训 |
|------|------|------|---------|------|
| MQ 消息积压 | 大促后消费积压 200 万条，处理要 4 小时 | 消费端单线程处理，批量写入 MySQL 时每条都单独建连接 | 改为批量写入 + 连接池 + 增加消费者线程数(从 10 到 50) | 消费能力要提前压测，留足冗余 |
| REPLACE INTO 导致死锁 | 高并发下频繁死锁 | REPLACE INTO 对唯一索引有 Gap Lock，并发时容易死锁 | 改用 INSERT ON DUPLICATE KEY UPDATE + 降低事务隔离级别为 RC | REPLACE INTO 在 InnoDB 下坑很多，RC 隔离级别下更友好 |
| 数据倾斜 | 某几个商品的数据量是其他商品的 100 倍 | 爆款商品流量集中 | 分表 key 加上商品 ID，预聚合提前到每 5 分钟 | 热点分散比加机器更有效 |
| 预聚合任务超时 | 每小时聚合任务执行超过 50 分钟 | 商品维度聚合笛卡尔积过大 | JDBC 分页 + 游标读取，避免一次性加载全量数据到内存 | 大数据量操作永远用流式 |
| 内存溢出 (OOM) | 补偿任务扫描日志文件时 OOM | 一次性加载整个小时的大日志文件到内存 | 改为按行流式读取，`BufferedReader.readLine()` | 处理大文件永远不要一次性加载 |

##### 技术方案


##### 面试话术

> "用户的一键同步可能要同时推送到 3-5 个平台，如果串行调用的话，每个平台 API 耗时 2-3 秒，总耗时就 10-15 秒——用户等不了。
>
> 我们用了 CompletableFuture 做并行同步。主线程把每个平台的同步请求包装成一个 CompletableFuture，用自定义线程池（核心 20，最大 50）异步执行，最后用 `allOf` 等待所有任务完成，设一个总超时 30 秒。如果某个平台超时了，不影响其他平台的结果返回。
>
> 失败处理用的是指数退避重试：第 1 次失败等 1 秒重试，第 2 次等 2 秒，第 3 次等 4 秒……最多重试 3 次。3 次都失败后写入重试任务表，由定时任务后续处理。"

### 2.4 面试官追问准备

#### 追问 1：策略模式有几种？你这个用的是哪种？

**回答：** "策略模式常见有三种写法：
1. 简单策略：外部直接 new 具体策略。
2. 策略 + 工厂：工厂根据条件返回具体策略，调用方不感知具体策略。
3. 策略 + 枚举：把策略封装在枚举里。

我们用的是第二种——策略 + 工厂模式。Spring 启动时，每个策略类通过 `@Component` 注册为 Bean，并标注 `@PlatformType(Shopify)` 注解标识所属平台。工厂类在初始化时扫描所有带 `@PlatformType` 注解的 Bean，构建一个 `Map<PlatformEnum, PlatformStrategy>` 的映射。调用时通过 `platformEnum` 直接从 Map 取策略。这样做的好处是新增平台只需加一个类，不需要改工厂代码——完全符合开闭原则。"

#### 追问 2：为什么不直接用 if-else？

**回答：** "当然可以，但 if-else 有两个致命问题：第一，每加一个平台就要改核心代码，违反开闭原则，容易引入回归 bug；第二，策略类的方法可能有几百行甚至上千行（比如 Amazon 的 SP-API 调用逻辑很复杂），全部堆在一个类里维护成本太高。策略模式把每个平台的逻辑隔离在自己的类里，职责单一，好测试、好维护。"

#### 追问 3：策略 + 工厂和策略 + 枚举有什么优缺点？

**回答：** "策略 + 枚举写起来最简单，但策略被限制在固定的枚举值里，不利于扩展（要改枚举定义）。而且如果策略类很重（需要注入 Spring Bean、连接数据库、调用外部 API），枚举不太方便管理这些依赖。我们用的策略 + 工厂 + Spring，每个策略独立成 Bean，可以注入依赖、AOP 切面拦截，更灵活。"

#### 追问 4：多平台并行同步中，如果 Shopify 同步完了，Amazon 还在同步，能不能先返回 Shopify 的结果？

**回答：** "可以的。两种方案：
方案一（简单）：不用 allOf 等待全部，而是每个 future 都 `.thenAccept(resultHandler)`，只要某个平台完成就立即回调处理，前端可以轮询或者用 WebSocket 推结果。
方案二（推荐）：用 CompletableFuture 的 `anyOf` 和 `allOf` 组合——每个 future 完成后往 Redis 写一个状态，前端轮询这个状态接口。当所有都完成（或超时）后汇总返回。
我们实际用的是方案二，前端 1 秒轮询一次，体验上是'逐个平台点亮完成状态'的效果。"

#### 追问 5：标准模型怎么设计的？如何做到兼容所有平台？

**回答：** "标准模型的设计原则是'取各平台字段的**超集**'——把主流平台（Shopify、Amazon、eBay）的所有字段都囊括进来，同时做好字段分类：
- **通用字段**：title、price、stock、images 等所有平台都有。
- **扩展字段**：用 JSON 字段 `extra_fields` 存储平台特有的数据，比如 Amazon 的 ASIN、FBA 信息、BulletPoints 等。
- **映射配置**：每加一个平台时，在 `t_field_mapping` 表里维护标准字段到平台字段的映射关系。

这样标准模型的表结构相对稳定，不会每接一个平台就加一堆字段。扩展性靠 JSON 列和映射表解决。"

#### 追问 6：如何处理请求频率限制（Rate Limit）？

**回答：** "各个平台都有 API 调用频率限制，比如 Shopify 是每秒 2 次，Amazon SP-API 根据销售账号和接口不同有不同配额。我们的处理方案：
1. **平台级限流器**：每个策略类内部维护一个 Guava RateLimiter，按平台官方文档配置 QPS。
2. **请求前判断**：每次调用 API 前 `rateLimiter.acquire()`，如果没令牌就阻塞等待。
3. **错误码兜底**：即便限流，有时平台还是会返回 429。我们捕获 429 后解析响应头里的 `Retry-After`，按指示时间等待后重试。
4. **公平调度**：多任务共享限流器，保证不会某个账号的同步任务霸占所有配额。"

#### 追问 7：Redis 怎么管理平台 Token？

**回答：** "OAuth2 的 access_token 有过期时间（Shopify 是永久有效的但需要 shop 验证，Amazon 的 refresh_token 用来换 access_token）。我们把 token 存在 Redis 里，Key 是 `platform_token:{platform}:{shop_id}`，Value 存储 token + 过期时间。每次调用 API 前先从 Redis 取，如果快过期了（提前 10 分钟），触发异步刷新，避免在请求中刷新导致延迟。"

#### 追问 8：同步过程中如果部分成功部分失败怎么办？

**回答：** "这个场景非常常见——比如一个商品推送到 3 个平台，2 个成功 1 个失败。我们的处理：
1. 不会回滚已成功的平台（因为已经对外暴露了数据，回滚反而引起混乱）。
2. 失败的商品记录到 `t_sync_failed` 表，关联到同步任务 ID。
3. 用户在界面上能看到 '部分成功' 状态，点击失败的平台可以看失败原因。
4. 用户可以手动重试，也可以等系统的定时重试（每 10 分钟）。"

#### 追问 9：如果目标平台返回的数据和我们标准模型差异特别大怎么办？

**回答：** "比如 Amazon 的商品变体（variations）概念和绝大多数平台都不一样——它以 ASIN 为唯一标识，父 ASIN 下挂一堆子 ASIN，每个子 ASIN 可以有不同的颜色/尺寸/价格。我们的策略是在策略类层面做**适配转换**，而不是在标准模型里迁就：
1. Amazon 策略类的 `toPlatformModel` 方法里，把标准模型的 variants 结构打散，构建成 Amazon 要求的父子关系。
2. `toStandardModel` 反过来聚合。
3. 策略类内部的适配逻辑可以很复杂，但对外暴露的接口（标准模型）保持简洁统一。
这就是策略模式的价值——把复杂性隔离在各自的策略类里。"

#### 追问 10：系统怎么测试的？Mock 外部 API？

**回答：** "测试分几层：
1. 单元测试：Mock 外部 API 调用，用 Mockito，验证策略类的映射逻辑是否正确。
2. 集成测试：在策略类里注入一个 `@Profile("test")` 的 API Client 实现，这个实现返回预先构造好的假数据，验证从模型转换到结果处理的完整链路。
3. 沙箱测试：Shopify 和 Amazon 都提供 Sandbox/测试环境，我们上线前在沙箱环境跑完整流程。

单元测试覆盖了核心的模型转换逻辑，这部分是最容易出 bug 的（字段映射错误、货币转换问题等）。"

### 2.5 遇到的坑和解决方案

| 问题 | 现象 | 根因 | 解决方案 | 教训 |
|------|------|------|---------|------|
| Amazon API 随机 403 | 调用 Amazon SP-API 偶尔返回 403，重试有时恢复 | Amazon 的 STS Token 过期机制比较特殊 | 加了一个 Token 有效期的前置校验，在请求前 5 分钟就刷新 | 境外平台的 Token 机制差异大，要仔细读文档 |
| 商品图片同步慢 | Shopify 上传图片耗时 5-10 秒 | 原图 10MB+ 上传到平台 CDN 慢 | 同步前先压缩到 2MB 以内，并且上传走异步（先返回商品信息，图片后台慢慢传） | 大文件上传必须异步化 |
| 价格精度丢失 | eBay 同步价格时 99.99 变成 99.98 | 使用 float 做货币转换时精度丢失 | 全部改用 BigDecimal，货币转换统一用 BigDecimal.multiply(exchangeRate) | 金融/价格计算必须用 BigDecimal |
| 订单状态不一致 | 用户在平台上改状态后，我们的系统显示旧状态 | 只做了单向同步（只推不拉） | 加了定时状态拉取任务，每 5 分钟拉取各平台最新订单状态，与本地对账 | 订单状态同步必须是双向的 |
| 策略 Bean 重复注册 | 启动时 Spring 报 Bean 冲突 | 两个平台用了同一个 Bean 名称 | 给每个策略类加 @Qualifier 和明确命名规则：`{platform}SyncStrategy` | Spring Bean 命名规范化 |

### 2.8 面试满分话术（项目二）

> **30 秒精讲版：**
> "我负责了一个多平台同步系统，用策略模式解决了多个电商平台商品和订单同步的问题。核心设计是：定义统一的标准商品/订单模型，每个平台实现自己的策略类来负责标准模型到平台私有模型的转换。通过策略工厂来管理和获取具体的策略实例。接新平台只需要新增一个策略类，不用改任何核心代码。同步支持多平台并行，用 CompletableFuture 把 3 个平台的串行 15 秒降到并行 5 秒。"

> **1~2 分钟详细版：**
> "这个项目的背景是：我们做跨境电商 SaaS，商家需要把商品和订单同步到 Shopify、Amazon、eBay 等多个平台。每个平台的 API 差异巨大——Shopify 用 GraphQL、Amazon 是 SP-API、eBay 又不一样。不能每接一个平台就写一套新流程，维护成本太高。
>
> 所以我用策略模式做了架构设计。核心有两个抽象：
> 第一是**标准模型**——我们内部统一用一个 StandardProduct 和 StandardOrder 类，它包含了所有主流平台字段的超集。字段设计分通用字段和扩展字段（JSON），新平台的特有字段塞扩展 JSON 里。
> 第二是**平台策略**——每个平台实现一个策略类，继承自一个抽象基类，基类用模板方法定义了固定流程：模型转换 → 字段校验 → API 调用 → 结果处理。策略类只需要实现模型转换和 API 客户端构建，其他都是继承的。
>
> 工厂用一个 Map 管理所有策略 Bean，通过枚举取策略。多平台同步的时候用 CompletableFuture 并行调用，总耗时 = max(各平台耗时)，而不是 sum。失败用指数退避重试 3 次，还失败就记录到重试表等后续处理。
>
> 这套设计的效果很直接：新增一个平台从 2 周缩减到 3 天，核心代码一次都没动过——完全对扩展开放、对修改关闭。接入了 5 个平台，运行很稳定。"

## 五、串联八股技巧

### 5.1 项目一串联表

| 项目中的设计/决策 | 自然引出的八股知识点 | 串词示例 |
|------------------|-------------------|---------|
| RabbitMQ 选型 | 消息队列对比 (RabbitMQ vs Kafka vs RocketMQ) | "为什么选 RabbitMQ？因为我们的订单场景需要单条确认，Kafka 更适合日志场景……" |
| 手动 ACK | 消息可靠性投递、ACK/NACK 机制 | "我们用了手动 ACK，因为自动 ACK 是消费者收到消息就确认，如果业务没处理完就挂了消息就丢了……" |
| 死信队列 | 死信队列的触发条件和使用场景 | "死信队列我们配了三种触发：消息过期、队列满了、被拒绝且不重新入队……" |
| Redis 幂等 (SETNX) | Redis 数据结构、分布式锁、setnx 原理 | "用 SETNX 做幂等是因为它是原子操作，适合做'第一次才算数'的场景……" |
| REPLACE INTO | MySQL 的锁机制、Gap Lock、隔离级别 | "REPLACE INTO 在 RR 隔离级别下会加 Gap Lock，高并发时容易死锁，我们把隔离级别降到了 RC……" |
| 联合索引 | 索引最左前缀原则、覆盖索引、回表 | "联合索引我们按 create_date, product_id, user_group 建，因为 90% 的查询都带日期条件……" |
| Redis 缓存 | 缓存穿透/击穿/雪崩、缓存一致性 | "缓存击穿我们用了互斥锁方案，热点 key 过期时只有一个线程去查 DB……" |

| 分库分表 | 分表策略、跨表查询、sharding | "按月分表的好处是数据天然按时间隔离，冷热数据分开管理……" |

### 5.3 串词技巧总结

1. **从"我们做了什么"过渡到"为什么这么做"**：
   - 差："我们用了 Redis 缓存"（只说了 what）
   - 好："查询量大，MySQL 扛不住，我们加了 Redis 缓存。这里要注意缓存穿透的问题，我们的方案是……"（说了 what + why + how）

2. **主动对比技术选型**：
   - "我们选 RabbitMQ 而不是 Kafka，是因为……"（展示你了解多个同类技术并能做出合理判断）

3. **顺带提到坑和优化**：
   - "一开始用 REPLACE INTO 频繁死锁，排查后发现是 RR 隔离级别下的 Gap Lock 导致的……"（展示你踩过坑并且能定位到原理层面）

4. **用八股术语自然包裹**：
   - 不说"把数据先存着再批量写"，而说"用 ConcurrentHashMap 做本地缓存异步批量写入来削峰填谷"
   - 不说"加了个去重的"，而说"用 Redis SETNX 做幂等性保证"

5. **引导面试官问你想答的问题**：
   - 在介绍项目时轻描淡写提到一个技术点但不说细节："这里用了一个有意思的设计……"，面试官大概率会追问
