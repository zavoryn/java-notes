# 内存管理

> [!tip] 核心速记
> **"FIFO 简单不顶用，OPT 最优做不到，LRU 最近没人碰，Clock 转圈最实用。"**
> 虚拟内存 = 图书馆目录索引，书架只有最常用的书，找不到就去仓库搬（缺页中断）。

---

## 1. 虚拟内存 → 物理内存映射

> **核心比喻**：**"图书馆书架的目录索引"**
>
> 你把全世界的书（虚拟内存）都列在目录卡片上，但实际上图书馆的书架（物理内存）只放了最常用的 1000 本。
> - 当你去找一本书（访问内存地址），先查目录（页表）
> - 如果目录上写着"在书架上"（在物理内存中），直接去拿
> - 如果目录上写着"在仓库"（在磁盘上），就去仓库搬过来放到书架上（缺页中断）
> - 你感觉好像全世界的书都在书架上——这就是虚拟内存的魔法

### 1.1 核心概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **虚拟地址** | 程序看到的地址（每个进程 0x0000...开始） | 你心中的"书的编号" |
| **物理地址** | 真实的RAM地址 | 书实际在书架第几排第几格 |
| **页 (Page)** | 虚拟内存的固定大小块（通常4KB） | 每本书都是固定大小 |
| **页框 (Frame)** | 物理内存的固定大小块（4KB） | 书架上的每个格子 |
| **页表 (Page Table)** | 虚拟页→物理页框的映射表 | 目录索引卡片 |
| **TLB (Translation Lookaside Buffer)** | 页表的高速缓存（MMU内部） | 手上常翻的"常用卡片" |

### 1.2 多级页表

为什么需要多级页表？如果32位地址空间、4KB页，页表有 2^20 = 1M 项，每项4字节 = 4MB。**每个进程4MB页表**，太浪费！

**二级页表**：只对实际使用的虚拟地址范围建立二级页表，未使用的区域只需一个"不存在"的一级页表项。

Linux 使用**四级页表**（PGD → PUD → PMD → PTE），64位地址空间需要更多层级。

## 2. 缺页中断流程

```
程序访问地址 → MMU查TLB(TLB命中?直接翻译:查页表)
  → 页表项有效位=1?直接翻译:触发缺页中断
  → 缺页中断: 从磁盘加载页面到物理内存 → 更新页表 → 重试指令
```

> [!note] 写时复制（Copy-on-Write）
> fork() 创建子进程时，父子共享同一物理页，只有当某一方进行写操作时才真正复制该页。
> 这就像图书馆里两个人借同一本书——不做副本，只有有人要写笔记时才复印一本新书给他。

## 3. 页面置换算法

> 比喻：图书馆书架只有 100 个格子（物理内存），但有 10000 本书（虚拟内存）。
> 当有新书要放进来、书架满了，就要决定把哪本旧书撤下去（换出到磁盘）。

| 算法 | 原理 | 优点 | 缺点 | 类比 |
|------|------|------|------|------|
| **OPT（最佳置换）** | 换掉未来最长时间不再访问的页 | 缺页率最低 | 无法实现（需要预知未来） | 知道哪本书以后不会再借——神仙才能做到 |
| **FIFO（先进先出）** | 换掉最早进入的那页 | 简单 | 可能换出频繁使用的页（Belady异常） | 最早进书架的先扔——不科学 |
| **LRU（最近最久未使用）** | 换掉最长时间没被访问的页 | 接近OPT | 实现成本高 | 最久没被借过的书——比较合理 |
| **LFU（最不常用）** | 换掉访问次数最少的那页 | 适合热点数据 | 新页容易被换出 | 被借次数最少的书——但有新书"冷启动"问题 |
| **Clock（时钟算法）** | 循环扫描，访问位=0就换出，=1就清零 | LRU的近似，实践常用 | 比LRU稍差 | 管理员转圈检查，没被翻过的就撤 |

### 3.1 Clock 算法详解（最实际）

```
页面组成环形链表 + 访问位(0/1)
指针循环移动：
  如果访问位==1 → 置0，继续走
  如果访问位==0 → 换出这一页
→ LRU 的性能，FIFO 的简单实现
```

**和缓存的关联**：Redis 的 volatile-lru / allkeys-lru 淘汰策略就是 LRU 思想。MySQL Buffer Pool 也用了 LRU 变种（冷热分离）。

## 4. 内存碎片

### 4.1 内部碎片 vs 外部碎片

| 类型 | 定义 | 类比 | 例子 |
|------|------|------|------|
| **内部碎片** | 分配给进程的内存大于实际需求，多余部分浪费在分配单元内部 | 你订了个10人餐桌只来了6人，4个座位空着 | 页式管理中，4KB页分配给只需3KB的数据，浪费1KB |
| **外部碎片** | 内存被频繁分配/释放后，剩余空间总量足够但无法合并成一块连续大区域 | 街上停车位散落各处，总数够但找不到连续3个空位 | 进程需要10MB连续空间，但剩余的10MB分散在各处 |

### 4.2 如何解决碎片？

| 方法 | 解决的碎片 | 说明 |
|------|-----------|------|
| **分页管理** | 外部碎片 | 页式管理消除了外部碎片（每个页框4KB，大小固定），但引入内部碎片 |
| **分段管理** | 内部碎片 | 段大小可变，按逻辑分段分配，减少内部碎片，但可能产生外部碎片 |
| **段页式管理** | 综合 | 先分段再分页，兼顾逻辑和物理 |
| **内存紧凑（Compaction）** | 外部碎片 | 移动进程使空闲区域合并，但开销大（需要暂停进程、更新地址） |
| **伙伴系统（Buddy System）** | 外部碎片 | 见下文详解 |
| **Slab 分配器** | 内部碎片 | 针对小对象的缓存分配，减少内部碎片 |

> [!warning] 面试易错点
> 分页管理解决了外部碎片问题，但引入了内部碎片。分段管理解决了内部碎片问题，但引入了外部碎片。

## 5. swap 机制

### 5.1 什么是 swap？

swap = 当物理内存不足时，OS将部分不常用的内存页写到磁盘的swap分区/文件中，腾出物理内存给需要的进程。

```
物理内存不足 → OS选择不活跃的页面 → 写到磁盘swap区 → 释放物理页框
  → 进程访问被swap出去的页面 → 缺页中断 → 从swap区读回物理内存
```

### 5.2 为什么 Java 服务要关 swap？

> [!warning] 重要！Java服务必须关闭swap

| 原因 | 说明 |
|------|------|
| **GC延迟飙升** | JVM的GC需要扫描整个堆，如果部分堆页被swap到磁盘，GC读这些页要走磁盘IO，延迟从毫秒级暴涨到秒级 |
| **STW时间暴增** | Full GC的Stop-The-World期间，所有线程暂停，但如果堆页在swap上，恢复时间也受影响 |
| **OOM Killer误杀** | Linux OOM Killer根据进程的"OOM score"决定杀谁，开启swap后JVM进程的score可能变高，容易被误杀 |
| **性能不确定性** | swap导致内存访问延迟从ns级变成ms级，完全不可预测 |

**关闭swap命令**：
```bash
swapoff -a   # 关闭所有swap分区
# 永久关闭：编辑 /etc/fstab，注释掉swap行
```

## 6. 大页内存（Huge Pages）

### 6.1 什么是大页？

传统页大小 = 4KB。大页 = 更大的页大小（x86-64 支持 2MB 和 1GB）。

| 页大小 | 页表项数量（2TB地址空间） | TLB覆盖范围 |
|--------|--------------------------|-------------|
| 4KB | ~500M项，页表 ~2GB | TLB 64项覆盖 256KB |
| 2MB | ~1M项，页表 ~4MB | TLB 64项覆盖 128MB |
| 1GB | ~2K项，页表 ~8KB | TLB 64项覆盖 64GB |

### 6.2 为什么使用大页？

- **减少TLB miss**：同样的TLB项数，大页覆盖的内存范围大500倍，TLB miss率大幅下降
- **减少页表大小**：页表从几GB降到几MB，节省内存
- **减少缺页中断**：大页一次性映射更大区域，缺页次数减少

### 6.3 Java与大页

```bash
# JVM启用大页
java -XX:+UseLargePages -XX:LargePageSizeInBytes=2m ...
```
- **Redis**：默认使用大页（`CONFIG SET hugepage on`）
- **MySQL**：`large-pages` 参数启用大页
- **Kafka**：在Linux上建议配置大页

> [!warning] 大页的缺点
> - 内部碎片更大：2MB大页即使只用了1KB也占2MB
> - 需要预分配连续的物理内存（2MB必须连续的2MB物理页框）
> - 配置复杂，需要系统管理员预配置

## 7. mmap 详解

### 7.1 什么是 mmap？

mmap = Memory Mapping，将文件映射到进程的虚拟地址空间，对映射区域的读写等同于对文件的读写。

```
mmap调用 → 内核将文件的页映射到进程的虚拟地址空间
  → 进程访问映射区域 → TLB/页表翻译 → 如果物理页不在内存 → 缺页中断 → 从磁盘加载
  → 修改映射区域 → 页面被标记为脏页 → 后续由内核写回磁盘（或手动msync）
```

### 7.2 mmap vs 普通 IO 对比

| 维度 | 普通 IO（read/write） | mmap |
|------|----------------------|------|
| **数据拷贝** | 磁盘 → 内核缓冲区 → 用户缓冲区（2次拷贝） | 磁盘 → 直接映射到进程地址空间（0次额外拷贝） |
| **系统调用** | 每次 read/write 都是一次系统调用 | 只需 mmap 一次，之后像访问内存一样无系统调用 |
| **适用场景** | 需要修改数据、随机读写少量数据 | 大文件顺序读、共享内存、不修改数据的传输 |
| **缺点** | 频繁系统调用开销大 | 映射区域占虚拟地址空间；修改后需要msync同步 |

### 7.3 Java中的 mmap

```java
// Java NIO的 MappedByteBuffer 就是 mmap
FileChannel channel = new FileInputStream("big.txt").getChannel();
MappedByteBuffer buffer = channel.map(FileChannel.MapMode.READ_ONLY, 0, channel.size());
// 直接像数组一样读取，底层是mmap
byte b = buffer.get(1024);
```

- **RocketMQ**：使用 mmap 写 CommitLog（`MappedByteBuffer`）
- **Kafka**：不使用 mmap 写日志（因为Kafka追求顺序写的吞吐量，mmap的脏页回写不可控），而是用普通FileChannel + PageCache

> [!warning] mmap 的坑
> - mmap 映射的区域大小受虚拟地址空间限制（32位进程只有3GB可用）
> - 大量小文件 mmap 会浪费页表空间（每个文件至少一个4KB页）
> - mmap 写入后如果不 msync，数据可能丢失（依赖内核回写时机）

## 8. 内存分配算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **First Fit（首次适应）** | 从低地址开始找，找到第一个够大的空闲块就分配 | 快（O(n)），低地址碎片利用率高 | 低地址区域产生大量小碎片 |
| **Best Fit（最佳适应）** | 找到最小的能满足需求的空闲块分配 | 碎片最小 | 搜索慢（O(n)遍历全部），产生大量极小碎片无法利用 |
| **Buddy System（伙伴系统）** | 见下文详解 | 合并快速，无外部碎片 | 内部碎片（只能分配2的幂次大小） |

### 8.1 Buddy System 详解

```
伙伴系统核心规则：
  1. 内存块大小必须是 2^k（k = 0,1,2,3...）
  2. 两个大小相同、地址连续的块称为"伙伴"
  3. 分配：如果当前没有合适大小的块，将大块一分为二（分裂），直到得到所需大小
  4. 释放：如果伙伴也空闲，合并为更大块（合并），递归向上合并

举例：请求分配 65KB
  → 找最小能容纳的块 128KB (2^7)
  → 如果没有128KB，分裂256KB → 两个128KB → 分配一个128KB
  → 实际只需要65KB，浪费63KB（内部碎片）

伙伴合并举例：
  释放 128KB块A → 检查伙伴128KB块B是否空闲？
    → B空闲 → 合并为256KB → 检查256KB的伙伴是否空闲？
      → 空闲 → 合并为512KB → 递归继续
```

> [!note] Linux 的伙伴系统
> Linux 内核使用伙伴系统管理物理页框。页框分配从 order 0（1页 = 4KB）到 order 10（1024页 = 4MB）。
> - `cat /proc/buddyinfo` 可以查看当前各order的空闲块数量
> - 伙伴系统解决了外部碎片，但内部碎片最大可达请求大小的一半

## 9. 与 Java 后端的关联

| OS概念 | Java关联 |
|--------|----------|
| 虚拟内存 | JVM 的 `-Xms`、`-Xmx` 管理的是虚拟内存 |
| 缺页中断 | 堆内存首次 `new 对象` 时，OS分配物理内存，类似缺页 |
| 页面置换 | GC后回收的堆空间是否返回OS取决于 `-XX:MinHeapFreeRatio` |
| mmap | `MappedByteBuffer` 直接利用OS虚拟内存机制 |
| swap | Java服务必须关闭swap，否则GC延迟飙升 |
| 大页 | JVM `-XX:+UseLargePages` 减少TLB miss |
| Buddy System | Linux内核用伙伴系统管理物理内存，JVM的TLAB是Buddy思想的简化 |

## 面试题速查

> [!example] 本模块高频面试题

1. **虚拟内存是什么？页表、TLB、缺页中断怎么工作的？**
   - 虚拟内存=进程看到的空间 vs 物理内存=真实RAM；页表做映射，TLB缓存映射，缺页中断处理不在内存的页

2. **页面置换算法有哪些？LRU是什么原理？**
   - OPT/FIFO/LRU/LFU/Clock；LRU换掉最久未用的页

3. **内存碎片：内部碎片vs外部碎片？如何解决？**
   - 内部=分配单元内的浪费，外部=空闲空间无法合并；分页解决外部、分段解决内部

4. **swap机制？为什么Java服务要关swap？**
   - swap=内存不足时写磁盘；Java关swap因为GC延迟飙升、OOM Killer误杀

5. **大页内存（Huge Pages）是什么？为什么能减少TLB miss？**
   - 2MB/1GB页；同样的TLB项覆盖更大范围，miss率大幅下降

6. **mmap原理？与普通IO对比？**
   - mmap将文件映射到虚拟地址空间，0次额外拷贝；普通IO需要2次拷贝+系统调用

7. **内存分配算法：First Fit/Best Fit/Buddy System？**
   - First Fit快但碎片多，Best Fit碎片最小但慢，Buddy无外部碎片但有内部碎片
