# 进程与线程

> [!tip] 核心速记
> **"进程管资源，线程做调度，进程独立线程共享，崩溃波及不一样。"**
> 进程 = 资源分配最小单位，线程 = CPU调度最小单位。

---

## 1. 进程 vs 线程

### 1.1 一句话定义

| 概念 | 定义 | 生活类比 |
|------|------|----------|
| **进程** | 资源分配的最小单位 | 一家公司（有自己的营业执照、银行账户、办公楼） |
| **线程** | CPU 调度的最小单位 | 公司里的员工（共享公司资源，每人干自己的活） |

### 1.2 为什么要有线程？

> 比喻：一家餐厅（进程）只有老板一个人（单线程）——他要迎客、点单、炒菜、结账。来了 10 桌客人，老板忙到崩溃。
>
> 后来老板雇了厨师、服务员、收银员（多线程），大家共享厨房和餐厅（进程资源），各司其职，效率飞升。

**线程共享**：代码段、数据段、堆、打开的文件描述符、信号处理器
**线程独有**：线程ID、栈、寄存器、程序计数器、错误码

### 1.3 进程 vs 线程对比表

| 对比维度 | 进程 | 线程 |
|----------|------|------|
| 地址空间 | 独立（进程隔离） | 共享（同进程内共享堆） |
| 通信方式 | IPC（管道/共享内存等） | 直接读写共享变量 |
| 创建开销 | 大（需要复制地址空间等） | 小（共享进程资源） |
| 崩溃影响 | 不影响其他进程 | 一个线程崩溃可能导致整个进程挂掉 |
| 切换开销 | 大（切换页表、刷新TLB等） | 小（只需切换栈和寄存器） |

> [!warning] 面试易错点
> 一个线程崩溃可能导致整个进程挂掉——因为线程共享地址空间，一个线程的栈溢出或非法访问会破坏整个进程的内存。

## 2. 进程间通信（IPC）

> 比喻：不同公司（进程）之间怎么沟通？不能直接进对方公司翻文件（地址空间隔离），必须通过"公共规则"通信。

| IPC 方式 | 原理 | 类比 | 后端关联 |
|----------|------|------|----------|
| **管道 (Pipe)** | 内核缓冲区，半双工 | 一根水管，一端进水一端出水 | `ps aux | grep java`，Shell 管线 |
| **消息队列 (MQ)** | 内核维护的消息链表 | 公司的收发室/信箱 | 解耦利器（Kafka、RabbitMQ 思想来源） |
| **共享内存 (Shared Memory)** | 映射同一块物理内存 | 多个公司共用一间会议室 | 最快的IPC，但需要同步机制配合 |
| **信号量 (Semaphore)** | 计数器，PV操作 | 停车场闸机（有空位才能进） | 并发控制（Java Semaphore） |
| **Socket** | 网络通信 | 跨城市的信件/快递 | 分布式系统的基石，RPC/http 全依赖它 |

> [!note] 重点记忆
> - 共享内存是最快的IPC（无数据拷贝），但需要信号量等配合解决同步问题。
> - Socket 是唯一支持跨机器的IPC方式。
> - 管道只能半双工（单向传输），且只能在父子进程间使用（匿名管道）。

**记忆口诀**：**"管道消息信号量，共享内存Socket双，共享最快跨机靠Socket。"**

## 3. 用户态 vs 内核态

### 3.1 为什么要区分？

> **比喻**：酒店的前台（用户态）和机房/保险库（内核态）。
>
> 客人可以自由在前台区域活动（应用程序在用户态），但不能直接进机房碰设备开关、不能打开保险库。
> 如果客人想取贵重物品（访问硬件、操作文件），必须找工作人员（系统调用），由工作人员进入保险库处理。
>
> **意义**：如果任何人都能进机房乱按，整个酒店就瘫痪了。同理，如果用户程序能直接操作硬件，一个 bug 就能让系统崩溃。

### 3.2 系统调用 → 切换开销

```
用户程序调用 read() → 触发软中断(int 0x80/syscall指令)
  → CPU切换到内核态 → 保存用户态上下文(寄存器、栈指针等)
  → 内核执行read系统调用 → 恢复上下文 → 切回用户态
```

**为什么开销大？**
- 上下文切换：保存/恢复寄存器、切换页表、刷新TLB
- 一次系统调用 = 微秒级延迟
- **这也是为什么 epoll / io_uring 要减少系统调用次数！**

**记忆口诀**：**"用户干活内核管，系统调用是门槛。切来切去开销大，epoll用事件减调用。"**

### 3.3 和 Java 的关联

- Java 线程是内核线程（1:1），线程切换 = 用户态 ↔ 内核态切换
- 虚拟线程（Project Loom）是用户态线程，切换无内核参与，超轻量
- `DirectByteBuffer` 分配的是堆外内存，使用 `Unsafe` 绕过 JVM 直接操作（本质上还是系统调用）

## 4. 进程调度算法

> [!note] 进程调度 = OS决定"哪个进程/线程获得CPU时间"

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **FCFS（先来先服务）** | 按到达顺序调度 | 简单公平 | 短作业被长作业阻塞（护航效应） | 批处理系统 |
| **SJF（最短作业优先）** | 优先调度执行时间最短的 | 平均等待时间最短 | 需预知执行时间；长作业可能饿死 | 非交互系统 |
| **时间片轮转（RR）** | 每个进程分配固定时间片，用完轮到下一个 | 响应快，公平 | 时间片太大→FCFS，太小→频繁切换 | 交互式系统 |
| **多级反馈队列** | 多个优先级队列，时间片逐级增大，用完时间片降级 | 综合了SJF和RR的优点 | 实现复杂 | 通用OS（Linux早期） |
| **CFS（完全公平调度）** | Linux默认，每进程维护"虚拟运行时间"，选最小的运行 | 公平，数学模型优雅 | 极短任务可能被抢占 | Linux 2.6+ 默认 |

> [!tip] CFS 核心思想
> CFS 的目标是"每个进程获得完全公平的CPU时间"。它维护一棵红黑树，key 是进程的 `vruntime`（虚拟运行时间），每次调度选 `vruntime` 最小的进程。
> - 实际运行时间 × (NICE_0_LOAD / 进程权重) = vruntime
> - 权重越高（优先级高），vruntime增长越慢，越容易获得CPU

**时间片大小选择**：
- 太大（>100ms）→ 退化成 FCFS，响应慢
- 太小（<10ms）→ 上下文切换开销占比过高
- 通常 10-100ms，Linux CFS 默认目标延迟 6ms

## 5. 中断机制

### 5.1 什么是中断？

> **比喻**：你正在写代码（正常工作），突然手机响了（中断信号），你必须暂停当前工作去接电话（处理中断），接完再回来继续写代码。

中断 = CPU暂停当前程序，转去执行特定处理程序，完成后返回被暂停的程序继续执行。

### 5.2 硬中断 vs 软中断

| 类型 | 来源 | 特点 | 举例 |
|------|------|------|------|
| **硬中断（外中断）** | 外部硬件设备 | 异步发生，不可预测 | 键盘输入、网卡收到数据包、时钟中断 |
| **软中断（内中断）** | 程序内部触发 | 同步发生，可预测 | 系统调用（`int 0x80`）、缺页中断、除零异常 |

### 5.3 中断处理流程

```
中断发生 → CPU响应中断信号
  → 保存当前上下文(压栈：PC、寄存器等)
  → 查找中断向量表，跳转到中断处理程序
  → 执行中断处理上半部（紧急操作，如拷贝网卡数据）
  → 设置软中断标记，延迟执行下半部（非紧急操作）
  → 恢复上下文 → 返回被中断程序继续执行
```

> [!note] 上半部 vs 下半部
> Linux 将中断处理分为两部分：
> - **上半部（Top Half）**：必须立即执行的紧急操作（如从网卡拷贝数据到内核），执行期间禁止其他中断
> - **下半部（Bottom Half）**：可延迟执行的非紧急操作（如数据包的协议解析），允许被新中断打断
> - 目的：减少中断屏蔽时间，保证系统响应性

## 6. 上下文切换详解

### 6.1 什么是上下文切换？

上下文切换 = CPU从一个进程/线程切换到另一个进程/线程，需要保存旧进程的状态、恢复新进程的状态。

### 6.2 上下文切换的详细步骤

```
1. 保存当前进程/线程的上下文
   → 保存寄存器（通用寄存器、浮点寄存器）
   → 保存程序计数器（PC）
   → 保存栈指针（SP）

2. 更新进程/线程状态
   → 从"运行"变为"就绪/阻塞"

3. 选择下一个进程/线程
   → 调度算法选择（CFS选vruntime最小的）

4. 恢复新进程/线程的上下文
   → 恢复寄存器、PC、SP
   → 进程切换还需：切换页表基址（CR3寄存器）、刷新TLB

5. 跳转到新进程/线程的恢复点继续执行
```

### 6.3 切换开销

| 开销来源 | 说明 | 量级 |
|----------|------|------|
| 寄存器保存/恢复 | ~20个通用寄存器 + 浮点寄存器 | ~几微秒 |
| 页表切换 + TLB刷新 | **仅进程切换需要**，线程切换不需要 | ~几十微秒 |
| 缓存失效 | L1/L2/L3缓存数据失效，需要重新预热 | ~几十到百微秒 |
| 调度算法计算 | CFS红黑树查找 | ~几微秒 |

> [!warning] 线程切换 vs 进程切换
> - **线程切换**：同进程内线程共享页表，不需要切换页表和刷新TLB，开销远小于进程切换
> - **进程切换**：需要切换页表基址、刷新TLB，开销大约是线程切换的 3-5 倍
> - **这就是为什么多线程比多进程更高效！**

### 6.4 如何减少上下文切换？

- 使用无锁编程（CAS）替代锁 → 减少线程阻塞和唤醒
- 合理设置线程池大小（避免过多线程争抢CPU）
- 使用协程/虚拟线程（用户态切换，无内核参与）
- 减少系统调用次数（批量IO，如 epoll 一次返回多个事件）
- CPU亲和性（`taskset`）→ 绑定线程到特定CPU，减少缓存失效

## 7. 孤儿进程 vs 僵尸进程

| 类型 | 定义 | 危害 | 解决方式 |
|------|------|------|----------|
| **孤儿进程** | 父进程退出，子进程还在运行 | 无危害——会被 init 进程（PID=1）"收养"，init 会负责回收 | OS自动处理 |
| **僵尸进程** | 子进程退出，父进程未调用 `wait()` 回收其状态 | **有害**——僵尸进程占用进程表项（PID等资源不释放），大量僵尸可耗尽PID | 父进程调用 `wait()/waitpid()`；或 kill 父进程让 init 回收 |

> [!tip] 判断方法
> ```bash
> ps aux | grep Z   # 状态列显示 Z = 像尸进程
> ```
> 僵尸进程的 STAT 列显示 `Z+`（Zombie）。

> [!warning] 面试陷阱
> - 孤儿进程**不有害**，会被 init 收养
> - 像尸进程**有害**，PID等内核资源不释放
> - `kill -9` 无法杀死僵尸进程——它已经死了！必须 kill 父进程或让父进程调用 wait()

## 8. 与 Java 后端的关联

| OS概念 | Java关联 |
|--------|----------|
| 进程/线程 | JVM进程中的Java线程 = OS内核线程（1:1模型） |
| 上下文切换 | 线程池过大 → 频繁切换 → CPU利用率低 |
| 用户态/内核态 | synchronized、Thread.sleep() 等都会触发内核态切换 |
| 进程调度 | Linux CFS → Java线程的CPU时间分配 |
| 中断机制 | 网卡中断 → epoll → Selector.select() |
| IPC | 分布式系统中 Socket 是唯一跨机器IPC方式 |

## 面试题速查

> [!example] 本模块高频面试题

1. **进程和线程的区别是什么？为什么要有多线程？**
   - 进程=资源分配最小单位，线程=CPU调度最小单位
   - 线程共享进程资源，创建和切换开销更小，但一个线程崩溃可能拖垮整个进程

2. **进程间通信有哪些方式？各自优缺点是什么？**
   - 管道/消息队列/共享内存/信号量/Socket
   - 共享内存最快，Socket唯一支持跨机器

3. **用户态和内核态的区别？为什么要区分？系统调用的开销在哪？**
   - 保护OS内核不被用户程序破坏；系统调用涉及上下文切换、页表切换、TLB刷新

4. **进程调度算法有哪些？CFS是什么？**
   - FCFS/SJF/RR/多级反馈队列/CFS
   - CFS用vruntime红黑树保证公平调度

5. **什么是中断？硬中断和软中断的区别？**
   - 中断=暂停当前工作去处理紧急事件；硬中断来自外部硬件，软中断来自程序内部

6. **上下文切换的步骤和开销？如何减少？**
   - 保存/恢复上下文 + 页表切换 + TLB刷新 + 缓存失效
   - 减少方式：无锁编程、合理线程池、协程、减少系统调用

7. **孤儿进程和僵尸进程的区别？**
   - 孤儿=父退出子还在（无害，init收养）；僵尸=子退出父未回收（有害，占PID）
