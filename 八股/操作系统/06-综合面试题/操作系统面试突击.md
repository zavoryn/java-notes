# 操作系统面试突击

> [!tip] 核心速记
> **面试时的操作系统题，三个字——"用比喻"。**
> **每个概念找一个生活场景对应，考官会觉得你"真正懂了"而不是"背了概念"。**
> **尤其是IO多路复用和零拷贝，一定要联系你熟悉的中间件（Redis/Netty/Kafka）说实际应用。**

## 2. 面试满分话术

### 话术 1：被问"进程和线程的区别"

> "进程是操作系统**资源分配**的最小单位，线程是**CPU调度**的最小单位。
>
> 打个比方：进程好比一家公司，有独立营业执照、银行账户；线程是公司里的员工，共享公司资源但各有各的工作台。
>
> 一个进程崩溃不会影响其他进程（进程隔离），但一个线程崩溃可能导致整个进程挂掉（共享地址空间）。
>
> 在 Java 后端里，我们开的多线程都在同一个 JVM 进程中，共享堆内存，所以并发安全问题特别重要。"

### 话术 2：被问"epoll 为什么比 select 快"

> "select 就像餐厅里服务员一个一个去问每桌客人要不要点菜，不管客人有没有需求都要问一遍（O(n)全量轮询）。
>
> epoll 相当于在每桌装了个按铃，客人有需求才按铃通知服务员（事件驱动），服务员的效率从 O(n) 变成 O(1)。
>
> 底层实现上：select 用固定大小位图存储 fd，每次调用都要把整个 fd_set 从用户态拷贝到内核态；epoll 用红黑树维护所有 fd，用就绪链表只返回有事件的 fd，避免了全量拷贝。
>
> 这就是为什么 Redis、Nginx、Netty 全部用 epoll 做网络模型。"

### 话术 3：被问"零拷贝是什么，Kafka 怎么用的"

> "传统 IO 发送一个文件到网络，数据要在内核态和用户态之间拷贝 4 次，CPU 大量时间花在数据搬运上。
>
> 零拷贝的核心思想：数据不经过用户态，直接在内核态完成传输。
>
> Linux 的 sendfile 系统调用把拷贝降到 2 次（DMA 直接传输），采用 scatter-gather DMA 甚至可以实现更彻底的零拷贝。
>
> Kafka 消费消息时，数据从磁盘的 PageCache 直接通过 sendfile 发到网卡，**完全不经过用户空间**，这就是 Kafka 能跑到百万级吞吐量的核心原因之一。"

### 话术 4：被问"为什么Redis单线程还这么快"

> "Redis 单线程快有三个原因：
>
> 第一，**纯内存操作**——Redis 所有数据都在内存中，读写延迟是纳秒级，不需要走磁盘IO。
>
> 第二，**epoll IO多路复用**——Redis 单线程用 epoll 同时处理数万个连接，不需要多线程就能高效管理网络IO。
>
> 第三，**无锁并发**——单线程没有锁竞争、没有上下文切换开销、没有死锁风险，CPU利用率极高。
>
> 不过 Redis 6.0 引入了多线程IO来处理网络读写（主要是解压和缓冲），但命令执行仍然是单线程的。"

## 3. 记忆口诀

### 进程线程
> **"进程管资源，线程做调度，进程独立线程共享，崩溃波及不一样。"**

### IPC通信
> **"管道消息信号量，共享内存Socket双，共享最快跨机靠Socket。"**

### 死锁四条件
> **"互斥持有不可夺，循环等待死锁成。破坏其一就能破，按序加锁是上策。"**

### 页面置换算法
> **"FIFO 简单不顶用，OPT 最优做不到，LRU 最近没人碰，Clock 转圈最实用。"**

### 零拷贝
> **"传统四拷四切换，sendfile 两拷两切换，Kafka 转发不用搬，数据直飞内核过。"**

### epoll 三问
> **"红黑树管全连接，就绪链表只返活，LT 反复催你拿，ET 只说一次话。"**

### 用户态内核态
> **"用户干活内核管，系统调用是门槛。切来切去开销大，epoll用事件减调用。"**

### 硬链接软链接
> **"硬链同inode不能跨区，软链独立号可跨区。删原文件硬链在，软链变死链。"**

## 4. 跨题串联

> 操作系统不是孤岛，这些知识和 JVM、MySQL、Redis、多线程编程紧密相连。面试时主动跨题串联，秒杀大多数候选人。

### 操作系统 → JVM

| OS 概念 | JVM 关联 |
|---------|----------|
| 进程/线程 | JVM 进程中的 Java 线程 = OS 内核线程（1:1 模型） |
| 虚拟内存 | JVM 堆内存、元空间都是虚拟地址空间中的一段 |
| 缺页中断 | 堆内存首次 `new 对象` 时，OS 分配物理内存，类似缺页 |
| 用户态/内核态 | GC 的 safepoint（安全点）、线程切换都涉及内核态 |
| CPU 缓存/伪共享 | volatile、CAS、`@Contended` 底层都依赖缓存一致性 |

### 操作系统 → MySQL

| OS 概念 | MySQL 关联 |
|---------|------------|
| IO 多路复用 | MySQL 8.0 网络层使用 epoll（通过 libevent） |
| 零拷贝 | InnoDB 的 `innodb_flush_method=O_DIRECT` 绕过 OS 缓存 |
| 页面置换 LRU | Buffer Pool 的 LRU 淘汰策略（冷热分离链表） |
| 文件系统 | `redo log`、`undo log`、`binlog` 都是顺序写，利用磁盘顺序IO优势 |

### 操作系统 → Redis

| OS 概念 | Redis 关联 |
|---------|------------|
| IO 多路复用 | Redis 单线程 + epoll 处理数万连接 |
| 零拷贝 | Redis 6.0+ 多线程 IO 也是基于 epoll |
| 页面置换 LRU | Redis 内存淘汰策略（volatile-lru / allkeys-lru） |
| 文件系统 | RDB 快照 = fork() + Copy-on-Write |
| CPU 缓存 | Redis 纯内存操作，充分利用 CPU 缓存命中的高效 |

### 操作系统 → Java 多线程

| OS 概念 | Java 多线程关联 |
|---------|----------------|
| 互斥锁/条件变量 | `synchronized` → OS mutex + `wait/notify` → OS condition variable |
| 信号量 | Java `Semaphore` → OS semaphore |
| CAS 无锁 | CPU 的 `cmpxchg` 指令 + `lock` 前缀（锁缓存行） |
| 伪共享 | `LongAdder` 优于 `AtomicLong`，Disruptor 框架的核心优化 |
| 零拷贝 | `FileChannel.transferTo()`、Netty 的 `FileRegion` |

## 5. 高频面试题补充（展开详解）

### 5.1 服务端产生大量 CLOSE_WAIT 或 TIME_WAIT 怎么排查？

**TCP连接状态回顾**：

```
CLOSE_WAIT：被动关闭方收到FIN后，等待应用层调用close()
TIME_WAIT：主动关闭方发送FIN+收到ACK后，等待2MSL（60秒）确保对方收到ACK

CLOSE_WAIT大量堆积 → 对方发了FIN，但本端应用没调用close()
  → 原因：代码bug（忘记关闭socket）、线程池满（来不及处理关闭请求）
  → 排查：ss -antp | grep CLOSE_WAIT → 找到进程 → 检查代码是否正确关闭连接
  → 解决：确保所有socket都有close()、设置合理的连接超时时间

TIME_WAIT大量堆积 → 本端主动关闭连接后等待2MSL
  → 原因：短连接频繁创建和关闭（如HTTP短连接）、本端主动关闭多
  → 排查：ss -s → 查看TIME_WAIT数量 → ss -antp | grep TIME_WAIT
  → 解决：
    → 内核参数：net.ipv4.tcp_tw_reuse=1（允许重用TIME_WAIT连接）
    → 内核参数：net.ipv4.tcp_max_tw_buckets=5000（限制TIME_WAIT数量）
    → 长连接替代短连接
    → 让客户端主动关闭（服务端少主动关）
```

> [!warning] CLOSE_WAIT vs TIME_WAIT 的关键区分
> - **CLOSE_WAIT大量** = 你的代码bug（没关闭连接）→ 必须修代码
> - **TIME_WAIT大量** = 正常现象但数量过多 → 调内核参数或改连接策略

### 5.2 什么是上下文切换？如何减少上下文切换？

> 上下文切换 = CPU从一个进程/线程切换到另一个，需要保存旧进程状态、恢复新进程状态。

**开销来源**：
- 保存/恢复寄存器、PC、SP
- 进程切换还需：切换页表（CR3）、刷新TLB
- 缓存失效（L1/L2/L3数据失效，需要重新预热）

**如何减少**：
- 无锁编程（CAS替代Mutex）→ 减少线程阻塞/唤醒
- 合理线程池大小（避免过多线程争抢CPU）
- 协程/虚拟线程（用户态切换，无内核参与）
- 减少系统调用次数（批量IO、epoll一次返回多个事件）
- CPU亲和性（`taskset`绑定线程到特定CPU，减少缓存失效）

**查看上下文切换次数**：
```bash
vmstat 1    # cs列 = 每秒上下文切换次数
pidstat -w  # 查看每个线程的切换次数
```

### 5.3 为什么 Redis 单线程还能这么快？

> 三个核心原因：**epoll + 内存操作 + 无锁**

| 因素 | 说明 |
|------|------|
| **epoll IO多路复用** | 单线程用epoll同时处理数万连接，不需要多线程 |
| **纯内存操作** | 所有数据在内存中，纳秒级读写，不需要磁盘IO |
| **无锁并发** | 单线程没有锁竞争、上下文切换、死锁风险 |
| **高效数据结构** | SDS、跳表、压缩列表等精心优化 |
| **避免上下文切换** | 单线程不需要线程切换，CPU利用率极高 |

> [!note] Redis 6.0 的多线程IO
> Redis 6.0 引入多线程处理**网络IO读写**（主要是接收命令和发送响应的缓冲区操作），但**命令执行仍然是单线程**。
> 原因：网络IO成为瓶颈时（大量连接同时发送大数据），单线程处理网络读写不够快。
> 好处：命令执行单线程 → 仍然无锁、无并发安全问题。

### 5.4 系统负载（Load Average）的含义？CPU核数和Load的关系？

```
Load Average = 系统在1分钟、5分钟、15分钟内的平均活跃进程数
  → 活跃进程 = 正在运行的 + 正在等待CPU的 + 正在等待IO的

查看：top 第一行 或 cat /proc/loadavg

Load与CPU核数的关系：
  → Load < 核数：系统空闲，CPU有余力
  → Load ≈ 核数：系统刚好满载，所有CPU都在工作
  → Load > 核数：系统过载，有进程在排队等待CPU

举例：4核CPU
  → Load=4 → 所有CPU刚好满载
  → Load=8 → 有4个进程在排队等待CPU（50%的进程在等待）
  → Load=2 → CPU利用率50%，还有余力

Load高的两种情况：
  → CPU Load高：进程争抢CPU时间 → 需要优化计算逻辑或增加CPU
  → IO Load高：进程等待磁盘IO → 需要优化IO或增加磁盘性能
```

> [!tip] 判断CPU还是IO瓶颈
> ```bash
> vmstat 1  # 看 r 列（运行队列）和 b 列（阻塞等待IO）
> # r高 → CPU瓶颈；b高 → IO瓶颈
> ```

### 5.5 fork() 的写时复制原理，为什么 Redis 用 fork 做 RDB？

**fork() + Copy-on-Write (COW)**：

```
fork()创建子进程 → 父子共享同一物理内存页（只读）
  → 某一方写入数据 → 内核发现写入只读页 → 触发写时复制
  → 复制该页给写入方 → 父子各自有独立副本 → 继续执行

优势：
  → fork()几乎零开销（不需要复制整个地址空间）
  → 只有实际修改的页才复制（大多数页不会修改）
```

**Redis 用 fork 做 RDB 的原因**：

```
Redis主进程执行 BGSAVE → fork()创建子进程
  → 子进程遍历内存数据 → 写入RDB文件到磁盘
  → 主进程继续服务客户端请求
  → 如果主进程修改了某个key → 对应的内存页触发COW → 复制给主进程
  → 子进程读到的仍然是fork时刻的数据（一致性保证）

为什么不用多线程做RDB？
  → 多线程需要加锁保护数据一致性（锁开销大）
  → fork+COW天然保证数据一致性（fork时刻的数据快照）
  → Redis的数据结构不适合并发遍历（各种不同数据结构）
```

### 5.6 线程池大小如何设置？CPU密集 vs IO密集？

| 任务类型 | 线程数公式 | 说明 |
|----------|-----------|------|
| **CPU密集** | `N_cpu + 1` | 线程数=CPU核数+1，多1个线程是为了在某个线程偶尔缺页中断时补位 |
| **IO密集** | `N_cpu * 2` 或 `N_cpu * (1 + W/C)` | W=等待时间，C=计算时间；IO密集时线程大量时间在等待，可以多开线程 |

**公式推导**：
```
最佳线程数 = CPU核数 * (1 + 等待时间/计算时间)

举例：8核CPU
  → CPU密集任务（W/C≈0）：8 * (1+0) = 8线程
  → IO密集任务（W/C≈10，等待10倍于计算）：8 * (1+10) = 88线程
```

> [!warning] 面试陷阱
> - 线程数不是越多越好——线程过多导致频繁上下文切换，CPU利用率反而下降
> - Java的线程是1:1模型（内核线程），线程池大小要考虑OS的上下文切换开销
> - 虚拟线程（Project Loom）是用户态线程，线程池大小可以不受CPU核数限制

### 5.7 Linux 查看端口占用、查看进程打开的文件、查看磁盘IO的命令？

| 场景 | 命令 | 说明 |
|------|------|------|
| **查看端口占用** | `lsof -i :8080` 或 `ss -antp \| grep 8080` | 查看哪个进程占用了8080端口 |
| **查看进程打开的文件** | `lsof -p <PID>` | 列出进程打开的所有文件（包括socket、pipe） |
| **查看磁盘IO** | `iostat -x 1` | await=IO等待时间(ms)，util=磁盘使用率(%) |
| **查看磁盘空间** | `df -h` | 磁盘分区使用率 |
| **查看大文件** | `du -sh * \| sort -rh \| head -10` | 当前目录下最大的10个文件/文件夹 |
| **查看网络连接统计** | `ss -s` | 各状态TCP连接数量汇总 |

## 6. 综合面试策略

1. **先给一句话定义**：进程=资源分配最小单位，线程=CPU调度最小单位
