# 线程同步与死锁

> [!tip] 核心速记
> **"互斥持有不可夺，循环等待死锁成。破坏其一就能破，按序加锁是上策。"**
> 三大同步机制：互斥锁（厕所门锁）、条件变量（奶茶叫号）、信号量（停车场闸机）。

---

## 1. 线程同步三大法宝

> 比喻：公司只有一个打印机（共享资源），多名员工（线程）都想用。如果没有"同步机制"，大家都往里塞纸，打印出来的东西乱成一团。

| 同步机制 | 作用 | Java 对应 | 生活类比 |
|----------|------|-----------|----------|
| **互斥锁 (Mutex)** | 保证同一时间只有一个线程访问临界区 | `synchronized`、`ReentrantLock` | 厕所门锁——进去了锁门，别人等，用完开锁 |
| **条件变量 (Condition)** | 允许线程在条件不满足时等待，条件满足时被唤醒 | `wait()/notify()`、`Condition.await()/signal()` | 奶茶店——做好了叫号，没叫到你继续等 |
| **信号量 (Semaphore)** | 控制同时访问资源的线程数 | `Semaphore` | 停车场——N个车位，满了后来的排队等 |

### 1.1 互斥锁详解

互斥锁是最基本的同步机制：
- **加锁（lock）**：进入临界区前加锁，其他线程阻塞等待
- **解锁（unlock）**：离开临界区后解锁，唤醒一个等待线程

Java 中 `synchronized` 和 `ReentrantLock` 底层都依赖 OS 的 Mutex 实现：
- `synchronized` → OS mutex + `wait/notify` → OS condition variable
- `ReentrantLock` → `AbstractQueuedSynchronizer`（AQS），底层也是 CAS + Mutex

## 2. 死锁

### 2.1 死锁四个必要条件（缺一不可）

> 比喻：两个人吃饭，每人拿一根筷子，都等对方放下筷子。永远等不到——这就是死锁。

| 条件 | 说明 | 类比 |
|------|------|------|
| **互斥** | 资源一次只能被一个线程使用 | 一根筷子只能一个人拿 |
| **持有并等待** | 线程持有资源同时等待新资源 | 拿着左筷子等右筷子 |
| **不可剥夺** | 不能强行抢走线程手中的资源 | 不能从别人手里抢筷子 |
| **循环等待** | 形成环形等待链 | A等B放筷子，B等A放筷子 |

### 2.2 死锁预防 vs 避免

**预防**：破坏四个条件之一
- 破坏"互斥"：用无锁数据结构（CAS）
- 破坏"持有并等待"：一次性申请所有资源（要么全给，要么都不给）
- 破坏"不可剥夺"：允许抢占（`Lock.lockInterruptibly()`）
- 破坏"循环等待"：按固定顺序申请锁

**避免（银行家算法）**：系统动态判断资源分配是否会导致死锁，如果会就不分配。

> 类比：银行家发放贷款——他要判断借出去这笔钱后，自己还能不能应对所有人的取款需求。如果会破产就不贷。

### 2.3 死锁检测与恢复

- **检测**：定期用资源分配图算法检测是否存在循环等待
- **恢复**：终止一个或多个进程（代价最小原则）；或回滚进程到安全状态

## 3. 读写锁（ReadWriteLock）

### 3.1 什么是读写锁？

> 比喻：图书馆的书——多人可以同时阅读（共享读），但有人要修改时必须独占（独占写）。

读写锁将锁分为**读锁（共享锁）**和**写锁（独占锁）**：
- **读锁**：多个线程可以同时持有读锁（共享读取）
- **写锁**：只有一个线程能持有写锁，且持有期间读锁也不能获取（独占写入）

### 3.2 读者写者问题

| 场景 | 策略 |
|------|------|
| 读-读 | 允许并发（共享读锁） |
| 读-写 | 互斥（写者独占） |
| 写-写 | 互斥（写者独占） |

**读者优先**：只要有人在读，后续读者可以直接进入，写者必须等所有读者读完。
- 问题：写者可能被饿死（持续有读者进来）

**写者优先**：有写者等待时，后续读者必须等写者完成。
- 问题：读者可能被饿死

**公平策略**：按请求顺序处理，既不饿死读者也不饿死写者。

### 3.3 Java ReadWriteLock

```java
ReadWriteLock rwLock = new ReentrantReadWriteLock();
// 读操作
rwLock.readLock().lock();
try { /* 读数据 */ } finally { rwLock.readLock().unlock(); }
// 写操作
rwLock.writeLock().lock();
try { /* 写数据 */ } finally { rwLock.writeLock().unlock(); }
```

> [!warning] 面试易错点
> `ReentrantReadWriteLock` 的写锁可以降级为读锁（持有写锁 → 获取读锁 → 释放写锁），但**读锁不能升级为写锁**（会导致死锁！）

## 4. 自旋锁（Spinlock）

### 4.1 什么是自旋锁？

自旋锁 = 线程在获取锁失败时，不进入阻塞状态，而是**循环检查锁是否释放**（"自旋"等待）。

```c
// 自旋锁伪代码
while (lock == 1) {
    // 忙等待，不断检查锁状态
}
lock = 1; // 获取锁
```

### 4.2 适用场景

> [!note] 自旋锁 vs 互斥锁
> - **互斥锁**：获取失败 → 线程阻塞 → 内核切换 → 被唤醒 → 再尝试获取。开销 = 2次上下文切换（微秒级）
> - **自旋锁**：获取失败 → 在CPU上忙等 → 锁释放后立即获取。开销 = CPU空转时间

| 场景 | 推荐 |
|------|------|
| 临界区极短（< 2次上下文切换的时间） | **自旋锁**（避免了2次切换开销） |
| 临界区较长 | **互斥锁**（自旋会浪费大量CPU时间） |
| 单核CPU | **互斥锁**（自旋锁在单核上无法工作——没有其他线程能释放锁） |

### 4.3 Java中的自旋

- `synchronized` 锁升级过程中，轻量级锁阶段使用**自适应自旋**（JVM根据上次自旋是否成功动态调整自旋次数）
- `AtomicInteger` 等原子类的 CAS 操作本质上也是一种"自旋+尝试"的模式

## 5. 乐观锁 vs 悲观锁

| 维度 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| **思想** | "肯定会有人和我争，我先锁住" | "大概率没人争，我先做，冲突了再重试" |
| **实现** | `synchronized`、`ReentrantLock`、OS Mutex | CAS（Compare-And-Swap）、版本号 |
| **适用** | 写多读少，冲突频繁 | 读多写少，冲突少 |
| **开销** | 锁获取/释放 → 内核态切换 | CAS（CPU指令级，无内核切换） |
| **问题** | 频繁加锁开销大 | CAS冲突时自旋浪费CPU；ABA问题 |

> [!tip] CAS 底层原理
> CAS = CPU 的 `cmpxchg` 指令 + `lock` 前缀（锁缓存行，不是锁总线！）
> - `cmpxchg`：比较并交换，原子操作
> - `lock` 前缀：锁定缓存行，确保多核下的缓存一致性
> - Java 的 `Unsafe.compareAndSwapInt()` 就是 CAS

**ABA 问题**：值从 A → B → A，CAS 认为"没变"，实际已变过。解决方案：版本号（AtomicStampedReference）。

## 6. 银行家算法详解

### 6.1 核心思想

> 类比：银行家发放贷款——他要判断借出去这笔钱后，自己还能不能应对所有人的取款需求。如果会破产就不贷。

银行家算法在分配资源前，先检查分配后系统是否仍处于**安全状态**。如果是就分配，否则拒绝。

### 6.2 详细步骤

```
已知：
  - Available[]：每类资源当前可用数量
  - Max[][]：每个进程最大需求矩阵
  - Allocation[][]：每个进程已分配矩阵
  - Need[][] = Max - Allocation：每个进程还需矩阵

步骤：
  1. 请求 Request[i] ≤ Need[i]？否则报错
  2. Request[i] ≤ Available？否则等待
  3. 假设分配：Available -= Request, Allocation += Request, Need -= Request
  4. 执行安全性检查：
     - 找到一个 Need ≤ Available 的进程 P
     - 假设 P 完成：Available += Allocation[P]
     - 重复直到所有进程都完成 → 安全，正式分配
     - 找不到 → 不安全，回滚假设分配，进程等待
```

### 6.3 安全性检查举例

| 进程 | Need | Allocation | Available |
|------|------|------------|-----------|
| P0 | (1,2,2) | (3,0,2) | (3,3,2) |
| P1 | (1,0,0) | (5,1,0) | |
| P2 | (1,3,1) | (2,1,1) | |

检查：P1的Need(1,0,0) ≤ Available(3,3,2) → P1完成 → Available=(8,4,2)
→ P0的Need(1,2,2) ≤ (8,4,2) → P0完成 → Available=(11,4,4)
→ P2的Need(1,3,1) ≤ (11,4,4) → P2完成 → 安全！

## 7. 活锁 vs 死锁 vs 饥饿

| 类型 | 定义 | 类比 | 区别 |
|------|------|------|------|
| **死锁** | 多个线程互相等待，都无法继续 | 两人对峙，都不退让 | **大家都停了** |
| **活锁** | 线程不断重试但始终无法成功 | 两人相遇，都礼貌让路，结果反复让路还是撞 | **大家都在动，但推进不了** |
| **饥饿** | 线程长期无法获得所需资源 | 一人排队永远排不到 | **某些线程停了，其他在跑** |

> [!tip] 活锁解决方案
> - 随机退避时间：冲突后随机等待一段时间再重试（以太网CSMA/CD就是这个思路）
> - 优先级调整：给长期等待的线程提升优先级

## 8. 与 Java 后端的关联

| OS概念 | Java关联 |
|--------|----------|
| 互斥锁/条件变量 | `synchronized` → OS mutex + `wait/notify` → OS condition variable |
| 信号量 | Java `Semaphore` → OS semaphore |
| 读写锁 | `ReentrantReadWriteLock` → 读者写者问题的Java实现 |
| CAS 无锁 | CPU 的 `cmpxchg` 指令 + `lock` 前缀（锁缓存行） |
| 自旋锁 | `synchronized` 轻量级锁的自适应自旋 |
| 避免死锁 | 按固定顺序加锁（Java中推荐`Lock.tryLock(timeout)`避免无限等待） |

## 面试题速查

> [!example] 本模块高频面试题

1. **线程同步有哪些方式？synchronized和ReentrantLock底层依赖什么？**
   - 互斥锁/条件变量/信号量；底层依赖OS Mutex和Condition Variable

2. **什么是死锁？四个必要条件是什么？如何预防和避免？**
   - 互斥/持有并等待/不可剥夺/循环等待；预防=破坏条件之一，避免=银行家算法动态判断

3. **读写锁是什么？读者写者问题怎么解决？**
   - 读锁共享、写锁独占；公平策略避免读者或写者饿死

4. **自旋锁适用场景？为什么临界区短时自旋锁更高效？**
   - 临界区极短（< 2次上下文切换时间）时，自旋避免了2次切换开销

5. **乐观锁vs悲观锁？CAS底层原理？ABA问题？**
   - 悲观锁先锁住，乐观锁先做再检查冲突；CAS=CPU cmpxchg+lock前缀；ABA用版本号解决

6. **银行家算法步骤？**
   - 假设分配 → 安全性检查 → 安全则正式分配，不安全则等待

7. **活锁vs死锁vs饥饿的区别？**
   - 死锁=都停了；活锁=都在动但推不动；饥饿=某些线程长期得不到资源
