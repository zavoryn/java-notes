## 1. 进程 vs 线程

### 1.1 一句话定义

| 概念 | 定义 | 生活类比 |
|------|------|----------|
| **进程** | 资源分配的最小单位 | 一家公司（有自己的营业执照、银行账户、办公楼） |
| **线程** | CPU 调度的最小单位 | 公司里的员工（共享公司资源，每人干自己的活） |

### 1.2 为什么要有线程？

> 比喻：一家餐厅（进程）只有老板一个人（单线程）——他要迎客、点单、炒菜、结账。来了 10 桌客人，老板忙到崩溃。
>
> 后来老板雇了厨师、服务员、收银员（多线程），大家共享厨房和餐厅（进程资源），各司其职，效率飞升。

**线程共享**：代码段、数据段、堆、打开的文件描述符、信号处理器
**线程独有**：线程ID、栈、寄存器、程序计数器、错误码

### 1.3 进程间通信（IPC）—— 后端必考

> 比喻：不同公司（进程）之间怎么沟通？不能直接进对方公司翻文件（地址空间隔离），必须通过"公共规则"通信。

| IPC 方式 | 原理 | 类比 | 后端关联 |
|----------|------|------|----------|
| **管道 (Pipe)** | 内核缓冲区，半双工 | 一根水管，一端进水一端出水 | `ps aux | grep java`，Shell 管线 |
| **消息队列 (MQ)** | 内核维护的消息链表 | 公司的收发室/信箱 | 解耦利器（Kafka、RabbitMQ 思想来源） |
| **共享内存 (Shared Memory)** | 映射同一块物理内存 | 多个公司共用一间会议室 | 最快的IPC，但需要同步机制配合 |
| **信号量 (Semaphore)** | 计数器，PV操作 | 停车场闸机（有空位才能进） | 并发控制（Java Semaphore） |
| **Socket** | 网络通信 | 跨城市的信件/快递 | 分布式系统的基石，RPC/http 全依赖它 |

**重点记忆**：
- 共享内存是最快的IPC（无数据拷贝），但需要信号量等配合解决同步问题。
- Socket 是唯一支持跨机器的IPC方式。

## 2. 线程同步与死锁

### 2.1 线程同步三大法宝

> 比喻：公司只有一个打印机（共享资源），多名员工（线程）都想用。如果没有"同步机制"，大家都往里塞纸，打印出来的东西乱成一团。

| 同步机制 | 作用 | Java 对应 | 生活类比 |
|----------|------|-----------|----------|
| **互斥锁 (Mutex)** | 保证同一时间只有一个线程访问临界区 | `synchronized`、`ReentrantLock` | 厕所门锁——进去了锁门，别人等，用完开锁 |
| **条件变量 (Condition)** | 允许线程在条件不满足时等待，条件满足时被唤醒 | `wait()/notify()`、`Condition.await()/signal()` | 奶茶店——做好了叫号，没叫到你继续等 |
| **信号量 (Semaphore)** | 控制同时访问资源的线程数 | `Semaphore` | 停车场——N个车位，满了后来的排队等 |

### 2.2 死锁

#### 死锁四个必要条件（缺一不可）

> 比喻：两个人吃饭，每人拿一根筷子，都等对方放下筷子。永远等不到——这就是死锁。

| 条件 | 说明 | 类比 |
|------|------|------|
| **互斥** | 资源一次只能被一个线程使用 | 一根筷子只能一个人拿 |
| **持有并等待** | 线程持有资源同时等待新资源 | 拿着左筷子等右筷子 |
| **不可剥夺** | 不能强行抢走线程手中的资源 | 不能从别人手里抢筷子 |
| **循环等待** | 形成环形等待链 | A等B放筷子，B等A放筷子 |

#### 死锁预防 vs 避免

**预防**：破坏四个条件之一
- 破坏"互斥"：用无锁数据结构（CAS）
- 破坏"持有并等待"：一次性申请所有资源（要么全给，要么都不给）
- 破坏"不可剥夺"：允许抢占（`Lock.lockInterruptibly()`）
- 破坏"循环等待"：按固定顺序申请锁

**避免（银行家算法）**：系统动态判断资源分配是否会导致死锁，如果会就不分配。

> 类比：银行家发放贷款——他要判断借出去这笔钱后，自己还能不能应对所有人的取款需求。如果会破产就不贷。

## 3. 内存管理

### 3.1 虚拟内存 → 物理内存映射

> **核心比喻**：**"图书馆书架的目录索引"**
>
> 你把全世界的书（虚拟内存）都列在目录卡片上，但实际上图书馆的书架（物理内存）只放了最常用的 1000 本。
> - 当你去找一本书（访问内存地址），先查目录（页表）
> - 如果目录上写着"在书架上"（在物理内存中），直接去拿
> - 如果目录上写着"在仓库"（在磁盘上），就去仓库搬过来放到书架上（缺页中断）
> - 你感觉好像全世界的书都在书架上——这就是虚拟内存的魔法

### 3.2 核心概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **虚拟地址** | 程序看到的地址（每个进程 0x0000...开始） | 你心中的"书的编号" |
| **物理地址** | 真实的RAM地址 | 书实际在书架第几排第几格 |
| **页 (Page)** | 虚拟内存的固定大小块（通常4KB） | 每本书都是固定大小 |
| **页框 (Frame)** | 物理内存的固定大小块（4KB） | 书架上的每个格子 |
| **页表 (Page Table)** | 虚拟页→物理页框的映射表 | 目录索引卡片 |
| **TLB (Translation Lookaside Buffer)** | 页表的高速缓存（MMU内部） | 手上常翻的"常用卡片" |

### 3.3 缺页中断流程（重要！）

```
程序访问地址 → MMU查TLB(TLB命中?直接翻译:查页表)
  → 页表项有效位=1?直接翻译:触发缺页中断
  → 缺页中断: 从磁盘加载页面到物理内存 → 更新页表 → 重试指令
```

> 写时复制（Copy-on-Write）：fork() 创建子进程时，父子共享同一物理页，只有当某一方进行写操作时才真正复制该页。这就像图书馆里两个人借同一本书——不做副本，只有有人要写笔记时才复印一本新书给他。

### 3.4 和 Java 的关联

- JVM 的 `-Xms`、`-Xmx` 管理的是虚拟内存
- GC 后回收的堆空间返回给 OS 还是保留？取决于 JVM 的 `-XX:MinHeapFreeRatio` 配置
- 内存映射文件（MappedByteBuffer）直接利用 OS 虚拟内存机制

## 4. 页面置换算法

> 比喻：图书馆书架只有 100 个格子（物理内存），但有 10000 本书（虚拟内存）。
> 当有新书要放进来、书架满了，就要决定把哪本旧书撤下去（换出到磁盘）。

| 算法 | 原理 | 优点 | 缺点 | 类比 |
|------|------|------|------|------|
| **OPT（最佳置换）** | 换掉未来最长时间不再访问的页 | 缺页率最低 | 无法实现（需要预知未来） | 知道哪本书以后不会再借——神仙才能做到 |
| **FIFO（先进先出）** | 换掉最早进入的那页 | 简单 | 可能换出频繁使用的页（Belady异常） | 最早进书架的先扔——不科学 |
| **LRU（最近最久未使用）** | 换掉最长时间没被访问的页 | 接近OPT | 实现成本高 | 最久没被借过的书——比较合理 |
| **LFU（最不常用）** | 换掉访问次数最少的那页 | 适合热点数据 | 新页容易被换出 | 被借次数最少的书——但有新书"冷启动"问题 |
| **Clock（时钟算法）** | 循环扫描，访问位=0就换出，=1就清零 | LRU的近似，实践常用 | 比LRU稍差 | 管理员转圈检查，没被翻过的就撤 |

### 4.1 Clock 算法详解（最实际）

```
页面组成环形链表 + 访问位(0/1)
指针循环移动：
  如果访问位==1 → 置0，继续走
  如果访问位==0 → 换出这一页
→ LRU 的性能，FIFO 的简单实现
```

**和缓存的关联**：Redis 的 volatile-lru / allkeys-lru 淘汰策略就是 LRU 思想。MySQL Buffer Pool 也用了 LRU 变种（冷热分离）。

## 5. 用户态 vs 内核态

### 5.1 为什么要区分？

> **比喻**：酒店的前台（用户态）和机房/保险库（内核态）。
>
> 客人可以自由在前台区域活动（应用程序在用户态），但不能直接进机房碰设备开关、不能打开保险库。
> 如果客人想取贵重物品（访问硬件、操作文件），必须找工作人员（系统调用），由工作人员进入保险库处理。
>
> **意义**：如果任何人都能进机房乱按，整个酒店就瘫痪了。同理，如果用户程序能直接操作硬件，一个 bug 就能让系统崩溃。

### 5.2 系统调用 → 切换开销

```
用户程序调用 read() → 触发软中断(int 0x80/syscall指令)
  → CPU切换到内核态 → 保存用户态上下文(寄存器、栈指针等)
  → 内核执行read系统调用 → 恢复上下文 → 切回用户态
```

**为什么开销大？**
- 上下文切换：保存/恢复寄存器、切换页表、刷新TLB
- 一次系统调用 = 微秒级延迟
- **这也是为什么 epoll / io_uring 要减少系统调用次数！**

### 5.3 和 Java 的关联

- Java 线程是内核线程（1:1），线程切换 = 用户态↔内核态切换
- 虚拟线程（Project Loom）是用户态线程，切换无内核参与，超轻量
- `DirectByteBuffer` 分配的是堆外内存，使用 `Unsafe` 绕过 JVM 直接操作（本质上还是系统调用）

## 6. IO 多路复用（重点）

> **核心场景**：一个餐厅服务员（一个线程）要同时服务 10 桌客人（10 个 socket 连接）。
>
> **没有 IO 多路复用之前**：
> - **BIO（阻塞IO）**：服务员站在 1 号桌前，等客人点菜（阻塞），其他 9 桌干等着。一个线程只能服务一个连接。
>
> **有了 IO 多路复用**：
> - 服务员注册到"餐厅总管"（内核的 epoll），哪桌有需求（数据就绪），总管通知服务员去处理。一个服务员可以高效服务全部 10 桌！

### 6.1 select / poll / epoll 对比

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| **数据结构** | 固定长度 `fd_set` 位图 | 动态数组 `pollfd` | 红黑树 + 就绪链表 |
| **最大连接数** | 1024（硬限制） | 无限制 | 无限制（数十万） |
| **时间复杂度** | O(n)，每次全量轮询 | O(n)，每次全量轮询 | O(1)，只返回就绪的 |
| **内核态→用户态拷贝** | 全量拷贝所有 fd | 全量拷贝所有 fd | 只拷贝就绪的 fd |
| **触发模式** | 仅水平触发 | 仅水平触发 | **LT（水平触发）+ ET（边缘触发）** |
| **适用场景** | 连接数少 | 连接数少 | **高并发（C10K/C100K）** |

### 6.2 epoll 的 LT（水平触发）vs ET（边缘触发）

> **比喻**：快递到了，物业通知你。
>
> **LT（水平触发）** = 物业一直打电话催你拿快递，直到你取走为止。你不取他就一直打。
> - 优点：不丢事件，编程简单
> - 缺点：可能重复通知
>
> **ET（边缘触发）** = 物业只打一次电话通知，然后不再管你。你不取是你的问题。
> - 优点：减少通知次数，高效
> - 缺点：必须一次读完数据（非阻塞IO + 循环 read），否则数据会丢

```c
// LT模式：每次epoll_wait都会返回就绪的fd，只要缓冲区有数据
// ET模式：只在状态变化时返回一次（无数据→有数据），必须循环读到EAGAIN
while (1) {
    n = read(fd, buf, sizeof(buf));     // 非阻塞读
    if (n == -1 && errno == EAGAIN)
        break;  // 读完
}
```

### 6.3 和 Java / 后端框架的关联

| 技术 | 使用的IO模型 | 说明 |
|------|-------------|------|
| **Netty** | epoll（Linux）/ kqueue（Mac） | Java NIO 底层可以通过 `EpollSelectorProvider` 使用 epoll |
| **Redis** | epoll（单线程） | Redis 6.0 前单线程用 epoll 处理所有连接，性能天花板 |
| **Nginx** | epoll（多进程） | master 进程 + 多个 worker 进程，每个 worker 独立用 epoll |
| **Tomcat NIO** | Java NIO Selector | 基于 epoll（Linux）实现非阻塞IO |
| **Kafka** | Java NIO | Broker 使用 Selector 处理生产者和消费者的连接 |

## 7. 零拷贝（Zero Copy）

> **核心问题**：传统 IO 读取文件并发送到网络，数据在内核态和用户态之间不停地"搬来搬去"，CPU 大量时间浪费在数据拷贝上。
>
> **零拷贝 = 数据不经过用户态，直接在内核态完成传输**

### 7.1 普通 IO vs 零拷贝对比

#### 普通 IO（read + write）—— 4 次拷贝 + 4 次上下文切换

```
磁盘 → (DMA Copy) → 内核缓冲区 → (CPU Copy) → 用户缓冲区  // read()
  → (CPU Copy) → 内核Socket缓冲区 → (DMA Copy) → 网卡      // write()
```

#### sendfile 零拷贝 —— 2 次拷贝 + 2 次上下文切换（Linux 2.4+）

```
磁盘 → (DMA Copy) → 内核缓冲区 → (DMA Copy,带SG-DMA) → 网卡
                      ↓
            Socket缓冲区只记录位置和长度(描述符信息)
```

#### mmap + write —— 3 次拷贝 + 4 次上下文切换

```
磁盘 → (DMA Copy) → 内核缓冲区 ←(映射)→ 用户缓冲区(同一物理内存)
  → (CPU Copy) → 内核Socket缓冲区 → (DMA Copy) → 网卡
```

### 7.2 零拷贝技术对比

| 方案 | 拷贝次数 | 上下文切换 | 适用场景 |
|------|---------|-----------|----------|
| 普通 read/write | 4 | 4 | 需要修改数据的场景 |
| mmap + write | 3 | 4 | 需读取文件内容再处理 |
| sendfile | 2 | 2 | 纯转发（不修改数据） |
| splice | 0（管道传输） | 2 | 两个 fd 之间管道传输 |

### 7.3 和 Java / 中间件的关联

| 中间件 | 使用场景 | 采用的零拷贝方案 |
|--------|---------|-----------------|
| **Kafka** | 消费消息：磁盘 → 网卡 | `FileChannel.transferTo()` = sendfile |
| **Netty** | 文件传输 | `FileRegion` 包装 transferTo |
| **RocketMQ** | 消息存储和消费 | mmap 写 CommitLog，sendfile 发给消费者 |
| **Tomcat/Nginx** | 静态文件服务 | sendfile（Nginx `sendfile on;`） |

```java
// Java中使用零拷贝
FileChannel fromChannel = new FileInputStream("source.txt").getChannel();
FileChannel toChannel = new FileOutputStream("target.txt").getChannel();
// 零拷贝：数据不经过用户态
fromChannel.transferTo(0, fromChannel.size(), toChannel);
```

## 8. Linux 常用命令

> 后端开发必备：线上排查问题的瑞士军刀。

### 8.1 系统资源监控

```bash
# CPU和内存
top          # 实时系统监控，按1看各CPU，按M按内存排序，按P按CPU排序
htop         # top的彩色增强版（需安装）
free -h      # 内存使用情况（-h 人类可读）
vmstat 1     # 每秒输出系统状态（r:运行队列, b:阻塞, cs:上下文切换）

# 磁盘
df -h        # 磁盘分区使用情况
du -sh *     # 当前目录下各文件/文件夹大小
iostat -x 1  # 磁盘IO监控（await: IO等待时间, util: 使用率）

# 网络
netstat -antp    # 所有TCP连接（LISTEN/ESTABLISHED/TIME_WAIT）
ss -antp         # netstat的替代品，更快（推荐）
sar -n DEV 1     # 网络流量统计
```

### 8.2 进程管理

```bash
ps aux | grep java              # 查Java进程
ps -eo pid,ppid,cmd,%mem,%cpu  # 按格式显示进程
kill -9 <PID>                   # 强制杀进程（-9 SIGKILL，不给清理机会）
kill -15 <PID>                  # 优雅停进程（SIGTERM，推荐先试试）
jps -l                          # 查看Java进程（JDK自带）
```

### 8.3 文件排查

```bash
lsof -p <PID>           # 列出进程打开的所有文件（包括socket、pipe）
lsof -i :8080           # 查看8080端口被谁占用
lsof /path/to/file      # 查看文件被哪些进程使用
tail -f app.log         # 动态追踪日志
tail -n 100 app.log     # 看最后100行
grep "ERROR" app.log    # 搜索错误日志
less app.log            # 大文件查看（不一次性加载到内存）
```

### 8.4 调优排查场景速查

| 场景 | 命令组合 |
|------|---------|
| 服务CPU飙高 | `top` → 找到PID → `top -Hp PID` → `jstack PID \| grep -A 10 线程ID(16进制)` |
| 内存飙高 | `free -h` → `jmap -histo:live PID \| head -20`（统计对象数） |
| 磁盘满了 | `df -h` → `du -sh /*` 逐层定位大文件 |
| 端口被占用 | `lsof -i :8080` 或 `ss -antp \| grep 8080` |
| 网络连接异常 | `ss -s`（统计连接状态）→ `ss -antp` 看具体连接 |
| OOM排查 | `dmesg -T \| grep -i oom` → 查看系统日志，找到被杀进程 |
| GC问题 | `jstat -gcutil PID 1000`（每秒输出GC情况） |

## 9. CPU 缓存与伪共享

### 9.1 CPU 缓存层级

> **比喻**：你办公桌上的东西。
>
> - **L1 缓存** = 你手里的笔（32KB，1ns，最快）
> - **L2 缓存** = 桌上的文件架（256KB，3ns）
> - **L3 缓存** = 背后的文件柜（8MB，12ns）
> - **主内存** = 资料室的档案柜（16GB，60ns）
> - **磁盘** = 城市档案馆（1TB，10ms）
>
> 每次你找资料（CPU取数据），先从手里找，找不到才去桌上，再找不到才去柜子……越远越慢。

### 9.2 缓存行 (Cache Line)

缓存不是按字节读取的，而是按 **缓存行**（通常 **64 字节**）读取。

> 比喻：你要书架上的一本书，图书管理员不是只给你那一本书，而是**把那一整排（64本）都抱过来**给你。因为根据"局部性原理"，你很可能接着要看旁边的书。

### 9.3 伪共享 (False Sharing) —— 重要！

**定义**：两个线程修改**不同**的变量，但这两个变量恰好落在同一缓存行中，导致缓存一致性协议（MESI）强制它们互相失效对方的缓存，性能急剧下降。

> 比喻：A和B坐在同一个长条沙发上（同一缓存行）。A想坐下去，B必须站起来；B想坐下去，A必须站起来。两人互相折腾，其实完全可以各坐各的单人沙发（不同缓存行）。

```java
// 伪共享问题示例
public class FalseSharingDemo {
    // 这两个变量很可能在同一缓存行上
    private volatile long x;  // 线程A频繁修改x
    private volatile long y;  // 线程B频繁修改y
}
```

**解决方案**：**缓存行填充 (Padding)**

```java
// Java 8+ 使用 @Contended 注解（需要 -XX:-RestrictContended）
@sun.misc.Contended
private volatile long x;

// 或者手动填充，使变量独占一个缓存行（64字节 = 8个long）
private volatile long p1, p2, p3, p4, p5, p6, p7; // padding
private volatile long x;                            // 实际变量
private volatile long p8, p9, p10, p11, p12, p13, p14; // padding
```

### 9.4 和 Java 并发编程的关联

- **volatile**：写操作会强制刷新缓存行到主存（写屏障），读操作会强制从主存重新加载（读屏障）。本质上操作的是 CPU 缓存一致性。
- **CAS (Compare-And-Swap)**：底层是 CPU 的 `cmpxchg` 指令，配合 `lock` 前缀锁定缓存行（不是锁总线！）。
- **Disruptor 框架**：高性能无锁队列，核心技巧之一就是用缓存行填充消除伪共享。
- **LongAdder vs AtomicLong**：LongAdder 通过分散热点（Cell 数组）减少伪共享冲突，高并发下性能更好。

## 10. 文件系统

### 10.1 inode（索引节点）

> **比喻**：图书馆的管理卡片。
>
> 每本书（文件）都有一张管理卡片（inode），卡片上记录了：
> - 书叫什么名字？—— 文件名（其实文件名在目录项里）
> - 书是哪个出版社的？—— 所有者、权限
> - 书在哪里（哪个书架第几排）？—— 数据块指针
> - 书有多大、什么时候入库的？—— 大小、时间戳
>
> **关键**：卡片的编号（inode号）才是书的唯一标识。书名只是"入口"，一本书可以有多个书名（硬链接），但卡片只有一张。

### 10.2 inode 包含的元数据

```
inode 存储的信息：
  ├── 文件类型和权限（rwx）
  ├── 所有者（UID）和所属组（GID）
  ├── 文件大小（字节数）
  ├── 时间戳（atime: 最后访问, mtime: 最后修改, ctime: 最后状态改变）
  ├── 链接数（硬链接计数）
  └── 数据块指针（直接指针 + 间接指针 + 双重间接 + 三重间接）
```

### 10.3 硬链接 vs 软链接

| 特性 | 硬链接 (Hard Link) | 软链接/符号链接 (Soft Link/Symlink) |
|------|-------------------|-----------------------------------|
| **本质** | 同一个 inode 的多个目录项 | 一个独立文件，内容是指向目标路径的字符串 |
| **inode** | 与原文件**共享**同一个 inode | 有自己**独立的** inode |
| **跨文件系统** | **不可以**（inode 号只在同一文件系统有效） | **可以** |
| **删除原文件** | **不影响**（inode还在，链接数-1，归0才删除） | **失效**（变成"死链接"） |
| **目录链接** | **不支持**（防止循环） | **支持** |
| **类比** | 图书馆有好几个门，每个门上都有书名标签，但指向同一张管理卡片 | 图书馆的"引导牌"："《XXX》请往A区查看"，牌子本身不是书 |

```bash
# 创建硬链接
ln source.txt hardlink.txt
# 创建软链接
ln -s source.txt softlink.txt
# 查看inode号和硬链接计数
ls -li  # 第一列是inode号，第三列是链接数
```

### 10.4 和日常开发的关联

- **Docker 镜像分层**：底层利用 UnionFS（Overlay2），多层文件系统叠加。每层的"白名单/黑名单"就是利用类似硬链接/软链接的文件引用机制。
- **删除文件磁盘空间不释放**：因为还有进程持有该文件的 fd（文件描述符），inode 引用计数不为0。用 `lsof | grep deleted` 排查。
- **日志轮转**：`mv app.log app.log.1` 后空间没释放？因为 Java 进程还在往原 inode 写，需要重新打开日志文件或发信号让进程重载。

## 面试满分话术

### 话术 1：被问"进程和线程的区别"

> "进程是操作系统**资源分配**的最小单位，线程是**CPU调度**的最小单位。
>
> 打个比方：进程好比一家公司，有独立营业执照、银行账户；线程是公司里的员工，共享公司资源但各有各的工作台。
>
> 一个进程崩溃不会影响其他进程（进程隔离），但一个线程崩溃可能导致整个进程挂掉（共享地址空间）。
>
> 在 Java 后端里，我们开的多线程都在同一个 JVM 进程中，共享堆内存，所以并发安全问题特别重要。"

### 话术 2：被问"epoll 为什么比 select 快"

> "select 就像餐厅里服务员一个一个去问每桌客人要不要点菜，不管客人有没有需求都要问一遍（O(n)全量轮询）。
>
> epoll 相当于在每桌装了个按铃，客人有需求才按铃通知服务员（事件驱动），服务员的效率从 O(n) 变成 O(1)。
>
> 底层实现上：select 用固定大小位图存储 fd，每次调用都要把整个 fd_set 从用户态拷贝到内核态；epoll 用红黑树维护所有 fd，用就绪链表只返回有事件的 fd，避免了全量拷贝。
>
> 这就是为什么 Redis、Nginx、Netty 全部用 epoll 做网络模型。"

### 话术 3：被问"零拷贝是什么，Kafka 怎么用的"

> "传统 IO 发送一个文件到网络，数据要在内核态和用户态之间拷贝 4 次，CPU 大量时间花在数据搬运上。
>
> 零拷贝的核心思想：数据不经过用户态，直接在内核态完成传输。
>
> Linux 的 sendfile 系统调用把拷贝降到 2 次（DMA 直接传输），采用 scatter-gather DMA 甚至可以实现更彻底的零拷贝。
>
> Kafka 消费消息时，数据从磁盘的 PageCache 直接通过 sendfile 发到网卡，**完全不经过用户空间**，这就是 Kafka 能跑到百万级吞吐量的核心原因之一。"

## 跨题串联

> 操作系统不是孤岛，这些知识和 JVM、MySQL、Redis、多线程编程紧密相连。面试时主动跨题串联，秒杀大多数候选人。

### 操作系统 → JVM

| OS 概念 | JVM 关联 |
|---------|----------|
| 进程/线程 | JVM 进程中的 Java 线程 = OS 内核线程（1:1 模型） |
| 虚拟内存 | JVM 堆内存、元空间都是虚拟地址空间中的一段 |
| 缺页中断 | 堆内存首次 `new 对象` 时，OS 分配物理内存，类似缺页 |
| 用户态/内核态 | GC 的 safepoint（安全点）、线程切换都涉及内核态 |
| CPU 缓存/伪共享 | volatile、CAS、`@Contended` 底层都依赖缓存一致性 |

### 操作系统 → MySQL

| OS 概念 | MySQL 关联 |
|---------|------------|
| IO 多路复用 | MySQL 8.0 网络层使用 epoll（通过 libevent） |
| 零拷贝 | InnoDB 的 `innodb_flush_method=O_DIRECT` 绕过 OS 缓存 |
| 页面置换 LRU | Buffer Pool 的 LRU 淘汰策略（冷热分离链表） |
| 文件系统 | `redo log`、`undo log`、`binlog` 都是顺序写，利用磁盘顺序IO优势 |

### 操作系统 → Redis

| OS 概念 | Redis 关联 |
|---------|------------|
| IO 多路复用 | Redis 单线程 + epoll 处理数万连接 |
| 零拷贝 | Redis 6.0+ 多线程 IO 也是基于 epoll |
| 页面置换 LRU | Redis 内存淘汰策略（volatile-lru / allkeys-lru） |
| 文件系统 | RDB 快照 = fork() + Copy-on-Write |
| CPU 缓存 | Redis 纯内存操作，充分利用 CPU 缓存命中的高效 |

### 操作系统 → Java 多线程

| OS 概念 | Java 多线程关联 |
|---------|----------------|
| 互斥锁/条件变量 | `synchronized` → OS mutex + `wait/notify` → OS condition variable |
| 信号量 | Java `Semaphore` → OS semaphore |
| CAS 无锁 | CPU 的 `cmpxchg` 指令 + `lock` 前缀（锁缓存行） |
| 伪共享 | `LongAdder` 优于 `AtomicLong`，Disruptor 框架的核心优化 |
| 零拷贝 | `FileChannel.transferTo()`、Netty 的 `FileRegion` |

---

> **总结一句话**：面试时的操作系统题，三个字——**"用比喻"**。每个概念找一个生活场景对应，考官会觉得你"真正懂了"而不是"背了概念"。尤其是 IO 多路复用和零拷贝，一定要联系你熟悉的中间件（Redis/Netty/Kafka）说实际应用，满分妥妥的。
