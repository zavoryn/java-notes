# 零拷贝

> [!tip] 核心速记
> **"传统四拷四切换，sendfile 两拷两切换，Kafka 转发不用搬，数据直飞内核过。"**
> 零拷贝 = 数据不经过用户态，直接在内核态完成传输。

---

## 1. 核心 问题

> 传统 IO 读取文件并发送到网络，数据在内核态和用户态之间不停地"搬来搬去"，CPU 大量时间浪费在数据拷贝上。

---

## 2. 普通 IO vs 零拷贝对比

### 2.1 普通 IO（read + write）—— 4 次拷贝 + 4 次上下文切换

```
磁盘 → (DMA Copy) → 内核缓冲区 → (CPU Copy) → 用户缓冲区  // read()
  → (CPU Copy) → 内核Socket缓冲区 → (DMA Copy) → 网卡      // write()
```

4次拷贝详情：
1. **DMA Copy**：磁盘 → PageCache（内核缓冲区）
2. **CPU Copy**：PageCache → 用户缓冲区（用户态）
3. **CPU Copy**：用户缓冲区 → Socket缓冲区（内核态）
4. **DMA Copy**：Socket缓冲区 → 网卡

4次上下文切换：read()调用 + read()返回 + write()调用 + write()返回

### 2.2 sendfile 零拷贝 —— 2 次拷贝 + 2 次上下文切换（Linux 2.4+）

```
磁盘 → (DMA Copy) → 内核缓冲区 → (DMA Copy,带SG-DMA) → 网卡
                      ↓
            Socket缓冲区只记录位置和长度(描述符信息)
```

- **DMA Copy 1**：磁盘 → PageCache
- **DMA Copy 2**（scatter-gather DMA）：PageCache → 网卡（CPU不参与拷贝！）
- Socket缓冲区只存储文件位置和长度等描述符信息，数据本身不经过Socket缓冲区

### 2.3 mmap + write —— 3 次拷贝 + 4 次上下文切换

```
磁盘 → (DMA Copy) → 内核缓冲区 ←(映射)→ 用户缓冲区(同一物理内存)
  → (CPU Copy) → 内核Socket缓冲区 → (DMA Copy) → 网卡
```

mmap 将文件映射到用户空间，用户缓冲区和内核缓冲区共享同一块物理内存，省去了一次 CPU Copy。但写操作仍需从用户空间拷贝到 Socket 缓冲区。

---

## 3. 零拷贝技术对比

| 方案 | 拷贝次数 | 上下文切换 | 适用场景 |
|------|---------|-----------|----------|
| 普通 read/write | 4 | 4 | 需要修改数据的场景 |
| mmap + write | 3 | 4 | 需读取文件内容再处理 |
| sendfile | 2 | 2 | 纯转发（不修改数据） |
| splice | 0（管道传输） | 2 | 两个 fd 之间管道传输 |

## 4. splice 和 tee 系统调用

### 4.1 splice

splice = 在两个文件描述符之间移动数据，**完全不经过用户空间**（零拷贝）。

```
splice(fd_in, fd_out)
  → 数据从 fd_in 通过内核管道缓冲区直接传输到 fd_out
  → 0次CPU拷贝，0次用户态参与
```

- **限制**：至少有一个 fd 必须是管道（pipe）
- **原理**：数据在内核管道缓冲区中传递，不经过用户空间也不经过 Socket 缓冲区

### 4.2 tee

tee = 从一个管道复制数据到另一个管道，**原始数据不消耗**（零拷贝复制）。

```
tee(fd_in, fd_out)
  → 数据从管道fd_in复制到管道fd_out，fd_in中的数据仍然可读
  → 类似Linux命令 tee：既输出到屏幕又保存到文件
```

- **限制**：两个 fd 都必须是管道
- **适用场景**：数据需要同时发送到多个目的地（如日志同时写磁盘和发到网络）

## 5. PageCache（页缓存）详解

### 5.1 什么是 PageCache？

PageCache = Linux 内核维护的磁盘缓存，将最近读写过的磁盘数据块缓存在物理内存中。

> 比喻：PageCache 就是你桌上常翻的文件夹——最近用过的资料不用每次都去档案室（磁盘）找，桌上（内存）就有副本。

### 5.2 PageCache 为什么能提升性能？

| 操作 | 无PageCache | 有PageCache |
|------|-------------|-------------|
| **读文件** | 每次都走磁盘IO（ms级） | 首次从磁盘加载到PageCache，后续直接从内存读（ns级） |
| **写文件** | 每次都同步写磁盘 | 先写到PageCache（延迟写），后续批量刷盘 |
| **顺序读写** | 依赖磁盘速度 | PageCache预读（readahead）提前缓存后续数据 |

**PageCache 的预读机制**：内核发现应用在顺序读文件，会提前把后续的数据页加载到 PageCache，应用下一次 read() 直接命中缓存。

### 5.3 PageCache 什么时候失效？

| 场景 | 说明 |
|------|------|
| **随机读写** | 预读机制失效，缓存的页可能不会被再次访问 |
| **大文件传输** | 大文件可能占满PageCache，把其他热点数据挤出缓存 |
| **内存压力大** | 系统内存不足时，PageCache会被回收（类似页面置换） |
| **Direct IO** | 绕过PageCache，直接读写磁盘 |
| **多进程竞争** | 多个进程读写不同文件，互相挤占PageCache空间 |

> [!warning] Kafka 的 PageCache 策略
> Kafka 大量依赖 PageCache：
> - **生产者写消息**：写到 PageCache（不是直接写磁盘），依赖内核的延迟写机制
> - **消费者读消息**：如果消息还在 PageCache 中（大概率），直接从内存读，不走磁盘
> - Kafka 不用 mmap 写日志的原因：mmap 的脏页回写时机不可控，Kafka 更倾向于通过 PageCache + flush 控制

## 6. Direct IO vs 缓冲IO

| 维度 | 缓冲IO（默认） | Direct IO（O_DIRECT） |
|------|---------------|----------------------|
| **是否经过PageCache** | 是（数据先到PageCache再到用户空间） | 否（直接读写磁盘，绕过PageCache） |
| **性能特点** | 小文件、顺序读写快（PageCache命中率高） | 大文件、自建缓存快（避免PageCache污染） |
| **适用场景** | 大多数场景 | 数据库（MySQL InnoDB的`O_DIRECT`）、自管理缓存的应用 |
| **限制** | 无 | 必须对齐到页大小（4KB），必须用对齐的缓冲区 |

> [!note] MySQL 为什么用 O_DIRECT？
> InnoDB 自己有 Buffer Pool 做缓存管理，如果再经过 PageCache = **双重缓存**（浪费内存 + 数据一致性难保证）。
> 所以 InnoDB 设置 `innodb_flush_method=O_DIRECT`，绕过 PageCache，直接管理缓存。

## 7. Kafka 高性能三要素详解

> [!warning] 这题面试必考！

| 要素 | 原理 | 为什么快 |
|------|------|----------|
| **零拷贝** | 消费者读消息：磁盘 → PageCache → sendfile → 网卡 | 数据不经过用户空间，4次拷贝降到2次 |
| **顺序写** | 追加写入CommitLog（append-only） | 顺序写磁盘速度接近内存（600MB/s vs 随机写100MB/s） |
| **PageCache** | 生产者写→PageCache（延迟写），消费者读→PageCache命中 | 大部分读写都在内存完成，极少走磁盘 |

### 7.1 详细解释

**① 零拷贝（sendfile）**
```java
// Kafka消费者传输消息的底层
FileChannel channel = new FileInputStream(logFile).getChannel();
channel.transferTo(0, channel.size(), socketChannel); // sendfile零拷贝
```
- 传统方式：磁盘 → PageCache → 用户缓冲区 → Socket缓冲区 → 网卡（4次拷贝）
- sendfile方式：磁盘 → PageCache → 网卡（2次拷贝，CPU不参与数据搬运）

**② 顺序写**
- 磁盘顺序写速度：600MB/s（接近内存读写速度）
- 磁盘随机写速度：100MB/s（因为需要频繁寻道）
- Kafka 的 CommitLog 是 append-only，永远顺序追加

**③ PageCache**
- 生产者写入：数据先到 PageCache，内核延迟批量刷盘（不是每次都 fsync）
- 消费者读取：如果消息还在 PageCache（最近写入的消息大概率还在），直接从内存读取
- 结果：大部分情况下，消费者读消息**完全不走磁盘**！

### 7.2 三要素的协同关系

```
生产者写入消息 → 顺序写追加到CommitLog → 先写入PageCache（延迟写）
  → PageCache中的数据通过内核回写机制（或主动flush）落到磁盘

消费者读取消息 → 从PageCache读（大概率命中）→ 如果命中：直接内存返回
  → 如果未命中：从磁盘读 → 同时加载到PageCache → 下次消费者可能命中
  → 数据从PageCache通过sendfile零拷贝发送到网卡 → 不经过用户空间！
```

> [!tip] 面试满分话术
> "Kafka 性能高的原因有三个：零拷贝、顺序写、PageCache。
> 零拷贝让消费者读消息时数据从 PageCache 直接通过 sendfile 发到网卡，不经过用户空间，拷贝次数从4降到2。
> 顺序写让磁盘写入速度接近内存水平，因为磁盘顺序写可以达到600MB/s。
> PageCache 让大部分读写操作都在内存中完成——生产者写先到PageCache，消费者读也在PageCache命中。
> 三者协同：写入时利用顺序写+PageCache保证高吞吐，消费时利用PageCache命中+零拷贝保证低延迟。"

## 8. 和 Java / 中间件的关联

| 中间件 | 使用场景 | 采用的零拷贝方案 |
|--------|---------|-----------------|
| **Kafka** | 消费消息：磁盘 → 网卡 | `FileChannel.transferTo()` = sendfile |
| **Netty** | 文件传输 | `FileRegion` 包装 transferTo |
| **RocketMQ** | 消息存储和消费 | mmap 写 CommitLog，sendfile 发给消费者 |
| **Tomcat/Nginx** | 静态文件服务 | sendfile（Nginx `sendfile on;`） |

```java
// Java中使用零拷贝
FileChannel fromChannel = new FileInputStream("source.txt").getChannel();
FileChannel toChannel = new FileOutputStream("target.txt").getChannel();
// 零拷贝：数据不经过用户态
fromChannel.transferTo(0, fromChannel.size(), toChannel);
```

## 面试题速查

> [!example] 本模块高频面试题

1. **什么是零拷贝？sendfile和mmap有什么区别？**
   - 零拷贝=数据不经过用户态；sendfile=2次拷贝2次切换纯转发，mmap=3次拷贝4次切换需修改数据

2. **PageCache是什么？为什么能提升性能？什么场景失效？**
   - PageCache=内核磁盘缓存，最近读写的数据缓存在内存；随机读写、大文件、DirectIO时失效

3. **Direct IO vs 缓冲IO？MySQL为什么用O_DIRECT？**
   - DirectIO绕过PageCache直接读写磁盘；MySQL有自己BufferPool，避免双重缓存

4. **Kafka高性能三要素？**
   - 零拷贝(sendfile) + 顺序写(append-only) + PageCache(读写都在内存)

5. **splice和tee系统调用？**
   - splice=两个fd通过管道零拷贝传输（至少一个fd是管道）；tee=管道间零拷贝复制

6. **为什么 Kafka 性能这么高？（零拷贝 + 顺序写 + PageCache）**
   - 三者协同：写入顺序写+PageCache保证高吞吐，消费PageCache+零拷贝保证低延迟
