## 🔴 缺失知识点：异常处理机制

> 你的笔记涵盖了参数、流程、拒绝策略、配置、状态，但**没涉及 `execute()` 和 `submit()` 的异常处理差异**——这是面试高频追问。

### execute() vs submit() 异常处理

| 提交方式 | 异常行为 | 后果 |
|----------|---------|------|
| **`execute(Runnable)`** | 任务抛出未捕获异常 → **线程直接终止** | 线程池检测到线程死亡，**自动创建新线程替代** |
| **`submit(Runnable/Callable)`** | 异常被**捕获并封装**在 `Future` 对象中 | 调用 `Future.get()` 时才抛出 `ExecutionException` |

```java
// execute() 异常——线程会死！
executor.execute(() -> {
    throw new RuntimeException("execute 异常");
    // 该线程终止！线程池会偷偷补一个新的
});

// submit() 异常——被吞了，get() 才暴露
Future<?> future = executor.submit(() -> {
    throw new RuntimeException("submit 异常");
});
// 此时异常已被封装，线程没死！
try {
    future.get();  // 这里才会抛出 ExecutionException
} catch (ExecutionException e) {
    e.getCause();  // 原始 RuntimeException
}
```

> ⚠️ **最佳实践**：无论用哪种方式，都在 `Runnable/Callable` 内部用 `try-catch` 兜底。既防止线程意外死亡，也避免异常被无声吞掉。

### 为什么 execute 异常会导致线程终止？

```java
// ThreadPoolExecutor.Worker.runWorker() 简化逻辑
final void runWorker(Worker w) {
    try {
        while (task != null || (task = getTask()) != null) {
            // 执行任务
            task.run();  // ← 这里抛未捕获异常
        }
        completedAbruptly = false;
    } finally {
        processWorkerExit(w, completedAbruptly);  // completedAbruptly=true
        // → 线程被移除，线程池补充新线程
    }
}
```

### Q1: 线程池相比每次新建线程有哪些实际好处？

- **降低资源消耗**：复用已创建线程，减少频繁创建/销毁的系统调用及内存开销
- **提升响应速度**：任务到达时直接分配空闲线程，无需等待线程创建
- **统一管理与控制**：可限制最大并发数，防止资源耗尽；便于监控与调优
- **避免线程泄漏**：集中管理生命周期，减少异常导致的线程残留

### Q2: ThreadPoolExecutor 的七个核心参数分别影响什么行为？

- `corePoolSize`：常驻线程数 → 影响日常吞吐
- `maximumPoolSize`：峰值线程数 → 影响突发应对能力
- `keepAliveTime`：非核心线程回收时间 → 影响资源释放速度
- `workQueue`：队列类型直接决定 `maximumPoolSize` 是否生效（无界队列使其失效）
- `threadFactory`：线程命名与属性 → 影响问题排查
- `handler`：饱和后的行为 → 影响系统稳定性

> 与你笔记第二节一致 ✅

### Q3: 任务提交到线程池时，其内部判定顺序是什么？

核心线程未满 → 创建核心线程 → 核心满则入队 → 队满且未达最大则创建非核心线程 → 全满则拒绝。

> 与你笔记第三节「核心→队列→最大→拒绝」一致 ✅

### Q4: 为何生产环境不建议用 Executors 创建线程池？

- `FixedThreadPool` / `SingleThreadExecutor`：无界 `LinkedBlockingQueue` → 任务堆积 OOM
- `CachedThreadPool`：`maximumPoolSize = Integer.MAX_VALUE` → 线程爆炸 OOM
- 缺少拒绝策略定制，默认 `AbortPolicy` 可能影响业务
- **最佳实践**：手动 `new ThreadPoolExecutor(...)` 显式指定所有参数

> 与你笔记第五节一致 ✅

### Q5: 关闭线程池时 shutdown 与 shutdownNow 有何差异？

| | shutdown() | shutdownNow() |
|:---|:---|:---|
| 新任务 | ❌ 拒绝 | ❌ 拒绝 |
| 队列任务 | ✅ 继续处理 | ❌ 清空并返回列表 |
| 执行中任务 | ✅ 等待完成 | ⚠️ 尝试 interrupt |
| 状态 | SHUTDOWN | STOP |

**优雅关闭流程**：先 `shutdown()` → `awaitTermination(timeout)` → 超时则 `shutdownNow()` 兜底。
