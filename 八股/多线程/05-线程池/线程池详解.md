# 线程池核心原理

> [!summary] 我的速记
> **线程池 = 核心线程（常驻） + 阻塞队列（缓冲） + 最大线程（救急） + 拒绝策略（兜底）。**
> **工作流程口诀：核心 → 队列 → 最大 → 拒绝。**
> **7 大参数：核心数、最大数、保活时间、时间单位、队列、线程工厂、拒绝策略。**
> **为什么不用 Executors？** FixedThreadPool/CachedThreadPool 的队列无限长或线程无限多 → OOM 风险。

---

## 一、为什么需要线程池？

```
创建线程的开销：
  分配内存 → 初始化线程栈（默认 1MB）→ 线程注册到 OS →
  用户态-内核态切换 → 线程调度

销毁线程的开销：
  回收线程栈 → OS 清理 → 通知调度器

线程池的思想：
  提前创建好一批线程 → 任务来了直接分配 → 用完回收（不销毁）
  = 控制资源 + 复用线程 + 提高响应速度
```

| 优势 | 说明 |
|------|------|
| **降低资源消耗** | 复用已有线程，避免反复创建/销毁 |
| **提高响应速度** | 任务到达时不需要等线程创建 |
| **统一管理** | 可以统一分配、监控、调优 |
| **防止资源耗尽** | 限制最大线程数，不会无限创建线程 |

## 二、ThreadPoolExecutor 7 大参数

```java
public ThreadPoolExecutor(
    int corePoolSize,              // ① 核心线程数（常驻的）
    int maximumPoolSize,           // ② 最大线程数
    long keepAliveTime,            // ③ 非核心线程空闲存活时间
    TimeUnit unit,                 // ④ 时间单位
    BlockingQueue<Runnable> workQueue,  // ⑤ 任务队列
    ThreadFactory threadFactory,   // ⑥ 线程工厂（命名、设daemon等）
    RejectedExecutionHandler handler // ⑦ 拒绝策略
)
```

| 参数 | 含义 | 记忆要点 |
|------|------|----------|
| corePoolSize | 核心线程数，即使空闲也存活 | "店里的正式员工" |
| maximumPoolSize | 最大线程数（核心 + 非核心） | "忙时能招的最大人数" |
| keepAliveTime | 非核心线程空闲多久后被回收 | "临时工闲置多久被辞退" |
| workQueue | 存放等待执行的任务 | "候客区" |
| threadFactory | 创建线程的工厂（可自定义线程名） | "员工编号规则" |
| handler | 线程和队列都满了怎么处理 | "客满拒客的规则" |

## 三、工作流程——面试必画的四步走

```
新任务进来
    │
    ├── 核心线程数 < corePoolSize？
    │      ├── 是 → 创建新线程（不论有没有空闲线程）
    │      └── 否 ↓
    │
    ├── 队列满了没？
    │      ├── 没满 → 放入队列等待
    │      └── 满了 ↓
    │
    ├── 当前线程数 < maximumPoolSize？
    │      ├── 是 → 创建非核心线程处理
    │      └── 否 ↓
    │
    └── 执行拒绝策略
```

```
举例（core=2, max=5, queue=3）：
  任务1 → 创建核心线程1
  任务2 → 创建核心线程2
  任务3 → 入队 (队列1)
  任务4 → 入队 (队列2)
  任务5 → 入队 (队列3)
  任务6 → 队列满了！创建线程3（非核心）
  任务7 → 队列满了！创建线程4（非核心）
  任务8 → 队列满了！创建线程5（非核心）
  任务9 → 队列满了 + 5个线程都在忙 → 拒绝！
```

> 🧠 **关键细节：核心线程数满了不是马上扩到最大，而是先放队列。队列才是第一道缓冲。非核心线程在空闲 keepAliveTime 后会被回收。**

## 四、四种拒绝策略

| 策略 | 行为 | 场景 |
|------|------|------|
| **AbortPolicy**（默认） | 抛 `RejectedExecutionException` | 需要感知任务被拒绝 |
| **CallerRunsPolicy** | 交给提交任务的线程自己执行 | 降级——让提交者慢下来，形成天然背压 |
| **DiscardPolicy** | 直接丢弃，不通知 | 不重要任务（非核心日志等） |
| **DiscardOldestPolicy** | 丢弃队列中最老的任务，重新提交 | 优先处理最新数据 |

```java
// CallerRunsPolicy 的妙用——天然背压
new ThreadPoolExecutor(
    2, 4, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(10),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
// 所有线程忙 + 队列满 → 任务由主线程自己执行
// 主线程在忙 → 不会继续提交新任务 → 系统压力自然下降
```

## 五、常见线程池 & 为什么不推荐用 Executors

| 线程池 | 核心=最大 | 队列 | 问题 |
|--------|-----------|------|------|
| **newFixedThreadPool(n)** | n=n | LinkedBlockingQueue（**无界**！） | 队列无限 → 任务堆积 → OOM |
| **newCachedThreadPool()** | 0=Integer.MAX_VALUE | SynchronousQueue（不存任务） | 线程无限 → CPU 耗尽 → OOM |
| **newSingleThreadExecutor** | 1=1 | LinkedBlockingQueue（**无界**！） | 队列无限 → OOM |
| **newScheduledThreadPool(n)** | n=Integer.MAX_VALUE | DelayedWorkQueue | 线程无限风险 |

> 🧠 **阿里规约（泰山）：** 禁止使用 Executors 创建线程池——必须用 `new ThreadPoolExecutor(...)` 显式指定参数。目的：让开发者清楚地知道线程池的参数和风险。

```java
// ❌ 禁止
ExecutorService pool1 = Executors.newFixedThreadPool(10);
ExecutorService pool2 = Executors.newCachedThreadPool();

// ✅ 正确
ExecutorService pool = new ThreadPoolExecutor(
    10,                        // 核心线程数
    20,                        // 最大线程数
    60L, TimeUnit.SECONDS,     // 空闲存活
    new LinkedBlockingQueue<>(100),  // 有界队列！
    new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build(),  // 有意义的线程名
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

## 六、如何合理配置线程数？

| 任务类型        | 公式                                 | 原因                         |
| ----------- | ---------------------------------- | -------------------------- |
| **CPU 密集型** | `N_cpu + 1`                        | CPU 核数够用，多 1 个应对偶尔的缺页中断/停顿 |
| **IO 密集型**  | `N_cpu * 2` 或 `N_cpu / (1 - 阻塞系数)` | IO 等待时 CPU 空闲，可以多开线程       |
| **混合型**     | 拆成 CPU 任务池 + IO 任务池                | 分别配置，互不影响                  |

```
实际经验：
  CPU 密集型（计算、加密、压缩）→ CPU 核数 + 1
  IO 密集型（数据库、网络、文件）→ CPU 核数 × 2 起，具体根据 RT 和 QPS 压测调优

阻塞系数 = 任务等待 IO 的时间 / 任务总时间
线程数 = CPU 核数 / (1 - 阻塞系数)
  例：8核CPU，任务 50% 时间等 IO → 8 / (1-0.5) = 16
```

> 🧠 **关键思想：CPU 密集宁可少开（避免上下文切换开销），IO 密集可以多开（让 CPU 在等 IO 时有事做）。**

## 七、线程池状态 + 生命周期

```
线程池 5 种状态：

RUNNING (-1) ← 初始状态，接收新任务 + 处理队列任务
    │  shutdown()
    ▼
SHUTDOWN (0) ← 不接收新任务，但处理队列中的任务
    │  shutdownNow()
    ▼
STOP (1) ← 不接收新任务，不处理队列任务，中断正在执行的任务
    │  队列空 + 线程全停
    ▼
TIDYING (2) ← 过渡状态，所有任务完成，线程数=0
    │  terminated() 钩子方法
    ▼
TERMINATED (3) ← 线程池终止
```

```java
// shutdown vs shutdownNow
pool.shutdown();       // 温柔：不接新活，把手头的活干完
pool.shutdownNow();    // 粗暴：不接新活，中断正在执行的线程，返回队列中的任务列表
```

## 八、面试满分话术

> **面试官：** 讲讲线程池的工作流程？

**你的回答：**

> **第一层（一句话定位）：** 线程池通过复用线程避免反复创建/销毁的开销，同时限制线程数防止资源耗尽。
>
> **第二层（参数+流程）：** ThreadPoolExecutor 有 7 个核心参数。工作流程四步走——核心→队列→最大→拒绝。新任务来了先看核心线程满了没，没满就创建核心线程；核心满了先放任务队列（第一道缓冲）；队列也满了且没到最大线程数，创建非核心线程救急；全满了就走拒绝策略。非核心线程空闲一段时间后会被回收。
>
> **第三层（选型陷阱）：** Executors 的 newFixedThreadPool 和 newCachedThreadPool 分别用了无界队列和无界线程数——前者任务堆积 OOM，后者线程爆炸 CPU 耗尽。阿里规约明确禁止用 Executors，必须手动创建 ThreadPoolExecutor 显式指定有界队列。
>
> **第四层（配置经验）：** CPU 密集型用 CPU 核数+1，IO 密集型用 CPU 核数×2 起——具体要根据压测调优。拒绝策略用 CallerRunsPolicy 可以形成天然背压——提交任务的线程自己执行，自然减速。

> **追问：核心线程可以回收吗？**
>
> 默认不可以。但 ThreadPoolExecutor 有一个 `allowCoreThreadTimeOut(true)` 方法——设为 true 后核心线程也会因为空闲超时被回收。一般不需要这么激进。
