# synchronized 锁升级机制

> [!summary] 我的速记
> **锁升级四阶段：无锁 → 偏向锁 → 轻量级锁（自旋） → 重量级锁（挂起）。不可降级！**
> **核心思想：长路浪费 vs 短跑热身。** 小竞争用自旋（轻量级），大竞争用挂起（重量级），单线程用偏向锁。
> **对象头 Mark Word 是这一切的物理基础——锁信息全存在这 64 位里。**

---

## 一、为什么要有锁升级？——从一次性能教训说起

**JDK5 及以前：** synchronized 直接是重量级锁 → 线程阻塞/唤醒涉及用户态-内核态切换 → 极其昂贵。

```
重量级锁的昂贵路径：
  Java 线程阻塞 → 系统调用 → 内核态 → OS 线程挂起 → 等待 →
  OS 线程唤醒 → 切换到用户态 → Java 线程恢复
  ↑ 一次切换约 5-10 微秒，并发高时开销惊人
```

**JDK6 的革命：** 引入锁升级——"根据竞争激烈程度，逐步升级锁的等级"。思路是：

> 🧠 **如果锁只被一个线程持有（大多数情况），为什么要走昂贵的系统调用？先用简单的 CAS 试试。**

## 二、Java 对象头——锁信息的物理载体

每个 Java 对象在堆内存中由三部分组成：

```
┌──────────────────────────────────────────────────┐
│                Java 对象内存布局                    │
├──────────────┬──────────────┬────────────────────┤
│  对象头(Header)│  实例数据(Data)│  对齐填充(Padding)  │
├──────┬───────┤              │                    │
│Mark  │Klass  │              │                    │
│Word  │Pointer│              │                    │
│64bit │64bit  │              │                    │
└──────┴───────┴──────────────┴────────────────────┘
```

> 🧠 **Mark Word（64位）在不同锁状态下的结构完全不同：**

| 锁状态 | bit 62 | 锁标志位(bits) | 存储内容 |
|--------|--------|----------------|----------|
| 无锁 | 0 | 01 | 对象的 hashCode、分代年龄 |
| 偏向锁 | 线程ID(54bit) + Epoch(2bit) | 01 | 偏向的线程 ID |
| 轻量级锁 | 指向栈中 Lock Record 的指针(62bit) | 00 | 指向线程栈的指针 |
| 重量级锁 | 指向 Monitor 对象的指针(62bit) | 10 | 指向 ObjectMonitor |
| GC 标记 | — | 11 | 仅供 GC 使用 |

> 🧠 **关键：偏向锁和轻量级锁的标志位不同——偏向锁存储的是线程 ID，轻量级锁存储的是栈指针。**

## 三、锁升级完整路径

```
     无锁 (001)
        │
        │ ① 当同一个线程反复获取该锁
        ▼
     偏向锁 (101, thread_id)
        │
        │ ② 另一个线程试图获取锁，但原持有线程已退出同步块
        ▼                                         OR 原持有线程还在执行
     轻量级锁 (00) ──────────────────────┐
        │  CAS 自旋获取                   │
        │                                 │
        │ ③ 自旋失败次数过多              │ ②' 交替竞争激烈
        │    (默认自旋10次或等待线程       │     (CAS 修改失败)
        │     数 > CPU核数/2)             │
        ▼                                 ▼
     重量级锁 (10)               直接升级！
        │
      线程挂起，进入 Monitor 等待队列
        │
      不可降级！（一旦变成重量级锁，不会变回轻量级）
```

---

### 3.1 偏向锁（Biased Locking）

**核心思想：** "这个锁大概率只有一个线程在用，把那个线程的 ID 记下来，下次它再来直接放行。"

**工作流程：**

```
1. 线程 A 第一次访问 synchronized 块
   → CAS 将自己的线程 ID 写入 Mark Word
   → 成功 → 获取偏向锁

2. 线程 A 再次访问同一个 synchronized 块
   → 检查 Mark Word 中的线程 ID == 自己？
   → 是 → 直接进入（不需要 CAS，零开销！）

3. 线程 B 尝试访问
   → 检查线程 ID ≠ 自己
   → 等待全局安全点（SafePoint）
   → 检查线程 A 是否还存活？
       ├── 存活 → 撤销偏向锁 → 升级为轻量级锁
       └── 不存活 → 重置为无锁 → 重新偏向给线程 B
```

> 🧠 **偏向锁的收益：** 适合"一个线程反复进出同一个锁"的场景（Vector、StringBuffer、Hashtable 的内部方法）。Java 15 默认关闭了偏向锁（JDK15 JEP 374），因为这些老类已被替代，偏向锁的维护成本 > 收益。

---

### 3.2 轻量级锁（Lightweight Locking）

**核心思想：** "有两个线程在争，但不见得同时争——用 CAS 自旋试试，别急着挂起。"

**工作流程：**

```
1. 线程 A 持有轻量级锁
   → 在线程栈中创建 Lock Record
   → 将对象头的 Mark Word 拷贝到 Lock Record
   → CAS 将对象头的 Mark Word 更新为指向 Lock Record 的指针

2. 线程 B 尝试获取
   → CAS 失败（Mark Word 已指向线程 A 的 Lock Record）
   → 自旋等待（空循环，反复 CAS 尝试）

3. 自旋 10 次失败 或 自旋等待的线程数 > CPU 核数/2
   → 膨胀为重量级锁
```

> 🧠 **自旋的取舍：** 自旋消耗 CPU（忙等待），但不涉及系统调用。如果锁持有时间极短（几个指令），自旋完直接拿到锁——比挂起+唤醒快。如果持有时间长 → 白白浪费 CPU → 必须膨胀。

**自适应自旋（JDK6+）：** JVM 根据前一次自旋的结果动态调整自旋次数。上次自旋成功拿到锁 → 下次多自旋一会儿；上次失败 → 下次少自旋。

---

### 3.3 重量级锁（Heavyweight Locking）

**核心思想：** "竞争太激烈了，别自旋了，直接挂起，用 OS 的 Monitor 来管。"

```
重量级锁 = ObjectMonitor 对象：

ObjectMonitor {
    _owner        = NULL;        // 当前持有锁的线程
    _WaitSet      = { ... };     // 调了 wait() 的线程队列
    _EntryList    = { ... };     // 等待获取锁的线程队列（BLOCKED）
    _count        = 0;           // 重入计数
    _recursions   = 0;
}
```

**工作流程：**

```
1. 线程进入 synchronized → cxq (ContentionList) 入队
2. 通过 CAS 竞争成为 _owner
3. 拿到锁 → 执行同步块
4. 退出 synchronized → 唤醒 _EntryList 中的等待线程
```

> 🧠 **一个对象一个 Monitor——每个 Java 对象都可以当锁。**

## 四、两个 JIT 优化：锁消除 + 锁粗化

| 优化 | 描述 | 示例 |
|------|------|------|
| **锁消除** | 编译器分析发现锁对象只被一个线程访问 → 直接去掉锁 | `StringBuffer.append()` 在局部变量中 |
| **锁粗化** | 连续对同一对象加锁解锁 → 合并成一次大锁 | `for` 循环里的 `synchronized` |

```java
// 锁消除示例
public String concat(String a, String b) {
    StringBuffer sb = new StringBuffer();  // 局部变量！只有一个线程能访问
    sb.append(a);
    sb.append(b);
    return sb.toString();
    // JIT 检测到 sb 没有逃逸 → 去掉 append() 内部的 synchronized
}

// 锁粗化示例
for (int i = 0; i < 100; i++) {
    synchronized (lock) { doSomething(); }  // 反复加锁解锁 100 次
}
// JIT 优化后相当于：
synchronized (lock) { for (int i = 0; i < 100; i++) doSomething(); }
```

## 五、synchronized vs ReentrantLock（面试高频对比）

| 对比维度 | synchronized | ReentrantLock |
|----------|--------------|---------------|
| **实现层面** | JVM 内置（关键字），C++ 实现 | JDK 类（java.util.concurrent），Java 实现 |
| **锁释放** | 自动释放（出同步块 / 异常） | 必须手动 `unlock()`，通常在 finally |
| **等待可中断** | ❌ 等不到锁就一直等 | ✅ `lockInterruptibly()` 可响应中断 |
| **公平锁** | ❌ 只有非公平 | ✅ 可选公平/非公平（构造器传 true） |
| **条件变量** | `wait/notify` 一个条件队列 | `Condition` 可多条件队列 |
| **尝试获取** | ❌ 不行 | ✅ `tryLock()` 立即返回 / `tryLock(time, unit)` |
| **性能** | JDK6 优化后差距小 | 略优于 synchronized（高并发下） |
| **使用建议** | 简单场景，默认用 | 需要高级特性（公平/超时/多条件）时用 |

> 🧠 **选型口诀：简单用 synchronized，高级用 Lock。JDK6 以后 synchronized 性能已不差。**

## 六、面试满分话术

> **面试官：** synchronized 的锁升级过程？

**你的回答：**

> **一句话定位：** JDK6 引入锁升级是为了避免每次都用重量级锁带来的内核态切换开销——根据竞争程度逐步升级。
>
> **完整路径（四阶段）：**
> 1. **偏向锁：** 锁对象头里记下第一个访问它的线程 ID。这个线程再来时，检查 ID 匹配直接放行——零开销。这是对"大多数锁只被一个线程用"的优化。JDK15 后默认关闭，因为收益递减。
> 2. **轻量级锁：** 第二个线程来了，撤销偏向锁。线程在线程栈里建 Lock Record，通过 CAS 尝试把对象头指向自己的 Lock Record。没拿到锁的线程 CAS 自旋等待——适合锁持有时间极短的场景。自旋消耗 CPU 但不涉及系统调用。
> 3. **重量级锁：** 自旋超过一定次数（默认 10 次，JVM 自适应调整）或等待线程过多 → 膨胀。对象头指向 ObjectMonitor，未拿到锁的线程进入阻塞队列被 OS 挂起——涉及用户态/内核态切换。锁一旦膨胀为重量级锁，不可降级。
> 4. **JIT 的两个优化：** 锁消除（局部变量上的锁直接删掉）和锁粗化（连续加锁合并成一次）。
>
> **面试亮点：** 能说出 Mark Word 在不同状态下存的内容（偏向存线程 ID、轻量存 Lock Record 指针、重量存 Monitor 指针），以及标志位（01 可能是无锁也可能是偏向，靠是否记录了线程 ID 区分）。

> **追问：synchronized 和 Lock 怎么选？**
>
> synchronized 自动释放锁（代码块结束或异常），Lock 必须手动 unlock。选型上——简单场景默认 synchronized，代码更安全；需要公平锁、可中断获取、超时尝试、多条件队列时用 Lock。JDK6 优化后二者性能差距已经不大。

## 常见面试问题清单

| #   | 问题                                   | 回答要点                                                              |
| --- | ------------------------------------ | ----------------------------------------------------------------- |
| Q1  | **synchronized 底层原理？**               | 基于 JVM Monitor，对象头 Mark Word 存锁信息。JDK6 锁升级：无锁→偏向→轻量→重量            |
| Q2  | **锁升级四个阶段分别是什么？**                    | 偏向（记线程ID，无竞争零开销）→ 轻量（CAS自旋，适合短临界区）→ 重量（OS挂起，内核态切换）                |
| Q3  | **Mark Word 在不同锁状态下存什么？**            | 无锁：hashCode+分代年龄；偏向：线程ID；轻量：Lock Record指针；重量：Monitor指针            |
| Q4  | **什么是锁消除和锁粗化？**                      | 锁消除：JIT 检测到局部变量不逃逸→去掉锁；锁粗化：连续加锁解锁→合并成一次                           |
| Q5  | **synchronized 和 ReentrantLock 区别？** | 自动释放 vs 手动 unlock；不可中断 vs 可中断；非公平 only vs 可选公平；单条件 vs 多 Condition |
| Q6  | **轻量级锁什么时候膨胀为重量级？**                  | 自旋超过阈值（默认10次自适应）或等待线程数 > CPU核数/2                                  |
