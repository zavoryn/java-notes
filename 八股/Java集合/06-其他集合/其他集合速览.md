# 其他集合速览

> [!summary] 我的速记
> **去重 HashSet、排序 TreeMap、LRU LinkedHashMap、队列 ArrayDeque。**
> - HashSet = HashMap + PRESENT 占位符，add(e) = map.put(e, PRESENT)
> - HashTable = 全表 synchronized，已淘汰，用 ConcurrentHashMap 替代
> - LinkedHashMap = HashMap + 双向链表，accessOrder=true 就是 LRU 缓存
> - TreeMap = 红黑树，按 key 自然顺序或 Comparator 排序，O(logn)
> - ArrayDeque = 循环数组双端队列，无 Node 开销；PriorityQueue = 完全二叉堆

## 一、HashSet

### 1.1 底层实现

```java
// HashSet 底层就是一个 HashMap！
public class HashSet<E> extends AbstractSet<E>
    implements Set<E>, Cloneable, Serializable {
    private transient HashMap<E, Object> map;   // 核心：HashMap 实例
    private static final Object PRESENT = new Object();  // 占位符

    public boolean add(E e) {
        return map.put(e, PRESENT) == null;    // add(e) = map.put(e, PRESENT)
    }

    public boolean remove(E e) {
        return map.remove(e) == PRESENT;
    }

    public boolean contains(E e) {
        return map.containsKey(e);              // contains = map.containsKey
    }
}
```

> [!important] HashSet 的每一个操作都是委托给 HashMap。key 是元素本身，value 统一存 PRESENT（`new Object()`）——一个极轻量的占位对象，只是为了让 HashMap 的 put 方法能工作（put 需要两个参数）。

### 1.2 为什么底层是 HashMap？

```text
HashSet 只需要 "元素是否存在" 的功能（Set 语义），
但 HashMap 的 put(key, value) 需要两个参数。

解决方案：value 统一用 PRESENT 占位：
  - PRESENT = new Object()，只占 16 字节（对象头），极轻量
  - map.put(e, PRESENT) 返回旧 value → 如果是 null 说明 key 不存在（插入成功）
  - map.put(e, PRESENT) 返回 PRESENT → 说明 key 已存在（add 返回 false）

这比自己实现一套 hash + 桶结构要简单得多，而且复用了 HashMap 的所有优化
```

> [!tip] 面试追问："为什么 HashSet 只存一个值却要两个参数？" → 底层复用了 HashMap 的整套 hash/桶/扩容机制，PRESENT 只是一个 16 字节的占位符，代价极小，换来的是不用重复实现整套 hash 逻辑。

### 1.3 HashSet 的特性（继承自 HashMap）

| 特性 | HashSet | 来源（HashMap） |
|------|---------|---------------|
| 去重 | 基于 equals + hashCode | HashMap 的 key 去重 |
| 无序 | 不保证插入顺序 | HashMap 桶分布无序 |
| 允许 null | 允许一个 null 元素 | HashMap 允许一个 null key |
| 线程不安全 | 不安全 | HashMap 本身不安全 |
| 扩容 | 同 HashMap（0.75 负载因子，2 倍扩容） | 完全复用 |

## 二、HashTable

### 2.1 与 HashMap 对比

```java
// HashTable —— 全表锁（已淘汰）
public class Hashtable<K, V> extends Dictionary<K, V>
    implements Map<K, V>, Cloneable, Serializable {

    public synchronized V put(K key, V value) { ... }    // synchronized 整个方法
    public synchronized V get(Object key) { ... }        // synchronized 整个方法
    public synchronized V remove(Object key) { ... }     // synchronized 整个方法
    public synchronized int size() { ... }               // synchronized 整个方法
    // 每个公开方法都加了 synchronized → 全表锁！
}
```

| 比较维度 | HashMap | HashTable |
|---------|---------|-----------|
| 线程安全 | 不安全 | 全表 synchronized（安全但性能差） |
| null key/value | 允许 1 个 null key + 多个 null value | 不允许任何 null |
| 继承 | AbstractMap | Dictionary（遗留类） |
| 初始容量 | 16 | 11 |
| 扩容因子 | 2 倍 | 2n+1 |
| 迭代器 | Iterator（fail-fast） | Enumerator（非 fail-fast） |
| 状态 | 主流 | 已淘汰（Legacy） |

> [!danger] HashTable 是遗留类，面试中提到 HashTable 时必须说：全表锁已淘汰，线程安全场景用 ConcurrentHashMap 替代。

### 2.2 HashTable 为什么不允许 null？

```text
HashTable 不允许 null key / null value 的原因：
  如果 put(null, value)，HashTable 内部会调用 key.hashCode()
  → null.hashCode() → NullPointerException！

HashMap 允许 null key 的原因：
  HashMap 对 null key 做了特殊处理，hash 值固定为 0，放在桶[0]
  → 不需要调用 key.hashCode()
```

> [!note] ConcurrentHashMap 同样不允许 null——并发场景下 get(key) 返回 null 时无法判断是"不存在"还是"value 就是 null"，语义不确定。

## 三、LinkedHashMap

### 3.1 数据结构

```text
LinkedHashMap = HashMap + 双向链表

  HashMap 的桶数组（hash 寻址） + 双向链表（维护顺序）

  双向链表贯穿所有 Entry：
    head ← EntryA ← EntryB ← EntryC ← tail

  每个 Entry 多了 before/after 两个指针：
    HashMap.Node  +  before/after  =  LinkedHashMap.Entry
```

```java
// LinkedHashMap.Entry 比 HashMap.Node 多了双向指针
static class Entry<K, V> extends HashMap.Node<K, V> {
    Entry<K, V> before, after;    // 双向链表指针
    // 仍然有 hash, key, value, next（桶内链表指针）
    // before/after 是跨桶的全局顺序链表
}
```

### 3.2 两种遍历顺序

```java
// 构造参数 accessOrder 决定遍历顺序
LinkedHashMap<K, V> map = new LinkedHashMap<>(16, 0.75f, accessOrder);

// accessOrder = false（默认） → 按插入顺序遍历
// accessOrder = true          → 按访问顺序遍历（最近访问的在链表尾部）
```

```text
accessOrder = false（插入顺序）：
  put("A", 1) → put("B", 2) → put("C", 3)
  遍历顺序：A → B → C（插入顺序不变）

accessOrder = true（访问顺序）：
  put("A", 1) → put("B", 2) → put("C", 3)
  get("A")     → A 被访问，移到链表尾部
  遍历顺序：B → C → A（最近访问的在最后）
```

> [!important] accessOrder = true 是 LRU（Least Recently Used）缓存的实现基础！最久没访问的在链表头部，最近访问的在尾部。删除头部元素就等于淘汰最久未用的缓存。

### 3.3 LRU 缓存实现

```java
// 经典 LRU 缓存：继承 LinkedHashMap，重写 removeEldestEntry
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    public LRUCache(int capacity) {
        super(capacity, 0.75f, true);  // accessOrder = true！
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;  // 当元素数超过 capacity，自动删除最老的（链表头部）
    }
}

// 使用
LRUCache<String, String> cache = new LRUCache<>(3);
cache.put("A", "1");
cache.put("B", "2");
cache.put("C", "3");
cache.get("A");              // A 移到尾部，顺序变为 B → C → A
cache.put("D", "4");         // 超过容量，自动删除最老的 B
// cache 中剩余：C → A → D
```

> [!tip] 面试考 LRU 缓存是高频题——直接用 LinkedHashMap + accessOrder=true + removeEldestEntry 是最简洁的实现。手写 LRU 的话需要 HashMap + 双向链表手动维护。

## 四、TreeMap

### 4.1 底层结构

```text
TreeMap = 红黑树（自平衡二叉搜索树）

  每个节点：
    Entry<K, V>
      ├── key
      ├── value
      ├── left / right / parent
      ├── color（RED / BLACK）

  保证：左子树 key < 根 key < 右子树 key
  红黑树规则：根黑、叶黑、红节点子必黑、任一路径黑节点数相同
```

```java
// TreeMap 核心操作复杂度
get(key)       → O(logn)    // 二叉搜索树查找
put(key, val)  → O(logn)    // 查找 + 插入 + 红黑树旋转调整
remove(key)    → O(logn)    // 查找 + 删除 + 红黑树旋转调整
containsKey    → O(logn)
firstKey/lastKey → O(logn) 或 O(1)（取决于实现）
```

### 4.2 排序方式

```java
// 方式1：key 自然顺序（key 实现 Comparable）
TreeMap<String, Integer> map1 = new TreeMap<>();
// String 按字典序排序

// 方式2：自定义 Comparator
TreeMap<String, Integer> map2 = new TreeMap<>(Comparator.reverseOrder());
// 逆序排列

// 方式3：传入 Comparator 比较复杂对象
TreeMap<User, String> map3 = new TreeMap<>(Comparator.comparingInt(u -> u.age));
```

> [!note] 如果 key 既没有实现 Comparable，也没有提供 Comparator，put 时会抛 ClassCastException。

### 4.3 TreeMap 特有方法

```java
// TreeMap 提供了大量范围查询方法（HashMap 不具备）
map.firstKey();                    // 最小 key
map.lastKey();                     // 最大 key
map.lowerKey(key);                 // 严格小于 key 的最大 key
map.higherKey(key);                // 严格大于 key 的最小 key
map.floorKey(key);                 // ≤ key 的最大 key
map.ceilingKey(key);               // ≥ key 的最小 key
map.subMap(fromKey, toKey);        // 范围子映射 [from, to)
map.headMap(toKey);                // < toKey 的子映射
map.tailMap(fromKey);              // ≥ fromKey 的子映射
```

> [!tip] TreeMap 的范围查询能力是 HashMap/LinkedHashMap 不具备的——需要"有序遍历"或"范围查找"时首选 TreeMap。

## 五、ArrayDeque vs LinkedList vs PriorityQueue

### 5.1 ArrayDeque

```java
// ArrayDeque：循环数组实现的双端队列
// 底层是 Object[] 数组 + head/tail 指针
// head 指向队首元素，tail 指向队尾元素下一个位置

public class ArrayDeque<E> extends AbstractCollection<E>
    implements Deque<E>, Cloneable, Serializable {
    transient Object[] elements;    // 循环数组
    transient int head;             // 队首指针
    transient int tail;             // 队尾指针
}

// 双端操作都是 O(1)
addFirst(e)   → elements[head = (head - 1) & (elements.length - 1)] = e
addLast(e)    → elements[tail] = e; tail = (tail + 1) & (elements.length - 1)
pollFirst()   → 返回 elements[head]; head = (head + 1) & mask
pollLast()    → 返回 elements[tail - 1]; tail = (tail - 1) & mask
```

```text
为什么用 & (length - 1) 而不是 % length？
  因为 length 始终是 2 的幂（和 HashMap 寻址一样）
  & 运算比 % 快，且保证 head/tail 在数组范围内循环
```

> [!important] ArrayDeque 是 Java 中栈/队列的最佳实现选择：无 Node 对象开销（只有数组），内存连续（CPU 缓存友好），双端操作全部 O(1)。

### 5.2 LinkedList 作为队列

```java
// LinkedList 也实现了 Deque 接口，可以作为队列/双端队列
// 但底层是双向链表，每个元素需要一个 Node 对象（额外 24 字节开销）

// Node 结构：
static class Node<E> {
    E item;           // 8 字节（引用）
    Node<E> next;     // 8 字节
    Node<E> prev;     // 8 字节
    // 对象头 16 字节
    // 总计 ~40 字节/元素 vs ArrayDeque ~8 字节/元素（引用）
}
```

| 比较维度 | ArrayDeque | LinkedList |
|---------|------------|------------|
| 底层结构 | 循环数组 | 双向链表 |
| 内存开销 | 每元素 ~8 字节引用 | 每元素 ~40 字节 Node |
| CPU 缓存 | 连续数组，缓存友好 | 链表分散，缓存不友好 |
| 随机访问 | 不支持 | 不支持（O(n） |
| 推荐 | **栈/队列首选** | 仅在需要 List 特性时考虑 |

> [!note] 面试高频考点：Stack 类已淘汰（全表 synchronized + 继承 Vector），栈应该用 `Deque<Integer> stack = new ArrayDeque<>()`。

### 5.3 PriorityQueue

```java
// PriorityQueue：基于堆（完全二叉树）的优先队列
// 底层是 Object[] 数组，用数组表示完全二叉树

// 堆的性质：
//   parent(i) = (i - 1) / 2
//   leftChild(i) = 2 * i + 1
//   rightChild(i) = 2 * i + 2
//   默认最小堆：queue[0] 是最小元素

// 核心操作
offer(e)    → O(logn)    // 插入，上浮（siftUp）
poll()      → O(logn)    // 删除堆顶，下沉（siftDown）
peek()      → O(1)       // 查看堆顶（queue[0]）
```

```text
最小堆示意：
        1（堆顶/最小）
       / \
      3   2
     / \ /
    5  4 6

数组表示：[1, 3, 2, 5, 4, 6]
  queue[0] = 1（最小值）
  poll() 返回 1 → 6 替换堆顶 → siftDown → [2, 3, 6, 5, 4]
```

> [!tip] PriorityQueue 是 Top-K 问题、定时任务调度等场景的核心数据结构。默认最小堆，最大堆需要传入 `Comparator.reverseOrder()`。

## 原理深挖

### HashSet 去重机制和 HashMap key 去重完全一致

```java
// HashSet.add(e) → map.put(e, PRESENT)
// HashMap.put(key, value) 的去重逻辑：
//   1. hash = key.hashCode()
//   2. 桶位置 = hash & (n - 1)
//   3. 遍历桶内链表/树：如果 key.equals(已有key) → 覆盖（add 返回 false）
//   4. 没有 equals 相等的 → 新建节点（add 返回 true）

// 所以 HashSet 的去重 = equals + hashCode
// 重写 equals 必须重写 hashCode，否则 HashSet/HashMap 会出 bug：
class BadSet {
    @Override
    public boolean equals(Object o) { ... }  // 重写了 equals
    // 没重写 hashCode → 两个 equals 相等的对象可能落在不同桶 → HashSet 无法去重！
}
```

> [!important] HashSet 和 HashMap 的 key 依赖同一套规则：先 hashCode 定桶，再 equals 判等。只重写 equals 不重写 hashCode，HashSet 去重失败——这是面试经典陷阱。

### LinkedHashMap.afterNodeAccess 的 LRU 机制

```java
// LinkedHashMap 重写了 HashMap 的 afterNodeAccess 回调
// 每次 get/put 已存在的 key 时，该 Entry 被移到双向链表尾部
void afterNodeAccess(Node<K, V> e) {
    // 把 e 从链表中摘出（before.after = after, after.before = before）
    // 把 e 追加到链表尾部（tail.after = e, e.before = tail, tail = e）
}
// 链表头部 = 最久未访问 = LRU 淘汰目标
// 链表尾部 = 最近刚访问 = 保留对象
```

> [!note] LinkedHashMap 通过重写 HashMap 的回调方法（afterNodeAccess / afterNodeInsertion / afterNodeRemoval）来维护双向链表，不需要修改 HashMap 本身的代码——这是 Template Method 设计模式的经典应用。

### TreeMap 红黑树插入调整

```text
红黑树插入新节点（默认红色）后的调整过程：
  1. 父节点黑色 → 不需要调整（红节点不影响黑高）
  2. 父节点红色 → 冲突，需要修复：
     - 叔叔红色 → 父+叔变黑，祖父变红，递归向上
     - 叔叔黑色（或 null）→ 根据方向旋转 + 变色
       - LL 型：祖父右旋，父变黑祖父变红
       - RR 型：祖父左旋，父变黑祖父变红
       - LR 型：父先左旋，转成 LL 型
       - RL 型：父先右旋，转成 RR 型
  3. 根节点强制变黑
```

> [!tip] TreeMap 的红黑树调整和 HashMap 中链表转红黑树后的调整原理完全相同——面试如果追问红黑树，可以从 HashMap 的树化聊到这里。

## 面试回答框架

> [!info] 问：除了 HashMap 还用过哪些集合？

**① 一句话定位：** 按场景选集合——去重用 HashSet、排序用 TreeMap、LRU 用 LinkedHashMap、队列用 ArrayDeque。

**② 核心展开：**

HashSet 底层就是 HashMap，value 用一个 PRESENT 占位符（`new Object()`），add(e) 就是 map.put(e, PRESENT)，去重机制和 HashMap 的 key 一样——先 hashCode 定桶再 equals 判等。

LinkedHashMap 在 HashMap 基础上加了双向链表维护插入/访问顺序，accessOrder=true 时最近访问的 Entry 移到链表尾部，头部就是最久未访问的——这就是 LRU 缓存的实现基础，重写 removeEldestEntry 就能自动淘汰。

TreeMap 是红黑树实现，所有操作 O(logn)，按 key 自然顺序或 Comparator 排序，有丰富的范围查询方法（subMap/floorKey/ceilingKey 等）。

HashTable 全表 synchronized，已淘汰，线程安全场景用 ConcurrentHashMap。

**③ 队列补充：** ArrayDeque 是循环数组双端队列，无 Node 开销、CPU 缓存友好，是栈和队列的最佳选择（Stack 类已淘汰）。PriorityQueue 是最小堆，Top-K 问题的核心数据结构。

> [!question] 追问：LRU 缓存怎么实现？

> 最简洁的实现是继承 LinkedHashMap，构造时 accessOrder=true，重写 removeEldestEntry 在 size() > capacity 时返回 true 自动淘汰。手写实现则需要 HashMap（O(1) 查找）+ 双向链表（O(1) 移动到尾部/删除头部）。

> [!question] 追问：HashSet 和 HashMap 的关系？

> HashSet 的底层就是 HashMap——add(e) = map.put(e, PRESENT)，contains(e) = map.containsKey(e)，remove(e) = map.remove(e)。PRESENT 是一个 `new Object()` 占位符，只为了让 put 方法能工作。复用 HashMap 意味着 HashSet 不需要自己实现 hash/桶/扩容逻辑，只需要一个 16 字节的占位对象代价。
