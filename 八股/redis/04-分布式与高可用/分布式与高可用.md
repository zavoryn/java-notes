## 一、分布式锁

### 1.1 手写版——SET NX EX + Lua 脚本释放

```bash
# 加锁：原子操作，一条命令完成
SET lock:order thread-001 NX EX 30
# NX：key 不存在才设（排他性）
# EX 30：30秒自动过期（防死锁）
# thread-001：value 放自己身份标识（防误删别人的锁）
```

> [!warning] 为什么不能用 SETNX + EXPIRE 两步？
> ```bash
> SETNX lock key   # 如果执行完这一步后程序崩溃/宕机
> EXPIRE lock 30   # 这步没执行 → 锁永远不会释放 → 死锁！
> ```

**释放锁——Lua 脚本原子操作：**

```lua
-- 先判断 value 是不是自己加的锁，是才删
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

> [!tip] 为什么判断 value？
> 防止 A 的锁过期被 B 拿到后，A 把 B 的锁删了。就像**宾馆房卡**——你只能退自己的房间，不能退别人的房间。

### 1.2 Redisson——看门狗（Watch Dog）自动续期

手写版的致命问题：**业务执行 35 秒，锁 30 秒就过期了**。锁自动释放，其他线程也能拿到锁 → 安全漏洞。

**看门狗机制：**

```
加锁成功
  ↓
启动 Watch Dog 定时任务（默认每 10 秒检查一次）
  ↓
如果线程还持有锁（判断 clientId 一致）→ 续期到 30 秒
  ↓
循环续期...直到业务完成
  ↓
unlock → 关闭看门狗 → 删除锁 key
```

> [!tip] 看门狗 = 保安
> 每隔一会来检查锁还在不在，在就给你续期。如果你（JVM）宕机了，看门狗也停了，锁自然过期。

**Redisson 加锁源码思想：**

```java
// 伪代码表达思想
RLock lock = redisson.getLock("lock:order");
lock.lock();  // 加锁 + 自动启动看门狗
try {
    // 业务逻辑（多久都行，看门狗会续期）
} finally {
    lock.unlock();  // 释放锁 + 关闭看门狗
}
```

### 1.3 RedLock——多节点场景

适用于**主从切换时锁丢失**的场景：

```
主节点：锁 order → 还没同步到从 → 主挂了 → 从变主 → 锁丢了！

RedLock 解法：
  N 个独立 Redis 节点（不是主从，是独立的，比如 5 个）

  加锁：向 N 个节点依次请求加锁
    → 超过半数（N/2+1）加锁成功 → 才算获取锁成功
    → 锁有效期 = TTL - 加锁总耗时

  释放：向所有节点发送释放命令
```

> [!warning] RedLock 争议
> RedLock 在实际生产中应用有限。工程实践更推荐用 **ZooKeeper / Etcd** 做分布式锁，因为 CP 系统比 AP 系统（Redis）更适合一致性锁场景。

### 1.4 分布式锁演进总结

```
阶段 1: SETNX + EXPIRE（两步非原子 → 死锁）
  ↓
阶段 2: SET NX EX（原子加锁，但仍要手动过期）
  ↓
阶段 3: SET NX EX + Lua 解锁（原子释放，防止误删）
  ↓
阶段 4: Redisson 看门狗（自动续期，长业务不慌）
  ↓
阶段 5: RedLock（多节点，防主从切换锁丢失）
  ↓
阶段 6: ZK/Etcd（强一致性场景终极方案）
```

## 二、高可用

### 2.1 演进路线

```
单机 Redis
  ↓ 容量不够 / 并发瓶颈
主从复制（读写分离）
  ↓ 主挂了谁来管？
哨兵 Sentinel（自动故障转移）
  ↓ 容量瓶颈（单机内存有上限）
集群 Cluster（数据分片，水平扩容）
```

### 2.2 主从复制

```
写请求 → 主节点
读请求 → 从节点（可以多个）

主从同步流程：
  从连上主 → 发 PSYNC 命令
    → 首次连接：全量复制（主 bgsave 生成 RDB → 从加载）
    → 断线重连：增量复制（主把断开期间的命令发给从）
```

**主从问题：**
- 主挂了不能自动切换（需要人工介入）
- 读写分离有数据延迟（主写完，从还没同步到）
- 所有写都在主节点，容量还是受限于单机内存

### 2.3 哨兵（Sentinel）

```
哨兵集群（至少 3 个，必须奇数）
  ├── 监控：定期 PING 主从节点
  ├── 通知：主挂了 → 哨兵之间投票确认 → 选新主
  ├── 自动故障转移：将旧主的从节点提升为新主
  └── 配置提供者：客户端问哨兵"现在主是谁？"
```

**故障转移流程：**

```
1. 主观下线（SDOWN）：
   一个哨兵 PING 不通主节点 → 认为主观下线

2. 客观下线（ODOWN）：
   多数哨兵（quorum）都认为主管下线 → 确认客观下线

3. 选主（Leader Sentinel）：
   哨兵之间投票选出一个 Leader 来执行故障转移

4. 选新主：
   优先级高的从 → 复制偏移量大的 → runid 小的

5. 通知：
   告知其他从节点新主是谁，告知客户端
```

> [!important] 为什么哨兵至少要 3 个？
> 1. **投票需要多数**：2 个哨兵中 1 个挂了就剩 1 个，无法形成多数决议
> 2. **奇数避免脑裂**：3 个哨兵可以容忍 1 个挂掉，仍能 2/3 形成多数

### 2.4 集群（Cluster）

```
数据分片：slot = CRC16(key) % 16384

┌─────────────────────────────────────────────────────┐
│  16384 个 slot 槽位                                 │
│  0-5460  │  5461-10922  │  10923-16383              │
│  节点A   │   节点B      │   节点C                    │
│  (主)    │   (主)       │   (主)                     │
│  + 从A   │   + 从B      │   + 从C                    │
└─────────────────────────────────────────────────────┘

请求任意节点 → 计算 slot →
  对路 → 直接处理
  不对路 → 返回 MOVED 重定向 → 客户端去正确节点
```

> [!important] 为什么是 16384 个槽（不是 65536）？

| 槽数 | 心跳报文大小 | 说明 |
|------|-------------|------|
| 16384 | 约 2KB | Redis 集群节点间通过 Gossip 协议同步状态，心跳包包含槽位 bitmap |
| 65536 | 约 8KB | 心跳包太大，浪费带宽。而且 1000 个节点的集群都够用，16384 绰绰有余 |

**集群扩容缩容：**

```
扩容（加新节点）：
  新节点加入 → 把部分 slot 迁移到新节点 → 完成扩容

缩容（下线节点）：
  把节点上的 slot 迁移到其他节点 → 下线该节点

迁移过程中：目标 key 在迁移途中 → 返回 ASK 重定向（临时重定向）
```

## 三、一致性哈希（补充概念）

并非 Redis 集群默认方案，但面试常考，用于**无中心节点的分布式系统**：

### 3.1 哈希环原理

```
哈希环：0 → 2^32-1 连接成环

         0
    Node A
   /     \
  /       \       Node A hash=100
 /         \      Node B hash=2000
|   哈希环    |     Node C hash=5000
 \         /
  \       /
   Node C ← Node B
   2^32-1

key 定位：计算 key 的 hash → 在环上顺时针找第一个节点
```

**节点变动时：**

```
新增节点（如 hash=300）：
  → 只影响 Node B 上 hash 在 (100, 300] 范围内的 key
  → 这些 key 重新路由到新节点
  → 其他 key 不受影响

删除节点（Node B 下线）：
  → Node B 上的 key 顺时针移到 Node C
  → 其他 key 不受影响
```

> [!important] 数据倾斜怎么办？——虚拟节点
> 如果只有 3 个实体节点，它们在哈希环上的位置不均匀，会导致有的节点数据特别多。
>
> **虚拟节点：** 每个实体节点对应 N 个虚拟节点（如 150 个），分散在哈希环上。key 先映射到虚拟节点，虚拟节点再映射到实体节点。就像**每个人买多个彩票号码**，中奖概率更均匀。
>
> ```
> 实体节点 A → 虚拟节点 A0, A1, A2, ..., A149（分散在环上）
> 实体节点 B → 虚拟节点 B0, B1, B2, ..., B149
> ```

### 3.2 一致性哈希 vs Redis Cluster

| 维度 | 一致性哈希 | Redis Cluster |
|------|-----------|---------------|
| 寻址方式 | 哈希环顺时针找 | CRC16 % 16384 → slot → 节点 |
| 扩缩容影响 | 只影响相邻节点 | slot 精细迁移 |
| 适用场景 | 无中心架构（如 Dubbo 负载均衡） | Redis 官方集群方案 |
| 是否中心化 | 否 | 否（Gossip 协议去中心） |

## 四、事务

### 4.1 MULTI / EXEC 基本用法

```bash
MULTI           # 开启事务
SET k1 v1       # 命令入队
SET k2 v2       # 命令入队
INCR counter    # 命令入队
EXEC            # 批量执行，返回结果数组
```

**Redis 事务的三不特性：**

| 特性 | MySQL 事务 | Redis 事务 |
|------|-----------|-----------|
| 原子性 | 全部成功或全部回滚 | **不支持回滚** |
| 隔离性 | 多版本（MVCC）、锁 | 串行执行（单线程天然隔离） |
| 持久性 | 写磁盘 | 取决于持久化配置 |
| 一致性 | 保证 | 保证 |

### 4.2 编译错误 vs 运行时错误

```bash
# 场景一：语法错误 → 全部不执行
MULTI
SET k1 v1
SET k2          # 参数不足！编译时就发现了
EXEC            # 整个事务不执行

# 场景二：运行时错误 → 正确命令照常执行，出错的不影响
MULTI
SET k1 v1
INCR k1         # k1 是 "v1"，不能 INCR，运行时才出错
SET k2 v2       # 这条会正常执行！
EXEC            # 结果：[OK, ERROR, OK]
```

> [!important] Redis 为什么不支持回滚？
> 1. **保持简单和高效**：回滚机制会极大增加复杂度
> 2. **Redis 的错误通常来自语法/类型问题**：开发阶段就该发现，不需要运行时回滚
> 3. **哲学差异**：Redis 追求极简高性能，不是 ACID 数据库

### 4.3 WATCH 乐观锁

```bash
WATCH mykey    # 监视 mykey
val = GET mykey
val = val + 1
MULTI
SET mykey val
EXEC           # 如果 EXEC 之前 mykey 被别人改过 → 返回 nil（事务不执行）
```

```
WATCH 机制：
  → 监视一个或多个 key
  → 执行 EXEC 时检查这些 key 在 WATCH 之后是否被修改
  → 被修改了 → 整个事务取消（返回 nil）
  → 没被修改 → 执行事务
```

> [!tip] WATCH 就是乐观锁 —— "图书馆借书"比喻
> 读数据 → 做计算 → 写之前看看数据有没有被人改过。就像**图书馆借书**——先查库存（WATCH），借走时（EXEC）再确认一次库存没被别人借走。

### 4.4 Lua 脚本 = 真正的原子性

```lua
-- Lua 脚本在 Redis 中执行时，整个脚本是原子操作
-- 不会被其他命令插入

local val = redis.call("GET", KEYS[1])
if val == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```bash
# Redis 执行 Lua 脚本
EVAL "return redis.call('SET', KEYS[1], ARGV[1])" 1 mykey "hello"
```

**Lua 脚本 vs MULTI/EXEC：**

| 维度 | MULTI/EXEC | Lua 脚本 |
|------|-----------|----------|
| 原子性 | 执行期间不插入其他命令 | 整个脚本不插入其他命令 |
| 条件判断 | 只在 WATCH 时能用（乐观锁） | 脚本内可以任意 if/else 逻辑 |
| 回滚 | 不支持 | 不支持 |
| 适用场景 | 简单批量操作 | 需要条件判断 + 原子性（如分布式锁释放） |

## 五、消息队列

### 5.1 三种方式对比

| 方案 | 实现方式 | 持久化 | 消费确认 | 推荐度 |
|------|---------|--------|---------|--------|
| **List** | LPUSH + BRPOP | 有（AOF/RDB） | 无（弹出即删除） | 简单场景可用 |
| **Pub/Sub** | PUBLISH + SUBSCRIBE | 无 | 无 | **不推荐（丢消息）** |
| **Stream** | XADD + XREADGROUP | 有 | 有（XACK） | **最推荐** |

### 5.2 List 实现消息队列

```bash
# 生产者
LPUSH queue:order message1
LPUSH queue:order message2

# 消费者（阻塞等待）
BRPOP queue:order 0   # 0=无限等待，直到有消息
```

> [!warning] List 的局限性
> - 不支持一对多（一条消息只能被一个消费者消费）
> - 不支持消费确认（消息被 BLPOP 弹出就丢了，如果消费者处理失败也没法重试）
> - 没有消息回溯能力

### 5.3 Pub/Sub 发布订阅

```bash
# 订阅者
SUBSCRIBE channel:news

# 发布者
PUBLISH channel:news "突发新闻！"
```

> [!warning] Pub/Sub 的三大致命缺陷
> 1. **消息不持久化**：没在线的订阅者收不到历史消息
> 2. **无法重试**：消息发出后没有确认机制，出了问题无法得知
> 3. **断线重连丢失**：网络抖动断开重连，断开期间的消息永久丢失
>
> **面试态度：明确表示不推荐用于生产环境的消息队列。**

### 5.4 Stream——5.0+ 真正的消息队列

```bash
# 生产消息
XADD mystream * key1 value1 key2 value2
# * 表示自动生成 ID（时间戳-序号）

# 消费者组消费
XREADGROUP GROUP mygroup consumer1 BLOCK 2000 STREAMS mystream >
# > 表示从最新未被消费的消息开始

# 确认消费
XACK mystream mygroup message-id

# 查看未确认的消息（消费者挂了可以重试）
XPENDING mystream mygroup
```

**Stream 的优势：**

| 特性 | 说明 |
|------|------|
| **消费者组** | 一条消息可被多个消费者组消费（广播），组内多个消费者分担（竞争） |
| **持久化** | 消息写入 AOF/RDB，重启不丢 |
| **消费确认** | XACK 确认，不确认的消息可被重新投递 |
| **消息回溯** | 可以按 ID 范围重新读取历史消息 |
| **阻塞读取** | BLOCK 参数支持阻塞等待 |

> [!tip] Stream = Kafka 精简版
> 有消费者组、持久化、确认机制，但不支持 partition 分片（单 Stream 单主）。

```
消费者组1（订单处理）：
  consumer_A → 消费消息
  consumer_B → 消费消息（与 A 共享消息，不重复）

消费者组2（订单通知）：
  consumer_C → 消费同一批消息（独立的组，消息重复投递）

每个消费者组维护自己的消费位置指针
```

## 六、内存管理

### 6.1 jemalloc 内存分配器

> [!important] jemalloc 是什么？
> Redis 默认使用 jemalloc 作为内存分配器，替代 Linux 默认的 glibc malloc。

**jemalloc 的优势：**

| 特性 | glibc malloc | jemalloc |
|------|-------------|----------|
| **内存碎片** | 高碎片（长时间运行后严重） | 低碎片（多 size class 精确匹配） |
| **多线程性能** | 锁争用严重 | arena 分区设计，锁争用少 |
| **内存统计** | 粗略 | 精确（可用 `MEMORY STATS` 查看） |
| **适用场景** | 通用 | 长时运行、频繁分配释放的服务端程序 |

**jemalloc 的 size class 设计：**

```
小对象（如 8B, 16B, 32B ...）→ 精确分配，不浪费
大对象 → 按页分配（4KB 对齐）

每种大小有独立的"篮子（bin）"：
  8B bin → 只存 8B 的对象
  16B bin → 只存 16B 的对象
  ...

释放的内存直接放回对应的 bin，下次分配直接从 bin 取 → 减少碎片
```

> [!tip] jemalloc = 快递柜
> 每个大小格子对应不同的快递大小，小包裹放小格子，大包裹放大格子。而不是把所有快递塞进一个大房间（容易乱），也不是每个快递精确打一个包裹（浪费空间）。

### 6.2 紧凑数据结构

Redis 使用多种紧凑结构来节省内存：

| 结构 | 压缩方式 | 节省内存原理 |
|------|---------|-------------|
| **ziplist** | 连续内存，每个 entry 只记录 prevlen + encoding + data | 省去链表指针（2x8字节） |
| **intset** | 连续整数数组，统一编码（int16/int32/int64） | 省去指针 + 统一整数位宽 |
| **quicklist** | ziplist + 链表混合 | 两者取其优 |
| **listpack** | ziplist 的替代（7.0+），解决了连锁更新问题 | 每个 entry 只记录自己的长度 |

**ziplist 连锁更新问题：**

```
ziplist 节点格式：[prevlen][encoding][data]

如果前面节点的长度从 253 变成 254：
  prevlen 从 1 字节变成 5 字节
  当前节点变大了 → 可能触发下一个节点的 prevlen 也变大
  → 连锁反应 → O(n^2) 级别的重写
```

Redis 7.0 引入 **listpack** 替代 ziplist，每个 entry 只记录自身长度，彻底解决连锁更新。

### 6.3 内存优化实战建议

| 优化手段 | 说明 |
|----------|------|
| **控制 key 数量** | key 本身占用内存，能用 hash 聚合就用 hash 聚合 |
| **短 key 名** | `user:1001:profile` 优于 `user_profile_with_detail_info_of_user_1001` |
| **选对数据结构** | 小集合用 intset/ziplist（默认配置下 512 个元素自动切换） |
| **设置 maxmemory** | 必须设，防止 OOM |
| **合理淘汰策略** | 缓存场景用 allkeys-lru，不要用默认 noeviction |
| **监控内存碎片率** | `INFO memory` 查看 `mem_fragmentation_ratio`，> 1.5 考虑启用碎片整理 |

## 七、Redis vs Memcached

### 7.1 12 维度核心对比

| 维度 | Redis | Memcached |
|------|-------|-----------|
| **数据类型** | String、List、Set、Hash、ZSet 等 | 仅 String（key-value） |
| **持久化** | RDB + AOF + 混合 | 不支持（纯内存） |
| **高可用** | 主从、哨兵、集群 | 客户端一致性哈希 |
| **线程模型** | 单线程处理命令（6.0 IO 多线程） | 多线程 |
| **内存管理** | jemalloc（多 size class 精细管理） | Slab Allocation（固定大小 chunk） |
| **网络模型** | epoll IO 多路复用 | libevent 多线程处理 |
| **过期策略** | 惰性删除 + 定期删除 | 惰性删除（访问时才检查）+ LRU |
| **事务** | 支持（MULTI/EXEC、Lua） | 不支持 |
| **发布订阅** | 支持 | 不支持 |
| **Lua 脚本** | 支持 | 不支持 |
| **集群** | 原生集群（16384 slot） | 客户端分片 |
| **性能** | QPS 10万+（单线程） | QPS 10万+（多线程） |
| **适用场景** | 复杂缓存 + 分布式锁 + 排行榜 + 消息队列 | 简单 key-value 缓存 |

### 7.2 深入对比

**Memcached 的 Slab Allocation：**

```
将所有内存切分成不同大小的 slab class：
  1KB slab → 存 1KB 的数据
  2KB slab → 存 2KB 的数据
  4KB slab → 存 4KB 的数据
  ...

优点：内存碎片少，分配快速
缺点：固定大小，浪费空间（存 1.1KB 的 value 要用 2KB slab）
```

**Redis jemalloc vs Memcached Slab：**

| 维度 | jemalloc（Redis） | Slab（Memcached） |
|------|-------------------|-------------------|
| 分配粒度 | 多级 size class（8B, 16B, ...） | chunk 固定大小 |
| 内存利用率 | 高（精确匹配） | 较低（可能有内部碎片） |
| 复杂度 | 复杂 | 简单 |

### 7.3 选型建议

```
选 Redis 的场景：
  ✅ 需要复杂数据类型（排行榜→ZSet，共同好友→Set）
  ✅ 需要持久化（数据不能丢）
  ✅ 需要高可用（主从+哨兵+集群）
  ✅ 需要分布式锁
  ✅ 需要消息队列 / 发布订阅

选 Memcached 的场景：
  ✅ 纯 key-value 缓存，数据可丢
  ✅ 多核 CPU 利用率更高（多线程）
  ✅ 部署运维极简
  ✅ 公司已有技术栈
```

> [!tip] 面试背诵
> "Redis 和 Memcached 都是高性能内存缓存。Redis 的优势在于**数据类型丰富、支持持久化和高可用、功能生态更全**，本质是一个数据结构服务器；Memcached 更纯粹，就是**多线程 key-value 缓存**，部署简单，但功能单一。现代 Java 后端项目首选 Redis。"

## 面试满分话术

### 话术一：分布式锁演进（展示深度）

> "分布式锁我理解它的演进过程：
>
> 第一版最简单，用 `SETNX + EXPIRE`，但两个命令不是原子操作，如果 SETNX 成功后宕机，锁永远不会释放。
>
> 第二版用 `SET key value NX EX 30`，一条命令原子完成加锁和设置过期时间。
>
> 第三版释放锁也不能简单 DEL——要判断 value 是不是自己设的。用 Lua 脚本原子执行 GET 判断 + DEL 删除，防止误删别人的锁。类比宾馆房卡——只能退自己的房间。
>
> 但这样还有问题：如果业务执行超过了锁的过期时间怎么办？这就引入了第四版——Redisson 的看门狗机制，后台线程每 10 秒自动续期到 30 秒，业务完成后再关闭。
>
> 再进一步，主从架构下主挂了锁还没同步到从——锁就丢了。这时候需要第五版 RedLock，向 N 个独立 Redis 节点申请锁，过半成功才算拿到。不过实际工程中，对强一致性要求的锁场景，更推荐用 ZK 或 Etcd。"

### 话术二：高可用演进（展示体系化思维）

> "Redis 高可用是一个完整的三层演进体系：
>
> 第一层是**主从复制**，实现读写分离——写请求走主节点，读请求走从节点。主从之间通过 PSYNC 进行全量或增量同步。但问题是主挂了不能自动切换。
>
> 第二层是**哨兵 Sentinel**，它解决了自动故障转移。哨兵集群至少 3 个节点（必须是奇数），通过 PING 监控主从，多数哨兵确认主观下线后触发客观下线，哨兵之间投票选出 Leader 来执行故障转移——从旧主的从节点中按优先级、偏移量、runid 选出新主。整个过程对客户端透明。
>
> 第三层是**集群 Cluster**，解决的是单机内存上限的容量瓶颈。通过 CRC16 对 key 取模，将数据分到 16384 个 slot 中，每个节点负责一部分 slot。节点间通过 Gossip 协议同步状态，心跳包约 2KB。客户端请求如果路由到错误节点，会收到 MOVED 重定向。"

### 话术三：消息队列 Stream（展示新特性理解）

> "Redis 做消息队列，我把它分为三个阶段：
>
> 第一阶段，最简单用 List 的 LPUSH + BRPOP，阻塞消费但消息弹出即删，不支持消费确认和一对多。
>
> 第二阶段，Pub/Sub 发布订阅，支持一对多广播但消息不持久化，订阅者不在线就收不到——生产环境不推荐。
>
> 第三阶段是 Redis 5.0 引入的 Stream，它真正补齐了消息队列的能力：支持消费者组（组间广播、组内竞争）、消息持久化到 AOF/RDB、XACK 消费确认机制（未确认可重投）、按 ID 范围消息回溯、阻塞读取。可以说是 Kafka 的精简版，只是暂不支持 partition 分片。对于不需要极端吞吐量的场景，Stream 完全够用。"

## 记忆口诀

> [!tip] 口诀一：分布式锁
> **NX EX 原子加，Lua 判断再释放。看门狗自动续，红锁过半才作数。**
>
> SET NX EX 原子加锁 → Lua 脚本(判断value→DEL)原子释放 → Redisson看门狗自动续期 → RedLock N个独立节点过半成功

> [!tip] 口诀二：高可用
> **主从读写分，哨兵自动选，集群分片散。一六三八四，心跳包大小。**
>
> 主从(读写分离) → 哨兵(自动选举,至少3奇数) → 集群(CRC16%16384分片) → 16384因心跳包仅2KB

> [!tip] 口诀三：一致性哈希
> **哈希环上顺时针，虚拟节点匀负载。扩缩只动相邻位，无中心时最佳选。**
>
> 哈希环顺时针定位 → 虚拟节点分散解决倾斜 → 扩缩容只影响相邻节点 → 适用于 Dubbo 等无中心架构

> [!tip] 口诀四：事务
> **MULTI 入队 EXEC 发，WATCH 看门乐观查。语法错误全不跑，运行错误各走各。Lua 脚本真原子，判断删除一把梭。**
>
> MULTI/EXEC 批量执行；WATCH=乐观锁(图书馆借书)；语法错全回滚，运行错各执行各的；Lua脚本真原子

> [!tip] 口诀五：消息队列
> **List 阻塞无确认，PubSub 不存丢消息。Stream 五零消费组，确认回溯像 Kafka。**
>
> List(BLPOP)阻塞消费无确认 → Pub/Sub 不持久不推荐 → Stream 5.0(消费组+XACK确认+持久化+回溯) = Kafka精简版

> [!tip] 口诀六：内存管理 + vs Memcached
> **快递柜 jemalloc 少碎片，压列表 intset 省内存。Redis 类型多持久化高可用全，MC 多线程纯缓存部署简。**
>
> jemalloc(快递柜比喻多 size class) + 紧凑结构(ziplist/intset/quicklist)；Redis=瑞士军刀(功能全)，MC=水果刀(只做 KV 但做得好)
