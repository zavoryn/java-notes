## 一、5 大数据类型 + 底层结构

### 1.1 速查表

| 类型 | 底层实现 | 适用场景 |
|------|----------|----------|
| **String** | SDS（简单动态字符串） | 缓存 value、计数器、分布式锁 token、session |
| **List** | quicklist（双向链表 + ziplist 混合结构） | 消息队列（左进右出）、最新列表、时间线 |
| **Set** | hashtable / intset（纯整数时用 intset） | 共同好友、标签系统、抽奖去重、交并差运算 |
| **Hash** | hashtable / ziplist（小 hash 时用压缩列表） | 对象属性缓存（如用户信息：name/age/email） |
| **ZSet** | skiplist（跳表） + hashtable（字典） | 排行榜、延时队列（score 存时间戳）、带权重的集合 |

### 1.2 String 底层：SDS（Simple Dynamic String）

Redis 不用 C 语言的 `\0` 结尾字符串，而是自己设计了 SDS 结构：

```
┌──────┬──────┬─────────────────────┐
│ len  │ free │ buf[]               │
│  5   │  5   │ 'h' 'e' 'l' 'l' 'o' │
└──────┴──────┴─────────────────────┘
```

**三大优势：**

| 特性 | C 字符串 | SDS |
|------|----------|-----|
| 获取长度 | O(n) 遍历 | O(1) 直接读 `len` 字段 |
| 拼接安全 | 需要手动 `realloc`，可能溢出 | 预分配空间（`free` 字段），自动扩容 |
| 二进制安全 | 遇到 `\0` 截断 | 用 `len` 判断结尾，可存图片/音频 |
| 内存分配次数 | 每次修改都要 `realloc` | 预分配 + 惰性释放，减少系统调用 |

> [!tip] 为什么 SDS 用 `free` 字段？
> 预分配策略：当 `buf` 长度 < 1MB 时，`free = len`（扩容到 2 倍）；当 >= 1MB 时，`free = 1MB`。这避免了频繁 `malloc/realloc`，就像**租房时一次性多交两个月押金**，而不是每次用水用电都去交钱。

### 1.3 List 底层：quicklist

**演进路线：** 早期是 linkedlist + ziplist → 3.2 版本统一为 quicklist。

```
quicklist 节点 1          quicklist 节点 2
┌─────────────────┐     ┌─────────────────┐
│    ziplist       │ ◄──►│    ziplist       │
│  [A,B,C,D,E]     │     │  [F,G,H,I,J]     │
└─────────────────┘     └─────────────────┘
```

quicklist = **双向链表（宏观）** + **ziplist 压缩列表（微观）**。

- **ziplist 是什么？** 一块连续内存，紧凑存储多个元素，省内存但不灵活（插入中间要 memmove）。
- **为什么两者结合？** 纯 linkedlist 指针开销大（每个节点有 prev/next 指针），纯 ziplist 大时插入慢。quicklist 取长补短，每个链表节点中放一个小 ziplist（默认 8KB），既不浪费内存，又不会插入太慢。

> 🧠 **类比：quicklist 就像火车——每节车厢（链表节点）里坐着一排人（ziplist），车厢之间用挂钩（双向指针）连接。**

### 1.4 Set / Hash 底层

| 类型 | 小数据量 | 大数据量（或元素够长） | 转换条件 |
|------|---------|---------------------|---------|
| **Set** | intset（整数集合） | hashtable | 元素全为整数 且 个数 <= 512 |
| **Hash** | ziplist（压缩列表） | hashtable | 键值对长度 <= 64字节 且 个数 <= 512 |

**intset 是什么？** 连续内存的整数数组，由小到大排列，二分查找 O(log n)。省内存（省去指针开销），但只存整数。

**hashtable 扩容——渐进式 rehash：** Redis 是单线程，如果一次性把所有 key 从旧桶搬到新桶会卡住其他命令。所以**每次操作时顺便搬几个桶**，分多次完成。就像**搬家不是一天搬完，而是每天下班顺路带一点东西去新家**。

### 1.5 ZSet 底层：skiplist（跳表）—— 重点中的重点

ZSet 同时使用 **跳表（按 score 排序）** + **hashtable（按 member 快速查找 score）**，两种数据结构指向同一份数据。

```
                           跳表 = 高速公路 ✈

Level 3:  1 ────────────────────────→ 15         高速公路（最少节点，跑得最快）
                           ↓                     ↓
Level 2:  1 ────────→ 7 ───────────→ 15         国道（略多一些出口）
                           ↓           ↓           ↓
Level 1:  1 → 3 → 5 → 7 → 9 → 12 → 15         省道（全量数据，每个节点都在）

索引节点（高速层）  普通节点（省道层）
```

**跳表核心原理：**

1. **多层索引结构**：底层（Level 1）包含所有数据，每往上一层节点数减半。
2. **层数概率**：每个节点随机生成层数，每个节点有 1/2 概率提升到 Level 2，1/4 概率提升到 Level 3...
3. **查找过程**：从最高层开始向右查找，遇到大于目标值的节点就下降一层，直到在底层找到。
4. **时间复杂度**：O(log n)，空间复杂度 O(n)（额外索引约为 n）。

**查找 9 的过程演示：**
```
Level 3: 1 → 15 → "15 比 9 大，下降！"
Level 2: 1 → 7 → 15 → "7 比 9 小继续，15 比 9 大，下降！"
Level 1: 7 → 9 ✅ 找到！
```

> 🧠 **为什么 ZSet 不用红黑树？**
> 1. **跳表实现更简单**：红黑树要处理旋转、变色等复杂操作，跳表只需调整指针。
> 2. **范围查询天然高效**：跳表找到起点后顺着底层链条走即可（O(log n + m)），红黑树需要中序遍历。
> 3. **并发友好**：跳表局部修改，红黑树旋转影响整棵树。
> 4. **Redis 是内存操作**：红黑树在磁盘场景下优势（减少磁盘 IO），但在内存场景下跳表更优。

> [!tip] 跳表 = 高速公路比喻
> **底层（Level 1）是省道**：每个节点都在，全量数据，走得最慢。
> **上层（Level N）是高速**：只有少数出口节点，跳过中间直接到达，走得最快。
> 查找时：先上高速狂奔到目标附近 → 下高速走省道精确定位 → **又快又准**。

## 二、线程模型

### 2.1 核心认知：单线程 ≠ 慢

> [!important] 一句话总结
> **Redis 单线程处理命令（CPU 不是瓶颈），IO 多路复用同时监听多个客户端连接，后台线程处理耗时操作（文件关闭/AOF 重写/数据落盘）。6.0+ 引入 IO 多线程只处理网络读写，命令执行仍然单线程。**

### 2.2 为什么单线程还这么快？

| 原因 | 解释 |
|------|------|
| **纯内存操作** | 读写都在内存中，微秒级响应（纳秒~微秒），CPU 根本跑不满 |
| **IO 多路复用** | epoll 一个线程同时监听成千上万个 socket，哪个有数据就处理哪个 |
| **避免上下文切换** | 多线程切换有 CPU 开销 + 竞争锁开销，单线程完全避免 |
| **数据结构简单** | 专门优化的 SDS、ziplist、skiplist，没有复杂的事务开销 |

> 🧠 **瓶颈在哪？** 单机 Redis QPS 可以达到 10 万+，瓶颈在**网络 IO 带宽**，不在 CPU。就像**百米短跑**——跑道够宽（IO），一个人跑（单线程）反而比一群人挤来挤去（多线程加锁）更快。

### 2.3 IO 多路复用详解

```
┌──────────────────────────────────────────────────────┐
│                    Redis 主线程                        │
│                                                      │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐       │
│   │  epoll    │ → │  事件    │ → │  命令    │       │
│   │ 监听     │    │  分发器  │    │  执行    │       │
│   └──────────┘    └──────────┘    └──────────┘       │
│        ↑                                ↓             │
│   socket1 socket2  ... socketN     返回结果           │
└──────────────────────────────────────────────────────┘

epoll_wait 监听所有 socket → 有数据到达 → 读数据 → 执行命令 → 写响应
```

**epoll 三板斧：**
1. `epoll_create`：创建 epoll 实例
2. `epoll_ctl`：注册/修改要监听的文件描述符（add/mod/del）
3. `epoll_wait`：阻塞等待就绪事件，返回了就处理

> [!tip] epoll vs select/poll
> - **select**：最多监听 1024 个 fd，每次要全量传入，全量扫描。O(n)
> - **poll**：无 fd 数量限制，但仍需全量扫描。O(n)
> - **epoll**：内核用红黑树+链表管理，只返回就绪的 fd。O(1) 获取就绪事件

### 2.4 后台线程（非主线程）

| 线程 | 职责 |
|------|------|
| **关闭文件** | `close` 操作单独线程处理，避免阻塞主线程 |
| **AOF 刷盘** | `fsync` 写磁盘，fork 子进程或后台线程完成 |
| **AOF 重写** | `bgrewriteaof` 后台重写，压缩 AOF 文件 |
| **RDB 持久化** | `bgsave` fork 子进程生成快照 |
| **异步删除** | `UNLINK` 异步删除大 key（4.0+），主线程只断链，后台线程回收内存 |

### 2.5 Redis 6.0 IO 多线程

```
Redis 6.0 之前：
  主线程：读 socket → 执行命令 → 写 socket → 读 socket → ...

Redis 6.0 之后：
  IO 线程组：读 socket（多个线程并行）
       ↓
  主线程：执行命令（单线程，保持原子性）
       ↓
  IO 线程组：写 socket（多个线程并行）
```

> [!important] IO 多线程只处理网络读写！
> 命令执行仍然是单线程，保证了原子性。默认关闭，需手动开启。就像**餐厅**——厨房（命令执行）只有一个厨师，但前厅可以有多个服务员（IO 线程）同时接待客人。

## 三、持久化

### 3.1 RDB vs AOF 对比

| 维度 | RDB | AOF |
|------|-----|-----|
| **是什么** | 某一时刻的内存**快照**（二进制文件） | 所有写命令的**日志**（文本文件） |
| **触发方式** | `save`（阻塞）/ `bgsave`（fork 子进程） | 每条写命令追加 |
| **文件大小** | 小（压缩的二进制） | 大（文本日志，会增长） |
| **恢复速度** | 快（直接加载二进制） | 慢（逐条回放命令） |
| **数据安全** | 可能丢最后一次快照后的数据 | `everysec` 最多丢 1 秒 |
| **对性能影响** | fork 时有短暂停顿（COW 减小影响） | 持续写盘有 IO 开销 |

### 3.2 RDB 详解

```bash
# 触发方式
save 900 1     # 900秒内至少1次修改 → 触发 bgsave
save 300 10    # 300秒内至少10次修改 → 触发 bgsave
save 60 10000  # 60秒内至少10000次修改 → 触发 bgsave
```

**bgsave 流程：**
```
主进程
  ↓ fork() 创建子进程（COW 机制）
子进程（共享父进程内存页）
  ↓ 遍历内存，写 RDB 文件
  ↓ 写完后用新文件替换旧文件
子进程退出
```

**COW（Copy On Write）机制：**
```
fork 之后：
  父子共享同一份物理内存页（标记为只读）

  父进程要修改某个页面：
    → 操作系统复制这个页面给父进程
    → 父进程在新页面修改
    → 子进程仍读取旧页面（不受影响）

  只有被修改的页面才真正复制 → 内存开销小
```

> 🧠 **COW 就像考试时老师发卷子**——所有人拿到的是一样的卷子（共享）。某同学要在卷子上打草稿时，老师给他一张新的草稿纸（复制），不影响原卷子内容。

### 3.3 AOF 详解

**三种刷盘策略：**

| 策略 | 行为 | 数据安全 | 性能 |
|------|------|---------|------|
| `always` | 每条命令都 fsync | 不丢数据 | 最低（纯 HDD 只有几百 QPS） |
| `everysec` | 每秒 fsync 一次 | 最多丢 1 秒 | 折中（**推荐**） |
| `no` | 由 OS 决定何时 fsync | 可能丢很多 | 最高 |

**AOF 重写（压缩）：**
```
重写前（AOF 膨胀）：        重写后（压缩）：
SET key a                   SET key c
SET key b         →         只保留最终值！
SET key c
LPUSH list 1 2 3
LPUSH list 4 5 6            LPUSH list 1 2 3 4 5 6
```

`bgrewriteaof`：fork 子进程，读内存数据，生成新的精简 AOF 文件，替换旧文件。

### 3.4 混合持久化（Redis 4.0+）

```
混合 AOF 文件结构：
┌──────────┬─────────────────────────────────┐
│  RDB 部分 │  AOF 增量部分                    │
│ (快照头)  │  (快照之后的写命令, AOF 格式)      │
└──────────┴─────────────────────────────────┘

恢复时：先加载 RDB 头（快）→ 再回放 AOF 尾部（补最新的数据）
```

> [!tip] 选型建议
> - 可以允许几分钟数据丢失：纯 RDB
> - 数据不能丢：RDB + AOF 同时开启（混合模式最好）
> - 追求极致安全：`always` 刷盘（但注意性能代价）

## 四、缓存模式

### 4.1 Cache Aside（旁路缓存）—— 最常用

```
读操作：
┌─────────┐   ①查缓存    ┌─────────┐
│ 应用    │ ──────────→ │ Redis   │
│         │ ←────────── │         │
└─────────┘   miss      └─────────┘
     │
     │ ②查 DB
     ↓
┌─────────┐
│  MySQL  │
└─────────┘
     │
     │ ③写缓存
     ↓
把数据放入 Redis

写操作：
┌─────────┐   ①更新DB    ┌─────────┐
│ 应用    │ ──────────→ │  MySQL  │
│         │              └─────────┘
│         │   ②删除缓存  ┌─────────┐
│         │ ──────────→ │  Redis  │
└─────────┘              └─────────┘
```

> [!important] 为什么先写 DB 后删缓存（而不先删缓存）？
>
> **场景分析：**
> - 线程 A：删缓存 → 刚要写 DB → 被抢断
> - 线程 B：查缓存 miss → 读 DB（此时 A 还没写完） → 写缓存放了旧数据
> - 结果：缓存里是旧数据，DB 是新数据 → 数据不一致
>
> **先写 DB 后删缓存**：即使删缓存失败，也可以重试；缓存 miss 时读的是最新的 DB 数据。

**兜底方案——延迟双删：**
```java
// 写操作
db.update(data);
redis.del(key);
Thread.sleep(200);        // 等一会儿，确保其他读线程结束
redis.del(key);           // 再删一次
```

### 4.2 Read / Write Through（读写穿透）

```
读: 缓存无数据 → 缓存负责从 DB 加载 → 返回 → 写入缓存
写: 缓存负责更新 → 缓存负责同步写 DB

应用层不直接接触 DB，缓存代理所有操作。
```

> 🧠 **优点：** 业务代码简化，不用写"查缓存→查 DB→写缓存"逻辑。
> **缺点：** Redis 本身不提供这个模式，需要自己实现缓存中间层。

### 4.3 Write Behind（异步写回）

```
写: 应用 → 写缓存 → 立即返回（异步批量写 DB）

缓存积累一批写操作 → 定时/定量 → 批量刷入 DB
```

> [!warning] 高性能但有数据丢失风险
> Redis 宕机会丢失还没刷到 DB 的数据。适用于对数据一致性要求不高、但性能要求极高的场景（如浏览计数）。

### 四种模式对比

| 模式 | 读 | 写 | 一致性 | 性能 | 复杂度 |
|------|----|----|--------|------|--------|
| **Cache Aside** | 先查缓存→miss 查 DB | 先写 DB→删缓存 | 强（最终一致） | 中 | 低 |
| **Read Through** | 缓存代理读 DB | 缓存代理写 DB | 强 | 中 | 中 |
| **Write Through** | — | 同步写 DB+缓存 | 强 | 低 | 中 |
| **Write Behind** | — | 异步批量写 DB | 弱 | 高 | 高 |

## 五、缓存穿透 / 击穿 / 雪崩

### 5.1 三穿对比速查

| 问题 | 现象 | 根因 | 生动比喻 |
|------|------|------|----------|
| **穿透** | 请求不存在的数据 | 缓存和 DB 都没有对应数据 | 打电话查一个**不存在的手机号**，每次都查不到 |
| **击穿** | 热点 key 过期，大量并发打该 key | 过期瞬间大量请求涌入 | **秒杀商品的缓存刚好过期**，全都冲到 DB |
| **雪崩** | 大量 key 同时过期或 Redis 宕机 | 大面积缓存失效 | **所有房子同时倒塌**，人们全涌向避难所（DB） |

### 5.2 穿透——解法

**方案一：布隆过滤器（Bloom Filter）**

```
请求 → 布隆过滤器判断 key 是否存在？
  → 不存在 → 直接返回（连缓存都不用查）
  → 可能存在 → 查缓存 → 查 DB
```

> [!tip] 布隆过滤器原理
> 一个 bitmap + 多个 hash 函数。插入时把所有 hash 位标为 1；查询时所有 hash 位都为 1 才返回"可能存在"。**有误判率（不存在判为存在），但绝对不会漏判**。
>
> 就像**机场安检**——宁可多拦几个好人（误判），也绝不放过一个坏人。

**方案二：缓存空值**

```java
// 查 DB 返回 null → 缓存一个空值，防止重复查 DB
if (dbResult == null) {
    redis.setex("null:" + key, 60, "");   // 空值缓存 60 秒
}
```

| 方案 | 优点 | 缺点 |
|------|------|------|
| 布隆过滤器 | 省内存，一次过滤 | 不能删除，有维护成本 |
| 缓存空值 | 简单直接 | 多占用缓存空间，可能被恶意攻击 |
| **最佳实践** | **两者结合**：布隆过滤器前置过滤 + 无法过滤的缓存空值兜底 | |

### 5.3 击穿——解法

**方案一：互斥锁（mutex lock）**

```java
public String getWithMutex(String key) {
    String value = redis.get(key);
    if (value != null) return value;

    // 抢锁：只有一个线程去查 DB
    String lockKey = "lock:" + key;
    if (redis.setnx(lockKey, "1", 10)) {   // SETNX + EXPIRE 原子
        try {
            // 双重检查：拿到锁后再查一次缓存
            value = redis.get(key);
            if (value != null) return value;

            // 查 DB 写缓存
            value = db.query(key);
            redis.setex(key, 3600, value);
        } finally {
            redis.del(lockKey);   // 释放锁
        }
    } else {
        Thread.sleep(50);
        return getWithMutex(key);  // 递归重试
    }
    return value;
}
```

> 🧠 **双重检查的必要性：** 抢到锁时，之前抢到锁的线程可能已经写好了缓存。如果不检查，会重复查 DB。

**方案二：逻辑过期（不设 TTL，用 value 中的字段判断过期）**

```java
// 缓存结构：{data: "...", expireTime: 1234567890}
// 永不过期，由后台线程异步更新
public String getWithLogicalExpire(String key) {
    CacheObject obj = redis.get(key);
    if (obj == null) return null;

    if (obj.expireTime > System.currentTimeMillis()) {
        return obj.data;   // 未过期
    }

    // 已过期 → 抢锁 → 异步更新
    if (redis.setnx("lock:" + key, "1")) {
        threadPool.execute(() -> {
            String newData = db.query(key);
            redis.setex(key, newData);   // 更新缓存（逻辑过期时间）
            redis.del("lock:" + key);
        });
    }
    return obj.data;  // 返回旧数据（让用户先用着，后台在更新了）
}
```

| 方案 | 优点 | 缺点 |
|------|------|------|
| 互斥锁 | 一致性高，实现简单 | 有等待时间，可能死锁 |
| 逻辑过期 | 响应快，不用等锁 | 返回的可能还是旧数据，实现复杂 |

### 5.4 雪崩——解法

**三大组合拳：**

1. **随机过期时间**：`TTL = 基础时间 + random(0, 基础时间*30%)`，避免集体到期
2. **Redis 高可用**：主从 + 哨兵 + 集群，避免单点故障
3. **服务降级 + 限流**：Redis 挂了也不让所有请求直接打 DB，限流熔断保底

```java
// 随机过期时间示例
int ttl = 3600 + ThreadLocalRandom.current().nextInt(0, 1800);  // 1小时 ± 30分钟
redis.setex(key, ttl, value);
```

> [!tip] 雪崩类比
> 就像**春节高速**——如果所有人都同一时间出发（TLL 一样），高速公路（DB）必然瘫痪。用随机过期时间就像让大家**错峰出行**，分散压力。

## 六、过期策略与淘汰策略

### 6.1 过期 key 的删除策略

| 策略 | 行为 | 评价 |
|------|------|------|
| **定时删除** | 每个 key 设一个定时器，到期就删 | CPU 开销大（每个 key 一个定时器），不采用 |
| **惰性删除** | 访问 key 时检查是否过期，过期就删 | 内存不友好（过期 key 占空间没人访问就一直留着） |
| **定期删除** | 每隔一段时间随机抽查一批 key，删除过期的 | **Redis 实际采用：惰性 + 定期 组合** |

**定期删除具体机制：**

```
每隔 100ms 执行一次：
  1. 从设置了过期时间的 key 中随机抽取 20 个
  2. 删除其中过期的
  3. 如果过期比例 > 25%，重复步骤 1（说明过期 key 密集）
  4. 但有时间上限（25ms），防止长期占用 CPU
```

> 🧠 **为什么是"随机抽查"而不是"全量扫描"？**
> Redis 是单线程，全量扫描会长时间阻塞其他命令。随机抽查是**用概率换性能**——抽查到的 key 够多时，遗漏的也会在惰性删除中处理。

### 6.2 8 种内存淘汰策略

当内存达到 `maxmemory` 时触发：

| 策略 | 范围 | 淘汰规则 |
|------|------|----------|
| `noeviction` | — | 不淘汰，写入报错（**默认**） |
| `allkeys-lru` | 所有 key | 最近最少使用（LRU 近似） |
| `allkeys-lfu` | 所有 key | 最不经常使用（LFU） |
| `allkeys-random` | 所有 key | 随机淘汰 |
| `volatile-lru` | 设了过期时间的 key | 最近最少使用 |
| `volatile-lfu` | 设了过期时间的 key | 最不经常使用 |
| `volatile-random` | 设了过期时间的 key | 随机淘汰 |
| `volatile-ttl` | 设了过期时间的 key | 最快过期的 |

**推荐使用：** `allkeys-lru`（通用场景）或 `allkeys-lfu`（热点数据明显）。

### 6.3 LRU vs LFU

| 维度 | LRU（最近最少使用） | LFU（最不经常使用） |
|------|---------------------|---------------------|
| 淘汰依据 | 时间：多久没被访问 | 频率：被访问了多少次 |
| 场景 | 热点随时间变化 | 热点相对固定 |
| 缺点 | 偶发性大批量访问会挤掉热点 | 历史高频 key 难以淘汰 |

> [!important] Redis LRU 是近似实现
> Redis 不会维护完整的访问时间链表（开销太大），而是**随机抽样 N 个 key（默认 5 个），淘汰其中最久未使用的**。`maxmemory-samples 10` 加大采样数可提高精度。就像**老师随机挑 5 个学生看有没有作弊，而非全班逐一检查。**

## 七、分布式锁

### 7.1 手写版——SET NX EX + Lua 脚本释放

```bash
# ✅ 加锁：原子操作，一条命令完成
SET lock:order thread-001 NX EX 30
# NX：key 不存在才设（排他性）
# EX 30：30秒自动过期（防死锁）
# thread-001：value 放自己身份标识（防误删别人的锁）
```

> [!warning] 为什么不能用 SETNX + EXPIRE 两步？
> ```bash
> SETNX lock key   # 如果执行完这一步后程序崩溃/宕机
> EXPIRE lock 30   # 这步没执行 → 锁永远不会释放 → 死锁！
> ```

**释放锁——Lua 脚本原子操作：**

```lua
-- 先判断 value 是不是自己加的锁，是才删
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

> 🧠 **为什么判断 value？** 防止 A 的锁过期被 B 拿到后，A 把 B 的锁删了。就像**宾馆房卡**——你只能退自己的房间，不能退别人的房间。

### 7.2 Redisson——看门狗（Watch Dog）自动续期

手写版的致命问题：**业务执行 35 秒，锁 30 秒就过期了**。锁自动释放，其他线程也能拿到锁 → 安全漏洞。

**看门狗机制：**

```
加锁成功
  ↓
启动 Watch Dog 定时任务（默认每 10 秒检查一次）
  ↓
如果线程还持有锁（判断 clientId 一致）→ 续期到 30 秒
  ↓
循环续期...直到业务完成
  ↓
unlock → 关闭看门狗 → 删除锁 key
```

> 🧠 **看门狗 = 保安**：每隔一会来检查锁还在不在，在就给你续期。如果你（JVM）宕机了，看门狗也停了，锁自然过期。

**Redisson 加锁源码思想：**
```java
// 伪代码表达思想
RLock lock = redisson.getLock("lock:order");
lock.lock();  // 加锁 + 自动启动看门狗
try {
    // 业务逻辑（多久都行，看门狗会续期）
} finally {
    lock.unlock();  // 释放锁 + 关闭看门狗
}
```

### 7.3 RedLock——多节点场景

适用于**主从切换时锁丢失**的场景：

```
主节点：锁 order → 还没同步到从 → 主挂了 → 从变主 → 锁丢了！

RedLock 解法：
  N 个独立 Redis 节点（不是主从，是独立的，比如 5 个）

  加锁：向 N 个节点依次请求加锁
    → 超过半数（N/2+1）加锁成功 → 才算获取锁成功
    → 锁有效期 = TTL - 加锁总耗时

  释放：向所有节点发送释放命令
```

> [!warning] RedLock 争议
> RedLock 在实际生产中应用有限。工程实践更推荐用 **ZooKeeper / Etcd** 做分布式锁，因为 CP 系统比 AP 系统（Redis）更适合一致性锁场景。

### 7.4 分布式锁演进总结

```
阶段 1: SETNX + EXPIRE（两步非原子 → 死锁）
  ↓
阶段 2: SET NX EX（原子加锁，但仍要手动过期）
  ↓
阶段 3: SET NX EX + Lua 解锁（原子释放，防止误删）
  ↓
阶段 4: Redisson 看门狗（自动续期，长业务不慌）
  ↓
阶段 5: RedLock（多节点，防主从切换锁丢失）
  ↓
阶段 6: ZK/Etcd（强一致性场景终极方案）
```

## 八、高可用

### 8.1 演进路线

```
单机 Redis
  ↓ 容量不够 / 并发瓶颈
主从复制（读写分离）
  ↓ 主挂了谁来管？
哨兵 Sentinel（自动故障转移）
  ↓ 容量瓶颈（单机内存有上限）
集群 Cluster（数据分片，水平扩容）
```

### 8.2 主从复制

```
写请求 → 主节点
读请求 → 从节点（可以多个）

主从同步流程：
  从连上主 → 发 PSYNC 命令
    → 首次连接：全量复制（主 bgsave 生成 RDB → 从加载）
    → 断线重连：增量复制（主把断开期间的命令发给从）
```

**主从问题：**
- 主挂了不能自动切换（需要人工介入）
- 读写分离有数据延迟（主写完，从还没同步到）
- 所有写都在主节点，容量还是受限于单机内存

### 8.3 哨兵（Sentinel）

```
哨兵集群（至少 3 个，必须奇数）
  ├── 监控：定期 PING 主从节点
  ├── 通知：主挂了 → 哨兵之间投票确认 → 选新主
  ├── 自动故障转移：将旧主的从节点提升为新主
  └── 配置提供者：客户端问哨兵"现在主是谁？"
```

**故障转移流程：**
```
1. 主观下线（SDOWN）：
   一个哨兵 PING 不通主节点 → 认为主观下线

2. 客观下线（ODOWN）：
   多数哨兵（quorum）都认为主管下线 → 确认客观下线

3. 选主（Leader Sentinel）：
   哨兵之间投票选出一个 Leader 来执行故障转移

4. 选新主：
   优先级高的从 → 复制偏移量大的 → runid 小的

5. 通知：
   告知其他从节点新主是谁，告知客户端
```

> [!important] 为什么哨兵至少要 3 个？
> 1. **投票需要多数**：2 个哨兵中 1 个挂了就剩 1 个，无法形成多数决议
> 2. **奇数避免脑裂**：3 个哨兵可以容忍 1 个挂掉，仍能 2/3 形成多数

### 8.4 集群（Cluster）

```
数据分片：slot = CRC16(key) % 16384

┌─────────────────────────────────────────────────────┐
│  16384 个 slot 槽位                                 │
│  0-5460  │  5461-10922  │  10923-16383              │
│  节点A   │   节点B      │   节点C                    │
│  (主)    │   (主)       │   (主)                     │
│  + 从A   │   + 从B      │   + 从C                    │
└─────────────────────────────────────────────────────┘

请求任意节点 → 计算 slot →
  对路 → 直接处理
  不对路 → 返回 MOVED 重定向 → 客户端去正确节点
```

**为什么是 16384 个槽（不是 65536）？**
| 槽数 | 心跳报文大小 | 说明 |
|------|-------------|------|
| 16384 | 约 2KB | Redis 集群节点间通过 Gossip 协议同步状态，心跳包包含槽位 bitmap |
| 65536 | 约 8KB | 心跳包太大，浪费带宽。而且 1000 个节点的集群都够用，16384 绰绰有余 |

**集群扩容缩容：**
```
扩容（加新节点）：
  新节点加入 → 把部分 slot 迁移到新节点 → 完成扩容

缩容（下线节点）：
  把节点上的 slot 迁移到其他节点 → 下线该节点

迁移过程中：目标 key 在迁移途中 → 返回 ASK 重定向（临时重定向）
```

### 8.5 一致性哈希（补充概念，通用方案）

并非 Redis 集群默认方案，但面试常考，用于**无中心节点的分布式系统**：

```
哈希环：0 → 2^32-1 连接成环

         0
    Node A
   /     \
  /       \       Node A hash=100
 /         \      Node B hash=2000
|   哈希环  |     Node C hash=5000
 \         /
  \       /
   Node C ← Node B
   2^32-1

key 定位：计算 key 的 hash → 在环上顺时针找第一个节点
```

**节点变动时：**
```
新增节点（如 hash=300）：
  → 只影响 Node B 上 hash 在 (100, 300] 范围内的 key
  → 这些 key 重新路由到新节点
  → 其他 key 不受影响 ✅

删除节点（Node B 下线）：
  → Node B 上的 key 顺时针移到 Node C
  → 其他 key 不受影响 ✅
```

> [!important] 数据倾斜怎么办？——虚拟节点
> 如果只有 3 个实体节点，它们在哈希环上的位置不均匀，会导致有的节点数据特别多。
>
> **虚拟节点：** 每个实体节点对应 N 个虚拟节点（如 150 个），分散在哈希环上。key 先映射到虚拟节点，虚拟节点再映射到实体节点。就像**每个人买多个彩票号码**，中奖概率更均匀。
>
> ```
> 实体节点 A → 虚拟节点 A0, A1, A2, ..., A149（分散在环上）
> 实体节点 B → 虚拟节点 B0, B1, B2, ..., B149
> ```

**一致性哈希 vs Redis Cluster：**

| 维度 | 一致性哈希 | Redis Cluster |
|------|-----------|---------------|
| 寻址方式 | 哈希环顺时针找 | CRC16 % 16384 → slot → 节点 |
| 扩缩容影响 | 只影响相邻节点 | slot 精细迁移 |
| 适用场景 | 无中心架构（如 Dubbo 负载均衡） | Redis 官方集群方案 |
| 是否中心化 | 否 | 否（Gossip 协议去中心） |

## 九、事务

### 9.1 MULTI / EXEC 基本用法

```bash
MULTI           # 开启事务
SET k1 v1       # 命令入队
SET k2 v2       # 命令入队
INCR counter    # 命令入队
EXEC            # 批量执行，返回结果数组
```

**Redis 事务的三不特性：**

| 特性 | MySQL 事务 | Redis 事务 |
|------|-----------|-----------|
| 原子性 | 全部成功或全部回滚 | **不支持回滚** |
| 隔离性 | 多版本（MVCC）、锁 | 串行执行（单线程天然隔离） |
| 持久性 | 写磁盘 | 取决于持久化配置 |
| 一致性 | 保证 | 保证 |

### 9.2 编译错误 vs 运行时错误

```bash
# 场景一：语法错误 → 全部不执行
MULTI
SET k1 v1
SET k2          # 参数不足！编译时就发现了
EXEC            # 整个事务不执行

# 场景二：运行时错误 → 正确命令照常执行，出错的不影响
MULTI
SET k1 v1
INCR k1         # k1 是 "v1"，不能 INCR，运行时才出错
SET k2 v2       # 这条会正常执行！
EXEC            # 结果：[OK, ERROR, OK]
```

> [!important] Redis 为什么不支持回滚？
> 1. **保持简单和高效**：回滚机制会极大增加复杂度
> 2. **Redis 的错误通常来自语法/类型问题**：开发阶段就该发现，不需要运行时回滚
> 3. **哲学差异**：Redis 追求极简高性能，不是 ACID 数据库

### 9.3 WATCH 乐观锁

```bash
WATCH mykey    # 监视 mykey
val = GET mykey
val = val + 1
MULTI
SET mykey val
EXEC           # 如果 EXEC 之前 mykey 被别人改过 → 返回 nil（事务不执行）
```

```
WATCH 机制：
  → 监视一个或多个 key
  → 执行 EXEC 时检查这些 key 在 WATCH 之后是否被修改
  → 被修改了 → 整个事务取消（返回 nil）
  → 没被修改 → 执行事务
```

> 🧠 **WATCH 就是乐观锁**：读数据 → 做计算 → 写之前看看数据有没有被人改过。就像**图书馆借书**——先查库存（WATCH），借走时（EXEC）再确认一次库存没被别人借走。

### 9.4 Lua 脚本 = 真正的原子性

```lua
-- Lua 脚本在 Redis 中执行时，整个脚本是原子操作
-- 不会被其他命令插入

local val = redis.call("GET", KEYS[1])
if val == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```bash
# Redis 执行 Lua 脚本
EVAL "return redis.call('SET', KEYS[1], ARGV[1])" 1 mykey "hello"
```

**Lua 脚本 vs MULTI/EXEC：**

| 维度 | MULTI/EXEC | Lua 脚本 |
|------|-----------|----------|
| 原子性 | 执行期间不插入其他命令 | 整个脚本不插入其他命令 |
| 条件判断 | 只在 WATCH 时能用（乐观锁） | 脚本内可以任意 if/else 逻辑 |
| 回滚 | 不支持 | 不支持 |
| 适用场景 | 简单批量操作 | 需要条件判断 + 原子性（如分布式锁释放） |

## 十、消息队列

### 10.1 三种方式的对比

| 方案 | 实现方式 | 持久化 | 消费确认 | 推荐度 |
|------|---------|--------|---------|--------|
| **List** | LPUSH + BRPOP | 有（AOF/RDB） | 无（弹出即删除） | 简单场景可用 |
| **Pub/Sub** | PUBLISH + SUBSCRIBE | 无 | 无 | **不推荐（丢消息）** |
| **Stream** | XADD + XREADGROUP | 有 | 有（XACK） | **最推荐** |

### 10.2 List 实现消息队列

```bash
# 生产者
LPUSH queue:order message1
LPUSH queue:order message2

# 消费者（阻塞等待）
BRPOP queue:order 0   # 0=无限等待，直到有消息
```

> [!warning] List 的局限性
> - 不支持一对多（一条消息只能被一个消费者消费）
> - 不支持消费确认（消息被 BLPOP 弹出就丢了，如果消费者处理失败也没法重试）
> - 没有消息回溯能力

### 10.3 Pub/Sub 发布订阅

```bash
# 订阅者
SUBSCRIBE channel:news

# 发布者
PUBLISH channel:news "突发新闻！"
```

> [!warning] Pub/Sub 的三大致命缺陷
> 1. **消息不持久化**：没在线的订阅者收不到历史消息
> 2. **无法重试**：消息发出后没有确认机制，出了问题无法得知
> 3. **断线重连丢失**：网络抖动断开重连，断开期间的消息永久丢失
>
> **面试态度：明确表示不推荐用于生产环境的消息队列。**

### 10.4 Stream——5.0+ 真正的消息队列

```bash
# 生产消息
XADD mystream * key1 value1 key2 value2
# * 表示自动生成 ID（时间戳-序号）

# 消费者组消费
XREADGROUP GROUP mygroup consumer1 BLOCK 2000 STREAMS mystream >
# > 表示从最新未被消费的消息开始

# 确认消费
XACK mystream mygroup message-id

# 查看未确认的消息（消费者挂了可以重试）
XPENDING mystream mygroup
```

**Stream 的优势：**

| 特性 | 说明 |
|------|------|
| **消费者组** | 一条消息可被多个消费者组消费（广播），组内多个消费者分担（竞争） |
| **持久化** | 消息写入 AOF/RDB，重启不丢 |
| **消费确认** | XACK 确认，不确认的消息可被重新投递 |
| **消息回溯** | 可以按 ID 范围重新读取历史消息 |
| **阻塞读取** | BLOCK 参数支持阻塞等待 |

> 🧠 **Stream 就像 Kafka 的精简版**——有消费者组、持久化、确认机制，但不支持 partition 分片（单 Stream 单主）。

```
消费者组1（订单处理）：
  consumer_A → 消费消息
  consumer_B → 消费消息（与 A 共享消息，不重复）

消费者组2（订单通知）：
  consumer_C → 消费同一批消息（独立的组，消息重复投递）

每个消费者组维护自己的消费位置指针
```

## 十一、内存管理

### 11.1 jemalloc 内存分配器

> [!important] jemalloc 是什么？
> Redis 默认使用 jemalloc 作为内存分配器，替代 Linux 默认的 glibc malloc。

**jemalloc 的优势：**

| 特性 | glibc malloc | jemalloc |
|------|-------------|----------|
| **内存碎片** | 高碎片（长时间运行后严重） | 低碎片（多 size class 精确匹配） |
| **多线程性能** | 锁争用严重 | arena 分区设计，锁争用少 |
| **内存统计** | 粗略 | 精确（可用 `MEMORY STATS` 查看） |
| **适用场景** | 通用 | 长时运行、频繁分配释放的服务端程序 |

**jemalloc 的 size class 设计：**
```
小对象（如 8B, 16B, 32B ...）→ 精确分配，不浪费
大对象 → 按页分配（4KB 对齐）

每种大小有独立的"篮子（bin）"：
  8B bin → 只存 8B 的对象
  16B bin → 只存 16B 的对象
  ...

释放的内存直接放回对应的 bin，下次分配直接从 bin 取 → 减少碎片
```

> 🧠 **jemalloc 就像快递柜**——每个大小格子对应不同的快递大小，小包裹放小格子，大包裹放大格子。而不是把所有快递塞进一个大房间（容易乱），也不是每个快递精确打一个包裹（浪费空间）。

### 11.2 紧凑数据结构

Redis 使用多种紧凑结构来节省内存：

| 结构 | 压缩方式 | 节省内存原理 |
|------|---------|-------------|
| **ziplist** | 连续内存，每个 entry 只记录 prevlen + encoding + data | 省去链表指针（2x8字节） |
| **intset** | 连续整数数组，统一编码（int16/int32/int64） | 省去指针 + 统一整数位宽 |
| **quicklist** | ziplist + 链表混合 | 两者取其优 |
| **listpack** | ziplist 的替代（7.0+），解决了连锁更新问题 | 每个 entry 只记录自己的长度 |

**ziplist 连锁更新问题：**
```
ziplist 节点格式：[prevlen][encoding][data]

如果前面节点的长度从 253 变成 254：
  prevlen 从 1 字节变成 5 字节
  当前节点变大了 → 可能触发下一个节点的 prevlen 也变大
  → 连锁反应 → O(n^2) 级别的重写
```

Redis 7.0 引入 **listpack** 替代 ziplist，每个 entry 只记录自身长度，彻底解决连锁更新。

### 11.3 内存回收相关

```
过期 key 删除（过期键空间）
    ↓
内存淘汰策略（maxmemory 达到上限）
    ↓
主动内存整理（内存碎片整理，4.0+）
```

**内存碎片整理（activedefrag，4.0+）：**
```
内存状态：
已分配: [AA_BB_CC___DD_]  (_ 表示空隙碎片)
整理后: [AABBCCDD_______]  合并碎片，形成连续可用空间
```

Redis 4.0+ 支持自动碎片整理，在 CPU 空闲时逐步完成，不影响正常操作。

### 11.4 内存优化实战建议

| 优化手段 | 说明 |
|----------|------|
| **控制 key 数量** | key 本身占用内存，能用 hash 聚合就用 hash 聚合 |
| **短 key 名** | `user:1001:profile` 优于 `user_profile_with_detail_info_of_user_1001` |
| **选对数据结构** | 小集合用 intset/ziplist（默认配置下 512 个元素自动切换） |
| **设置 maxmemory** | 必须设，防止 OOM |
| **合理淘汰策略** | 缓存场景用 allkeys-lru，不要用默认 noeviction |
| **监控内存碎片率** | `INFO memory` 查看 `mem_fragmentation_ratio`，> 1.5 考虑启用碎片整理 |

## 十二、Redis vs Memcached

### 12.1 核心对比

| 维度 | Redis | Memcached |
|------|-------|-----------|
| **数据类型** | String、List、Set、Hash、ZSet 等 | 仅 String（key-value） |
| **持久化** | RDB + AOF + 混合 | 不支持（纯内存） |
| **高可用** | 主从、哨兵、集群 | 客户端一致性哈希 |
| **线程模型** | 单线程处理命令（6.0 IO 多线程） | 多线程 |
| **内存管理** | jemalloc（多 size class 精细管理） | Slab Allocation（固定大小 chunk） |
| **网络模型** | epoll IO 多路复用 | libevent 多线程处理 |
| **过期策略** | 惰性删除 + 定期删除 | 惰性删除（访问时才检查）+ LRU |
| **事务** | 支持（MULTI/EXEC、Lua） | 不支持 |
| **发布订阅** | 支持 | 不支持 |
| **Lua 脚本** | 支持 | 不支持 |
| **集群** | 原生集群（16384 slot） | 客户端分片 |
| **性能** | QPS 10万+（单线程） | QPS 10万+（多线程） |
| **适用场景** | 复杂缓存 + 分布式锁 + 排行榜 + 消息队列 | 简单 key-value 缓存 |

### 12.2 深入对比

**Memcached 的 Slab Allocation：**
```
将所有内存切分成不同大小的 slab class：
  1KB slab → 存 1KB 的数据
  2KB slab → 存 2KB 的数据
  4KB slab → 存 4KB 的数据
  ...

优点：内存碎片少，分配快速
缺点：固定大小，浪费空间（存 1.1KB 的 value 要用 2KB slab）
```

**Redis jemalloc vs Memcached Slab：**
| 维度 | jemalloc（Redis） | Slab（Memcached） |
|------|-------------------|-------------------|
| 分配粒度 | 多级 size class（8B, 16B, ...） | chunk 固定大小 |
| 内存利用率 | 高（精确匹配） | 较低（可能有内部碎片） |
| 复杂度 | 复杂 | 简单 |

### 12.3 选型建议

```
选 Redis 的场景：
  ✅ 需要复杂数据类型（排行榜→ZSet，共同好友→Set）
  ✅ 需要持久化（数据不能丢）
  ✅ 需要高可用（主从+哨兵+集群）
  ✅ 需要分布式锁
  ✅ 需要消息队列 / 发布订阅

选 Memcached 的场景：
  ✅ 纯 key-value 缓存，数据可丢
  ✅ 多核 CPU 利用率更高（多线程）
  ✅ 部署运维极简
  ✅ 公司已有技术栈
```

> [!tip] 面试背诵
> "Redis 和 Memcached 都是高性能内存缓存。Redis 的优势在于**数据类型丰富、支持持久化和高可用、功能生态更全**，本质是一个数据结构服务器；Memcached 更纯粹，就是**多线程 key-value 缓存**，部署简单，但功能单一。现代 Java 后端项目首选 Redis。"

## 面试满分话术

### 话术一：Redis 整体介绍（开场话术）

> "Redis 是一个基于内存的高性能 key-value 数据库，我主要从几个方面来理解它：
>
> 第一，**数据类型丰富**——不仅有 String，还有 List、Set、Hash、ZSet 等，底层分别用 SDS、quicklist、hashtable、跳表等结构实现。其中跳表是一个重点，通过多层索引实现 O(log n) 的范围查询，比红黑树实现简单、并发友好。
>
> 第二，**高性能**——单线程处理命令，配合 epoll IO 多路复用，有效避免锁竞争和上下文切换。瓶颈不在 CPU 而在网络 IO，所以 6.0 引入了 IO 多线程处理网络读写，命令执行保持单线程。
>
> 第三，**高可用体系**——从主从复制（读写分离）到哨兵（自动故障转移）再到集群（数据分片），实现了从单机到分布式的完整演进。集群使用 CRC16 对 16384 个槽取模进行数据分片。
>
> 第四，**持久化方案**——RDB 快照恢复快但可能丢数据，AOF 日志更安全但文件大，4.0 的混合持久化兼顾两者。
>
> 第五，**丰富的功能生态**——分布式锁（SET NX EX + 看门狗自动续期）、消息队列（Stream 支持消费组和确认机制）、缓存策略（Cache Aside 先写 DB 后删缓存）等。"

### 话术二：缓存三穿（高频必问）

> "缓存问题我总结为三个场景——穿透、击穿、雪崩。
>
> **穿透**是查不存在的数据，每次请求都穿过缓存打到 DB。就像查一个不存在的手机号，永远查不到。解法用布隆过滤器做前置过滤，再加缓存空值兜底。
>
> **击穿**是热点 key 刚好过期，此时大量并发请求直接打到 DB。就像秒杀商品的缓存刚好失效。解法有两种：互斥锁——只让一个线程去查 DB 回写，其他线程等待；逻辑过期——缓存不设 TTL，由后台线程异步刷新。
>
> **雪崩**是大面积 key 同时过期或者 Redis 整体宕机。就像所有房子同时倒塌，人们全涌向避难所。解法是过期时间加随机值错峰、Redis 高可用部署、以及服务降级限流保底。"

### 话术三：分布式锁演进（展示深度）

> "分布式锁我理解它的演进过程：
>
> 第一版最简单，用 `SETNX + EXPIRE`，但两个命令不是原子操作，如果 SETNX 成功后宕机，锁永远不会释放。
>
> 第二版用 `SET key value NX EX 30`，一条命令原子完成加锁和设置过期时间。
>
> 第三版释放锁也不能简单 DEL——要判断 value 是不是自己设的。用 Lua 脚本原子执行 GET 判断 + DEL 删除，防止误删别人的锁。
>
> 但这样还有问题：如果业务执行超过了锁的过期时间怎么办？这就引入了第四版——Redisson 的看门狗机制，后台线程每 10 秒自动续期到 30 秒，业务完成后再关闭。
>
> 再进一步，主从架构下主挂了锁还没同步到从——锁就丢了。这时候需要第五版 RedLock，向 N 个独立 Redis 节点申请锁，过半成功才算拿到。不过实际工程中，对强一致性要求的锁场景，更推荐用 ZK 或 Etcd。"

## 记忆口诀

> [!tip] 口诀一：五大类型
> **S（String）List（快速列表）Set（哈希/整数）Hash（值压），ZSet 跳表排行杀**
>
> String→SDS(len+free+buf)，List→quicklist，Set→hashtable/intset，
> Hash→hashtable/ziplist，ZSet→skiplist+hashtable

> [!tip] 口诀二：线程模型
> **单线处理命与令，epoll 监听多路行。后台线程干重活，6.0 IO 多线读。**
>
> 单线程处理命令 + epoll IO 多路复用 + 后台线程(close/bgsave/bgrewrite) + 6.0 IO 多线程只处理网络读写

> [!tip] 口诀三：持久化
> **快照 RDB 恢复快，日志 AOF 更安全。COW 复制在修改，混合持久两边来。**
>
> RDB 二进制快照快但有丢失窗口；AOF 日志安全但文件大；COW 只复制修改的页；混合持久化=RDB 头+AOF 尾

> [!tip] 口诀四：缓存三穿
> **不存穿，热点破，大雪崩。布隆防穿锁防破，随机过期高可防雪崩。**
>
> 穿透=布隆过滤器+缓存空值，击穿=互斥锁+逻辑过期，雪崩=随机过期+高可用+限流

> [!tip] 口诀五：分布式锁
> **NX EX 原子加，Lua 判断再释放。看门狗自动续，红锁过半才作数。**
>
> SET NX EX 原子加锁 → Lua 脚本(判断value→DEL)原子释放 → Redisson看门狗自动续期 → RedLock N个独立节点过半成功

> [!tip] 口诀六：高可用
> **主从读写分，哨兵自动选，集群分片散。一六三八四，心跳包大小。**
>
> 主从(读写分离) → 哨兵(自动选举,至少3奇数) → 集群(CRC16%16384分片) → 16384因心跳包仅2KB

> [!tip] 口诀七：淘汰策略
> **惰性访问才检查，定期百毫随二十。8 种淘汰分两类，全键和有过期键。**
>
> 惰性删除+定期删除(100ms随机20个); 8种淘汰分allkeys和volatile两类, 推荐allkeys-lru

> [!tip] 口诀八：Redis vs Memcached
> **Redis 类型多持久化分布式锁一把抓，MC 多线程纯缓存部署简答。**
>
> Redis=瑞士军刀(功能全); Memcached=水果刀(只做KV缓存但做得好)
