# Redis 缓存策略

> [!summary] 速记总纲
> **三大块：缓存模式（怎么读写缓存）、缓存三穿（出问题怎么办）、过期淘汰（内存不够怎么办）。逐一击破，面试不慌。**

---

## 一、缓存模式

缓存模式决定了**应用层如何与缓存（Redis）和数据库（MySQL）交互**。主要有四种模式。

### 1.1 Cache Aside（旁路缓存）—— 最常用

**读操作流程：**

```
应用 → ① 查缓存
         ├── 命中（hit）→ 直接返回
         └── 未命中（miss）→ ② 查数据库 → ③ 把结果写入缓存 → 返回
```

**写操作流程：**

```
应用 → ① 先更新数据库 → ② 删除缓存（不是更新缓存！）
```

**伪代码：**

```java
// ========== 读 ==========
public String get(String key) {
    // 1. 先查缓存
    String value = redis.get(key);
    if (value != null) {
        return value;                    // 命中，直接返回
    }
    // 2. 缓存未命中 → 查数据库
    value = db.select(key);
    if (value != null) {
        redis.setex(key, 3600, value);   // 3. 写入缓存
    }
    return value;
}

// ========== 写 ==========
public void update(String key, String value) {
    // 1. 先更新数据库
    db.update(key, value);
    // 2. 删除缓存（不是更新缓存！）
    redis.del(key);
}
```

> [!important] 核心问题：为什么先写 DB 后删缓存，而不是先删缓存后写 DB？
>
> **先删缓存再写 DB 的并发问题：**
> - 线程 A：删缓存 → 刚要写 DB → 被操作系统抢断
> - 线程 B：查缓存（已空，miss）→ 读 DB（A 还没来得及写完，读到旧数据）→ 把旧数据写回缓存
> - 结果：**缓存里是旧值，DB 是新值 → 数据不一致**
>
> **先写 DB 后删缓存（推荐）：**
> - 写 DB 期间，缓存里还是旧值，读请求读到旧值（短暂不一致，可以接受）
> - DB 写完后删缓存，后续读请求就会从 DB 读到新值再写入缓存
> - 即使删缓存失败，也可以通过重试机制兜底

**兜底方案 —— 延迟双删：**

```java
// 写操作：先写 DB，删缓存，等几百毫秒再删一次
db.update(data);
redis.del(key);
Thread.sleep(200);         // 等待其他读线程结束（读到旧值的可能仍在写缓存）
redis.del(key);            // 再删一次，确保不会残留旧值
```

> [!warning] 延迟双删的局限性
> 1. `Thread.sleep` 阻塞业务线程，影响性能（通常用异步延迟删除优化）
> 2. 等待时间不好确定，太短没效果，太长影响性能
> 3. 如果第二次删除也失败，依然可能不一致
>
> 更推荐配合 **Canal 监听 binlog** 异步删除缓存，以及 **最终一致性兜底**。

### 1.2 Read / Write Through（读写穿透）

```
Read Through：
  应用 → 查缓存 → miss → 缓存层自动查 DB → 加载到缓存 → 返回给应用
  应用层不感知 DB，由缓存层代理所有读操作

Write Through：
  应用 → 写缓存 → 缓存层自动同步写 DB → 返回
  应用层不直接操作 DB，由缓存层代理所有写操作
```

> [!note] 特点
> - **优点**：业务代码极度简洁，不需要自己写"查缓存 → miss → 查 DB → 写缓存"逻辑
> - **缺点**：Redis 本身不提供这个模式，需要自己开发缓存中间层（如封装一层 CacheService）
> - **适用场景**：对一致性要求高的系统，缓存层统一管理读写，避免业务层的遗漏

### 1.3 Write Behind（异步写回）

```
Write Behind：
  应用 → 写缓存 → 立即返回（不写 DB！）
           ↓
       缓存层在后台 → 积累一批写操作 → 定时/定量 → 批量刷入 DB
```

> [!warning] 高性能但数据有丢失风险
> - **优点**：写入性能极高（只写缓存，不等待 DB），适合高并发写入场景
> - **缺点**：Redis 宕机会丢失还没刷到 DB 的数据，数据一致性弱
> - **适用场景**：浏览计数、点赞数、PV/UV 统计等对绝对精确性要求不高的场景
> - **不是 Redis 内置功能**：需要自己实现批量刷盘逻辑

### 1.4 四种模式对比

| 模式 | 读策略 | 写策略 | 一致性 | 写入性能 | 复杂度 | 谁操作 DB |
|------|--------|--------|--------|----------|--------|-----------|
| **Cache Aside** | 先查缓存，miss 查 DB | 先写 DB，删缓存 | 最终一致（强） | 中 | **低** | 业务层 |
| **Read Through** | 缓存代理读 DB | — | 强 | — | 中 | 缓存层 |
| **Write Through** | — | 同步写 DB + 缓存 | 强 | **低** | 中 | 缓存层 |
| **Write Behind** | — | 异步批量写 DB | **弱** | **高** | 高 | 缓存层 |

> [!tip] 面试一句话总结
> "实际项目中 **Cache Aside 是最主流的选择**，读先查缓存 miss 查 DB，写先更新 DB 再删缓存。Read/Write Through 需要封装缓存中间层，Write Behind 类似 MySQL 的 write-back，性能高但可能丢数据。"

## 二、缓存穿透 / 击穿 / 雪崩

### 2.1 三穿区别速查表

| 问题 | 现象 | 根因 | 生动比喻 |
|------|------|------|----------|
| **穿透** | 请求的数据**缓存和 DB 都没有** | 恶意攻击或业务逻辑漏洞 | 打电话查一个**不存在的手机号**，每次查都查不到，但每次都要翻通讯录 |
| **击穿** | **单个热点 key** 刚好过期，大量并发同时访问 | 过期瞬间，所有请求同时打向 DB | **秒杀商品的缓存刚好失效**，所有人同时冲向收银台 |
| **雪崩** | **大面积 key** 同时过期或 Redis 宕机 | 集中过期 / 单点故障 | **所有房子同时倒塌**，人们全涌向唯一避难所（DB） |

> [!tip] 一句话区分
> - **穿透**：数据根本不存在，每次都要穿透到底层查一遍
> - **击穿**：单个热点 key 过期，一瞬间大量请求"击穿"缓存打到 DB
> - **雪崩**：大面积 key 过期或 Redis 挂了，像雪崩一样不可阻挡

### 2.2 穿透 —— 解法

#### 方案一：布隆过滤器（Bloom Filter）

```
请求 → 布隆过滤器判断 key 是否存在？
  ├── 不存在 → 直接返回空（缓存都不用查！）
  └── 可能存在 → 查缓存 → 查 DB
```

> [!tip] 布隆过滤器原理
> 一个 **bitmap** + 多个 **hash 函数**。
> - **插入**：对 key 做 N 次 hash，把 bitmap 中对应位置标为 `1`
> - **查询**：对 key 做 N 次 hash，所有位都是 `1` 才返回"可能存在"
> - **误判率**：可能把不存在的 key 判为存在（但概率可控），**绝不会把存在的 key 判为不存在**
>
> 就像**机场安检**——宁可多拦几个好人（误判），也绝不放过一个坏人（漏判）。

**布隆过滤器的局限性：**
- **不能删除元素**（因为 hash 位可能被其他 key 共用）
- 需要提前把所有合法 key 加载到过滤器中（初始化成本）
- 有误判率（通常控制在 1% 以内，通过调节 bit 数组大小和 hash 函数数量）

#### 方案二：缓存空值

```java
// 查 DB 返回 null → 缓存一个空值，防止重复查 DB
String value = db.select(key);
if (value == null) {
    redis.setex("null:" + key, 60, "");   // 空值缓存 60 秒
    return null;
}
```

| 方案 | 优点 | 缺点 |
|------|------|------|
| 布隆过滤器 | 省内存，一次过滤，前置拦截 | 不能删除元素，有维护成本，需要预热 |
| 缓存空值 | 实现简单，直接有效 | 多占用缓存空间（大量空 key），可能被恶意不同 key 攻击 |
| **最佳实践** | **两者结合**：布隆过滤器前置过滤 + 无法过滤的缓存空值兜底 | |

### 2.3 击穿 —— 解法

#### 方案一：互斥锁（Mutex Lock）

**核心思想：** 缓存过期时，只有**一个线程**去查 DB 重建缓存，其他线程等待或自旋。

```java
public String getWithMutex(String key) {
    // 1. 先查缓存
    String value = redis.get(key);
    if (value != null) return value;

    // 2. 缓存过期 → 尝试抢锁（只有一个线程能拿到）
    String lockKey = "lock:" + key;
    if (redis.set(lockKey, "1", "NX", "EX", 10)) {  // ← SET NX EX 原子操作
        try {
            // 3. 双重检查（DCL）：抢到锁后再查一次缓存
            //    因为可能之前抢到锁的线程已经重建好了
            value = redis.get(key);
            if (value != null) return value;

            // 4. 查 DB 回写缓存
            value = db.query(key);
            redis.setex(key, 3600, value);     // 缓存 1 小时
        } finally {
            redis.del(lockKey);                // 5. 释放锁
        }
    } else {
        // 6. 没抢到锁 → 等待 + 重试
        Thread.sleep(50);
        return getWithMutex(key);              // 递归重试（也可用自旋）
    }
    return value;
}
```

> [!important] 双重检查（DCL）的必要性
> 线程 A 抢到锁时，线程 B 可能**刚刚释放锁并写好了缓存**。如果不做二次检查，会重复查 DB，失去锁的意义。

#### 方案二：逻辑过期（不设 Redis TTL）

**核心思想：** 缓存不设 Redis 过期时间（永不过期），而是在 value 内部存储过期时间戳。过期时返回旧值，由后台线程异步刷新。

```java
// 缓存结构（JSON）：{"data": "实际数据", "expireTime": 1714780800000}

public String getWithLogicalExpire(String key) {
    // 1. 查缓存（永不过期，所以一定能拿到）
    CacheObject obj = redis.get(key);
    if (obj == null) return null;            // 真没有，返回 null

    // 2. 判断逻辑时间是否过期
    if (obj.expireTime > System.currentTimeMillis()) {
        return obj.data;                     // 未过期，直接返回
    }

    // 3. 已过期 → 抢锁，异步重建缓存
    String lockKey = "lock:" + key;
    if (redis.set(lockKey, "1", "NX", "EX", 10)) {
        // 异步重建（用线程池，不阻塞当前请求）
        threadPool.execute(() -> {
            try {
                String newData = db.query(key);
                CacheObject newObj = new CacheObject(
                    newData,
                    System.currentTimeMillis() + 3600_000  // 1小时后过期
                );
                redis.set(key, newObj.toJson());
            } finally {
                redis.del(lockKey);
            }
        });
    }
    // 4. 返回旧的缓存数据（让用户先用着，后台在更新了）
    return obj.data;
}
```

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 互斥锁 | 一致性高，实现简单，数据一定是最新的 | 有等待时间，抢不到锁的要等待或重试 | 对一致性要求高 |
| 逻辑过期 | 响应极快，不阻塞用户请求 | 返回的可能是旧数据，实现复杂 | 对一致性要求不高，追求极致性能 |
| **最佳实践** | **热点数据用逻辑过期，普通数据用互斥锁** | | |

### 2.4 雪崩 —— 解法

**三大组合拳：**

```
┌────────────────────────────────────────────────────┐
│  1. 随机过期时间（错峰）                              │
│     TTL = 基础时间 + random(0, 基础时间 × 30%)        │
│                                                      │
│  2. Redis 高可用（防挂）                              │
│     主从 + 哨兵 + 集群，避免单点故障                   │
│                                                      │
│  3. 服务降级 + 限流（保底）                           │
│     Redis 挂了也绝不裸奔，限流熔断守住 DB 底线         │
└────────────────────────────────────────────────────┘
```

**随机过期时间示例：**

```java
// 1 小时 ± 30 分钟随机偏移
int baseTTL = 3600;
int randomOffset = ThreadLocalRandom.current().nextInt(0, 1800);
redis.setex(key, baseTTL + randomOffset, value);
```

**多级缓存架构（终极方案）：**

```
请求 → 本地缓存（Caffeine） → Redis → DB
        ↓                    ↓       ↓
    命中直接返回           命中返回  最终兜底
```

> [!tip] 雪崩的生动类比
> 就像**春运高速**——如果所有人同一时间出发（TTL 一样），高速公路（DB）必然瘫痪。
> 用**随机过期时间 = 错峰出行**，分散压力。**高可用 = 多条高速**，一条堵了还有别的路。

### 2.5 三穿解法速查总表

| 问题 | 核心解法 | 辅助解法 | 一句话口诀 |
|------|----------|----------|------------|
| **穿透** | 布隆过滤器 + 缓存空值 | 参数校验、限流 | 不存在就给它个"假的" |
| **击穿** | 互斥锁 / 逻辑过期 | 热点永不过期 | 热点过期，只让一个人去重建 |
| **雪崩** | 随机过期 + 高可用 | 多级缓存、限流降级 | 错峰出行 + 多条公路 |

## 三、过期删除策略

### 3.1 三种理论策略

| 策略 | 行为 | 优点 | 缺点 |
|------|------|------|------|
| **定时删除** | 每个 key 设一个定时器，到期立即删 | 内存友好（过期立即释放） | CPU 负担大（每个 key 一个定时器），Redis 不采用 |
| **惰性删除** | 访问 key 时检查是否过期，过期则删除 | CPU 友好（不主动扫描） | 内存不友好（没人访问的过期 key 一直占着内存） |
| **定期删除** | 每隔一段时间轮询抽查一批 key | 折中方案 | 需要平衡频率和数量 |

### 3.2 Redis 实际策略：惰性删除 + 定期删除

Redis **结合两者**，互相弥补：

> **惰性删除 = 日常随手扔垃圾（访问时才检查）**
> **定期删除 = 保洁阿姨定期巡查（每隔 100ms 随机抽查）**

#### 定期删除的具体机制

```
每隔 100ms 执行一次定期删除循环：

  1. 随机抽取 20 个设置了过期时间的 key
  2. 检查是否过期 → 过期就删除
  3. 计算过期比例：
     ├── 如果过期比例 > 25% → 说明过期 key 还很密集 → 回到步骤 1 继续
     └── 如果过期比例 ≤ 25% → 本轮结束
  4. 硬性限制：单轮最多执行 25ms（防止长期占用 CPU 阻塞其他命令）
```

> [!important] 为什么是"随机抽查"而不是"全量扫描"？
> **Redis 是单线程处理命令。** 如果全量扫描所有设置了过期时间的 key（可能有几百万个），会长时间阻塞其他命令的执行，整个 Redis 在此期间就像死了一样。
>
> **随机抽查是用概率换性能**——抽查 20 个中过期比例高就继续查，低就停。即使有漏网之鱼，也会在惰性删除中处理。

## 四、内存淘汰策略

当 Redis 内存达到 `maxmemory` 阈值时，触发内存淘汰机制。

### 4.1 8 种淘汰策略速查

| # | 策略 | 作用范围 | 淘汰规则 |
|---|------|----------|----------|
| 1 | `noeviction` | — | **不淘汰**，写入时直接报错（**Redis 默认策略**） |
| 2 | `allkeys-lru` | 所有 key | 淘汰**最近最少使用**的（LRU 近似） |
| 3 | `allkeys-lfu` | 所有 key | 淘汰**最不经常使用**的（LFU） |
| 4 | `allkeys-random` | 所有 key | **随机淘汰** |
| 5 | `volatile-lru` | 设了过期时间的 key | 淘汰最近最少使用的 |
| 6 | `volatile-lfu` | 设了过期时间的 key | 淘汰最不经常使用的 |
| 7 | `volatile-random` | 设了过期时间的 key | 随机淘汰 |
| 8 | `volatile-ttl` | 设了过期时间的 key | 淘汰**最快过期的**（TTL 最短） |

**分类记忆法：**

```
allkeys-*   →  范围是所有 key（4 种：lru / lfu / random / —）
volatile-*  →  范围是设了过期时间的 key（4 种：lru / lfu / random / ttl）
noeviction  →  不淘汰（1 种：默认值）
```

**推荐策略：**

```bash
# 通用缓存场景 —— 推荐
maxmemory-policy allkeys-lru

# 热点数据明显的场景（如电商商品详情）
maxmemory-policy allkeys-lfu
```

> [!warning] 生产环境注意
> **不要把 `noeviction` 默认策略带上生产！** 内存满了写入直接报错，可能导致大量业务异常。务必在配置中显式设置淘汰策略为 `allkeys-lru`。

### 4.2 LRU vs LFU

| 维度 | LRU（Least Recently Used） | LFU（Least Frequently Used） |
|------|---------------------------|------------------------------|
| **淘汰依据** | **时间**：多久没被访问 | **频率**：被访问了多少次 |
| **核心逻辑** | 最近没被访问的 → 优先淘汰 | 访问次数最少的 → 优先淘汰 |
| **判断标准** | 最后一次访问距今的时间差 | 一段时间内的访问计数 |
| **适用场景** | 热点随时间变化（如新闻热点） | 热点相对固定（如电商热销商品） |
| **缺点** | 偶发性大批量访问会挤掉真正的热点 | 历史高频 key 访问量一旦上去就很难被淘汰 |
| **Redis 实现** | 近似 LRU（随机抽样） | 近似 LFU（Morris 计数器） |

> [!important] Redis 的 LRU 是近似实现
> 标准 LRU 需要维护完整的访问时间链表（双向链表），每次访问都要移动到链表头部，开销太大。
>
> **Redis 的做法：随机抽样 N 个 key（默认 5 个，`maxmemory-samples` 控制），淘汰其中最久未使用的。**
>
> 就像**老师随机挑 5 个学生检查作业**，而不是全班逐一检查。加大 `maxmemory-samples`（如设为 10）可提高精度，但会增加 CPU 开销。

```
标准 LRU：                     Redis 近似 LRU：
┌──┐                          随机抽 5 个 key：
│A │ ← 最近使用                [A(hit:0s), B(hit:10s), C(hit:30m), D(hit:2h), E(hit:1d)]
│B │                           淘汰 E（最久未使用）
│C │
│D │                          开销 O(N×M)而非 O(1)，但 N 很小（默认 5），M 是总 key 数
│E │ ← 最久未使用 → 淘汰
└──┘
```

### 4.3 LFU 计数器的特殊设计

**标准 LFU 的问题：** 一个 key 被访问了 1000 万次，即使现在不再被访问，访问计数也很难被超越，难以淘汰。

**Redis LFU 的改进：** 使用 **概率性计数器（Morris 计数器）**，并且访问计数会**随时间衰减**：

```
日志计数法（对数增长）：
  访问 1 次   → 计数器 +1
  访问 10 次  → 计数器 +1（概率 ~1/10）
  访问 100 次 → 计数器 +1（概率 ~1/100）
  ...
  次数越高，增长越慢（对数增长，不会无限膨胀）

衰减机制（lfu-decay-time）：
  每隔一段时间（默认 1 分钟），计数器减 1
  → 历史高频 key 如果不再被访问，计数器会慢慢降到 0
```

## 六、面试满分话术

> "关于 Redis 缓存策略，我从三个方面来回答：缓存模式、缓存三穿、过期淘汰。
>
> **第一，缓存模式方面**，我们项目最常用的是 **Cache Aside** 旁路缓存模式。读的时候先查缓存，miss 再查 DB 并回写；写的时候**先更新 DB，再删除缓存**，而不是更新缓存或先删缓存。这样做的原因是：先删缓存再写 DB 存在并发场景下读到旧值覆盖的问题，而先写 DB 后删缓存即使删缓存失败也可以通过重试机制兜底。其他模式比如 Read Through、Write Through 需要缓存层代理，Write Behind 是异步写回，性能高但可能丢数据。
>
> **第二，缓存三穿方面**，穿透是查不存在的数据——用**布隆过滤器前置拦截**加**缓存空值兜底**；击穿是热点 key 刚好过期——用**互斥锁**只让一个线程去重建缓存，或者用**逻辑过期**异步刷新让用户先用旧数据；雪崩是大面积 key 过期或 Redis 宕机——给过期时间加**随机偏移量错峰**，配合**高可用部署**（主从+哨兵+集群），以及**服务降级限流**保底。
>
> **第三，过期淘汰方面**，Redis 采用**惰性删除 + 定期删除**的组合策略。惰性删除是访问 key 时检查过期，定期删除是每 100ms 随机抽 20 个 key，过期比例超过 25% 就继续清理。当内存满了触发淘汰时，有 8 种策略，我们生产环境推荐 **allkeys-lru**。Redis 的 LRU 是近似实现，不是维护完整的访问链表，而是随机抽样 N 个 key 选最久未使用的，用概率换性能。"

## 七、记忆口诀

> [!tip] 口诀一：缓存模式
> **旁路读先查 miss 再写，写先更新 DB 再删。穿透穿读写，备异迟迟刷。**
>
> Cache Aside = 读先缓存 miss 查 DB 回写 + 写先 DB 再删缓存。Read Through 缓存代理读。Write Through 同步写。Write Behind 异步批量刷 DB。

> [!tip] 口诀二：缓存三穿
> **不存穿，热点破，大面崩。布隆防穿锁防破，随机过期高可防崩。**
>
> 穿透 = 布隆过滤器 + 缓存空值。击穿 = 互斥锁 + 逻辑过期。雪崩 = 随机过期 + 高可用 + 限流降级。

> [!tip] 口诀三：过期淘汰
> **惰性访问才删，定期百毫随抽。二十过五继续，二十五毫封顶。八种淘汰分全有过，最推全 LRU。**
>
> 惰性删除(访问时) + 定期删除(每 100ms 随机抽 20 个，过期比例 >25% 继续，限制 25ms)。
> 8 种淘汰策略分 allkeys 和 volatile 两类，推荐 allkeys-lru。

> [!tip] 口诀四：LRU vs LFU
> **LRU 看久别，LFU 看常来。近似随机抽，采样定精度。**
>
> LRU = 淘汰最久没访问的。LFU = 淘汰访问次数最少的。Redis 都是近似实现，随机抽样 N 个 key。
