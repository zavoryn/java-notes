# Redis 持久化

> [!summary] 一句话总结
> **Redis 提供三种持久化方案：RDB 快照（二进制、恢复快、可能丢数据）、AOF 日志（文本、更安全、文件大）、混合持久化（4.0+，RDB 头 + AOF 尾，兼顾速度与安全）。核心机制是 COW（写时复制），fork 子进程时父子共享内存页，修改时才复制。**

---

## 一、RDB vs AOF 对比表

| 维度        | RDB                            | AOF                            |
| --------- | ------------------------------ | ------------------------------ |
| **是什么**   | 某一时刻的内存**快照**（二进制文件）           | 所有写命令的**日志**（文本文件）             |
| **触发方式**  | `save`（阻塞）/ `bgsave`（fork 子进程） | 每条写命令追加                        |
| **文件大小**  | 小（压缩的二进制）                      | 大（文本日志，会增长）                    |
| **恢复速度**  | 快（直接加载二进制到内存）                  | 慢（逐条回放命令）                      |
| **数据安全**  | 可能丢最后一次快照后的数据                  | `everysec` 最多丢 1 秒；`always` 不丢 |
| **对性能影响** | fork 时有短暂停顿（COW 减小影响）          | 持续写盘有 IO 开销                    |
| **文件可读性** | 不可读（二进制）                       | 可读（文本命令）                       |
| **适用场景**  | 备份、灾备、全量复制                     | 数据安全性要求高                       |

---

## 二、RDB 详解

### 2.1 触发条件

```bash
# redis.conf 配置
save 900 1      # 900秒内至少1次修改 → 触发 bgsave
save 300 10     # 300秒内至少10次修改 → 触发 bgsave
save 60 10000   # 60秒内至少10000次修改 → 触发 bgsave
```

触发方式有三种：

| 触发方式     | 命令                      | 是否阻塞  | 说明                                |
| -------- | ----------------------- | ----- | --------------------------------- |
| **手动同步** | `save`                  | 阻塞主线程 | 主线程直接执行，期间不处理任何命令。**生产禁用**        |
| **手动异步** | `bgsave`                | 不阻塞   | fork 子进程生成快照，**生产推荐**             |
| **自动触发** | 配置 `save`               | 不阻塞   | 满足条件自动执行 bgsave                   |
| **其他触发** | `shutdown` / `flushall` | —     | 关闭时执行 save；flushall 前自动 save（若开启） |
| **主从复制** | 从连主                     | 不阻塞   | 主节点执行 bgsave 发给从节点                |

### 2.2 bgsave 流程

```
主进程（处理命令）
  │
  ├─ fork() ──────────────────────→ 子进程
  │  (COW 机制，共享内存页)              │
  │  │                               ├─ 遍历所有键空间，序列化数据
  │  │  继续处理客户端命令               ├─ 写入临时 RDB 文件
  │  │  （写操作触发 COW 复制）          ├─ 写入完毕，原子 rename 替换旧文件
  │  │                               └─ 子进程退出，通知主进程
  │  │
  ▼  ▼
  N 秒后 RDB 文件生成完成
```

> [!important] fork 阻塞
> `fork()` 操作本身会阻塞主线程，但时间极短（通常微秒级）。真正的耗时是子进程写文件，这期间主进程照常工作。

### 2.3 COW（Copy On Write）机制 —— 核心中的核心

```
fork 之后的内存布局：

┌─────────────────────────────────────────────────────┐
│              物理内存页（共享）                        │
│  [页0]  [页1]  [页2]  [页3]  [页4]  [页5]  ...       │
│    ↑      ↑      ↑      ↑      ↑      ↑              │
│    └──────┴──────┴──────┴──────┴──────┘              │
│         父进程 & 子进程 都指向同一页                     │
│         这些页被标记为"只读"                            │
└─────────────────────────────────────────────────────┘

父进程要修改 [页2] 时：

┌─────────────────────────┐  ┌─────────────────────────┐
│    父进程（新的页2）       │  │    子进程（旧的页2）       │
│  [页0][页1][页2*][页3]   │  │  [页0][页1][页2][页3]     │
│                          │  │                          │
│  页2* = 操作系统复制的新页   │  │  页2 = 原始页面           │
│  父进程在新页上修改          │  │  子进程继续读旧页          │
└─────────────────────────┘  └─────────────────────────┘

只有被修改的页才真正复制 → 内存开销很小
```

> 🧠 **COW 就像考试时老师发卷子**
>
> 所有人拿到的是一样的卷子（共享）。某同学要在卷子上打草稿时，老师给他一张**新的草稿纸**（复制），不影响原卷子内容。其他人继续看原卷子不受影响。
>
> - 不写不改 = 完全不复制（零内存开销）
> - 修改得越多 = 复制得越多（但通常远小于全量内存）
> - 子进程始终看到的是 fork 那一瞬间的快照（一致性保证）

**COW 三大关键点：**

| 关键点                | 说明                                                |
| ------------------ | ------------------------------------------------- |
| **fork 前预留内存**     | 虽然有 COW，但如果父进程修改了大量数据，仍需额外内存。建议预留 50% 内存空间        |
| **避免频繁 bgsave**    | 如果父子进程同时大量写操作，触发大量 COW 复制，可能导致内存翻倍                |
| **大 value 写操作最危险** | 一个 value 占用多个页，修改一条数据可能触发多页复制                     |
| **透明大页（THP）问题**    | Linux 透明大页会导致 COW 复制整个 2MB 大页，即使只改了 1 字节。建议关闭 THP |

```bash
# 关闭透明大页（生产环境建议）
echo never > /sys/kernel/mm/transparent_hugepage/enabled
```

### 2.4 RDB 优缺点

> [!tip] RDB 优点
> - **文件紧凑**：压缩的二进制，适合备份和灾难恢复
> - **恢复极快**：直接 load 到内存，比 AOF 逐条回放快几个数量级
> - **对性能影响小**：bgsave fork 子进程，主进程继续处理命令
> - **适合大数据量场景**：全量数据快照，适合冷备、异地容灾

> [!warning] RDB 缺点
> - **数据丢失窗口**：两次快照之间的数据可能丢失（取决于 save 配置）
> - **fork 开销**：内存越大，fork 越慢（虽然有 COW），巨型实例可能卡住数秒
> - **频繁全量写盘**：IO 压力大（相比 AOF 增量追加）

## 三、AOF 详解

### 3.1 AOF 写流程

```
客户端写命令 → Redis 主线程
                    │
                    ├─ ① 执行命令（修改内存数据）
                    │
                    ├─ ② 把命令追加到 AOF 缓冲区（aof_buf）
                    │
                    ├─ ③ 把缓冲区的命令写入 AOF 文件
                    │      │
                    │      ├─ always:    每条命令都 write + fsync
                    │      ├─ everysec:  每秒 write + fsync 一次（后台线程）
                    │      └─ no:        只是 write，由 OS 决定何时 fsync
                    │
                    └─ ④ 响应客户端
```

> [!important] write vs fsync 的区别
> - `write`：数据从用户态缓冲区写入内核页缓存（Page Cache），快到微秒级
> - `fsync`：把内核页缓存的数据强制刷到磁盘，**这才是真正的落盘**，HDD 需要毫秒级
> - **Redis 的"刷盘"指的是 fsync**，三个策略的区别就在于 fsync 的频率

### 3.2 三种刷盘策略

| 策略 | 行为 | 数据安全性 | 性能 | 适用场景 |
|------|------|-----------|------|----------|
| `always` | 每条写命令都 fsync | 不丢数据（最高安全） | 最低（HDD 只有几百 QPS） | 金融、支付等极致安全场景 |
| **`everysec`** | 每秒 fsync 一次 | 最多丢 1 秒数据 | 折中（**推荐**） | **绝大多数生产场景** |
| `no` | 只 write 不主动 fsync，OS 决定 | 可能丢 30 秒以上数据 | 最高 | 纯缓存、数据可丢失 |

> [!tip] 如何选择？
> ```
> 金融/支付    → always  （数据完整第一，性能第二）
> 一般业务    → everysec（鱼与熊掌兼得，生产首选）
> 纯缓存      → no      （丢了也无所谓，追求极致 QPS）
> ```

**everysec 的实现细节：**

```
主线程：写命令 → 追加到 aof_buf → write 到内核页缓存
                                         │
后台线程（每秒）：───────────────────────┘
  → fsync() 把页缓存刷到磁盘
  → 如果上次 fsync 还没完成（磁盘慢），本次跳过
  → 保证每秒最多刷一次，不阻塞主线程
```

### 3.3 AOF 重写（压缩）

**问题：** AOF 文件随时间增长，里面充斥着大量冗余命令。

```
重写前（AOF 膨胀）：              重写后（精简）：
SET key a                        SET key c
SET key b           →            只保留最终值！
SET key c
INCR counter                     SET counter 3
INCR counter        →
INCR counter
LPUSH list 1 2 3                 LPUSH list 1 2 3 4 5 6
LPUSH list 4 5 6    →
```

**AOF 重写过程（bgrewriteaof）：**

```
主进程
  │
  ├─ fork() ─────────────────────→ 子进程
  │  (COW 机制)                       │
  │  │                               ├─ 遍历所有键空间
  │  │  写命令                        ├─ 每个 key 只生成一条"当前状态"命令
  │  │    │                          ├─ 写入新的 AOF 文件
  │  │    ├─ 执行命令（修改内存）       │
  │  │    └─ 写入 AOF 重写缓冲区       │
  │  │           （记录子进程期间的增量）  │
  │  │                               │
  │  │  子进程完成 → 通知主进程         │
  │  │  ←────────────────────────────┘
  │  │
  │  ├─ 把 AOF 重写缓冲区的内容追加到新文件末尾
  │  ├─ 原子 rename 替换旧 AOF 文件
  │  └─ 继续使用新 AOF 文件
```

> [!important] 重写期间的数据一致性
> 子进程读的是 fork 瞬间的内存快照（相当于 RDB），但主进程在子进程写文件期间仍可能接收新命令。这些**增量命令**被写入 AOF 重写缓冲区，等子进程完成后追加到新文件末尾，保证不丢数据。

**自动触发重写条件：**

```bash
auto-aof-rewrite-percentage 100   # 当前 AOF 文件比上次重写后增长了 100%
auto-aof-rewrite-min-size 64mb    # 且 AOF 文件至少 64MB
```

### 3.4 AOF 优缺点

> [!tip] AOF 优点
> - **数据安全性高**：everysec 最多丢 1 秒，always 不丢
> - **文件可读性强**：文本格式，出问题时可以人工修复
> - **写入性能稳定**：追加写（顺序写），比 RDB 随机写快

> [!warning] AOF 缺点
> - **文件体积大**：同等数据量下 AOF 远大于 RDB
> - **恢复速度慢**：逐条回放命令，大数据量下比 RDB 慢很多
> - **持续 IO 开销**：每次写命令都要追加日志
> - **重写开销**：fork + 写盘，和 bgsave 类似

## 四、混合持久化（Redis 4.0+）

### 4.1 设计思路

RDB 恢复快但丢数据，AOF 安全但恢复慢。**混合持久化取长补短：**

```
混合 AOF 文件结构：

┌────────────────────┬────────────────────────────────────────────────┐
│                    │                                                │
│    RDB 二进制部分    │         AOF 增量部分（文本）                     │
│    (fork 时刻的快照)  │         (快照之后的写命令, AOF 格式)              │
│                    │                                                │
│    恢复速度极快       │         补最新数据，确保不丢                     │
│                    │                                                │
└────────────────────┴────────────────────────────────────────────────┘

恢复流程：
  Step 1: 加载 RDB 头部（二进制快照）→ 瞬间恢复大部分数据
  Step 2: 逐条回放 AOF 尾部（少量的最新命令）→ 补上 fork 后的增量
  → 最终效果：RDB 的速度 + AOF 的安全性
```

### 4.2 启用混合持久化

```bash
# redis.conf
aof-use-rdb-preamble yes   # 开启混合持久化（4.0+ 默认开启）
```

**混合持久化的触发：** 在执行 AOF 重写（`bgrewriteaof`）时，子进程不再逐条写 AOF 命令，而是先写 RDB 二进制快照到 AOF 文件头部，再把重写缓冲区的增量命令以 AOF 格式追加到尾部。

### 4.3 混合模式文件示例

```
文件内容示意：

REDIS0009ú...           ← RDB 二进制头部（大部分数据）
*2\r\n$6\r\nSELECT\r\n$1\r\n0\r\n     ← AOF 格式增量
*3\r\n$3\r\nSET\r\n$3\r\nkey\r\n...   ← AOF 格式增量
```

### 4.4 三种方案终极对比

| 维度 | 纯 RDB | 纯 AOF | 混合持久化（推荐） |
|------|--------|--------|-------------------|
| **文件大小** | 小 | 大 | 中 |
| **恢复速度** | 极快 | 慢 | 快 |
| **数据安全** | 可能丢几分钟 | everysec 丢 1 秒 | everysec 丢 1 秒 |
| **CPU 开销** | fork + 写盘 | 持续追加 | fork + 写盘 |
| **版本要求** | 全部 | 全部 | 4.0+ |
| **文件可读性** | 不可读 | 可读（纯 AOF） | 半可读（头部二进制 + 尾部文本） |

## 五、选型建议

```
数据可以丢几分钟
  └─ → 纯 RDB（省资源，适合备份/灾备/纯缓存）

数据不能丢
  └─ → RDB + AOF 同时开启
        └─ → Redis 4.0+：开启混合持久化（最优方案）
              ├─ aof-use-rdb-preamble yes
              ├─ appendonly yes
              └─ appendfsync everysec

极致安全（金融/支付）
  └─ → 混合持久化 + appendfsync always
        └─ → 但注意性能代价（SSD 可缓解）
```

### 生产环境推荐配置

```bash
# redis.conf 持久化相关推荐配置

# RDB（备用，虽然混合模式下 AOF 为主）
save 900 1
save 300 10
save 60 10000
stop-writes-on-bgsave-error yes   # bgsave 失败时停止写入，避免不知情
rdbcompression yes                # 压缩 RDB 文件
rdbchecksum yes                   # 校验和

# AOF
appendonly yes                    # 开启 AOF
appendfsync everysec              # 每秒刷盘
auto-aof-rewrite-percentage 100   # 增长 100% 触发重写
auto-aof-rewrite-min-size 64mb    # 至少 64MB

# 混合持久化
aof-use-rdb-preamble yes          # 开启混合持久化
```

> [!warning] 重要提醒
> - 如果同时开启 RDB 和 AOF，**Redis 重启时优先用 AOF 恢复**（因为 AOF 数据更完整）
> - 如果启用了混合持久化，AOF 文件中包含 RDB 头部，恢复时会自动识别
> - 关闭 THP（透明大页）是生产必须，避免 COW 时内存翻倍

## 面试满分话术

> "Redis 持久化我理解有三个层次：
>
> **第一层，RDB 快照**——它是某一时刻内存的二进制全量快照。优点是文件小而紧凑，恢复快（直接加载二进制到内存），适合做备份和全量复制。缺点是两次快照之间的数据可能丢失。触发方式有 save（阻塞，禁用）和 bgsave（fork 子进程，生产用）。配置如 `save 900 1` 表示 900 秒内至少 1 次修改就触发 bgsave。
>
> 这里有个关键机制叫 COW（写时复制）。fork 子进程时父子共享同一份物理内存页，都标记为只读。只有当主进程要修改某个页时，操作系统才复制该页给主进程修改，子进程继续读旧页。这样既保证了子进程看到的是一份一致性快照，又最大程度节省了内存。我用一个类比来记：就像**考试发卷子**——所有人共享同一份卷子，有人要打草稿老师才给一张新草稿纸，不影响原卷子。
>
> **第二层，AOF 日志**——它记录所有写命令。有三种刷盘策略：`always` 每条命令都 fsync，不丢数据但性能最低；`everysec` 每秒 fsync，最多丢 1 秒数据，是生产环境的推荐选择；`no` 交给操作系统，性能最高但不安全。AOF 的问题是文件会不断膨胀，所以有 AOF 重写机制：fork 子进程读取当前内存数据，把每个 key 的多条历史命令精简为一条当前状态的命令。重写期间主进程的增量命令写到一个重写缓冲区，子进程完成后追加到新文件末尾，保证不丢数据。
>
> **第三层，混合持久化（4.0+）**——它是 RDB 和 AOF 的融合方案。在执行 AOF 重写时，文件的头部存放 RDB 二进制快照，尾部存放后续增量 AOF 命令。恢复时先快速加载 RDB 头部，再回放少量 AOF 尾部。这样就同时获得了 RDB 的快速恢复和 AOF 的数据安全性，是当前生产环境的最佳实践。
>
> 在实际配置上，我会开启 AOF + everysec + 混合持久化，同时保留 RDB 配置作为备份。另外一定要关闭 Linux 的透明大页（THP），防止 COW 时内存翻倍的问题。"

## 记忆口诀

> [!tip] 口诀一：持久化三剑客
> **快照 RDB 恢复快，日志 AOF 更安全。COW 复制在修改，混合持久两边来。**
>
> RDB=二进制快照(快但有丢失窗口)；AOF=命令日志(安全但文件大)；COW=写时复制(fork 时共享，修改时才复制)；混合=RDB 头+AOF 尾(4.0+，速度+安全)

> [!tip] 口诀二：AOF 刷盘三策略
> **always 条条落，everysec 秒秒刷，no 靠 OS。生产推荐 everysec，一秒丢数据。**
>
> always=每条 fsync(安全但慢)；everysec=每秒 fsync(折中，推荐)；no=OS 决定(快但不安全)

> [!tip] 口诀三：COW 核心
> **fork 共内存，只读双共享。父写才复制，子读不动摇。**
>
> fork 后父子共享只读内存页 → 父进程修改时 OS 复制新页 → 子进程始终读旧页 → 一致性快照 + 省内存

> [!tip] 口诀四：选型速记
> **能丢用 RDB，不丢开 AOF。混合最推荐，4.0 以上行。everysec 日常用，always 金融选。THP 关了才放心。**
>
> 纯缓存→RDB；数据安全→AOF；两者都要→混合持久化；生产标配=everysec+混合+关闭THP
