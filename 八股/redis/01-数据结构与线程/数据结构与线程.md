# Redis 数据结构与线程模型

> [!summary] 本章速记
> **两大核心：数据结构（5 种类型 + 底层实现）与线程模型（单线程 + epoll + 6.0 IO 多线程）。数据结构决定 Redis 能做什么，线程模型决定 Redis 为什么这么快。**

---

## 一、5 大数据类型速查表 + 底层结构

| 类型         | 底层实现                               | 适用场景                            | 一句话定位 |
| ---------- | ---------------------------------- | ------------------------------- | ----- |
| **String** | SDS（简单动态字符串）                       | 缓存 value、计数器、分布式锁 token、session | 万能基石  |
| **List**   | quicklist（双向链表 + ziplist 混合结构）     | 消息队列（左进右出）、最新列表、时间线             | 火车运人  |
| **Set**    | hashtable / intset（纯整数时用 intset）   | 共同好友、标签系统、抽奖去重、交并差运算            | 不重复的兜 |
| **Hash**   | hashtable / ziplist（小 hash 时用压缩列表） | 对象属性缓存（如用户信息：name/age/email）    | 对象的衣柜 |
| **ZSet**   | skiplist（跳表） + hashtable（字典）       | 排行榜、延时队列（score 存时间戳）、带权重的集合     | 高速排行  |
|            |                                    |                                 |       |

> [!tip] 记忆技巧
> 五种类型像五件兵器：**String 是长剑（通用主战）、List 是九节鞭（一节一节伸缩自如）、Set 是盾（去重防御）、Hash 是盔甲（包裹对象）、ZSet 是弓（精确瞄准 score）**。

## 二、String 底层：SDS（Simple Dynamic String）

Redis 不用 C 语言的 `\0` 结尾字符串，而是自己设计了 SDS 结构。

### 2.1 SDS 结构图示

```
┌──────┬──────┬─────────────────────┐
│ len  │ free │ buf[]               │
│  5   │  5   │ 'h' 'e' 'l' 'l' 'o' │
└──────┴──────┴─────────────────────┘
```

**三个核心字段：**
- **len**：已使用的字节数（5），O(1) 获取长度
- **free**：预分配的空闲空间（5），减少内存重分配次数
- **buf[]**：实际的字符数组，存储二进制数据

### 2.2 SDS 三大优势

| 特性         | C 字符串              | SDS                   | 为什么重要           |
| ---------- | ------------------ | --------------------- | --------------- |
| **获取长度**   | O(n) 遍历            | O(1) 直接读 `len` 字段     | `STRLEN` 命令毫无压力 |
| **拼接安全**   | 需手动 `realloc`，可能溢出 | 预分配空间（`free` 字段），自动扩容 | 不会缓冲区溢出         |
| **二进制安全**  | 遇到 `\0` 截断         | 用 `len` 判断结尾，可存图片/音频  | 能存任意二进制数据       |
| **内存分配次数** | 每次修改都要 `realloc`   | 预分配 + 惰性释放，减少系统调用     | 减少碎片和系统开销       |

> [!tip] 为什么 SDS 用 `free` 字段？
> **预分配策略**：当 `buf` 长度 < 1MB 时，`free = len`（扩容到 2 倍）；当 >= 1MB 时，`free = 1MB`。这避免了频繁 `malloc/realloc`。
>
> 比喻：就像**租房时一次性多交两个月押金**，而不是每次用水用电都去交钱。虽然预占了一些空间，但省去了频繁的"去银行"（系统调用）。

### 2.3 SDS 扩容与惰性释放

```
扩容：APPEND 字符串 → 空间不够 → 计算新长度 → 预分配 free → 执行拼接
惰性释放：DEL 字符串 → 不立即释放内存 → free 字段变大 → 留给下次使用
```

> 潜意识：**SDS 的策略是"用空间换时间"**。Redis 是内存数据库，内存不是瓶颈，CPU 才是。多占点内存没关系，关键是别让 `malloc` 拖慢速度。

## 三、List 底层：quicklist

### 3.1 演进路线

```
早期 Redis：linkedlist（双向链表） + ziplist（压缩列表）两种结构，单独使用各有弊端
    ↓
Redis 3.2+：统一为 quicklist，取两者之长
```

### 3.2 quicklist 结构

```
quicklist 节点 1          quicklist 节点 2
┌─────────────────┐     ┌─────────────────┐
│    ziplist       │ ◄──►│    ziplist       │
│  [A,B,C,D,E]     │     │  [F,G,H,I,J]     │
└─────────────────┘     └─────────────────┘
         ↑                       ↑
    每个节点是一个               双向指针连接
    ziplist（默认8KB）
```

**quicklist = 双向链表（宏观结构） + ziplist 压缩列表（微观存储）**

| 对比维度 | 纯 linkedlist                | 纯 ziplist                  | quicklist          |
| ---- | --------------------------- | -------------------------- | ------------------ |
| 内存效率 | 差（每个节点有 prev/next 指针，2x8字节） | 好（连续内存）                    | **好**（折衷）          |
| 插入性能 | 好（O(1) 修改指针）                | 差（大 ziplist 中间插入需 memmove） | **好**（拆成小 ziplist） |
| 删除性能 | 好                           | 差                          | **好**              |

### 3.3 核心类比

> [!important] quicklist = 火车运人
> **每节车厢（链表节点）里坐着一排人（ziplist compact 存储），车厢之间用挂钩（双向指针 prev/next）连接。**
>
> - 要找人？先定位车厢（O(1) 跳转），再在车厢里一个一个看（遍历 ziplist）
> - 要加人？塞进对应车厢就行，塞不下了新开一节车厢
> - 比每个人单独开车（纯 linkedlist）省路费（内存），比所有人挤一辆大巴（纯 ziplist）灵活高效

## 四、Set / Hash 底层结构与转换

### 4.1 双底层设计

| 类型       | 小数据量          | 大数据量（或元素够长） | 转换条件                          |
| -------- | ------------- | ----------- | ----------------------------- |
| **Set**  | intset（整数集合）  | hashtable   | 元素全为整数 **且** 个数 <= 512        |
| **Hash** | ziplist（压缩列表） | hashtable   | 键值对长度 <= 64字节 **且** 个数 <= 512 |

> 潜意识：**Redis 用"小结构省内存，大结构换性能"的策略**。数据量小时用紧凑结构省内存，数据量大时自动切换到标准哈希表保证操作效率。

### 4.2 intset 是什么

```
intset 内存布局：
┌──────────┬──────────┬─────────────────────────┐
│ encoding │ length   │ [v1][v2][v3][v4]...[vn]   │
│ 编码方式  │ 元素个数  │ 连续内存，由小到大排列      │
└──────────┴──────────┴─────────────────────────┘
```

- **本质**：一块连续内存的整数数组，元素按从小到大排列
- **查找**：二分查找 O(log n)
- **优势**：极大节省内存（省去 hashtable 的指针开销和哈希冲突空间）
- **限制**：只能存整数，一旦插入字符串就升级为 hashtable

### 4.3 渐进式 rehash —— 搬家比喻

> [!important] 为什么叫"渐进式"？
> Redis 是单线程，如果一次性把所有 key 从旧哈希桶搬到新哈希桶，期间整个 Redis 会卡住（CPU 阻塞在搬数据上），所有客户端请求全部等待。
>
> **解决方案**：每次操作时只搬几个桶，"顺便"完成迁移。
>
> **就像搬家不是一天全部搬完，而是每天下班顺路带几件东西去新家。** 旧居（旧哈希表）还有东西但越来越少，新家（新哈希表）慢慢充实，一段时间后旧居彻底清空。

```
渐进式 rehash 流程：
┌───────────┐     ┌───────────┐
│ 旧 ht[0]   │     │ 新 ht[1]   │
│ (ht[0])   │ ──→ │ (ht[1])   │
└───────────┘     └───────────┘
      ↓                  ↑
  每次操作（增删改查）带几个桶过去

操作时：
  - 查询：先查 ht[0]，没找到再查 ht[1]
  - 新增：直接写入 ht[1]
  - 搬运完 → ht[1] 变为 ht[0] → 释放旧表
```

## 五、ZSet 跳表 —— 重点中的重点

### 5.1 ZSet 的双重人格

ZSet 同时使用**两套数据结构**指向同一份数据：

| 结构                | 职责                   | 时间复杂度    |
| ----------------- | -------------------- | -------- |
| **skiplist（跳表）**  | 按 score 排序，支持范围查询    | O(log n) |
| **hashtable（字典）** | 按 member 快速查找其 score | O(1)     |


> 就像**图书馆的两种索引**：一张是按书名首字母排序的目录（skiplist，适合查找"A-H 开头的所有书"），一张是精确索书号（hashtable，快速找到某一本具体书）。

### 5.2 跳表结构 —— 高速公路比喻

```
                           跳表 = 高速公路 ✈

Level 3:  1 ────────────────────────→ 15         高速公路（最少出口，跑得最快）
                           ↓                     ↓
Level 2:  1 ────────→ 7 ───────────→ 15         国道（略多一些出口）
                           ↓           ↓           ↓
Level 1:  1 → 3 → 5 → 7 → 9 → 12 → 15         省道（全量数据，每个节点都在）

索引节点（高速层）          普通数据节点（省道层）
```

> [!important] 高速公路比喻
> **底层（Level 1）是省道**：每个节点都在，全量数据，一个一个走最慢。
> **上层（Level N）是高速**：只有少数"出口节点"，跳过中间直达远方，最快。
>
> **查找过程**：先上高速狂奔到目标附近 → 快到的时候下高速 → 走省道精确定位。**又快又准！**
>
> 就像开车去一个远方的城市：先在高速上以 120km/h 冲到城市附近 → 下匝道 → 走市区小路精确找到目的地。

### 5.3 查找过程演示（找 9）

```
Level 3: 1 → 15 → "15 比 9 大，下降！"
Level 2: 1 → 7 → 15 → "7 比 9 小继续，15 比 9 大，下降！"
Level 1: 7 → 9 ✅ 找到！

跳了 4 步找到目标，而省道上需要走 5 步（1→3→5→7→9）。
数据量越大，优势越明显：10 万个节点省道上要走一半（5万步），高速最多走 log(100000) ≈ 17 步。
```

### 5.4 核心原理

1. **多层索引**：底层（Level 1）包含所有数据，每往上一层节点数约减半
2. **层数概率**：每个节点随机生成层数 —— 1/2 概率升到 L2，1/4 到 L3，1/8 到 L4 ...
3. **查找算法**：从最高层开始向右查找，遇到比目标大的就下降一层
4. **复杂度**：时间 O(log n)，空间 O(n)（额外索引约等于 n 个节点）

### 5.5 为什么不用红黑树？—— 四大理由

| 理由        | 跳表                        | 红黑树               | 谁更优             |
| --------- | ------------------------- | ----------------- | --------------- |
| **实现复杂度** | 只需调整指针，逻辑直观               | 需要处理旋转、变色、多种 case | **跳表**          |
| **范围查询**  | 找到起点后顺着底层链条走 O(log n + m) | 需要中序遍历，不直观        | **跳表**          |
| **并发友好**  | 局部修改，只影响相邻几个节点            | 旋转可能影响整棵树         | **跳表**          |
| **场景适配**  | 内存操作，跳表效率足够               | 磁盘场景下有优势（减少IO）    | Redis 场景下**跳表** |

> [!tip] 面试一句话
> "跳表实现简单（不用旋转）、范围查询天然高效（顺链走就行）、局部修改并发友好。而红黑树设计初衷是减少磁盘 IO（树矮），Redis 是纯内存数据库不存在磁盘 IO 问题，所以跳表是更优选择。"

## 六、线程模型

### 6.1 核心认知：单线程不等于慢

> [!important] 一句话总结
> **Redis 单线程处理命令（CPU 不是瓶颈），epoll IO 多路复用同时监听多个客户端连接，后台线程处理耗时操作（AOF 刷盘 / RDB 持久化 / 异步删除）。6.0+ 引入 IO 多线程只处理网络读写，命令执行仍然单线程保证原子性。**

### 6.2 为什么单线程还这么快？

| 原因                 | 解释                         | 比喻                |
| ------------------ | -------------------------- | ----------------- |
| **纯内存操作**          | 读写都在内存中，微秒级响应              | 直接在脑子里算，不需要翻书（磁盘） |
| **IO 多路复用（epoll）** | 一个线程同时监听成千上万个 socket       | 一个调度员同时指挥 100 个工人 |
| **避免上下文切换**        | 多线程切换有 CPU 开销 + 竞争锁开销      | 一个人专注干活 vs 一群人抢工位 |
| **数据结构简单**         | 专门优化的 SDS、ziplist、skiplist | 用顺手的工具，不玩花活       |

> [!tip] 瓶颈究竟在哪？
> 单机 Redis QPS 可达 **10 万+**，瓶颈在**网络 IO 带宽**，不在 CPU。就像**百米短跑**——跑道够宽（IO 带宽），一个人跑（单线程）反而比一群人挤来挤去（多线程加锁 + 上下文切换）跑得更快更稳。

### 6.3 IO 多路复用 —— epoll 三板斧

```
┌──────────────────────────────────────────────────────┐
│                    Redis 主线程                        │
│                                                      │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐       │
│   │  epoll    │ → │  事件    │ → │  命令    │       │
│   │  监听     │    │  分发器  │    │  执行    │       │
│   └──────────┘    └──────────┘    └──────────┘       │
│        ↑                                ↓             │
│   socket1 socket2  ... socketN      返回结果          │
└──────────────────────────────────────────────────────┘

epoll_wait 监听所有 socket → 有数据到达 → 读数据 → 执行命令 → 写响应
```

**epoll 三板斧：**

| 调用 | 作用 | 比喻 |
|------|------|------|
| `epoll_create` | 创建一个 epoll 实例 | 建立一个"监控室" |
| `epoll_ctl` | 注册/修改要监听的文件描述符（add/mod/del） | 把摄像头接到监控室 |
| `epoll_wait` | 阻塞等待就绪事件，返回了就处理 | 值班看屏幕，哪个动就处理哪个 |

### 6.4 epoll vs select vs poll

| 维度          | select        | poll          | epoll             |
| ----------- | ------------- | ------------- | ----------------- |
| **fd 数量限制** | 最多 1024（硬限制）  | 无限制           | 无限制               |
| **扫描方式**    | 全量遍历 O(n)     | 全量遍历 O(n)     | **只返回就绪** O(1)    |
| **内核实现**    | 每次调用重新传入所有 fd | 每次调用重新传入所有 fd | 内核用**红黑树+就绪链表**管理 |
| **适用场景**    | 连接数少          | 连接数中等         | **高并发海量连接**       |

> 比喻：
> - **select**：老师一个一个问全班 50 个学生"你举手了吗"，每次都要问全部人
> - **poll**：同上，但班级没有人数上限（但还是要一个一个问）
> - **epoll**：学生主动举手（事件驱动），老师只关注已经举手的

### 6.5 后台线程（非主线程）

| 线程/进程       | 职责                       | 为什么不能在主线程做            |
| ----------- | ------------------------ | --------------------- |
| **关闭文件**    | `close` 操作               | 文件关闭可能耗时数毫秒，阻塞命令执行    |
| **AOF 刷盘**  | `fsync` 写磁盘              | 磁盘 IO 是百毫秒级操作，远超内存微秒级 |
| **AOF 重写**  | `bgrewriteaof` 压缩 AOF    | 需要遍历全量生成新 AOF，耗时久     |
| **RDB 持久化** | `bgsave` fork 子进程生成快照    | fork 后子进程独立处理，不影响主线程  |
| **异步删除**    | `UNLINK` 异步删除大 key（4.0+） | 主线程只断链，后台回收内存，避免卡顿    |

> 角色分工比喻：**主线程是"前台柜员"**（2 分钟办好一笔业务），**后台线程是"后勤部门"**（花半小时搬货入库）。前台不用自己搬货，否则后面的客户全得排队等着。

### 6.6 Redis 6.0 IO 多线程 —— 餐厅比喻

```
Redis 6.0 之前：
  主线程：读 socket → 执行命令 → 写 socket → 读 socket → ...
         （一个人干三个角色的活）

Redis 6.0 之后：
  IO 线程组：读 socket（多个线程并行解析网络包）
       ↓
  主线程：执行命令（单线程，保持原子性）
       ↓
  IO 线程组：写 socket（多个线程并行发送响应）
```

> [!important] 餐厅比喻（核心）
> **厨房（命令执行）只有一个厨师（主线程），保证每道菜的味道（数据一致性/原子性）不变。**
>
> **前厅有多个服务员（IO 线程）**：同时接待多个客人（并发连接），记录点菜单（读取 socket），上菜（写回响应）。服务员再多，菜怎么做还是厨师一个人说了算。
>
> **默认不开启 IO 多线程**，需要在配置中手动开启。因为对于大多数场景，单个厨师（单线程）配一个跑得飞快的服务员（epoll）已经够用。

## 面试满分话术

### 话术一：5 大数据类型 + 底层结构

> "Redis 有 5 种基本数据类型，每种背后都有精心设计的底层结构：
>
> **String** 的底层是 SDS，不是 C 语言的 \0 字符串。SDS 有三个字段：len（O(1)获取长度）、free（预分配空间，减少 malloc 次数）、buf（实际数据）。三大优势：O(1) 取长度、预分配防溢出、二进制安全。
>
> **List** 底层是 Redis 3.2 统一后的 quicklist，本质是双向链表 + ziplist 的混合结构。每个链表节点里是一个不超过 8KB 的 ziplist，像火车——车厢之间用挂钩连接，车厢里坐着一排人。
>
> **Set** 和 **Hash** 都有两套底层：小数据用紧凑结构（intset/ziplist）、大数据自动切换 hashtable。切换时用渐进式 rehash（每次搬几个桶避免阻塞），就像搬家不是一天搬完。
>
> **ZSet** 是重点，同时用跳表和哈希表——跳表负责按 score 排序和范围查询，哈希表负责按 member 快速定位。跳表像高速公路：上层索引少跑得快，底层全量数据做精确查找，时间 O(log n)。不用红黑树是因为跳表实现简单、范围查询高效、并发友好。"

### 话术二：线程模型

> "Redis 的线程模型我总结为四个层次：
>
> **第一层：单线程处理命令。** Redis 是纯内存操作，CPU 不是瓶颈。单线程避免了锁竞争和上下文切换，配合专门优化的数据结构，QPS 达到 10 万+。
>
> **第二层：IO 多路复用。** 使用 epoll 实现一个线程同时监听成千上万个 socket 连接。epoll 内核用红黑树 + 就绪链表管理 fd，只返回有数据到达的 socket（O(1)），不像 select/poll 那样全量遍历（O(n)）。
>
> **第三层：后台线程。** 对于耗时操作（如 AOF 刷盘、RDB 持久化、异步删除大 key），Redis fork 子进程或使用后台线程处理，避免阻塞主线程。主线程是前台柜员，后台线程是后勤部门。
>
> **第四层：6.0 IO 多线程。** 6.0 版本引入 IO 多线程，但只处理网络数据包的读写（解析和发送），命令执行仍然单线程，保证原子性不变。就像餐厅——多个服务员接待客人，但后厨只有一个厨师做菜。"
>
> "值得注意的是，IO 多线程默认关闭，因为对于绝大部分场景，单线程 + epoll 已经足够快。瓶颈在网卡带宽而非 CPU。"

## 记忆口诀

> [!tip] 口诀一：五大类型底层
> **S String SDS 带缓冲，List 快列火车装。**
> **Set Hash 小压大哈希，ZSet 跳表高速跑。**
>
> String→SDS(len+free+buf 三字段)；List→quicklist(双向链表+ziplist 火车比喻)；
> Set→intset(全整数小量)/hashtable；Hash→ziplist(小键小值)/hashtable；
> ZSet→skiplist+hashtable 双重结构

> [!tip] 口诀二：线程模型
> **单线处理命与令，epoll 监听多路行。**
> **后台线程干重活，六零 IO 多线读。**
>
> 单线程命令执行 + epoll(epoll_create/ctl/wait) IO 多路复用 +
> 后台线程(close/bgsave/bgrewriteaof/unlink) + 6.0 IO 多线程只处理网络读写

> [!tip] 口诀三：跳表四优势
> **实现简单不旋转，范围查询顺链走。**
> **并发友好局部改，内存场景更高效。**
>
> 跳表 vs 红黑树四大优势：①实现简单(只调指针不旋转) ②范围查询O(log n+m)
> ③并发友好(局部修改) ④内存数据库不需要减磁盘IO

> [!tip] 口诀四：渐进式 rehash
> **每次操作搬几个桶，分多次完成不卡壳。**
> **好比搬家不一天搬完，下班顺路带几箱。**
>
> 渐进式 rehash：两个哈希表(ht[0]旧→ht[1]新)，每次CRUD操作顺便搬几个桶，
> 查时先旧后新，增时直入新表。扩容/缩容在无声无息中完成

> [!tip] 口诀五：epoll vs select/poll
> **select 限千全遍历，poll 不限也全扫。**
> **epoll 只回就绪态，红黑链表效率高。**
>
> select(1024限制+O(n)遍历) < poll(无限制但O(n)遍历) < epoll(红黑树+就绪链表，O(1)获取就绪)
